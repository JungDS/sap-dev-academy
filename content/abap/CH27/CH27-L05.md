## 핸들러를 한 곳에

[Double Click Event](CH27-L01.html)부터 [USER_COMMAND 처리](CH27-L04.html)까지 이벤트를
하나씩 배웠다. 하지만 실제 화면엔 더블클릭·hotspot·툴바·user command가 **동시에** 있다.
이벤트 메서드와 `SET HANDLER`가 프로그램 곳곳에 흩어지면 "어떤 이벤트가 어디로 연결되는지"
추적이 어렵다 — 버튼 하나가 안 먹힐 때 *화면 생성 문제인지·핸들러 등록 문제인지·메서드
로직 문제인지* 구분이 안 된다.

그래서 이 챕터의 마지막은 **전용 핸들러 클래스**다. 거대한 클래스를 만드는 게 아니라,
*어떤 이벤트가 어떤 메서드로 연결되는지*를 한 곳에서 보이게 만든다([MVC 기반 Report
구조화](CH26-L04.html)의 Controller가 이 핸들러를 들고 Model·View를 조율하면 더 깔끔하다).

## 한 클래스에 모으기

생성자에서 ALV Grid와 화면 데이터를 받아 두고, **모든 `SET HANDLER`를 한 번에** 건다.

```abap
CLASS lcl_alv_handler DEFINITION.
  PUBLIC SECTION.
    METHODS constructor
      IMPORTING io_grid    TYPE REF TO cl_gui_alv_grid
                it_booking TYPE ztt_booking.
    METHODS on_double_click
      FOR EVENT double_click OF cl_gui_alv_grid
      IMPORTING e_row e_column.
    METHODS on_hotspot
      FOR EVENT hotspot_click OF cl_gui_alv_grid
      IMPORTING e_row_id e_column_id.
    METHODS on_toolbar
      FOR EVENT toolbar OF cl_gui_alv_grid
      IMPORTING e_object.
    METHODS on_user_command
      FOR EVENT user_command OF cl_gui_alv_grid
      IMPORTING e_ucomm.
  PRIVATE SECTION.
    DATA mo_grid    TYPE REF TO cl_gui_alv_grid.
    DATA mt_booking TYPE ztt_booking.
ENDCLASS.

CLASS lcl_alv_handler IMPLEMENTATION.
  METHOD constructor.
    mo_grid    = io_grid.
    mt_booking = it_booking.
    SET HANDLER me->on_double_click
                me->on_hotspot
                me->on_toolbar
                me->on_user_command
      FOR mo_grid.
  ENDMETHOD.
  " on_double_click / on_hotspot / on_toolbar / on_user_command —
  " 각 메서드는 화면 상태를 읽어 업무 서비스로 위임
ENDCLASS.
```

화면 초기화 코드는 **핸들러 객체를 만들기만** 하면 배선이 끝난다.

```abap
DATA(go_handler) = NEW lcl_alv_handler(
  io_grid    = go_grid
  it_booking = lt_booking ).
```

- `mo_grid` — 선택 행 조회·화면 갱신에 필요. `mt_booking` — 화면 데이터 읽기용.
- 핸들러가 *모든 업무 규칙을 직접 갖기보다*, 컨트롤러·서비스 객체에 **위임**하면 유지보수가
  쉽다.

## 이벤트 → 핸들러 배선 살펴보기

왼쪽 이벤트를 클릭하면 어느 핸들러 메서드로 연결되는지, 그 배선이 생성자 `SET HANDLER`
한 곳에서 이뤄지는지 확인해 보라.

## 흔한 실수와 점검

- **핸들러에 업무 로직을 다 밀어넣기** → 클래스가 금방 비대해진다. 핸들러는 "이벤트를 받아
  *화면 상태와 업무 서비스를 잇는 층*"으로 둔다.
- **`SET HANDLER` 일부 누락** → `on_double_click`만 걸고 `on_user_command`를 빠뜨리면
  더블클릭은 되는데 버튼은 무반응. 생성자 한 곳에 모아 두면 누락이 한눈에 보인다.
- **그리드 참조 미보관** → 핸들러가 `mo_grid`를 들고 있어야 선택 조회·갱신이 된다. 동적
  화면이면 해제·재등록 정책도 함께 생각한다.

> ☁️ **ABAP Cloud 경계** — `CL_GUI_ALV_GRID`·SAP GUI Control Framework는 **Classic ABAP /
> SAP GUI** 기술이다. ABAP Cloud(ABAP for Cloud Development)는 released API·RAP·Fiori
> 중심이라 SAP GUI 접근이 없다 — Cloud에선 RAP/UI5의 이벤트·액션 설계로 넘어간다.

## 🧩 도전 과제 (서브퀘스트)

> 선택 과제 — 예매 목록 ALV에 상호작용을 더한다.

1. 예매 행을 **더블클릭**하면 그 예매 상세를 띄운다([Double Click](CH27-L01.html)).
2. 툴바에 "예매 취소" 버튼을 더하고([Toolbar](CH27-L03.html)), 선택 행을 취소
   처리한다([USER_COMMAND](CH27-L04.html)).
3. (심화) 공연ID 컬럼을 **hotspot**으로 만들어 한 번 클릭으로 공연 상세로([Hotspot](CH27-L02.html)).
4. 위 핸들러들을 **한 클래스**로 모아 생성자에서 `SET HANDLER`로 배선한다.

## 정리

- 이 챕터의 완성형은 이벤트 메서드를 많이 아는 게 아니라, **이벤트 발생 시점과 배선을 설명할
  수 있는 것**.
- `double_click`=행/셀 더블클릭, `hotspot_click`=링크 셀 한 번 클릭, `toolbar`=버튼 추가,
  `user_command`=버튼 클릭 후 명령 처리.
- 이 넷을 **한 핸들러 클래스**에서 `SET HANDLER`로 연결하면 ALV 화면의 사용자 반응 구조가
  선명해진다.
