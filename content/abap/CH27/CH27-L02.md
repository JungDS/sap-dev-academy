## 단 하나여야 하는 객체

설정·[[캐시]]·연결처럼 **프로그램에 하나만** 있어야 하는 객체가 있다. 매번 NEW하면
상태가 분산된다. **Singleton**은 인스턴스를 하나로 보장한다.

```abap
CLASS zcl_config DEFINITION PUBLIC CREATE PRIVATE.
  PUBLIC SECTION.
    CLASS-METHODS get_instance RETURNING VALUE(ro) TYPE REF TO zcl_config.
  PRIVATE SECTION.
    CLASS-DATA go_instance TYPE REF TO zcl_config.
ENDCLASS.

CLASS zcl_config IMPLEMENTATION.
  METHOD get_instance.
    IF go_instance IS INITIAL.
      go_instance = NEW #( ).        " 최초 1회만 생성
    ENDIF.
    ro = go_instance.
  ENDMETHOD.
ENDCLASS.
```

- `CREATE PRIVATE` — 외부에서 `NEW` 금지(생성 통제).
- `get_instance( )` — 항상 같은 인스턴스를 돌려준다.

## 같은 하나임을 직접 확인

`get_instance()`를 여러 번 부르면 **같은 주소**(인스턴스 1개)가, `NEW`를 여러 번 하면
**매번 다른 주소**가 나오는 걸 비교해 보라.

## 흔한 실수와 점검

- **`CREATE PRIVATE` 누락** → 외부에서 `NEW`가 가능해 '하나' 보장이 깨진다(`get_instance`만으론 부족).
- **사용자별·요청별·트랜잭션별 상태를 Singleton에** → 다른 사용자/실행 흐름과 섞인다. 그런 상태는 user/session/
  request 범위가 필요하지 Singleton이 아니다.
- **"하나면 편하다"로 남용** → *하나면 편하다*와 *반드시 하나여야 한다*는 다르다. 모든 service class를
  Singleton으로 만들지 말고, 진짜 단일성이 필요한 설정/공유 자원에만 쓴다(Clean ABAP도 singleton 최소화 권장).
- **테스트 고려 안 함** → Singleton은 한 번 만들어진 상태가 테스트 사이에 남는다. 테스트가 필요한 class면
  interface 분리·생성자 주입(DI)을 검토한다 → [Testable Class 설계와 ABAP Unit](CH27-L05.html).

## 정리

- Singleton = 인스턴스를 **단 하나**로 보장(`get_instance()` + `CREATE PRIVATE`).
- 설정·캐시·연결처럼 상태가 한 곳에 모여야 할 때 유용 — 단, 남용 금지.
