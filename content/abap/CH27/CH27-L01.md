## NEW를 여기저기 흩뿌리면

코드 곳곳에서 `NEW zcl_...( )`로 객체를 만들면, 생성 방식이 바뀔 때(예: 종류에 따라
다른 클래스) 전부 고쳐야 한다. 생성을 **한 곳(팩토리)** 에 모은다.

```abap
CLASS zcl_booking_factory DEFINITION PUBLIC.
  PUBLIC SECTION.
    CLASS-METHODS create
      IMPORTING iv_type       TYPE c
      RETURNING VALUE(ro_obj) TYPE REF TO zif_booking.
ENDCLASS.

CLASS zcl_booking_factory IMPLEMENTATION.
  METHOD create.
    CASE iv_type.
      WHEN 'V'.    ro_obj = NEW zcl_vip_booking( ).      " VIP 예매
      WHEN OTHERS. ro_obj = NEW zcl_booking_manager( ).  " 일반 예매
    ENDCASE.
  ENDMETHOD.
ENDCLASS.
```

- 호출부는 `zcl_booking_factory=>create( 'V' )`만 — 어느 클래스인지 몰라도 된다.
- 인터페이스(`zif_booking`) 반환으로 [다형성 — CAST와 CASE TYPE OF](CH21-L08.html)과 결합: static type은
  `zif_booking`이지만 runtime object는 `zcl_vip_booking`/`zcl_booking_manager`다.

## 타입을 바꿔 보기

타입을 바꿔도 **호출부 코드는 그대로**인데, 팩토리가 만들어 주는 **구체 클래스**만 달라지는
걸 직접 확인해 보라.

## 흔한 실수와 점검

- **팩토리 우회** → 어떤 곳은 직접 `NEW`, 어떤 곳은 팩토리 → 일관성 깨짐. 생성은 팩토리로 일원화(팀 규칙으로).
- **거대 팩토리** → 예매·결제·할인·로그가 한 팩토리에 모이면 잡동사니 창고가 된다. 업무 영역별로 나눈다.
- **팩토리에 업무 로직** → 팩토리는 *생성 규칙*만. 잔여석 계산·결제 승인 같은 업무 동작은 *생성된 객체*의 책임이다.
- **알 수 없는 타입을 조용히 기본값으로** → `OTHERS`를 일반 예매로 보내면 잘못된 코드 값을 숨긴다. 타입이
  customizing/DDIC check table에서 온다면 **유효성 검사 먼저**, 아니면 예외를 던질지 정책을 정한다.

## 정리

- 흩뿌린 `NEW`를 **한 곳(팩토리)** 으로 모은다 → 생성 규칙 변경이 한 곳만.
- 팩토리는 **인터페이스**를 반환 → 호출부는 구체 클래스를 몰라도 됨(다형성).
