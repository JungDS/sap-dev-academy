## 표준을 직접 고치면 안 된다

SAP 표준 프로그램을 직접 수정하면 **업그레이드 때 덮어써지고**, SAP Note·패치 후 원인
분석도 어렵다. 그런데 회사마다 규칙은 다르다 — *판매오더 저장 전 VIP 메모, 특정 자재 조합
금지, 출고 직전 자체 번호 검증*… SAP가 이걸 다 표준에 넣을 순 없다. 그래서 표준 흐름 안에
**빈 자리(확장 지점)** 를 미리 마련해 둔다.

여기서 감각 하나 — *"내가 표준을 호출하는 게 아니라, 표준이 나를 호출한다."* 평소엔 내가
`CALL FUNCTION`으로 부르지만, exit는 **표준이 자기 흐름 중간에 내 코드 자리를 부른다.**
그 옛 방식이 User/Customer Exit다.

| 종류 | 형태 | 관리 |
|---|---|---|
| **User Exit** | 표준 include의 빈 `FORM userexit_...`(주로 SD) | 고객 영역에 작성 |
| **Customer Exit** | `CALL CUSTOMER-FUNCTION` → `FUNCTION EXIT_...` | `SMOD`/`CMOD` |

표준은 평소처럼 일하다가 확장 지점을 부른다.

```abap
" SAP 표준 쪽: 확장 지점 호출(단순화)
CALL CUSTOMER-FUNCTION '001'
  EXPORTING is_sales_order = ls_sales_order.
```

고객은 표준 본문을 안 고치고, **그 자리(함수 모듈)** 에 내 로직만 채운다.

```abap
" 고객 구현 쪽: 표준이 부르는 빈 함수에 내 검증
FUNCTION exit_saplxxxx_001.
  IF is_sales_order-netwr > 100000.
    " 고객 규칙: 검증·메시지·데이터 가공
  ENDIF.
ENDFUNCTION.
```

- `SMOD` — SAP가 제공한 enhancement **정의를 보는** 곳. `CMOD` — 고객 **프로젝트로
  선택·활성화**하는 곳.
- **활성화 안 하면 호출 위치가 있어도 실행 안 된다.** 오래된 방식이라 신규는
  [BAdI](CH29-L03.html)·[Enhancement Point](CH29-L02.html)를 먼저 본다.

## 자리에 끼워 보기

표준이 마련한 Exit 자리에 내 코드를 넣고 빼 보라 — **표준 소스는 한 줄도 안 바뀌는데**
동작만 확장된다.

## 흔한 실수와 점검

- **표준 include 직접 수정** → User Exit란 이름만 듣고 표준을 바로 고치면, 업그레이드
  충돌·패치 후 원인 분석 지옥. 제공된 빈 자리에만 쓴다. 목표는 *"표준 소스 변경 0줄."*
- **`SMOD`/`CMOD` 역할 혼동** → `SMOD`에서 *봤다고 구현된 게 아니다*. `CMOD` 프로젝트로
  **활성화**해야 동작한다. "코드 썼는데 안 돈다"의 절반은 활성화·프로젝트 연결 문제.
- **obsolete 경계 무시** → Customer Exit는 *레거시 이해용*이다. 신규 요구엔 같은 시점에
  [BAdI](CH29-L03.html)가 있는지 먼저 — 더 구조적이고 테스트하기 쉽다.
- **exit에 업무 과부하** → exit는 표준 트랜잭션 중간에서 불린다. 느린 `SELECT`·외부 통신·화면
  의존 로직을 넣으면 표준 전체가 불안정해진다. exit엔 *판단·위임*만, 복잡한 건 별도 Z
  클래스로 분리한다.

## 정리

- User/Customer Exit = 표준을 안 고치고 **빈 자리에 고객 로직**을 꽂는 옛 확장 방식.
- Customer Exit = `CALL CUSTOMER-FUNCTION` → `FUNCTION EXIT_...` + `SMOD`(정의)/`CMOD`(활성화).
- *레거시 이해엔 중요*하지만 신규 최우선은 아니다 — BAdI·Enhancement가 있으면 그쪽 먼저.
