## 표준 코드 라인 사이에 끼우기

[Customer Exit](CH29-L01.html)는 SAP가 함수 exit를 *만들어 둔 곳*에서만 쓸 수 있었다. 하지만
표준엔 "이 줄 다음에 한 줄 더 끼우고 싶다", "이 계산 블록만 우리 것으로 바꾸고 싶다"는
자리가 더 많다. **Enhancement Framework**가 이걸 더 일반적인 *소스 코드 확장*으로 다룬다.

핵심은 **추가(Point) vs 대체(Section)의 책임 차이**다. Point는 표준 흐름 *옆에* 내 코드를
놓고, Section은 표준 흐름 *일부를 대신 맡는다*. 대체가 더 강력하지만 — **테스트·업그레이드
책임도 그만큼 커진다.**

| 종류 | 동작 | 책임 |
|---|---|---|
| **Enhancement Point** | 그 자리에 코드 **추가** | 표준 그대로 + 내 코드 |
| **Enhancement Section** | 표준 블록을 내 것으로 **교체** | 표준 책임을 *대신* 맡음 |

표준에는 **확장 지점**이 표시돼 있고(`SPOTS`), 고객은 별도 구현(plug-in)을 채운다.

```abap
" SAP 표준: 추가 지점(Point)
ENHANCEMENT-POINT ep_before_save SPOTS es_order.

" 고객 plug-in: 그 지점에 끼우는 내 코드
ENHANCEMENT zenh_vip_before_save.
  IF ls_order-netwr > 100000.
    " 저장 전 VIP 검증 추가
  ENDIF.
ENDENHANCEMENT.
```

Section은 *구간 자체*를 정의한다 — plug-in이 없으면 표준 코드가, 있으면 내 코드가 그 구간을
실행한다.

```abap
" SAP 표준: 교체 가능한 구간(Section)
ENHANCEMENT-SECTION es_amount_check SPOTS es_order.
  IF ls_order-netwr > gv_limit.
    MESSAGE e001(zsd) WITH '한도 초과'.
  ENDIF.
END-ENHANCEMENT-SECTION.
```

- 메서드·함수의 시작/끝엔 [Implicit Enhancement](CH29-L04.html)가 늘 있다(SAP가 표시
  안 해도).
- 구현은 손으로 막 쓰지 않고 **Enhancement Builder로 생성·활성화**한다 — 활성화해야 동작.

## 흔한 실수와 점검

- **Point와 Section을 같은 걸로** → Point는 *삽입*, Section은 *대체*다. 금액 계산·권한·업데이트
  같은 핵심 로직을 Section으로 바꾸면 **회귀 테스트 범위가 폭증**한다. 가능하면 Point.
- **explicit 지점이 없는데 억지로** → 명시적 point가 시점과 안 맞으면 [확장 방식
  판단](CH29-L04.html)으로 돌아간다(BAdI 있나 → implicit로 충분한가 → 그래도 안 되면
  modification 심사).
- **표준 내부 변수에 과의존** → enhancement를 "어디서든 값에 접근하는 편한 구멍"으로 쓰면,
  표준 패치로 변수·흐름이 바뀔 때 바로 깨진다. *안정적 계약*인지 *우연히 보이는 내부 상태*인지
  구분한다.
- **활성화·운송을 가볍게** → enhancement 구현은 repository object다. transport가 누락되면
  *개발은 되는데 운영은 안 도는* 사고. 이름·package·transport·활성화 상태를 체크리스트로.

## 정리

- Enhancement Point = 위치에 코드 **추가**, Enhancement Section = 구간을 코드로 **대체**.
- 둘 다 표준 소스를 안 고치고 plug-in을 단다(`SPOTS`·`ENHANCEMENT`).
- 신규는 **BAdI 같은 더 구조적인 지점 먼저** → source 확장이면 **Point 우선**(Section은
  책임·테스트 범위가 크다).
