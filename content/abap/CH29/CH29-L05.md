## 표준은 깨끗하게, 확장은 분리

CH23-L08의 **Clean Core** 원칙이 확장에도 그대로 적용된다. 표준을 직접 고치지 않고,
**공개 확장점(BAdI)·Released API**로만 확장해야 업그레이드에 강하다.

## 기준

- **확장점(BAdI) 우선** — SAP이 보장한 자리만.
- **Released API만** — 비공개 객체·테이블 직접 접근 금지(클라우드에서 깨짐).
- **Key User / Developer Extensibility** — S/4HANA Cloud의 공식 확장 계층을 쓴다.
- 확장 코드는 **별도 패키지**로 분리해 추적·이송을 명확히.

> 온프레미스의 유연함(Implicit·Modification)과 클라우드의 규율(BAdI·Released) 사이에서,
> **가능한 한 Clean Core 쪽**을 택하는 게 미래지향적이다.

## 흔한 실수와 점검

- **비공개 API 확장** → 업그레이드·클라우드 전환에서 깨진다.
- **확장 추적 불가** → 별도 패키지·문서로 무엇을 왜 확장했는지 남긴다.
