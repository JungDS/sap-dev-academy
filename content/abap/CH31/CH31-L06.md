## 화면 검증과 저장 검증은 다르다

셀 단위 검증([Grid 입력값 검증과 오류 표시](CH31-L05.html))을 통과해도, **저장 직전** 다시
봐야 할 게 있다. ① 사용자가 마지막 셀에 값을 넣고 *아직 확정(Enter) 안 했을* 수 있다.
② 합계·중복처럼 **여러 행을 합쳐야 아는 규칙**(예: 좌석 합계가 정원 초과)은 셀 하나만 봐선
모른다. ③ 그새 *다른 사용자가 먼저 바꿨을* 수 있다.

그래서 저장 버튼은 데이터 무결성의 **마지막 방어선**이다. 화면 검증은 입력 편의, 저장 검증은
무결성 — **역할이 다르다.**

```abap
" 1) 편집 중인 값을 내부 테이블에 반영(마지막 셀까지)
go_grid->check_changed_data( ).

" 2) 저장 전 전체 규칙 재검증(합계·중복 등 포함)
DATA(lv_has_error) = abap_false.
LOOP AT lt_booking INTO DATA(ls_booking).
  IF ls_booking-seats < 1 OR ls_booking-seats > 10.
    lv_has_error = abap_true.
    EXIT.
  ENDIF.
ENDLOOP.
IF lv_has_error = abap_true.
  MESSAGE '입력값을 확인하세요.' TYPE 'I'.
  RETURN.                            " DB에 쓰지 않음
ENDIF.

" 3) 통과한 경우에만 DB 반영
MODIFY zbooking FROM TABLE @lt_booking.
IF sy-subrc = 0.
  COMMIT WORK.
ELSE.
  ROLLBACK WORK.
ENDIF.
```

- `check_changed_data( )` — 편집 중인 값을 내부 테이블로 **강제 반영**(마지막 셀 누락 방지).
- 최종 검증을 **통과한 경우에만** [DML](CH25-L01.html)·[잠금](CH26-L01.html)·커밋 경계로
  저장한다.

## 저장 흐름 체험

값을 고치고 **저장**을 눌러 보라. 잘못된 값이 있으면 저장이 **거부**되고, 모두 통과하면
`MODIFY + COMMIT`이 실행되는 흐름을 확인할 수 있다.

## 흔한 실수와 점검

- **`check_changed_data` 누락** → 사용자는 마지막 셀에 값을 넣었다고 생각하지만, 내부
  테이블엔 이전 값이 남아 그대로 저장될 수 있다. 저장 직전 반드시 호출한다.
- **화면 검증을 최종 검증으로 착각** → 앞서 본 셀 검증은 입력을 빨리 고치게 돕는 것.
  합계·중복·권한·잠금·DB 최신 상태는 **저장 직전 다시** 확인한다.
- **`COMMIT WORK`를 너무 쉽게** → 한 번 확정하면 화면에서 "취소"를 눌러도 DB가 자동으로
  되돌아오지 않는다. 검증·잠금·`MODIFY` 결과(`sy-subrc`) 확인이 끝난 뒤에만 commit한다.

## 정리

- Editable ALV의 저장은 입력칸에서 끝나지 않는다.
- `check_changed_data`(마지막 편집 반영) → **전체 규칙 재검증** → 통과 시에만
  `MODIFY`+`COMMIT`(실패 시 `ROLLBACK`).
- 화면 검증은 편의, 저장 검증은 **무결성의 마지막 방어선** — 오류면 DB에 쓰지 않는다.

## 🧩 도전 과제 (서브퀘스트)

> 선택 과제 — 예매 좌석수를 표에서 직접 편집한다.

1. 예매 Grid의 `SEATS`를 편집 가능하게 만들고 입력 이벤트를 등록한다([Editable Field
   Catalog](CH31-L01.html)).
2. `DATA_CHANGED`에서 좌석수가 잔여석을 넘으면 그 셀에 오류를 등록한다([DATA_CHANGED](CH31-L02.html)
   · [오류 표시](CH31-L05.html)).
3. (심화) 저장 직전 `check_changed_data` 후 정원 규칙을 최종 검증하고 [DML로
   저장](CH25-L01.html)한다.
