## 왜 — display( )만으론 부족하다 🧰

[SALV의 목적과 CL_SALV_TABLE 개요](CH11-L01.html)에서 SALV는 내부 테이블을 *빨리 보여 주는* 데 충분했다. 그런데 실제 업무 표는 그냥 보이기만 하면 부족하다 — 날짜가 뒤섞여 있거나, 취소된 회차가 섞여 있거나, 사용자가 엑셀·합계 기능을 못 찾으면 "덜 만든 표"처럼 느껴진다.

이제 [객체지향](CH21-L01.html)을 배웠으니, SALV 객체에게 **`display( )` 전에** "표준 기능을 켜라", "이 컬럼으로 먼저 정렬하라", "이 상태는 빼라"를 지시할 수 있다.

## SALV 하위 객체 — 표시 조건을 꺼내 쓴다 🔧

SALV 객체는 여러 **하위 설정 객체**를 돌려준다.

| 하위 객체 | 대표 메서드 | 의미 |
|---|---|---|
| functions | `get_functions( )->set_all( )` | 표준 툴바 기능(정렬·필터·인쇄·엑셀) |
| sorts | `get_sorts( )->add_sort( )` | 첫 표시 때 적용할 **기본 정렬** |
| filters | `get_filters( )->add_filter( )` | 첫 표시 때 적용할 **기본 필터** |
| aggregations | `get_aggregations( )->add_aggregation( )` | 숫자 컬럼의 합계/집계 |

> ⚠️ `CL_SALV_TABLE=>FACTORY`는 값을 *돌려주는* 식이 아니라 **static method 호출**이다. SALV 객체는 **`IMPORTING r_salv_table`**으로 받는다(`DATA(lo_salv) = factory(...)`가 아니다).

```abap
DATA lt_perf TYPE STANDARD TABLE OF zperf.
SELECT * FROM zperf WHERE concert_id = @p_concert INTO TABLE @lt_perf.

TRY.
    cl_salv_table=>factory(
      IMPORTING r_salv_table = DATA(lo_salv)
      CHANGING  t_table      = lt_perf ).

    lo_salv->get_functions( )->set_all( abap_true ).        " 표준 기능 켜기

    lo_salv->get_sorts( )->add_sort( 'PERF_DATE' ).         " 공연일 오름차순

    lo_salv->get_filters( )->add_filter(                    " 취소(C) 제외
      columnname = 'STATUS' sign = 'I' option = 'NE' low = 'C' ).

    lo_salv->display( ).
  CATCH cx_salv_msg INTO DATA(lx_salv).
    MESSAGE lx_salv->get_text( ) TYPE 'E'.
ENDTRY.
```

## 흔한 실수와 점검 🚧

- **`display( )` 후에 설정** → 정렬·기능·필터는 화면을 띄우기 **전에** 지정한다.
- **컬럼명을 화면 제목으로** → `add_sort`·`add_filter`의 컬럼명은 내부 테이블 **필드명**(보통 대문자 `PERF_DATE`). 틀리면 SALV가 컬럼을 못 찾는다.
- **필터를 DB 조건으로 착각** → 여기 필터는 *이미 읽은* 내부 테이블을 표시 단계에서 거른다. 데이터를 줄이려면 `SELECT … WHERE`에서 먼저.
- **표준 기능을 전부 켜기** → 도구는 많아지지만 화면 목적이 흐려질 수 있다. 업무 화면은 필요한 기능만 켜는 설계도 고려.
