## DB에서 정렬해 받기

이미 읽은 [[Internal Table]]을 ABAP `SORT`로 정렬할 수도 있지만, DB에서 **바로 정렬해
받을** 수도 있다. 이것이 `ORDER BY`다.

## ORDER BY (classic)

```abap
SELECT persid name age
  FROM ztperson
  INTO CORRESPONDING FIELDS OF TABLE lt_person
  ORDER BY age DESCENDING name ASCENDING.
```

- `ORDER BY 필드 [ASCENDING|DESCENDING]`, 여러 필드 나열 가능.
- `ORDER BY PRIMARY KEY` — 기본 키 순(단, JOIN 결과엔 못 쓴다 → 컬럼 명시).

## 정렬을 안 하면? — 순서 보장 없음

`ORDER BY`가 **없으면 결과 순서는 보장되지 않는다.** 오늘 우연히 이름순처럼 보여도
내일은 다를 수 있다. "상위 매출", "최근 공연일"처럼 순서가 의미를 가지면 반드시 명시한다.

## 여러 기준 — 앞이 우선

`ORDER BY age DESCENDING name ASCENDING`은 **나이 내림차순**으로 먼저 정렬하고,
**나이가 같으면** 이름 오름차순으로 정한다. 앞에 적은 기준일수록 우선순위가 높다.
`ORDER BY`는 행을 버리지 않고 **순서만** 바꾼다(`sy-dbcnt`는 그대로).

정렬 기준을 바꿔 가며 결과가 어떻게 달라지는지, 동률일 때 두 번째 기준이 어떻게
끼어드는지 직접 보자.

## ORDER BY vs ABAP SORT

- **소량·이미 읽은 데이터** → ABAP `SORT`로도 충분.
- **DB가 정렬에 유리(인덱스)** 하거나 정렬해 바로 쓰면 → `ORDER BY`.
- 무분별한 ORDER BY는 DB 부하가 될 수 있으니 상황에 맞게.

## 흔한 실수와 점검

- **대량 ORDER BY 부하** → 인덱스 없는 정렬은 비싸다. 필요한 만큼만.
- **콤마/@** → classic이라 공백 구분, `@` 없이.
