## 🎫 콘서트앱을 모던으로 — 단, 결과는 그대로

문법은 작은 예제에서 배우지만, 실무 감각은 **관통 예제**에서 굳는다.
[잔여석 계산·예매 판정 모듈화](CH10-L07.html)에서 만든 콘서트 예매 흐름에, 지금까지 배운 모던 문법을 입힌다.
목표는 앱을 새로 만드는 게 아니라, **이미 잘 돌아가는 classic 코드의 일부를 더 읽기 좋게** 바꾸는 것이다.

> 🚧 **이 실습의 경계 — 넘지 않는다.**
> - 예매 조회 `SELECT` 자체는 classic 그대로(모던 SQL `@`·콤마는 [Classic Open SQL과 Modern ABAP SQL 비교](CH19-L01.html)에서).
> - 객체지향으로 다시 설계하지 않는다.
> - 예외 처리 문법을 새로 끌어오지 않는다.
> - **업무 규칙(예약 판정)은 절대 바꾸지 않는다.**
>
> 바꾸는 건 인라인 선언·`VALUE`·Table Expression 보호·String Template·누적 계산, 이 다섯뿐이다.

## ① 좌석 합계 — LOOP를 모던으로

```abap
" classic
DATA: ls_book TYPE zbooking, lv_sum TYPE i.
LOOP AT lt_book INTO ls_book.
  lv_sum = lv_sum + ls_book-seats.
ENDLOOP.

" modern
DATA(lv_total) = 0.
LOOP AT lt_book INTO DATA(ls_book).   " 선언이 INTO 위치로
  lv_total += ls_book-seats.          " 누적은 +=
ENDLOOP.
```

바뀐 건 셋. `ls_book` 선언이 루프의 `INTO`로 내려왔고, 합계 변수는 필요한 자리에서 `DATA(lv_total) = 0`으로 시작하고, 누적은 `+=`로 짧아졌다.

## ② 새 예매 행 — VALUE 한 식으로

```abap
DATA(ls_new) = VALUE zbooking(
  concert_id = 'C001'
  perf_no    = '01'
  customer   = '정훈영'
  seats      = 2
  status     = 'N' ).
```

여러 줄 필드 대입보다, **"새 예매 행은 이런 필드를 가진다"**는 의도가 더 빨리 보인다.

## ③ 한 줄 조회 — 반드시 보호하고 읽는다 🛡️

업무 입력값(`lv_id`)은 **언제든 틀릴 수 있다.** 그래서 Table Expression을 보호 없이 바로 읽지 않는다. `line_exists( )`로 먼저 막는다.

```abap
IF line_exists( lt_book[ booking_id = lv_id ] ).
  DATA(ls_hit) = lt_book[ booking_id = lv_id ].
  DATA(lv_msg) = |예매자 { ls_hit-customer }님의 좌석 수는 { ls_hit-seats }석입니다.|.
  WRITE / lv_msg.
ELSE.
  WRITE / |예매번호 { lv_id }를 찾지 못했습니다.|.
ENDIF.
```

이 한 토막에 이 챕터의 모던 문법이 다 들어 있다 — `line_exists`로 실패를 막고, `DATA(ls_hit)`로 필요한 자리에서 선언하고, String Template으로 사용자 메시지를 만든다. **업무 결과는 그대로, 표현만 좋아진다.**

::embed CH18-L07-S01 | classic ↔ modern 코드를 짝지어 비교::

## 결과가 같은지 확인 — 특히 "없는 번호" ✅

같은 `lt_book`으로 classic과 modern을 돌려 결과를 맞춰 본다.

| booking_id | customer | seats | status |
|---|---|---:|---|
| B001 | 정훈영 | 2 | N |
| B002 | 손흥민 | 1 | N |
| B003 | 아이유 | 4 | N |

| 확인 항목 | 기대 결과 |
|---|---|
| 좌석 합계 | `2 + 1 + 4 = 7` |
| 있는 번호 `B001` | 예매자·좌석 수 메시지 출력 |
| **없는 번호 `B999`** | "찾지 못했다" 메시지, **프로그램 중단 없음** |
| `SELECT` 문장 | 모던으로 바꾸지 않음(다음 챕터 범위) |

> 💡 **`B999`(없는 번호) 테스트가 제일 중요하다.** Table Expression을 막 배우면 *있는 값*만 넣어 성공 화면만 보기 쉽다. 프로그램의 진짜 안정성은 **없는 값**에서 드러난다. `line_exists( )`가 그 차이를 만든다.

## 직접 해 볼 과제

1. `get_remaining`의 `LOOP`를 인라인 `DATA`·`+=`로 바꾼다.
2. 새 예매 구조체를 `VALUE`로 한 식에 채운다.
3. `READ TABLE`을 `lt_book[ … ]`로 바꾸고 `line_exists( )`로 보호한 뒤, **`B999`로 안전 분기를 확인**한다.

## 흔한 실수와 점검 🚧

- **이름을 너무 짧게** → 모던이 짧아져도 업무 이름은 정확해야 한다. `DATA(ls)`보다 `DATA(ls_book)`.
- **String Template에 업무 판단 넣기** → 상태에 따라 문구를 바꾸는 로직은 먼저 `IF`/`CASE`로 정하고, template은 **최종 문장 조립**만 맡긴다.
- **보호 없는 조회** → `DATA(ls_hit) = lt_book[ booking_id = lv_id ].`만 쓰면 없는 값에서 예외. 완성 코드엔 `line_exists( )`가 기본 방어선([Table Expression 안전하게 읽기](CH18-L04.html)).
- **섞여도 괜찮다** → 한 파일에 오래된 SQL·classic LOOP·modern VALUE가 공존해도 된다. 중요한 건 팀 기준·학습 경계·읽는 사람의 이해다.
