## READ TABLE의 식 버전

[Internal Table 기초](CH06-L01.html)에서 한 행을 찾는 기본 도구는 `READ TABLE`이었다.
성공 여부를 `sy-subrc`로 확인하고, 성공했을 때만 값을 썼다. modern에서는 한 행을
**배열처럼 대괄호로** 꺼낸다. 이것이 Table expression이다.

```abap
" classic
READ TABLE lt_booking INTO ls_book WITH KEY booking_id = '0001'.

" modern — 키로 콕 집기
DATA(ls_book) = lt_booking[ booking_id = '0001' ].

" 인덱스로
DATA(ls_first) = lt_booking[ 1 ].
```

## 없으면 예외가 난다 — line_exists로 보호

짧지만 **중요한 차이**가 있다. `READ TABLE`은 실패하면 `sy-subrc`만 세팅하고 지나간다.
Table expression은 일반 **값 표현식**이라, 없는 행을 직접 읽으면 `CX_SY_ITAB_LINE_NOT_FOUND`
**예외**가 난다. 이 차이를 모르면 modern 문법이 오히려 더 위험하다.

```abap
IF line_exists( lt_booking[ booking_id = lv_id ] ).
  DATA(ls) = lt_booking[ booking_id = lv_id ].
  WRITE: / ls-customer.
ENDIF.

DATA(lv_idx) = line_index( lt_booking[ booking_id = lv_id ] ).  " 위치(못 찾으면 0)
```

- `line_exists( )` — 그 행이 있는지 먼저 확인(성공 여부는 **`sy-subrc`가 아니라** 이걸로).
- `line_index( )` — 찾은 **위치**. 못 찾으면 `0`(hashed key 등 순서를 말할 수 없으면 `-1`).

> 예외 처리(`TRY/CATCH`)는 [Exception Class — TRY / CATCH / CX 계층](CH20-L06.html)에서
> 정식으로 다룬다. 이 챕터에서는 `line_exists( )`로 안전하게 피한다.

## 직접 비교해 보기 — READ TABLE vs Table Expression

같은 데이터로 **존재하는 키(B001)**, **다른 키(B002)**, **없는 키(B999)**를 검색하며
`READ TABLE`(sy-subrc), 직접 `tab[ ]`(예외!), `line_exists` 보호, `line_index`를 비교하자.

## 흔한 실수와 점검

- **`tab[ ]` 뒤에 `IF sy-subrc = 0`** → table expression은 `sy-subrc`를 세우지 않는다.
  성공 여부는 `line_exists`, 위치는 `line_index`로 본다.
- **없는 행 직접 접근** → 예외 발생. `line_exists( )`로 먼저 확인하고 읽는다.
- **여러 행에 남발** → table expression은 **한 행 조회** 도구다. 여러 행 순회·전체 처리는
  여전히 `LOOP AT`이 자연스럽다.
- **인덱스 접근 `tab[ 1 ]`** → 정렬·테이블 종류에 민감하다. "첫 행"이 업무적으로 의미 있을
  때만. 예매번호로 찾을 수 있으면 key 조건이 더 명확하다.

## 정리

- Table expression은 Internal Table 한 행을 **값처럼** 읽는 modern syntax — 존재가 확실할 땐 매우 간결하다.
- 없는 행을 직접 읽으면 `READ TABLE`과 달리 **예외**가 난다. `line_exists`·`line_index`로
  "짧지만 안전한 조회"를 한다.
