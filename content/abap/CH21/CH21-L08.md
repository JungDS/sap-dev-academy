## 부모 타입 변수에 자식 객체 🎭

상속·인터페이스 덕에 **부모(또는 인터페이스) 타입 변수**에 여러 자식 객체를 담아 **똑같이** 다룰 수 있다. 자식을 부모 타입에 담는 건 자연스럽다(**upcast**). 그런데 가끔 부모 타입에는 없는 **실제 타입 전용 기능**이 필요하다 — 예: VIP 전용 `apply_vip_coupon( )`.

## 내려보내기 — CAST ( ?= ) 🔽

```abap
DATA lo_mgr TYPE REF TO zcl_booking_manager.
lo_mgr = NEW zcl_vip_booking( iv_concert = 'C001' iv_perf = '01' ).   " upcast

TRY.
    DATA(lo_vip) = CAST zcl_vip_booking( lo_mgr ).   " downcast (부모→자식)
    lo_vip->apply_vip_coupon( ).
  CATCH cx_sy_move_cast_error.
    MESSAGE 'VIP 예약 객체가 아닙니다.' TYPE 'I'.
ENDTRY.
```

- `CAST 타입( ref )`(모던) = `ref ?= ...`(classic) — **down-cast**.
- **실제 객체의 동적 타입이 안 맞으면** `CX_SY_MOVE_CAST_ERROR` → `TRY/CATCH`로 대비한다. CAST는 정적 타입을 바꾸는 듯 보이지만, **실제(동적) 타입이 맞아야** 성공한다.

## 타입 분기 — CASE TYPE OF 🔀

객체의 실제 타입에 따라 분기한다. `WHEN TYPE ... INTO ...`는 **타입 검사와 안전한 참조 확보를 한 번에** 한다(예외를 던지지 않음).

```abap
CASE TYPE OF lo_mgr.
  WHEN TYPE zcl_vip_booking INTO DATA(lo_vip).
    lo_vip->apply_vip_coupon( ).
  WHEN TYPE zcl_booking_manager INTO DATA(lo_base).
    lo_base->book( iv_seats = 1 ).
  WHEN OTHERS.
    MESSAGE '처리할 수 없는 예약 객체입니다.' TYPE 'I'.
ENDCASE.
```

`CAST`는 실패하면 예외지만, `CASE TYPE OF`는 **타입에 맞는 가지로 안전하게** 들어간다. 위 체험에서 둘을 바꿔 보자.

## 흔한 실수와 점검 🚧

- **무분별한 CAST** → 타입 불일치 예외. 기본은 **다형성**(공통 메서드 호출), CAST는 최소화.
- **downcast 실패 무시** → 실제 타입이 불확실하면 `TRY/CATCH cx_sy_move_cast_error` 또는 `CASE TYPE OF`로.
- **`CASE TYPE OF` 순서** → **구체 타입부터**. 부모 타입을 먼저 두면 자식 객체가 부모 조건에 걸려 자식 전용 분기까지 못 간다.
- **타입별 분기가 자꾸 늘어남** → 공통 인터페이스·재정의 메서드로 풀 수 없는지 **설계를 재검토**한다.
