## 같은 규약, 다른 구현 🤝

종류가 달라도 **같은 약속**을 지켜야 할 때가 많다. 예매 내역·공연 정보·오류 로그는 서로 다른 클래스지만, **"화면에 출력할 수 있다"**는 점은 같다. 이때 셋을 같은 부모에서 상속시키는 건 억지다. 필요한 건 **"출력할 수 있다"는 규약**이다.

그 규약을 정하는 게 [[Interface]](인터페이스)다. 메서드 이름만 약속하고, 실제 방법은 각 클래스가 알아서 한다.

```abap
INTERFACE zif_printable PUBLIC.
  METHODS print.
ENDINTERFACE.
```

## 클래스에서 구현 — INTERFACES 🧩

클래스는 `INTERFACES`로 규약을 받아들이고, 구현 메서드는 **`인터페이스~메서드`** 이름으로 쓴다.

```abap
CLASS zcl_booking DEFINITION PUBLIC CREATE PUBLIC.
  PUBLIC SECTION.
    INTERFACES zif_printable.          " 이 규약을 따른다
ENDCLASS.

CLASS zcl_booking IMPLEMENTATION.
  METHOD zif_printable~print.          " 인터페이스~메서드
    WRITE: / '예매 내역 출력'.
  ENDMETHOD.
ENDCLASS.
```

## 인터페이스 참조 — 다형성의 기반 🎭

핵심은 여기다. **인터페이스 타입 참조 변수**(`TYPE REF TO zif_printable`)에 어떤 구현 클래스를 담느냐에 따라, **같은 호출이 다르게** 실행된다.

```abap
DATA lo_printable TYPE REF TO zif_printable.

lo_printable = NEW zcl_booking( ).
lo_printable->print( ).              " "예매 내역 출력"
```

호출자는 *실제 클래스가 무엇인지 몰라도* "이 객체는 출력 가능하다"는 약속만 보고 `print( )`를 부른다. 클래스 이름에 덜 묶일수록 나중에 구현을 갈아끼우기 쉽다. 이게 [[다형성]]이다.

## 흔한 실수와 점검 🚧

- **메서드 구현 누락** → `INTERFACES`로 규약을 받았으면 그 메서드를 **전부** 구현해야 활성화된다.
- **`~`를 빠뜨림** → 구현·호출 모두 `인터페이스~메서드` 형식(`zif_printable~print`).
- **인터페이스를 "빈 부모 클래스"처럼** → 상태를 상속받는 도구가 아니라 **외부 규약을 고정**하는 도구다.
- **한 클래스 = 인터페이스 하나로 오해** → 한 클래스는 `INTERFACES`를 여러 줄 써서 인터페이스를 여러 개 받아들일 수 있다. 다만 너무 많으면 그 클래스가 무슨 책임을 지는지 흐려진다.
- **인터페이스가 비대해짐** → `zif_printable`이면 출력에 필요한 약속만. 예매·취소·검증까지 넣으면 모든 구현 클래스가 그 의무를 떠안는다.
- **인터페이스 참조로 구체 메서드** → 인터페이스 참조로는 인터페이스에 선언된 것만 보인다. 구체 클래스 전용 메서드는 그 클래스 타입 참조라야 보인다.
