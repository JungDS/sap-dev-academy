## 왜 — 갱신이 흐름을 끊지 않게 🔄

업무 화면은 한 번 띄우고 끝나지 않는다. 예매가 추가되면 잔여석이 바뀐다. 이때 80번째 줄을 보던 사용자가 갱신 후 **첫 줄로 튕기면** 화면은 불편한 도구가 된다. [Refresh와 Stable Refresh 기초](CH17-L08.html)를 심화한다.

> 📌 **`refresh_table_display`는 DB를 다시 읽지 않는다.** 내부 테이블 값을 *먼저* 바꾼 뒤, "내부 테이블이 바뀌었으니 화면을 다시 그려라"에 가깝다.

```abap
" 1) 내부 테이블을 먼저 갱신
READ TABLE lt_perf_alv ASSIGNING FIELD-SYMBOL(<perf>)
  WITH KEY concert_id = p_concert perf_no = p_perf.
IF sy-subrc = 0.
  <perf>-seatsocc   = <perf>-seatsocc + p_new_seats.
  <perf>-seats_left = <perf>-capacity - <perf>-seatsocc.
ENDIF.

" 2) 같은 grid에 화면만 다시 그리기 — 위치는 유지
go_grid->refresh_table_display(
  EXPORTING
    is_stable      = VALUE lvc_s_stbl( row = abap_true col = abap_true )
    i_soft_refresh = abap_true ).
```

| 옵션 | 의미 |
|---|---|
| `is_stable-row` | 세로 스크롤이 튀지 않게(행 위치 유지) |
| `is_stable-col` | 가로 스크롤/컬럼 위치 유지 |
| `i_soft_refresh` | field catalog/layout을 다시 짜지 않는 *가벼운* 갱신 |

::embed CH21-L07-S01 | Refresh 흔들림 비교기 (is_stable on/off)::

## 흔한 실수와 점검 🚧

- **매 갱신 hard refresh** → 정렬·스크롤 위치가 다 풀린다. 갱신이 잦으면 `is_stable`을.
- **데이터부터 갱신** → `refresh_table_display`는 자동 재조회가 아니다. 내부 테이블을 먼저 바꾼다.
- **grid를 매번 새로 생성** → stable refresh의 의미가 사라진다. 같은 `go_grid`·같은 output table 흐름을 유지.
- **`i_soft_refresh`가 만능 아님** → 컬럼 구성·layout 자체가 바뀌면(구조 변경) **초기 표시를 다시 구성**해야 한다. 값 변경과 구조 변경을 구분.
