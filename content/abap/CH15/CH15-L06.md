## 오래된 리포트에서 마주치는 이벤트

기존 SAP 시스템에는 `END-OF-SELECTION`이 들어간 오래된 리포트가 아주 많다.
처음 이 코드를 보면 자연스럽게 이렇게 오해하기 쉽다.

> "모든 리포트는 조회를 `START-OF-SELECTION`에 쓰고,
> 출력은 `END-OF-SELECTION`에 써야 하는 거구나?"

결론부터 말하면 **그렇지 않다.** `END-OF-SELECTION`은 신규 리포트가 꼭 갖춰야 할
마무리 이벤트가 아니다. 이 레슨의 목표는 이 이벤트를 *권장*하는 것이 아니라,
기존 코드를 읽을 때 만나는 이벤트로 **정확히 위치시키는 것**이다.

## END-OF-SELECTION의 정체

[[END-OF-SELECTION]]은 원래 [[Logical Database|Logical Database(LDB)]]와 연결된
executable program(실행형 프로그램)에서, **데이터가 전부 선택된 뒤** 발생하는
이벤트다. LDB(선택화면·데이터 읽기를 미리 정의해 둔 옛 데이터 공급 구조)가 일을
끝내면 그 다음 차례로 `END-OF-SELECTION`이 열린다.

공식 문서는 이 이벤트를 **Obsolete(낡아서 권장하지 않는) 문법**으로 분류한다.
그리고 핵심 한 줄이 있다.

> LDB를 쓰지 않는 프로그램에서는 `END-OF-SELECTION` 이벤트 블록을
> **구현할 필요가 없다.** (없으면 `START-OF-SELECTION` 직후에 그냥 지나간다.)

즉 우리가 [Chapter 11 · SALV 1차 (간단 ALV)](CH11-L02.html)부터 만들어 온,
직접 `SELECT`로 조회하는 일반 리포트에는 `END-OF-SELECTION`이 **필수가 아니다.**

## 제한적 예제 — 조회와 표시를 나눠 보기

그래도 "조회 단계와 표시 단계를 의도적으로 나눈다"는 생각을 보여 주는 용도로는
쓸 수 있다. 단, 이건 **신규 표준 패턴이 아니라 "기존 코드 독해 + 단계 분리 예제"**다.

```abap
PARAMETERS pa_conc TYPE zbooking-concert_id.

DATA gt_booking TYPE STANDARD TABLE OF zbooking.
DATA lo_alv     TYPE REF TO cl_salv_table.

START-OF-SELECTION.
  SELECT *
    FROM zbooking
    INTO TABLE gt_booking
    WHERE concert_id = pa_conc.

END-OF-SELECTION.
  IF gt_booking IS INITIAL.
    MESSAGE '표시할 예매가 없습니다' TYPE 'S'.
    RETURN.
  ENDIF.

  TRY.
      cl_salv_table=>factory(
        IMPORTING r_salv_table = lo_alv
        CHANGING  t_table      = gt_booking ).
      lo_alv->display( ).
    CATCH cx_salv_msg.
      MESSAGE 'ALV 표시 중 오류가 발생했습니다' TYPE 'E'.
  ENDTRY.
```

`TYPE REF TO`·`TRY ... CATCH`·[[SALV]] 호출은
[Chapter 11 · SALV 1차 (간단 ALV)](CH11-L02.html)에서 결과 표시를 위해 이미 써 본
흐름을 그대로 재사용한 것이다. 이 레슨에서 그 문법을 다시 깊게 설명하지는 않는다.

> 💡 변수 접두어: 프로그램 맨 위에서 선언한 `gt_booking`은 이 리포트 전체에서 쓰는
> 전역(global) 내부 테이블이라 `gt_`로 시작한다.

## 직접 읽어 보기 — 기존 리포트 독해 모드

세 가지 리포트를 나란히 두고 비교해 보자. **LDB 연결 리포트**, **일반 SELECT
리포트(START 한 곳)**, **END로 표시만 분리한 리포트**가 각각 이벤트를 어떻게
호출하고 출력을 어디에 두는지, 공식 문서 경고 배지가 언제 붙는지 확인한다.
신규(비-LDB) 리포트에서 `END-OF-SELECTION`을 *필수*로 보면 어떤 경고가 뜨는지도 보라.

## 실수와 주의

- **신규 리포트의 권장 마무리 이벤트로 오해** → `END-OF-SELECTION`은 LDB/legacy
  중심 이벤트다. LDB가 없으면 구현할 이유가 없다.
- **`END-OF-SELECTION`에 입력 검증을 넣기** → 사용자는 이미 조회 흐름을 지나온
  뒤다. 입력 검증은 선택화면 단계(`AT SELECTION-SCREEN`)에서 한다.
- **SALV 예제 때문에 OO·예외 클래스로 빠지기** → 이 레슨의 주제는 report event의
  **경계**다. 표시 코드는 재사용일 뿐, 여기서 확장하지 않는다.

## 정리

- `END-OF-SELECTION`은 원래 **Logical Database**와 연결된 이벤트이고, 공식 문서상
  **obsolete** 성격이다.
- **LDB가 없는 일반 리포트에는 필수가 아니다** — `START-OF-SELECTION` 한 곳에서
  조회·표시를 명확히 구성하면 된다.
- 그래도 기존 시스템 코드에는 자주 나오므로, "조회/후처리를 나눈 legacy 패턴"으로
  **읽을 줄** 알아야 한다.
