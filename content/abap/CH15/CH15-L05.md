## 검증이 끝나면, 이제 본 처리

입력 검증이 끝나면 실제 업무 처리를 시작한다. 이 조회 로직을 [[INITIALIZATION]]에 쓰면
사용자가 조건을 입력하기 **전에** 조회하고, [[AT SELECTION-SCREEN OUTPUT]]에 쓰면 화면을
그릴 때마다 조회한다. 둘 다 좋은 자리가 아니다.

[[START-OF-SELECTION]]은 selection screen 처리(입력·검증)가 끝난 뒤 **본 처리로 들어가는
표준 자리**다. 지금까지 코드를 여기 몰아 둔 이유다.

## START-OF-SELECTION

```abap
DATA gv_status TYPE zbooking-status.

PARAMETERS p_conc TYPE zbooking-concert_id.
SELECT-OPTIONS s_stat FOR gv_status.

DATA gt_booking TYPE STANDARD TABLE OF zbooking.
DATA gs_booking TYPE zbooking.

START-OF-SELECTION.
  SELECT * FROM zbooking
    INTO TABLE gt_booking
    WHERE concert_id = p_conc
      AND status     IN s_stat.

  IF sy-subrc <> 0.
    MESSAGE '조회된 예매가 없습니다' TYPE 'S'.   " 빈 결과 안내(흐름 안 막음)
  ENDIF.

  LOOP AT gt_booking INTO gs_booking.
    WRITE: / gs_booking-booking_id, gs_booking-customer_name, gs_booking-status.
  ENDLOOP.
```

- 선택화면 입력([[PARAMETERS]]·[[SELECT-OPTIONS]])이 이 시점엔 모두 채워져 있다.
- 조회·계산·집계 같은 **무거운 로직**을 여기 둔다. ([Chapter 12 · Range Table](CH12-L01.html))의
  `SELECT-OPTIONS`, ([Chapter 13 · JOIN](CH13-L01.html))의 classic `SELECT`, `sy-subrc` 감각이 여기서 합쳐진다.
- event 키워드 없이 executable statement만 있으면 그 문장들은 **암묵적으로** `START-OF-SELECTION`에 묶인다.

아래에서 검증을 통과해야만 `START-OF-SELECTION`이 열리고, 조건에 따라 결과·`sy-subrc`가
어떻게 달라지는지 보자.

::embed CH15-L05-S01 | 조회 시작 게이트 (검증 통과 → SELECT)::

## 출력은?

- 간단하면 여기서 `WRITE`나 [Chapter 11 · SALV](CH11-L01.html)로 바로 출력한다.
- "조회 후 마무리"를 명시적으로 나누려면 `END-OF-SELECTION`(다음 레슨, 단 레거시).

## 흔한 실수와 점검

- **화면 속성 제어를 여기서** → 화면은 이미 지나간 뒤다. 화면 제어는 `AT SELECTION-SCREEN OUTPUT`.
- **빈 결과 ≠ 잘못된 입력** → 없는 코드 입력은 검증(`AT SELECTION-SCREEN`)에서 막고, 정상
  조건인데 결과가 0건이면 `S` 메시지나 빈 표 안내로 처리한다.
- **모든 코드를 여기에** → 기본값·검증·화면 제어는 각자의 이벤트로 분리한다.

## 정리

- [[START-OF-SELECTION]]은 **본 처리의 시작점**이다(검증 통과 후 조회·가공).
- 이 위치를 정확히 잡으면 기본값·화면 제어·검증·조회의 책임이 섞이지 않는다.
- 결과 유무는 `sy-subrc`·`sy-dbcnt`로 확인한다.

---

본 처리는 됐다. 마지막으로, 기존 코드에서 만나는 마무리 이벤트의 경계를 잡자 → 다음 레슨(END-OF-SELECTION).
