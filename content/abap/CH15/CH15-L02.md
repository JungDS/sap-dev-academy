## 빈 화면에서 시작하고 싶진 않다

운영 리포트라면 오늘 날짜, 자주 보는 부서, 기본 상태 같은 값이 미리 채워져 있으면
편하다. 단, 기본값은 "언제든 덮어써도 되는 값"이 아니다. 사용자가 화면에서 값을
바꿨는데 화면이 다시 그려질 때마다 프로그램이 그걸 되돌리면, 사용자는 입력을 잃는다.

[[INITIALIZATION]]은 이 문제의 첫 자리다. 선택화면이 **처음 나타나기 전 1회** 기본값을 준비한다.

## INITIALIZATION

```abap
DATA gv_dept TYPE c LENGTH 4.

PARAMETERS pa_date TYPE d.
SELECT-OPTIONS so_dept FOR gv_dept.

INITIALIZATION.
  pa_date = sy-datum.              " 오늘 날짜 기본값

  so_dept-sign   = 'I'.
  so_dept-option = 'EQ'.
  so_dept-low    = '1000'.
  APPEND so_dept.                  " header line → selection table에 조건 1건
```

- `pa_date`는 단일 입력값, `so_dept`는 selection table이다.
- `[[SELECT-OPTIONS]]`는 **header line**을 가진 selection table을 만들어, `sign`·`option`·
  `low`를 채운 뒤 `APPEND so_dept.`로 조건 행을 추가한다.
- [Chapter 12 · Range Table](CH12-L01.html)에서 배운 네 칸(`SIGN/OPTION/LOW/HIGH`) 구조가 그대로 이어진다.

## 첫 제안일 뿐, 매번 강제가 아니다

`INITIALIZATION`은 **1회**라 사용자가 화면에서 바꾼 값을 유지한다. 반면 매 화면마다 도는
[[AT SELECTION-SCREEN OUTPUT]]에서 값을 대입하면, 사용자가 입력한 값을 **덮어써** 입력이
사라진다. 아래에서 두 방식의 차이를 직접 보자.

## 흔한 실수와 점검

- **START-OF-SELECTION에서 기본값** → 이미 화면을 지난 뒤라 화면엔 안 보인다. 기본값은 `INITIALIZATION`.
- **`APPEND` 누락** → header line만 채우고 `APPEND` 안 하면 조건 행이 안 들어간다.
- **`LOW`에 표시용 텍스트** → `LOW`·`HIGH`에는 필드 타입에 맞는 **내부 값**이 들어가야 한다.
- **"매번 초기화"로 오해** → `INITIALIZATION`은 **첫 실행 기본값**이지 사용자 입력을 계속 이기는 규칙이 아니다.

## 정리

- [[INITIALIZATION]]은 사용자에게 **좋은 출발점**을 제안하는 event다(화면 뜨기 전 1회).
- 오늘 날짜·기본 조직·기본 상태를 미리 넣기 좋다.
- 사용자 입력을 계속 덮어쓰는 자리가 아니다. 반복 화면 제어는 `AT SELECTION-SCREEN OUTPUT`이다.

---

기본값은 채웠다. 그런데 화면을 **그릴 때마다** 필드를 숨기거나 잠그려면? → 다음 레슨(AT SELECTION-SCREEN OUTPUT).
