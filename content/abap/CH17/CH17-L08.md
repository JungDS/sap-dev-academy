## 데이터 변경 ≠ 화면 갱신

한 번 띄운 ALV 표는 자동으로 계속 최신이 되지 않는다. 초보자는 internal table이 바뀌면
화면도 바로 바뀐다고 생각하기 쉽지만, **화면 컨트롤은 이미 그려진 상태**다. 예매 취소·상태
변경·재조회 같은 화면에서는 **데이터 변경**과 **화면 갱신**이 별개의 단계임을 알아야 한다.

이미 떠 있는 Grid를 다시 그리려면 `refresh_table_display`를 부른다.

```abap
go_grid->refresh_table_display( ).
```

## Stable Refresh — 보던 위치를 지키며

단순 refresh는 사용자가 보던 **위치를 잃게** 할 수 있다. 200번째 행 근처를 보고 있었는데
refresh 후 맨 위로 튀면 매우 불편하다. **Stable Refresh**는 행·컬럼 위치를 가능한 유지하며 다시 그린다.

```abap
DATA ls_stable TYPE lvc_s_stbl.

ls_stable-row = abap_true.    " 행 위치 유지
ls_stable-col = abap_true.    " 컬럼 위치 유지

go_grid->refresh_table_display(
  EXPORTING is_stable = ls_stable
  EXCEPTIONS finished = 1
             OTHERS   = 2 ).

IF sy-subrc <> 0.
  MESSAGE 'ALV Grid 새로고침 중 문제가 발생했습니다.' TYPE 'S'.
ENDIF.
```

세 상황을 한눈에 구분하자.

| 상황 | 호출 |
|---|---|
| 표를 **처음** 그림 | `set_table_for_first_display` |
| 데이터 바뀌어 **다시** 그림 | `refresh_table_display` |
| 다시 그리되 **보던 위치 유지** | `refresh_table_display` + `is_stable` |

## 직접 갱신해 보기 — 데이터 vs 화면

왼쪽은 내부 테이블, 오른쪽은 화면 Grid다. **예약 상태 변경**을 누르면 **왼쪽만** 바뀐다
(화면은 아직 옛 값). **일반 Refresh**는 화면을 맞추지만 스크롤이 맨 위로 튀고,
**Stable Refresh**는 화면을 맞추면서 보던 위치를 지킨다.

## 흔한 실수와 점검

- **refresh 안 함** → 내부 테이블만 바꾸고 화면은 그대로다. 디버거엔 바뀌었는데 사용자는
  안 바뀐 것처럼 느낀다. `refresh_table_display`를 호출한다.
- **매번 display 재호출** → Grid를 만든 뒤에는 display 재호출이 아니라 **refresh**가 맞다.
  처음 구성(display)과 이후 갱신(refresh)을 분리해야 화면이 안정적이다.
- **stable 미사용** → 갱신마다 맨 위로 튄다. 편집·갱신이 잦으면 `is_stable`을 준다.
  (단 정렬이 크게 바뀌거나 행이 삭제되면 위치 유지가 어려울 수 있다. 그래도 일반 refresh보다 친절하다.)

## 정리

- **데이터 변경과 화면 갱신은 별개** — 내부 테이블을 바꿨으면 `refresh_table_display`로 다시 그린다.
- **display는 처음, refresh는 이후 갱신.** 위치를 지키려면 `is_stable`(row·col)을 준다.
