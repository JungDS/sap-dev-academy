## 읽기에서 쓰기로

[SELECT 4요소 · `*` vs 필드](CH08-L02.html)에서 배운 `SELECT`는 **읽기 전용**이었다. 데이터를 화면에 보여 줄 수는 있었지만,
"예매가 들어왔다", "취소됐다" 같은 **변화를 DB에 남기지는** 못했다. 실무 프로그램의
절반은 결국 **데이터를 바꾸는** 일이다.

DB의 행을 직접 바꾸는 네 개의 문장을 묶어 [[DML]](Data Manipulation Language)이라고
부른다. 읽기(`SELECT`)와 짝이 되는 **쓰기 4형제**다.

| 문 | 한 줄 동작 | 키가 이미 있으면 | 키가 없으면 |
|---|---|---|---|
| `INSERT` | 새 행 추가 | **삽입 안 됨**(`sy-subrc=4`) | 추가 |
| `UPDATE` | 기존 행 수정 | 수정 | 아무 일 없음(`sy-subrc=4`) |
| `MODIFY` | 있으면 수정·없으면 추가(**upsert**) | 수정 | 추가 |
| `DELETE` | 행 삭제 | 삭제 | 아무 일 없음(`sy-subrc=4`) |

```abap
INSERT zbooking FROM @ls_booking.                          " 새 예매 1건
UPDATE zbooking SET status = 'X' WHERE booking_id = @lv_id. " 상태만 수정
MODIFY zbooking FROM @ls_booking.                          " 있으면 수정·없으면 추가
DELETE FROM zbooking WHERE booking_id = @lv_id.            " 예매 취소
```

> 네 문장의 **차이가 헷갈리는 게 정상**이다. 아래에서 직접 실행해 보면 한 번에 잡힌다.

## 한 건이 아니라 통째로 — FROM TABLE

한 행씩 넣는 대신, **메모리의 내부 테이블을 한 번에** DB로 보낼 수 있다. [Transparent Table 생성
(SE11)](CH07-L01.html)에서 만든 표에 한 줄씩 손으로 넣던 번거로움, [SELECT 형태 — SINGLE · INTO TABLE · UP
TO n ROWS](CH08-L03.html)에서 "조회 결과 itab을 통째로 저장하고 싶다"던 아쉬움이 여기서 풀린다. 행마다
왕복하지 않아 **훨씬 빠르다**.

```abap
INSERT zgugudan FROM TABLE @lt_gugu.        " 81행을 한 번에
MODIFY zbooking FROM TABLE @lt_booking.     " 여러 건 upsert
```

> 단, `INSERT ... FROM TABLE`에서 **중복 키가 섞이면** 단건과 달리 런타임 오류가 난다(필요하면
> `ACCEPTING DUPLICATE KEYS`로 건너뛴다). 대량 처리의 실패 회수는 뒤 레슨에서 다시 잡는다.

## 누가·언제 바꿨나 — 감사필드 자동 stamp

업무 테이블에는 보통 "이 행을 **누가·언제** 만들었나"를 남기는 컬럼(`created_by`·
`created_on`)을 둔다. 나중에 문제를 추적하거나 책임을 가릴 때 필수다. 값은 [[SY 시스템 변수|SY 변수]]로
채운다 — 시스템이 로그온 사용자·날짜·시각을 알아서 들고 있다.

```abap
ls_booking-created_by = sy-uname.   " 로그온 사용자 ID
ls_booking-created_on = sy-datum.   " 오늘 날짜
ls_booking-created_at = sy-uzeit.   " 현재 시각
INSERT zbooking FROM @ls_booking.
```

> 위 플레이그라운드에서 `INSERT`/`MODIFY`로 새 행을 만들면 `created_by`·`created_on`이
> 자동으로 찍히는 걸 볼 수 있다.

## ⚠️ 표준 테이블은 직접 쓰지 않는다

`SCARR`·`SFLIGHT` 같은 **SAP 표준 테이블**은 `INSERT`/`UPDATE`로 **직접 바꾸면 안 된다**.
표준 데이터는 보이지 않는 검증·후속 처리(다른 테이블 동기화, 번호 채번 등)가 딸려 있는데,
직접 DML은 그걸 **모두 건너뛰어** 데이터를 조용히 망가뜨린다. 표준 데이터는 반드시
**BAPI/표준 API**를 경유한다(자세히는 [BAPI 호출과 Return 처리](CH33-L01.html)). 직접 DML은 **내가 만든
테이블**(`ZBOOKING` 등)에만 쓴다.

## 흔한 실수와 점검

- **INSERT 중복키** → 단건 `INSERT FROM wa`는 같은 키가 있으면 **삽입 안 됨·`sy-subrc=4`**(덤프 아님 — 중복은
  성공이 아니므로 실패로 분기). 대량 `INSERT FROM TABLE`은 런타임 오류. "있으면 갱신"이 목적이면 `MODIFY`.
- **`WHERE` 없는 `UPDATE`/`DELETE`** → **전체 행**이 바뀌거나 사라진다. 사고의 단골 — 조건을 반드시.
- **표준 테이블 직접 DML** → 금지. [BAPI 호출과 Return 처리](CH33-L01.html) 경유.
- **`sy-subrc`/`sy-dbcnt` 미확인** → 실행 전 "몇 행이 바뀌어야 정상인지" 예측하고, 실행 후 `sy-subrc`(성공 여부)·
  `sy-dbcnt`(영향 행 수)로 확인한다. `sy-dbcnt`가 예상과 다르면 `WHERE`가 너무 넓거나 대상이 없는 것 —
  그대로 다음 레슨의 COMMIT 판단으로 넘기면 안 된다.

## 정리

- 쓰기 4형제 = `INSERT`(추가)·`UPDATE`(수정)·`MODIFY`(upsert)·`DELETE`(삭제).
- 여러 건은 `FROM TABLE`로 한 번에 — 빠르다.
- 업무 테이블엔 감사필드(`created_by`·`created_on`)를 `SY` 변수로 stamp.
- 표준 테이블은 직접 DML 금지 → BAPI.

데이터를 바꾸는 법은 익혔다. 그런데 이 변경은 **언제 진짜 확정**되고, 중간에 실패하면
어떻게 **되돌릴까**? `COMMIT`/`ROLLBACK` → 다음 레슨.
