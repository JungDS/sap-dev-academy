## 동시에 같은 예매를 고치면

CH24에서 데이터를 안전하게 **바꾸는** 법을 배웠다. 그런데 실무 시스템은 **여러 사람이
동시에** 쓴다. 두 직원이 같은 예매(`ZBOOKING`)를 **동시에** 열어 수정하면 어떻게 될까?
한쪽이 저장한 변경이 다른 쪽 저장에 **덮여 사라지거나**, 데이터가 어긋난다.

`COMMIT`/`ROLLBACK`은 *내 트랜잭션 안*만 챙길 뿐, **남이 동시에 건드리는 것**은 못 막는다.
그래서 "이 행은 지금 내가 작업 중"이라고 알리는 **잠금** 장치가 필요하다 — [[Lock Object]]다.

## Lock Object — SE11에서 만든다

`SE11` → Lock Object → `EZ_BOOKING`(관례: `E` + 테이블명)을 만든다. 구성은 단 두 가지다.

- **Primary Table** — 잠글 대상 테이블(`ZBOOKING`).
- **Lock Argument** — **무엇을 기준으로** 잠글지. 보통 키 필드(`booking_id`) — "1001번 예매만 잠금".
- 활성화하면 잠그고/푸는 **ENQUEUE/DEQUEUE 함수가 자동 생성**된다(`ENQUEUE_EZ_BOOKING`·
  `DEQUEUE_EZ_BOOKING`). 우리가 함수 본문을 짤 필요 없이 **호출만** 하면 된다(다음 레슨).

## 잠금 모드 — E / S / X

잠금에는 "어떤 동시 접근을 허용할지" 정하는 **모드**가 있다.

| 모드 | 뜻 | 다른 사용자에게 허용 |
|---|---|---|
| `E` | Exclusive(쓰기) | 아무도 못 — 가장 흔함 |
| `S` | Shared(읽기) | 다른 **S(읽기)** 는 OK, **E(쓰기)** 는 차단 |
| `X` | eXclusive 비누적 | 같은 사용자도 중첩 불가(가장 강함) |

말로는 헷갈리니, A가 어떤 모드로 잡고 있을 때 B의 요청이 **허용되는지/거절되는지** 직접
바꿔 보자.

::embed CH25-L01-S01 | 잠금 모드 호환 매트릭스 | 470::

> 대부분의 **변경** 잠금은 `E`(Exclusive)다 — "내가 고치는 동안 아무도 손대지 마".

## 흔한 실수와 점검

- **Lock Argument 과다** → 너무 넓게(테이블 전체 등) 잠그면 동시성이 급락. 보통 **키 단위**로 좁게.
- **잠금 없이 변경** → 동시 충돌 위험. 변경 **전에** 잠근다(L02의 표준 패턴).
- **모드 오해** → 단순 조회 보호엔 `S`, 변경엔 `E`. X는 특수 상황.

## 정리

- 동시 변경 충돌은 `COMMIT`으로 못 막는다 → **Lock Object**(SE11)로 잠근다.
- 구성 = **Primary Table** + **Lock Argument**(보통 키). 활성화 시 ENQUEUE/DEQUEUE 자동 생성.
- 모드 = `E`(쓰기·기본)·`S`(읽기 공유)·`X`(강한 배타). S끼리만 동시 허용.
