## 잠금은 언제 풀리나

잠금을 거는 것만큼 **푸는 타이밍**이 중요하다. 안 풀면 다른 사용자가 영영 막히고, 너무
일찍 풀면 보호가 안 된다. 잠금이 풀리는 경우는 셋이다.

- **`DEQUEUE`** 명시 호출 시 — 내가 직접 푼다.
- **`COMMIT WORK` / `ROLLBACK WORK`** 시 — 보통 트랜잭션 종료와 함께 **자동 해제**된다(실무에서 가장 흔함).
- **[[세션]] 종료** 시 — 그 세션이 쥔 잠금이 모두 풀린다.

`DEQUEUE_ALL`로 현재 프로그램이 건 **모든 잠금**을 한 번에 풀 수도 있다.

반대로 단순 `SELECT`(읽기)나 **다른 키**의 작업은 내 잠금에 영향이 없다. 아래 퀴즈로
"풀림 / 유지"를 판별해 보라.

## foreign_lock 예외 — 정중히 거절

이미 남이 잠갔으면 `foreign_lock`이 난다. 무시하지 말고 사용자에게 안내한다.

```abap
CALL FUNCTION 'ENQUEUE_EZ_BOOKING'
  EXPORTING  booking_id = lv_id
  EXCEPTIONS foreign_lock = 1 OTHERS = 2.

IF sy-subrc = 1.
  MESSAGE '다른 사용자가 편집 중입니다' TYPE 'E'.
ENDIF.
```

> ⚠️ **누가 잠갔는지**(잠금 소유자)는 `sy-msgv1` 같은 시스템 변수에 **항상 채워진다고
> 가정하지 말 것.** 필요하면 `SM12`(잠금 목록)나 잠금 조회 함수로 별도로 얻는다.

## 흔한 실수와 점검

- **장시간 잠금** → 사용자가 편집 화면을 켜 둔 채 자리를 비우면 남이 막힌다. 가능하면 **짧게** 잠그고 푼다.
- **예외 안 잡음** → `foreign_lock`을 무시하면 사용자 경험이 나쁘다. 반드시 안내한다.
- **누가 잠갔나 가정** → 잠금 소유자는 시스템 변수에 항상 채워진다고 믿지 말고 `SM12`/조회 함수로 확인.

## 정리

- 잠금은 **`DEQUEUE`·`COMMIT`/`ROLLBACK`·세션 종료**로 풀린다(자동 해제가 흔함). `SELECT`·다른 키는 영향 없음.
- `DEQUEUE_ALL`로 내 모든 잠금을 한 번에.
- `foreign_lock`은 **정중히 거절**하고, 잠금은 **짧게** 유지한다. 소유자는 `SM12`로 확인.
