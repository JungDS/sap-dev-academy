## 잃어버린 갱신 (Lost Update)

A와 B가 같은 예매를 읽고, A가 저장한 직후 B가 (옛 값 기준으로) 저장하면 **A의 변경이
사라진다**. 이걸 막는 두 전략이 있다.

| 전략 | 방식 | 언제 |
|---|---|---|
| **Pessimistic**(비관) | 미리 ENQUEUE로 **잠그고** 작업 | 충돌 잦음·편집 길음 |
| **Optimistic**(낙관) | 안 잠그고, 저장 직전 **변경 여부 확인** | 충돌 드묾·읽기 위주 |

## Optimistic — 타임스탬프 비교

읽을 때의 변경시각을 들고 있다가, 저장 직전 DB의 현재 변경시각과 **비교**한다. 다르면
"그 사이 누가 바꿨다" → 거절·재조회.

```abap
SELECT SINGLE changed_at FROM zbooking
  WHERE booking_id = @lv_id INTO @DATA(lv_now).
IF lv_now <> lv_read_at.            " 읽은 뒤 누가 바꿈
  MESSAGE '데이터가 변경되었습니다. 다시 조회하세요' TYPE 'E'.
ENDIF.
```

## 흔한 실수와 점검

- **무조건 비관적 잠금** → 동시성↓. 충돌 빈도를 보고 전략 선택.
- **타임스탬프 미비** → 낙관적 잠금은 변경 추적 컬럼이 필요.
