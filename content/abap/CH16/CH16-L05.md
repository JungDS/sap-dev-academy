## 외우지 못하는 값, 화면값에 따라 달라지는 후보

사용자는 공연ID `C001` 같은 코드를 외우지 못한다. 입력칸에서 **F4를 누르면 목록**이 떠야
하고, **F1을 누르면** 그 칸이 무엇인지 설명이 나와야 한다.

[[Search Help]]는 [Data Element에 Search Help 붙이기](CH09-L04.html)에서 배웠다. 대부분은
그걸로 끝난다. 그런데 이런 경우가 있다.

- 프로그램 안에서만 만드는 필드라 **DDIC Search Help가 없다.**
- **다른 칸 값에 따라 후보가 달라진다.** 예를 들어 공연ID를 먼저 고르면, F4에는 **그 공연의
  회차만** 떠야 한다.

이럴 때 화면(Dynpro)에서는 flow logic의 [[PROCESS ON VALUE-REQUEST]]가 F4를,
[[PROCESS ON HELP-REQUEST]]가 F1을 맡는다.

## 먼저 규칙 — 입력 도움에는 순서가 있다

이 레슨의 핵심은 "F4를 직접 코딩하는 법"이 아니라 **순서**다. 좋은 classic 화면은 아래
순서로 내려간다. **위에서 해결되면 아래는 건드리지 않는다.**

| 순서 | 방법 | 언제 |
|---|---|---|
| 1 | DDIC 입력 도움 | Data Element의 [[Search Help]]·Check Table·Domain 고정값으로 충분할 때 (코드 0줄) |
| 2 | 화면 필드에 Search Help 직접 연결 | 그 칸에만 별도 Search Help가 필요할 때 |
| 3 | Search Help exit | Search Help의 동작 자체를 손봐야 할 때 (심화 · 여기선 다루지 않음) |
| 4 | `PROCESS ON VALUE-REQUEST` 직접 구현 | 다른 화면값으로 후보를 만들거나, 표준으로 안 될 때 (**최후수단**) |

> ⚠️ DDIC로 충분한 칸에 습관적으로 직접 F4를 만들지 말자. 직접 만든 건 **유지보수 책임**이
> 그만큼 커진다. 4번은 정말 필요할 때만.

## 그래도 직접 만들어야 한다면

flow logic에 처리할 칸과 [[dialog module|모듈]]을 이어 준다.

```abap
PROCESS ON VALUE-REQUEST.              " F4 요청 처리
  FIELD gv_perf MODULE value_perf.

PROCESS ON HELP-REQUEST.               " F1 요청 처리
  FIELD zconcert-concert_id MODULE help_conc.
```

`FIELD gv_perf MODULE value_perf.`는 "**`gv_perf` 칸에서 F4를 누르면** `value_perf` 모듈을
부르라"는 뜻이다. 모듈은 ABAP 쪽에 `MODULE value_perf INPUT.`으로 정의한다(PBO의
`OUTPUT`과 짝이 되는 `INPUT`이다).

## 다른 화면값을 읽어 후보 좁히기

F4를 누른 순간, **지금 화면에 입력된 다른 값**(공연ID)을 읽어야 회차 후보를 만들 수 있다.
그 값을 읽어 오는 화면 전용 함수가 `DYNP_VALUES_READ`다.

```abap
DATA: gt_dynp TYPE STANDARD TABLE OF dynpread,
      gs_dynp TYPE dynpread.

MODULE value_perf INPUT.
  gs_dynp-fieldname = 'ZCONCERT-CONCERT_ID'.   " 읽고 싶은 칸
  APPEND gs_dynp TO gt_dynp.

  CALL FUNCTION 'DYNP_VALUES_READ'              " 화면에서 현재 값 읽기
    EXPORTING dyname = sy-repid
              dynumb = sy-dynnr
    TABLES    dynpfields = gt_dynp.

  READ TABLE gt_dynp INTO gs_dynp INDEX 1.
  " gs_dynp-fieldvalue 에 사용자가 고른 공연ID가 들어 있다
ENDMODULE.
```

읽은 공연ID로 **후보 목록**을 만들어 F4 창으로 띄운다. 목록을 띄우는 함수는
`F4IF_INT_TABLE_VALUE_REQUEST`다.

```abap
TYPES: BEGIN OF ts_perf,
         perf_no TYPE n LENGTH 3,
         text    TYPE c LENGTH 30,
       END OF ts_perf.
DATA: gt_perf TYPE STANDARD TABLE OF ts_perf,
      gs_perf TYPE ts_perf.

" 공연이 C001이면 그 공연의 회차만 후보로
IF gs_dynp-fieldvalue = 'C001'.
  gs_perf-perf_no = '001'. gs_perf-text = '1회차 19:00'.
  APPEND gs_perf TO gt_perf.
  gs_perf-perf_no = '002'. gs_perf-text = '2회차 21:00'.
  APPEND gs_perf TO gt_perf.
ENDIF.

CALL FUNCTION 'F4IF_INT_TABLE_VALUE_REQUEST'
  EXPORTING retfield    = 'PERF_NO'      " 고른 값을 돌려줄 컬럼
            dynpprog    = sy-repid
            dynpnr      = sy-dynnr
            dynprofield = 'GV_PERF'       " 값을 넣어 줄 화면 칸
            value_org   = 'S'
  TABLES    value_tab   = gt_perf.
```

> 지금 이 코드를 통째로 외울 필요는 없다. 기억할 건 **"직접 F4는 최후수단이고, 그때는
> `DYNP_VALUES_READ`로 다른 칸 값을 읽어 후보를 좁힌다"** 는 큰 그림이다.

## F1은 PROCESS ON HELP-REQUEST

F1(도움말)은 대부분 [[Data Element]] 문서에 맡기는 게 낫다. DDIC 타입을 참조한 칸은 F1을
누르면 그 Data Element 설명이 **자동으로** 뜬다([Data Element](CH03-L03.html)에서 붙여 둔
그 설명이다). 화면 맥락에 맞는 별도 설명이 정말 필요할 때만 `PROCESS ON HELP-REQUEST`로
직접 만든다.

## 선택화면의 F4와 무엇이 다른가

[Report Event·선택화면 심화](CH15-L08.html)에서 리포트의 F4를
`AT SELECTION-SCREEN ON VALUE-REQUEST FOR ...`로 만들어 봤다. 뿌리는 **같은 POV**다.
다만 위치가 다르다.

| | 일반 화면(Dynpro) | 선택화면(Report) |
|---|---|---|
| 어디에 | flow logic `PROCESS ON VALUE-REQUEST` | ABAP 이벤트 `AT SELECTION-SCREEN ON VALUE-REQUEST` |
| flow logic | 직접 작성 | 자동 생성(감춰짐) |

`F4IF_INT_TABLE_VALUE_REQUEST` 같은 함수는 **양쪽에서 똑같이** 쓴다. 한 번 익히면 재사용된다.

## 흔한 실수와 점검

- **DDIC로 될 걸 직접 만든다** → 1번(Search Help)이 되면 4번(직접 POV)은 만들지 않는다.
- **POV를 일반 PAI처럼 쓴다** → POV 모듈은 **후보를 보여 주고 값을 돌려줄 뿐**이다. 저장·
  잠금 같은 일은 하지 않는다(그건 뒤 챕터의 주제다).
- **`ON VALUE-REQUEST`(리포트)와 `PROCESS ON VALUE-REQUEST`(화면)를 섞는다** → 뿌리는 같아도
  쓰는 위치가 다르다.
- **`FIELD` 이름 불일치** → `FIELD gv_perf ...`의 이름은 화면 요소 이름과 같아야 한다.

## 정리

- Dynpro F4의 기본은 DDIC [[Search Help]]다. **직접 구현은 최후수단.**
- 직접 만들 땐 flow logic `PROCESS ON VALUE-REQUEST`(F4)·`PROCESS ON HELP-REQUEST`(F1)에
  칸과 모듈을 잇는다.
- 다른 칸 값이 필요하면 `DYNP_VALUES_READ`로 읽고, `F4IF_INT_TABLE_VALUE_REQUEST`로 목록을
  띄운다.
- 리포트의 `AT SELECTION-SCREEN ON VALUE-REQUEST`와 뿌리는 같고 위치만 다르다.
