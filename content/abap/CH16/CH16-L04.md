## 같은 걸 두 번 만드는 불편

지난 레슨에서 화면 요소는 **같은 이름의 전역 변수**와 값을 주고받는다고 배웠다.
그래서 입력칸마다 `DATA gv_conc ...`, `DATA gv_seats ...`를 화면 필드 하나하나와
이름·타입을 맞춰 **일일이** 선언했다.

그런데 공연 마스터 [[Transparent Table]] `ZCONCERT`에는 이미 `concert_id`(공연ID)·
`title`(공연명)·`hall_id`(공연장) 같은 필드가 **타입도, 길이도, 한글 라벨도, F4 도움말도**
완성된 채로 들어 있다. 화면에 이 필드들을 올리려고 프로그램에 `gv_conc`·`gv_title`을
다시 만들면, **DDIC에 있는 걸 베껴 두 번 만드는** 셈이다. 라벨과 F4는 재사용할 방법도
막막하다.

> 💡 DDIC 필드를 통째로 화면에 얹고, 프로그램에도 **똑같은 그릇을 한 번에** 만들 수 없을까?

## TABLES — DDIC와 같은 모양의 그릇

그 그릇을 만들어 주는 classic 문장이 [[TABLES]]다.

```abap
PROGRAM sapmzconcert.

TABLES zconcert.        " ZCONCERT와 같은 이름·모양의 work area 1개
```

`TABLES zconcert.`는 [[Transparent Table]] `ZCONCERT`와 **같은 이름·같은 구조**의
[[table work area|work area]](작업 영역)를 프로그램에 만든다. 핵심은 이것이 **한 줄짜리
[[Structure]]**라는 점이다. 여러 행을 담는 [[Internal Table]]이 아니다.

- `TABLES zconcert.` → `zconcert-concert_id`, `zconcert-title`, `zconcert-hall_id` 같은
  [[Component]](구성 필드)를 가진 전역 work area가 생긴다.
- 필드 접근은 [[Structure]]에서 배운 그대로 `zconcert-concert_id`처럼 **하이픈**으로 한다.

> 🔎 CH12에서 `SELECT-OPTIONS ... FOR zbooking-concert_id`를 쓰려고 `TABLES zbooking`을
> **[선행 사용]**으로 잠깐 봤다. 같은 문장이다. 그때는 "필드를 참조하려고" 썼고, 지금은
> **화면과 프로그램 사이에서 값을 나르는 통로**로 쓴다. 정식 쓰임을 여기서 배운다.

## 이름이 같으면 자동으로 나른다

Screen Painter에서 `ZCONCERT-CONCERT_ID` 필드를 **DDIC에서 복사해** 화면에 올리면,
화면 요소 이름도 `ZCONCERT-CONCERT_ID`가 된다. 그러면 `TABLES` work area의
`zconcert-concert_id`와 **이름이 같아져** 값이 저절로 오간다.

```abap
MODULE prepare_0100 OUTPUT.        " PBO — 화면 뜨기 직전
  zconcert-concert_id = 'C001'.    " work area에 넣으면
  zconcert-title      = 'Rock Fest'.
ENDMODULE.                          " 화면의 같은 이름 필드에 표시된다
```

운반 방향은 앞서 배운 PBO/PAI 두 박자 그대로다.

| 시점 | 방향 | 뜻 |
|---|---|---|
| PBO 끝 | work area → 화면 | `zconcert-concert_id` 값이 화면에 뜬다 |
| PAI 시작 | 화면 → work area | 사용자가 고친 값이 `zconcert-concert_id`로 들어온다 |

직접 만든 필드와 DDIC 필드는 **섞어 써도** 된다. 연결 원리는 똑같이 "이름 일치"다.

```abap
TABLES zconcert.                    " DDIC 필드 → work area로 연결
DATA   gv_seats TYPE i.             " 직접 만든 필드 → 같은 이름 변수로 연결
```

화면의 `ZCONCERT-CONCERT_ID`는 `TABLES` work area와, `GV_SEATS`는 `gv_seats`와 묶인다.

## 라벨·F4가 공짜로 따라온다

DDIC에서 복사했기 때문에, 그 필드의 **한글 라벨**과 **F4 입력 도움말**이 화면에 그대로
따라온다. `gv_`로 직접 선언한 필드는 이 이점을 받지 못한다. 이것이 Dictionary 필드를
화면에 쓰는 가장 큰 매력이다.

## 흔한 실수와 점검

- **`TABLES`를 조회 문장으로 오해** → `TABLES zconcert.`는 DB를 읽지 않는다. 값을 나를
  **그릇 하나**를 만들 뿐이다.
- **`TABLES`를 Internal Table로 착각** → 이름은 TABLES지만 **한 줄짜리 구조**다.
- **이름이 어긋남** → 화면 요소 이름과 work area(또는 변수) 이름이 다르면 운반이
  아예 일어나지 않는다.
- **아무 데나 `TABLES` 남용 금지** → `TABLES`는 **classic 화면 프로그램 전용**의 레거시
  방식이다. 라벨·F4 재사용이 목적일 때만 쓴다. DDIC 이점이 필요 없으면 `DATA gs_concert
  TYPE zconcert.`처럼 전역 구조를 직접 선언하는 편이 낫다.

> 여기서는 **화면 ↔ 프로그램 값 운반**까지만 다룬다. 나른 값을 DB에 저장하거나 잠그는
> 일은 뒤 챕터의 주제다.

## 정리

- Screen Painter에서 DDIC 필드를 복사하면 라벨·타입·F4가 함께 온다.
- 그 필드가 프로그램과 값을 주고받으려면 **같은 이름의 `TABLES` work area**가 있어야 한다.
- `TABLES`는 한 줄짜리 구조이고, 운반은 PBO(→화면)·PAI(→프로그램) 두 박자 그대로다.
- `TABLES`는 classic 화면 전용 레거시다. 필요할 때만 쓴다.
