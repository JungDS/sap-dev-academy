## 여기까지의 ABAP SQL 🗺️

- **콤마·`@`·strict** — classic과 다른 modern SQL의 기본.
- **`@DATA( )`** — 결과 테이블을 SELECT 자리에서 인라인 선언.
- **SQL 식**(`CASE`·`CAST`·`COALESCE`)·**문자/날짜 함수** — DB에서 값 가공.
- **`SELECT FROM @itab`** — 메모리 테이블도 SQL로.

## 언제 무엇을 쓸까 — 선택 기준 🧭

이 챕터의 핵심은 문법 암기가 아니라 **선택 기준**이다. "modern이니까 무조건 SQL로"가 아니라, **DB가 잘하는 집합 연산·결과 컬럼 계산은 SQL에**, **절차적 판단·사용자 상호작용은 ABAP에** 둔다.

| 상황 | 선택 | 이유 |
|---|---|---|
| DB 컬럼과 ABAP 변수 경계가 헷갈린다 | `@` host variable | ABAP 값임을 SQL 안에서 명확히 |
| SELECT list가 길다 | 콤마 필드 목록 | 필드 경계 분명 · strict mode |
| 짧은 조회 결과를 바로 쓴다 | `INTO TABLE @DATA( )` | target을 SELECT 가까이 |
| 단순 상태 컬럼이 필요하다 | SQL `CASE` | DB에서 결과 컬럼 계산 |
| null을 표시값으로 바꾼다 | `COALESCE` | DB null 대체 |
| 날짜·문자열 일부 가공 | SQL function | 결과 컬럼으로 계산 |
| 메모리 데이터를 SQL식으로 재가공 | `SELECT FROM @itab` | internal table을 source로 |
| 결과를 key로 자주 조회한다 | 명시 타입 + key | `@DATA( )`는 empty key |
| 복잡한 업무 규칙을 표현한다 | ABAP 코드 | SQL에 과적하지 않기 |

## 흔한 실수와 점검 🚧

- **classic/modern 혼용** → 한 SELECT 안에서 섞지 않는다(콤마를 쓰면 `@`까지 일관되게).
- **SQL에 업무 로직 과적** → `CASE`가 된다고 모든 조건을 SELECT list에 넣지 않는다. 복잡한 규칙은 ABAP/이후 구조화로.
- **`@DATA( )`가 항상 최선은 아님** → key로 자주 조회하거나 타입을 여러 모듈이 공유하면 명시 타입이 낫다.
- **`@itab`은 "DB SELECT 없이 SQL 쓰는 멋진 방법"이 아님** → 메모리에 있는 적당한 데이터에 SQL을 적용하는 선택지일 뿐.

## 다음은 — 코드 구조를 객체로 🧱

데이터 접근(읽기·가공)은 이제 현대적이다. 곧 배울 [CDS View](CH23-L01.html)는 이 ABAP SQL을 **재사용 가능한 뷰**로 끌어올린 것이다. 여기서 익힌 SQL 감각이 그대로 쓰인다.

그리고 문법(지난 챕터)·SQL(이 챕터)을 현대화했지만, [ALV를 만들 때](CH17-L01.html)는 객체·메서드를 "블랙박스"로 빌려 썼을 뿐 **직접 클래스를 설계**하진 않았다. 이제 코드 구조 자체를 객체로 설계하는 [OO ABAP](CH21-L01.html)로 간다.
