## 메모리 테이블도 SQL로 🧠

modern SQL은 DB 테이블뿐 아니라 **Internal Table**(메모리에 있는 표)도 소스로 조회할 수 있다. 이미 가져온 데이터를, DB를 다시 읽지 않고 **SQL 문법(필터·정렬·집계)으로 재가공**할 때 편하다.

먼저 내부 테이블을 하나 만든다(지금까지 배운 `TYPES`·`VALUE`).

```abap
TYPES: BEGIN OF ty_flight,
         carrid   TYPE sflight-carrid,
         connid   TYPE sflight-connid,
         seatsocc TYPE sflight-seatsocc,
       END OF ty_flight.
TYPES ty_flight_tab TYPE STANDARD TABLE OF ty_flight WITH EMPTY KEY.

DATA(lt_flight) = VALUE ty_flight_tab(
  ( carrid = 'LH' connid = '0400' seatsocc = 120 )
  ( carrid = 'LH' connid = '0401' seatsocc = 80  )
  ( carrid = 'AA' connid = '0017' seatsocc = 50  ) ).
```

이제 이 메모리 테이블을 **SQL 소스**로 조회한다.

```abap
SELECT f~carrid, f~connid, f~seatsocc
  FROM @lt_flight AS f          " 내부 테이블이 소스
  WHERE f~carrid = @lv_carr
  ORDER BY f~connid
  INTO TABLE @DATA(lt_filtered).
```

- `FROM @lt_flight` — 메모리의 내부 테이블을 소스로(**DB 왕복 없음**).
- `AS f` … `f~carrid` — alias로 출처를 분명히 한다.

## 집계도 된다 ➕

```abap
SELECT f~carrid,
       SUM( f~seatsocc ) AS total_occ
  FROM @lt_flight AS f
  GROUP BY f~carrid
  INTO TABLE @DATA(lt_sum).
```

`carrid`별로 묶어 좌석을 합산한다(LH = 120 + 80 = 200). 같은 일을 ABAP에선 `LOOP`/`COLLECT`로 했지만, 여기선 SQL 한 문장이다.

## 언제 쓰나 · 언제 쓰지 말까 ⚖️

- **쓸 때**: 이미 가져온 데이터를 **다시 집계/정렬**하거나, 파일·인터페이스로 받은 데이터를 SQL식으로 다룰 때.
- **쓰지 말 때**: DB에 있는 **대량 데이터를 다 끌어와** 메모리에서 `SELECT FROM @itab`으로 집계하는 것. 보통 나쁜 방향이다 — 가능하면 **DB에서 직접** 조건·집계를 처리한다.

> 🔧 internal table을 source로 쓰는 SELECT는, **ABAP SQL engine**이 처리하는 경우와 임시로 DB에 데이터를 넘겨야 하는 경우로 나뉜다. 집계·join·복잡한 식이 섞이면 시스템이 경고를 낼 수 있는데, **pragma로 숨기기 전에 "왜 나는지"부터** 이해하자.

## 흔한 실수와 점검 🚧

- **`@` 누락** → 소스 내부 테이블도 ABAP 값이라 `@lt_flight`. `FROM lt_flight`는 안 된다.
- **alias 생략** → `AS f` … `f~carrid`로 출처를 분명히(DB 테이블과 섞일 때 특히).
- **DB 대용으로 남용** → 대량은 DB에서 직접. `@itab` SELECT는 **메모리 데이터 재가공용**.
- **elementary row 테이블** → 컬럼 이름이 `table_line`이 될 수 있어, inline target과 함께면 alias가 필요. 구조화된 row type으로 시작하는 게 안전하다.
