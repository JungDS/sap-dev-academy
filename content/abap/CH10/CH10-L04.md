## Subroutine 말고, 클래스라는 묶음

지금까지 코드를 [[Subroutine]]·[[Function Module]]로 묶었다. 또 하나의 묶음 도구가
[[Class]](클래스)와 [[Method]](메서드)다. 본격 OO(객체지향) 설계는 한참 뒤(CH20)에
배우니, 여기선 **맛보기**만 — 그것도 객체를 안 만들어도 되는 **정적(static) 메서드**부터.

> 💡 이 시기의 코드는 **보통 Subroutine/Function Module**로 묶고, 클래스는 특수한 경우에
> 쓴다. 클래스가 본격적으로 주인공이 되는 건 CH20이다.

## 클래스의 두 부분 — DEFINITION / IMPLEMENTATION

[[Class]]는 "무엇이 있는지"(DEFINITION)와 "실제 코드"(IMPLEMENTATION) 둘로 나뉜다.
공개할 것은 `PUBLIC SECTION`, 감출 것은 `PRIVATE SECTION`에 둔다.

```abap
CLASS lcl_calc DEFINITION.
  PUBLIC SECTION.
    CLASS-METHODS add_tax
      IMPORTING iv_amount        TYPE p
      RETURNING VALUE(rv_result) TYPE p.
ENDCLASS.

CLASS lcl_calc IMPLEMENTATION.
  METHOD add_tax.
    rv_result = iv_amount * '1.1'.
  ENDMETHOD.
ENDCLASS.
```

- `CLASS-METHODS` — **정적 메서드**(객체 없이 클래스로 직접 부르는 메서드).
- `IMPORTING`(입력)·`RETURNING`(반환 한 개)으로 파라미터를 선언한다.

## 정적 메서드 호출 — `=>`

정적 메서드는 **객체를 만들 필요 없이** "클래스이름`=>`메서드"로 부른다.

```abap
DATA lv_out TYPE p LENGTH 8 DECIMALS 2.

lv_out = lcl_calc=>add_tax( 1000 ).   " 객체 생성 불필요
WRITE lv_out.
```

## 설계도와 객체 (개념만)

[[Class]]는 "설계도", 그 설계도로 찍어낸 실체가 **객체(Object)**다(쿠키틀과 쿠키).
객체를 만들어 "객체`->`메서드"로 부르는 **인스턴스 메서드**도 있지만, 객체 생성과
참조 변수는 본격 OO인 **CH20**에서 다룬다. 지금은 **정적 메서드(`=>`)면 충분**하다.

## 흔한 실수와 점검

- **DEFINITION만, IMPLEMENTATION 없음** → 메서드 본문이 없어 오류.
- **정적인데 객체 만들기** → `CLASS-METHODS`는 객체 없이 `=>`로 부른다.
- **`=>` vs `->`** → 정적은 `=>`(클래스로 직접), 인스턴스는 `->`(객체로, CH20).
