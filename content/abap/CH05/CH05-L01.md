## 사람 한 명에 변수가 셋

지난 챕터의 [구구단 프로그램](CH04-L07.html)에서 `gv_dan`·`gv_mul`·`gv_res` **변수
세 개**를 따로 들고 다녔다. 이번엔 더 실전적인 데이터인 **사람**을 다뤄 보자. 이름(문자)·
나이(정수)·금액(소수)을 담으려면 역시 변수 셋(`gv_name`·`gv_age`·`gv_amount`)이
필요하다.

한 명이면 참을 만하다. 그런데 사람이 **두 명, 세 명**이 되면? 변수 이름에 **번호를
붙이기 시작**한다. `gv_name2`, `gv_age2`, `gv_name3`… 선언은 사람마다 3줄씩
불어나고, "이 셋이 한 사람"이라는 묶음은 코드 어디에도 없이 **머릿속에만** 있다.
게다가 같은 선언을 반복해서 적다 보면 **오타·타입 실수**가 끼어드는데, 이를 막아
줄 장치도 없다. 아래에서 직접 눈으로 확인해 보자. **숨은 실수 세 개**도 찾아보자.

::embed CH05-L01-S07 | 단일 값 변수만으로 사람 셋 — 선언의 벽 | 560::

관련된 값을 **하나로 묶는 그릇**이 필요하다. 그게 [[Structure]](구조체)다.

## BEGIN OF ~ END OF

[[DATA]] 문에서 `BEGIN OF` 뒤에 Structure 이름을 적고, `END OF`로 닫기 전까지 묶을 필드를 나열하면 [[Structure]] 변수가 선언된다.

```abap
DATA: BEGIN OF gs_person,
        name   TYPE string,
        age    TYPE i,
        amount TYPE p LENGTH 8 DECIMALS 2,
      END OF gs_person.
```

`BEGIN OF`와 `END OF` 사이에 적은 `name`·`age`·`amount`는 [[Structure]]를 구성하는 [[Component]](구성 요소)라 부른다.
ABAP에서는 이렇게 만든 Structure 변수를 [[Work Area]](작업 영역)라고 부르기도 하며, 주로 한 건([[레코드]])을 담는 목적으로 쓰인다.

변수를 선언할 때는 아래처럼 **접두어**만으로 변수의 모양을 파악할 수 있도록 한다.

| 접두어 | 뜻 | 담는 것 | 예 |
|---|---|---|---|
| `gv_` | **g**lobal **v**ariable | **값 하나** (단일 값) | `gv_age` |
| `gs_` | **g**lobal **s**tructure | **값 여러 개** (Component 칸으로 묶음) | `gs_person` |

## Component 접근 — 하이픈(-)

[[Structure]] 안의 [[Component]]는 **하이픈 `-`** 으로 접근한다. 아래 예제로 사용법을 확인해 보자.

```abap
gs_person-name   = '정훈영'.
gs_person-age    = 30.
gs_person-amount = '1500.50'.

WRITE: / gs_person-name, 
         gs_person-age, 
         gs_person-amount.
```

## 여러 명 담기 — 이번엔 모양 선언이 반복된다

`gs_person`으로 한 명은 깔끔하게 묶었다. 그런데 사람이 **두 명, 세 명**이 되면?
첫 섹션의 번호 참사로 돌아갈 수는 없으니 사람마다 Structure 변수를 만들 텐데, 그러면
`BEGIN OF ~ END OF` **모양 선언이 통째로 반복**된다. 사람마다 5줄씩이다. 모양은
**한 번만** 적고, 변수만 여러 개 찍어내고 싶다.

얼마나 나아지는지는 규모로 체감된다. 실무의 "사람"은 정보가 셋일 리 없다.
성별·생일·부서·직급·입사일·연봉·연락처만 더해도 **정보 10개**다. 단일 값 변수는
**사람 수 × 정보 수**로 폭발하지만(겨우 3명인데 **변수 30개**), Structure와 타입을
쓰면 얘기가 달라진다. 아래 비교와 "실무 스케일"을 열어 확인해 보자.

## 타입으로 빼두기 (재사용)

Structure의 **모양**을 [[TYPES]]로 선언하면 **여러 변수가 같은 모양을 공유**한다. 모양은
한 번만 선언하고, 변수는 `TYPE ts_person`으로 여러 개 만든다. 한 명이 아니라 여러
명을 같은 틀로 찍어내는 셈이다. Structure Type 이름은 **`ts_`**(**t**ype **s**tructure)로
시작하는 게 SAP 표준 교재의 관례다.

```abap
TYPES: BEGIN OF ts_person,
         name   TYPE string,
         age    TYPE i,
         amount TYPE p LENGTH 8 DECIMALS 2,
       END OF ts_person.

" 같은 타입으로 변수 여러 개, 모양은 매번 다시 안 적는다
DATA: gs_p1 TYPE ts_person,
      gs_p2 TYPE ts_person,
      gs_p3 TYPE ts_person.

gs_p1-name   = '정훈영'.
gs_p1-age    = 30.
gs_p1-amount = '1500.50'.

gs_p2-name   = '손흥민'.
gs_p2-age    = 32.

gs_p3-name   = '유재석'.
gs_p3-age    = 51.
gs_p3-amount = '980.00'.
```

## LIKE — 기존 변수와 똑같은 모양으로

`TYPE`이 **타입을 보고** 만든다면, `LIKE`는 **이미 선언된 변수를 보고** 똑같은
모양으로 만든다. 타입을 따로 선언하지 않았어도, 잘 만든 변수 하나가 있으면 같은
모양을 바로 찍어낼 수 있다.

```abap
gs_p1-name   = '정훈영'.     " 원본 gs_p1에 값이 채워진 상태에서
gs_p1-age    = 30.
gs_p1-amount = '1500.50'.

DATA gs_p4 LIKE gs_p1.      " gs_p1과 똑같은 "모양"의 새 변수

gs_p4-name = '강호동'.       " 값이 필요하면 이렇게 따로 넣는다
```

`LIKE gs_p1`은 "`gs_p1`이 가진 **모양 그대로**"라는 뜻이다. Component 이름·타입·길이가 모두 같다.
여기서 **오해하기 쉬운 것 하나**: LIKE가 변수를 통째로 "복사"한다고 생각하면 안 된다.
복사되는 건 **모양뿐**이고, 원본 `gs_p1`에 들어 있던 값(정훈영·30·1500.50)은 **한 글자도
넘어오지 않는다.** `gs_p4`는 선언되는 순간 모든 Component가 **타입별 초기값**(정수 `0`,
문자열 `''`, `p`는 `0.00`)으로 시작한다. 아래 디버거에서 직접 확인해 보자.

> 💡 **`TYPE`이든 `LIKE`이든, 변수는 선언되는 순간 반드시 초기값을 가진다.** ABAP에는
> "값이 없는(미지정) 변수"가 없다. 안 넣었으면 그건 그냥 그 타입의 **초기값**이다.
