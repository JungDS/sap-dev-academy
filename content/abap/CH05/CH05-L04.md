## 묶었으면 다뤄야 한다

Structure를 통째로 복사하거나, 비우거나, **이름이 같은 필드만 골라 옮기는** 일이
자주 생긴다. 기본기 몇 가지를 익힌다.

## 통째로 대입 / 초기화

같은 타입이면 Structure끼리 **통째로** 대입된다. `CLEAR`로 전체를 비운다.

```abap
DATA: gs_a TYPE zst_person,
      gs_b TYPE zst_person.

gs_a-name = '정훈영'.
gs_b = gs_a.        " 모든 Component 복사
CLEAR gs_a.         " gs_a를 초기값으로
```

## MOVE-CORRESPONDING — 이름이 같은 필드만

타입이 다른 두 Structure 사이에서 **이름이 같은 Component만** 골라 옮긴다. 어떤 필드가
옮겨지고 어떤 필드가 남는지, 세 가지 경우가 다 들어 있는 예로 보자.

```abap
DATA: BEGIN OF gs_src,
        name   TYPE c LENGTH 20,
        age    TYPE i,
        amount TYPE p LENGTH 8 DECIMALS 2,   " 대상엔 없는 필드
      END OF gs_src.

DATA: BEGIN OF gs_tgt,
        name TYPE c LENGTH 20,
        age  TYPE i,
        city TYPE c LENGTH 20,               " 원본엔 없는 필드
      END OF gs_tgt.

gs_src-name   = '정훈영'.
gs_src-age    = 30.
gs_src-amount = '1500.50'.
gs_tgt-city   = 'Seoul'.

MOVE-CORRESPONDING gs_src TO gs_tgt.
" name·age만 복사된다. amount는 갈 곳이 없고, city는 건드리지 않는다
```

- **이름이 같은** `name`·`age`만 복사된다.
- 원본에만 있는 `amount`는 갈 곳이 없어 **무시**되고, 대상에만 있는 `city`는
  **건드리지 않는다**.
- 화면·테이블·DB의 Structure가 조금씩 다를 때 아주 유용하다.

> ⚠️ **"건드리지 않는다"가 함정이 된다** — 대상에만 있는 필드(`city`)는 **이전 값이
> 그대로 남는다.** 새로 채우는 Structure라면 옮기기 전에 `CLEAR gs_tgt.`부터. 안 그러면
> **지난 건의 값이 이번 건에 섞인다.**

## 매핑을 눈으로 — 같은 이름만 건너간다

## 흔한 실수와 점검

- **타입 다른데 통째 대입** → Component 구성이 다르면 오류/엉뚱한 결과. 다르면
  [[MOVE-CORRESPONDING]].
- **대상 잔존값** → 대상에만 있는 필드는 이전 값이 남는다. 새 건을 채울 땐
  `CLEAR` 먼저.
- **CLEAR vs REFRESH** → Structure는 `CLEAR`. (여러 행을 담는 내부 테이블 비우기는
  다음 챕터에서.)
