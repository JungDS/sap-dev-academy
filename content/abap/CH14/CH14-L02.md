## 모든 필드를 다 보여줄 필요는 없다

테이블을 만들면 모든 필드를 그대로 보여 주는 게 자연스러워 보인다. 하지만 실무는
다르다. 공연 테이블에 가격·내부 상태·생성자·변경일·기술 키가 함께 있어도, 어떤
사용자는 **공연 코드와 공연명만** 필요하다. 다 보여 주면 화면이 복잡해지고, 안 봐도
될 정보가 노출되며, 엉뚱한 필드를 업무 필드로 착각한다.

[[Projection View]]는 "같은 테이블을 **더 좁게** 보여 주는 관점"이다.

## Projection View — 한 테이블, 열 솎아내기

[[Projection View]]는 **하나의 basis table**에서 일부 필드만 보이게 하는 DDIC View다.
조인도 계산도 하지 않고, 행을 거르는 selection condition도 두지 않는다. 공식문서상
별도의 database SQL view도 만들어지지 않는다. 한마디로 **"한 테이블에서 열을 줄이는 View"** 다.

`zconcert`에 `concert_id, perf_id, date, hall_id, base_price, internal_status,
created_by, changed_at`이 있다고 하자. 업무 담당자가 기본 목록만 보면 된다면
Projection View `ZPV_CONCERT_BASIC`은 앞 네 필드만 노출할 수 있다. 읽는 모습은
테이블 읽기와 거의 같다.

```abap
DATA gt_basic TYPE STANDARD TABLE OF zpv_concert_basic.

SELECT * FROM zpv_concert_basic
  INTO TABLE gt_basic.
```

아래에서 노출할 필드를 켜고 끄며 사용자가 보게 되는 결과가 어떻게 바뀌는지,
key 필드를 빼면 무슨 일이 생기는지 직접 보자.

## 한계와 오해

- **행을 거르는 필터 View가 아니다** → 특정 `hall_id`만 보이게 하는 **행 조건**은
  Projection View의 역할이 아니다. 이 레슨에서는 **열을 줄이는** 도구로만 본다.
- **key 필드를 빼지 말 것** → 결과가 깔끔해 보여도 어떤 행인지 식별 못 하면 못 쓴다.
  최소한 업무 식별 key와 설명 필드는 남긴다.
- **권한 설계를 대신하지 않는다** → "안 보이게 했으니 보안 끝"이 아니다. 권한·유지보수
  허용·프로그램 권한 체크는 별도 주제다.
- **여러 테이블 결합** → Projection View는 단일 테이블 전용. 결합은 [[Database View]]나
  코드 JOIN. **계산·가공 컬럼**은 클래식 뷰가 아니라 CDS의 영역이다
  ([Chapter 22 · CDS View Entity](CH23-L01.html) 예고).

## 정리

- [[Projection View]]는 단일 테이블에서 **필요한 필드만** 드러내는 DDIC View다.
- 데이터를 복사하지 않고, 행을 새 조건으로 거르지도 않는다.
- "사용자에게 필요한 컬럼만 보여 주는 것도 설계"라는 감각이 핵심이다.
