## 마스터 데이터를 담당자가 직접

[Chapter 7 · Transparent Table](CH07-L01.html)에서 SE11의 Create Entries로 한 건씩 넣어 봤다.
학습 초반엔 편하지만, 실무에서는 개발자가 매번 운영 값을 대신 넣을 수 없다. 공연장
코드·좌석 등급·가격 구간처럼 **업무 담당자가 관리할 기준 데이터**는 표준 화면으로
유지보수할 수 있어야 한다. 게다가 서로 연결된 테이블이면 어느 순서로 열고 어떤 key가
이어지는지 기억해야 해서 실수도 잦다.

[[Maintenance View]]는 관련 테이블을 **하나의 유지보수 관점**으로 묶는 도구다.

## Maintenance View — 조회가 아니라 유지보수

[[Maintenance View]]는 **extended table maintenance**(확장 테이블 유지보수)를 위한 DDIC
view다. 공식문서 기준으로 **ABAP SQL로 읽는 대상이 아니며**, SE54에서 maintenance
dialog를 만들고 SM30·SM31 같은 표준 유지보수 흐름에서 쓴다.

핵심은 [[Database View]]와 **목적부터** 다르다는 점이다.

| 구분 | Database View | Maintenance View |
|---|---|---|
| 핵심 목적 | 프로그램 **조회**용 공통 관점 | 표준 **유지보수**용 관점 |
| ABAP SQL 읽기 | 가능 | **대상 아님** |
| 연결 근거 | join condition | **Foreign Key dependency** |
| 사용 화면 | SELECT·Data Browser | SE54·SM30·SM31 |
| 설계 질문 | 어떤 필드를 읽을까 | 어떤 데이터를 같이 관리할까 |

> ⚠️ Maintenance View를 "left outer join 조회 대안"으로 외우지 말자(흔한 오해). 그건
> 조회 관점이고, Maintenance View는 **유지보수 관점**이다. (outer join 성격은
> [Chapter 14 · Help View](CH14-L03.html)에 해당.)

## Foreign Key가 설계 기준

여러 테이블을 아무 조건으로나 붙이는 게 아니라 **[[Foreign Key]] 관계**로 연결한다.
([Chapter 9 · DDIC 관계](CH09-L01.html)의 관계가 여기서 또 쓰인다. 나선 재방문이다.) Foreign Key는
**input check·input help·view 생성**에서 평가된다. 단, ABAP SQL의 `SELECT`는 Foreign
Key를 **자동으로 평가하지 않는다**(DDIC 화면·표준 도구에서 강한 의미).

아래에서 `zconcert-hall_id → zhall-hall_id` Foreign Key를 켜고 끄며, 유지보수 화면에서
존재하지 않는 공연장 코드를 입력했을 때 input check가 어떻게 막는지 보자.

## 흔한 실수와 점검

- **유지보수 = 조회로 혼동** → Database View는 읽기, Maintenance View는 유지보수 토대.
  이름이 같아도 목적이 다르면 완전히 다른 도구다.
- **Foreign Key를 문서용 선으로** → input check·input help·view 생성에서 실제로 평가된다.
  (단 `SELECT` 조건이 자동으로 안전해지는 건 아니다.)
- **권한 없이 판단** → View·dialog가 있어도 authorization이 부족하면 못 연다. 표준
  유지보수는 **설정 + 권한**이 함께 필요하다.

## 정리

- [[Maintenance View]]는 조회 편의가 아니라 **유지보수 흐름**을 위해 존재한다.
- [[Foreign Key]]로 연결된 테이블을 표준 Table View Maintenance에서 함께 다룬다.
- ABAP SQL 일반 data source로 읽는 View가 **아니다**.
