## 입력 화면을 자동 생성

테이블이나 [[Maintenance View]]에 **표준 입력 화면**을 자동으로 만들어 주는 게
[[Table Maintenance Generator]]다. 만든 뒤엔 **SM30**으로 데이터를 입력·수정한다.

## 생성 (SE11 → Utilities)

`SE11` → 테이블/뷰 → Utilities → **Table Maintenance Generator**.

- Authorization Group(권한 그룹), 화면 번호, Maintenance Type(1단계/2단계) 지정.
- 생성하면 표준 유지보수 화면(추가·수정·삭제)이 만들어진다.

## 운영 (SM30)

```text
SM30 → 테이블/뷰 입력 → Maintain → 행 추가·수정·삭제 → 저장
```

- **SM30** — 단일 뷰/테이블 유지보수.
- 저장 시 (이송 대상이면) [[이송요청]]에 기록된다(CH01-L06의 그 흐름).

## One Step vs Two Step — 화면 단계 수

Generator를 만들 때 **Maintenance Type**(유지보수 유형)을 고른다.

- **One Step**(1단계) — **목록 화면 하나**에서 바로 입력·수정한다. 필드가 적은 단순 테이블에 알맞다.
- **Two Step**(2단계) — **목록(개요) 화면 + 상세 화면** 둘로 나뉜다. 한 행에 필드가 많아 한 화면에 다 안 들어갈 때, 목록에서 행을 고르고 상세 화면에서 편집한다.

> 💡 필드가 적으면 One Step, 많으면 Two Step. 나중에 바꿔 재생성할 수도 있다.

## Maintenance Dialog — 장단점

Generator가 만들어 주는 표준 유지보수 화면(Maintenance Dialog)은 **공짜로 얻는 게 많다.**

| 장점 | 단점 |
|---|---|
| 코딩 없이 입력·수정·삭제 화면 자동 생성 | 레이아웃 커스터마이즈가 제한적 |
| 입력검증·F4·이송 연동이 그대로(CH09) | 복잡한 업무 로직엔 부족 |
| 권한 그룹으로 접근 통제 | (일부는 이벤트로 보완) |

> 단순 마스터데이터 유지보수엔 이걸로 충분하다. 화면을 마음대로 꾸미거나 복잡한 로직이 필요하면 별도 화면(CH16 Dynpro)을 고려한다.

## View Cluster — 관련 뷰를 묶어 한 흐름으로

마스터 하나가 아니라 **연관된 여러 테이블**(예: 공연 → 회차 → 좌석등급)을 **함께** 유지보수해야 할 때가 있다. 여러 유지보수 뷰/테이블을 **하나의 트랜잭션으로 계층적으로** 묶은 게 [[View Cluster]](뷰 클러스터, `SE54`로 정의)다.

- **장점** — 헤더-상세-상세상세처럼 관련 데이터를 한 화면 흐름에서 일관되게 유지보수.
- **단점** — 설정이 복잡하다. **단일 테이블엔 과하다**(그냥 SM30).
- **언제** — 마스터 + 여러 종속 테이블을 늘 함께 손봐야 할 때.

## 흔한 실수와 점검

- **Generator 없이 SM30** → "유지보수 불가" 오류. 먼저 Generator로 화면 생성.
- **권한 그룹 미지정** → 운영에서 권한 통제가 안 된다.

빠르게 조회**하고 싶을 때 쓰는 도구가 따로 있다 → SE16N(다음 레슨).
