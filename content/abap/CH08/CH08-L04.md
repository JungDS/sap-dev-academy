## 같은 SELECT, 다른 담는 그릇

읽은 결과를 담는 방법은 여러 가지다. 상황에 맞게 고른다.

## INTO wa / INTO TABLE

```abap
SELECT SINGLE * FROM sflight INTO gs_flight WHERE carrid = 'KE' AND connid = '0701' AND fldate = '20260623'.
SELECT * FROM sflight INTO TABLE gt_flight WHERE carrid = 'KE'.
```

한 건은 Structure(`INTO gs_`), 여러 건은 Internal Table(`INTO TABLE`).

## INTO (필드들) — 개별 변수로

몇 개 값만 변수로 직접 받는다. **SELECT 필드 순서와 변수 순서가 1:1로** 맞아야 한다.

```abap
DATA: gv_occ TYPE sflight-seatsocc,
      gv_max TYPE sflight-seatsmax.

SELECT SINGLE seatsocc seatsmax FROM sflight
  INTO (gv_occ, gv_max)
  WHERE carrid = 'KE' AND connid = '0701' AND fldate = '20260623'.
```

## CORRESPONDING FIELDS OF — 이름 맞춰 담기

SELECT 필드와 받는 Structure의 **같은 이름 필드끼리** 자동으로 맞춰 담는다(순서 무관).

```abap
SELECT carrid connid fldate FROM sflight
  INTO CORRESPONDING FIELDS OF TABLE gt_flight
  WHERE carrid = 'KE'.
```

## APPENDING — 기존 테이블에 덧붙이기

`INTO TABLE`은 기존 내용을 **지우고** 채우지만, `APPENDING TABLE`은 **이어 붙인다.**

```abap
SELECT * FROM sflight APPENDING TABLE gt_flight WHERE carrid = 'LH'.
```

탭을 눌러 같은 결과가 **도착지별로 어떻게 담기는지** 직접 확인해 보자(순서 어긋남·이름 불일치·덮어쓰기 함정 포함).

## 흔한 실수와 점검

- **INTO (v1,v2) 순서 어긋남** → SELECT 필드 순서와 변수 순서가 1:1로 맞아야 한다.
- **CORRESPONDING 이름 불일치** → 이름이 다르면 안 담긴다. 이름을 맞추거나 별칭([Chapter 13 · JOIN](CH13-L01.html)의 `AS`).
- **INTO vs APPENDING** → INTO는 덮어쓰기, APPENDING은 누적.
- **sy-dbcnt를 누적 행 수로 오해** → `sy-dbcnt`는 **직전 SELECT가 넘긴 행 수**다(테이블 총 누적 아님). APPENDING으로 이어 붙였다면 총 행 수는 `DESCRIBE TABLE … LINES`로 따로 센다.
