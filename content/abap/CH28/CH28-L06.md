## 화면 검증과 저장 검증은 다르다

셀 단위 검증(L05)을 통과해도, **저장 직전** 전체 규칙(중복·정원 합계·권한)을 다시
본다. 화면 검증은 입력 편의, 저장 검증은 데이터 무결성의 **마지막 방어선**이다.

```abap
" 1) 변경분 확보(편집 내용을 내부 테이블에 반영)
go_grid->check_changed_data( ).

" 2) 전체 규칙 최종 검증
LOOP AT lt_perf INTO DATA(ls).
  IF ls-seats_sum > ls-capacity.
    MESSAGE '정원을 초과했습니다' TYPE 'E'.
  ENDIF.
ENDLOOP.

" 3) 통과하면 DML + 잠금 + 커밋(CH24/25)
MODIFY zperf FROM TABLE @lt_perf.
COMMIT WORK.
```

- `check_changed_data( )` — 편집 중인 값을 내부 테이블로 강제 반영.
- 최종 검증 통과 후에만 DML.

## 흔한 실수와 점검

- **check_changed_data 누락** → 마지막 셀 편집이 반영 안 된 채 저장될 수 있다.
- **화면 검증만 믿기** → 저장 전 전체 규칙을 다시 확인(무결성).


## 🧩 도전 과제 (서브퀘스트)

> 선택 과제 — 예매 좌석수를 표에서 직접 편집한다.

1. 예매 목록 Grid의 `SEATS` 컬럼을 편집 가능(`edit`)으로 만들고 입력 이벤트를 등록한다.
2. `DATA_CHANGED`에서 좌석수가 잔여석을 넘으면 그 셀에 오류를 등록(`add_protocol_entry`)한다.
3. (심화) 저장 직전 `check_changed_data` 후 정원 규칙을 최종 검증하고 `MODIFY`+`COMMIT`(CH24).

힌트: L01(edit)·L02(DATA_CHANGED)·L05(검증/오류표시)·L06(저장 전 검증).
