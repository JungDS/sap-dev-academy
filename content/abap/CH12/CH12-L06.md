## 화면 없이 조건 만들기

지금까지는 사용자가 선택화면에 입력하면 [[Range Table]]이 만들어졌다. 하지만 모든
조건이 화면에서 오지는 않는다. **배치 프로그램, 내부 호출, 기본 조건 보정, 권한에
따른 자동 제한**처럼 화면 없이 조건을 만들어야 할 때가 있다.

이때 필요한 게 Range Table **직접 조작**이다. 구조(`SIGN/OPTION/LOW/HIGH`)는 이미
[Range Table 구조](CH12-L01.html)에서 배웠다. 코드로도 결국 그 네 칸을 채운 행을 만들어 `APPEND`하면 된다.

## TYPE RANGE OF로 코드에서 만들기

```abap
DATA: lr_stat TYPE RANGE OF zbooking-status,
      ls_stat LIKE LINE OF lr_stat,
      lt_book TYPE TABLE OF zbooking.

CLEAR ls_stat.
ls_stat-sign   = 'I'.
ls_stat-option = 'EQ'.
ls_stat-low    = 'N'.
APPEND ls_stat TO lr_stat.

SELECT * FROM zbooking INTO TABLE lt_book
  WHERE status IN lr_stat.
```

줄별 의미는 이렇다.

| 코드 | 의미 |
|---|---|
| `lr_stat TYPE RANGE OF zbooking-status` | 상태 필드와 같은 타입의 Range Table을 만든다. |
| `ls_stat LIKE LINE OF lr_stat` | Range Table **한 행**을 담을 work area를 만든다. |
| `CLEAR ls_stat.` | 이전 행 값이 남지 않게 비운다. |
| `ls_stat-sign/option/low` | 포함(I)·정확히 같음(EQ)·상태 N으로 한 행을 채운다. |
| `APPEND ls_stat TO lr_stat.` | 완성한 조건 행을 Range Table에 **추가**한다. |

> 💡 옛 코드에는 `RANGES r_stat FOR zbooking-status.` 형태도 보인다 — 화면 없이 Range
> Table을 선언하는 **레거시**(헤더라인식) 방식이다. 공식 문서상 *obsolete*라, 읽을 수는
> 있어야 하지만 새 코드는 위처럼 **`TYPE RANGE OF`**를 쓴다. 참고로 [[SELECT-OPTIONS]]는
> 같은 Range Table을 *화면 입력칸과 함께* 만들고, `TYPE RANGE OF`는 **코드 안의 table만**
> 만든다.

## 한 줄씩 따라가기

핵심은 **`APPEND` 전에는 `lr_stat`이 비어 있다**는 점이다. `ls_stat`에 값을 채워도
그건 work area일 뿐, `APPEND`해야 비로소 Range Table에 행이 생긴다. 한 줄씩 실행하며
`ls_stat`와 `lr_stat`이 어떻게 바뀌는지 보자.

## 흔한 실수와 점검

- **`SIGN`/`OPTION`을 비운 채 `APPEND`** → 행은 네 칸이지만 최소 `SIGN`·`OPTION`·`LOW`는
  조건에 맞게 채워야 한다.
- **`CLEAR` 생략** → work area를 재사용할 때 안 비우면 앞 행 값이 남는다. 특히 앞 행이
  `BT`라 `HIGH`를 썼고 다음이 `EQ`라면, `HIGH`가 남아 해석을 헷갈리게 만든다.
- **`RANGES`를 새 기준처럼 쓴다** → 레거시 인지용일 뿐, 새 코드는 `TYPE RANGE OF`.

## 정리

- 화면 없이도 `TYPE RANGE OF`로 [[Range Table]]을 만들 수 있다.
- 한 조건 행은 work area에 `SIGN`·`OPTION`·`LOW`(·`HIGH`)를 채운 뒤 `APPEND`한다.
- `SELECT-OPTIONS`는 화면+selection table, `TYPE RANGE OF`는 코드 안 range table만.
- `RANGES`는 레거시 인지용이다.
