## 1번 + N번 = 재앙

LOOP를 돌며 매 행마다 SELECT하면, 행이 1만이면 SELECT가 1만 번이다(DB 왕복 폭발).
이게 가장 흔한 성능 사고다.

```abap
" ❌ 안티패턴
LOOP AT lt_booking INTO DATA(b).
  SELECT SINGLE artist FROM zconcert
    WHERE concert_id = @b-concert_id INTO @DATA(lv_artist).   " N번 왕복
ENDLOOP.
```

## 고치는 법 — 한 번에 읽기

```abap
" ✅ 미리 한 번에 읽어 두고 메모리에서 찾기
SELECT concert_id, artist FROM zconcert
  FOR ALL ENTRIES IN @lt_booking
  WHERE concert_id = @lt_booking-concert_id
  INTO TABLE @DATA(lt_concert).
SORT lt_concert BY concert_id.

LOOP AT lt_booking INTO DATA(b2).
  READ TABLE lt_concert INTO DATA(c) WITH KEY concert_id = b2-concert_id BINARY SEARCH.
ENDLOOP.
```

- **FOR ALL ENTRIES**(CH13) 또는 **JOIN**으로 한 번에.
- 또는 사전 조회 후 **내부 테이블에서 READ**(BINARY SEARCH, CH06).

## N을 늘려 보면 — 왕복 폭발

예매 건수 N을 키우며 DB 왕복이 어떻게 폭발하는지(1+N) vs 사전 조회(2)를 비교해 보라.

## 흔한 실수와 점검

- **FAE 빈 테이블** → 드라이버가 비면 전체 조회된다. 비었으면 건너뛴다.
- **중첩 SELECT** → 절대 금지(CH08). JOIN/FAE로.
