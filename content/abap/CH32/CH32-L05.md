## 규모가 커지면 설계가 다르다

수백만 건은 "잘 짠 SELECT" 이상의 **설계**가 필요하다.

- **패키지 처리**(CH24-L05) — 일정 건수마다 처리·COMMIT로 메모리·잠금 관리.
- **병렬 처리** — 작업을 나눠 동시에(`aRFC`·병렬 프레임워크). 단 복잡·자원 고려.
- **Code Pushdown** — 가공을 ABAP이 아니라 **DB(HANA)에서** 하도록 — CDS·AMDP(CH33)·
  modern SQL 식(CH19). "데이터를 ABAP으로 끌어와 LOOP" 대신 **DB에서 집계해 결과만**.

```abap
" 끌어와서 ABAP 집계(느림) 대신
SELECT carrid, SUM( seatsocc ) AS occ      " DB에서 집계(빠름)
  FROM sflight GROUP BY carrid
  INTO TABLE @DATA(lt).
```

## 흔한 실수와 점검

- **전부 메모리로** → 수백만 건을 한 번에 itab에 담으면 메모리 폭발. 패키지·스트리밍.
- **ABAP에서 집계** → DB가 잘하는 일(집계·정렬)은 DB로 내린다.
