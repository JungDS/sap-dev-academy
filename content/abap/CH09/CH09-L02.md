## 비슷해 보이는 둘

[[Domain]]을 만들 때 **Value Table**을 지정할 수 있고, 테이블 필드엔
[[Foreign Key]]를 건다. 둘 다 "어떤 테이블의 값으로 제한"하는 것 같은데 뭐가
다른가? 헷갈리기 쉬운 지점이라 분명히 짚는다.

## Value Table = Domain의 '제안'

- [[Value Table]]은 **Domain 수준**에서 "이 도메인 값은 보통 이 테이블에서 온다"고
  **제안**하는 것이다.
- 그 자체로는 검증을 **강제하지 않는다.** 어디까지나 기본 제안.

## Foreign Key = 테이블 필드의 '실제 관계'

- [[Foreign Key]]는 **테이블 필드 수준**에서 거는 **실제 관계**다.
- 이게 있어야 입력검증·F4가 **실제로 동작**한다.
- Foreign Key를 만들 때, Domain에 Value Table이 있으면 시스템이 그걸
  **Check Table로 자동 제안**한다 → 둘은 협력 관계.

| | Value Table | Foreign Key |
|---|---|---|
| 정의 위치 | Domain | 테이블 필드 |
| 역할 | 제안(기본값) | 실제 관계·검증 |
| 검증 강제 | ✗ | ✓ |

## 보너스 — Domain의 변환 루틴(ALPHA)

[[Domain]]에는 **변환 루틴(Conversion Routine)** 도 달 수 있다. 대표가 `ALPHA`다.
공연번호·사번처럼 숫자로 된 ID를, **내부 저장은 앞자리 0을 채우고**(`0000000042`)
**화면 표시는 0을 떼서**(`42`) 보여 주는 식으로 형식을 자동 변환한다.

- 같은 ID라도 "내부 형식"과 "표시 형식"이 다를 수 있다는 점을 기억해 둔다.
- 코드에서 조회·비교 시 형식이 안 맞아 "없음"이 나오면 변환 루틴(ALPHA) 때문일 수
  있다. 표준 변환 기능으로 내부/표시 형식을 맞춘다.

> CH03에서 "변환 루틴은 개념만"이라 했던 그 ALPHA가 여기서 실제로 쓰인다 — 나선형 재방문.

## Text Table — 코드 옆에 이름표

마스터 테이블은 보통 **코드**(예: 공연번호 `C001`)만 키로 들고, 사람이 읽을 **이름**(공연명)은 **별도 테이블**에 둔다. 그 별도 테이블이 [[Text Table]](텍스트 테이블)이다.

- 키 = **마스터 키 + 언어(`SPRAS`)**. 그래서 같은 코드의 이름을 **언어별**로 가질 수 있다(한국어 "콘서트", 영어 "Concert").
- 마스터와 Foreign Key로 잇고, SE11에서 그 관계를 **Text Table로 표시**한다.
- 그러면 화면·[[F4]]·Search Help가 코드 옆에 **이름을 자동으로** 보여 준다(코드만 보면 무슨 공연인지 모르니까).

| 테이블 | 키 | 담는 것 |
|---|---|---|
| `ZCONCERT`(마스터) | `CONCERT_ID` | 정원·장소 등 |
| `ZCONCERT_T`(Text) | `CONCERT_ID` + `SPRAS` | 공연명(언어별) |

> 💡 표준 예: 항공사 마스터 `SCARR`의 이름은 그 안에, 많은 표준 마스터는 코드와 **언어별 텍스트**를 텍스트 테이블로 나눠 둔다. "코드는 마스터, 이름은 Text Table"로 기억하자.

## 흔한 실수와 점검

- **Value Table만 믿기** → 그것만으론 검증 안 된다. 필드에 Foreign Key를 걸어야.
- **Check Table ≠ Value Table** → 보통 같게 쓰지만, Foreign Key에서 다른 Check
  Table을 지정할 수도 있다.
