## 검증은 누가 하나

DDIC의 [[Foreign Key]]·[[고정값]]·[[Search Help]]는 **선언적 검증**을 해준다.
그런데 "퇴직자는 못 고른다", "금액 > 한도면 거부" 같은 **업무 규칙**은? 어디까지
DDIC가 하고 어디부터 코드가 해야 하는지 가른다.

## 두 계층

| 검증 | 담당 | 예 |
|---|---|---|
| **존재/형식** | DDIC(선언적) | 국가코드가 실제 존재? 고정값 범위? |
| **업무 규칙** | 프로그램(코드) | 한도 초과? 상태 전이 허용? 권한? |

## DDIC 검증의 한계

- DDIC 검증(Foreign Key 등)은 **화면/유지보수** 레벨에서 동작한다.
- 프로그램이 `INSERT`/`MODIFY`로 **직접 DB에 쓰면** 그 검증은 **자동 적용되지
  않는다.** 직접 쓰기 전엔 프로그램이 책임지고 확인해야 한다.

```abap
" 직접 쓰기 전, 존재 여부를 코드로 확인(예)
SELECT SINGLE country FROM ztcountry
  INTO lv_dummy WHERE country = ls_person-country.
IF sy-subrc <> 0.
  " 잘못된 국가코드 — 업무 처리(메시지/중단)
ENDIF.
```

## 원칙

- **할 수 있으면 DDIC로**(선언적·재사용·일관). 존재·형식·코드 목록은 DDIC.
- **DDIC로 표현 못 하는 업무 규칙만** 코드로. 중복 검증은 피한다.

## 흔한 실수와 점검

- **코드로 다 검증** → DDIC가 해줄 걸 중복 구현. 유지보수 부담.
- **DDIC만 믿고 직접 쓰기** → 직접 DB 쓰기는 DDIC 검증을 우회한다. 코드로 보완.
