## 테스트가 어려운 코드

패턴을 잘 적용해도 **검증할 수 없으면** 유지보수는 여전히 불안하다. 특히 DB·화면에 직접
붙은 코드는 자동 테스트가 어렵다. 이런 메서드를 보자.

```abap
METHOD remaining.
  SELECT SINGLE seats_total, seats_used
    FROM zbooking
    WHERE booking_id = @iv_booking_id
    INTO @DATA(ls_booking).
  rv_remaining = ls_booking-seats_total - ls_booking-seats_used.
ENDMETHOD.
```

이걸 테스트하려면 실제 `ZBOOKING`에 데이터가 있어야 한다. 다른 개발자·QA 데이터에 흔들리고,
DB가 비면 실패한다. *계산 로직만* 확인하고 싶은데 DB 준비가 더 큰 일이 된다.

→ **Testable Class**(테스트 가능한 클래스)는 계산 로직과 외부 의존성(DB·화면·파일·네트워크)을
**분리**한다. 그리고 **ABAP Unit**(코드를 자동 실행해 기대값↔실제값을 비교해 주는 단위 테스트
도구)으로 그 로직만 빠르게 검증한다.

## 의존성을 주입한다

핵심은 **의존성 주입**(Dependency Injection — 필요한 외부 객체를 class 안에서 *직접 만들지
않고* 밖에서 넣어 주는 설계)이다. 그래야 테스트에서 진짜 DB 객체 대신 가짜를 넣을 수 있다.

먼저 DB 접근을 **interface**로 분리한다.

```abap
INTERFACE zif_booking_repo.
  METHODS get_booking
    IMPORTING iv_booking_id     TYPE zbooking_id
    RETURNING VALUE(rs_booking) TYPE zbooking.
ENDINTERFACE.
```

업무 class는 구체 DB class를 직접 만들지 않고 이 interface를 **주입받는다**.

```abap
CLASS zcl_booking_manager DEFINITION.
  PUBLIC SECTION.
    METHODS constructor
      IMPORTING io_repo TYPE REF TO zif_booking_repo.
    METHODS remaining
      IMPORTING iv_booking_id       TYPE zbooking_id
      RETURNING VALUE(rv_remaining) TYPE i.
  PRIVATE SECTION.
    DATA mo_repo TYPE REF TO zif_booking_repo.
ENDCLASS.

CLASS zcl_booking_manager IMPLEMENTATION.
  METHOD constructor.
    mo_repo = io_repo.
  ENDMETHOD.
  METHOD remaining.
    DATA(ls_booking) = mo_repo->get_booking( iv_booking_id = iv_booking_id ).
    rv_remaining = ls_booking-seats_total - ls_booking-seats_used.
  ENDMETHOD.
ENDCLASS.
```

이제 `remaining`은 DB가 아니라 **주입받은 `mo_repo`** 에게 데이터를 묻는다 — 그 자리에 진짜를
넣든 가짜를 넣든 코드는 그대로다.

## 가짜로 갈아끼우고 — ABAP Unit

테스트에선 DB 대신 **Mock**(목 — 테스트용 가짜 구현)을 넣는다. 필요한 값만 돌려주면 된다.

```abap
CLASS ltd_booking_repo DEFINITION FOR TESTING.
  PUBLIC SECTION.
    INTERFACES zif_booking_repo.
ENDCLASS.

CLASS ltd_booking_repo IMPLEMENTATION.
  METHOD zif_booking_repo~get_booking.
    rs_booking-seats_total = 100.
    rs_booking-seats_used  = 50.
  ENDMETHOD.
ENDCLASS.
```

그리고 **ABAP Unit 테스트 class**가 이 Mock을 주입해 로직을 검증한다.

```abap
CLASS ltcl_booking_manager DEFINITION
  FOR TESTING RISK LEVEL HARMLESS DURATION SHORT.
  PRIVATE SECTION.
    METHODS remaining_calc FOR TESTING.
ENDCLASS.

CLASS ltcl_booking_manager IMPLEMENTATION.
  METHOD remaining_calc.
    DATA(lo_mgr) = NEW zcl_booking_manager(
      io_repo = NEW ltd_booking_repo( ) ).
    DATA(lv_rem) = lo_mgr->remaining( iv_booking_id = '1001' ).
    cl_abap_unit_assert=>assert_equals(
      act = lv_rem
      exp = 50 ).
  ENDMETHOD.
ENDCLASS.
```

- `FOR TESTING` class에 `FOR TESTING` 메서드를 둔다.
- `cl_abap_unit_assert=>assert_equals( act exp )`로 기대값을 검증한다.
- **Mock 덕에 DB 없이** 100−50=50 계산만 빠르게 확인한다.

## 테스트를 돌려 보기

테스트를 실행하면 초록 ✓가 뜬다. 그 상태에서 **버그 주입**을 켜면(계산이 잘못된 값을 반환)
해당 테스트가 빨갛게 실패하는 걸 보라. *좋은 테스트는 정상일 때 통과하고, 버그를 넣으면
반드시 실패한다* — 이렇게 **회귀를 자동으로 잡는 것**이 테스트의 진짜 가치다.

## 흔한 실수와 점검

- **DB 붙은 class에 테스트만 억지로** → 데이터 준비에 의존해 느리고 불안정. 먼저 의존성을
  interface로 분리한다.
- **Mock이 실제보다 복잡** → Mock은 *테스트에 필요한 값만* 반환한다. 운영 DB를 다 흉내 내면
  또 하나의 유지보수 대상이 된다.
- **테스트가 구현 세부를 너무 앎** → 결과(`remaining`=50)만 검증하면 충분. 내부 변수까지
  파고들면 리팩터링이 막힌다.
- **Singleton·static 상태 방치** → 이전 테스트가 바꾼 정적 상태가 다음 테스트에 남아 순서에
  따라 결과가 달라진다. 의존성은 **생성자 주입**으로 명시한다.
- **"테스트는 QA 일"** → 단위 테스트는 *설계 피드백*이다. 테스트가 어렵다면 class가 외부에
  너무 붙어 있다는 신호다.

## 정리

- 테스트 가능한 설계는 **외부 의존성 분리**에서 시작한다.
- 업무 class가 DB를 직접 만들지 않고 interface를 주입받으면 **Mock으로 교체**할 수 있다.
- ABAP Unit = `FOR TESTING` class·메서드 + `cl_abap_unit_assert`로 결과 검증.
- Factory·Singleton·Strategy·MVC도 결국 **테스트 가능한 구조**로 이어져야 가치가 크다.

> 선택 과제 — 이 챕터의 네 가지 설계 판단으로 예매 로직을 정리한다.

1. 일반/VIP 예매 객체 생성을 [Factory](CH26-L01.html)로 모은다.
2. 요금 계산을 [Strategy](CH26-L03.html)(인터페이스)로 분리해 정책을 갈아끼운다.
3. 화면·로직·데이터를 [MVC](CH26-L04.html)로 나눈다(작은 리포트면 과분리 주의).
4. (심화) `remaining()`이 repository interface를 주입받게 만들고, **ABAP Unit**에서 Mock으로
   테스트한다.

> 💬 한 문장으로 말해 보기 — *"Factory=생성 규칙을 한 곳에 · Singleton=하나여야만 하는
> 것에만 · Strategy=자주 바뀌는 알고리즘 분리 · MVC=변경 영향 범위를 예측 가능하게 · ABAP
> Unit은 의존성을 먼저 분리해야 산다."*
