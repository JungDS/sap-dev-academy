## 안전한 변경의 정석

[직접 DB 변경(DML·COMMIT)](CH25-L01.html)과 이 챕터의 잠금을 묶으면 **동시성까지 안전한 변경 흐름**이
완성된다. 먼저 전체 흐름을 한눈에 보자 — 어디서 갈라지고, 잠금이 언제 풀리는지.

::embed CH26-L05-S01 | 안전 변경 패턴 흐름도 | 460::

그대로 코드로 옮기면 이렇다.

```abap
" 1) 잠금 (변경 전에 편집권 확보)
CALL FUNCTION 'ENQUEUE_EZ_BOOKING'
  EXPORTING  mode_zbooking = 'E'
             booking_id    = lv_id
             _scope        = '2'
  EXCEPTIONS foreign_lock  = 1 OTHERS = 2.
IF sy-subrc <> 0.
  MESSAGE '편집 중입니다' TYPE 'E'.
ENDIF.

" 2) 읽기 → 3) 변경
SELECT SINGLE * FROM zbooking WHERE booking_id = @lv_id INTO @DATA(ls).
ls-seats = lv_new_seats.
ls-changed_by = sy-uname. ls-changed_at = sy-uzeit.
UPDATE zbooking FROM @ls.

" 4) 확정 / 취소
IF sy-subrc = 0.
  COMMIT WORK.
ELSE.
  ROLLBACK WORK.
ENDIF.

" 5) 해제(_scope=2면 COMMIT/ROLLBACK이 보통 자동 해제하나 명시도 안전·_scope는 ENQUEUE와 동일하게)
CALL FUNCTION 'DEQUEUE_EZ_BOOKING'
  EXPORTING mode_zbooking = 'E' booking_id = lv_id _scope = '2'.
```

## 흔한 실수와 점검

- **순서 어긋남** → 잠금은 변경 **전**, 해제는 커밋 **후**.
- **롤백 시 잠금** → `ROLLBACK`도 잠금을 해제한다(보통). 흐름을 일관되게.
- **거절 후 진행** → `foreign_lock`인데 무시하고 변경하면 동시성 보호가 무의미.

## 정리

- 안전 변경 = **ENQUEUE → READ → UPDATE → COMMIT →(DEQUEUE)** 한 흐름.
- 잠금은 변경 전, 해제는 커밋 후. `COMMIT`/`ROLLBACK`이 자동 해제(정확히는 `_scope`).
- 이 패턴 하나로 [원자성(COMMIT/ROLLBACK)](CH25-L02.html)과 이 챕터의 동시성(잠금)이 합쳐진다 — DML·COMMIT만
  있으면 동시 변경을 못 막고, ENQUEUE만 있고 COMMIT 판단이 없으면 원자성이 깨진다.

## 🧩 도전 과제 (서브퀘스트)

> 선택 과제 — 예매 변경에 동시성 제어를 입힌다.

1. 예매 수정 전 `ENQUEUE_EZ_BOOKING`으로 잠그고, 끝나면 `DEQUEUE`(또는 COMMIT 시 자동 해제).
2. 다른 세션이 같은 예매를 잠그면(`foreign_lock`) "편집 중" 메시지로 정중히 거절.
3. (심화) 낙관적 잠금 — 읽을 때의 `changed_at`을 저장 직전 DB와 비교해 충돌을 감지.
