## 검증 가능하게 설계

DB·화면에 딱 붙은 코드는 자동 테스트가 어렵다. **인터페이스로 의존성을 분리**하면
가짜 구현(Mock)으로 갈아끼워 **단위 테스트**할 수 있다.

```abap
" 의존성을 인터페이스로 주입받기
METHOD constructor.
  mo_repo = io_repo.        " zif_booking_repo (실제 or Mock)
ENDMETHOD.
```

## ABAP Unit — 테스트 클래스

```abap
CLASS ltcl_test DEFINITION FOR TESTING DURATION SHORT RISK LEVEL HARMLESS.
  PRIVATE SECTION.
    METHODS remaining_calc FOR TESTING.
ENDCLASS.

CLASS ltcl_test IMPLEMENTATION.
  METHOD remaining_calc.
    DATA(lo) = NEW zcl_booking_manager( io_repo = NEW ltd_repo_mock( ) ).
    cl_abap_unit_assert=>assert_equals( act = lo->remaining( ) exp = 50 ).
  ENDMETHOD.
ENDCLASS.
```

- `FOR TESTING` 클래스에 테스트 메서드를 둔다.
- `cl_abap_unit_assert`로 기대값 검증.
- Mock(가짜 의존성)으로 DB 없이 로직만 빠르게 검증.

## 흔한 실수와 점검

- **DB 직접 의존** → 테스트가 느리고 불안정. 인터페이스로 분리·Mock.
- **테스트 없음** → 리팩터링이 두려워진다. 핵심 로직엔 단위 테스트를.


## 🧩 도전 과제 (서브퀘스트)

> 선택 과제 — 예매 로직을 OO 패턴으로 다듬는다.

1. `ZCL_BOOKING_MANAGER`(CH20-L10)를 **Factory**로 생성하게 바꾼다(일반/VIP 예매).
2. 요금 계산을 **Strategy**(인터페이스)로 분리해 정책을 갈아끼운다.
3. (심화) `remaining()` 로직에 **ABAP Unit** 테스트를 작성하고 DB는 Mock으로 분리한다.

힌트: L01(Factory)·L03(Strategy)·L05(ABAP Unit/의존성 주입).
