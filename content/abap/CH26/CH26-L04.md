## 잃어버린 갱신 (Lost Update)

앞 레슨까지 "잠그면 막힌다"를 봤다. 그럼 **안 잠그면** 무슨 일이 날까? A와 B가 같은 예매를
각자 읽고, A가 저장한 **직후** B가 (자기가 읽었던 **옛 값** 기준으로) 저장하면 — **A의 변경이
조용히 사라진다**. 이게 **Lost Update**(잃어버린 갱신)다.

아래에서 전략을 바꿔 가며, 같은 순서의 작업이 어떻게 다른 결과를 내는지 직접 비교해 보라.

::embed CH26-L04-S01 | Lost Update 시뮬 — 전략별 결과 비교 | 470::

이걸 막는 두 전략이 있다.

| 전략 | 방식 | 언제 |
|---|---|---|
| **Pessimistic**(비관) | 미리 ENQUEUE로 **잠그고** 작업 | 충돌 잦음·편집 긺 |
| **Optimistic**(낙관) | 안 잠그고, 저장 직전 **변경 여부 확인** | 충돌 드묾·읽기 위주 |

## Optimistic — 타임스탬프 비교

읽을 때의 변경시각을 들고 있다가, 저장 직전 DB의 현재 변경시각과 **비교**한다. 다르면
"그 사이 누가 바꿨다" → 거절·재조회.

```abap
SELECT SINGLE changed_at FROM zbooking
  WHERE booking_id = @lv_id INTO @DATA(lv_now).
IF lv_now <> lv_read_at.            " 읽은 뒤 누가 바꿈
  MESSAGE '데이터가 변경되었습니다. 다시 조회하세요' TYPE 'E'.
ENDIF.
```

> **핵심은 "저장 직전 비교"다.** `changed_at` 컬럼이 *있다는 사실만*으로는 충돌을 막지 못한다 — 저장 직전에
> 읽은 값과 현재 값을 비교해 다르면 거절해야 비로소 낙관적 잠금이다. 그리고 `changed_at = sy-uzeit`는 학습용
> 예시일 뿐 — 날짜가 빠지고 같은 초에 두 변경이 들어오면 구분이 약하다. 운영에선 **timestamp·version
> counter·ETag** 같은 더 안정적인 변경표식을 쓴다. ABAP Cloud/RAP에서는 ETag와
> [Behavior Definition 기초](CH24-L04.html)의 `lock master`가 이 문제를 체계적으로 다룬다(이 챕터는 classic 중심).

## 흔한 실수와 점검

- **무조건 비관적 잠금** → 동시성↓. 충돌 빈도를 보고 전략 선택.
- **타임스탬프 미비** → 낙관적 잠금은 `changed_at` 같은 **변경 추적 컬럼**이 있어야 한다.
- **읽은 값으로 덮어쓰기** → 낙관 전략 없이 옛 값을 그대로 `UPDATE`하면 Lost Update.

## 정리

- **Lost Update** = 안 잠그고 옛 값 기준으로 저장 → 남의 변경이 사라짐.
- **Pessimistic**(미리 잠금) vs **Optimistic**(저장 직전 `changed_at` 비교) — 충돌 빈도로 선택.
- 낙관 전략은 **변경 추적 컬럼**이 전제.
