## 한 프로그램에 다 섞으면

전통적인 ABAP Report는 빠르다. `PARAMETERS` → `SELECT` → 계산 → 표시(`WRITE`나 ALV)를
한 파일에 순서대로 적으면 끝이다. **작은 보고서엔 이게 가장 실용적**이다. 문제는 보고서가
커질 때다.

예매 현황 리포트에 기능이 하나씩 붙는다고 해 보자.

- 공연별 조회 조건, 잔여석 계산
- 취소/확정 상태별 색상
- 더블클릭 상세, 엑셀 다운로드, 권한별 버튼

조회 SQL·계산·화면·이벤트·메시지가 **한 report에 뒤엉키면** 고칠 때 영향 범위를 알 수 없다.
*화면만 바꾸려는데 계산이 깨지고, 계산만 바꾸려는데 이벤트가 깨진다.* **MVC**(Model·View·
Controller — 데이터/로직·표시·흐름을 나누는 구조)는 이 섞임을 푸는 방법이다.

## 세 역할로 나누기

| 계층 | 책임 | ABAP 예 |
|---|---|---|
| **Model** | 데이터 조회·업무 계산 | `get_bookings( )`·`remaining( )` |
| **View** | 화면 표시 | ALV display·list 출력·화면 필드 갱신 |
| **Controller** | 흐름 조율 | 조건 수집 → Model 호출 → View 갱신 |

각 역할을 class로 잡으면 이렇게 된다(뼈대만).

```abap
CLASS zcl_booking_model DEFINITION.
  PUBLIC SECTION.
    METHODS constructor
      IMPORTING iv_concert TYPE zconcert_id.
    METHODS get_bookings
      RETURNING VALUE(rt_bookings) TYPE ztt_booking.
  PRIVATE SECTION.
    DATA mv_concert TYPE zconcert_id.
ENDCLASS.

CLASS zcl_booking_view DEFINITION.
  PUBLIC SECTION.
    METHODS display
      IMPORTING it_bookings TYPE ztt_booking.
ENDCLASS.
```

흐름은 **Controller**가 조율한다 — Model로 데이터를 얻어 View에 넘길 뿐이다.

```abap
METHOD run.
  DATA(lo_model) = NEW zcl_booking_model( iv_concert = p_concert ).
  DATA(lt_bookings) = lo_model->get_bookings( ).

  DATA(lo_view) = NEW zcl_booking_view( ).
  lo_view->display( it_bookings = lt_bookings ).
ENDMETHOD.
```

중요한 건 class **이름**이 아니다. *"조회·계산은 Model, 표시는 View, 실행 순서는
Controller"* 라는 **책임 분리**다.

## 한 계층을 바꾸면 어디까지?

MVC가 잘 됐는지는 **변경 시나리오**로 확인한다. 한 가지를 바꿀 때 *바뀌어야 하는 곳*은
분명하고, *나머지는 그대로*여야 한다.

| 변경 요구 | 바뀌어야 하는 곳 | 여기까지 바뀌면 = 분리 깨짐 신호 |
|---|---|---|
| ALV → list 출력 | View | Model SQL |
| 잔여석 계산식 변경 | Model | View 표시 코드 |
| 버튼 클릭 시 재조회 | Controller | Model 계산식 |

아래에서 한 계층을 바꿔 보고, **나머지가 그대로**인지 직접 확인해 보라 — 이게 역할을
나누는 진짜 이유다.

## 흔한 실수와 점검

- **Model에 화면 코드** → Model이 ALV field catalog·색상·메시지를 알기 시작하면 View와 다시
  붙는다. Model은 **순수한 데이터·업무 규칙**만 돌려준다.
- **View에서 DB 조회** → View가 표시 중에 `SELECT`를 또 하면 *"화면만 바꿨는데 조회 결과가
  달라지는"* 이상한 구조가 된다. 조회는 Model 몫.
- **Controller가 다 앎** → Controller가 계산 세부까지 들고 있으면 Model이 텅 빈다. Controller는
  실행 **순서만** 조율한다.
- **과분리** → 30줄짜리 일회성 리포트에 class 3개는 구조화가 아니라 의식(ritual)이 된다.
  기준은 *"변경이 예상되는가 · 테스트가 필요한가 · 화면/로직이 섞여 고통이 생겼는가."*

> 🔍 **코드 리뷰 체크** — ① Model 안에 `WRITE`·ALV display가 있나 ② View가 직접 `SELECT`
> 하나 ③ Controller가 계산 세부를 너무 아나 ④ **Model만 떼어 테스트**할 수 있나.

## 정리

- **Model**(로직)·**View**(표시)·**Controller**(흐름)로 관심사를 나눈다.
- 변경 요구별로 **수정 범위가 예측 가능**해지는 게 핵심 — 한 계층 변경이 나머지로 안 번진다.
- Model에 화면, View에 DB 조회가 들어가면 분리가 깨진다. 작은 리포트엔 과분리 주의.
