## 내부 테이블을 정하는 세 가지

`TYPE TABLE OF ty_person` 한 줄에는 사실 **세 가지 속성**이 들어 있다. 이 셋을
알면 테이블을 의도대로 설계할 수 있다.

1. **Line Type(행 모양)** — 한 행의 구조.
2. **Primary Key(기본 키)** — 행을 구분하는 기준.
3. **Table Kind(종류)** — 내부적으로 어떻게 저장·탐색하나.

## ① Line Type — 행은 꼭 구조가 아니어도 된다

행 모양은 보통 구조체지만, **숫자 하나·문자 하나** 같은 단일(elementary) 타입일 수도 있다.

```abap
DATA lt_num    TYPE TABLE OF i.           " 행이 정수 하나
DATA lt_person TYPE TABLE OF ty_person.   " 행이 구조체
```

행 모양은 단일형·구조 말고도 **Data Element·View·투명 테이블**(그 테이블의 한 행 모양)을 줄 수 있다 — CH05-L02의 "DDIC 타입으로 쓸 수 있는 것"과 같은 원리다.

```abap
DATA lt_amt   TYPE TABLE OF zde_amount.   " 행이 Data Element
DATA lt_fli   TYPE TABLE OF zsflight.     " 행이 투명 테이블의 행 모양
```

## ② Primary Key — 행을 구분하는 기준

키는 "어떤 필드로 행을 식별하나"이다. `TYPE TABLE OF`만 쓰면 기본은 **standard key**
(여러 필드 기준의 느슨한 키)가 잡힌다. 필요하면 키를 명시할 수도 있다(`WITH KEY ...`).

## ③ Table Kind — 세 종류

| 종류 | 저장 / 탐색 | 인덱스 접근 | 언제 |
|---|---|---|---|
| STANDARD | 입력 순서 유지, 순차 탐색 | O | 기본·순서 중요 |
| SORTED | 키로 자동 정렬, 이진 탐색 | O | 정렬 유지·빠른 범위 |
| HASHED | 해시로 단건 초고속 | X | 유일 키 대량 단건 |

```abap
TYPES ty_person_tab TYPE STANDARD TABLE OF ty_person.
DATA  lt_person     TYPE ty_person_tab.
```

- STANDARD·SORTED는 **Index Table**(번호 `sy-tabix`로 접근 가능), HASHED는 인덱스가 없다.
- 처음엔 **STANDARD**로 충분하다. SORTED/HASHED는 성능이 절실할 때 다시 본다.

## 로컬 vs 전역(DDIC) Table Type

구조체처럼 테이블 타입도 로컬(`TYPES`)·전역(SE11 Table Type)으로 둘 수 있다.

`SE11` → Data Type → **Table Type** → `ZTT_PERSON` → 행 타입에 구조 `ZST_PERSON` →
활성화. 그러면 여러 프로그램이 같은 테이블 타입을 공유한다.

```abap
DATA lt_person TYPE ztt_person.   " 전역 Table Type
```

## 흔한 실수와 점검

- **처음부터 SORTED/HASHED 남발** → 기본은 STANDARD. 측정 후 바꾼다.
- **HASHED에 인덱스 접근** → HASHED는 `sy-tabix`/INDEX 접근 불가(키로만 찾는다).
- **행 타입 먼저** → 테이블 타입의 "행"은 구조(또는 단일형). 행 모양을 먼저 정한다.

---

테이블을 어떻게 정의하는지 알았다. 이제 **한 행을 콕 집어** 넣고·찾고·고치는 법이다
→ 단일행 접근(다음 레슨).
