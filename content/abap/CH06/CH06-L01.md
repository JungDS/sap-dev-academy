## 한 건으로는 부족하다

[Chapter 5 · Structure](CH05-L01.html)에서 만든 Structure는 사람 **한 명**을 담는다.
[구구단 캡스톤](CH05-L05.html)에서도 봤듯, 반복이 돌 때마다 `gs_line`은 **덮어써졌다**.
화면에는 `WRITE` 줄이 9줄 남지만, 메모리에는 늘 마지막 한 줄뿐이었다.

그런데 실제 업무는 거의 항상 **여러 건**을 다룬다.

| 상황 | 한 건 | 여러 건 |
|---|---|---|
| 사원 | 사원 한 명 | 사원 명단 |
| 주문 | 주문 한 건 | 오늘의 주문 목록 |
| 구구단 | `2 x 1 = 2` 한 줄 | 2～9단 81줄 |

같은 모양의 행을 **여러 개** 쌓아 두는 그릇이 필요하다. 그게
[[Internal Table]](내부 테이블)이다.

> ⚠️ **Internal Table ≠ Database Table** — 가장 흔한 입문 오해다. Internal Table은 프로그램이
> 도는 **동안만 메모리**에 있다가 끝나면 사라진다. 영구 저장은 DB 테이블([Chapter 7 ·
> Transparent Table](CH07-L01.html))의 몫이다. 이 둘을 구별하지 못하면 로직 설계 전체가 흔들린다.

## 선언 — TYPE TABLE OF

행의 모양(Structure 타입) 위에 `TABLE OF`를 붙인다.

```abap
TYPES: BEGIN OF ts_person,
         name TYPE string,
         age  TYPE i,
       END OF ts_person.

DATA: gs_person TYPE ts_person,          " 한 건(Work Area)
      gt_person TYPE TABLE OF ts_person. " 여러 건(Internal Table)
```

- `gt_` 접두어가 관례다(**g**lobal **t**able: 아직 전역 스코프). `it_`는 Header Line을 가진 옛 방식 테이블에만 쓴다.
- 한 건을 담는 [[Work Area]](`gs_`)와 짝지어 쓴다. Work Area는 "지금 만들고 있는 한 행"을 담는 그릇이다.

## 행 추가 — APPEND

[[Work Area]]에 값을 채우고 `APPEND`로 테이블 **끝에** 쌓는다.

```abap
gs_person-name = '정훈영'. gs_person-age = 30.
APPEND gs_person TO gt_person.

CLEAR gs_person.
gs_person-name = '홍길동'. gs_person-age = 25.
APPEND gs_person TO gt_person.
```

이제 `gt_person`에는 행이 둘이다. 여기서 중요한 사실 하나. `APPEND`는 Work Area를
테이블 안으로 옮기는 게 아니라 **현재 값을 복사해 새 행을 만든다**. 그래서 쌓은 뒤에
`gs_person`을 바꿔도 이미 들어간 행은 그대로다.

행이 몇 개인지는 `DESCRIBE TABLE`로 센다.

```abap
DATA gv_cnt TYPE i.
DESCRIBE TABLE gt_person LINES gv_cnt.   " gv_cnt = 2
```

## 비우기 — CLEAR / FREE

```abap
CLEAR gt_person.   " 행 전체 삭제(초기 할당 메모리는 남겨 둠)
FREE  gt_person.   " 행 삭제 + 초기 할당분까지 메모리 전부 반환
```

- 곧 **다시 채울** 테이블은 `CLEAR`. 남겨 둔 초기 메모리 덕에 다시 채우기 유리하다.
- 대용량 테이블을 **다 쓰고 더 안 쓸 때**는 `FREE`.

> 🕰️ **옛 코드에서 보이는 두 가지** — 지금은 안 쓰지만 알아본다.
> - `REFRESH gt_...` — 행 전체를 비우는 옛 문장(공식 문서 기준 obsolete). Header Line 없는
>   테이블에서는 `CLEAR`와 효과가 같다. 새 코드는 `CLEAR`/`FREE`만 쓴다.
> - **Header Line** — 테이블에 같은 이름의 Work Area가 딸리는 옛 단축 방식(역시 obsolete).
>   이 방식에서는 `CLEAR gt_...`가 그 헤더만 비워 혼란스러웠다. 우리는 Work Area를
>   `gs_`로 분리하므로 `CLEAR gt_...`가 행 전체를 비운다.

## 흔한 실수와 점검

- **APPEND 전에 CLEAR 안 함** → 이전 [[Work Area]] 값이 남아 섞일 수 있다(name만 바꾸면
  age는 앞사람 값 그대로). 새 행을 채우기 전에 `CLEAR gs_person` 습관.
- **Internal Table에 넣었으니 저장됐다는 착각** → 메모리일 뿐이다. 프로그램이 끝나면 사라진다.
- **헤더라인 옛 방식 흉내** → 오래된 코드를 읽을 때만 알아보면 된다. 지금은 `INTO gs_`
  작업영역 방식이 명확하고 안전하다.
