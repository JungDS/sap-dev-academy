## 행을 콕 집어 다루기

전체 순회 말고 **특정 한 행**을 넣고·찾고·고치는 일이 잦다. 그때마다 결과는
`sy-subrc`로 확인한다(0=성공).

## 넣기 — APPEND vs INSERT

- `APPEND ls TO lt.` — 맨 끝에 추가(STANDARD 테이블).
- `INSERT ls INTO lt INDEX n.` — n번째 위치에 끼워 넣기.
- `INSERT ls INTO TABLE lt.` — **키 순서에 맞게** 넣기(SORTED/HASHED의 표준 방식).

```abap
APPEND ls_person TO lt_person.            " 끝에
INSERT ls_person INTO lt_person INDEX 1.  " 맨 앞에
INSERT ls_person INTO TABLE lt_person.    " 키 기준(종류에 따라)
```

> `INTO`(위치 기반)와 `INTO TABLE`(키 기반)은 **다른 문장**이다. SORTED/HASHED엔
> `INTO TABLE`을 쓴다.

## 찾기 — READ TABLE

```abap
READ TABLE lt_person INTO ls_person WITH KEY name = '정훈영'.
IF sy-subrc = 0.
  WRITE: / '찾음:', ls_person-age.
ENDIF.
```

- `INDEX n` — n번째 행을 **번호로 직접** 읽기(예: `READ TABLE lt_person INTO ls_person INDEX 1.`).
- `WITH KEY field = ...` — 아무 필드로나 찾기(STANDARD는 순차 탐색).
- `WITH TABLE KEY ...` — **정의된 키**로 찾기(SORTED/HASHED에 최적).
- 못 찾으면 `sy-subrc = 4`. **반드시 확인**한다.

값은 필요 없고 **있는지만** 확인할 땐 `TRANSPORTING NO FIELDS`로 복사를 생략한다(빠름).

```abap
READ TABLE lt_person TRANSPORTING NO FIELDS WITH KEY name = '정훈영'.
IF sy-subrc = 0.  " 존재만 확인(Work Area 복사 없음)
ENDIF.
```

### 빠른 탐색 — BINARY SEARCH

STANDARD 테이블에서 단건을 자주 찾으면 순차 탐색이 느리다. **키로 정렬해 두고**
`BINARY SEARCH`를 붙이면 이진 탐색으로 빨라진다.

```abap
SORT lt_person BY name.                  " 반드시 키로 먼저 정렬
READ TABLE lt_person INTO ls_person
     WITH KEY name = '정훈영' BINARY SEARCH.
```

> ⚠️ `BINARY SEARCH`는 **그 키로 정렬돼 있어야** 정확하다. 정렬 안 하고 쓰면
> 있는 행도 "없음(sy-subrc=4)"으로 나온다.

## 고치기 · 지우기 — MODIFY · DELETE

```abap
MODIFY lt_person FROM ls_person INDEX sy-tabix.  " 현재 행 수정(LOOP 안)
MODIFY TABLE lt_person FROM ls_person.           " 키로 수정
MODIFY lt_person FROM ls_person INDEX sy-tabix
       TRANSPORTING age.                         " age 필드만 갱신(나머지 보존)
DELETE lt_person WHERE age < 20.                 " 조건 삭제
```

> `TRANSPORTING f1 f2`를 붙이면 **그 필드만** 갱신하고 나머지는 그대로 둔다.

## 흔한 실수와 점검

- **sy-subrc 미확인** → READ 실패 후 옛 Work Area 값을 그대로 쓰는 버그.
- **BINARY SEARCH 전에 SORT 안 함** → 잘못된 "없음" 결과.
- **INTO vs INTO TABLE 혼동** → 위치 기반인지 키 기반인지 구분.
