## 버튼이 눌렸을 때

[Toolbar Event](CH30-L03.html)에서 만든 버튼(`ZCANCEL`)이 눌리면 `user_command` 이벤트가
온다. 버튼을 만들어 놓고 처리하지 않으면 **버튼은 장식일 뿐**이다. 반대로 *어느 행을
대상으로 할지 확인도 없이* 취소를 수행하면 위험하다. `user_command`는 누른 버튼의 **명령
코드(`e_ucomm`)** 를 받아 업무로 분기한다.

```abap
METHODS on_user_command
  FOR EVENT user_command OF cl_gui_alv_grid
  IMPORTING e_ucomm.

METHOD on_user_command.
  CASE e_ucomm.
    WHEN 'ZCANCEL'.
      mo_grid->get_selected_rows(
        IMPORTING et_index_rows = DATA(lt_rows) ).

      IF lt_rows IS INITIAL.
        MESSAGE '먼저 취소할 행을 선택하세요.' TYPE 'I'.
        RETURN.
      ENDIF.

      LOOP AT lt_rows INTO DATA(ls_row).
        READ TABLE mt_booking ASSIGNING FIELD-SYMBOL(<ls_booking>)
          INDEX ls_row-index.
        IF sy-subrc = 0.
          <ls_booking>-status = 'CANCELLED'.
        ENDIF.
      ENDLOOP.

      mo_grid->refresh_table_display( ).
  ENDCASE.
ENDMETHOD.
```

- `get_selected_rows( )` — 사용자가 체크한 행들(`et_index_rows`).
- **선택이 없으면 안내 메시지로 돌려보낸다** — "선택 없음"은 오류가 아니라 조작 미완료다.
- 내부 테이블만 바꾸면 화면은 옛 상태 → `refresh_table_display( )`로 갱신해야 보인다.

이 코드는 **화면 실습용**이다. 진짜 예매 취소라면 [실무 데이터 변경과 트랜잭션
제어](CH25-L01.html)의 DML, [Lock Object와 동시성 제어](CH26-L01.html)의 잠금, 권한·오류
메시지·커밋/롤백 경계를 함께 설계해야 한다. 여기선 "어느 행을 어떤 업무 서비스로 넘길지"를
잇는 데 집중한다.

## 선택 → 취소 → 갱신 체험

행을 체크해 고르고 **예매 취소** 버튼을 누르면 `user_command`가 `ZCANCEL`로 분기해 선택
행을 취소하고 화면을 갱신한다. **아무것도 안 고르고** 누르면 어떻게 되는지도 꼭 확인해 보라.

## 흔한 실수와 점검

- **`CASE e_ucomm` 없이 바로 처리** → 버튼이 늘어나면 위험하다. 명령 코드는 반드시 분기하고,
  모르는 명령은 무시·로그로 남긴다.
- **선택 없음 피드백 누락** → 안 골랐는데 아무 반응이 없으면 멈춘 줄 안다. 더 위험한 건
  *선택이 없는데 전체 행을 처리*하는 것. 친절한 정보 메시지로 돌려보낸다.
- **갱신 누락** → 내부 테이블만 바꾸고 `refresh_table_display( )`를 안 하면 화면이 옛
  상태로 남는다.

## 정리

- `user_command` = 커스텀 버튼을 눌렀을 때 **명령 코드(`e_ucomm`)** 로 업무를 분기.
- `get_selected_rows( )`로 고른 행을 가져오고, **선택 없음은 안내**, 처리 후 **`refresh`로
  갱신**한다.
- 실제 데이터 변경은 DML·잠금·권한·커밋 경계가 필요 — 이벤트는 그 업무 서비스로 *잇는* 역할.
