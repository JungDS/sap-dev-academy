## 표준 버튼만으론 부족

ALV를 업무 화면으로 쓰다 보면 표 위에 "예매 취소"·"메일 발송"·"엑셀 다운로드" 같은
**화면 고유 기능**을 놓고 싶어진다. 표 밖에 버튼을 둘 수도 있지만, 표에서 행을 고른 뒤
바로 실행할 기능이면 **ALV 툴바 안**에 버튼을 두는 게 흐름이 자연스럽다.

`toolbar` 이벤트가 그 지점이다. ALV가 툴바를 구성할 때 끼어들어 버튼을 추가한다. 단 —
**`toolbar`는 버튼을 *만드는* 이벤트일 뿐, *눌렀을 때 처리*하는 건 다음 레슨의
[USER_COMMAND 처리](CH30-L04.html)** 다.

## 툴바에 버튼 추가 — toolbar

핸들러는 `e_object`를 받는다. 그 안의 `mt_toolbar`에 버튼 한 줄을 추가하면 툴바에 나타난다.

```abap
METHODS on_toolbar
  FOR EVENT toolbar OF cl_gui_alv_grid
  IMPORTING e_object.

METHOD on_toolbar.
  APPEND VALUE #(
    function  = 'ZCANCEL'
    icon      = icon_delete
    text      = '예매 취소'
    quickinfo = '선택한 예매를 취소합니다'
    butn_type = 0
  ) TO e_object->mt_toolbar.
ENDMETHOD.
```

- **`function`(`ZCANCEL`)이 핵심** — 버튼을 누르면 이 코드가 다음 이벤트(`user_command`)의
  `e_ucomm`으로 넘어온다.
- `text`·`quickinfo` = 사용자에게 보이는 이름·설명, `function` = 코드가 분기할 **내부 명령
  키**(둘은 다르다).
- 동적으로 툴바가 바뀌면 `go_grid->set_toolbar_interactive( )`로 다시 그리기도 한다.

이 코드는 버튼을 **추가만** 한다. `ZCANCEL`은 아직 취소 로직이 아니다.

## 추가된 버튼 확인

표준 버튼 옆에 끼워 넣은 커스텀 버튼(`ZCANCEL`)을 눌러, 어떤 `function` 코드로 등록됐는지
확인해 보라. 로그는 "버튼 추가"까지만 — 실제 취소·갱신이 아직 안 일어나는 게 **정상**이다
(다음 레슨).

## 흔한 실수와 점검

- **`toolbar`에서 바로 업무 처리(DB 취소 등)** → ⚠️ 툴바는 **여러 번 다시 그려진다**. 그릴
  때마다 취소가 반복되면 치명적이다. `toolbar`에선 **버튼 정의만**, 실제 처리는
  `user_command`에서.
- **`function` 코드 누락·모호** → 눌러도 분기할 키가 없다. SAP 표준 코드와 안 겹치게
  `ZCANCEL`처럼 **커스텀임을 드러내는** 고유 코드를 쓴다.
- **버튼이 안 보임** → 보통 세 곳: `SET HANDLER` 누락 · `mt_toolbar`에 `APPEND` 안 함 ·
  동적 갱신 누락. 순서대로 점검한다.

## 정리

- `toolbar` 이벤트 = ALV 툴바를 구성할 때 **커스텀 버튼을 추가**하는 지점.
- `e_object->mt_toolbar`에 버튼 정의를 `APPEND`하고, 그 `function`을 다음 단계의 명령 코드로
  쓴다.
- **버튼 만들기(`toolbar`)와 처리(`user_command`)는 분리** — 업무 처리는 절대 `toolbar`에서
  하지 않는다.
