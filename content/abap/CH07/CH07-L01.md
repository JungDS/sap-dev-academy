## 끝나면 사라진다

CH06에서 구구단 81행을 [[Internal Table]] `lt_gugu`에 담았지만, 프로그램이 끝나면
**메모리째 사라졌다.** 다시 실행하면 또 처음부터다. 값을 **디스크에 영구 저장**하고
다른 사용자와도 공유하고 싶다 — 그게 [[Transparent Table]](투명 테이블)다.

## 투명 테이블이란

[[Transparent Table]]은 DDIC에 정의하는 테이블로, **DB의 실제 물리 테이블과 1:1로
대응**한다. "투명(transparent)"은 DDIC 정의와 DB 테이블이 같은 모양으로 들여다보인다는 뜻.

## SE11에서 ZGUGUDAN 만들기

구구단을 영구 저장할 테이블 `ZGUGUDAN`을 만든다. `SE11` → Database Table →
`ZGUGUDAN` → Create.

```text
MANDT  (Key)  → 클라이언트(자동)
DAN    (Key)  → 단
MUL    (Key)  → 곱하는 수
RESULT        → 결과
```

| 항목 | 내용 |
|---|---|
| Delivery Class | 보통 `A`(application data) |
| Key | 행을 유일하게 식별(여기선 MANDT+DAN+MUL) |
| Technical Settings | 데이터 클래스·크기 지정 후 활성화 |

활성화하는 순간 실제 DB 테이블이 생성된다.

## 필드 타입 — Data Element vs Built-In

각 컬럼의 타입을 주는 방법은 두 가지다.

- **Data Element 참조(권장)** — `DAN`에 `ZDE_DAN` 같은 [[Data Element]]를 주면, CH03에서
  봤듯 **라벨·F4·도움말까지** 따라온다. 화면·ALV가 똑똑해진다.
- **Built-In 타입 직접** — `INT4`(정수) 같은 내장 타입을 바로 지정. 간단하지만 라벨·도움말이 없다.

> 업무 테이블은 거의 항상 **Data Element**로 준다. 그래야 전사적으로 의미·라벨이 일관된다.

## MANDT — 클라이언트 분리

대부분 업무 테이블의 첫 키는 [[MANDT]](클라이언트)다. CH01에서 본 "방(Client)"별로
데이터가 분리되는 장치다. 보통 직접 다루지 않아도 시스템이 챙긴다.

## 흔한 실수와 점검

- **키 미지정** → 최소 하나의 키 필드 필요. 보통 MANDT + 업무 키.
- **활성화 안 함** → 활성화해야 DB 테이블이 실제로 생긴다.
- **필드를 Built-In으로만** → 라벨·도움말을 잃는다. 가능하면 Data Element.
