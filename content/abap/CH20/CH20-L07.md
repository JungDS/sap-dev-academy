## 🎫 한 화면에서 만나는 고급 SQL

지금까지는 문법을 따로 봤다. 실무에선 이 문법들이 **한 조회 화면**에 함께 나온다. 콘서트 운영자는 이런 질문을 한 화면에서 보고 싶어 한다.

- 공연별 정원·예약 좌석·잔여석은?
- 예약이 있는 공연은?
- 예약이 없는 공연은?
- 예약 상세마다 공연 안 순번과 합계는?

> 🚧 **이 실습이 하지 않는 것**: 예약 저장·취소(DML)·CDS/RAP. 모두 다음 챕터들의 주제다. 여기선 **읽기·검증**까지.

## 실습 데이터

`zconcert`

| concert_id | title | capacity |
|---|---|---:|
| C001 | 정훈영 밴드 | 100 |
| C002 | ABAP Trio | 80 |
| C003 | UI5 Night | 50 |
| C004 | Closed Show | 60 |

`zbooking`

| booking_id | concert_id | seats | status |
|---|---|---:|---|
| B001 | C001 | 2 | N |
| B002 | C001 | 3 | N |
| B003 | C002 | 1 | **C** |
| B004 | C004 | 4 | N |

## 과제 1: CTE로 잔여석 조회

```abap
WITH
  +booked AS (
    SELECT concert_id, SUM( seats ) AS booked_seats
      FROM zbooking
      WHERE status <> 'C'
      GROUP BY concert_id )
  SELECT c~concert_id, c~title, c~capacity,
         COALESCE( b~booked_seats, 0 )              AS booked,
         c~capacity - COALESCE( b~booked_seats, 0 ) AS remaining
    FROM zconcert AS c
    LEFT OUTER JOIN +booked AS b ON b~concert_id = c~concert_id
    INTO TABLE @DATA(lt_capacity).
```

예상 결과:

| concert_id | booked | remaining |
|---|---:|---:|
| C001 | 5 | 95 |
| C002 | 0 | 80 |
| C003 | 0 | 50 |
| C004 | 4 | 56 |

검증 포인트: C002의 취소 예약 1석은 합계에서 빠진다. C003은 예약이 없어도 `LEFT OUTER JOIN` 덕분에 남는다. `COALESCE`가 없으면 예약 없는 공연의 합계가 null이 되어 잔여석이 어긋난다.

## 과제 2: EXISTS로 예약 있는 공연 찾기

```abap
SELECT c~concert_id, c~title
  FROM zconcert AS c
  WHERE EXISTS ( SELECT concert_id
                   FROM zbooking AS b
                   WHERE b~concert_id = c~concert_id
                     AND b~status    <> 'C' )
  INTO TABLE @DATA(lt_has_booking).
```

예상 결과는 **C001, C004**다. C002는 취소 예약만 있어 제외, C003은 예약이 없어 제외.

## 과제 3: EXCEPT로 예약 없는 공연 찾기

```abap
SELECT concert_id FROM zconcert
EXCEPT DISTINCT
SELECT concert_id FROM zbooking WHERE status <> 'C'
INTO TABLE @DATA(lt_without_booking).
```

예상 결과는 **C002, C003**다. C002는 취소 예약만 있으므로 "취소 아닌 예약이 있는 공연" 집합에 들지 않는다.

## 과제 4: Window로 상세에 합계·순번 붙이기

```abap
SELECT concert_id, booking_id, seats,
       SUM( seats ) OVER( PARTITION BY concert_id )                      AS booked_total,
       ROW_NUMBER( ) OVER( PARTITION BY concert_id ORDER BY booking_id ) AS row_in_concert
  FROM zbooking
  WHERE status <> 'C'
  INTO TABLE @DATA(lt_window).
```

예상 결과:

| booking_id | concert_id | seats | booked_total | row_in_concert |
|---|---|---:|---:|---:|
| B001 | C001 | 2 | 5 | 1 |
| B002 | C001 | 3 | 5 | 2 |
| B004 | C004 | 4 | 4 | 1 |

## 직접 해 볼 과제

1. 과제 1의 CTE에서 `WHERE status <> 'C'`를 빼 보고, 취소 예약이 합계에 섞이는지 확인한다.
2. 과제 2의 `EXISTS`를 `INNER JOIN`으로 바꿔 보고, 공연 행이 예약 건수만큼 중복되는지 비교한다.
3. 과제 3의 `EXCEPT` 좌우를 바꿔 보고 결과가 달라지는지 확인한다.

## 흔한 실수와 점검 🚧

- **취소 조건 누락** → 합계·존재·차집합 모두 `status <> 'C'`가 빠지면 결과가 달라진다.
- **`COALESCE` 오해** → DB null을 대체하는 것이다. 이미 `0`인 값을 바꾸는 게 아니다.
- **`EXCEPT` 방향** → 왼쪽에서 오른쪽을 뺀다. 방향을 뒤집으면 다른 답이 나온다.
- **window를 필터로** → window는 `SELECT` list 계산 컬럼이다. `WHERE`에 직접 쓰지 않는다.
