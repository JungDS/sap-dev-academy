## 🎫 배운 OO를 한 클래스에 모으기

[잔여석 계산·예매 판정 모듈화](CH10-L07.html)에서 절차적으로 만들었던 잔여석·예매 판정을, 이제 하나의 업무 객체 `ZCL_BOOKING_MANAGER`로 **캡슐화**한다. 상태는 숨기고(PRIVATE), 외부는 메서드로만 다루며, **실패는 예외**로, **사건은 이벤트**로 표현한다. (목표는 멋진 OO를 많이 쓰는 게 아니라, 이 챕터에서 배운 것을 *제자리에* 놓는 것.)

> 🚧 실제 예약 행을 `INSERT`하거나 취소를 `UPDATE`하는 건 **이후 DML 챕터**의 영역이다. 여기선 **읽기·검증·예외/이벤트 설계**까지가 경계다.

## 클래스 — 외부 계약과 내부 상태 🧱

```abap
CLASS zcl_booking_manager DEFINITION PUBLIC CREATE PUBLIC.
  PUBLIC SECTION.
    EVENTS sold_out
      EXPORTING VALUE(iv_concert) TYPE zconcert-concert_id
                VALUE(iv_perf)    TYPE zbooking-perf_no.

    METHODS constructor IMPORTING iv_concert TYPE zconcert-concert_id
                                  iv_perf    TYPE zbooking-perf_no.
    METHODS remaining  RETURNING VALUE(rv_remaining) TYPE i.
    METHODS can_book   IMPORTING iv_seats TYPE i
                       RETURNING VALUE(rv_ok) TYPE abap_bool.
    METHODS book       IMPORTING iv_seats TYPE i
                       RAISING   zcx_fully_booked.
  PRIVATE SECTION.
    DATA mv_concert TYPE zconcert-concert_id.
    DATA mv_perf    TYPE zbooking-perf_no.
    DATA mv_cap     TYPE i.
ENDCLASS.
```

## 구현 — 초기화·조회·검증·예외/이벤트 ⚙️

```abap
CLASS zcl_booking_manager IMPLEMENTATION.
  METHOD constructor.
    mv_concert = iv_concert.
    mv_perf    = iv_perf.
    SELECT SINGLE capacity FROM zconcert
      WHERE concert_id = @iv_concert INTO @mv_cap.
  ENDMETHOD.

  METHOD remaining.
    SELECT COALESCE( SUM( seats ), 0 ) FROM zbooking
      WHERE concert_id = @mv_concert AND perf_no = @mv_perf
        AND status <> 'C'                       " 취소 제외
      INTO @DATA(lv_booked).
    rv_remaining = mv_cap - lv_booked.
  ENDMETHOD.

  METHOD can_book.
    rv_ok = xsdbool( iv_seats > 0 AND iv_seats <= me->remaining( ) ).
  ENDMETHOD.

  METHOD book.
    DATA(lv_remaining) = me->remaining( ).
    IF iv_seats <= 0 OR iv_seats > lv_remaining.
      RAISE EXCEPTION TYPE zcx_fully_booked.     " 실패 = 예외
    ENDIF.
    IF iv_seats = lv_remaining.
      RAISE EVENT sold_out                       " 사건 = 이벤트
        EXPORTING iv_concert = mv_concert iv_perf = mv_perf.
    ENDIF.
    " 실제 INSERT는 이후 DML 챕터에서.
  ENDMETHOD.
ENDCLASS.
```

호출부는 **public 메서드만** 쓴다(정원·회차를 직접 못 바꾼다).

```abap
DATA(lo_mgr) = NEW zcl_booking_manager( iv_concert = 'C001' iv_perf = '01' ).
TRY.
    lo_mgr->book( iv_seats = 2 ).
    MESSAGE '예약 가능' TYPE 'S'.
  CATCH zcx_fully_booked INTO DATA(lx_booked).
    MESSAGE lx_booked->get_text( ) TYPE 'I'.
ENDTRY.
```

## 직접 해 볼 과제

1. `ZCX_FULLY_BOOKED` 예외 클래스(`CX_STATIC_CHECK` 상속)를 SE24에서 만든다.
2. 객체를 만들고 `remaining( )`이 취소 제외 좌석 합계를 정원에서 빼는지 확인한다.
3. 잔여석보다 큰 좌석 수로 `book`을 불러 `ZCX_FULLY_BOOKED`가 `CATCH`로 잡히는지 확인한다.
4. (도전) `lo_monitor`의 `on_sold_out`을 `SET HANDLER`로 등록하고, 잔여석과 *같은* 좌석을 예약해 `sold_out` 이벤트가 호출되는지 본다.

## 흔한 실수와 점검 🚧

- **`book`이 DB INSERT까지 한다고 오해** → 이 챕터는 OO *설계* 장. 저장(DML)은 이후 챕터. 여기선 검증·예외·이벤트까지.
- **`remaining( )`에서 취소 미제외** → `status = 'C'`까지 합산하면 잔여석이 실제보다 적게 나온다.
- **`can_book`과 `book` 기준 불일치** → 둘이 다른 잔여석 판단을 가지면 호출자가 혼란. `book`이 내부에서 `remaining( )`을 다시 확인한다.
- **너무 넓은 예외 이름** → `ZCX_ERROR`는 이유 구분이 안 된다. `ZCX_FULLY_BOOKED`처럼 *업무 상황*을 말하는 이름.
- **상태 캡슐화** → 정원·회차는 PRIVATE, 외부는 메서드로만([Attribute / Method / Visibility](CH20-L02.html)). (없는 공연 ID는 [선택화면 검증](CH15-L07.html)에서 걸러진다는 전제.)
