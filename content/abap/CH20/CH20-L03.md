## 왜 — 행을 붙일 필요가 없을 때 🎯

[JOIN](CH13-L01.html)은 두 표의 행을 옆으로 붙인다. 하지만 어떤 요구는 행을 붙일 필요가 없다.

- 예약이 **있는** 공연인가?
- 취소되지 않은 예약이 **하나라도** 있는가?

여기서 필요한 건 예약 행의 상세 컬럼이 아니라 **존재 여부**다. 이때 [[Subquery]](조건 안에 넣는 SELECT)가 자연스럽다.

## 무엇 — EXISTS · correlated 🔍

`EXISTS`는 subquery 결과가 **하나라도 있으면 참**이 된다.

```abap
SELECT c~concert_id,
       c~title
  FROM zconcert AS c
  WHERE EXISTS ( SELECT concert_id
                   FROM zbooking AS b
                   WHERE b~concert_id = c~concert_id
                     AND b~status    <> 'C' )
  INTO TABLE @DATA(lt_has_booking).
```

안쪽 `SELECT`가 바깥 `SELECT`의 `c~concert_id`를 참조한다. 이렇게 **바깥 행을 기준으로 안쪽 질문을 다시 던지는** subquery를 correlated subquery(상관 서브쿼리)라고 한다.

읽는 법은 이렇다.

1. 바깥 공연 `c`를 한 행 고른다고 상상한다.
2. 그 공연 ID를 가진 예약 `b`가 있는지 안쪽 `SELECT`가 확인한다.
3. 취소 아닌 예약이 하나라도 있으면 바깥 공연 행이 결과에 남는다.

`EXISTS`는 존재만 판정하므로 안쪽 `SELECT`의 컬럼이 무엇이든 중요하지 않다(공식 문서도 컬럼 하나만 쓰길 권한다).

`IN` subquery는 "목록에 포함되는가"를 묻는다.

```abap
SELECT c~concert_id, c~title
  FROM zconcert AS c
  WHERE c~concert_id IN ( SELECT concert_id
                            FROM zbooking
                            WHERE seats >= 3 )
  INTO TABLE @DATA(lt_big_booking).
```

3석 이상 예약이 있는 공연 ID 목록에 드는 공연만 남는다.

## 어떻게 확인 — 존재 판정 표 ✅

`EXISTS`는 결과 컬럼보다 **조건**이 중요하다. 표로 검증한다.

| 공연 | 예약(N) | 취소(C) | `status <> 'C'` 존재 | 결과 포함 |
|---|---:|---:|---|---|
| C001 | 2건 | 1건 | 예 | 포함 |
| C002 | 0건 | 1건 | 아니오 | 제외 |
| C003 | 0건 | 0건 | 아니오 | 제외 |

같은 결과를 JOIN으로 만들면 예약 건수만큼 **공연 행이 중복**될 수 있다. `EXISTS`는 존재 여부만 보므로 바깥 공연 행은 **한 번만** 남는다.

## 흔한 실수와 점검 🚧

- **바깥·안쪽 별칭 혼동** → `c~concert_id = b~concert_id`처럼 양쪽 출처를 `~`로 분명히.
- **`NOT EXISTS`에서 조건 빠짐** → "없다"를 묻는 강력한 도구지만, 조건 하나가 빠지면 엉뚱한 판정이 난다. 조건을 빠짐없이.
- **EXISTS를 값 가져오기로 오해** → `EXISTS`는 값을 반환하지 않는다. 참·거짓만 판정한다. 값이 필요하면 JOIN이나 subquery 값 비교를 쓴다.
- **성능 단정** → subquery 성능은 DB·상황마다 다르다. 실제 시스템에서 측정한다.
