## 왜 — "할 수 있다"가 "해야 한다"는 아니다 🧭

CTE·subquery·집합·window를 배우면 모든 문제를 SQL 한 문장으로 풀고 싶어진다. 그러나 좋은 개발자는 "SQL로 **할 수 있는가**"보다 "SQL로 표현하는 것이 더 **읽기 쉽고 검증 가능한가**"를 먼저 묻는다.

## 무엇 — 상황별 선택 🗂️

| 상황 | 우선 고려 | 이유 |
|---|---|---|
| 두 표의 컬럼을 옆으로 붙인다 | `JOIN` | 컬럼 결합이 목적 |
| 관련 행 **존재 여부**만 필요하다 | `EXISTS` | 중복 없이 조건으로 판단 |
| 한 문장에서 중간 결과 재사용·단계 이름이 필요 | CTE | 읽기 쉬운 단계로 나눔 |
| 두 목록을 합·교·차 | Set operation | 집합 질문을 직접 표현 |
| 상세 행을 **유지**하며 그룹 계산 | Window | `GROUP BY`로는 상세 행이 사라짐 |
| 계산 규칙이 업무 정책이고 여러 곳에서 재사용 | ABAP 코드 / 후속 [CDS](CH23-L01.html) | SQL 한 문장에 정책을 숨기지 않음 |

## 어떻게 확인 — 리뷰 체크리스트 ✅

| 질문 | 확인 방법 |
|---|---|
| 결과 행 수가 예상과 맞나? | 입력표로 수작업 예상표를 먼저 만든다 |
| 중복이 업무적으로 의도됐나? | `JOIN`·`UNION ALL`의 중복을 확인 |
| null 처리가 명시됐나? | outer join·계산 컬럼에서 `COALESCE` 필요성 |
| 조건 위치가 맞나? | outer join 조건은 `ON`과 `WHERE` 차이([지난 챕터](CH19-L08.html)) |
| SQL이 업무 정책을 숨기지 않나? | 읽는 사람이 한 화면에서 설명할 수 있는지 |
| 후속 장 주제를 앞당기지 않나? | CDS·RAP·동적 SQL이면 해당 장으로 |

## 흔한 실수와 점검 🚧

- **CTE 남발** → 많아질수록 절차적 코드처럼 보이지만, 디버거로 한 줄씩 중간 표를 보긴 어렵다. 중간 결과는 **머릿속 표**로 검증한다.
- **성능 단정** → subquery·window의 성능은 DB·상황마다 다르다. 측정 없이 단정하지 않는다.
- **단순 합계에 window** → 상세 행이 필요 없으면 `GROUP BY`가 더 낫다.
- **집합 연산 모양 안 맞춤** → 좌우 `SELECT` 컬럼·타입을 먼저 맞춘다.
