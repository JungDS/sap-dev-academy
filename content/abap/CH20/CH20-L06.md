## 왜 — 오류를 객체로 🎯

절차적 코드는 오류를 숫자 코드나 화면 메시지로 처리하곤 했다.

```abap
IF iv_seats > lv_remaining.
  MESSAGE '매진입니다' TYPE 'E'.    " 클래스가 직접 화면 메시지?
ENDIF.
```

이러면 같은 예약 로직을 **배치·테스트에서 재사용하기 어렵다**(화면이 없으니까). 또 호출자는 *무엇이* 실패했는지 구조적으로 구분하기 힘들다. **예외**는 오류를 **객체**로 전달한다. 클래스는 "정원 초과라 예약 불가"라는 사실을 [[예외]] 객체로 던지고, 호출자가 화면·배치·테스트 상황에 맞게 잡는다. ([SALV FACTORY](CH11-L02.html)에서 `cx_salv_msg`를 `TRY/CATCH`로 잡았던 그 메커니즘 — 이제 *내 예외*도 만든다.)

## 내 예외 만들기 · 던지기 🚀

`ZCX_FULLY_BOOKED`(예외 클래스, SE24에서 `CX_STATIC_CHECK` 상속)를 만든다. 예외를 던질 수 있는 메서드는 **시그니처에 `RAISING`으로 선언**한다.

```abap
METHODS book
  IMPORTING iv_seats TYPE i
  RAISING   zcx_fully_booked.        " 이 메서드는 이 예외를 던질 수 있다
```

```abap
METHOD book.
  IF iv_seats > me->remaining( ).
    RAISE EXCEPTION TYPE zcx_fully_booked.
  ENDIF.
  " 실제 INSERT(예약 저장)는 뒤 챕터(DML)에서 다룬다.
ENDMETHOD.
```

## 잡기 — TRY / CATCH / CLEANUP 🪤

```abap
TRY.
    lo_mgr->book( iv_seats = 100 ).
  CATCH zcx_fully_booked INTO DATA(lx_booked).
    MESSAGE lx_booked->get_text( ) TYPE 'I'.
  CLEANUP.
    " TRY 블록이 예외로 정리될 때 필요한 뒷정리(자원 해제 등)
ENDTRY.
```

## CX 계층 — 세 갈래 🌳

모든 class-based 예외는 최상위 `CX_ROOT`에서 내려오고, 처리 강제 수준에 따라 셋으로 나뉜다.

| 계층 | 처리 강제 | 입문자 판단 기준 |
|---|---|---|
| `CX_STATIC_CHECK` | 시그니처 선언·처리를 강하게 요구(checked) | 업무상 예상 가능한 복구 오류 |
| `CX_DYNAMIC_CHECK` | 선언은 필요, 일부 검사는 런타임 | 동적 상황에서 날 수 있는 오류 |
| `CX_NO_CHECK` | 처리 강제 안 함 | 어디서나 발생 가능·강제 부적절 |

`CX_ROOT`의 `get_text( )`로 메시지를 얻는다. `RESUME`은 *resumable 예외*를 `CATCH BEFORE UNWIND`로 잡은 뒤 발생 지점부터 재개하는 특수 문장 — **존재와 제약만 알고**, 일반 업무 예외 패턴으로는 쓰지 않는다.

## 흔한 실수와 점검 🚧

- **예외 무시** → `TRY/CATCH` 없이 던지면 덤프. 적절히 잡는다.
- **`CATCH cx_root` 먼저** → 상위 예외가 모든 걸 가로채 구체 handler가 **도달 불가**가 된다. **구체 예외부터, 일반 예외는 뒤로.** (위 체험에서 직접 확인.)
- **`cx_root`로 전부 삼킴** → 무엇이 실패했는지 사라진다. 정말 필요한 경계에서만 좁게.
- **`CLEANUP`을 finally로 오해** → 항상 실행되는 후처리가 아니라, TRY 블록이 *예외로* 정리될 때 실행되는 블록이다.
- **던지기와 메시지를 한 곳에** → 클래스는 예외로 *알리고*, 화면 프로그램이 사용자 메시지로 *바꾼다*.
