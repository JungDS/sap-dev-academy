## 절차적 코드의 한계

지금까지 데이터(테이블)와 그걸 처리하는 코드(서브루틴·함수)가 **따로** 놀았다.
"공연 예약"이라는 개념을 — 데이터(잔여석)와 행동(예약·취소)을 **한 덩어리**로 묶고
싶다. 그게 객체지향([[OO]])이고, 그 틀이 [[Class]](클래스)다.

> 콘서트 예매 앱의 핵심 로직을 `ZCL_BOOKING_MANAGER` 클래스로 만들어 간다.

## 전역 클래스 — SE24

`SE24`(또는 ADT)에서 `ZCL_BOOKING_MANAGER`를 만든다. CH10에서 봤듯 클래스는
**DEFINITION**(무엇이 있는지)과 **IMPLEMENTATION**(실제 코드)으로 나뉜다.

```abap
CLASS zcl_booking_manager DEFINITION PUBLIC.
  PUBLIC SECTION.
    METHODS book IMPORTING iv_seats TYPE i.
ENDCLASS.

CLASS zcl_booking_manager IMPLEMENTATION.
  METHOD book.
    " 예약 처리
  ENDMETHOD.
ENDCLASS.
```

## 객체 만들기 — NEW (드디어)

CH18에서 미뤄 둔 `NEW`를 이제 객체 생성에 쓴다. 클래스(설계도)로 **객체(인스턴스)**
를 찍어낸다.

```abap
DATA(lo_mgr) = NEW zcl_booking_manager( ).   " modern
lo_mgr->book( iv_seats = 2 ).                " -> 로 인스턴스 메서드 호출
```

```abap
" classic(여전히 유효)
DATA lo_mgr TYPE REF TO zcl_booking_manager.
CREATE OBJECT lo_mgr.
```

## 흔한 실수와 점검

- **객체 없이 인스턴스 메서드** → `NEW`(또는 CREATE OBJECT)로 객체부터.
- **REF TO 타입** → 객체를 담는 변수는 `TYPE REF TO 클래스`.

---

클래스와 객체를 만들었다. 이제 그 안에 담을 **속성·메서드와 공개 범위**를 보자 → 다음 레슨.
