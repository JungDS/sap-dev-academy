## 왜 — modern SQL 기본으론 벅찬 질문 🧗

[지난 챕터](CH19-L01.html)에서 `@`·콤마·`@DATA( )`·SQL 식·SQL 함수·`SELECT FROM @itab`을 배웠다. 그런데 CH19는 일부러 **CTE·복잡한 subquery·집합 연산·window**를 미뤘다. 이 문법들은 *한 줄을 짧게 쓰는 기술*이 아니라, **데이터베이스에 질문을 설계하는 방식**이기 때문이다.

콘서트 예매 데이터를 예로 들면, 곧 이런 요구가 나온다.

- 공연별 예약 합계를 **먼저 구한 뒤** 공연 마스터와 합치고 싶다.
- 예약이 **하나라도 있는** 공연만 찾고 싶다.
- 전체 공연에서 이미 **예약된 공연을 빼고** 싶다.
- 예약 상세 행은 유지하면서 같은 공연의 **합계와 순번**을 함께 보고 싶다.

이걸 전부 ABAP `LOOP`로 처리할 수도 있다. 하지만 데이터가 DB에 있고 조건·집계가 SQL로 자연스럽게 표현된다면, **DB가 잘하는 일을 DB에 맡기는** 편이 더 명확할 때가 많다.

## 무엇 — 네 묶음 도구 🧰

이 챕터가 다루는 고급 SQL은 네 가지다. 모두 **읽기 전용 `SELECT`**다(저장·변경 없음).

| 도구 | 한 줄 정의 | 대표 질문 |
|---|---|---|
| [[CTE]] | `SELECT` 안의 중간 결과에 **이름**을 붙인다 | "예약 합계를 먼저 만들고, 그 결과를 공연과 합치자" |
| [[Subquery]] | 조건 안에 `SELECT`를 넣어 **존재·포함**을 묻는다 | "예약이 하나라도 있는 공연만" |
| [[Set operation]] | 두 `SELECT` 결과를 집합처럼 **합·교·차** | "전체 공연에서 예약된 공연을 빼면?" |
| [[Window]] | 상세 행을 유지하며 그룹의 **합계·순번·순위**를 붙인다 | "예약 한 건마다 공연 누적 좌석과 순번을" |

## 어떻게 확인 — 눈으로만 보면 위험하다 ✅

고급 SQL은 결과를 눈대중으로 넘기면 틀리기 쉽다. **네 가지를 함께** 본다.

1. 입력 데이터 표를 먼저 본다.
2. SQL이 만든 중간 개념을 **말로** 설명한다.
3. 결과 **행 수**를 예상한다.
4. 실제 결과에서 누락된 행·중복된 행·계산값을 확인한다.

예를 들어 "예약이 있는 공연만"이라는 요구는 행 수보다 **행의 의미**가 중요하다.

| 공연 | 예약 상태 | 결과에 있어야 하나 |
|---|---|---|
| C001 | 예약 있음 | 예 |
| C002 | 예약 없음 | 아니오 |
| C003 | 취소 예약만 있음 | 업무 조건에 따라 다름 |

취소(`status = 'C'`) 예약을 예약으로 볼지 말지에 따라 SQL 조건이 달라진다. 고급 SQL에서 실수는 문법보다 **업무 조건 누락**에서 더 자주 난다.

## 흔한 실수와 점검 🚧

- **SQL이 길어졌다고 좋은 코드는 아님** → 읽기 쉬운 단계로 나뉘지 않으면 ABAP `LOOP`보다 위험하다.
- **JOIN·subquery·CTE를 같은 것으로 착각** → 대체 가능해 보여도 **결과 행 수가 달라질 수 있다.** 반드시 행 수를 비교한다.
- **이 장은 CDS가 아님** → `define view entity`·association은 [다음 단계](CH23-L01.html)의 주제다. 여기선 `SELECT` 문장만.
- **이 장은 RAP가 아님** → 데이터 저장·변경은 [실무 데이터 변경](CH25-L01.html) 이후. 여기선 읽기만.
