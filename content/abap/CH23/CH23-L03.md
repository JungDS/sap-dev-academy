## 왜 관계를 모델에 새길까

[INNER JOIN 기본 개념과 구현](CH13-L01.html)에서 본 JOIN은 반드시 필요하지만, 프로그램마다 같은 관계를
반복해서 쓰면 유지보수가 어려워진다. 콘서트 앱을 떠올려 보자. 공연 `ZCONCERT`, 회차 `ZPERF`, 예매
`ZBOOKING`은 계속 함께 등장한다.

공연 목록에서 회차를 따라가야 하고, 회차에서 예매를 따라가야 하며, 예매에서 다시 공연 정보를 보여 줘야
한다. 이 관계를 매번 JOIN 조건으로 쓰면 `concert_id`·`perf_no`를 잇는 규칙이 여러 곳에 흩어진다. 누군가
한 곳에서 조건을 잘못 쓰면 같은 업무 관계가 화면마다 다르게 해석된다.

CDS [[Association]](뷰 사이 관계 선언)은 이 관계를 데이터 모델 안에 새겨 둔다. 핵심은 "JOIN을 없앤다"가
아니라 **"관계를 이름 붙여 재사용한다"**이다.

## 무엇인가 — 관계에 이름을 붙이기

Association의 기본 구조는 이렇다.

```cds
@AccessControl.authorizationCheck: #NOT_REQUIRED
define view entity ZI_Concert
  as select from zconcert
  association [0..*] to ZI_Perf as _Perf
    on $projection.concert_id = _Perf.concert_id
{
  key concert_id,
      artist,
      venue,
      capacity,
      _Perf
}
```

읽는 순서는 다음과 같다.

| 코드 | 뜻 |
|---|---|
| `association [0..*] to ZI_Perf as _Perf` | 현재 뷰에서 `ZI_Perf`로 가는 관계를 `_Perf`라는 이름으로 선언. 한 공연은 0개 이상의 회차를 가질 수 있다. |
| `on $projection.concert_id = _Perf.concert_id` | 현재 뷰가 노출하는 `concert_id`와 대상의 `concert_id`를 연결. |
| `_Perf` (select list 안) | association을 **노출**한다. 다른 CDS·소비처가 이 경로를 따라갈 수 있게. |

- **`[0..*]`는 [[Cardinality|target cardinality]](대상 쪽 개수 기대치)다.** 한 공연이 **대상인 회차를 몇 개**
  가리키는지 — 없을 수도(0), 여러 개일 수도(`*`) 있다는 뜻. 단순 주석이 아니라 모델을 읽는 사람·도구에게
  관계의 형태를 알려 주므로 **데이터 현실에 맞춰야** 한다. (숫자 대신 단어로 `association of one to many
  to ZI_Perf`처럼 쓰는 표기도 있다.)
- **`$projection`은 "현재 뷰가 노출하는 요소를 기준으로 ON 조건을 쓴다"는 표시**다. association을 select
  list에 노출하면, ON 조건의 source 쪽 필드(`concert_id`)는 반드시 `{ }` 안에 있어야 한다.

공연→회차→예매가 어떻게 이름 붙은 경로로 이어지는지 먼저 관계도로 보자.

## 직접 따라가 보기

아래 시뮬레이터에서 공연을 골라 `_Perf` 경로로 회차를 따라가 보자. `_Perf` 노출을 끄면 소비자가 경로를 못
보고, `concert_id`를 빼면 ON 조건의 source가 사라진다. cardinality를 `[1..1]`로 바꿨는데 회차가 여러 개면
어떤 경고가 뜨는지, 같은 관계를 JOIN으로 보면 어떤 모습인지도 비교해 보자.

::embed CH23-L03-S02 | Association 경로 시뮬레이터::

> **association은 JOIN과 무관한 마법 통로가 아니다.** path를 실제로 사용하면 내부적으로 JOIN으로 변환될 수
> 있다. 차이는 JOIN 조건을 소비처마다 반복하지 않고 **모델에 한 번 선언**한다는 점이다.

## 어떻게 확인하는가

1. **활성화** — ON 조건에 쓴 필드가 source와 target 양쪽에 있어야 한다. 대상 필드는 `_Perf.concert_id`처럼
   association 이름으로 접근한다.
2. **ADT outline** — `_Perf`가 단순 필드가 아니라 *관계 항목*으로 보이면 의도대로 선언된 것이다.
3. **소비 뷰로 전달** — 소비 계층이 관계를 이어받는지 본다.

```cds
@AccessControl.authorizationCheck: #NOT_REQUIRED
define view entity ZC_Concert
  as select from ZI_Concert
{
  key concert_id,
      artist,
      venue,
      capacity,
      _Perf
}
```

`ZC_Concert`에서도 `_Perf`가 보인다면 기반 뷰가 관계를 노출한 것이고, 이후 다른 CDS가 그 경로를 쓸 수 있다.

## 흔한 실수와 점검 🚧

- **선언만 하고 노출 안 함** → association을 만들고 `{ }`에 `_Perf`를 넣지 않으면 소비자가 따라갈 수 없다.
  이 레슨의 가장 중요한 지점이다.
- **cardinality를 대충 쓰기** → 공연→회차는 보통 `[0..*]`. 반대로 예매→공연은 한 예매가 정확히 한 공연에
  속하면 `[1..1]`에 가깝다. 외래키·업무 규칙을 보고 정한다.
- **성능 무료 쿠폰으로 착각** → path를 쓰면 JOIN이 발생한다. CDS는 관계를 *선언*하는 도구지 DB 비용을
  없애는 도구가 아니다. 성능은 SQL Trace·결과 행 수로 확인한다.
- **`$projection`/source 필드 누락** → association을 노출하는데 ON source 필드가 select list에 없으면
  활성화 오류나 사용 제한으로 이어진다.
