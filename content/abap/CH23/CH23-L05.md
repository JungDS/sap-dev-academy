## 왜 주석을 따로 뺄까

[Annotation과 의미 부여](CH23-L04.html)처럼 annotation을 뷰 본문에 직접 쓰면 처음에는 이해하기 쉽다.
그런데 UI annotation이 늘어나면 문제가 생긴다. `ZC_Concert`의 핵심은 *어떤 데이터를 소비자에게 노출할지*인데,
본문이 `@UI.lineItem`·`@UI.identification`·여러 `position` 값으로 가득 차면 **데이터 구조를 읽기 어려워진다**.

또 프로젝트가 커지면 같은 데이터 모델을 여러 UI나 고객별 확장 계층에서 다르게 꾸미고 싶어진다. 이때 CDS
본문을 계속 수정하면 기반 모델과 표현 설정이 강하게 엮인다. **Metadata Extension**은 annotation을 별도
DDLX source로 분리해 이 문제를 줄인다.

## 무엇인가 — annotation만 별도 파일로

Metadata Extension은 기존 CDS entity에 **annotation만** 덧붙이는 별도 소스다. 두 가지가 짝을 이룬다.

먼저 **대상 CDS entity가 확장을 허용**해야 한다.

```cds
@AccessControl.authorizationCheck: #NOT_REQUIRED
@Metadata.allowExtensions: true
define view entity ZC_Concert
  as select from ZI_Concert
{
  key concert_id,
      artist,
      venue,
      capacity
}
```

그 다음 별도 [[Metadata Extension]](DDLX) source에서 annotation을 붙인다.

```metadata
@Metadata.layer: #CORE
annotate entity ZC_Concert with
{
  @UI.lineItem: [{ position: 10 }]
  concert_id;

  @UI.lineItem: [{ position: 20 }]
  artist;

  @UI.lineItem: [{ position: 30 }]
  venue;
}
```

여기서 입문자가 꼭 짚어야 할 차이가 있다. DDL Source의 select list는 **쉼표(`,`)**로 필드를 나열하지만,
Metadata Extension의 `annotate` 블록에서는 요소명 뒤에 **세미콜론(`;`)**을 쓴다. `ZC_Concert`를 *정의하는*
파일이 아니라, 이미 존재하는 `ZC_Concert`의 요소에 *주석을 붙이는* 파일이기 때문이다.

`@Metadata.layer: #CORE`는 이 extension이 어느 layer에 속하는지 나타낸다. layer는 여러 metadata extension이
있을 때 **평가 우선순위**에 영향을 준다. 이 챕터에서는 `#CORE`를 기본 예제로 쓰고, 고객별 덮어쓰기 전략은
뒤로 미룬다.

> ⚠️ **extension의 annotation이 본문보다 우선한다.** 같은 annotation을 본문 뷰와 metadata extension에 둘 다 두면,
> **extension 쪽 값이 이긴다**(그래서 본문을 안 건드리고 표현만 바꿀 수 있는 것이다). 우선순위를 모르면 "본문에서
> 바꿨는데 화면이 안 변한다"에서 막힌다 — 그 annotation이 extension에도 있는지 먼저 본다.

## 직접 분리해 보기

아래 실험실에서 UI 주석을 본문에 둘 때와 DDLX로 뺄 때를 비교해 보자. `@Metadata.allowExtensions`를 끄면
활성화가 어떻게 실패하는지, DDLX 구분자를 쉼표로 바꾸면 어떤 오류가 나는지도 확인하자.

## 어떻게 확인하는가

1. **대상 먼저** — `ZC_Concert`가 먼저 활성화되어 있어야 한다. 존재하지 않는 entity는 annotate할 수 없다.
2. **허용 여부** — 대상에 `@Metadata.allowExtensions: true`가 있는지 본다. 없으면 extension이 붙지 않는다.
3. **DDLX 활성화** — 활성화 오류가 나면 요소명 오타, 세미콜론 누락, layer 누락, 허용되지 않는 annotation을
   먼저 본다.
4. **중복 제거** — 같은 `@UI.lineItem`을 본문과 extension에 동시에 두지 않는다. UI 주석을 extension으로
   옮겼다면 본문에는 필드 구조만 남긴다.

## 흔한 실수와 점검 🚧

- **`@Metadata.allowExtensions: true` 빼기** → Metadata Extension은 "아무 CDS에나 밖에서 주석을 붙이는 기능"이
  아니다. 대상이 확장을 허용해야 한다.
- **DDL 문법과 DDLX 문법 섞기** → `define view entity` 안은 필드 뒤 **쉼표**, `annotate entity` 안은 요소 뒤
  **세미콜론**.
- **데이터 구조 변경 도구로 착각** → Metadata Extension은 annotation을 *추가*하는 도구다. 필드를 새로 만들거나
  select source를 바꾸는 곳이 아니다.
- **layer를 단순 장식으로 보기** → layer는 여러 extension의 평가 우선순위와 관련된 관리 정보다.
