## 왜 코드가 따로 필요할까

BDEF는 "무엇을 할 수 있는가"를 선언한다. 하지만 정원 초과 검증, 취소 action, 상태 자동 설정 같은 업무 로직은
**실제 코드**가 필요하다. 이 코드를 아무 report나 class에 흩뿌리면 RAP runtime이 호출할 수 없다. BDEF와 연결된
**정해진 구현 위치**가 필요하다.

그 위치가 **Behavior Pool**이다. [Global Class 생성과 객체](CH20-L01.html)에서 배운 class·method 지식이 여기서
다시 등장한다. 다만 일반 class와 달리 RAP handler method는 BDEF의 operation과 연결되는 **특수 signature**를 가진다.

## 무엇인가 — 집합 지향 handler

Behavior Pool은 ABAP behavior implementation을 위한 특수 class pool이다. global class가 모든 behavior를 직접
구현하는 게 아니라, 보통 그 안의 **local handler class**가 동작을 구현한다. validation handler의 기본 모양은 이렇다.

```abap
CLASS lhc_booking DEFINITION INHERITING FROM cl_abap_behavior_handler.
  PRIVATE SECTION.
    METHODS validate_capacity FOR VALIDATE ON SAVE
      IMPORTING keys FOR Booking~validate_capacity.
ENDCLASS.

CLASS lhc_booking IMPLEMENTATION.
  METHOD validate_capacity.
    READ ENTITIES OF ZI_Booking IN LOCAL MODE
      ENTITY Booking
      FIELDS ( booking_id concert_id perf_no seats )
      WITH CORRESPONDING #( keys )
      RESULT DATA(lt_bookings).

    LOOP AT lt_bookings INTO DATA(ls_booking).
      " 정원 초과 여부를 계산하고, 실패 시 failed/reported에 기록한다.
    ENDLOOP.
  ENDMETHOD.
ENDCLASS.
```

| 코드 | 뜻 |
|---|---|
| `INHERITING FROM cl_abap_behavior_handler` | RAP handler class의 기반 클래스를 상속한다. |
| `FOR VALIDATE ON SAVE` | 이 method가 save 시점 validation 구현임을 나타낸다. |
| `IMPORTING keys FOR Booking~validate_capacity` | validation 대상 instance key 집합을 받는다. |
| `READ ENTITIES OF ZI_Booking IN LOCAL MODE` | 같은 RAP BO 구현 안에서 현재 transactional buffer를 고려해 entity를 읽는다. |
| `WITH CORRESPONDING #( keys )` | handler가 받은 key 구조를 read request에 맞게 매핑한다. |
| `failed`, `reported` | 실패 인스턴스와 메시지를 RAP runtime에 돌려주는 응답 구조다. |

여기서 가장 중요한 감각 — **`keys`는 단건이 아니라 internal table이다.** RAP는 여러 instance를 한 번에 처리하는
집합 지향 모델이므로, method 안에서 `keys`를 한 건이라고 가정하면 안 된다.

## 직접 따라가 보기

아래 추적기에서 저장 요청 → `READ ENTITIES`(집합 read) → 정원 검사 → `failed/reported` 기록 흐름을 따라가
보자. `keys`가 3행이라는 점, `loop 안 EML`이 왜 anti-pattern인지 확인하자.

## 어떻게 확인하는가

1. **이름 일치** — behavior pool class가 BDEF header의 `managed implementation in class zbp_i_booking unique;`와 맞는지 본다.
2. **method 이름 일치** — handler method 이름이 BDEF에 선언한 validation/action 이름과 맞아야 연결된다.
3. **signature** — `FOR VALIDATE ON SAVE`·`FOR MODIFY`·`FOR ACTION` 같은 구문이 정확해야 한다.
4. **loop 안 EML 금지** — 먼저 필요한 데이터를 한 번에 읽고, 그 결과를 loop로 처리한다.

## 흔한 실수와 점검 🚧

- **behavior pool을 일반 utility class처럼** → RAP runtime이 호출하는 구현 위치다. 외부에서 마음대로 부르는
  helper와 구분한다.
- **`keys`를 단건으로 처리** → 사용자가 UI에서 여러 행을 동시에 저장할 수 있고, runtime은 여러 instance를 한
  번에 넘긴다.
- **validation 안에서 데이터 수정** → 검증은 *실패를 알리는* 역할이다. 값 자동 변경은 determination·action에서
  설계한다(validation에서 `MODIFY ENTITIES` 금지).
- **`failed/reported`를 안 채우기** → 문제를 발견해도 runtime에 돌려주지 않으면 사용자는 왜 저장이 막혔는지
  알 수 없다.
