## 왜 root부터 정할까

RAP에서 가장 먼저 안정되어야 하는 것은 업무 객체의 **뿌리(root)**다. 예매 앱이라면 "예매"가 중심인지,
"공연"이 중심인지, "회차"가 중심인지 정해야 한다. 취소 action, 정원 validation, 상태 determination이 모두
**예매 단위**로 일어난다면 root는 `Booking`이 자연스럽다.

root가 모호하면 뒤 계층이 전부 흔들린다. projection이 무엇을 노출해야 할지 불명확해지고, BDEF가 어떤 entity에
붙어야 할지 헷갈리며, action의 대상도 불안정해진다. 그래서 RAP의 첫 설계 질문은 **"이 앱의 트랜잭션 단위는
무엇인가"**다.

## 무엇인가 — Root Interface View

Root Interface View는 RAP BO를 대표하는 CDS view entity다. [CDS View Entity 기본 구조](CH22-L01.html)의 Interface
View를 RAP에 맞게 확장한 것이다.

```abap
@AccessControl.authorizationCheck: #NOT_REQUIRED
define root view entity ZI_Booking
  as select from zbooking
{
  key booking_id,
      concert_id,
      perf_no,
      customer,
      seats,
      status
}
```

읽는 순서는 이렇다.

| 코드 | 뜻 |
|---|---|
| `define root view entity ZI_Booking` | 이 CDS view entity가 RAP BO의 **root entity**임을 선언한다. |
| `as select from zbooking` | 지속 데이터의 기반은 `zbooking` 테이블이다. |
| `key booking_id` | OData entity key이자 RAP instance 식별의 중심이 된다. |
| `concert_id`, `perf_no` | 공연·회차를 찾기 위한 업무 연결 필드. |
| `customer`, `seats`, `status` | 예매자·좌석 수·예매 상태 같은 트랜잭션 필드. |

root entity는 **parent가 없다**(BO를 대표한다). child entity가 있을 수는 있지만, 이 챕터의 입문 실습에서는
예매 root 하나를 중심으로 단순하게 시작한다. 공연(`ZI_Concert`)·회차(`ZI_Perf`)와의 관계는
[Association 기초](CH22-L03.html)에서 배운 association으로 잇는다.

## 어떻게 확인하는가

1. **활성화** — `define root view entity`는 `define view entity`와 비슷하지만 `root`가 RAP BO 설계 의미를 더한다.
2. **key 안정성** — `booking_id`가 비거나 중복될 수 있으면 root key로 부적합하다. key는 "사용자가 한 건을
   열었을 때 어떤 행인가"를 정확히 가리켜야 한다.
3. **Data Preview** — `booking_id`·`concert_id`·`perf_no`·`customer`·`seats`·`status`가 보이면 최소 모델이 준비된 것이다.

## 흔한 실수와 점검 🚧

- **모든 CDS view를 root로** → root는 업무 객체를 대표하는 entity다. 보조 조회용·value help view까지 root로
  만들면 구조가 흐려진다.
- **root key를 화면 번호처럼 가볍게** → RAP에서 key는 update·action·validation·OData URL이 인스턴스를 찾는
  기준이다. 나중에 바뀔 수 있는 상태 필드를 key로 삼으면 안 된다.
- **root view에 UI 표현 과다** → `ZI_Booking`은 기반 모델이다. 목록 컬럼 순서 같은 표현은 `ZC_Booking`이나
  Metadata Extension에서 다룬다.
- **root entity = DB table 동일시** → `zbooking`은 저장소이고, `ZI_Booking`은 RAP가 이해하는 업무 모델의 시작점이다.
