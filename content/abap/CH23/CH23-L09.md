## 🎫 12단계 — RAP로 트랜잭션

CH22의 CDS 위에, 예매 RAP 앱의 핵심 동작을 직접 선언·구현한다 — 정원 초과 검증과
취소 action.

```abap
" BDEF
define behavior for ZI_Booking alias Booking
persistent table zbooking
lock master
{
  create; update; delete;
  validation validateCapacity on save { field seats, concert_id, perf_no; }
  action cancel result [1] $self;
}
```

```abap
" Behavior Pool — 정원 검증
METHOD validateCapacity.
  READ ENTITIES OF zi_booking IN LOCAL MODE
    ENTITY Booking FIELDS ( concert_id perf_no seats )
    WITH CORRESPONDING #( keys ) RESULT DATA(bks).
  LOOP AT bks INTO DATA(b).
    " 회차 정원·기존 합계 조회 → 초과면 APPEND TO failed/reported
  ENDLOOP.
ENDMETHOD.
```

## 직접 해 볼 과제

1. `validateCapacity`에서 정원 초과 예매를 `failed`/`reported`로 막는다.
2. `cancel` action으로 상태를 'C'로 바꾼다.
3. (도전) `setStatus` determination으로 생성 시 기본 상태를 'N'으로.

## 흔한 실수와 점검

- **IN LOCAL MODE 누락** → 핸들러 내부 읽기/수정은 보통 `IN LOCAL MODE`(CH23-L07).
- **failed/reported 누락** → 검증 실패를 안 알리면 저장이 안 막힌다.
