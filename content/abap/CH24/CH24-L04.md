## 왜 동작을 따로 선언할까

CDS view는 데이터를 *보여줄* 수 있지만, 사용자가 *무엇을 할 수 있는지*는 말해 주지 않는다. 예매를 만들 수
있는가, 수정할 수 있는가, 삭제할 수 있는가, 취소 버튼이 있는가, 저장 전에 검증해야 하는가 같은 동작은 별도의
**계약**이 필요하다.

이 계약이 없으면 소비자는 "이 entity가 읽기 전용인지, 수정 가능한지, 어떤 action을 제공하는지"를 알 수 없다.
RAP에서는 [[Behavior Definition]](줄여서 BDEF)이 이 역할을 맡는다.

## 무엇인가 — RAP BO의 동작 계약

BDEF는 RAP BO의 transactional behavior를 정의하는 repository object다. root entity를 기준으로 어떤 operation을
허용하는지 선언한다.

```abap
managed implementation in class zbp_i_booking unique;

define behavior for ZI_Booking alias Booking
persistent table zbooking
lock master
{
  create;
  update;
  delete;

  field ( readonly:update ) booking_id;

  mapping for zbooking
  {
    booking_id = booking_id;
    concert_id = concert_id;
    customer   = customer;
    seats      = seats;
    status     = status;
  }
}
```

각 줄의 의미는 이렇다.

| 코드 | 뜻 |
|---|---|
| `managed implementation in class zbp_i_booking unique;` | managed 시나리오이며, custom 구현이 필요하면 `zbp_i_booking` behavior pool을 쓴다. |
| `define behavior for ZI_Booking alias Booking` | `ZI_Booking` root의 behavior를 정의하고 내부에서 `Booking` alias로 부른다. |
| `persistent table zbooking` | managed RAP BO의 지속 데이터 테이블을 지정한다. |
| `lock master` | root entity가 lock master임을 선언한다(managed에서는 framework가 잠금을 처리). |
| `create; update; delete;` | 소비자가 쓸 수 있는 표준 operation을 허용한다. |
| `field ( readonly:update ) booking_id;` | update 시 key 필드를 바꾸지 못하게 한다. |
| `mapping for zbooking { ... }` | CDS 요소와 persistent table 필드의 대응을 명시한다. |

여기서 **`create/update/delete`는 직접 DB SQL이 아니다.** RAP BO에 대해 가능한 표준 operation을 선언하는 BDL
문법이다. 직접 `INSERT`·`UPDATE`·`DELETE` SQL과 transaction control은
[INSERT / UPDATE / MODIFY / DELETE 실무 기준](CH25-L01.html)에서 따로 배운다.

## 어떻게 확인하는가

1. **root 연결** — BDEF는 적어도 하나의 root entity로 구성된 CDS data model에 기반하며, root entity와 연결된다.
2. **persistent table 일치** — `ZI_Booking`이 `zbooking` 기반인데 BDEF가 다른 테이블을 가리키면 저장 모델이 어긋난다.
3. **lock master** — 이 챕터는 root 하나만 다루므로 `lock master`가 자연스럽다.
4. **활성화 오류** — mapping 필드명 오타, alias 불일치, persistent table 누락, root entity 이름 오류는 활성화에서 잡힌다.

## 흔한 실수와 점검 🚧

- **BDEF를 ABAP class로 착각** → BDEF는 behavior를 선언하는 BDL source이고, 실제 ABAP 구현은 behavior pool에 둔다.
- **`mapping` 생략해도 항상 괜찮다고 생각** → 필드명이 완전히 같고 규칙이 단순하면 덜 복잡할 수 있지만, 학습
  단계에서는 entity 요소와 DB 필드의 대응을 눈으로 확인하는 편이 안전하다.
- **key를 수정 가능하게 두기** → `booking_id`가 update 중 바뀌면 어떤 예매를 수정하는지 기준이 흔들린다. key는
  보통 read-only로 둔다.
- **`lock master`를 classic lock object와 동일시** → 여기서는 RAP BDEF의 locking 선언을 배우고, classic
  enqueue/dequeue와 동시성 심화는 [Lock Object 설계 기준](CH26-L01.html)에서 다룬다.
