## 왜 root를 그대로 노출하지 않을까

[Interface View ZI_* 설계 (Root)](CH24-L02.html)에서 만든 root view를 그대로 외부에 노출하면 안 되는 경우가
많다. 기반 모델은 내부 안정성을 위해 존재하고, 외부 소비자는 화면·API 목적에 맞춘 모델을 원한다.

내부 감사 필드를 숨기고 싶을 수도 있고, UI annotation을 분리하고 싶을 수도 있으며, RAP transactional service에서
쓸 **projection 계약**을 명확히 하고 싶을 수도 있다. [Interface View와 Consumption View 계층](CH23-L02.html)에서
배운 `ZI_`/`ZC_` 분리를, 이제 RAP 트랜잭션 모델에 맞게 적용한다.

## 무엇인가 — transactional projection

RAP의 Projection View는 root entity를 소비자에게 노출하기 위한 CDS projection view다.

```abap
@AccessControl.authorizationCheck: #NOT_REQUIRED
@Metadata.allowExtensions: true
define root view entity ZC_Booking
  provider contract transactional_query
  as projection on ZI_Booking
{
  key booking_id,
      concert_id,
      perf_no,
      customer,
      seats,
      status
}
```

핵심은 세 가지다.

| 코드 | 뜻 |
|---|---|
| `define root view entity ZC_Booking` | 기반 `ZI_Booking`이 root이므로 projection도 **root 위치**를 반영한다. |
| `provider contract transactional_query` | 이 projection이 RAP transactional query 시나리오에 쓰인다는 **계약**을 드러낸다. |
| `as projection on ZI_Booking` | 기반 root view를 외부 소비 목적에 맞게 투영한다. |

transactional projection은 RAP BO의 projection layer를 모델링한다. transactional query는 business object의
일부여야 하며, root entity이거나 root를 포함한 composition tree 안에 있어야 한다. 이 챕터에서는 root 하나를
쓰는 단순 구조로 이 조건을 이해한다.

## 어떻게 확인하는가

1. **활성화** — `provider contract transactional_query` 위치, `as projection on ZI_Booking`, 필드 목록이 올바른지 본다.
2. **root 위치 일치** — `ZI_Booking`이 root인데 `ZC_Booking`이 root가 아니면 projection tree 위치가 맞지 않는다.
   "기반이 root면 소비 projection도 root로 맞춘다"로 기억한다.
3. **필드 노출** — `ZC_Booking`은 외부 소비 모델이므로 어떤 필드를 보여 줄지 의도적으로 결정한다. 내부 감사
   필드는 이 projection에서 뺄 수 있다.
4. **Metadata Extension 계획** — DDLX를 붙일 거면 `@Metadata.allowExtensions: true`가 있는지 본다([Metadata
   Extension 기초](CH23-L05.html)).

## 흔한 실수와 점검 🚧

- **provider contract를 장식처럼** → transactional projection에서는 어떤 runtime 시나리오용인지 *계약*을 드러낸다.
  contract가 없으면 runtime-specific syntax check를 기대하기 어렵다.
- **`ZI_Booking`을 서비스에 직접 expose** → 작은 실습에서는 되어 보여도, 실무에서는 소비 계층 `ZC_*`를 따로
  두는 편이 변경에 강하다.
- **모든 내부 필드를 무조건 올리기** → projection은 *소비 계약*이다. 외부에 보여 줄 필요 없는 필드는 숨긴다.
- **CDS 계층 분리와 transactional projection을 동일시** → [Interface View와 Consumption View 계층](CH23-L02.html)에서
  배운 건 계약 없이 뷰를 쌓는 **읽기 전용** 소비 계층(nesting)이다. 여기서는 그 위에 `provider contract`를 붙여
  RAP transactional BO에 소속시키는 **Projection View**를 배운다 — 계약이 있고 없고가 갈림길이다.
