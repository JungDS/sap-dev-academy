## 🎫 조각을 하나의 예매 흐름으로

이제 이 챕터의 조각을 하나의 콘서트 예매 흐름으로 묶는다. 목표는 "RAP 앱 전체를 완벽히 운영 배포한다"가
아니라, RAP 구조를 실제 예매 시나리오에 연결해 보는 것이다. 실습의 핵심은 둘이다 — **정원 초과 예매는
validation으로 막고, 취소는 action으로 명시 실행한다**. 여기에 생성 시 기본 상태를 determination으로 채우면
세 로직의 차이를 한 번에 확인할 수 있다.

| 단계 | 산출물 | 역할 |
|---|---|---|
| 1 | `ZI_Booking` | 예매 root interface view |
| 2 | `ZC_Booking` | transactional projection view |
| 3 | BDEF | create/update/delete · validation · determination · action 선언 |
| 4 | Behavior Pool | validation/determination/action 구현 위치 |
| 5 | Service Definition | `ZC_Booking` expose |
| 6 | Service Binding | OData로 활성화하고 Preview 확인 |

## BDEF — 세 로직을 선언

```abap
managed implementation in class zbp_i_booking unique;

define behavior for ZI_Booking alias Booking
persistent table zbooking
lock master
{
  create;
  update;
  delete;

  validation validate_capacity on save { field seats, concert_id, perf_no; }
  determination set_status_new on modify { create; }
  action cancel result [1] $self;

  field ( readonly:update ) booking_id;
  mapping for zbooking { booking_id = booking_id; }
}
```

정원 validation의 behavior pool 흐름은 이렇게 설계한다(골격).

```abap
METHOD validate_capacity.
  READ ENTITIES OF ZI_Booking IN LOCAL MODE
    ENTITY Booking FIELDS ( booking_id concert_id perf_no seats )
    WITH CORRESPONDING #( keys )
    RESULT DATA(lt_bookings).

  LOOP AT lt_bookings INTO DATA(ls_booking).
    " 1. concert_id·perf_no로 회차 정원을 찾는다.
    " 2. 이미 예약된 좌석 수 + 이번 요청 좌석 수를 비교한다.
    " 3. 초과하면 failed-booking·reported-booking에 기록한다.
  ENDLOOP.
ENDMETHOD.
```

취소 action은 상태를 바꾸는 명시 동작이다(전체 구현은 줄이고, action이 validation과 *다른 종류*임을 확인한다).

```abap
METHOD cancel.
  " 1. keys로 취소 대상 예매를 읽는다.
  " 2. 이미 취소된 건인지 확인한다.
  " 3. 취소 가능하면 status를 'C'로 바꾸는 modify request를 수행한다.
ENDMETHOD.
```

## 확인 체크리스트

| 케이스 | 입력 | 기대 결과 |
|---|---|---|
| 정상 생성 | 좌석 2, 잔여 10 | 저장 성공, 상태 `N` |
| 정원 초과 | 좌석 12, 잔여 10 | validation 실패, `failed/reported` 메시지 |
| 취소 | 상태 `N` 예매에 cancel | 상태 `C` |
| 중복 취소 | 이미 `C`인 예매에 cancel | 실패 메시지 또는 no-op 정책 |
| service preview | binding 활성화 후 preview | Booking 목록·`cancel` action 노출 |

## 흔한 실수와 점검 🚧

- **validation 실패를 화면 메시지로만** → 실패 instance를 `failed`에, 사용자 메시지를 `reported`에 기록해야
  runtime·UI가 제대로 반응한다.
- **정원 계산을 현재 입력 좌석만 보기** → 이미 예약된 좌석·취소 제외·같은 회차 합계·현재 transaction 변경분을
  함께 봐야 한다(실습에선 이 기준을 설계 질문으로 적어 둔다).
- **action과 determination 섞기** → 취소는 사용자가 명시 실행하므로 action, 생성 시 기본 상태는 자동 결정이므로
  determination이 자연스럽다([Validation / Determination / Action 개요](CH24-L07.html)).
- **`IN LOCAL MODE` 남용** → behavior implementation에서 해당 RAP BO 자체를 읽는 특수 EML 형태다([Behavior
  Implementation 기초](CH24-L05.html)). 일반 소비 프로그램에 습관적으로 붙이는 문법이 아니다.
- **직접 DML로 우회** → behavior pool 안에서 테이블을 직접 바꾸면 RAP buffer·save sequence를 우회할 수 있다.
  직접 DB 변경은 [INSERT / UPDATE / MODIFY / DELETE 실무 기준](CH25-L01.html)에서 따로 다룬다.
