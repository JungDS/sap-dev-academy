## 왜 서비스로 노출할까

RAP BO가 시스템 안에서만 존재하면 사용자가 앱으로 쓸 수 없다. Fiori 화면, 외부 [[OData]] client, 테스트 도구가
접근하려면 **business service**로 노출해야 한다. 여기서 두 단계가 필요하다 — 무엇을 노출할지 정하는 **[[Service Definition]]**, 어떤 protocol로 실제 접근 가능하게 만들지 정하는 **[[Service Binding]]**이다.

이 둘을 섞으면 문제 해결이 어려워진다. `expose`를 안 해서 entity가 서비스에 없는데 binding만 만지거나,
binding을 활성화하지 않았는데 service definition만 보고 URL을 찾는 식의 혼란이 생긴다.

## 무엇인가 — 노출(Definition)과 연결(Binding)

**① Service Definition**은 CDS entity를 RAP business service로 노출하는 SDL source다.

```service
define service ZUI_Booking {
  expose ZC_Booking as Booking;
}
```

`ZC_Booking`은 [Projection View ZC_* 설계](CH24-L03.html)에서 만든 소비 projection이고, `Booking`은 서비스에서
보일 노출 이름이다. **Service Definition에 넣은 entity만 외부에 노출된다.**

**② Service Binding**은 ADT의 form-based repository object로, Service Definition을 OData 같은 RESTful protocol과
연결한다. binding type에는 OData V2/V4, UI용·Web API용 변형이 있다. **UI용**은 SAPUI5/Fiori가 쓸 control
information을 포함할 수 있고, **Web API용**은 data-only 성격이 강하다. ADT에서 binding을 **Activate**하면 URL과
preview를 쓸 수 있다.

## 어떻게 확인하는가

1. **expose 확인** — `expose ZC_Booking as Booking;`이 있는지. `ZI_Booking`이 아니라 소비 projection `ZC_Booking`을
   노출하는 구조가 명확해야 한다.
2. **binding 참조** — binding이 어떤 service definition을 참조하는지. 다른 definition에 연결돼 있으면 원하는 entity가
   안 보인다.
3. **Activate** — binding을 활성화하면 protocol에 따라 접근이 가능해지고 ADT가 URL·testing을 제공한다.
4. **$metadata / Preview** — service definition의 alias `Booking`이 metadata에 노출되는지 본다.

## 흔한 실수와 점검 🚧

- **Definition과 Binding을 같은 것으로** → Definition은 *무엇을* 노출할지, Binding은 *어떻게* 접근하게 할지다.
- **binding 활성화 빼먹기** → source가 활성화돼도 binding이 inactive면 URL·preview를 못 쓴다.
- **아래 모델이 미완성인데 service만 테스트** → service는 아래 모델의 문제를 그대로 드러낸다. service 오류가
  나면 root·projection·BDEF·annotation·key를 함께 본다.
- **UI용과 Web API용 binding 목적 섞기** → UI는 annotation·control information의 영향을 크게 받고, Web API는 data
  contract에 더 집중한다.
