## 왜 로직을 셋으로 나눌까

트랜잭션 앱은 저장 버튼만 있으면 안 된다. 사용자가 잘못된 좌석 수를 입력하면 **저장 전에 막아야** 하고,
새 예매의 기본 상태는 **자동으로 채워야** 하며, 취소처럼 사용자가 **명시적으로 실행**하는 업무 버튼도 필요하다.
이 세 종류의 로직을 같은 곳에 섞으면 *언제 실행되는지* 알기 어렵다.

RAP는 이 로직을 **Validation · Determination · Action**으로 나눈다. 이 구분을 이해해야 BDEF를 읽을 수 있고,
behavior pool의 구현 위치도 찾을 수 있다.

## 무엇인가 — 목적과 실행 시점이 다르다

| 종류 | 목적 | 실행 시점 | 콘서트 예매 예 |
|---|---|---|---|
| **Validation** | 저장해도 되는지 검사 | save sequence | 정원 초과 예매 거부 |
| **Determination** | 값을 자동으로 계산·채움 | on modify 또는 on save | 생성 시 `status = 'N'` 기본값 |
| **Action** | 사용자가 명시적으로 실행 | 버튼·API 호출 | 예매 취소 |

BDEF 선언은 이렇게 한다.

```abap
define behavior for ZI_Booking alias Booking
{
  validation validate_capacity on save { field seats, concert_id, perf_no; }
  determination set_status_new on modify { create; }
  action cancel result [1] $self;
}
```

- `validation validate_capacity on save { field ... }` — 해당 필드가 관련된 create/update 이후 **save 시점**에
  정원 초과를 검사한다.
- `determination set_status_new on modify { create; }` — 생성 요청이 들어왔을 때 **상태 기본값**을 자동으로 잡는다.
- `action cancel result [1] $self;` — 사용자가 취소를 호출하고, 결과로 **자기 자신 한 건**을 돌려받는다.

정원 초과 검증은 [Exception Class — TRY / CATCH / CX 계층](CH21-L06.html)에서 만든 `ZCX_FULLY_BOOKED`의 RAP
버전이라고 볼 수 있다 — 다만 RAP에서는 예외를 던지는 대신 `failed/reported`로 실패를 알린다.

## 어떻게 확인하는가

1. **목적 분류** — "잘못된 데이터면 막는다"=validation, "값을 자동으로 채운다"=determination, "사용자가
   버튼으로 실행한다"=action.
2. **BDEF 선언** — behavior pool에 method만 만들어서는 RAP runtime이 그 동작을 알 수 없다. BDEF에 선언이 있어야 한다.
3. **behavior pool method** — `validate_capacity`는 `FOR VALIDATE ON SAVE`, `set_status_new`는 `FOR DETERMINE ON
   MODIFY`, `cancel`은 action handler로 구현한다([Behavior Implementation 기초](CH24-L05.html)).
4. **실패 반응** — validation이 실패하면 `failed`·`reported`에 인스턴스·메시지를 기록한다. validation 실패는
   transactional buffer를 거부할 수 있으므로 "오류 메시지만 띄우고 계속 저장"이 아니다.

## 흔한 실수와 점검 🚧

- **모든 로직을 action으로** → 사용자가 눌러야 하는 동작은 action이지만, 저장 전 반드시 지킬 규칙은 validation이
  더 자연스럽다.
- **validation에서 값 고치기** → validation은 검사·실패 보고가 중심이다. 값 자동 채움은 determination에 둔다.
- **determination 실행 순서에 과도하게 의존** → 여러 로직이 같은 조건으로 트리거될 때 실행 순서를 전제로 짜면
  위험하다. 각 로직은 필요한 데이터를 명확히 읽고 결과를 명확히 남긴다.
- **action 결과를 설계 안 함** → `result [1] $self`는 action 후 자기 자신 한 건을 결과로 돌려준다. UI가 갱신할
  데이터를 기대한다면 result 설계를 함께 본다.
