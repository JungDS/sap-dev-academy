## CH08은 읽기였다 — 이제 쓰기

CH08의 Open SQL은 **읽기 전용**이었다. 실무는 데이터를 **바꿔야** 한다. DB를 직접
변경하는 네 문장이 [[DML]](Data Manipulation Language)이다.

| 문 | 동작 |
|---|---|
| `INSERT` | 새 행 추가(이미 있으면 오류) |
| `UPDATE` | 기존 행 수정 |
| `MODIFY` | 있으면 수정, 없으면 추가(upsert) |
| `DELETE` | 행 삭제 |

```abap
INSERT zbooking FROM @ls_booking.
UPDATE zbooking SET status = 'X' WHERE booking_id = @lv_id.
MODIFY zbooking FROM @ls_booking.
DELETE FROM zbooking WHERE booking_id = @lv_id.
```

## 통째로 — FROM TABLE

CH07에서 손입력이 번거롭고, CH08에서 "메모리 itab을 통째로 넣고 싶다"고 했다. 이제
**내부 테이블을 한 번에** 넣는다.

```abap
INSERT zgugudan FROM TABLE @lt_gugu.        " 81행을 한 번에
MODIFY zbooking FROM TABLE @lt_booking.
```

## 감사필드 자동 stamp

"누가·언제" 바꿨는지 기록할 컬럼(생성자·일시)을 둔다. CH04의 SY 변수로 채운다.

```abap
ls_booking-created_by = sy-uname.
ls_booking-created_on = sy-datum.
ls_booking-created_at = sy-uzeit.
INSERT zbooking FROM @ls_booking.
```

## ⚠️ 표준 테이블은 직접 쓰지 않는다

`SCARR`·`SFLIGHT` 같은 **SAP 표준 테이블**은 `INSERT`/`UPDATE`로 직접 바꾸면 안 된다
(검증·후속 로직 우회 → 데이터 깨짐). 반드시 **BAPI/표준 API**를 경유한다. 직접 DML은
**내 테이블**(`ZBOOKING` 등)에만.

## 흔한 실수와 점검

- **INSERT 중복키** → 이미 있으면 덤프. upsert가 필요하면 `MODIFY`.
- **WHERE 없는 UPDATE/DELETE** → 전체가 바뀐다. 조건을 반드시.
- **표준 테이블 직접 DML** → 금지. BAPI 경유.
