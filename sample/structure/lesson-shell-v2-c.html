라벨을 누르면 SE11이 무엇이고 어떻게 여는지, 화면 구성과 주의점까지 한 장으로 설명합니다.

🎯학습 목표

이번 Lesson에서는 Domain을 "값의 기술 규격"으로 설계합니다. 수업이 끝나면 Data Type, Length, Output Length, Fixed Value의 역할을 나눠 설명하고, SE11에서 작은 Domain을 저장·검사·활성화하는 흐름을 직접 시뮬레이션할 수 있어야 합니다.

1. Domain은 업무 의미가 아니라 값의 재질을 정한다

지난 Lesson에서 DDIC는 SAP 시스템이 공유하는 데이터 정의의 기준점이라고 봤습니다. Domain은 그 기준점 안에서 가장 아래쪽에 놓이는 부품입니다.

"주문 상태"라는 업무 의미보다 먼저 "한 글자 코드인가, 숫자인가, 허용값이 고정되어 있는가"라는 값의 재질을 정합니다.

비유: Domain은 "그릇의 규격(재질·크기)", 업무 의미를 붙이는 건 다음 단계(Data Element)에서 "그릇에 이름표 붙이기"입니다.

2. 네 가지 속성을 나눠서 본다

Domain을 만들 때 정하는 핵심 속성은 네 가지예요. 지금은 "이런 게 있구나" 정도면 충분합니다.

Data Type & Length — 값의 종류와 칸의 크기
계산할 값인지(숫자), 코드인지(문자)를 먼저 정하고, 저장 길이(자릿수)를 정합니다. 예: 상태 코드는 CHAR 1.
Output Length — 화면 표시 폭
저장 길이와 별개로 화면에 보여줄 폭입니다. 보통은 같지만, 통화·부호 등으로 표시가 더 길어질 수 있어요.
Fixed Value — 허용값 고정
값을 정해진 목록으로만 받게 제한합니다. 작고 안정적인 코드 목록(A/B/C)에 적합하며 입력 도움말과 기본 검증에 쓰입니다.

3. SE11에서 만드는 흐름 (저장 → 검사 → 활성화)

SE11에서 Domain을 만들면 저장(Save) → 검사(Check) → 활성화(Activate) 3단계를 거칩니다. 활성화까지 끝나야 다른 객체가 이 Domain을 가져다 쓸 수 있어요.

주의: 저장만 하고 활성화를 안 하면 "왜 안 보이지?"가 됩니다. 활성화는 "이제 진짜 쓸게요" 선언이에요.

🧾한눈에 정리

Domain값의 기술 규격과 작은 허용 범위를 정하는 DDIC 하위 부품입니다.
Data Type과 Length계산할 값인지 보존할 코드인지 먼저 보고 저장 길이를 정합니다.
Fixed Value작고 안정적인 코드 목록에 적합하며 입력 도움과 기본 검증에 쓰입니다.
Value Table자동 검증이 아니라 Foreign Key 후보를 제안하는 성격으로 이해합니다.
