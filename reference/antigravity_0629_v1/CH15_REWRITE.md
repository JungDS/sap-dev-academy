# CH15_REWRITE - Report Event·Selection Screen 심화 v1

> 목적: `content/abap/CH15`의 12개 레슨을 "IT 비전공자 입문자가 왜 필요한지부터 실습 확인까지 직관적으로 이해하고 따라갈 수 있는 완성 강의자료" 수준으로 재작성한 기준 원고다. 이 문서는 아직 `content/abap` 원본을 덮어쓴 결과가 아니라, 재작업 판정 이후 실제 반영 전 검수 가능한 v1 기준안이다.

## CH15 전체 설계

CH15의 한 문장 목표는 "실행형 리포트 프로그램의 생명주기 이벤트(INITIALIZATION, OUTPUT, PAI, START-OF-SELECTION, LDB-END)를 정밀하게 격리하고, 화면 구성 요소와 MODIF ID를 통한 동적 필드 제어, 존재와 권한의 2단 검증 가드, 커스텀 F1/F4 및 여러 화면번호(CALL)와 Variant 설정을 조립해 완결성 있는 예매현황 리포트 캡스톤을 조립한다"이다.

IT 비전공자 입문자는 이벤트 선언을 누락해 모든 코드가 START-OF-SELECTION에 묶여 순서가 꼬이거나, PBO 시점에 사용자가 입력한 필드를 덮어쓰고, 에러 메시지 검증 위치를 혼동하며, Obsolete 문법인 END-OF-SELECTION을 필수 마무리 이벤트로 오인하는 실수를 유발한다.
따라서 본 챕터는 다음과 같은 불편 해결의 단계적 설계로 전개한다.

1. **절차적 꼬임 해소**: 위에서 아래로 흐르는 단층 코드가 아니라, 약속된 이벤트에 맞춰 각 구획을 호출하는 **Report Event 전체 흐름** 체화.
2. **좋은 출발점 제공 (INITIALIZATION)**: 매번 강제 덮어쓰기가 아닌 화면 뜨기 전 딱 1회 오늘 날짜나 기본 조직 코드를 준비하고, SELECT-OPTIONS의 헤더라인을 APPEND로 적재해 주는 **INITIALIZATION** 원리 정립.
3. **상황별 동적 화면 제어 (OUTPUT)**: PBO 시점에 `LOOP AT SCREEN INTO gs_screen`을 가동해 필드를 숨기거나 잠그고, `MODIFY SCREEN FROM gs_screen`으로 반영하는 **화면 제어** 실무.
4. **잘못된 입력의 원천 차단 (PAI)**: 실행 직후 입력값을 확인하는 전체 검증(`AT SELECTION-SCREEN`)과 특정 필드만 콕 집어 화면을 잠그는 개별 검증(`AT SELECTION-SCREEN ON <field>`)의 동작 차이 분석.
5. **메시지 클래스 규격 (MESSAGE)**: SE91을 통한 메시지 재사용, `&1`~`&4`와 `WITH` 자리표시자 활용법 및 메시지 타입(I/S/W/E/A/X)에 따른 시스템 거동 격리.
6. **조회 본체의 시작 (START-OF-SELECTION)**: 검증을 무사히 통과한 알짜배기 조건을 가지고 DB를 스캔하는 **START-OF-SELECTION**의 책임 정의.
7. **END-OF-SELECTION의 아키텍처 경계**: Logical Database(LDB)의 역사적 배경과 obsolete 특성을 이해하고, 신규 리포트에서의 불필요성 및 레거시 독해용 격리.
8. **존재와 권한의 2단 검문소**: DB에 실존하는지 묻는 **존재 검증**(`SELECT SINGLE`)과 안전하게 볼 수 있는 계정인지 묻는 **권한 검증**(`AUTHORITY-CHECK`)의 2단계 보안 결합.
9. **UI 요소 조립**: 선택화면을 보기 좋게 구성하는 `BLOCK`, `COMMENT FOR FIELD`, `POSITION` 문법과 라디오 배타 그룹 체결법.
10. **버튼과 툴바 ucomm**: 본문 푸시버튼과 툴바 단축키 이벤트 수납 시 `SY-UCOMM` 대신 **`SSCRFIELDS-UCOMM`** 을 사용하는 이유와 제어법.
11. **옵션 카탈로그**: `OBLIGATORY`, `LOWER CASE`(문자형 전용 제약), `MEMORY ID`, 그리고 To칸을 숨기는 `NO INTERVALS`와 복수 입력을 막는 `NO-EXTENSION` 옵션 대조.
12. **여러 화면과 CALL**: 기본 1000번 화면 외에 보조 화면을 AS WINDOW로 빚어내어 **`CALL SELECTION-SCREEN`** 으로 팝업 호출하고 `sy-subrc`로 실행/취소를 갈라내는 화면 기획.
13. **Variant와 예매현황 실습**: 입력 세트를 보관하는 Variant 개념을 이해하고, ZCONCERT와 ZBOOKING을 연동해 모든 이벤트가 제자리에서 가동되는 종합 예매 리포트 캡스톤 구현.

### 범위와 금지선

- **R6 classic-first 철저 수호**:
  - `SELECT * FROM zbooking INTO TABLE gt_booking` 와 같이 classic Open SQL과 전역 변수 선언(TYPE 참조)만으로 소스를 채우며, `@`나 `@DATA(...)` 같은 modern New Open SQL 문법은 완벽히 차단한다.
- **변수 스코프 접두어**: 모든 스크립트는 전역 범위로 간주하며, 일반 변수는 `gv_`, 구조체는 `gs_`, 내부 테이블은 `gt_`, 클래스 참조 변수는 `lo_`/`go_` 접두어를 엄수한다.
- **이름 풀 준수**: 예제 이름은 지정된 이름 풀에서만 가져오며, 주인공은 항상 **정훈영**으로 통일한다.

## CH15-L01 - ABAP Report Event 전체 흐름

### 왜 필요한가

우리가 프로그램을 짤 때, 코드는 보통 위에서 아래로 한 줄씩 순서대로 실행된다고 배운다. 
하지만 화면이 있는 리포트 프로그램은 그렇게 단순하게 흘러가지 않는다. 

사용자가 프로그램을 켜자마자 기본값이 채워져 있어야 하고, 화면에서 조회 조건을 치고 실행 버튼을 누르면 DB를 조회하기 전에 값에 오타가 없는지 먼저 검문해야 하며, 검문에 통과해야 비로소 실제 무거운 DB 조회 로직이 돌아가야 한다. 
만약 이 모든 코드를 하나의 위에서 아래로 흐르는 길쭉한 소스 파일에 몰아넣으면, 프로그램은 "화면이 뜨기도 전에 조회를 먼저 시작해 버리거나", "입력값이 잘못되었는데도 조회를 멈추지 않고 끝내 버리는" 처참한 순서 꼬임 장애를 겪게 된다. 

**각각의 시점에 맞춰 알맞은 방(Block)을 미리 준비해 두고, 약속된 상황(화면 준비, 실행 클릭 등)이 올 때마다 SAP 시스템이 해당 방을 정교하게 똑똑 노크해 실행시키는 이벤트 중심 실행 구조**가 필요하다. 그것이 **[[Report Event]]** (리포트 이벤트)이다.

### 무엇인가

#### 1. Report Event (리포트 이벤트)의 정의
- **정의**: 실행형 프로그램(Executable Program)이 가동될 때, 화면을 그리기 전부터 프로그램이 끝날 때까지의 단계별 생명주기(Life Cycle) 시점에 맞춰 아밥 시스템이 호출해 주는 구획 단위의 약속된 키워드 블록이다.
- **🍳 비유 - 식당의 하루**:
  - `LOAD-OF-PROGRAM`: 식당 아침 오픈 준비 (메모리 적재)
  - `INITIALIZATION`: 주문지 기본 메뉴 세팅 (화면 뜨기 전 기본값 설정)
  - `AT SELECTION-SCREEN OUTPUT`: 메뉴판 꾸미기 (화면 그리기 직전 PBO)
  - `AT SELECTION-SCREEN`: 주문 검수 (입력값 검증 PAI)
  - `START-OF-SELECTION`: 주방 조리 시작 (본격 DB 조회 및 가공)
  - `END-OF-SELECTION`: 마감 및 정리 (출력 마무리)

#### 2. 이벤트 블록 선언 규칙
- 소스 코드에 이벤트 키워드(예: `START-OF-SELECTION.`)를 기입하면, 그 아래에 적히는 코드들은 다음 이벤트 키워드가 나타날 때까지 해당 시점 방에 묶이게 된다.
- 만약 프로그램 맨 위에 아무 키워드도 안 쓰고 명령어를 적어두면, 그 명령어들은 아밥 시스템이 암묵적으로 `START-OF-SELECTION` (조회 실행 시점)의 방에 묶어서 강제로 가둬 버리므로, 시점별 키워드를 똑바로 명시하는 설계 버릇이 핵심이다.

### 어떻게 확인하는가

에디터에서 이벤트 키워드를 여러 개 배치해 두고, 프로그램 실행 시 콘솔에 찍히는 호출 순서가 소스 코드 위치와 무관하게 시스템 정의 순서대로 출력되는지를 검증한다.

```abap
REPORT zch15_l01_event_flow.

INITIALIZATION.
  WRITE: / '1. INITIALIZATION 실행됨 (화면 뜨기 전)'.

START-OF-SELECTION.
  WRITE: / '3. START-OF-SELECTION 실행됨 (본 조회)'.

AT SELECTION-SCREEN.
  " 이 구획은 사용자가 화면을 보고 실행을 클릭해야 돌지만,
  " 개념상 시점 순서는 INITIALIZATION과 START-OF-SELECTION 사이다.
  WRITE: / '2. AT SELECTION-SCREEN 실행됨 (검증)'.
```

#### 체험/시뮬레이터 설계 (점진적 빌드업)
- **프로세스 플로우**:
  1. 학습자가 [1단계: 이벤트 없는 소스] 탭을 클릭한다. 모든 코드가 암묵적으로 본처리에 묶여 한 번에 출력되는 밋밋한 흐름을 본다.
  2. [5단계: 전체 생명주기 소스] 탭을 누른다. 
  3. 신호등 모양의 이벤트 실행 바가 `INITIALIZATION` 녹색등 -> `OUTPUT` 노란색등 -> `AT SELECTION-SCREEN` 주황색등 -> `START-OF-SELECTION` 파란색등 순으로 통과하며, 우측의 코드 구획이 시점 순으로 번쩍번쩍 하이라이트 활성화되는 모션을 감상한다.
- **상태 및 데이터**:
  - `AT SELECTION-SCREEN` 내부에서 오류 메시지 트리거 -> 신호등 바가 `START-OF-SELECTION`으로 넘어가지 못하고, 빨간색 점선 화살표를 그리며 다시 `OUTPUT` 화면으로 회귀 복귀하는 모션 피드백.
- **피드백**: 이벤트 자리를 틀려 조회 구획에 검증 코드를 넣으면 데이터 유출 방어가 불가능해짐을 빨간 경고 메시지로 인지시킨다.

### 실수/주의

- **검증 코드를 START-OF-SELECTION에 두기**: "어차피 SELECT 하기 전에 검사하면 되니까 조회 방에 검증 코드를 넣어야지" 하면, 사용자가 값을 잘못 쳤을 때 화면이 복귀되는 것이 아니라 프로그램 자체가 이미 본처리를 타버려 빈 페이지로 굳어 버리거나 오류를 뿜게 되므로, **검증은 반드시 PAI인 `AT SELECTION-SCREEN`** 에 봉인해야 한다.

### 정리

- 실행형 프로그램은 위에서 아래로 그냥 흐르지 않고, **약속된 이벤트 생명주기 순서**대로 방을 돌며 호출한다.
- 키워드를 누락하면 코드는 암묵적으로 **`START-OF-SELECTION`** 에 감금되므로 시점을 명확히 기재한다.
- 기본값은 `INITIALIZATION`, 검증은 `AT SELECTION-SCREEN`, 조회는 `START-OF-SELECTION` 이 자리가 진리다.

## CH15-L02 - INITIALIZATION 기본값 설정

### 왜 필요한가

매일 아침 출근한 대리 정훈영 씨가 공연 예매 리포트 프로그램을 켭니다. 
그런데 켤 때마다 조회 조건 입력창이 완전히 하얀 빈칸으로 텅 비어있다면 어떨까? 
정훈영 씨는 매번 키보드를 탁탁 치며 "오늘 날짜인 `2026-06-30`"을 손으로 기입하고, 늘 조회하는 "기본 공연 ID `C001`"을 타자 쳐서 입력하는 귀찮은 수작업을 매일 무한 반복해야 한다. 

프로그램을 켜자마자 가장 대중적으로 자주 사용하는 오늘 날짜나 디폴트 코드 값을 **입력창에 기본 제안으로 미리 예쁘게 얹어서 채워주는 센스 있는 초기 세팅 장치**가 필요하다. 그것이 **[[INITIALIZATION]]** (초기화 시점)이다.

### 무엇인가

#### 1. INITIALIZATION (초기화 시점)의 정의
- **정의**: 선택화면(Selection Screen)이 사용자의 모니터 모니터 화면에 **최초로 렌더링되어 켜지기 직전 단 1회만 실행**되는 특수 초기화 이벤트다.
- **주요 용도**: `PARAMETERS` 단건 입력창의 시작 기본값을 주입하거나, `SELECT-OPTIONS` 레인지 테이블의 헤더라인을 조립해 디폴트 범위 조건을 채워두는 목적으로 활약한다.

#### ⚠️ [SELECT-OPTIONS 의 헤더라인 APPEND 철칙]
- *초보 개발자들이 가장 흔하게 겪는 "기본값이 안 채워져요" 버그의 원인이다.*
- SELECT-OPTIONS(범위 조회 필터)는 구조가 단일 변수가 아니라 내부 테이블(Range Table) 구조다.
- 따라서 단순히 `s_dept-low = '1000'.` 이라고 값만 기입하고 멍하니 있으면 화면에 아무것도 안 나온다.
- **반드시 값을 채운 뒤 `APPEND s_dept.` 문장을 명시적으로 기입하여 내부 테이블에 조건 레코드를 1건 집어넣어 주는 물리적 APPEND 절차를 밟아야만** 화면에 기본값이 상속되어 인쇄된다.

### 어떻게 확인하는가

에디터에서 INITIALIZATION 이벤트 블록을 개설하여, 오늘 날짜와 레인지 테이블 기본값을 채워 화면에 안착시키는 소스를 검증한다.

```abap
REPORT zch15_l02_initialization.

DATA gv_dept TYPE c LENGTH 4.

PARAMETERS p_date TYPE d.
SELECT-OPTIONS s_dept FOR gv_dept.

INITIALIZATION.
  " 1. 단일 파라미터는 변수에 바로 대입하면 1000번 화면에 즉시 제안됨!
  p_date = sy-datum.

  " 2. 레인지 필터는 Range 4대 필드를 채우고 반드시 APPEND 철칙 준수!
  s_dept-sign   = 'I'.
  s_dept-option = 'EQ'.
  s_dept-low    = '1000'.
  APPEND s_dept.
```

#### 체험/시뮬레이터 설계 (기본값 주입기)
- **프로세스 플로우**:
  1. 학습자가 [INITIALIZATION 적용] 모드와 [매 화면 덮어쓰기 적용] 모드를 선택하는 스위치 판을 본다.
  2. [INITIALIZATION] 모드에서 화면 날짜 값을 `2026-07-05` 로 수정하고 [엔터]를 치면, 값이 유지되는 것을 확인한다. (최초 1회만 초기화하므로 사용자 입력을 존중함).
  3. [매 화면 덮어쓰기 (OUTPUT 시점 대입)] 모드로 스위치를 전환한다. 날짜 값을 고치고 엔터를 치자, 화면이 다시 PBO를 타며 사용자가 고친 `2026-07-05`를 무참히 지워버리고 다시 오늘 날짜로 강제 복원 덮어쓰기 해버리는 버그 연출을 감상한다.
- **상태 및 데이터**:
  - `s_dept` 레인지 테이블 대입 후 APPEND 생략 상태 -> 화면 로우가 텅 비어 아무 기본값도 표시되지 않는 현상 데이터 하이라이트.
- **피드백**: 기본값 제안은 단 1회만 실행되어 사용자의 수정 권리를 보호하는 INITIALIZATION 방에 봉인해야 함을 인지시킨다.

### 실수/주의

- **기본값 주입을 START-OF-SELECTION에서 하기**: "시작값을 주는 거니까 본처리 방에 넣어야지" 했다가는, 이미 사용자가 화면을 보고 실행을 누른 뒤인 본처리 시점이라, 텅 빈 화면만 사용자에게 노출되고 프로그램이 끝난 뒤에 혼자 변수 값을 채우는 뒷북 버그를 겪게 되므로 반드시 **선택화면 노출 전인 `INITIALIZATION`** 에서 처리한다.

### 정리

- **`INITIALIZATION`** 은 선택화면이 첫 노출되기 전 딱 1회 돌아가는 기본값 주입 전용 시점이다.
- `SELECT-OPTIONS` 레인지 기본값을 제안할 때는 **Range 필드 대입 후 반드시 `APPEND`** 를 실행해야 수납된다.
- 사용자가 바꾼 화면 값을 덮어쓰지 않고 온전히 지켜주기 위해 1회성 초기화 자리를 사수한다.

## CH15-L03 - AT SELECTION-SCREEN OUTPUT 동적 화면 제어

### 왜 필요한가

우리가 선택화면을 만들면, 선언한 모든 입력 상자(Parameters)들이 한눈에 훤하게 다 노출된다. 
하지만 비즈니스 요구사항은 늘 변덕스럽다. 
"사용자가 '간편 조회' 라디오 버튼을 선택했을 때는 복잡한 상세 필터 상자들을 싹 숨겨서 화면을 슬림하게 보여주고, '고급 상세 조회' 라디오 버튼을 클릭하는 순간 숨어있던 비밀 번호나 세부 일자 입력창이 스르륵 살아나게 해주고 싶다" 
조회 버튼이나 라디오 버튼 등 사용자의 액션이 감지되어 **화면이 다시 그려져 나가기 직전(PBO)에, 입력창들의 투명도와 비활성화 속성을 쥐락펴락하여 화면 인터페이스 레이아웃을 다이내믹하게 켜고 꺼주는 스위칭 가드**가 필요하다. 그것이 **[[AT SELECTION-SCREEN OUTPUT]]** (화면 출력 직전 시점)이다.

### 무엇인가

#### 1. AT SELECTION-SCREEN OUTPUT (화면 그리기 직전)의 정의
- **정의**: SAP 화면 프레임워크의 **PBO (Process Before Output - 화면 출력 전 이벤트)** 시점과 100% 매칭되는 선택화면 전용 이벤트 구획이다.
- **주요 용도**: 화면이 사용자 눈에 뿌려지기 직전마다 호출되므로, 특정 입력 상자를 숨기거나(`active = '0'`), 읽기 전용으로 얼려 잠그는(`input = '0'`) 등 화면 요소의 속성을 지배하는 용도로 쓰인다.

#### ⚠️ [MODIFY SCREEN 필수 갱신 철칙]
- *아밥 개발자들이 가장 많이 헤매는 "화면 속성을 바꿨는데 왜 그대로죠?" 에러의 주범이다.*
- 아바 내부의 화면 정보는 `screen` 이라는 명시적 구조체로 돌게 된다.
- `LOOP AT SCREEN INTO gs_screen` 루프 안에서 `gs_screen-active = '0'.` 으로 속성을 아무리 백날 수정해 봐야,
- **루프 내부에서 수정 즉시 `MODIFY SCREEN FROM gs_screen.` 이라는 아바 지시어로 화면에 도로 밀어 넣어 반영해주지 않으면 메모리 상의 수정 내역이 최종 화면에 절대로 투사되지 않고 소실**된다.

#### 2. screen-name 은 대문자로 대조
- SAP 시스템 사전의 모든 화면 필드 이름은 **전부 대문자(Uppercase)** 로 저장된다.
- 따라서 비교 구문을 짤 때 소문자로 `IF gs_screen-name = 'p_secret'` 이라고 적으면 백날 돌아도 매칭이 안 되어 속성 제어가 안 먹으므로, 반드시 대문자인 `'P_SECRET'` 으로 대조해야 하는 사양 제약이 존재한다.

### 어떻게 확인하는가

에디터에서 AT SELECTION-SCREEN OUTPUT 이벤트를 열어, 특정 파라미터 필드를 동적으로 숨김/잠금 처리하는 로직을 검증한다.

```abap
REPORT zch15_l03_screen_output.

PARAMETERS: p_mode   TYPE c LENGTH 1,
            p_secret TYPE c LENGTH 10.

" screen 구조를 받아낼 work area 명시적 선언!
DATA gs_screen TYPE screen.

AT SELECTION-SCREEN OUTPUT.
  " 1. 루프를 돌며 화면 구성원들을 하나씩 꺼내 검사!
  LOOP AT SCREEN INTO gs_screen.
    " [대문자 매치 철칙] 필드명은 무조건 대문자 대조!
    IF gs_screen-name = 'P_SECRET' AND p_mode <> 'A'.
      " 2. 속성을 0 (비활성/숨김) 으로 조정!
      gs_screen-active = '0'.
      " [반영 철칙] 수정한 속성을 반드시 화면 객체에 밀어 넣기!
      MODIFY SCREEN FROM gs_screen.
    ENDIF.
  ENDLOOP.
```

#### 체험/시뮬레이터 설계 (SCREEN 속성 조정 패널)
- **프로세스 플로우**:
  1. 학습자가 [p_mode = 'B'] 상태로 둔다. 화면에 `p_secret` 입력창이 보이지 않는다.
  2. [p_mode 값을 'A' 로 변경] 단추를 누른다.
  3. 번쩍하는 PBO 신호가 흐르더니, 숨겨져 있던 `p_secret` 입력 슬롯이 스르륵 모습을 드러내며 활성화되는 동적 렌더링을 확인한다.
- **상태 및 데이터**:
  - `MODIFY SCREEN 문장을 주석 처리한 상태` -> 속성 gv_screen-active가 0으로 바뀌었음에도 화면 상에서는 p_secret 창이 숨겨지지 않고 뻔뻔하게 계속 노출되는 오작동 하이라이트.
- **피드백**: 화면 제어는 루프 안에서 수정한 뒤 반드시 MODIFY SCREEN으로 쾅 인장을 찍어야 최종 렌더링에 반영됨을 전달한다.

### 실수/주의

- **OUTPUT 이벤트 안에서 변수 값 기본값 초기화**: "화면을 그리는 시점이니까 여기서 기본값을 대입해 채워야지" 하고 `p_secret = '1234'.`를 적어버리면, 사용자가 화면에서 값을 수정하고 엔터를 칠 때마다 PBO가 매번 돌며 사용자가 입력한 소중한 값을 다시 기본값으로 덮어써 날려버리는 업무 마비 버그가 나므로, 기본값 대입은 반드시 `INITIALIZATION`에 유치해야 한다.

### 정리

- **`AT SELECTION-SCREEN OUTPUT`** 은 화면 출력 직전인 **PBO** 시점마다 돌며 화면을 성형하는 곳이다.
- `LOOP AT SCREEN INTO gs_screen` 안에서 속성을 바꾼 후, **반드시 `MODIFY SCREEN`** 을 쳐야 갱신된다.
- 비교할 화면 오브젝트 이름은 반드시 **대문자(Uppercase)** 로 매칭하여 설계한다.

## CH15-L04 - AT SELECTION-SCREEN 입력 검증

### 왜 필요한가

사용자가 선택화면에 날짜나 코드 값을 적고 신나게 [실행 (F8)] 단추를 눌렀습니다. 
그런데 사용자가 실수로 "시작 날짜를 `2026-07-30`으로 적고 종료 날짜를 `2026-06-30`"으로 뒤집어서 기입했습니다. 
프로그램이 아무 생각 없이 이 모순된 데이터 조건 그대로 무거운 데이터베이스 SELECT 쿼리를 날려버리면, 서버 리소스만 허비하고 "조회 결과 0건" 이라는 차가운 화면만 남긴 채 사용자는 왜 조회가 안 되었는지 알 길이 없어 헤매게 된다. 

본격적으로 무거운 DB 처리를 시작하기 직전(PAI), **사용자가 입력한 조건들의 정합성을 꼼꼼하게 검문하여 모순이나 오타가 발견되면 즉시 경고등을 켜고 실행을 중단시켜 화면으로 돌려보내는 게이트 키퍼 검증 장치**가 필요하다. 그것이 **[[AT SELECTION-SCREEN]]** (입력 검증 시점)이다.

### 무엇인가

#### 1. AT SELECTION-SCREEN (입력 검증 - PAI)의 정의
- **정의**: 선택화면에서 사용자가 실행(F8)이나 엔터(Enter)를 치는 찰나인 **PAI (Process After Input - 입력 후 처리)** 시점에 즉시 가동되는 입력 정합성 검증 이벤트다.
- **동작**: 이 구획 내부에서 `MESSAGE ... TYPE 'E'` (Error)를 트리거하면, 시스템은 즉시 후속 조회 단계로 넘어가는 동작을 차단하고 선택화면으로 사용자를 강제 복귀(Rollback) 시킨다.

#### 2. 2대 검증 기어: 전체 검증 vs 개별 필드 검증 (ON <필드>) 차이
에러가 났을 때 화면을 통제하는 방식에 따라 두 가지 키워드로 명확히 쪼개진다.

- **전체 검증 (`AT SELECTION-SCREEN` 단독)**:
  - **시점**: 모든 필드 입력이 완전히 끝나고 마지막에 단 1회 실행된다.
  - **오류 발생 시**: 화면의 **모든 입력 필드**가 다시 입력 가능한 열린 상태로 복귀한다.
  - **용도**: "시작일이 종료일보다 늦으면 안 됨" 처럼 **여러 필드를 동시에 대조하는 조합 검증**에 최적이다.
- **개별 필드 검증 (`AT SELECTION-SCREEN ON <필드>` 콕 집어 검증)**:
  - **시점**: 전체 검증보다 **먼저** 호출되며, 해당 필드 데이터가 들어오는 순간 핀포인트로 실행된다.
  - **오류 발생 시 [필드 잠금의 특권]**: 오직 **에러가 난 그 필드 상자 하나만 하이라이트 입력 가능으로 열어두고, 나머지 멀쩡한 다른 필드들은 전부 회색(Read-Only)으로 얼려 잠가 버린다.**
  - **용도**: "공연 코드는 비어있으면 안 됨" 처럼 **단일 필드 단독 형식 검증**에 최적이며, 사용자의 시선과 키보드 마우스를 오류 난 칸으로 핀포인트 집중시키는 강력한 UX를 제공한다.
  - *제약*: 라디오 버튼 멤버 필드는 개 개별 `ON para`로 묶을 수 없으며, 반드시 `ON RADIOBUTTON GROUP group`으로 묶어 검증해야 하는 문법 제한이 있다.

#### 3. MESSAGE (메시지) 타입과 매핑 규칙
- 메시지를 띄울 때는 타입에 따라 프로그램 목숨을 가른다.
  - `E` (Error): 처리를 중단하고 입력 화면으로 되돌려 **막음**.
  - `I` (Information): 팝업창을 띄워 알려주지만, 닫으면 조회를 **계속 진행시킴**. (차단 실패).
  - `S` (Status): 하단 바에 조용히 성공 메시지를 띄우며 조회를 **계속 진행시킴**.
- **SE91 메시지 클래스**: 매번 하드코딩 문자를 쓰지 않고 `MESSAGE e001(zmsg)` 형태로 번호 재사용. `WITH` 옵션을 쓰면 메시지 내부의 `&1`~`&4` 자리에 변수 값이 순서대로 쏙쏙 장착되어 출력된다.

### 어떻게 확인하는가

에디터에서 두 종류의 검증 이벤트와 메시지 매핑을 설계해 정상 차단되는지 검증한다.

```abap
REPORT zch15_l04_screen_validation.

PARAMETERS: p_conc TYPE zconcert-concert_id,
            p_from TYPE d,
            p_to   TYPE d.

" 1. 단일 필드 핀포인트 검증! 오류 시 오직 p_conc 필드만 열리고 나머지는 회색 잠금!
AT SELECTION-SCREEN ON p_conc.
  IF p_conc IS INITIAL.
    MESSAGE e001(zmsg). " "공연 코드를 입력하세요" 메시지 출력 후 잠금 가드!
  ENDIF.

" 2. 여러 필드의 모순 비교 조합 검증! 오류 시 전체 필드가 다시 열림!
AT SELECTION-SCREEN.
  IF p_from > p_to.
    MESSAGE e002(zmsg) WITH p_from p_to. " "시작일 &1 이 종료일 &2 보다 늦습니다"
  ENDIF.
```

#### 체험/시뮬레이터 설계 (검증 게이트와 메시지 콘솔)
- **프로세스 플로우**:
  1. 학습자가 [p_conc] 칸을 빈칸으로 두고 실행(F8)을 누른다.
  2. `AT SELECTION-SCREEN ON p_conc` 검문관이 호루라기를 분다.
  3. [p_conc] 박스만 붉은색 테두리로 활성화되고, [p_from]과 [p_to] 입력 박스는 철컥 회색으로 질려 잠기면서 아래 상태 바에 "공연 코드를 입력하세요" 경고가 뜨는 하이라이트 집중 모션을 관찰한다.
  4. 날짜를 `from 7월 5일, to 7월 1일` 로 어긋나게 치고 실행한다. 전체 검증 `AT SELECTION-SCREEN`이 감지해 "시작일 7월 5일이 종료일 7월 1일보다 늦습니다" 라고 WITH 변수명이 한글 날짜로 실시간 변환 매핑되어 전체 필드 복귀 모션으로 막아내는 피드백을 본다.
- **상태 및 데이터**:
  - `MESSAGE 타입을 E가 아닌 I로 둔 상태` -> 팝업 경고가 뜨지만 [확인]을 누르면 조회가 차단되지 않고 START-OF-SELECTION으로 폭주해 넘어가 버리는 제어 실패 연출.
- **피드백**: 잘못된 입력은 반드시 PAI 검증 단에서 에러(E) 메시지로 강력히 차단 억류해야 데이터 정합성이 사수됨을 알려준다.

### 실수/주의

- **검증 시점에 'I' (Information) 나 'S' (Status) 타입 메시지 쓰기**: "경고창을 띄워줬으니 사용자가 알아채고 멈추겠지" 하는 것은 완전한 오해다. `I` 나 `S` 메시지는 화면에 글자만 뿌릴 뿐, 프로그램을 멈추지 않고 후속 `START-OF-SELECTION` 조회를 그대로 뚫고 지나가 버리므로, **강제 복구 차단은 오직 'E' (Error) 타입만 가능**함을 명심한다.

### 정리

- **`AT SELECTION-SCREEN`** 은 사용자가 실행을 클릭한 직후인 **PAI** 단계에서 값의 정합성을 수호하는 검문소다.
- **`ON <field>`** 검증은 단일 필드용이며, 에러 시 **해당 입력 상자만 살리고 나머지는 잠그는** 핀포인트 UX를 제공한다.
- 강제 차단은 **반드시 메시지 TYPE 'E'** 를 사용해야 뚫리지 않는다.

## CH15-L05 - START-OF-SELECTION 조회 실행

### 왜 필요한가

우리가 화면에 초기 제안 기본값도 잘 채웠고, 동적 숨김 제어도 마쳤으며, 시작일과 종료일 모순 검증 검문까지 무사히 통과했다. 
이제 모든 보안 검문과 형식 검사가 안전하게 완료되었으니, **진짜 본격적으로 데이터베이스 디스크 창고의 문을 열고 들어가서 예약 데이터를 SELECT 쿼리로 왕창 긁어모아 오고 가공 연산을 가동해야 하는 본 게임 시점**이 와야 한다. 

이 본 처리를 화면을 그릴 때마다 도는 OUTPUT 이벤트 방에 넣어두면 사용자가 클릭하지도 않았는데 서버가 계속 DB를 조회하는 리소스 부하를 겪고, INITIALIZATION에 넣어두면 사용자가 조건을 칠 기회도 없이 빈 조건으로 선조회를 해버린다. 
**선택 화면 처리(입력, 검증)가 100% 무사히 끝난 직후에만 단 한 번 신호탄을 쏘아 올려 무거운 비즈니스 로직과 DB 쿼리를 안전하게 폭발시키는 본 처리 전용 방**이 필요하다. 그것이 **[[START-OF-SELECTION]]** (조회 실행 시점)이다.

### 무엇인가

#### 1. START-OF-SELECTION (본 처리 시작점)의 정의
- **정의**: 선택화면 PAI 검증 단계를 무사통과한 합격 조건들만 유입되어, **실제 메인 비즈니스 로직(DB SELECT, 루프 계산, 가공 등)을 집행하는 표준 본처리 이벤트 블록**이다.
- **위치**: 모든 화면 입력 매커니즘이 종료되고, 내부 파라미터 변수(`p_conc`, `s_stat`)에 실물 데이터가 100% 안착하여 대기 중인 상태에서 호출된다.
- **묵시적 규칙**: 키워드가 없는 모든 평문의 데이터 가공 실행문은 아밥 시스템이 묵시적으로 이 `START-OF-SELECTION` 구획에 포함된 것으로 처리한다.

#### 2. 조회 결과 처리와 sy-subrc 분기
- DB에서 SELECT 쿼리를 수납해 온 뒤, 결과물이 0건인지 아닌지에 따라 후속 흐름을 명확하게 쪼개야 한다.
- **결과가 0건일 때 (`sy-subrc <> 0` 혹은 `gt_booking IS INITIAL`)**: 화면을 빈 화면으로 넘겨 덤프를 내지 않고, 하단에 잔잔하게 **`MESSAGE ... TYPE 'S'`** (조회된 예매가 없습니다) 상태바를 조용히 인쇄하며 즉시 프로그램을 종료(`RETURN`)해 방어한다.
- **결과가 존재할 때 (`sy-subrc = 0`)**: 내부 테이블에 담긴 레코드를 돌며 화면에 쓰거나(`WRITE`), [CH11]에서 정복한 SALV 뷰어 객체를 출산해 화면에 뿌려준다.

### 어떻게 확인하는가

검증을 통과한 조건으로 DB를 조회해 안전하게 출력하는 정석 본처리 소스를 검증한다.

```abap
REPORT zch15_l05_selection_run.

DATA gv_status TYPE zbooking-status.
PARAMETERS p_conc TYPE zbooking-concert_id.
SELECT-OPTIONS s_stat FOR gv_status.

DATA: gt_booking TYPE STANDARD TABLE OF zbooking,
      gs_booking TYPE zbooking.

START-OF-SELECTION.
  " 1. 검증이 끝난 진짜 알짜배기 조건으로만 DB 조회 집행!
  SELECT * FROM zbooking
    INTO TABLE gt_booking
    WHERE concert_id = p_conc
      AND status     IN s_stat.

  " 2. 결과 유무 분기 철칙! 0건이면 팝업으로 막지 않고 상태바(S)로 흘리기!
  IF sy-subrc <> 0.
    MESSAGE '조회된 예매 정보가 없습니다' TYPE 'S'.
    RETURN. " 무의미한 루프나 화면 출력을 타지 않게 즉시 복귀 탈출!
  ENDIF.

  " 3. 조회 성공 시 안전한 출력 루프 작동!
  LOOP AT gt_booking INTO gs_booking.
    WRITE: / gs_booking-booking_id, gs_booking-customer_name.
  ENDLOOP.
```

#### 체험/시뮬레이터 설계 (조회 시작 게이트)
- **프로세스 플로우**:
  1. 학습자가 [실행]을 누른다. 데이터 쿼리 기차가 검증 게이트를 하이패스로 통과한다.
  2. 기차가 `START-OF-SELECTION` 본처리 가공 공장에 들어서자, 디스크 아이콘에서 다량의 예약 로우 상자들이 컨베이어 벨트를 타고 올라와 내부 테이블 `gt_booking`으로 쏟아지는 비주얼을 본다.
  3. `sy-subrc = 0` 신호가 파란 불로 켜지고, 목록 판에 예약번호와 고객명 **"정훈영"** 이 나란히 출력 완료되는 전체 유통 단계를 확인한다.
- **상태 및 데이터**:
  - `조건에 매칭되는 데이터가 없는 조건 입력` -> sy-subrc <> 0 빨간 불이 들어오고, 에러 화면으로 폭발 덤프를 내는 대신 하단 상태 바에 "조회된 예매가 없습니다"가 인쇄되며 조용히 리포트가 안전 종료되는 방어막 작동 피드백.
- **피드백**: 무거운 DB 조작과 결과 분기는 철저하게 앞단의 보안/형식 검증을 다 뚫고 온 뒤인 START-OF-SELECTION에서 평화롭게 시작해야 함을 알려준다.

### 실수/주의

- **화면 속성 제어(MODIFY SCREEN 등)를 여기서 시도**: "조회해 보고 모드가 틀리면 여기서 입력 필드를 숨겨야지" 하고 본처리 방에 적어봤자, 화면은 이미 과거 속성을 타서 굳어있고 본처리가 돌고 있는 중이라 아무 영향도 미치지 못하는 헛발질 코딩이 되므로, 화면 제어는 무조건 `OUTPUT` 시점으로 이식한다.

### 정리

- **`START-OF-SELECTION`** 은 모든 검증을 무사히 통과한 청정 조건으로 본격 DB 조회와 연산을 집행하는 본처리 방이다.
- 쿼리 완료 후 **반드시 `sy-subrc` 나 `IS INITIAL` 분기**를 타서 0건일 때의 방어막(Return)을 세운다.
- 묵시적 평문들은 다 여기에 묶이므로, 다른 이벤트 방과 물리 공간을 명확히 쪼개서 관리한다.

## CH15-L06 - END-OF-SELECTION의 위치와 경계

### 왜 필요한가

오래된 선배들의 ABAP 레거시 리포트 프로그램을 열어보면, 짠 듯이 `START-OF-SELECTION` 아래에 조회 쿼리를 두고, 그 아래에 다시 **`END-OF-SELECTION.`** 이라는 키워드를 개설한 뒤 그 안에서 `WRITE` 나 출력을 구동하는 구조가 10개 중 9개 꼴로 사방에 널려있다. 
이걸 본 아바 초학자들은 "아하! SAP 리포트를 짤 때는 조회를 스타트에 쓰고, 화면 출력은 무조건 엔드 오브 셀렉션이라는 방에 따로 나눠서 넣는 게 공식이구나!" 라고 철석같이 오해해, 굳이 안 써도 될 이벤트를 억지로 끼워 넣는 코딩 낭비를 자초하곤 한다. 

**이 이벤트의 정체가 사실은 옛날 SAP 시스템이 쓰던 Logical Database(LDB)라는 낡은 유물 수혈 장치에서 비롯된 것임을 명확하게 규명하고, 내가 만드는 현대의 일반적인 SELECT 리포트에서는 이 방을 굳이 개설할 필요가 전혀 없음을 명확하게 선 긋는 아키텍처적 격리 기준**이 필요하다. 그것이 **[[END-OF-SELECTION]]** 의 올바른 경계선 획정이다.

### 무엇인가

#### 1. END-OF-SELECTION (마무리 시점)의 역사적 기원
- **기원**: 옛날 SAP는 DB에서 직접 데이터를 SELECT 하지 않고, SAP가 미리 정의해 둔 거대한 데이터 공급 테이블 묶음인 **`Logical Database` (LDB - 논리 데이터베이스)** 를 링크해서 화면을 빌드했다.
- **과거의 거동**: LDB 프레임워크가 데이터를 다 긁어모아 공급하는 작업을 안전하게 완료(End)하고 물러가면, 바통을 이어받아 화면에 출력하기 위해 시스템이 약속대로 노크해 주던 마감 시점 이벤트가 바로 `END-OF-SELECTION` 이었다.

#### ⚠️ [현대적 Obsolete (낡은 문법) 분류 및 무용성]
- **현대적 기준**: 현대의 모든 리포트 프로그램은 LDB를 쳐다보지 않고, 우리가 직접 `SELECT ... FROM ... INTO TABLE` 문장으로 다이렉트 쿼리를 날려 수납한다.
- **불필요성**: LDB를 쓰지 않는 신규 리포트 프로그램에서는 **`END-OF-SELECTION` 블록을 구현할 가치가 0.001%도 없다.** 
- **거동**: 이 블록을 안 적어도 어차피 `START-OF-SELECTION` 구획의 코드가 다 실행된 직후 순차적으로 자연스럽게 출력이 마무리되기 때문이다. 공식 키워드 문서에서도 이를 명백히 낡은 Obsolete 계열로 못 박아 두고 있다.

#### 2. 레거시 독해 관점의 유효성
- 비록 신규 개발에는 절대 쓰지 말아야 할 대상이지만, 회사 내 15년 전에 짜여 가동 중인 레거시 소스에는 여전히 단골로 나오므로 "아하, 이건 옛날 LDB 흔적이거나 단순 조회/출력의 시각적 분할을 위해 억지로 적어둔 낡은 패턴이구나" 라고 **읽어내고 해독하는(Read-only) 독해 무기**로만 머릿속에 이식해 둔다.

### 어떻게 확인하는가

LDB가 없는 리포트에서는 `END-OF-SELECTION`을 생략하고 `START-OF-SELECTION` 한 방으로 조회와 SALV 렌더링 출력을 유기적으로 종결하는 클린 아키텍처를 확인한다.

```abap
REPORT zch15_l06_no_end_event.

PARAMETERS p_conc TYPE zbooking-concert_id.
DATA gt_booking TYPE STANDARD TABLE OF zbooking.
DATA lo_alv     TYPE REF TO cl_salv_table.

START-OF-SELECTION.
  " 1. 본 처리 조회
  SELECT * FROM zbooking
    INTO TABLE gt_booking
    WHERE concert_id = p_conc.

  " 2. LDB가 없는 현대적 리포트이므로, END-OF-SELECTION 구획을 굳이 개설하지 않고
  " START-OF-SELECTION의 꽁무니에서 곧바로 SALV 객체를 안전하게 display 종결!
  IF gt_booking IS NOT INITIAL.
    TRY.
        cl_salv_table=>factory(
          IMPORTING r_salv_table = lo_alv
          CHANGING  t_table      = gt_booking ).
        lo_alv->display( ).
      CATCH cx_salv_msg.
        MESSAGE '표시 오류' TYPE 'S'.
    ENDTRY.
  ENDIF.
```

#### 체험/시뮬레이터 설계 (기존 리포트 독해 모드)
- **프로세스 플로우**:
  1. 학습자가 [1. LDB 연결 레거시 소스] 카드를 본다. LDB 시스템이 루프를 돌며 데이터를 수혈하고 `END-OF-SELECTION`에 와서야 깃발을 흔들며 최종 WRITE 출력이 동작하는 구식 흐름을 본다.
  2. [2. 현대식 다이렉트 SELECT 소스] 카드를 누른다. 
  3. `END-OF-SELECTION` 블록이 아예 삭제된 상태에서 `START-OF-SELECTION` 한 칸 안에서 기차가 출발해 DB 조회와 SALV 출력을 막힘없이 단번에 끝마치는 수려한 단층 흐름을 감상한다.
- **상태 및 데이터**:
  - `신규 리포트 소스코드 중간에 END-OF-SELECTION 강제 삽입` -> "Obsolete event block in non-LDB context" 주의 알림 딱지 하이라이트 노출.
- **피드백**: "END-OF-SELECTION은 신규 개발의 의무가 아니라, 오직 지나간 옛 소스를 읽기 위해 장착하는 돋보기일 뿐" 임을 못 박아 전달한다.

### 실수/주의

- **신규 프로그램에 의무적으로 END-OF-SELECTION 적기**: "출력 코드는 마무리에 있어야 멋있으니까 엔드를 써야지" 했다가, 불필요하게 전역 변수 공유 생명주기만 길어지고 소스 코드가 산발되어 가독성을 떨어뜨리므로, 신규 리포트에는 **`START-OF-SELECTION` 내부에서 일괄 종결**하는 정석을 따른다.

### 정리

- **`END-OF-SELECTION`** 은 과거 LDB(Logical Database)와 연동하던 시절에나 의무로 쓰던 유물 마감 이벤트다.
- 현대의 일반 SELECT 리포트에서는 이 이벤트를 **절대 개설해 구현할 필요가 없다 (Obsolete).**
- 기존 프로젝트의 레거시 소스 분석 시 "조회와 출력을 시각적으로 쪼개놓은 옛날 흔적" 으로 읽어내는 용도로만 활용한다.

## CH15-L07 - Selection Screen 권한/존재 여부 검증 기초

### 왜 필요한가

사용자가 선택화면에 존재하지도 않는 공연 코드 `C999` 를 기입하고 실행을 눌렀을 때, 프로그램이 "존재하지 않는 공연입니다" 라고 막아주는 가드는 당연하다. 
그런데 만약, 실제로 데이터베이스에 멀쩡히 들어있는 대외비 VIP 특별 공연 코드인 `C002` 를 입력하고 실행을 눌렀는데, 프로그램이 그냥 똑같이 **"조회된 데이터가 존재하지 않습니다"** 라고 뭉뚱그려서 퉁쳐서 메시지를 내보내면 어떨까? 
정훈영 대리는 "어라? 어제 부장님이 등록했다고 하셨는데 왜 데이터가 없지? 내가 오타를 냈나?" 헷갈려하며 무의미하게 글자를 고쳐쳐서 재조회를 무한 반복하는 삽질 업무 병목을 겪게 된다. 
게다가 **권한이 없어서 조회가 막힌 보안 위배 상황과, 데이터 자체가 존재하지 않는 정합성 에러 상황을 명확하게 쪼개지 않으면, 타인의 대외비 정보를 훔쳐보려는 위험한 보안 침해 시도를 걸러내어 모니터링할 도리가 아예 사라진다.** 

**실제 존재하는 코드인지 검사하는 '존재 검문소'와, 사용자의 계정(ID) 권한을 직접 조회해 튕겨내는 '보안 검문소'를 2단계로 확실하게 격리하여 검증하는 이중문 가드 장치**가 필요하다. 그것이 **존재 검증과 권한 검증([[AUTHORITY-CHECK]])의 분리 설계**다.

### 무엇인가

#### 1. 1차 관문: 존재 검증 (`SELECT SINGLE`)
- **목적**: 사용자가 입력한 키가 실제로 데이터베이스 테이블에 존재하는 실존 데이터인지 무결성을 입증한다.
- **구현**: `SELECT SINGLE` 조회를 던져 `sy-subrc = 0` 이면 합격, 아니면 **"없는 코드입니다 (E)"** 에러를 뿜어 리포트를 잠근다.
- **위치**: 필드 하나만 콕 집어 검증해야 하므로 `AT SELECTION-SCREEN ON <필드>` 방에 가둔다.

#### 2. 2차 관문: 권한 검증 (`AUTHORITY-CHECK`)
- **목적**: 존재하는 데이터일지라도, 현재 로그인한 사용자가 그 값을 열어볼 수 있는 안전한 사람인지 보안 등급을 체크한다.
- **구현**: SAP 표준 보안 명령인 **`AUTHORITY-CHECK OBJECT`** 문장을 사용하여, 해당 권한 객체(Authorization Object)의 액티비티(ACTVT = '03' 조회) 권한이 계정에 박혀있는지 스캔한다.
- **결과 분기 (`sy-subrc`)**:
  - `0`: 통과 (권한 적합).
  - `4`: 권한은 있으나 **그 특정 값(예: 대외비 등)에 대한 세부 권한이 부족**해 탈락.
  - `12`: 해당 권한 객체 자체를 계정에 아예 부여받지 못한 **보안 미달자**로 탈락.
  - *처리*: `sy-subrc <> 0` 이 감지되면 즉시 **"이 공연을 볼 권한이 없습니다 (E)"** 라는 강력한 보안 경고 메시지로 차단해 되돌려 보낸다.

### 어떻게 확인하는가

에디터에서 존재 검증과 권한 검증 이중문을 순차 배치하고 입력 오류를 각각 쪼개어 가두는 소스를 검증한다.

```abap
REPORT zch15_l07_auth_check.

PARAMETERS p_conc TYPE zconcert-concert_id.
DATA gv_concert_id TYPE zconcert-concert_id.

AT SELECTION-SCREEN ON p_conc.
  " [1차 존재 검문] 실존하는 데이터인지 DB에 노크!
  SELECT SINGLE concert_id FROM zconcert
    INTO gv_concert_id
    WHERE concert_id = p_conc.
  IF sy-subrc <> 0.
    MESSAGE e003(zmsg) WITH p_conc. " "없는 공연 코드입니다: &1" 로 튕기기!
  ENDIF.

  " [2차 보안 검문] 존재 무결성이 검증된 값을 볼 권한이 계정에 있는지 대조!
  AUTHORITY-CHECK OBJECT 'Z_CONCERT'
    ID 'CONCERT' FIELD p_conc
    ID 'ACTVT'   FIELD '03'. " 03 = 조회 권한 체크
  IF sy-subrc <> 0.
    MESSAGE e004(zmsg) WITH p_conc. " "이 공연을 볼 권한이 없습니다: &1" 로 튕기기!
  ENDIF.
```

#### 체험/시뮬레이터 설계 (존재·권한 이중문)
- **프로세스 플로우**:
  1. 학습자가 `C999` (없는 공연)을 치고 F8을 클릭한다.
  2. 1차 존재 검문소 셔터가 빨간색으로 내려앉으며 "없는 공연 코드입니다" 에러 팝업이 터진다.
  3. 실존하지만 권한이 없는 `C002` 를 치고 실행한다. 1차 존재 검문소는 무난히 통과하지만, 2차 권한 검문소의 열쇠 구멍에 빨간불이 깜빡이며 "이 공연을 볼 권한이 없습니다" 자물쇠가 철컥 잠기는 2단계 연동 모션을 체감한다.
- **상태 및 데이터**:
  - `AUTHORITY-CHECK 호출 후 sy-subrc 판단 분기를 생략한 상태` -> 권한 체크 결과 sy-subrc가 4나 12가 나왔음에도 무사히 통과되어 대외비 데이터가 화면에 출력 유출되어 버리는 심각한 보안 붕괴 데모 노출.
- **피드백**: 보안 체크는 명령어를 치는 것보다 `sy-subrc <> 0` 분기문으로 에러(E) 메시지를 터트려 되돌리는 제어가 핵심임을 전달한다.

### 실수/주의

- **데이터가 없을 때 그냥 에러 메시지 없이 화면을 통과시키기**: "SELECT 해보고 데이터가 안 나오면 권한이 없는 거겠죠" 라고 대충 넘기면, 보안 구멍이 발생해 악의적인 사용자가 다른 공연 코드를 무작위로 때려 넣어 볼 수 있는 백도어가 되므로, 존재성과 권한은 반드시 PAI 진입 입구인 `AT SELECTION-SCREEN`에서 철저하게 분리 검문해야 한다.

### 정리

- **`존재 검증`** 은 `SELECT SINGLE` 조회를 던져 실존 키 여부를 가리는 업무 정합성 검사다.
- **`권한 검증`** 은 **`AUTHORITY-CHECK`** 로 로그인 ID의 권한 등급을 스캔하는 보안 검사다.
- 두 단계는 원인이 다르므로 반드시 **서로 다른 에러 메시지(E)를 부여**해 프로그램을 억류 설계한다.

## CH15-L08 - Selection Screen 고급 — 블록·그룹 검증과 커스텀 F1·F4

### 왜 필요한가

필드 하나가 올바른지 검사하는 `ON <필드>` 검증은 마스터했다. 
하지만 실제 현업의 화면은 더 입체적이다. 
"날짜 From과 To를 모아둔 '날짜 블록' 전체를 검사해서 둘 중 하나라도 비어있으면 막아야 하고, 라디오 버튼에서 '상세'를 골랐을 때만 비밀 일자 필드가 비어있지 않은지 정합성을 묶어서 검사해 주고 싶다" 
또한 사용자가 입력창에서 키보드 F1(도움말)을 누르면 나만의 특별한 설명창을 띄우고, F4(도움말)를 누르면 체크 테이블 사전 기준이 아니라 **"내가 프로그램 소스 코드 내부에서 직접 DB 쿼리로 가공해 조립한 팝업 리스트 목록"** 을 화면에 짠 보여주고 싶을 수 있다. 
**선택 화면이 단순한 static 텍스트 상자 더미를 넘어, 블록 묶음 단위로 검증하고 사용자 키보드(F1, F4) 모션에 다이내믹하게 팝업으로 즉각 대꾸하는 똑똑한 Dialog 화면 제어 장치**가 필요하다. 그것이 **블록/그룹 검증과 커스텀 도움말([[ON HELP-REQUEST]]/[[ON VALUE-REQUEST]])** 이다.

### 무엇인가

#### 1. 블록 및 라디오 배타 그룹 검증
- **`AT SELECTION-SCREEN ON BLOCK <블록명>`**:
  - **역할**: 선언해 둔 블록 전체 영역을 한 단위로 감시한다. 블록 내의 From/To 날짜가 서로 순서가 뒤집혔거나, 필수 쌍이 깨졌는지의 조합 검증을 단번에 수행한다.
- **`AT SELECTION-SCREEN ON RADIOBUTTON GROUP <그룹명>`**:
  - **역할**: 배타적인 라디오 그룹의 어떤 기어 버튼이 켜졌는지에 따라, 다른 입력창들의 공백 정합성을 묶어서 단숨에 판정한다.

#### 2. 코드로 직접 빚는 도움말 F1 & F4
- **F1 도움말 커스텀 (`AT SELECTION-SCREEN ON HELP-REQUEST FOR <필드>`)**:
  - 해당 입력 필드 위에서 키보드 `F1` 키를 클릭하면 트리거되는 시점이다. 데이터 엘리먼트 문서 외에 개발자가 직접 상세 도움 정보를 팝업 메시지로 출력할 때 애용한다.
- **F4 리스트 커스텀 (`AT SELECTION-SCREEN ON VALUE-REQUEST FOR <필드>`)**:
  - 해당 입력 필드에서 `F4` 키를 누르는 찰나(POV)에 동적 DB 조회를 가동해 팝업 리스트를 뿌려준다.
  - **구현**: 아바 내장 함수인 **`F4IF_INT_TABLE_VALUE_REQUEST`** 를 호출하며, 내가 쿼리해 둔 내부 테이블(`value_tab`)을 넘기고, 사용자가 고른 값이 어느 화면 필드(`dynprofield`)로 복귀 반사될지를 선언한다.

#### ⚠️ [POH/POV 시점의 화면 값 자동 복사 불가능 사양 제약]
- *아밥 개발자들이 가장 많이 걸려 넘어지는 복잡 도움말 함정이다.*
- 사용자가 F4 나 F1 도움말 키를 클릭하는 찰나(POH/POV)에는, **화면에 쳐져 있는 다른 입력창들의 타이핑 값이 아밥 프로그램 내부 변수 테이블 공간으로 자동으로 복사 전송(Transport)되지 않는다.**
- 따라서 만약 "상위 입력창에 적힌 '서울'이라는 값을 보고 F4 팝업 리스트를 동적으로 걸러서 보여주고 싶다" 고 그냥 쿼리를 짜면, 서울이라는 값이 변수에 복사되지 않아 텅 빈 F4가 나와 버린다.
- 이 경우 현재 화면 값을 강제로 읽어 들이는 특수 API 함수인 `DYNP_VALUES_READ` 등을 명시적으로 결합 호출해 수동으로 값을 수혈해 가져오는 고급 연동이 필수적이다.

### 어떻게 확인하는가

F4IF 함수를 활용하여, 소스 상에서 즉석 빚어낸 내부 테이블 목록을 화면 필드에 복귀시키는 커스텀 POV 구성을 검증한다.

```abap
REPORT zch15_l08_advanced_screen.

PARAMETERS p_carr TYPE scarr-carrid.

" F4 팝업창에 채워줄 보조 구조체 및 내부 테이블 선언!
DATA: BEGIN OF gs_carr,
        carrid   TYPE scarr-carrid,
        carrname TYPE scarr-carrname,
      END OF gs_carr.
DATA gt_carr LIKE STANDARD TABLE OF gs_carr.

" F4 버튼 클릭 찰나(Value Request)를 핀포인트 제어!
AT SELECTION-SCREEN ON VALUE-REQUEST FOR p_carr.
  " 1. DB에서 리스트 목록 고속 쿼리 수납!
  SELECT carrid carrname FROM scarr
    INTO TABLE gt_carr.

  " 2. SAP 표준 F4 팝업 서빙 함수 가동!
  CALL FUNCTION 'F4IF_INT_TABLE_VALUE_REQUEST'
    EXPORTING
      retfield    = 'CARRID'     " 선택 행에서 이 컬럼 값을 적출!
      dynpprog    = sy-repid
      dynpnr      = sy-dynnr
      dynprofield = 'P_CARR'     " 최종적으로 이 화면 변수에 쏙 배달!
    TABLES
      value_tab   = gt_carr.     " 화면 팝업 그리드에 띄울 원천 데이터!
```

#### 체험/시뮬레이터 설계 (Selection Screen 반응 실험실)
- **프로세스 플로우**:
  1. 학습자가 [시작일: 텅 빔 | 종료일: 텅 빔] 상태로 두고 실행한다. 
  2. `ON BLOCK b_date` 방의 검문관이 작동하여 블록 영역 전체가 빨간색 상자로 쾅 감싸지며 "기간을 모두 입력하세요" 에러가 뜬다.
  3. [p_carr] 필드 우측 돋보기(F4)를 누른다. 번쩍하며 POV 기차가 DB에서 항공사명 "정훈영 항공"을 셀렉트해 가져와 임시 팝업 표에 렌더링하고, 더블클릭하자 화면 p_carr 창에 `JY` 코드가 자동 기입 복귀하는 과정을 본다.
- **상태 및 데이터**:
  - `F4IF retfield 명칭에 대문자 'CARRID'가 아닌 소문자 'carrid'를 기입한 상태` -> 사용자가 더블클릭해도 화면 필드에 아무 값도 돌아오지 않고 멍하니 있는 매핑 단절 현상 연출.
- **피드백**: 커스텀 F4는 테이블 원천 컬럼명과 타깃 필드명칭을 명확하게 파라미터 매핑해주어야 데이터 순환 벨트가 작동함을 인지시킨다.

### 실수/주의

- **Search Help가 이미 걸려있는데 굳이 커스텀 F4 함수 코딩하기**: "코딩하는 게 제일 확실하니까 그냥 다 F4IF 함수를 써야지" 했다가, 불필요한 DB SELECT 부하와 수백 줄의 유지보수 스파게티 코드가 생겨 낭패를 겪는다. F4는 **가급적 데이터 사전(DDIC)에 선언적으로 Search Help를 박아두는 것이 표준**이며, 오직 팝업 목록이 매번 상황에 따라 유동적으로 요동치는 극한의 상황에서만 이 커스텀 POV 함수를 개설해야 한다.

### 정리

- **`ON BLOCK`** 과 **`ON RADIOBUTTON GROUP`** 은 필드 여러 개의 배타 조건 및 조합 정합성을 묶어 감시하는 검문소다.
- **`ON HELP-REQUEST`** 와 **`ON VALUE-REQUEST`** 는 F1/F4 키보드 액션 시 코드로 즉석 응대 팝업을 빚어주는 반응 지점이다.
- 도움말 시점에는 **화면 타이핑 값이 자동으로 전송되지 않는** 사양 한계를 기억하고 동적 연동을 꾀한다.

## CH15-L09 - Selection Screen UI 구성

### 왜 필요한가

우리가 PARAMETERS와 SELECT-OPTIONS를 소스 코드에 나열하여 화면에 입력창들을 배치했다. 
하지만 필드가 15개가 되는 대형 리포트 화면을 그냥 쌩으로 열어놓으면, 사용자는 "이 중에서 뭐가 필수이고, 뭐와 뭐가 같은 그룹 조건인지" 눈이 사방으로 찢어져서 피로를 느끼고, 엉뚱한 값을 조합해 조회 실패를 반복하게 된다. 

마치 보기 좋은 가구 카탈로그나 서류 신청서 양식처럼, **관련 조건들을 네모난 외곽 테두리 프레임(Block)으로 단정하게 모으고, 중간중간 자로 긋듯 가로 구분선(ULINE)을 긋고, 각 입력 칸 옆에는 한글 설명 이름표(Comment)를 이쁘게 부착하여, 사용자가 한눈에 화면의 레이아웃 구조를 3초 만에 꿰뚫어 볼 수 있게 돕는 사용자 중심 UI 디렉션 기술**이 필요하다. 그것이 **[[SELECTION-SCREEN]] UI 빌더** 이다.

### 무엇인가

#### 1. 테두리 블록과 설명 텍스트
- **`SELECTION-SCREEN BEGIN OF BLOCK <블록명> WITH FRAME TITLE <텍스트>`**:
  - **동작**: 네모난 테두리 선을 긋고 그 외곽선 상단에 한글 제목을 예쁘게 인쇄하여 한 세트로 가둔다.
- **`SELECTION-SCREEN COMMENT /1(50) <텍스트> FOR FIELD <필드명>`**:
  - **동작**: 화면 상에 보조 한글 설명 텍스트 라벨을 출력한다.
  - **[FOR FIELD 매핑 필수 규칙]**: 주석문 뒤에 반드시 **`FOR FIELD`** 옵션을 통해 타깃 입력창을 선언적으로 결합해주어야 한다. 이 관계가 박혀있어야 사용자가 해당 한글 설명 텍스트 라벨을 마우스로 클릭하고 F1을 눌렀을 때도 타깃 입력창의 F1 도움말이 정상 작동하는 SAP 표준 접근성 명세가 준수된다.

#### 2. 한 줄에 여러 요소 도킹하기 (`BEGIN OF LINE` ~ `END OF LINE`)
- 기본적으로 입력창들은 한 줄에 하나씩 수직으로 줄을 바꿔 가며 배치된다.
- From/To 처럼 한 가로 줄에 라벨 설명 글씨와 체크박스, 그리고 입력 필드를 징검다리처럼 나란히 정렬하려면 `BEGIN OF LINE` 블록을 열고 **`POSITION n`** (n번째 가로 열로 커서 이동) 지시어를 사용해 물리 위치를 콕콕 찍어서 가로로 배열한다.

#### 3. 선택화면 버튼 (PUSHBUTTON vs FUNCTION KEY)
선택화면에 장착할 수 있는 버튼은 가동 위치에 따라 2대 축으로 나뉜다.

- **본문 버튼 (`SELECTION-SCREEN PUSHBUTTON ... USER-COMMAND <코드>`)**:
  - 화면 정중앙 영역에 넓게 빚어내는 버튼이다.
- **툴바 버튼 (`SELECTION-SCREEN FUNCTION KEY <번호>`)**:
  - 화면 맨 위쪽 표준 툴바 라인에 단축 아이콘 버튼으로 매치해 준다.
- **⚠️ [sscrfields-ucomm 제어 규칙]**:
  - *초보 개발자들이 일반 Dynpro와 혼동해 백발백중 오작동을 터트리는 변수 차이다.*
  - **선택화면 안의 버튼들을 클릭했을 때 날아오는 고유 단축 코드(ucomm)는 시스템 변수인 `SY-UCOMM`에 안전하게 대입되지 않는 물리 사양이 존재한다.**
  - **반드시 `TABLES sscrfields.` 구조체 선언을 프로그램 상단에 미리 기입해 두고, `sscrfields-ucomm` 변수 방을 CASE 문으로 분기 검사**하여 이벤트를 처리해야 함을 명심한다.

### 어떻게 확인하는가

본문 푸시버튼을 개설하고 sscrfields 구조체를 선언하여 버튼 이벤트를 정확하게 처리하는 소스를 검증한다.

```abap
REPORT zch15_l09_screen_ui.

" [변수 선언 철칙] 선택화면 ucomm 가동을 위한 sscrfields 구조체 도킹 선언!
TABLES sscrfields.

" 1. 테두리 블록 개설!
SELECTION-SCREEN BEGIN OF BLOCK b_main WITH FRAME TITLE TEXT-001.
  PARAMETERS p_conc TYPE zconcert-concert_id.
  
  " [FOR FIELD 체결 철칙] 주석 라벨과 입력창 연결!
  SELECTION-SCREEN COMMENT /1(50) TEXT-002 FOR FIELD p_conc.
  SELECTION-SCREEN ULINE. " 가로 실선 긋기
  
  " 2. 본문 푸시버튼 장착! 클릭 시 'REF' 코드가 발송됨!
  SELECTION-SCREEN PUSHBUTTON /1(20) gv_btn USER-COMMAND ref.
SELECTION-SCREEN END OF BLOCK b_main.

INITIALIZATION.
  gv_btn = '조건 새로고침'. " 버튼 얼굴에 글씨 쓰기

AT SELECTION-SCREEN.
  " [UCOMM 대조 철칙] sy-ucomm이 아닌 sscrfields-ucomm을 분기 비교!
  CASE sscrfields-ucomm.
    WHEN 'REF'.
      MESSAGE '화면 입력 조건이 새로고침 되었습니다' TYPE 'S'.
  ENDCASE.
```

#### 체험/시뮬레이터 설계 (Selection Screen 레이아웃 빌더)
- **프로세스 플로우**:
  - 학습자가 화면 설계 보드에서 `PUSHBUTTON` 요소를 마우스로 클릭한다.
  - [클릭 이벤트 발생] 시 신호선이 `SY-UCOMM` 방으로 들어가려다 차단막에 막혀 튕겨 나가고, `SSCRFIELDS-UCOMM` 방의 스위치 보드로 안전하게 도킹되어 녹색등이 켜지는 유통 과정을 본다.
  - 화면 정중앙에 "새로고침 되었습니다" 상태 메시지가 뜨는 피드백을 본다.
- **상태 및 데이터**:
  - `SY-UCOMM 변수로 'REF'를 검사하게 코딩한 오작동 상태` -> 버튼을 백날 클릭해도 sy-ucomm 방이 텅 빈 공백이라 아무 반응을 보이지 않고 먹통이 되는 버그 시뮬레이션 표출.
- **피드백**: 선택화면의 버튼 조작 처리는 반드시 sscrfields 선언과 sscrfields-ucomm 비교 매커니즘을 타야만 맥박이 통한다는 점을 상기시킨다.

### 실수/주의

- **PUSHBUTTON을 메인 조회 실행 버튼 대용으로 오용**: "F8 실행 단추가 위에 있어 누르기 머니까 본문 화면 한가운데에 커다란 조회실행 버튼을 만들어서 조회를 돌려야지" 하고 푸시버튼 이벤트를 타서 무거운 SELECT 조회를 직접 호출하면 안 된다. 푸시버튼은 화면 조건 리셋이나 팝업 도움말 띄우기 등의 보조 제어용이며, **조회 실행은 무조건 SAP 표준 Execute(F8) 라인을 타서 START-OF-SELECTION으로 빠지게 설계하는 것이 표준 양식**이다.

### 정리

- **`SELECTION-SCREEN`** UI 빌더를 사용하여 테두리 블록을 묶고 구분선을 쳐서 화면을 구조화한다.
- `COMMENT` 설명 라벨에는 반드시 **`FOR FIELD`** 매핑을 붙여 F1 접근성 무결성을 챙긴다.
- 버튼 클릭 ucomm 코드는 **`sscrfields-ucomm`** 시스템 변수를 경유해 처리하므로 사전 도킹 선언을 수호한다.

## CH15-L10 - PARAMETERS · SELECT-OPTIONS 옵션 총정리

### 왜 필요한가

우리가 입력창을 만들 때, 사용자의 사소한 입력 실수를 막고 싶다. 
"이 필드는 필수 입력이게 하고, 소문자로 쳐도 그대로 저장되게 하고, 마지막에 쳤던 값을 기억하게 하고, From-To 사이 범위(Intervals) 입력은 막고 딱 단일 값만 치게 락을 걸고 싶다" 
이 요구사항들을 해결하겠다고 `AT SELECTION-SCREEN` 검증 방에 들어가서 "필드가 비어있으면 막아라", "소문자를 대문자로 변환해라", "범위가 들어오면 튕겨내라" 하고 수십 줄짜리 구차한 예외 코드 모듈을 키보드로 짜기 시작하면, 프로그램은 쓸데없이 무거워지고 가독성은 끔찍해진다. 

아바 소스 코드 정의 단에서 단 한 글자의 단축어 선언 키워드만 꼬리에 척 얹어주면, **코딩 한 줄 없이 아바 프레임워크가 필수 입력 검증과 대소문자 보존 및 화면 레이아웃 락을 선언적(Declarative)으로 한 방에 처리해 주는 강력한 기술 카탈로그 스위치**가 필요하다. 그것이 **PARAMETERS 및 SELECT-OPTIONS 옵션 패키지**다.

### 무엇인가

#### 1. PARAMETERS 핵심 옵션 명세
- **`OBLIGATORY`**: 해당 입력창을 필수 입력으로 선포한다. 비우고 실행하면 아밥이 화면 아래에 빨간 체크 표시를 띄우며 진입을 원천 방어한다.
- **`LOWER CASE` [문자형 전용 제약]**:
  - 아밥 화면은 기본적으로 입력을 받으면 소문자를 전부 강제로 대문자로 치환해 전송한다.
  - 이 변환을 막고 소문자를 살리기 위해 LOWER CASE를 얹는다.
  - **제약**: 이 옵션은 오직 **문자형(Character-like: `C`, `String` 등) 필드에만 유효**하며, 숫자형(`I`, `P` 등)이나 쌩뚱맞은 날짜형에 붙이면 사전 빌드 오류를 만나므로 엄격히 격리해 쓴다.
- **`MEMORY ID <pid>`**: 사용자가 마지막에 입력했던 값을 SAP 메모리 버퍼에 기억(Save)해 두었다가, 다음에 프로그램을 다시 켤 때 그 값을 자동으로 채워 복원해 준다.
- **`MODIF ID <그룹명>`**:
  - 화면의 특정 필드들에 하나의 그룹 이름표(예: `adv`)를 붙여준다.
  - 훗날 `OUTPUT` PBO 시점에 `LOOP AT SCREEN` 을 돌 때 `screen-group1 = 'ADV'` 대문자 이름으로 검색해 한꺼번에 숨기거나 여는 대량 일괄 통제가 가능해진다.

#### 2. SELECT-OPTIONS 범위 필터 특수 제어 옵션
- **`NO INTERVALS`**: From과 To로 나뉜 넓은 범위 입력 상자 중에서 **우측의 'To' 입력 박스만 눈앞에서 싹 지워 숨겨버린다.** (사용자가 단일값만 치는 것처럼 유도).
- **`NO-EXTENSION`**: 레인지 필터 우측 끝에 찰딱 달라붙어 있는 **'다중 선택(여러 줄 입력 화살표)' 버튼을 완전히 삭제해 숨겨버린다.** (사용자가 오직 1줄 조건만 입력하도록 통제).
- *주의*: 이 둘은 엄연히 목적이 다르다. `NO INTERVALS`는 To 상자만 숨길 뿐 우측 다중 버튼은 살아있으므로 팝업을 열면 여전히 범위를 칠 수 있다. 둘 다 확실하게 막으려면 두 옵션을 동시에 도킹시켜야 한다.

### 어떻게 확인하는가

에디터에서 선언적 옵션을 정확한 타입 제약에 맞춰 부여하고 화면에 락이 결합되는 상태를 검증한다.

```abap
REPORT zch15_l10_screen_options.

DATA gv_date TYPE d.

" 1. 문자형 필드에 소문자 보존 및 메모리 ID 탑재!
PARAMETERS p_note TYPE c LENGTH 20 LOWER CASE MEMORY ID znt.

" 2. 범위 필터에서 To 입력창과 다중선택 팝업 화살표 버튼을 철저히 삭제 락 처리!
SELECT-OPTIONS s_date FOR gv_date
  OBLIGATORY
  NO INTERVALS
  NO-EXTENSION.
```

#### 체험/시뮬레이터 설계 (옵션 스위치보드)
- **프로세스 플로우**:
  1. 학습자가 [NO INTERVALS 켜기] 토글을 켠다. s_date의 우측 To 상자가 연기처럼 뿅 사라지고 From 상자만 심플하게 남는 렌더링을 본다.
  2. [NO-EXTENSION 켜기] 토글을 켠다. s_date 끝자락의 노란색 화살표 다중선택 단추가 촥 접혀 사라져 복수 행 차단 락이 걸리는 레이아웃을 감상한다.
  3. p_note 칸에 영어 소문자로 `jhoonyeong` 을 타이핑하고 실행한다. LOWER CASE 옵션 작동으로 대문자로 변환되지 않고 소문자 날것 그대로 변수 방에 수납 완료되는 피드백을 확인한다.
- **상태 및 데이터**:
  - `숫자형 변수 p_num (TYPE I) 뒤에 LOWER CASE 옵션을 강제로 부여해 빌드 시도` -> "LOWER CASE is only allowed for character-like types" 컴파일 경고 에러 하이라이트 노출.
- **피드백**: 선언적 옵션은 복잡한 화면 제어 코드를 수십 줄 줄여주는 최고의 가성비 스위치이므로, 코딩을 들이밀기 전에 카탈로그 옵션을 먼저 확인해야 함을 선사한다.

### 실수/주의

- **OBLIGATORY(필수)를 걸었으니 값의 비즈니스 검증은 끝났다고 방심하기**: "필수 입력을 걸었으니 이상한 값은 안 들어오겠지" 하는 것은 착각이다. 필수 입력은 단순히 **"입력창이 비어있지 않음"** 을 보장할 뿐, 사용자가 가짜 코드 `H999` 를 대충 쳐서 통과하는 것은 막지 못하므로, 유효성 검증은 여전히 PAI 인 `AT SELECTION-SCREEN`에서 수행해야 한다.

### 정리

- **`OBLIGATORY`** 로 필수 입력을 선포하고, **`MEMORY ID`** 로 마지막 작성 값을 복원한다.
- **`LOWER CASE`** 는 오직 **문자형(Character-like) 필드에서만 대문자 강제 변환을 막는 사양 제약**을 준수한다.
- To 칸을 숨기는 **`NO INTERVALS`** 와 복수 선택 단추를 날리는 **`NO-EXTENSION`** 은 쪼개어 제어한다.

## CH15-L11 - 여러 선택화면 — 화면번호·CALL·Variant

### 왜 필요한가

지금까지 우리가 짰던 모든 Parameters와 Select-Options 입력창들은 프로그램이 켜지자마자 나오는 단 하나의 넓은 첫 화면에 다닥다닥 수직으로 모여서 쌓였다. 
하지만 실무 화면을 짤 때는 조건이 30개가 넘어가기 일쑤다. 
"첫 메인 화면은 기본 필수 조건 3개만 심플하게 배치해 두고, 사용자가 '고급 조건 설정' 버튼을 누르는 순간 예쁘고 조그마한 서브 입력 팝업창(Modal Window)이 화면 위에 짠 튀어나와 세부 로그 여부나 한계 건수를 추가로 받아서 닫히게 구성하고 싶다" 
메인 조건 공간을 난잡하게 오염시키지 않고, **독자적인 방 번호를 가진 비밀 선택화면들을 서브 창으로 보관해 두었다가 필요할 때 팝업으로 점프해 호출하고, 사용자가 그 안에서 작업을 완료했는지 취소했는지를 안전하게 분기해 유통하는 다중 화면 기획**이 필요하다. 그것이 **여러 선택화면 개설과 호출([[CALL SELECTION-SCREEN]])** 이다.

### 무엇인가

#### 1. 선택화면 번호와 다중 개설 규칙
- **표준 1000번**: 우리가 아무 번호도 선언하지 않고 적은 기본 선택화면은 SAP가 내부적으로 **1000번**이라는 전용 번호를 붙여 메인 화면으로 가동한다.
- **보조 화면 선언**: `SELECTION-SCREEN BEGIN OF SCREEN <번호> AS WINDOW ...` 지시어를 써서 독자적인 방 번호(예: `1100`)를 가진 보조 선택화면을 정의한다. `AS WINDOW` 옵션을 붙여야 팝업 대화상자(Modal Popup) 크기로 알맞게 출산된다.

#### 2. CALL SELECTION-SCREEN 번호 호출과 sy-subrc 리턴 분기
- **호출**: 아바 프로그램 내부에서 **`CALL SELECTION-SCREEN 1100 STARTING AT 열 행`** 명령어를 가동하면, 1000번 화면 위에 1100번 선택화면 팝업창이 modal 상태로 즉시 도킹되어 열린다.
- **⚠️ [물리적 CALL SCREEN 호출 금지령 (문법 철칙)]**:
  - *초보 개발자들이 일반 Dynpro 화면 호출 방식과 혼동해 컴파일을 터트리는 단골 함정이다.*
  - **선택화면 객체는 일반 화면 호출 지시어인 `CALL SCREEN 1100` 으로 부르면 아예 런타임 덤프나 오류가 나므로, 반드시 'CALL SELECTION-SCREEN' 지시어 명세를 사수해야 한다.**
- **사용자 액션 판정 (`sy-subrc`)**:
  - 사용자가 팝업창 아래의 [실행 (Execute)] 단추를 클릭해 창을 닫으면 **`sy-subrc = 0`** 이 반환된다 (사용자가 입력 값을 반영하겠다는 동의).
  - 사용자가 우측 상단의 [X] 나 [취소 (Cancel / Back)]를 눌러 창을 닫으면 **`sy-subrc = 4`** 가 반환된다. (입력값을 반영하지 않고 무효화하겠다는 거부).
  - *제어*: 이 리턴 코드로 분기하지 않으면 사용자가 취소했는데도 값이 저장되는 억지 버그가 나므로 반드시 IF 분기문을 쳐서 방어해야 한다.

#### 3. Selection Screen Variant (입력값 세트 저장소)
- 매번 화면에 수십 개 조건을 손으로 치기 번거로우므로, 현재 채워진 조건 세트를 금고에 넣고 이름을 붙여두는 기능이 **`Variant` (베리언트)** 이다.
- 백엔드 배치 잡([CH35]에서 상세 배움)으로 프로그램을 스케줄러에 등록할 때는 사용자가 화면을 칠 기회가 없으므로, **반드시 미리 저장해 둔 Variant 이름을 대입해 기입하여 백그라운드 쿼리 조건을 공급하는 것이 필수 개발 철칙**이다.

### 어떻게 확인하는가

1100번 보조 팝업 화면을 선언하고, 버튼 클릭 시 CALL 하여 sy-subrc 결과에 따라 안전하게 값을 대입 통제하는 과정을 확인한다.

```abap
REPORT zch15_l11_multi_screen.

TABLES sscrfields.

" 1. 기본 1000번 화면에 푸시버튼 장착!
SELECTION-SCREEN PUSHBUTTON /1(20) gv_btn USER-COMMAND open.

" 2. 보조 1100번 팝업 화면 개설!
SELECTION-SCREEN BEGIN OF SCREEN 1100 AS WINDOW TITLE TEXT-010.
  PARAMETERS p_max TYPE i DEFAULT 100.
SELECTION-SCREEN END OF SCREEN 1100.

INITIALIZATION.
  gv_btn = '고급조건 설정'.

AT SELECTION-SCREEN.
  CASE sscrfields-ucomm.
    WHEN 'OPEN'.
      " [호출 철칙] CALL SCREEN이 아닌 CALL SELECTION-SCREEN 사용!
      CALL SELECTION-SCREEN 1100 STARTING AT 10 5.
      
      " [분기 철칙] 사용자가 실행(0)했는지 취소(4)했는지 철저 대조!
      IF sy-subrc = 0.
        MESSAGE '고급 최대 건수 조건이 적용되었습니다' TYPE 'S'.
      ELSE.
        MESSAGE '설정이 취소되어 무효화 되었습니다' TYPE 'S'.
      ENDIF.
  ENDCASE.
```

#### 체험/시뮬레이터 설계 (보조 선택화면과 Variant 금고)
- **프로세스 플로우**:
  1. 학습자가 [고급조건 설정] 본문 버튼을 클릭한다.
  2. 1000번 메인 화면 위로 투명도가 어두워지며 `1100번 팝업 대화상자`가 팝업 모션을 내며 부드럽게 튀어 오르는 비주얼을 본다.
  3. 학습자가 팝업의 최대 건수에 `500` 을 적고 [Execute (실행)]을 누른다. sy-subrc = 0 파란 불이 들어오며 "500건 조건이 최종 반영되었습니다" 피드백을 확인한다.
  4. 다시 열고 값을 `999` 로 바꾼 뒤 우측 상단 [X]를 눌러 닫는다. sy-subrc = 4 노란 불이 켜지며 "설정이 취소되었습니다"가 출력되고, 값은 이전 상태인 500으로 안전하게 방어 복원되는 롤백 모션을 관찰한다.
- **상태 및 데이터**:
  - `CALL SCREEN 1100 문장으로 팝업 호출을 유도하는 코딩 상태` -> 컴파일하자마자 화면 경고등이 켜지며 "Selection screen 1100 cannot be called via CALL SCREEN" 컴파일 덤프 경고 하이라이트 노출.
- **피드백**: 다중 화면의 유통 벨트는 반드시 CALL SELECTION-SCREEN의 전용 지시어와 sy-subrc 리턴 분기의 합작으로만 안전하게 제어됨을 인지시킨다.

### 실수/주의

- **Variant가 모든 보안/입력 검증을 우회 처리해 준다고 오해하기**: "Variant 조건 파일을 불러왔으니 입력 검증(AT SELECTION-SCREEN)은 실행 안 해도 통과되겠지" 하는 것은 착각이다. Variant는 단순히 **화면 타자를 대신 쳐주는 자동완성 매크로**일 뿐이며, 불러온 직후 실행 단추를 누르면 아밥 생명주기인 `AT SELECTION-SCREEN` 검증 이중문이 100% 동일하게 실행되므로, 검증 로직은 꼼꼼하게 똑같이 굴러가야 한다.

### 정리

- executable report의 기본 메인 화면은 **`1000번`** 이며, 보조 화면은 **`AS WINDOW`** 로 빚어 개설한다.
- 선택화면 팝업 호출은 일반 Dynpro 지시어 대신 반드시 **`CALL SELECTION-SCREEN`** 명세를 사용한다.
- 팝업 종료 후 **`sy-subrc = 0` (실행) 과 `sy-subrc = 4` (취소)** 의 흐름을 명확하게 분기해 값을 가공한다.

## CH15-L12 - 실습 — 예매현황 리포트 (이벤트·검증)

### 왜 필요한가

우리가 이 장에서 배운 INITIALIZATION 초기 제안, OUTPUT 화면 동적 숨김 제어, PAI 입력 검증, 존재성 SELECT SINGLE 검사, AUTHORITY-CHECK 권한 이중 관문, sscrfields-ucomm 버튼 연계 기술을 파편적으로 다 정복했다. 

이제 이 파편들을 하나로 엮어, **🎫 실제 현업이 운영 시스템에서 매일 사용하는 정식 A급 '예매현황 리포트 프로그램'의 완결 파이프라인**을 조립해야 할 시간이다. 
프로그램을 켜면 오늘 날짜와 제안 코드가 쫙 채워져 있고(INITIALIZATION), 모드가 아닐 때는 세부 입력 필드가 안전하게 숨어있으며(OUTPUT), 사용자가 실행을 클릭하면 공백 오타 검사(PAI)와 실존 데이터 검사(존재) 및 권한 검문(보안)까지 촘촘하게 3단 바리케이드를 돌파한 뒤, 비로소 안전이 입증된 데이터만 DB에서 긁어와(START-OF-SELECTION) 깔끔한 ALV 표(SALV)로 인쇄하는 **종합 캡스톤 프로세스**의 완성이 필수적이다.

### 무엇인가

#### 🎫 A급 예매 조회 리포트 종합 아키텍처 명세
우리는 데이터 사전에 등록된 `ZCONCERT` (공연 마스터)와 `ZBOOKING` (예매 테이블)을 연동해 다음 4단계 이벤트 흐름을 빌드한다.

1. **초기값 세팅 (`INITIALIZATION`)**:
   - 화면 최초 가동 시 공연 코드 `p_conc` 에 `C001`을 제안 기입.
   - 예매 상태 필터 `s_stat` 에 기본 Range 값인 `I / EQ / R` (예약 완료 상태) 레코드를 조립해 APPEND 장착.
2. **동적 화면 필터 제어 (`AT SELECTION-SCREEN OUTPUT`)**:
   - 사용자가 `p_mode` 에 `'A'` (고급 모드)를 기입하지 않았을 경우, 고급용 세부 텍스트 창인 `p_note` 필드를 화면에서 숨김(`active = '0'`) 처리 후 `MODIFY SCREEN` 반영.
3. **3단 검문 가드 (`AT SELECTION-SCREEN ON p_conc`)**:
   - **1단 공백 검문**: `p_conc` 가 비어있으면 "공연 코드를 입력하세요 (E)" 차단.
   - **2단 존재 검문**: `ZCONCERT` 마스터 테이블에 실존하는 코드인지 `SELECT SINGLE` 하여 없으면 "없는 공연 코드입니다 (E)" 차단.
   - **3단 보안 검문**: `AUTHORITY-CHECK` 로 권한 객체 `Z_CONCERT` 의 `ACTVT = '03'`(조회) 권한 등급을 스캔해 부족 시 "볼 권한이 없습니다 (E)" 차단.
4. **조회 및 ALV 표 서빙 (`START-OF-SELECTION`)**:
   - 3단 검문을 무사통과한 합격 조건으로 `ZBOOKING` 을 고속 SELECT 수납.
   - 결과가 0건이면 상태 바에 "조회된 예매가 없습니다 (S)" 를 뿌리고 즉시 탈출(`RETURN`).
   - 결과가 존재하면 `CL_SALV_TABLE` 팩토리를 출산해 화면에 ALV 격자판으로 최종 display 렌더링. (신규 리포트이므로 불필요한 `END-OF-SELECTION` 은 아예 생략 격리).

### 어떻게 확인하는가

완성된 종합 캡스톤 리포트 프로그램을 실행하고, 5대 검증 시나리오에 따라 각 이벤트 방이 제자리에서 톱니바퀴처럼 연동되어 구동되는지를 최종 검수한다.

```abap
REPORT zch15_concert_booking.

" 화면 구성원 정의
PARAMETERS: p_conc TYPE zconcert-concert_id,
            p_mode TYPE c LENGTH 1,
            p_note TYPE c LENGTH 20.

DATA: gv_chk     TYPE zconcert-concert_id,
      gv_status  TYPE zbooking-status,
      gt_booking TYPE STANDARD TABLE OF zbooking,
      lo_alv     TYPE REF TO cl_salv_table,
      gs_screen  TYPE screen.

SELECT-OPTIONS s_stat FOR gv_status.

" ① 1단계: 초기 제안 기본값 장착!
INITIALIZATION.
  p_conc = 'C001'.
  s_stat-sign = 'I'. s_stat-option = 'EQ'. s_stat-low = 'R'.
  APPEND s_stat.

" ② 2단계: 화면 렌더링 직전 동적 속성 조율!
AT SELECTION-SCREEN OUTPUT.
  LOOP AT SCREEN INTO gs_screen.
    IF gs_screen-name = 'P_NOTE' AND p_mode <> 'A'.
      gs_screen-active = '0'.
      MODIFY SCREEN FROM gs_screen.
    ENDIF.
  ENDLOOP.

" ③ 3단계: PAI 시점에 3단 바리케이드 검문 집행!
AT SELECTION-SCREEN ON p_conc.
  " 1단: 공백 방어
  IF p_conc IS INITIAL.
    MESSAGE '공연 코드를 입력하세요' TYPE 'E'.
  ENDIF.

  " 2단: 존재 방어
  SELECT SINGLE concert_id FROM zconcert INTO gv_chk
    WHERE concert_id = p_conc.
  IF sy-subrc <> 0.
    MESSAGE '없는 공연 코드입니다' TYPE 'E'.
  ENDIF.

  " 3단: 보안 방어
  AUTHORITY-CHECK OBJECT 'Z_CONCERT'
    ID 'CONCERT' FIELD p_conc
    ID 'ACTVT'   FIELD '03'.
  IF sy-subrc <> 0.
    MESSAGE '이 공연을 볼 권한이 없습니다' TYPE 'E'.
  ENDIF.

" ④ 4단계: 검문을 다 통과한 안전 조건으로 본 처리 조회 및 SALV 인쇄!
START-OF-SELECTION.
  SELECT * FROM zbooking INTO TABLE gt_booking
    WHERE concert_id = p_conc
      AND status IN s_stat.

  " 결과 0건 방어막
  IF gt_booking IS INITIAL.
    MESSAGE '조회된 예매가 없습니다' TYPE 'S'.
    RETURN.
  ENDIF.

  " 안전한 SALV 화면 출력
  TRY.
      cl_salv_table=>factory(
        IMPORTING r_salv_table = lo_alv
        CHANGING  t_table      = gt_booking ).
      lo_alv->display( ).
    CATCH cx_salv_msg.
      MESSAGE 'ALV 표시 중 오류가 발생했습니다' TYPE 'E'.
  ENDTRY.
```

#### 체험/시뮬레이터 설계 (예매 리포트 실행 시뮬레이터)
- **프로세스 플로우**:
  1. 학습자가 [코드 누락 시나리오] 단추를 누른다. p_conc를 빈칸으로 만들고 F8을 때린다. PAI의 1단 바리케이드 에러등이 켜지며 화면으로 억류 복귀한다.
  2. [권한 없는 C002 입력 시나리오] 단추를 클릭한다. C002 코드가 들어가며 실행을 시도한다. 1, 2단을 통과하지만 3단 보안 검문관이 계정을 대조하고 붉은빛 쉴드를 켜서 차단 복귀시키는 흐름을 본다.
  3. [정상 조건 C001 시나리오]를 누른다. 3단 검문을 스르륵 통과하더니 본 처리 공장으로 내려가 DB를 SELECT 하고, 아름다운 SALV 표가 팝업으로 연출 출력되는 예매현황 조회의 전 유통 파이프라인 지도를 완결해서 감상한다.
- **상태 및 데이터**:
  - `s_stat 상태 필터를 예약완료(R)에서 취소(C)로 조정한 상태` -> 조건에 맞는 취소 데이터가 DB에 없어 sy-subrc 가 0이 아닌 상태로 돌아와 "조회된 예매가 없습니다" 하단 조용한 안내와 함께 화면이 굳지 않고 소프트 랜딩하는 상태 데이터 피드백 표출.
- **피드백**: "이 리포트의 완성 DoC는 단순히 코드가 돌아가는 것이 아니라, 프로그램의 생명주기 생명주기 이벤트 방마다 알맞은 비즈니스적 책임이 엄격하게 분배되어 있느냐이다" 임을 나선형 종합 완료 마크와 함께 뿌듯하게 선사한다.

### 실수/주의

- **보안 검증을 START-OF-SELECTION이나 SALV 출력 직전에 배치**: "어차피 출력하기 전에 권한 없음을 체크하면 데이터는 안 나가니까 괜찮겠지" 하는 것은 보안상 아주 위험한 아키텍처다. 권한이 없는 비인가자라 하더라도 조회 구획에 도킹되면 이미 무거운 DB 테이블 조회가 가동되어 서버에 엄청난 트랜잭션 부하를 얹고 난 뒤에야 차단되는 시스템 낭비가 초래되므로, **모든 보안/형식 검증은 무조건 DB 조회 쿼리가 돌기 전인 `AT SELECTION-SCREEN` 검문소에서 1순위로 조기 차단(Early Exit)** 해야 한다.

### 정리

- **`INITIALIZATION`** (기본값) $\rightarrow$ **`OUTPUT`** (화면제어) $\rightarrow$ **`ON <field>`** (3단검문) $\rightarrow$ **`START-OF-SELECTION`** (DB조회 및 ALV) 순서로 책임을 격리한다.
- LDB와 엮이지 않는 현대식 리포트이므로 **`END-OF-SELECTION` 은 완전히 생략**하는 것이 클린 아키텍처 규칙이다.
- 존재 검증(SELECT SINGLE)과 권한 검증(AUTHORITY-CHECK)은 서로 성격이 전혀 다르므로 **메시지 타입과 문구를 철저히 격리**해 사용자 시선에 맞춰 표출한다.
