## 4. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. SELECT-OPTIONS s_conc FOR zbooking-concert_id. 구문을 선언할 때, 왜 TABLES zbooking. 구문을 '반드시' 세트로 윗줄에 먼저 적어주어야 하나요?
**A**: **선택 화면(Selection Screen)의 입력 필드가 참조할 'Dictionary 필드의 구조 모델'을 컴파일러가 사전에 인지해야 하기 때문**입니다.
`SELECT-OPTIONS`는 선언과 동시에 SAP 스크린 시스템에 입력창 상자를 그리게 됩니다. 이때 아밥 컴파일러는 화면에 그릴 입력 박스의 데이터 타입(문자인지 숫자일지, 길이는 몇 자리인지)과 훗날 화면에서 F4 키를 누르면 띄워줄 검색 도움말(Search Help) 등의 메타데이터 속성을 통째로 알고 있어야 상자를 안전하게 렌더링할 수 있습니다.
이를 위해 `zbooking-concert_id` 필드의 뼈대 메타데이터를 통째로 상속 참조하게 되는데, 이 사전 계약 구조를 공유하는 통로가 바로 **`TABLES 테이블명.`** 선언입니다. 이 선언이 누락되면 컴파일러는 zbooking이라는 존재 자체를 사전에서 참조할 주소로 인지하지 못해 빌드 구문 에러를 냅니다.

### Q2. 빈 범위(Empty Range)를 조건에 넣고 실행하면, 왜 아무 필터가 안 걸리는 참(True) 상태가 되어 테이블의 '전체 데이터'를 긁어오나요? 0건 조회가 되는 게 더 논리적이지 않나요?
**A**: **"검색 필터 조건을 명시하지 않았으니, 해당 필드에 대한 조회 제한을 해제(제한 없음)하겠다" 는 대화형 화면의 실무적인 기본 약속 때문**입니다.
우리가 인터넷 쇼핑몰에서 상품을 검색할 때, '제조사' 필터를 아무것도 고르지 않고 빈칸으로 비워둔 채 검색을 누르면 모든 제조사의 상품이 다 쏟아져 나오는 경험을 해보셨을 것입니다.
SAP의 `SELECT-OPTIONS` 역시 이와 동일하게 작동합니다. 만약 빈 범위를 0건으로 처리해 버리면, 사용자가 아무 필터도 안 썼을 때 빈 화면만 보게 되어 매번 모든 필터를 수작업으로 가득 채워 검색해야 하는 심각한 사용성 불편을 겪게 됩니다.
따라서 SAP DB 엔진은 **"Range Table이 텅 비어있음(Initial) = 필터 제한 없음(전체 통과)"** 으로 암묵적 룰을 정했습니다. 편리한 반면, 초대용량 운영 DB에서 이 규칙을 방치하면 풀스캔 조회로 서버가 정지될 수 있으므로, 실무에서는 꼭 필수 조건 검증을 엮어 방어해야 합니다.

### Q3. 다중 선택 팝업창에서 녹색 포함(Include) 조건 2줄과 빨강 제외(Exclude) 조건 1줄을 엮어 넣었을 때, 내부적으로 🟢 포함과 🔴 제외 조건이 계산되는 최종 연산 순서를 수학적으로 명쾌하게 밝혀주세요.
**A**: **"포함(I) 조건들의 합집합(OR) 연산을 먼저 묶어 수행하여 1차 합격자를 뽑은 뒤, 제외(E) 조건들의 차집합(AND NOT)을 나중에 삭감 차감" 하는 2단계 순서로 판정합니다.**
수학식으로 보면 다음과 같습니다.
$$\text{최종 합격 데이터} = (\text{포함행 1} \cup \text{포함행 2}) \setminus \text{제외행 1}$$
예를 들어 `s_conc`에 `I EQ C001`, `I EQ C003`, `E EQ C002` 세 행이 얹혀 있다면:
1. **1단계 (Include 합산)**: 공연 번호가 C001 이거나 또는(`OR`) C003 에 해당하는 로우 상자들을 다 쓸어모아 1차 통과 후보군 주머니에 적재합니다. (합집합 C001, C003 확보).
2. **2단계 (Exclude 삭감)**: 확보된 통과 후보군 주머니 내부에서, 제외 조건인 C002와 일치하는 것이 있는지 샅샅이 뒤져 만나는 즉시 칼같이 잘라내 제외시킵니다. (C002는 원래 후보군에 없었으므로 영향 없음).
만약 포함 조건에 C002도 포함되어 있었더라도, 이 2단계 제외 삭감 컷트라인에서 칼자루를 쥔 붉은색 E 조건에 의해 **C002는 결국 최종 불합격 탈락** 처리됩니다.

### Q4. CP(Contains Pattern) 옵션에서 쓰는 와일드카드 `*` 와 `+` 의 물리 거동 차이를 예시를 들어 보여주시고, 쿼리의 `LIKE` 와일드카드 기호와 절대로 섞어 쓰면 안 되는 이유를 설명해 주세요.
**A**: **`*` 은 자릿수 무제한 가변 매칭이고, `+` 는 정확히 단 1자리 글자 공간의 점유를 뜻합니다.**
- **`*` (무한 가변)**: `LOW = 'C*'` 로 기입하면 C 뒤에 글자가 안 붙은 `C` 도 통과, 3글자인 `C001` 도 통과, 10글자인 `C001002003` 도 전부 매칭되어 통과합니다.
- **`+` (1자리 홀더)**: `LOW = 'C+01'` 로 기입하면 C와 01 사이에 정확히 한 글자 방(`+`)만 허용합니다. 즉 4자리인 `C001`, `CA01` 은 자릿수가 딱 맞아서 통과되지만, 5자리인 `C0010` 이나 3자리인 `C01` 은 자릿수가 어긋나서 무참히 탈락합니다.
- **LIKE와 혼용 금지**: 데이터베이스 일반 쿼리 `LIKE` 절에서는 `%` (가변)와 `_` (1자리)를 문법으로 쓰지만, ABAP의 Range Table `CP` 비교 엔진은 오직 **`*` 와 `+`** 만을 와일드카드 제어 문자로 분석하도록 설계되어 있습니다. 만약 CP에 `C%` 를 적어 던지면 시스템은 퍼센트 기호 자체를 진짜 글자로 대조하는 비정합성 오작동을 하므로 기호의 경계를 칼같이 가리셔야 합니다.

### Q5. 예전 레거시 구문인 RANGES r_stat FOR zbooking-status. 구문을 현업 신규 코딩 시 절대 쓰지 말아야 할 치명적인 아키텍처적 위험성은 무엇인가요?
**A**: **구식 RANGES 문은 '테이블의 헤더 라인(Header Line)'을 무단으로 강제 복제하여 선언하므로, 현대 Clean Core 표준인 Local Class 및 전역 Method 내부에서 구문 덤프 에러를 일으켜 프로그램 재사용성을 완전히 파괴하기 때문입니다.**
과거의 `RANGES` 구문은 Range Table 변수를 개설하면서 동시에 테이블의 한 행을 담는 작업 영역(Work Area)의 이름까지 똑같은 `r_stat` 이라는 동명의 포인터로 강제 연결하는 구시대적인 이중성 선언(헤더라인 방식)을 내부적으로 집행합니다.
이 이중 선언은 소스 분석을 헷갈리게 만들 뿐 아니라, 현대 표준 개발 아키텍처인 클래스의 메서드(`METHOD`) 안으로 진입하게 되면 **"메서드 내부에서는 헤더 라인을 가진 테이블 변수를 절대 생성하거나 전달받을 수 없다"** 는 표준 컴파일 규칙에 꽝 들이받혀 빌드 경고와 함께 활성화 자체가 원천 차단됩니다.
따라서 신규 개발 시에는 화면 없이 Range를 개설할 때 반드시 1인 1역 독립 선언인 **`TYPE RANGE OF`** 와 **`LIKE LINE OF`** 형식을 사용해 조립하는 것이 Clean 표준입니다.
