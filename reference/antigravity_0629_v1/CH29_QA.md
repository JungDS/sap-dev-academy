## 3. 공식 문서 수동 근거 명세

`abapget_badi.htm` 을 대조하여 BAdI 의 물리 예외 및 안전 제약 요건을 철저히 규명했다.

- **Single-use BAdI 에러 제약 규칙**:
  `GET BADI` 호출 시 해당 BAdI 정의가 **Single-use** 로 지정된 경우, 메모리에 올라와 활성화된 구현 클래스가 **아예 없거나(0개)** 혹은 **2개 이상 중복 존재(Multiple)** 할 시 즉시 catchable exception 인 **`CX_BADI_NOT_IMPLEMENTED`** 또는 **`CX_BADI_MULTIPLY_IMPLEMENTED` 캐치 예외 덤프**를 터트리고 트랜잭션을 강제 다운시킨다.
- **Fallback Class 지명 의무 규칙**:
  0개 매칭 시의 시스템 폭사를 미연에 방지하기 위해 BAdI 정의 빌더(`SE18`)상에서 구현 클래스가 0개일 때 대신 호출될 기본 대체 구현체인 **`Fallback Class`** 안전 지지대를 반드시 사전에 지정해 주어야 함을 대조 및 수록 완료.

## 4. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. Single-use BAdI 를 호출하기 위해 GET BADI badi 문을 쐈는데 왜 가끔 CX_BADI_NOT_IMPLEMENTED 나 CX_BADI_MULTIPLY_IMPLEMENTED 예외 덤프가 터지며 시스템이 뻗나요? 예방하는 방법은 무엇인가요?
**A**: **Single-use BAdI 는 시스템 상에 활성 구현 클래스가 '오직 정확히 1개(Exactly one)' 상주해 돌아가는 사양이므로, 0개(결핍)이거나 2개 이상(중복 충돌) 매칭되면 아바 엔진이 누구를 실행할지 판단하지 못해 예외를 던지기 때문이며, SE18 상에 Fallback Class 를 지정해 방어해야 합니다.**
- **덤프의 물리 원인**:
  1. **구현 0개 (`CX_BADI_NOT_IMPLEMENTED`)**: 
     표준 프로그램이 이벤트를 실행하려고 BAdI 를 깨웠는데, 아무 부서도 구현 클래스를 안 만들어 활성화해 두지 않았습니다. 
     BAdI 가 빈 깡통 상태이므로 `GET BADI` 에서 예외가 터져 다운됩니다.
  2. **구현 2개 이상 (`CX_BADI_MULTIPLY_IMPLEMENTED`)**: 
     예를 들어 한국 지사도 구현을 켜고, 본사도 구현을 켜서 활성 칩이 2개가 중복 도킹되어 있습니다. 
     BAdI 가 어느 녀석을 골라 1순위로 실행해 줄지 다형성 충돌을 감지해 강격발 예외를 때립니다.
- **예방 설계 (정석)**:
  BAdI 정의 빌더(`SE18`) 상에 **`Fallback Class` (기본 대체 구현체)** 를 의무 등록해 둡니다. 
  이렇게 해 두면, 0개 구현 상황일 때 덤프를 터트리는 대신 Fallback Class 의 기본 메서드를 가동해 덤프 없이 안전하게 흘러가게 해 줍니다. (다중 중복 시에는 SE19 에서 불필요한 구현을 찾아 비활성화(Deactive) 조치해야 수습됩니다.)

### Q2. 표준을 확장하라고 해서 BAdI 구현 메서드(또는 User Exit include) 안방에 들어갔습니다. 내가 임의로 DB 데이터를 수정하고 commit 을 날리기 위해 COMMIT WORK 이나 ROLLBACK WORK 문장을 기재해 두었는데 왜 갑자기 덤프가 쾅 터지나요?
**A**: **표준 BAdI 와 Exit 은 이미 표준의 DB 락 큐가 묶이고 트랜잭션 버퍼가 돌고 있는 저장 시퀀스 한가운데에서 작동하므로, 내 로직이 쌩으로 커밋/롤백을 질러 트랜잭션을 찢어발기면 DB 정합성이 박살 나며 커널 덤프가 터지기 때문입니다.**
- **물리적 모순**:
  SAP 표준 프로그램은 오더를 저장할 때 `VA01` 내부에서 DB UPDATE 문들을 돌리고, 최종 정합성 검증이 모두 통과되면 마지막 최선단 PAI 에서 단 한 번 `COMMIT WORK` 를 때려 트랜잭션을 마칩니다.
  그런데 이 중간 다리 길목에서 돌고 있는 내 BAdI 검증 로직이 쌩뚱맞게 중간에 끼어들어 `COMMIT WORK` 도장을 무단으로 쾅 찍어버리면, 아직 채 다 돌지 않은 표준 업데이트 큐가 강제로 조기 커밋 확정되어 데이터 조각들이 쪼개져 저장됩니다. 
  반대로 에러 났다고 `ROLLBACK WORK` 를 때려버리면 표준이 쥐고 있던 전체 세션 세팅 메모리가 싹 날아가 표준 프로그램 자체가 뇌사 덤프를 뿜습니다.
- **철칙**: BAdI/Exit 안방에서는 **절대로 COMMIT WORK 나 ROLLBACK WORK 를 직접 타이핑해선 안 되며**, 오직 비즈니스 계산과 에러 판정만 하고 트랜잭션 확정 권한은 전적으로 **표준 엔진에게 위임**하는 것이 절대 철칙입니다.

### Q3. Customer Exit 을 사용하기 위해 SMOD 에 들어가 빈 include 파일 안에 멋지게 검증 Z코드를 채우고 빌드도 성공했습니다. 그런데 왜 운영 서버에 배포하니 내 검증 코드가 무시당한 채 쌩 스킵되어 넘어가나요?
**A**: **SMOD 는 단지 SAP 가 제공해 준 Customer Exit 정의 설계도를 구경하는 공간일 뿐이며, 반드시 CMOD (고객 프로젝트 빌더)를 켜서 엑시트를 프로젝트 주머니에 쓸어 담고 '활성화(Activation)' 전원을 켜주어야 바인딩이 연결되기 때문입니다.**
- **바인딩 누락 원인**:
  아바 컴파일러는 빈 include 소스가 채워져 있어도, CMOD 에 등록된 고객 활성 마크(Project Activation)가 체크되어 있지 않으면 "이 고객사는 이 엑시트 기능을 작동시킬 의사가 없다" 고 판단해 런타임 시 엑시트 호출부(`CALL CUSTOMER-FUNCTION`)를 쌩 까고 바이패스(Bypass) 우회 스킵합니다.
  반드시 개발 서버와 운영 서버 모두에서 **`CMOD` 에 들어가 해당 엑시트를 묶어둔 [Project] 가 활성 상태(Active)인지 검문**해야 배선 락이 연결됩니다.

### Q4. 표준 라인 사이에 코드를 심는 Explicit Enhancement Point 와 Explicit Enhancement Section 의 작동 차이와, 왜 Section 이 OCP 원칙을 짓밟는 고위험군인지 비유를 들어 설명해 주세요.
**A**: **Point(추가)는 기존 표준 코드 전광판 옆에 내 스티커 한 장을 '덧붙이는' 안전한 추가 방식인 반면, Section(대체)은 기존 표준 소스 10줄을 뜯어내고 내 소스로 '통째 대체 교체' 하는 방식이므로, 향후 패치 때 표준이 바꾼 보안 패치가 내 Section 에 가로막혀 영구 누락되기 때문입니다.**
- **동작 차이**:
  - **`ENHANCEMENT-POINT`**: 표준이 `ep_point` 깃발을 꽂은 자리에 내 코드를 추가합니다. 기존 표준 소스도 돌고, 내 소스도 돌아서 OCP 에 완벽히 부합하고 충돌이 적습니다.
  - **`ENHANCEMENT-SECTION`**: 이 구간은 "내 코드가 작동하면 표준 구간을 무시하라" 고 선언합니다. 내 plug-in 이 활성화되는 순간, 그 안에 들어있던 원래 SAP 사의 표준 코드들은 컴파일러에 의해 영구 삭제 취소선이 그어져 무력화되고 오직 내 코드만 기동합니다.
- **OCP 파괴 리스크**:
  내년에 세법이나 국가 보안 규정이 강화되어 SAP 사가 패치 패키지를 통해 그 Section 구간 안방에 "해외 송금 한도 1억 제한 가드" 라는 핵심 보안 표준 코드를 추가해 배포했다고 합시다.
  내 시스템에서는 여전히 내가 예전에 작성해 둔 Section 커스텀 칩이 표준을 대체 가로막아 덮어쓰고 있기 때문에, 새로 패치된 1억 제한 가드가 작동하지 않고 무단 우출금이 터지는 정합성 붕괴를 초래합니다. 
  이로 인해 업그레이드할 때마다 모든 Section 코드를 수동 전수 대조 검사해야 하는 유지보수 헬게이트가 열립니다.

### Q5. Implicit Enhancement 는 표준 소스에 겉보기에 아무런 Spot 깃발도 없는데 어떻게 틈새 개입이 가능한가요? 그리고 왜 이 구멍을 남용하면 Clean Core 아키텍처에 위배되나요?
**A**: **Implicit (암묵적) 인핸스먼트는 아바 컴파일러가 모든 Function, Method, FORM 구문의 '맨 첫 줄(시작)' 과 '맨 끝 줄(종료)' 틈새에 숨겨진 묵시적 컴파일 구멍을 자동 개설해 주기 때문이며, 이는 SAP 와 공증된 '공개 Released API 계약' 이 아니어서 패치 시 예고 없이 폭사하기 때문입니다.**
- **암묵적 구멍의 메커니즘**:
  아바 에디터에서 [Show Implicit Enhancement Options] 를 켜면, 소스에 깃발이 없어도 모든 `METHOD ... ENDMETHOD.`, `FUNCTION ... ENDFUNCTION.`, `FORM ... ENDFORM.` 의 시작점(첫 줄)과 끝점(막줄) 자리에 눈물 모양 숨겨진 슬롯이 펼쳐집니다. 이 틈새에 내 코드를 주입할 수 있습니다.
- **Clean Core 위배 이유**:
  이것은 SAP 사가 공식적으로 "여기 데이터를 고치셔도 무결성을 평생 보증합니다" 라고 공증한 **GET BADI/CALL BADI 같은 약속된 계약이 아닙니다.**
  단순히 컴파일러 구멍을 비집고 들어간 뒷구멍이므로, 향후 SAP 가 표준 업그레이드 때 그 메서드의 파라미터 규격(Interface Signature)을 바꾸거나 include 파일명을 바꾸는 즉시 내 Z코드는 허공에 붕 뜬 좀비 코드가 되어 즉각 컴파일 덤프를 터트립니다. 
  미래 S/4HANA Cloud 전환 시 이 묵시적 인핸스먼트는 보안 및 Clean Core 미준수 판정을 받아 실행이 전면 차단되는 수모를 겪습니다.
  **규칙**: 임플리시트는 BAdI 나 Explicit Point 가 눈 씻고 찾아봐도 전혀 없는 최악의 빈손 상황에서만 극도로 제한적으로 수량/가드 1줄 얹는 데 그쳐야 정석입니다.
