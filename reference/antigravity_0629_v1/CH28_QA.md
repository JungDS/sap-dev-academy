## 3. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. 화면에서 좌석 수를 '5' 로 입력하고 마우스 포커스가 그 입력창에 깜빡거리는 상태에서 곧바로 우측 상단 [저장] 단추를 눌렀습니다. 왜 저장 완료 후 확인해 보니 seats 값이 반영 안 되고 이전 값으로 돌아가며 유실되나요?
**A**: **사용자가 Enter 나 마우스 포커스 아웃을 치지 않고 버튼을 다이렉트로 누르면, 마지막 셀의 임시 입력 상태는 grid 버퍼 큐에만 방치되어 내부 테이블(mt_booking)로 Flush 적재되지 못하기 때문이며, 저장 PAI 로직 1순위에 check_changed_data( ) 를 강제 기격발하여 수거해야 합니다.**
- **데이터 유실 메커니즘**:
  사용자가 키보드로 값을 타이핑하고 있는 도중에는 해당 데이터는 ALV 컨트롤의 '물리 윈도우 스크린 임시 메모리' 에만 둥둥 떠 있습니다. 
  사용자가 엔터를 때리거나 포커스를 다른 행으로 튕겨주어야 비로소 이 임시 버퍼 값이 회수되어 아바 내부 테이블(`mt_booking`)에 정식 라이팅 갱신됩니다.
  그런데 입력 완료 엔터도 안 때리고 포커스 아웃도 안 한 채 곧바로 화면 헤더의 [저장] 버튼(PAI 이벤트)을 클릭하면, 아바 프로그램은 내부 테이블 데이터를 통째로 DML 칠 준비를 마치지만, 5번 셀의 seats = 5 값은 아직 버퍼 큐에서 탈출하지 못해 수집에서 누락됩니다. 결국 이전 값인 2석인 상태로 `MODIFY TABLE` 이 가동되어 유실됩니다.
- **방어 규칙 (정석)**:
  ```abap
  " [ 저장 버튼 PAI 로직 진입 즉시 최상단 강제 회수 철칙! ]
  go_grid->check_changed_data(
    " 이 메서드가 격발되어 큐를 Flush 하고 변경 완료 플래그를 mt_booking 에 안전 안착시킴!
  ).
  ```
이 메서드 호출을 PAI 최선단에 심어두어야 비로소 마지막 1칸 입력 유실 사고를 원천 방어할 수 있습니다.

### Q2. data_changed 와 data_changed_finished 이벤트는 둘 다 셀이 변경되었을 때 격발되는 이벤트인데 왜 용도를 섞어서 finished 내벽에서 오류 검증을 태우면 안 되나요?
**A**: **data_changed 는 값이 내부 테이블에 반영되기 '전' 변경 후보를 검문해 거절하는 [순수 검증 시점]이고, data_changed_finished 는 값이 내부 테이블에 반영 완료된 '후' 에 총합 등 파생값을 리프레시하는 [순수 후처리 시점]으로, 시점과 책임이 180도 남남이기 때문입니다.**
- **이유**:
  `data_changed` 시점에는 `er_data_changed->add_protocol_entry` 메서드를 가동해 잘못된 값의 내부 테이블 이송 자체를 긴급 차단하고 셀을 빨간 락으로 채워 방어할 수 있습니다.
  반면 `data_changed_finished` 는 이미 검증이 다 끝나서 예약 테이블(`mt_booking`) 내부에 값 적재가 최종 완수되어 끝난 시점입니다. 
  이 시점에서 "어라? 값이 10석 초과네? 저장하면 안 되는데!" 하고 오류를 튕겨내려 들면 이미 값은 내 램 테이블에 적재 완료되어 오염된 뒤입니다. 
  여기서 뒤늦게 값을 수정하려 EML 이나 쿼리로 테이블을 비틀면 무한 루프 변경 이벤트가 꼬리를 물고 격발되어 스택 오버플로우 덤프가 납니다.
**철칙**: 
- **검증 및 거절**은 무조건 **`data_changed`** 에서만 수행합니다.
- **총합 재계산 및 stable refresh** 는 무조건 **`data_changed_finished`** 에서만 집행합니다.

### Q3. Cell Style 을 사용해 특정 매진 행의 seats(수량) 셀만 입력을 Disabled 락으로 잠그려고 style = mc_style_disabled 를 심었습니다. 그런데 왜 셀이 안 잠기고 여전히 컬럼 전체가 수정 가능한 하얀색 상자로 열려 있나요?
**A**: **Cell Style 은 'Field Catalog edit = true' 라는 대문이 선행 개방되어 있는 전제 하에서만 행별 잠금 예외를 씌우는 우선순위 하위 속성기이므로, 필드캣의 edit 속성을 꺼두면 스타일 제어가 통째로 묵살당하기 때문입니다.**
ALV 레이아웃 엔진의 우선순위 철칙은 아래와 같습니다:
1. Field Catalog 에서 seats 컬럼에 `edit = abap_true` 를 주어 **"이 칼럼 전체는 기본적으로 하얀색 입력칸으로 문을 연다"** 고 선언(1순위 개방)해야 합니다.
2. 그 뒤에 Layout `stylefname` 으로 엮인 행별 `cellstyles` 테이블을 훑어내려가며, **"다만, 매진 행 너는 mc_style_disabled 배지를 달았으니 예외적으로 문을 쾅 잠그겠다"** 고 덮어쓰는(2순위 예외 락) 순서로 작동합니다.
필드카탈로그 `edit` 전원 자체를 꺼버리면 아바 엔진은 "이 칼럼은 문을 열어줄 생각조차 없는 영구 폐쇄 구역" 으로 규정하므로, 밑단에서 아무리 cellstyles 스타일을 심어봤자 문고리조차 닿지 않아 작동을 멈춥니다.

### Q4. data_changed 검증 메서드 안방에서 입력 값이 틀렸을 때 그냥 MESSAGE ... TYPE 'E' 문장으로 경고창을 때리면 왜 안 되나요?
**A**: **쌩 TYPE 'E' 메시지를 때리면 팝업창은 뜨지만 정작 화면 상에서 '어느 줄의 어느 셀 칸' 이 틀렸는지 ALV 가 인지하지 못해 빨간 테두리 하일라이트를 칠하지 못하며, 사용자는 팝업을 닫고 고치면 다음 팝업이 또 뜨는 팝업 지옥에 갇히기 때문입니다.**
- **add_protocol_entry 의 가치**:
  이 메서드를 타야만 `i_fieldname = 'SEATS'` 와 `i_row_id = ls_cell-row_id` 좌표를 ALV 엔진에게 배달하여 "이 3행 2열 셀에 에러 징표를 채워라" 고 지시할 수 있습니다. 
  그러면 ALV 는 해당 셀만 정교하게 빨간색 테두리로 묶어 키보드 입력을 임시 고정하고, 하단 스풀 트레이에 "3행 seats: 좌석수 한도 초과" 라는 친절한 안내를 올려줍니다.
  사용자는 팝업창에 가로막히지 않고 다른 셀들을 자유롭게 편집한 뒤, 에러 셀들만 한눈에 확인해 보정할 수 있게 됩니다.
  반면 쌩 `MESSAGE ... TYPE 'E'` 는 화면 전체를 잠가버려 다른 셀의 편집 흐름까지 싹둑 끊어버리는 최악의 방해 공작입니다.

### Q5. 화면 Data Changed 이벤트와 Cell Style 로 입력 검증을 이미 다 채워두어 잘못된 값이 다 차단되는데, 왜 굳이 [저장] 단추 PAI 로직에서 2차로 전체 데이터 검증을 또 돌려야 하나요?
**A**: **Data Changed 와 Cell Style 은 화면 상에서 실시간 입력을 편리하게 고치도록 돕는 'UI 단 프론트엔드 피드백' 일 뿐이며, 백엔드 DB 무결성을 영구 보증하는 최종 안전 방어선이 아니기 때문입니다.**
- **보안/동시성 위배**:
  화면 UI 제어는 악의적인 직원이 SAP 디버거를 켜서 런타임에 cell style 변수 값을 disabled 에서 enabled 로 우회 변조해 덮어쓰거나, 화면을 거치지 않고 RFC 함수로 다이렉트 SQL UPDATE DML 을 찔러 들어오는 해킹 침범에 대해 100% 무력하게 뚫립니다.
  또한, 조회를 뜬 시점에는 seats = 3 이 정상 입력 범위였으나, 저장 단추를 누르는 0.1초의 짧은 찰나에 다른 사용자가 티켓 10석을 먼저 선점해 결제 완료해 버렸다면, 화면 단은 알지 못하는 '정원 초과' 가 최종 발생하게 됩니다.
  **규칙**: 따라서 입력 단계의 화면 가이드는 UI 편의용으로 엮고, **최종 저장 PAI 단에는 반드시 백엔드 DB 의 최신 정원 현황과 데이터 전체 무결성을 2중으로 단단히 재스캔 검문한 뒤에만 MODIFY 와 COMMIT WORK 을 격발**해야 영구 데이터 파손을 방지합니다.
