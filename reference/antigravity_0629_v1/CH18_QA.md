## 4. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. DATA(data) = ... 와 같이 인라인 선언을 썼을 때, 왜 변수 선언이 안 되고 오프셋/길이 substring 연산으로 컴파일러가 꼬여서 오류를 내나요?
**A**: **ABAP 사전의 기본 선언 연산자 키워드인 `DATA` 와, 런타임이 식별하는 일반 변수명인 `data` 가 완벽하게 동일한 스펠링을 공유하여 컴파일 파서(Parser)가 우선순위를 잃기 때문입니다.**
만약 프로그램 상단에 `DATA data TYPE string.` 과 같이 `data` 라는 이름을 가진 변수가 이미 선언되어 존재하고 있다면,
아랫줄에 기입한 `DATA(data) = '1'.` 이나 `DATA(var) = ...` 라는 코드를 만나는 순간, 컴파일러는 "오라? 기존에 선언된 `data` 라는 변수의 괄호 `( )` 영역에 접근하는구나" 라고 판단합니다. 즉, 클래식 아바의 문자 자르기 문법인 **`data(offset)`** 이나 **`data(length)`** 로 오해하여 해독하는 것입니다. 
이로 인해 "data 변수 뒤에 기입된 var 라는 명칭이 오프셋 숫자가 아니므로 문법이 틀렸다" 며 컴파일을 거부하고 굳어버립니다. 
따라서 모던 아바를 개발할 때는 변수 이름으로 **`data` 라는 단독 단어를 쓰는 것을 완벽히 배제**하고, 대신 `gv_data`, `lv_data`, `ls_data` 처럼 명확한 스코프 접두어를 결합해 명명해야 이 런타임 컴파일 트랩을 피할 수 있습니다.

### Q2. VALUE 연산자로 내부 테이블을 대입할 때 BASE 지시어를 누락하면 왜 기존 테이블 데이터가 소멸해 버리는지 물리적 메모리 초기화 매커니즘을 설명해 주세요.
**A**: **VALUE 연산자는 기존 공간에 덧대는 것이 아니라, 백엔드 런타임 메모리 상에 완전히 새로운 임시 독립 테이블 버퍼를 생성한 뒤 좌측 변수로 통째로 assignment(대입)하는 물리 절차를 타기 때문입니다.**
아바 엔진에서 `lt_booking = VALUE #( ( seats = 2 ) )` 를 실행하면 아래와 같은 물리적 절차가 가동됩니다.
1단계: 메모리 임시 구역에 단 1건의 로우만 들어있는 완전히 새로운 익명 테이블 버퍼를 임시로 빚어냅니다.
2단계: 좌측의 목적지 테이블 `lt_booking` 을 **완전히 초기화(포맷 - Clear)하여 비워버립니다.**
3단계: 1단계에서 빌드한 임시 테이블의 데이터를 `lt_booking` 방으로 통째로 밀어 복사(Over-write)해 넣습니다.
이 과정에서 기존에 `lt_booking` 에 들어있던 1,000건의 예매 데이터는 2단계 포맷 과정에서 흔적도 없이 증발하게 됩니다. 
하지만 **`BASE lt_booking`** 을 기입해 주면, 2단계 포맷 과정이 생략되고 **기존 `lt_booking` 의 메모리 주소를 1단계 임시 버퍼의 부모 기반(Base)으로 먼저 이송 복사한 뒤에 추가 행을 덧대어 갱신**해 주므로 기존 데이터가 무사히 생포 존속됩니다. 따라서 누적 작업 시에는 BASE 가 절대적인 가드라인입니다.

### Q3. Table Expression lt_booking[ booking_id = ... ] 이 없는 키 검색 시 CX_SY_ITAB_LINE_NOT_FOUND 예외 덤프를 내며 폭사하는 물리적 사양은 무엇이며, line_exists 와 line_index 는 어떻게 덤프를 방어하나요?
**A**: **Table Expression은 sy-subrc 신호등을 세우지 않고 결과 '값' 자체를 반환해야 하는 식(Expression)이므로, 결과가 없으면 빈 값을 줄 수 없어 복구 불가능한 시스템 예외 덤프를 발생시키기 때문이며, 두 함수는 덤프가 나기 전 존재 여부나 위치 번호로 우회시켜 방어합니다.**
- **덤프 유발의 이유**:
  `DATA(ls_hit) = lt_booking[ booking_id = 'B999' ]` 은 대입문입니다. 만약 `'B999'` 에 해당하는 행이 존재하지 않는다면, `ls_hit` 라는 새로운 주머니 변수방에 도대체 어떤 타입의 어떤 쓰레기 값을 채워 넣어 탄생시켜야 할지 컴퓨터가 결정할 수 없습니다. 따라서 복구 불가능한 강한 런타임 예외인 **`CX_SY_ITAB_LINE_NOT_FOUND`** 를 뿜고 강제 셧다운됩니다.
- **`line_exists( )` 의 방어 메커니즘**:
  이 함수는 실제로 대괄호 조회를 실행해 값을 꺼내오지 않고, 테이블 내부 인덱스 노드를 순간 스캔하여 **"그 행이 존재하는가?" 에 대한 결과만 논리값(True/False)으로 리턴**해 줍니다. 값이 꺼내지지 않으므로 덤프가 날 틈새가 없이 안전하게 `IF` 문으로 분기 검문할 수 있습니다.
- **`line_index( )` 의 방어 메커니즘**:
  마찬가지로 실제 행 데이터를 꺼내지 않고, 해당 조건의 행이 몇 번째 행에 위치해 있는지 **인덱스 숫자(1~N)만 반환**해 주며, 찾지 못하면 평화롭게 **`0`** 을 리턴하므로 덤프 폭발을 원천 가드합니다.

### Q4. VALUE #( FOR ... ) 루프 식 안에서 사용하는 임시 인덱스 변수를 식 밖에서 호출하면 왜 컴파일러가 문법 오류를 내며 변수 수명 범위(Scope)를 제한하나요?
**A**: **FOR 식의 루프 변수는 VALUE 연산자가 메모리 상에서 테이블을 조립해 내는 찰나의 연산 스택(Expression Stack) 내부에서만 유효한 초단기 로컬 수명(Local Scope)으로 제한 설계되었기 때문입니다.**
`VALUE ty_tab( FOR line IN lt_src ( dan = line-dan ) )` 에서 가동되는 임시 변수 `line` 은, 괄호 `( )` 내부에서 원천 데이터를 한 건씩 집어서 대입 식에 전달해 주는 일종의 **임시 포인터 주소** 역할만 담당합니다.
이 변수방은 VALUE 연산자가 테이블 조립을 끝마치고 최종 테이블을 완성해 던져주는 순간, 메모리 스택에서 **즉각 소멸(Garbage Collected)** 됩니다. 
만약 괄호 바깥인 아랫줄에서 `gv_val = line-dan.` 이라고 쓰면, 이미 소멸한 유령 메모리 주소를 호출하는 꼴이 되므로 컴파일러는 "line 이라는 변수는 존재하지 않는다" 며 컴파일을 거부해 버립니다. 따라서 FOR 루프 변수는 오직 괄호 식 내부에서만 서식하는 1회성 구성원임을 인지해야 합니다.

### Q5. VALUE 식 우측 피연산자 내부에서 좌측의 대상 테이블을 직접 FOR 로 순회할 수 없는 제약은 무엇이고, LET 지시어를 통한 해결 원리는 무엇인가요?
**A**: **VALUE 대입문이 실행되는 즉시 좌측 대상 테이블이 먼저 포맷(Clear)되어 비워지므로 우측의 FOR 루프가 돌 원천 데이터가 사라지는 모순이 발생하기 때문이며, LET 은 포맷 전에 데이터를 임시 대피시켜 해결합니다.**
- **모순의 원리**: `itab = VALUE #( FOR line IN itab ( result = line-val * 2 ) )`
  이 구문이 실행되면, 시스템은 대입을 수행하기 위해 우측 식을 평가하기 시작하는데, 대입 기호 `=` 과 VALUE 규칙에 따라 **좌측의 `itab` 테이블이 먼저 텅 비워집니다.**
  이로 인해 우측 식의 `FOR line IN itab` 은 루프를 채 돌기 시작하기도 전에 순회할 원천 `itab` 의 알맹이가 다 증발해 버려 빈 껍데기 루프만 돌고 값이 모두 유실되는 치명적인 데이터 정합성 파괴 버그가 납니다.
- **`LET` 을 통한 해결 원리**:
  **`LET` 식은 대입문 가동 및 좌측 테이블 포맷이 일어나는 시점보다 "무조건 가장 최우선"으로 동작하여 데이터를 사전 대피시킵니다.**
  `itab = VALUE #( LET lt_temp = itab IN FOR line IN lt_temp ... )` 
  이렇게 짜주면, 좌측 테이블이 포맷되기 직전 찰나에 `itab` 원본의 복사본을 임시 주머니 변수인 `lt_temp` 에 꽁꽁 싸매어 대피시켜 둡니다. 그 후 좌측 `itab` 이 비워지더라도 우측 `FOR` 는 대피소인 `lt_temp` 를 안전하게 돌며 연산을 완수하고 새 테이블을 빚어낼 수 있어 모순이 해결됩니다.
