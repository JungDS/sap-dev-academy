# CH13_REWRITE - Open SQL 2차: JOIN·집계 v1

> 목적: `content/abap/CH13`의 8개 레슨을 "IT 비전공자 입문자가 왜 필요한지부터 실습 확인까지 직관적으로 이해하고 따라갈 수 있는 완성 강의자료" 수준으로 재작성한 기준 원고다. 이 문서는 아직 `content/abap` 원본을 덮어쓴 결과가 아니라, 재작업 판정 이후 실제 반영 전 검수 가능한 v1 기준안이다.

## CH13 전체 설계

CH13의 한 문장 목표는 "단일 테이블 조회의 한계를 넘어, 물결표 별칭 구분자(`~`)를 사용하는 classic INNER/LEFT OUTER JOIN과 GROUP BY/HAVING/ORDER BY 집계 체인, 그리고 FOR ALL ENTRIES 내부 목록 필터 성능 가이드를 활용하여 데이터베이스 레벨에서 집계 연산이 수행되는 공연별 예매현황 종합 리포트를 완성한다"이다.

IT 비전공자 입문자는 LEFT JOIN 하단에 WHERE 오른쪽 테이블 필터를 수동 추가해 보존 데이터를 증발시키는 버그를 내고, FAE 가드(`IS NOT INITIAL`)를 누락해 전체 조회 폭탄을 유발하며, FAE의 DISTINCT 중복 제거 성질을 잊어 데이터를 손실하는 설계를 범한다.
따라서 본 챕터는 다음과 같은 불편 해결의 단계적 설계로 전개한다.

1. **여러 테이블 흩어짐의 비효율**: 예매 테이블에는 `concert_id`만 있고 정원이나 아티스트명은 공연 마스터 테이블에 가 있어, 필요한 데이터를 보려면 매번 LOOP 안에서 쿼리를 난발해야 하는 무거운 연산 고통 $\rightarrow$ 키로 두 장부를 하나로 묶는 **JOIN** 도입.
2. **양방향의 짝맞춤 (INNER JOIN)**: 두 테이블에 매칭 키가 칼같이 다 존재하는 짝꿍 데이터만 뽑아내는 **INNER JOIN**과 classic **`~`** (물결표) 별칭 매핑법.
3. **왼쪽 보존의 규칙 (LEFT OUTER JOIN)**: 예매 내역이 단 1건도 없는 신규 공연이라도 0건 예약으로 보존해 리포트에 띄우는 **LEFT OUTER JOIN** 명세.
4. **오른쪽 WHERE 수축 함정**: LEFT JOIN 후 WHERE 절로 오른쪽 필드를 함부로 걸었을 때, NULL 매칭 행들이 싹 걸러져 INNER JOIN으로 강제 쪼그라드는 **WHERE 필터 오염 버그** 구조 분석.
5. **요약 봉투 조립 (GROUP BY & 집계)**: 수만 건의 예매 행들을 공연 ID 그룹으로 묶는 **GROUP BY**와 그룹 외 평필드의 **집계 함수(SUM/COUNT/MAX/MIN)** 감싸기 강제 규칙.
6. **2단 가드라인 (HAVING)**: 그룹을 묶기 전 개별 행을 발라내는 **WHERE**와 묶은 후 집계 결과를 검문하는 **HAVING**의 순서도 정비.
7. **줄 맞춤의 원칙 (ORDER BY)**: DB 인덱스를 활용해 조회 결과 정렬 순서를 강제하는 **ORDER BY** 다중 필드 매치.
8. **내 주머니 카드로 조회 (FOR ALL ENTRIES)**: 메모리 내부 테이블의 값을 조건 인수로 전달하는 FAE 문법 및 FAE 적용 시 반드시 수반되어야 할 **`IS NOT INITIAL`** 안전장치와 FAE 특유의 **자동 DISTINCT(중복 제거)** 제거 원리 폭로.
9. **3대 조회 전략 이정표**: JOIN, FAE, ABAP 루프 중 성능과 실무에 가장 적절한 아키텍처 판단 기준 카드 정립.
10. **예매현황 리포트 실습**: ZCONCERT 와 ZBOOKING을 LEFT JOIN하고 GROUP BY SUM으로 묶어 총 예약석과 잔여석을 한 표로 완성해 SALV로 띄우는 콘서트앱 5단계 실습.

### 범위와 금지선

- **R6 classic-first 철저 수호**: 
  - `FROM ztperson AS p INNER JOIN ztdept AS d ON p~dept_id = d~dept_id` 와 같이 classic 조인 구문과 `~` 물결표 별칭 체결을 엄수한다. (modern 구문의 콤마 필드 분리나 호스트 변수 `@` 대입, inline data declaration 및 modern 조인 지시어는 철저히 배제한다.)
  - FAE 호출 시 classic host variable 이식 없이 `lt_dept-dept_id` 형태로 매칭하는 classic 구문을 엄수한다.
- **변수 스코프 접두어**: 모든 스크립트는 전역 범위로 간주하며, 일반 변수는 `gv_`, 구조체는 `gs_`, 내부 테이블은 `gt_`, 구조체 타입 선언은 `ty_` 접두어를 엄수한다.
- **이름 풀 준수**: 예제 이름은 지정된 이름 풀에서만 가져오며, 주인공은 항상 **정훈영**으로 통일한다.

## CH13-L01 - INNER JOIN 기본 개념과 구현

### 왜 필요한가

우리가 지금까지 하나의 테이블에서 데이터를 SELECT 문으로 긁어오는 법을 공부했다. 
하지만 실제 현업 데이터베이스 구조는 성격에 따라 테이블이 잘게 찢어져 있다. 

예를 들어, `ZBOOKING` 예매 대장에는 공연 번호(`C001`)만 외롭게 적혀 있고, 그 공연의 가수 이름이나 전체 좌석 정원(`1,000석`) 정보는 마스터 대장인 `ZCONCERT` 테이블에 가 있다. 
사용자에게 "공연명과 예약 신청석 수, 정원을 한 표에 합쳐서 리포트로 보여주세요" 라는 지시가 떨어졌을 때, 두 장부를 합치는 방법을 모르면 개발자는 예매 테이블을 루프 돌리며 그 안에서 매번 SELECT 문을 쳐서 마스터 데이터를 긁어와 합쳐주는 끔찍하게 느린 노가다 코딩을 해야 한다. 

데이터베이스 창고 단선에서, **두 장부에 적힌 공통 연결 키(공연 ID)를 가이드로 삼아 두 테이블을 옆으로 찰떡같이 묶어서 한 번에 완성형 긴 줄로 읽어오는 조립 기술**이 필요하다. 그것이 **[[JOIN]]** (조인)의 시작이다.

### 무엇인가

#### 1. JOIN (조인)의 직관적 비유
조인은 **"두 장부를 공통 연결고리(키)로 나란히 묶어 단 하나의 큰 엑셀 시트처럼 옆으로 길게 이어 붙이는 기술"** 이다.
- **연결고리**: 사람 명단의 '부서 번호'와 부서 대장의 '부서 번호'가 같은 것끼리 나란히 짝지어 한 줄로 합친다.

#### 2. INNER JOIN (내부 조인)의 정의
- **INNER JOIN**: **"양쪽 장부 모두에 매칭되는 짝이 명확히 존재하는 행들만 최종 합격시켜 남기는 조인"** 이다. 
- *예시*: 사람 테이블에 부서 코드가 99번으로 적혀 있는데 부서 마스터 테이블에 99번 부서가 아예 존재하지 않는다면, 짝이 맞지 않으므로 그 사람 데이터는 최종 리포트 명단에서 완전히 탈락되어 빠진다.

#### 3. Classic INNER JOIN 문법 규칙
*classic SQL은 modern 문법과 달리 콤마가 없고, 필드를 별칭과 물결표(`~`)로 체결한다.*
- **별칭 지정 (`AS 별칭`)**: `FROM ztperson AS p` 처럼 테이블 이름 뒤에 간단한 닉네임을 달아준다.
- **물결표 구분자 (`~`)**: `p~persid` 와 같이 **`별칭~필드명`** 기호로 연결해 해당 컬럼이 어느 테이블에서 온 것인지 좌표를 찍어준다.
- **합체 기준 (`ON`)**: `ON p~dept_id = d~dept_id` 구절을 작성해 두 장부를 붙일 접착제 키를 정의한다.
- **중복 뻥튀기 방지**: 만약 매칭할 기준 키가 공연 ID(`concert_id`)와 회차 번호(`perf_no`) 2개라면, `ON` 절 우측에 **`AND` 로 2개 키를 다 적어주어야 정합성이 보장**된다. 키 중 일부를 누락하고 1개 키로만 조인을 걸면 데이터 짝이 문어발식으로 중복 매칭되어 결과 행 수가 뻥튀기(의도치 않은 데이터 증발/복제)되는 버그를 낸다.

### 어떻게 확인하는가

에디터에서 ZBOOKING과 ZCONCERT를 엮는 classic INNER JOIN 구문을 작성하고 조인을 검증한다.

```abap
REPORT zch13_l01_inner_join.

" 두 장부의 합작 컬럼을 고스란히 담아낼 구조체 정의
TYPES: BEGIN OF ty_out,
         booking_id TYPE zbooking-booking_id,
         concert_id TYPE zbooking-concert_id,
         artist     TYPE zconcert-artist,
       END OF ty_out.

DATA gt_out TYPE TABLE OF ty_out.

START-OF-SELECTION.
  " [classic 조인 철칙] 골뱅이@ 금지, 콤마 분리 금지, 별칭 연결은 물결표(~) 사용!
  SELECT b~booking_id b~concert_id c~artist
    FROM zbooking AS b
    INNER JOIN zconcert AS c
      ON b~concert_id = c~concert_id
    INTO CORRESPONDING FIELDS OF TABLE gt_out.

  IF sy-subrc = 0.
    WRITE: / '조인 성공 총 건수(sy-dbcnt):', sy-dbcnt.
  ENDIF.
```

#### 체험/시뮬레이터 설계 (JOIN 짝 맞추기 보드)
- **프로세스 플로우**:
  1. 학습자가 좌측 [사람 카드]의 `dept_id = 99` 카드를 마우스로 집는다.
  2. 우측 [부서 대장] 테이블에는 10번(영업), 20번(개발) 부서만 등록되어 있다.
  3. [INNER JOIN 가동] 버튼을 클릭하면, 99번 카드가 짝을 찾으러 갔다가 짝을 만나지 못해 아래 쓰레기통으로 떨어져 탈락하고, 10번과 20번 짝꿍 데이터만 나란히 맞물려 완성 표 라인에 안착되는 모션을 본다.
- **상태 및 데이터**:
  - 부서 정보를 고의로 숨김 -> 짝이 없는 사람 행이 최종 `gt_out` 내부 테이블 결과에서 즉각 제거되어 `sy-dbcnt`가 차감 상태로 변동 렌더링.
- **피드백**: INNER JOIN은 "양쪽 테이블 모두에 키 짝꿍이 확실히 숨 쉬고 있는 데이터만 살아남는 구조" 임을 시각적으로 각인시킨다.

### 실수/주의

- **복합 키 매칭 시 ON 조건 누락**: 테이블의 기본 키(Primary Key)가 2개 이상인 테이블을 조인하면서 귀찮다고 `ON`에 1개 키 조건만 쓰면, DB 엔진은 키가 다른 데이터까지 마구잡이로 옆으로 덧대어 1행의 데이터가 3~4행으로 폭증하는 치명적인 정합성 복제 버그를 초래하므로 모든 키 컬럼을 `AND`로 체결해야 한다.

### 정리

- **`INNER JOIN`** 은 두 테이블 간에 공통 키 짝꿍이 양쪽에 다 존재하는 데이터만 결과에 남기는 조인이다.
- classic SQL 표기는 공백 구분 및 **`별칭~필드`** 기호를 사용하여 매핑한다.
- 뻥튀기 버그를 막기 위해 `ON` 절에는 대상 테이블의 **기본 키 전체를 AND로 다 연결**해 준다.

## CH13-L02 - LEFT OUTER JOIN 기본 개념과 NULL 처리

### 왜 필요한가

짝이 있는 데이터만 걸러서 깔끔하게 가져오는 `INNER JOIN`을 마스터했다. 
하지만 현실 비즈니스 리포트 요구사항에서는 이 엄격한 짝맞춤 규칙이 오히려 큰 문제가 된다. 

"신규 개설한 공연 `C003` 번호가 있습니다. 이 공연은 방금 태어났기 때문에 아직 예매한 사람이 단 한 명도 없어서 `ZBOOKING` 예매 대장에는 관련 데이터가 단 1줄도 없습니다. 하지만 현업에서는 예매가 0건인 이 신규 공연 정보도 리포트 첫 줄에 누락 없이 **'C003 공연 / 예매합계 0석'** 이라고 떳떳하게 남겨서 보여달라고 합니다"
INNER JOIN을 쓰면 예매가 없다는 이유로 이 귀중한 신규 공연이 리포트에서 통째로 지워져 누락되는 중대한 업무 사고가 터진다. 
**짝이 없더라도 기준 테이블(왼쪽)의 명단은 철저히 100% 다 보존하여 살려두고, 짝이 없는 우측 빈칸은 상냥하게 빈 값(초기값)으로 채워 리포트를 서빙하는 보존형 조인 기술**이 요구된다. 그것이 **`LEFT OUTER JOIN`** (왼쪽 외부 조인)이다.

### 무엇인가

#### 1. LEFT OUTER JOIN의 원리와 NULL
- **왼쪽 보존**: `FROM` 절에 적힌 왼쪽(기준) 테이블의 명단은 짝의 유무를 불문하고 **무조건 전부 리포트에 잔류**시킨다.
- **NULL과 초기값 변환**: 매칭되는 오른쪽 테이블 정보가 없는 빈칸 영역은 데이터베이스상에서는 **`NULL`** (값 없음) 상태가 된다. 하지만 ABAP 내부 테이블로 이 데이터 화물이 넘어오는 순간, 아밥 엔진은 이를 감지해 자료형에 맞게 문자는 **빈칸(' ')**, 숫자는 **`0`** 이라는 친숙한 **초기값(Initial Value)** 으로 변환 수납해 준다.

#### ⚠️ [LEFT OUTER JOIN을 무력화시키는 WHERE절 필터 함정]
- *실무 현업 리포트 개발 시 10명 중 8명이 겪고 퇴근을 못 하는 치명적인 단골 함정이다.*
- 왼쪽을 살리려고 `LEFT OUTER JOIN`을 걸어놓고, 아랫줄 `WHERE` 절에다가 오른쪽 테이블의 조건(예: `WHERE b~status = 'N'`)을 기입해 대조하면 대참사가 터진다.
- **이유**: 예매가 없어 오른쪽이 NULL(빈칸)로 살아남은 `C003` 공연 행이, `WHERE b~status = 'N'` 이라는 필터 검문을 받는 순간 "빈칸은 'N'과 같지 않다"며 조건 불일치로 **필터 탈락**해 버린다.
- **결과**: 결국 힘들게 걸어놓은 LEFT OUTER JOIN이 찌그러져 **INNER JOIN과 완전히 똑같은 결과**로 오염된다.
- **해결책**: 오른쪽 테이블의 필터 조건은 절대로 WHERE에 쓰지 말고, **`ON` 절의 조인 결합부 우측에 `AND b~status = 'N'` 형태로 도킹 체결**해야 왼쪽 보존의 법칙이 온전하게 수호된다.

### 어떻게 확인하는가

에디터에서 WHERE절 필터 함정이 개입했을 때와 ON 절에 가두었을 때의 행 수 변화를 검증한다.

```abap
REPORT zch13_l02_left_join.

TYPES: BEGIN OF ty_out,
         concert_id TYPE zconcert-concert_id,
         artist     TYPE zconcert-artist,
         booking_id TYPE zbooking-booking_id,
       END OF ty_out.

DATA gt_out TYPE TABLE OF ty_out.

START-OF-SELECTION.
  " 1. LEFT OUTER JOIN 체결 (예매가 없는 공연도 다 살림)
  " 2. [함정 가드] 오른쪽 테이블 조건인 status <> 'C'를 WHERE가 아닌 ON 절 뒤에 도킹!
  SELECT c~concert_id c~artist b~booking_id
    FROM zconcert AS c
    LEFT OUTER JOIN zbooking AS b
      ON c~concert_id = b~concert_id
     AND b~status    <> 'C' " 여기에 적어야 NULL(예매 0건) 행이 살아남음!
    INTO CORRESPONDING FIELDS OF TABLE gt_out.

  " 만약 WHERE b~status <> 'C' 라고 적으면 예매 없는 C003 공연은 NULL이라 즉사 탈락함!
  WRITE: / '조회된 공연 행 수:', sy-dbcnt.
```

#### 체험/시뮬레이터 설계 (LEFT 보존 스위치)
- **프로세스 플로우**:
  1. 학습자가 [조인 스위치]를 [INNER JOIN]으로 둔다. 예매가 전혀 없는 공연 `C003` 행이 리포트 캔버스에서 흔적도 없이 소실된다.
  2. [LEFT OUTER JOIN] 스위치로 변경한다. `C003` 행이 부활해 리포트 좌측에 드러나고, 우측의 예매 번호 칸은 텅 빈 초기값(' ') 상태로 반짝 렌더링된다.
  3. 이 상태에서 [WHERE 절에 b~status 필터 가동] 단추를 누른다. 부활했던 `C003` 행이 쿼리 검문선에서 붉은 불을 내며 다시 바스러져 사라지는 함정을 본다.
- **상태 및 데이터**:
  - `status <> 'C' 가 ON 절에 도킹됨` -> C003 행 잔류 통과.
  - `status <> 'C' 가 WHERE 절에 쓰임` -> C003 행 즉각 소멸 탈락.
- **피드백**: "LEFT OUTER JOIN에서 오른쪽 테이블 조건 필터링은 반드시 ON 절에 함께 엮어 수호해야 한다" 는 철칙 경고를 제시한다.

### 실수/주의

- **오른쪽 테이블 조건의 WHERE절 오기입**: "LEFT JOIN을 썼는데도 예매 없는 공연이 왜 리포트에서 다 사라졌는지 모르겠습니다" 라며 선배들에게 질문을 던지는 버그는 백이면 백 WHERE 절에 오른쪽 조건 컬럼을 대입했기 때문이다. 반드시 ON의 우측 결합부로 이동 적재해야 한다.

### 정리

- **`LEFT OUTER JOIN`** 은 매칭 짝꿍이 우측에 없더라도 기준(왼쪽) 테이블 행을 전부 살려내는 보존 조인이다.
- 매칭 실패로 비어버린 우측 공간은 DB 상에선 NULL이며, 아밥으로 넘어오면 빈칸이나 0 같은 **초기값**이 된다.
- 오른쪽 테이블의 조건절 가드는 **반드시 ON 절 내부에 엮어야** LEFT JOIN 보존 뼈대가 수축 붕괴되지 않는다.

## CH13-L03 - GROUP BY와 Aggregate

### 왜 필요한가

두 테이블을 조인해서 모든 상세 데이터를 한 줄로 합쳐 가져오는 데 성공했다. 
하지만 비즈니스 보고의 최종 종착지는 상세 내역 나열이 아닌 **"요약 통계"** 다. 

임원들이 원하는 보고서는 "예매 건수 10만 줄을 다 보여달라" 가 아니라, 딱 5줄로 정제된 **"공연별 총 예매 좌석 수 합계"** 와 **"공연별 총 예매 건수"** 다. 
이 10만 행의 데이터를 아밥 메모리로 다 긁어와서 루프 돌며 더하는 코딩은 서버 리소스를 극심하게 낭비한다. 
데이터베이스 레벨에서 **데이터 행들을 동일한 공연 그룹별로 한 주머니에 모아 묶어두고, 그 안에서 바로 합산(SUM)하고 개수(COUNT)를 세어 클린하게 단 한 행의 요약본으로 추출하는 압축 기술**이 필요하다. 그것이 **`GROUP BY`** 와 **[[Aggregate]]** (집계 함수)이다.

### 무엇인가

#### 1. GROUP BY (그룹화) 와 요약 주머니
- **GROUP BY**: 특정 기준 컬럼(예: `concert_id`)의 값이 같은 행들을 메모리상에서 하나의 끈으로 질질 묶어 **그룹 봉투 안에 쑤셔 넣는 동작**이다.

#### 2. Aggregate Function (집계 함수)의 종류
그룹 봉투 내부에 적재된 다중 행 데이터를 요약하는 DB 엔진의 정규 함수들이다.
- **`COUNT( * )`**: 그룹 내의 총 행 개수를 카운트한다.
- **`SUM( 필드 )`**: 지정된 컬럼 값들을 전부 더해 총합을 낸다. (금액, 수량용).
- **`MAX( 필드 ) / MIN( 필드 )`**: 그룹 내의 최고값 / 최저값을 솎아낸다.
- **`AVG( 필드 )`**: 평균값을 산출한다.

#### ⚠️ [데이터베이스 컴파일러의 절대적 평필드 차단 규칙]
- *GROUP BY 적용 시 문법적으로 어기면 즉사하는 철칙이다.*
- `SELECT` 목록에 올리는 필드 중, **`GROUP BY` 절에 적지 않은 평필드(일반 컬럼)가 존재한다면 그 필드는 100% 반드시 집계 함수(`SUM` 등)로 감싸여 보호되어야만 한다.**
- *이유*: 공연 ID(`c~concert_id`)로 그룹을 묶었는데, 집계 함수도 안 씌우고 GROUP BY에도 없는 일반 고객명(`b~name`)을 SELECT에 적어놓으면, DB 엔진은 "공연 ID 하나에 고객명 100명이 엉켜 있는데 화면에 누구 이름 1개를 출력해야 하냐?" 라고 모호성 에러를 내며 빌드를 폭사시킨다.

#### 3. 중복 행의 물리적 제거: SELECT DISTINCT
집계 연산이나 덧셈 계산은 전혀 필요 없고, 단순하게 "우리 예매 대장에 단 한 번이라도 예매를 등록했던 고객들의 사번 목록(중복 제거)"만 깔끔하게 뽑고 싶다면, 굳이 무거운 GROUP BY 대신 SELECT 뒤에 **`DISTINCT`** (디스틴트) 키워드 한 단어만 명시해 주면 중복 행이 깔끔하게 지워져서 반환된다.

### 어떻게 확인하는가

에디터에서 GROUP BY와 SUM 집계를classic 쿼리로 엮어 요약 값을 산출한다.

```abap
REPORT zch13_l03_groupby.

TYPES: BEGIN OF ty_sum,
         concert_id TYPE zbooking-concert_id,
         cnt        TYPE i,
         total      TYPE i,
       END OF ty_sum.

DATA gt_sum TYPE TABLE OF ty_sum.

START-OF-SELECTION.
  " [평필드 가드 철칙] GROUP BY에 적힌 concert_id 외의 모든 SELECT 필드는
  " COUNT나 SUM과 같은 집계 함수로 꽁꽁 싸매어 보호 처리 필수!
  SELECT concert_id 
         COUNT( * )      AS cnt
         SUM( seats )    AS total
    FROM zbooking
    INTO CORRESPONDING FIELDS OF TABLE gt_sum
    GROUP BY concert_id. " 이 그룹 키 외에는 다 집계 처리됨

  IF sy-subrc = 0.
    " 10만 행 데이터가 공연 ID 개수만큼 단 몇 행으로 축소 출력됨
    WRITE: / '집계된 요약 그룹 행 수:', sy-dbcnt.
  ENDIF.
```

#### 체험/시뮬레이터 설계 (행 묶기 실험실)
- **프로세스 플로우**:
  1. 학습자가 [GROUP BY concert_id] 레버를 당긴다.
  2. 사방에 흩어져 있던 ZBOOKING의 예매 로우 박스 6개가 다가와 `C001` 그룹 주머니, `C002` 그룹 주머니 속으로 분리되어 골인한다.
  3. 주머니 입구가 묶이면서, 주머니 표면에 `COUNT: 4건, SUM: 12석` 이라는 요약 딱지가 찰칵 붙으며 단 2개의 결과 행으로 압축 변환되는 모션을 목격한다.
- **상태 및 데이터**:
  - `SELECT c~concert_id, b~name(평필드) GROUP BY c~concert_id` 세팅 가동 -> "Field name is not in GROUP BY clause or aggregate function" 컴파일 빨간불 점등 상태 확인.
- **피드백**: GROUP BY에 없는 필드는 무조건 집계 함수로 감싸거나 GROUP BY절에 같이 묶어주어야 컴파일러 통과가 됨을 도식으로 매칭한다.

### 실수/주의

- **SELECT 목록에 비그룹 평필드 방치**: GROUP BY 코딩을 짤 때 가장 많이 저지르는 구문 에러다. SELECT 절에 공연 ID와 좌석 수 합계를 쓰고 아무 생각 없이 공연 ID로만 묶었을 때, 깜빡하고 옆에 적어둔 공연 날짜(`perf_date`) 필드를 집계 처리하지 않아 활성화 실패 경고를 마주하므로 꼼꼼히 그룹 절을 대조해야 한다.

### 정리

- **`GROUP BY`** 는 동일한 값을 가진 행들을 묶어 그룹 봉투 속에 격리 압축하는 기술이다.
- 그룹 내의 덧셈은 **`SUM`**, 카운트는 **`COUNT`**, 최대/최소는 **`MAX/MIN`** 집계 함수로 처리한다.
- SELECT 목록 중 GROUP BY 키가 아닌 필드는 **반드시 집계 함수로 감싸주어야** 문법 에러를 방어한다.

## CH13-L04 - HAVING과 집계 조건

### 왜 필요한가

우리가 `WHERE` 절에 필터 조건을 꽂는 법도 배웠고, `GROUP BY`로 그룹을 묶어 합산(`SUM`)하는 법도 마스터했다. 
하지만 비즈니스 업무 리포트의 최종 필터 요건은 한 단계 더 까다롭다. 

"예매 좌석 수의 합계(`SUM`)를 다 구한 다음에, **총 예매 합계가 1,000석 이상으로 매진 직전인 인기 공연 그룹**만 솎아내서 리포트에 올려주세요"
이 요구사항을 받고 평소대로 `WHERE SUM(seats) >= 1000` 이라고 적으면, 데이터베이스 엔진은 즉사하며 구문 에러를 뿜는다. 
왜냐하면 `WHERE` 절은 그룹을 묶기 **전**에 작동하는 거름망이라, 그룹이 묶여서 더해진 최종 합계 점수(`SUM`)가 몇 점인지 아직 알 도리가 없기 때문이다. 
**그룹을 다 지어 집계 연산을 끝내놓은 그 최종 그룹 결과를 대상을 삼아 2차로 정밀 검문을 수행하는 그룹 전용 필터 기술**이 요구된다. 그것이 **`HAVING`** (해빙)이다.

### 무엇인가

#### 1. WHERE 와 HAVING의 확실한 결투 기준
두 거름망은 작동하는 시간 대역과 검문 대상이 물리적으로 완전히 격리되어 있다.

| 필터 키워드 | 작동 시점 | 검문 대상 | 예시 용도 |
| --- | --- | --- | --- |
| **`WHERE`** | GROUP BY로 그룹 묶기 **전** | 테이블의 **개별 로우 1행** | 20세 이상 성인 고객만 먼저 수용 |
| **`HAVING`** | GROUP BY로 그룹 묶은 **후** | 집계 완료된 **그룹 요약본** | 총 예매석 합계(SUM)가 10석 이상인 그룹만 통과 |

#### 2. 두 번 거르는 2단 필터 파이프라인 정석 구문
- **순서 규칙**: SQL 문장 작성 시 순서는 반드시 **`WHERE` $\rightarrow$ `GROUP BY` $\rightarrow$ `HAVING`** 체인을 따라야 한다.
  ```abap
  SELECT concert_id SUM( seats ) AS total
    FROM zbooking
    WHERE status <> 'C'           " 1단계: 취소 건은 그룹 묶기 전에 아예 버림 (개별 행 필터)
    GROUP BY concert_id
    HAVING SUM( seats ) >= 1000.  " 2단계: 합산 끝난 그룹 중 1000석 넘은 그룹만 잔류 (그룹 필터)
  ```

### 어떻게 확인하는가

에디터에서 2단 거름망 구문을 빌드하여 개별 행 필터와 그룹 합계 필터의 수동 검수를 집행한다.

```abap
REPORT zch13_l04_having.

TYPES: BEGIN OF ty_sum,
         concert_id TYPE zbooking-concert_id,
         total      TYPE i,
       END OF ty_sum.

DATA gt_sum TYPE TABLE OF ty_sum.

START-OF-SELECTION.
  " WHERE로 1차 필터링 -> GROUP BY로 묶기 -> HAVING으로 집계 결과 2차 필터링 체인 작동
  SELECT concert_id SUM( seats ) AS total
    FROM zbooking
    INTO CORRESPONDING FIELDS OF TABLE gt_sum
    WHERE status <> 'C'
    GROUP BY concert_id
    HAVING SUM( seats ) >= 10. " 요약 합계가 10석 이상인 그룹만 생존!

  IF sy-subrc = 0.
    WRITE: / '생존한 인기 공연 그룹 수:', sy-dbcnt.
  ENDIF.
```

#### 체험/시뮬레이터 설계 (WHERE·HAVING 2단 필터 파이프라인)
- **프로세스 플로우**:
  1. 학습자가 [1단계: WHERE status <> 'C'] 가동 스위치를 누른다.
  2. ZBOOKING의 6행 중 취소(`status = 'C'`) 딱지가 붙은 1행이 그룹 벨트에 올라타기도 전에 먼지가 되어 사그라진다.
  3. [2단계: GROUP BY] 톱니바퀴가 돌아 남은 5행이 공연별로 합산 묶인다.
  4. [3단계: HAVING SUM >= 10] 검문망이 가동되자, 합산 값이 5석밖에 안 되는 비인기 공연 그룹이 검문 바리케이드에 걸려 튕겨 나가고 인기 그룹만 최종 통과함판에 떨어지는 3차원 전개를 체험한다.
- **상태 및 데이터**:
  - `WHERE SUM(seats) >= 10` 기입 후 컴파일 -> "Aggregate function not allowed in WHERE clause" 에러 하이라이트.
- **피드백**: "WHERE는 그룹 전 개별 행의 거름망이고, HAVING은 그룹 완성 후 집계 결과의 최종 가드" 임을 2단계 계단 벨트형 그래픽으로 보증한다.

### 실수/주의

- **집계 조건을 WHERE 절에 적는 실수**: SQL 입문자들이 가장 많이 던지는 버그 코드다. `WHERE SUM(seats) >= 10` 처럼 기재하면 아예 컴파일 에러를 내며 빌드가 차단되므로, 합계/개수 비교는 무조건 **`HAVING`** 영역에 수납해야 한다.

### 정리

- **`WHERE`** 는 그룹을 짓기 전 개별 행을 대상을 삼아 거르는 1차 필터 장치다.
- **`HAVING`** 은 그룹화 연산이 완료된 후, 집계 결과(`SUM`, `COUNT` 등)를 검문하는 2차 그룹 전용 거름망이다.
- 반드시 **`WHERE` $\rightarrow$ `GROUP BY` $\rightarrow$ `HAVING`** 의 아키텍처적 조립 순서를 준수한다.

## CH13-L05 - ORDER BY 정렬 조회

### 왜 필요한가

두 테이블을 묶고 집계하여 근사한 요약 데이터를 산출하는 파이프라인을 다 꿰뚫었다. 
하지만 이 결과 표를 사용자 화면에 띄웠을 때, 정렬 기준 없이 뒤죽박죽 인쇄된다면 업무적 가치는 뚝 떨어진다. 

"예매율이 가장 높은 공연부터 순서대로 보여주세요", "가장 최근에 열린 예매 일자순으로 정렬하되, 예매 일자가 같으면 예매 번호 순서대로 계층 정렬해 주세요" 
조회된 데이터를 메모리에 적재한 뒤, 아밥 코드로 `SORT gt_table` 명령을 추가로 때려 정렬할 수도 있지만, 이왕이면 **데이터베이스 창고에서 데이터를 꺼내올 때부터 정렬 기준에 맞춰 완벽히 줄이 선 상태로 고속 수납해 오는 스마트한 쿼리 정렬 기술**이 요망된다. 그것이 **`ORDER BY`** (오더 바이)이다.

### 무엇인가

#### 1. ORDER BY 의 정렬 규칙과 기어
- **정렬 방향**:
  - **`ASCENDING`** (오름차순 - 디폴트): 숫자 1부터 10으로, 한글 가부터 하로 커지는 정방향 정렬이다. (생략 가능).
  - **`DESCENDING`** (내림차순): 큰 값부터 거꾸로 내려가는 정렬이다. (매출 상위, 최신순에 사용).
- **기본 키 정렬**:
  - `ORDER BY PRIMARY KEY` 지시어를 쓰면, 데이터 사전에 등록된 테이블의 기본 키(PK) 컬럼 순서대로 알아서 자동 정렬해 들여온다. (단, 두 테이블을 엮은 JOIN 결과문에는 이 PK 자동 정렬을 쓸 수 없어 컬럼명을 명시해야 한다).

#### 2. 여러 정렬 기준의 꼬리물기 우선순위
- **다중 정렬**: `ORDER BY age DESCENDING name ASCENDING`
- **거동 규칙**: 소스 코드의 **왼쪽에 먼저 적어둔 필드일수록 절대적인 우선순위**를 점유한다. 즉, 나이 순서대로 먼저 내림차순 정렬을 집행하고, **나이가 완벽히 똑같은 동률 데이터가 존재할 때만** 그 동률 그룹 안에서 이름 오름차순 정렬을 2차로 집행해 교통정리를 완성한다.

#### 3. ORDER BY가 누락되었을 때의 무서운 무질서
- **순서 무보장**: 쿼리 뒤에 `ORDER BY`를 명시하지 않으면, 조회 결과의 정렬 순서는 데이터베이스 내부의 데이터 분산 상태에 따라 **내 맘대로 무질서하게 섞여서 반환**된다. 오늘 우연히 1번부터 이쁘게 정렬되어 나왔다고 해서, 내일도 그렇게 나오리라는 보장이 절대 없다. 순서가 의미 있는 화면은 반드시 ORDER BY를 박아야 한다.

### 어떻게 확인하는가

에디터에서 ORDER BY 다중 필드 정렬을 구현하고 우선순위 순번 거동을 대조 확인한다.

```abap
REPORT zch13_l05_orderby.

DATA gt_book TYPE TABLE OF zbooking.

START-OF-SELECTION.
  " [classic ORDER BY] 호스트 골뱅이@ 및 콤마 없이 정렬 기준 나열
  SELECT * FROM zbooking
    INTO TABLE gt_book
    ORDER BY concert_id ASCENDING  " 1순위: 공연 ID로 먼저 정렬
             seats      DESCENDING. " 2순위: 공연 ID가 같으면 신청 좌석 많은 순서대로!

  " sy-dbcnt는 정렬로 인해 행이 탈락하지 않으므로 그대로 전체 건수가 반환됨
  WRITE: / '정렬 완료 수입 행 수:', sy-dbcnt.
```

#### 체험/시뮬레이터 설계 (정렬 우선순위 조작기)
- **프로세스 플로우**:
  1. 학습자가 [1순위: concert_id ASC] 와 [2순위: seats DESC] 레일 노드를 세팅한다.
  2. 예매 데이터들이 레일 위를 지나며 `C001` 공연 데이터끼리 먼저 한 블록으로 묶인다.
  3. 그 `C001` 블록 안에서 좌석 수 `5석`, `3석`, `1석` 순서대로 내림차순 미끄러져 재배치되는 정렬 모션을 확인한다.
- **상태 및 데이터**:
  - `ORDER BY` 기입 안 함 -> F8 누를 때마다 DB 저장 순서대로 뒤섞여서 무질서하게 출력되는 불안정 텍스트 렌더링.
- **피드백**: "정렬은 순서 보장의 기본 철칙이며, 앞에 쓴 필드가 왕권을 쥐는 다단계 정렬 구조" 임을 가이드한다.

### 실수/주의

- **인덱스 없는 대량 데이터의 무분별한 ORDER BY**: 정렬은 DB 엔진 입장에서도 메모리를 사용해 데이터를 뒤집는 꽤 비싼 연산이다. 데이터가 1,000만 건인 대형 테이블에 인덱스(Index) 가 안 깔려있는 텍스트 필드를 기준으로 ORDER BY를 때리면 DB 서버 과부하의 원인이 되므로, 성능 튜닝 검토를 동반해야 한다.

### 정리

- **`ORDER BY 필드 [ASCENDING|DESCENDING]`** 구문으로 DB 조회 단계부터 정렬 상태로 수납한다.
- 여러 정렬 기준 기입 시, **왼쪽에 먼저 명시한 필드**가 최우선 순위로 정렬을 지배한다.
- 정렬을 선언하지 않으면 결과 순서는 결코 보장되지 않으므로, 순서가 중요한 업무 화면은 ORDER BY를 명시한다.

## CH13-L06 - FOR ALL ENTRIES 사용 기준

### 왜 필요한가

지금까지 우리는 흩어진 테이블들을 하나로 엮는 조인(JOIN)과 집계 기술을 습렵했다. 
하지만 실무 현장에서는 DB 테이블끼리 엮는 것 외에 다음과 같은 난처한 요구사항이 수반된다.

"화면이나 이전 로직에서 가공하여 메모리에 올려둔 **'10명의 특별 VIP 고객 사번 리스트 내부 테이블'** 이 이미 프로그램 RAM에 떠 있습니다. 이 메모리에 기적재되어 있는 10명의 사번 값을 검색 조건인 `WHERE` 절의 인수로 전달하여, DB의 `ZBOOKING` 테이블에서 이 10명에 속하는 예매 정보만 쏙 빼서 골라 조회해 오고 싶습니다"
이미 메모리에 기차로 떠 있는 이 데이터 덩어리는 DB 테이블이 아니기 때문에 조인(JOIN)을 걸 수도 없고, 그렇다고 10번 루프 돌며 SELECT를 때리자니 성능이 박살 난다. 
**메모리에 실려 있는 내부 테이블 목록을 쿼리의 동적 조건절로 활용하여, 대량의 DB 테이블에서 해당 조건에만 매칭되는 데이터만 한꺼번에 고속 인양해 오는 특수 필터 기술**이 필수적이다. 그것이 **[[FOR ALL ENTRIES]]** (포 올 엔트리즈, FAE) 이다.

### 무엇인가

#### 1. FOR ALL ENTRIES (내부 테이블 대입 조회)
- **FAE의 기본 거동**: `FOR ALL ENTRIES IN 내부테이블` 을 SELECT 문장 중간에 장착하면, DB 엔진은 내부 테이블에 탑재된 모든 행의 조건 키 값들을 모아서 하나의 동적 IN 조건 묶음으로 컴파일해 DB 조회를 일괄 집행한다.

#### ⚠️ [FAE를 가동할 때 어기면 시스템 폭사를 초래하는 2대 철칙]

1. **`IS NOT INITIAL` 안전장치의 절대적 필수화 (1순위 함정)**:
   - *FAE를 쓰면서 이 가드를 누락하는 것은 대형 시스템 다운 사고를 일으키는 폭탄이다.*
   - FAE의 기준이 되는 내부 테이블(예: `lt_dept`)이 텅 빈 initial(0행) 상태에서 조회를 돌리면, DB 엔진은 **`WHERE` 절 조건을 통째로 없애고 무제한으로 전체 테이블 데이터를 풀스캔 조회**해 버린다.
   - **해결**: FAE를 호출하기 직전에 반드시 **`IF 내부테이블 IS NOT INITIAL. ... ENDIF.`** 보호막으로 SELECT 문 전체를 감싸서 빈 테이블일 때는 아예 DB 조회를 가동조차 못 하도록 사전 가드를 단단히 쳐야 한다. (업계 철칙).
2. **결과의 자동 중복 제거 (DISTINCT) 성질**:
   - FAE는 데이터를 메모리로 수집해 올 때, 시스템 기본 사양으로 **조회 결과 행들 중 완전히 똑같이 생긴 중복 행을 자동 압축해 날려버리는 `DISTINCT` 거동**을 탑재하고 있다.
   - *함정*: 만약 SELECT 목록에 예매 번호(`booking_id`) 같은 유니크한 기본 키 필드를 빠뜨리고 공연 ID(`concert_id`)만 적으면, 동일한 공연 ID를 가진 여러 사람의 예매 행들이 중복이라는 이유로 단 1행으로 압축 증발해 버려 데이터 누락 오차를 유발한다. 따라서 유니크 키 필드를 SELECT 대상에 필히 포함시켜 중복 판정을 회피해야 한다.

### 어떻게 확인하는가

에디터에서 FAE 전제 조건 검증과 SELECT 문을 클래식 구절로 엮어 빌드한다.

```abap
REPORT zch13_l06_for_all_entries.

" 조회 기준이 될 내부 테이블 선언 및 데이터 적재
DATA: BEGIN OF ty_dept,
        dept_id TYPE ztperson-dept_id,
      END OF ty_dept.
DATA: gt_dept   LIKE TABLE OF ty_dept,
      gs_dept   LIKE ty_dept,
      gt_person TYPE TABLE OF ztperson.

START-OF-SELECTION.
  " 기준 테이블 세팅 (예: 10번 부서만 타깃)
  gs_dept-dept_id = '10'.
  APPEND gs_dept TO gt_dept.

  " ------------------------------------------------------------------
  " [FAE 절대 수칙] 기준이 되는 gt_dept 테이블이 비어있지 않은지 철저 가드!
  " ------------------------------------------------------------------
  IF gt_dept IS NOT INITIAL.
    " classic FAE 매칭 (골뱅이@ 금지)
    SELECT persid name dept_id
      FROM ztperson
      INTO CORRESPONDING FIELDS OF TABLE gt_person
      FOR ALL ENTRIES IN gt_dept " 기준 내부 테이블 지목
      WHERE dept_id = gt_dept-dept_id. " FAE 조건 대조 체결
  ELSE.
    WRITE: / '기준 조건 테이블이 비어 있어 조회를 취소 우회합니다.'.
  ENDIF.
```

#### 체험/시뮬레이터 설계 (FAE 안전장치 시뮬레이터)
- **프로세스 플로우**:
  1. 학습자가 [gt_dept 테이블 비우기] 단추를 누른다. gt_dept 기차 칸이 하얗게 비워진다.
  2. 이 상태에서 [안전장치(IS NOT INITIAL) OFF] 해제 스위치를 내리고 [SELECT 가동]을 클릭한다.
  3. 경보음과 함께 빨간 경고등이 돌며, FAE 쿼리가 WHERE 필터를 무시하고 1억 건 대장 DB 전체를 풀스캔하여 서버 랙에 과부하 연기가 뿜어져 나오는 비상 상태 애니메이션을 관찰한다.
  4. [안전장치 ON]을 복구하자, 빈 테이블 상태에서 쿼리를 사전에 차단 우회하여 녹색 안전등이 켜지는 상태 변화를 대조한다.
- **상태 및 데이터**:
  - `IS NOT INITIAL` 미작동 시 -> DB 풀스캔 플래그 활성화 및 `sy-subrc = 0` (전체 조회 성공이라는 기만적 상태) 유발 데이터 출력.
- **피드백**: FAE의 빈 테이블 대입은 전체 데이터 풀스캔 폭탄으로 이어진다는 시스템 아키텍처 경고를 알린다.

### 실수/주의

- **FAE 호출 전 IS NOT INITIAL 생략**: 신입 개발자가 짠 소스 코드 감사(Code Review) 시 가장 먼저 뒤져보는 단골 검출 대상이다. FAE 전후에 `IS NOT INITIAL` 방어막이 안 씌워진 코드는 운영 배포 즉시 반려 처리되므로 몸에 각인시켜야 한다.

### 정리

- **`FOR ALL ENTRIES IN 내부테이블`** 은 메모리에 로드된 리스트 데이터를 검색 조건 인수로 삼아 DB를 일괄 필터링해오는 고속 조회 기술이다.
- 기준 내부 테이블이 비었을 때 전체 조회가 유발되는 덤프 재앙을 막기 위해 반드시 **`IS NOT INITIAL`** 가드로 쿼리를 꽁꽁 감싸 보위한다.
- FAE는 조회 결과의 중복 행을 자동 삭감하는 **DISTINCT** 성질을 품고 있으므로, 유니크 키 필드를 SELECT 목록에 매핑하여 데이터 누락 오차를 차단한다.

## CH13-L07 - JOIN / FAE / ABAP 처리 선택 기준

### 왜 필요한가

여러 갈래에 흩어져 있는 테이블 데이터를 하나로 엮어서 읽는 3대 무기인 **DB 조인(JOIN)**, **메모리 대입(FOR ALL ENTRIES)**, 그리고 아밥 코드 상에서 직접 버무리는 **ABAP 직접 루프 제어** 기술을 완벽하게 마스터했다. 

이제 리포트 소스 코드 설계를 작성하기 위해 연필을 잡았을 때, "지금 이 상황에서는 조인을 걸어야 하는가? 아니면 FAE로 찢어서 2번 읽어야 하는가? 아니면 아밥 루프로 합쳐야 하는가?" 명확한 **성공 성능 아키텍처 기준 기어**가 서 있지 않으면, 실무에서 가장 기피하는 성능의 주범인 'LOOP 내 SELECT 중첩 조회' 라는 최악의 똥코드를 생산하여 시스템 성능을 말아먹게 된다. 3대 전략의 명확한 판단 공식을 뇌에 이식해 두어야 한다.

### 무엇인가

#### 1. 데이터 합치기 3대 전략의 성능 및 적용 공식

우리는 개발을 시작할 때, 질문의 방향과 데이터가 얹혀있는 위치를 쳐다보고 전략 기어를 짚어 결정한다.

1. **1순위 정석 기어 $\rightarrow$ 무조건 DB JOIN (조인)**:
   - **원칙**: 합쳐야 할 대상 테이블들이 **둘 다 데이터베이스 디스크 창고 안에 고이 모셔져 있을 때**는 한 치의 망설임도 없이 **JOIN**을 걸어 DB 엔진이 한 번에 조립해 가져오도록 만든다. 데이터가 네트워크 통로를 타고 앱 서버로 움직이는 물리 이동량을 극적으로 줄여주므로 성능상 최고의 1순위 정석이다.
2. **2순위 기어 $\rightarrow$ FOR ALL ENTRIES (FAE)**:
   - **언제**: 합쳐야 할 조건 기준 목록이 DB가 아닌 **내 프로그램 메모리 내부 테이블(`gt_itab`)에 이미 실려 올라와 존재할 때**는 조인을 못 하므로 FAE를 쓴다. (단, FAE 안전장치 `IS NOT INITIAL`과 DISTINCT 누락 방어는 세트로 체결).
3. **최후의 보루 $\rightarrow$ ABAP 직접 루프 매칭**:
   - **언제**: 대조할 데이터가 단순 소량이거나, JOIN과 FAE를 쓰기에는 비즈니스 계산식이 너무 복잡해 DB 단에서 쿼리로 도저히 연산해 낼 수 없을 때, 각각 따로 SELECT해 가져와 아밥 프로그램 내부에서 `READ TABLE` 과 `LOOP` 구문 조합으로 머리를 써서 손수 짝을 맞춘다.

#### ⚠️ [절대 금지선: LOOP 내부의 중첩 SELECT (성능 안티패턴)]
- *실무 프로젝트 개발 표준 가이드라인에서 절대적으로 금지하는 범죄적 코딩이다.*
- 내부 테이블을 루프 돌면서, 그 루프 바퀴 수만큼 DB 창고로 SELECT 호출을 반복 왕복하는 중첩 조회 코드는 대량 배치 작동 시 백퍼센트 타임아웃 덤프로 서버를 중단시키므로, 무조건 **JOIN**이나 **FAE** 구조로 뜯어고쳐 단 한 번의 단선 조회로 리포트를 뿜어내야 한다.

```text
  [ 안티패턴: LOOP 안의 SELECT ]              [ 개선 패턴: 단일 SELECT FAE ]
  LOOP AT gt_data.                        IF gt_data IS NOT INITIAL.
    SELECT SINGLE ... FROM DB   <── (폭사)   SELECT ... FOR ALL ENTRIES IN gt_data
  ENDLOOP.                                ENDIF.
```

### 어떻게 확인하는가

비즈니스 업무 요구사항 시나리오를 바탕으로 아키텍처 판단 기어를 대조 조율한다.

```text
요구사항 1: "예매 마스터 내역 ZBOOKING을 읽으면서, 공연 날짜와 가수명(ZCONCERT)을 같이 합쳐 표로 보여주세요."
  -> 판정: 조인의 대상이 되는 두 테이블이 모두 DB에 상존하므로 1순위 [DB JOIN] 장착!

요구사항 2: "화면에서 엑셀 업로드로 긁어온 '사용자 사번 리스트 100건'에 해당하는 예매 정보만 골라 읽어주세요."
  -> 판정: 조건 기준 목록이 이미 메모리 내부 테이블에 실려 올라와 있으므로 [FOR ALL ENTRIES] 적용!

요구사항 3: "가져온 예매 좌석 총 합계 수량을, 아밥 복잡 할인율 수식 10단계를 거쳐 짝맞춰 가공 출력해주세요."
  -> 판정: DB 쿼리로 가공할 수 없는 복잡 아밥 연산이므로 각자 조회 후 [ABAP 루프 직접 가공] 수렴!
```

#### 체험/시뮬레이터 설계 (조회 전략 의사결정 카드)
- **프로세스 플로우**:
  1. 학습자가 "LOOP 문 내에 SELECT SINGLE 쑤셔 넣기" 카드를 든다.
  2. 이를 [정석 설계] 바구니에 넣으려고 시도하면 "경고! 1만 번 루프 돌 때 DB 1만 번 노크하여 서버가 비명을 지릅니다. 즉시 FAE나 JOIN으로 뜯어고치십시오" 라고 경고등이 켜지는 거름 장치를 본다.
  3. 카드를 [금지 안티패턴] 바구니로 던져 넣자 통과가 되는 분류 모션을 확인한다.
- **상태 및 데이터**:
  - `대량 JOIN` 카드 -> 무분별한 10개 조인 체인 역시 비싸므로 튜닝 검토 팝업 상태 렌더링.
- **피드백**: 루프 내 SELECT 중첩 왕복은 성능을 죽이는 독배이며, JOIN과 FAE의 명확한 상황 경계를 이정표 지도로 정리한다.

### 실수/주의

- **개발 편의주의적 LOOP SELECT 작성**: "조인 문법이 길고 복잡하니까 그냥 루프 돌면서 SELECT SINGLE로 하나씩 꺼내 쓰는 게 코딩하기 편하다" 는 안일한 타성으로 프로그램을 짰다가는, 데이터가 조금만 쌓여도 리포트가 모래시계만 돌리다 뻗어버려 전면 재작업 처분을 받으므로 반드시 **JOIN**으로 묶는 정공법을 고수해야 한다.

### 정리

- 여러 테이블을 엮을 때 두 대상이 모두 DB에 상존한다면 무조건 **`DB JOIN`** 을 1순위 표준으로 삼는다.
- 이미 내부 테이블로 가공된 리스트를 필터 조건으로 쓸 때만 **`FOR ALL ENTRIES`** 를 장착한다.
- 루프 내부에서 SELECT 쿼리를 날려 DB 왕복을 반복하는 중첩 조회 구문은 **절대 엄금 안티패턴**으로 취급해 봉쇄한다.

## CH13-L08 - 실습 — 공연별 예매현황 리포트

### 왜 필요한가

우리가 이 장에서 배운 classic INNER/LEFT OUTER JOIN의 문결표(`~`) 체결법, `GROUP BY`와 `SUM` 집계 연산 규칙, `HAVING`을 통한 2단 검문, 그리고 `ORDER BY` 정렬 조회의 대단원을 다 거쳤다. 

이제 콘서트 예매 앱의 최종 결정체인 **🎟️ 5단계 공연별 예매현황 리포트**의 엔진을 구축할 차례다. ZCONCERT 공연 마스터 대장을 기준으로, ZBOOKING 예매 대장에 기록된 좌석 수들을 묶어서 합산(`SUM`)하되, 예매가 전혀 없는 신형 공연도 0석 예약으로 떳떳하게 누락 없이 살려내고, 취소된 예약석(`status = 'C'`)은 원천 배제하여 요약한 뒤 정렬하여 깔끔한 SALV 표로 서빙해 내는 진짜 실무 현업 수준의 종합 리포트 소스 코딩을 내 손으로 완성해야 한다.

### 무엇인가

#### 🎫 5단계 예매현황 종합 리포트의 설계 질문과 도구 짝맞춤
종합 설계서의 세 가지 질문에 따라 아키텍처 도구를 엮어낸다.

1. **질문 1**: "예매 내역이 단 1건도 없는 갓 태어난 신규 공연도 누락 없이 표에 보여야 하나요?"
   - **대답**: 예 $\rightarrow$ 뼈대를 살려주는 **`LEFT OUTER JOIN`** 채택. (FROM `zconcert` 기준).
2. **질문 2**: "예약 신청했다가 취소(`status = 'C'`)한 좌석 수량은 집계 합산에서 확실히 빠져야 하나요?"
   - **대답**: 예 $\rightarrow$ **`ON` 절 우측 결합부에 `AND b~status <> 'C'` 조건 도킹.** (WHERE에 쓰면 예매 0건 공연이 싹 날아가 버리는 함정 차단).
3. **질문 3**: "공연 하나당 예매 기록 500줄을 1줄의 요약본 합계로 압축해 인쇄해야 하나요?"
   - **대답**: 예 $\rightarrow$ **`GROUP BY` + `SUM( b~seats )`** 가동.

### 어떻게 확인하는가

에디터에서 3대 설계 질문이 모두 실현된 예매현황 리포트를 코딩하고 활성화해 F8 구동한 뒤 요약 및 누적 값을 최종 검수한다.

1. `/nSE38`로 이동해 프로그램 `ZREP_CONCERT_STATISTICS`를 생성한다.
2. 아래의 정석 소스 코드를 기입하고 활성화(Ctrl+F3)한다.
3. 프로그램을 실행(`F8`)하여 공연 ID별 아티스트명, 전체 정원, 그리고 예매 완료석(`booked`) 합계가 깔끔히 렌더링된 SALV 표를 확인한다.
4. 예매 내역이 없는 신형 공연 `C003` 행이 낙오되지 않고 온전하게 생존하여 `booked = 0` (또는 빈 값) 형태로 정상 출력되는지 검수한다.
5. 취소된 예매 건의 좌석 3석이 총합계 계산에서 정합성 오차 없이 배제되어 정확한 집계치로 노출되는지 수동으로 대조 확인한다.

```abap
REPORT zrep_concert_statistics.

" 1. DB에서 조인 집계해 올린 4대 요약 데이터를 담을 전역 타입 선언
TYPES: BEGIN OF ty_stat,
         concert_id TYPE zconcert-concert_id,
         artist     TYPE zconcert-artist,
         capacity   TYPE zconcert-capacity,
         booked     TYPE i, " SUM(seats) 수납용
       END OF ty_stat.

DATA: gt_stat TYPE TABLE OF ty_stat,
      go_alv   TYPE REF TO cl_salv_table.

START-OF-SELECTION.
  " 2. [종합 조인 집계 쿼리 가동]
  " - c~concert_id 외의 평필드 artist와 capacity는 GROUP BY절에 같이 묶어 가드!
  " - status <> 'C' 조건은 LEFT JOIN 보존을 지키기 위해 반드시 ON 절 내부에 엮음!
  SELECT c~concert_id c~artist c~capacity SUM( b~seats ) AS booked
    FROM zconcert AS c
    LEFT OUTER JOIN zbooking AS b
      ON c~concert_id = b~concert_id
     AND b~status    <> 'C'
    INTO CORRESPONDING FIELDS OF TABLE gt_stat
    GROUP BY c~concert_id c~artist c~capacity.

  " 3. 조립된 요약 기차 테이블을 덤프 방어막 깔고 SALV로 화면 빔 투사!
  TRY.
      cl_salv_table=>factory(
        IMPORTING r_salv_table = go_alv
        CHANGING  t_table      = gt_stat
      ).
      go_alv->get_functions( )->set_all( abap_true ).
      go_alv->display( ).
    CATCH cx_salv_msg.
      MESSAGE '현황 ALV 조립 도중 예외가 발생했습니다.' TYPE 'I'.
  ENDTRY.
```

#### 체험/시뮬레이터 설계 (공연별 예매현황 실습)
- **프로세스 플로우**:
  1. 학습자가 ZREP_CONCERT_STATISTICS 소스를 F8 실행한다.
  2. 쿼리 트랙을 따라 `ZCONCERT` 디스크의 3개 공연 행이 끈을 들고 출발한다.
  3. `ZBOOKING` 대장과 조인을 걸 때, 취소 딱지가 붙은 예약석을 피하며 `C001` 공연 좌석 수들을 끄집어내어 `SUM` 요약 봉투에 담는다.
  4. 예매 기록이 없던 `C003` (신규공연) 행이 짝꿍이 없음에도 탈락하지 않고 0석 딱지를 단 채 무사히 통과 캔버스에 안착하여, 총 3줄의 요약 행으로 ALV 격자판에 출력되는 과정을 최종 검수 통과한다.
- **상태 및 데이터**:
  - `status <> 'C' 조건을 WHERE 절로 잘못 적고 실행했던 상태` -> C003 공연 행이 최종 ALV 표에서 홀연히 실종되어 버리는 누락 에러 상태 출력.
- **피드백**: "3대 아키텍처 질문(보존, 필터, 그룹화)을 쿼리로 완벽하게 엮었을 때 비로소 흠집 없는 실무 현황판 리포트가 탄생한다" 는 요약 주머니 렌더링을 선사한다.

### 실수/주의

- **SELECT 필드와 내부 테이블 컴포넌트의 기하학적 매핑 불일치**: `SELECT` 문장에서 컬럼을 나열한 순서(`concert_id` $\rightarrow$ `artist` $\rightarrow$ `capacity` $\rightarrow$ `booked`)와 내부 테이블 `ty_stat`에 선언된 변수들의 나열 순서가 꼬여 어긋나면, 엉뚱한 정원 데이터가 좌석 합계 칸으로 흘러 들어가는 컬럼 변조 오차가 발생하므로 순서를 칼같이 맞추어야 한다. (또는 `INTO CORRESPONDING FIELDS` 옵션을 통해 이름 기준 매핑을 장착하는 것이 정석이다).

### 정리

- **`LEFT OUTER JOIN`** + **`GROUP BY`** + **`SUM`** 의 3중 콤보를 통해 예매 무실적 공연까지 포괄하는 공연별 좌석 합계 리포트를 조립한다.
- 예매 없는 공연 보존을 깨뜨리지 않기 위해 세부 상태 가드는 반드시 **`ON` 절 내부에 AND로 장착**한다.
- **`SALV`** 표준 툴바 활성화와 연동하여 현업 부서에 즉시 납품할 수 있는 **콘서트 예매 현황판 대화형 리포트의 5단계 실무 구현**을 완수한다.
