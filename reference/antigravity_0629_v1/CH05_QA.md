## 3. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. Flat Structure (평평한 구조체) 와 Deep Structure (깊은 구조체) 는 램 메모리에 데이터가 적재될 때 구체적으로 어떤 구조적 차이가 있나요?
**A**: **Flat 구조체는 모든 필드가 고정 크기여서 램에 파편화 없이 연속된 바이트 블록으로 저장되어 카피 속도가 빠르지만, Deep 구조체는 가변 STRING 필드로 인해 실제 텍스트 내용물이 저장된 힙(Heap) 주소 번지수(Pointer)를 저장하므로 포인터 역참조 비용이 발생합니다.**
- **물리 메모리 배치 차이**:
  - **Flat Structure (평평한 구조체)**: 내벽 컴포넌트들이 `I`(4바이트), `C LENGTH 10`(10바이트) 처럼 크기가 완전 고정된 필드로만 짜인 구조입니다. 아바 커널은 이를 램에 올릴 때 정확히 **14바이트의 연속된 메모리 공간 (Contiguous Byte Block)** 에 물안개처럼 이어서 적재합니다. 데이터 전체를 전송하거나 초기화할 때 이 14바이트 블록만 쌩 복사하면 끝나므로 성능이 극대화됩니다.
  - **Deep Structure (깊은 구조체)**: 필드 성분에 가변 크기 `STRING` 이나 `Table Type` (내부 테이블)이 단 1개라도 섞여 있는 구조체입니다. `STRING` 은 글자 수에 따라 크기가 계속 요동칩니다. 
  따라서 아바 커널은 구조체 내부에 쌩 데이터를 박아넣지 못하고, 실제 텍스트가 저장되어 있는 외곽 힙(Heap) 메모리 영역의 주소 값인 **포인터 참조 번지수 (Pointer Reference - 8바이트 주소)** 만 구조체 칸에 저장합니다. 
  데이터를 읽으려면 구조체 주소를 짚은 뒤, 그 주소가 가리키는 힙 번지로 가차 없이 날아가는 '포인터 역참조(Dereferencing)' 단계를 2차로 밟아야 해, 메모리 할당 및 복사 시 상대적으로 높은 자원 오버헤드가 누적됩니다.

### Q2. 아바 구조체 연산 시 편리해 보이는 MOVE-CORRESPONDING 을 사용하여 두 구조체 간 필드 복사를 때릴 때, 런타임에 구체적으로 어떤 내부 성능 오버헤드가 발생하나요?
**A**: **런타임 엔진이 송/수신 대상 구조체의 메타데이터 명세표를 뒤지며 '이름이 같은 필드명' 이 존재하는지 실시간으로 문자열 루프 대조 탐색(Run-time Name Matching)을 하기 때문에 심각한 CPU 탐색 비용이 발생합니다.**
- **실시간 이름 매칭 탐색**:
  `MOVE-CORRESPONDING gs_a TO gs_b.` 는 소스가 아주 깔끔해 보입니다. 
  하지만 아바 인터프리터 런타임은 이 라인을 실행할 때, "gs_a 의 첫 컴포넌트인 ID 가 gs_b 에도 존재하느냐?" 고 gs_b 의 필드 명세 사전을 쌩으로 루프 돌며 문자 비교합니다. 
  이어서 "gs_a 의 두 번째 필드인 NAME 이 gs_b 에도 존재하느냐?" 를 대조 검색합니다. 
  이 짓을 매 실행 시점마다 단행하므로, 필드 개수가 20개로 늘어나고 이 복사 처리가 10만 건 반복되는 배치 루프 속에 상주해 가동되면, CPU 연산 시간이 개별 대입보다 10배 이상 튀어 오르는 성능 저하 참사를 겪습니다.

### Q3. SAP 표준 DB 테이블이나 표준 구조체의 필드를 확장하려 할 때, 그냥 테이블 구조 내벽에 필드를 직접 쑤셔 넣지 말고 왜 귀찮게 .APPEND 구조체를 따로 빚어서 도킹 장착해야 하나요?
**A**: **표준 원본 구조를 직접 수정하면 SAP 시스템 패키지 업그레이드(Upgrade) 시 원본이 강제 덮어쓰기 오버라이트되어 내 커스텀 필드가 영구 삭제되는 형상 유실이 터지지만, `.APPEND` 구조체는 내 Z필드들을 별도 컨테이너로 격리 보존하여 기존 뼈대를 다치지 않고 안전 면역되기 때문입니다.**
- **업그레이드 덮어쓰기 방어 (.APPEND Structure)**:
  `SFLIGHT` 테이블에 내 예약 할인율 필드(`ZZDISC`)를 직접 도려내 추가했습니다. 
  본사 Basis 팀이 SAP 버전을 업그레이드하는 순간, SAP 본사가 배포한 순수 오리지널 `SFLIGHT` 데이터 사본이 내 테이블 위에 쌩 덮어쓰기 복사됩니다. 내가 짠 `ZZDISC` 필드는 그 찰나 흔적도 없이 삭제됩니다. 
  이를 막기 위해, Z로 시작하는 별도의 Append Structure `ZAP_SFLIGHT` 를 개설하고 그 안에 `ZZDISC` 필드를 담아 저장 활성화합니다. 
  시스템은 `SFLIGHT` 테이블 꼬리에 `ZAP_SFLIGHT` 를 자석처럼 찰딱 붙여 하나의 테이블로 병합 기동합니다. 
  버전 업그레이드가 SFLIGHT 본체를 덮어쓰더라도, 외곽 꼬리에 부착된 Append Structure `ZAP_SFLIGHT` 는 시스템 형상 제어가 건드리지 않아 내 튜닝 필드가 100% 무결 보존되는 평화가 지켜집니다.

### Q4. 구조체 ZST_PERSON 을 상속받아 DATA gs_p4 LIKE gs_p1 로 변수를 선언했을 때, 변수 생성 시점에 gs_p1 이 머금고 있던 데이터 값들까지 함께 복사되어 채워지나요?
**A**: **아니오, 변수의 값은 전혀 복사되지 않으며, `gs_p4` 내벽의 모든 필드는 아바 런타임 램에 적재되는 그 찰나에 각 타입에 규정된 깨끗한 초기값(정수 0, 문자 공백, P타입은 0.00)으로 제로 셋업되어 시작합니다.**
- **LIKE 의 형상 복제 제약**:
  `LIKE gs_p1` 은 **"gs_p1 이 점유하고 있는 메모리의 설계 명세(Descriptor)"** 만 본뜨는 컴파일러 지시어입니다. 
  방을 복제해 짓는 선언 구문일 뿐이므로, 램 공간이 할당되는 찰나에는 `gs_p1` 의 실시간 데이터 값(예: '정훈영', 30세)은 일절 넘어가지 않고 차단됩니다. 
  만약 `gs_p1` 의 내용물까지 복사하고 싶다면, 선언 이후 런타임 본 로직 시점에 대입 연산자 **`gs_p4 = gs_p1.`** 을 명시적으로 격발해 주어야 비소로 메모리 복제가 체결됩니다.

### Q5. 100만 건 대량 연산 루프 속에서 구조체 복사를 해야 합니다. MOVE-CORRESPONDING 을 쓰는 것이 좋나요, 아니면 귀찮더라도 gs_b-name = gs_a-name 처럼 하이픈으로 개별 적는 게 맞나요?
**A**: **무조건 귀찮더라도 하이픈(-)으로 개별 1:1 대입문을 명시적으로 적어주는 것이 정답이며, 이 방식은 컴파일할 때 물리 메모리 주소지가 고정 바인딩되어 O(1) 에 근접한 기계 복사를 단행하므로 압도적으로 빠릅니다.**
- **실무 고속 성능 코딩 룰**:
  `MOVE-CORRESPONDING` 은 런타임에 문자열 비교 루프를 타므로 대량 루프 내에서는 절대 금기시됩니다.
  `gs_b-name = gs_a-name.` 과 같이 하이픈 개별 대입을 치면, 컴파일러가 빌드 시점에 gs_b-name 의 램 주소와 gs_a-name 의 램 주소지를 직접 결선 결선(Static Binding)해 버립니다.
  런타임 시 CPU 는 필드명을 대조하는 헛손질 없이, 해당 주소의 바이트 데이터를 타깃 주소로 바로 복사 전송하므로 연산 속도가 지연되지 않습니다.
  **철칙**: 실무 대량 트랜잭션 연산 루프 내벽에서는 **`하이픈 개별 대입`** 을 필사적으로 사수해 코딩해야 합니다.
