## 3. 공식 문서 수동 근거 명세

`abensap_lock.htm` 을 대조하여 SAP Lock 의 물리 제약 요건을 철저히 규명했다.

- **DEQUEUE _SCOPE 매칭 제약 규칙**:
  `DEQUEUE_*` 함수를 호출하여 락을 수동 해제할 때, **`DEQUEUE` 의 `_SCOPE` 파라미터는 무조건 `ENQUEUE` 호출 시점 지정한 것보다 같거나 큰 값**이어야만 해제가 인정된다. 예를 들어, `ENQUEUE` 시 `_scope = 2` 로 지정한 락을 `DEQUEUE` 시 `_scope = 1` 로 던져 해제를 날리면, 엔진은 이 해제 신호를 No-op 쌩까고 락을 해제하지 않고 잠궈둔 채 방치하는 심각한 좀비 락 제약이 존재함을 확인 및 수록 완료.

## 4. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. ENQUEUE_EZ_BOOKING 을 호출할 때 _scope = '2' 로 잠갔습니다. 이후 수동으로 DEQUEUE_EZ_BOOKING 을 때려 풀기 위해 _scope = '1' 을 파라미터로 넘겨 던졌는데 왜 잠금이 여전히 안 풀리고 잠겨 있나요?
**A**: **아바 락 엔진 규칙 상, DEQUEUE 해제 호출 시 지정하는 _SCOPE 의 레벨은 무조건 ENQUEUE 시점의 scope 값보다 같거나 높아야만 정상 수락되기 때문이며, 동일하게 _scope = '2' 로 맞춰서 해제해야 합니다.**
- **동작 불일치 원인**:
  `_scope = '2'` 는 락을 SAP LUW 업데이트 태스크 세션에 바인딩하겠다는 상위 결계입니다. 
  이 상위 결계가 씌워진 락을 풀 때는 해제 함수도 동급 이상의 해제 권한인 `_scope = '2'` (또는 양쪽 다 푸는 '3')를 갖고 노크해야 합니다.
  그런데 이보다 해제 파워가 낮은 일반 프로그램 세션 수준의 `_scope = '1'` 을 해제 인수로 넘기면, 락 엔진은 "상위 결계 락을 하위 기어로 조기 해제하려 드는 안전 위배 요청" 으로 간주하고 해제 명령을 쌩까버립니다. 
  이때 에러 덤프를 내뿜지 않고 그냥 침묵으로 락을 해제하지 않은 채 방치하기 때문에 개발자가 알아채기 매우 힘든 함정입니다.
- **해결책**:
  따라서 수동으로 디큐(DEQUEUE)를 때릴 때는 무조건 처음 락을 잡을 때 기재했던 scope 와 **완벽히 동일한 scope(여기서는 '2')** 를 매칭하여 기입해야 안전하게 풀립니다.

### Q2. 이미 다른 사용자가 잠그고 있어 foreign_lock = 1 예외가 터졌습니다. 이때 도대체 어떤 로그인 사용자 ID 가 나를 가로막고 있는지 출력하기 위해 sy-msgv1 시스템 변수를 화면에 뿌려봤는데, 왜 자꾸 빈 칸(space)으로 나오나요?
**A**: **SAP 커널 명세 상 foreign_lock 충돌 시 sy-msgv1 에 소유자 ID 정보를 채워주는 행위는 의무 조항이 아니므로, 상황에 따라 빈 값으로 채워져 반환되는 경우가 빈번하며, 소유자를 꼭 봐야 한다면 ENQUEUE_READ 함수를 따로 써야 합니다.**
디버깅이나 사용자 친화적 메시지를 짤 때 `WRITE: / sy-msgv1, '님이 편집 중입니다.'` 라고 코딩을 많이 짭니다.
실제로 몇 번은 소유자 이름이 잘 찍힐 수 있지만, 락 징표의 상태나 업데이트 타스크 세션 상태에 따라 `sy-msgv1` 은 텅 빈 스페이스로 떨어지는 경우가 매우 많습니다. 이 변수를 맹신해 화면 팝업을 짜면 사용자에게 " space 님이 편집 중입니다" 라는 조접한 문구가 나가게 됩니다.
**정석 방어**:
정확한 락 소유자 정보를 끄집어내야 한다면, 시스템 변수에 의존하지 말고 백엔드 락 조회 전용 펑션인 **`ENQUEUE_READ`** 를 가동하여 해당 Key 에 락을 쥔 소유자 ID 필드를 꺼내오거나, T-code **`SM12`** 에 가서 수동 대조하는 것이 정석입니다.

### Q3. 분명히 ENQUEUE_EZ_BOOKING 으로 1001번 예약 건을 잠갔습니다. 그런데 다른 사용자가 SE38 로 만든 이상한 프로그램으로 1001번 예약을 UPDATE 쳤더니 잠금을 뚫고 그냥 저장 변경이 완료되었습니다. 락이 고장난 건가요?
**A**: **SAP Lock 은 물리 DB 레벨의 강제 락이 아니라 SM12 중앙 Lock Table 에 징표를 적고 대조하는 '논리적 자발 규약' 이기 때문이며, 상대방 프로그램이 DML 실행 전 ENQUEUE 선행 체크를 무시하고 쌩 UPDATE 를 때렸기 때문에 뚫린 것입니다.**
이 개념을 입문자들이 가장 많이 헷갈려합니다. 
"내가 자물쇠를 걸어 잠갔는데 왜 문을 열고 들어가서 데이터를 훔쳐 가냐" 고 억울해합니다.
다시 한번 강조하지만, SAP Lock 은 DB 디스크 레코드를 물리적으로 콘크리트 쳐 막아두는 장치가 아닙니다.
"1001번 레코드는 내가 손대고 있음" 이라고 **중앙 게시판(SM12 락 테이블)에 대고 분필로 써놓은 논리적 선언**입니다.
따라서, 정상적인 비즈니스 프로그램들은 모두 "1001번을 고치기 전에 먼저 게시판(ENQUEUE 호출)을 확인하고, 다른 사람의 분필 징표가 있으면 작업을 양보하고 멈춘다" 는 규칙을 신사적으로 준수해야 합니다.
만약 어떤 불량 CBO 프로그램이 이 게시판 확인(ENQUEUE)을 건너뛰고 그냥 `UPDATE zbooking SET ...` SQL 쿼리를 날려버리면 DB 는 게시판 따위 안 보기 때문에 묵묵히 덮어쓰고 맙니다.
**교훈**: 동시성 보호를 완성하려면 내 프로그램이 잠그는 것만큼, **다른 모든 동료 프로그램들도 변경 전 선행 ENQUEUE 락 획득 시도를 하도록 개발 표준 규율을 강제 통일**해야 안전합니다.

### Q4. 낙관적 락(Optimistic Lock)을 타고 저장할 때, 변경 대조 컬럼을 changed_at = sy-uzeit 로만 짜면 왜 동일 1초 이내 충돌 상황에서 데이터가 파손되어 날아가나요?
**A**: **동일한 1초(예: 14시 10분 05초)라는 극도로 좁은 찰나에 User A 와 User B 가 연쇄적으로 저장을 찌르면, 둘 다 changed_at 이 '141005' 로 완벽히 일치하여 대조 검문소가 충돌을 인지하지 못하고 패스시켜 버리기 때문입니다.**
- **유실 타임라인**:
  1. DB 테이블의 B001 번 행의 `changed_at` 이 '141005' 입니다.
  2. User A 와 User B 가 동시에 조회를 떠갔고, 둘 다 자신이 가진 이전 버전의 변경시각으로 '141005' 를 메모리에 쥐고 있습니다.
  3. User A 가 14시 10분 05초에 저장을 때렸습니다. DB 세션이 열리고 저장 직전 찰나 대조를 하니 `DB('141005') = 내 메모리('141005')` 가 성립해 무사 통과되어 저장하고, DB 의 changed_at 을 현재 시각인 '141005' 로 세팅합니다.
  4. 0.1초 뒤, 여전히 14시 10분 05초인 상태에서 User B 가 저장을 누릅니다. 저장 직전 찰나 대조를 하니 `DB('141005') = 내 메모리('141005')` 가 성립해 통과되고, User B 의 옛 값으로 DB 가 덮어써져 User A 의 데이터가 공중 폭사합니다.
- **방어 규칙**:
  낙관적 락의 대조 징표는 이와 같은 동시 찰나 충돌에 취약한 초단위 시각을 절대 쓰면 안 됩니다.
  변경할 때마다 1씩 순차 증가하는 **`Version Counter` (1 -> 2 -> 3)** 나 밀리세컨드 단위 **`Timestamp`** 를 락 아규먼트로 채택해야 100% 안전하게 충돌을 소거해 낼 수 있습니다.

### Q5. 락을 거는 구문(ENQUEUE)을 'SELECT 데이터 조회' 쿼리를 타기 전이 아니라, 굳이 값 변경 UPDATE DML 쿼리를 치기 바로 직전 라인에 꽂아 넣으면 왜 Lost Update 충돌을 막지 못하나요?
**A**: **락을 잡기도 전에 이미 User A 와 User B 가 락이 풀려있는 무방비 DB 레코드를 나란히 조회해가 버렸기 때문에, 뒤늦게 업데이트 칠 때 락을 거는 것은 조회 시점의 동시 진입을 전혀 방어하지 못해 덮어쓰기가 터집니다.**
- **안티패턴 순서**: `1. READ -> 2. 값 고침 -> 3. 뒤늦게 ENQUEUE -> 4. UPDATE`
- **충돌 시나리오**:
  1. User A 와 User B 가 1001번 예약석(2석)을 동시에 READ 조회해 가져갔습니다. (이때 락을 안 걸어 놨으므로 둘 다 2석 데이터를 쥔 채 조회 통과 완료).
  2. User A 가 3석으로 고치고 ENQUEUE 를 잡아 락을 딴 뒤 UPDATE 를 때리고 커밋해 락을 풀었습니다. (DB 는 3석이 됨).
  3. User B 가 자기가 쥔 옛날 2석 기준으로 1석으로 고친 뒤, 뒤늦게 ENQUEUE 를 겁니다. User A 가 이미 커밋하고 락을 풀었으므로 User B 는 락을 손쉽게 획득 성공하고 UPDATE 를 때려 DB 를 1석으로 파괴해 버립니다.
- **철칙**: 
  자물쇠는 방을 나오기 전이 아니라, 방에 들어가기 전인 **`SELECT READ 를 수행하기 직전`** 에 1순위로 잠가 걸어야만, 상대방이 옛날 데이터를 복사해 가 원인 제공하는 시점 자체를 완벽하게 차단해 낼 수 있습니다.
