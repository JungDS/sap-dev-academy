- **`abapcall_screen.htm`**: 
  - 화면 호출 시 `dynnr` 파라미터는 4자리 `n` 형식의 데이터 객체 형식을 엄수해야 한다는 물리 명세 검증 완료.

  - 팝업으로 여는 화면(Modal Window)은 반드시 Screen Attributes 속성 탭에서 `Modal dialog box`로 화면 타입을 수동 변경해야 정상 기동된다는 가이드 확보.
- **`abapset_screen.htm`**: `SET SCREEN` 지시어는 다음 화면(next screen) 예약 표식만 꽂아둘 뿐, PAI 프로세싱이 완전히 완료되는 마지막 시점에 비로소 자동 이동이 가동된다는 지연 실행(Deferred Execution) 스펙 수동 대조 완료.
- **`abapleave_screen.htm`**:
  - `LEAVE TO SCREEN` 구문은 `SET SCREEN` + `LEAVE SCREEN`을 압축한 숏폼(Short form) 문법이라는 명세 대조 완료.
  - `LEAVE SCREEN` 실행 즉시 현재 아밥 모듈 및 딘프로 Flow Logic 구동이 즉각 중단(exited immediately)되고 예약 화면으로 점프한다는 물리 실행 거동 수립.

## 4. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. PAI 모듈 내부에서 ok_code를 보조 변수 save_ok에 대피시키자마자 즉시 CLEAR ok_code. 문장을 실행하여 비워주어야 하는 물리적 이유는 무엇인가요?
**A**: **사용자가 키보드 [엔터 (Enter)] 키를 칠 때마다, 이전 버튼 클릭 시 발생했던 ucomm 명령(예: 'SAVE')이 변수 방에 그대로 잔류해 있어 '중복 저장' 오작동을 터트리는 대참사를 막기 위해서입니다.**
사용자가 화면 상의 [저장 (SAVE)] 버튼을 클릭하면, 화면 요소 목록에 바인딩된 `ok_code` 전역 변수 방으로 `'SAVE'` 라는 문자열이 수송 유입됩니다.
만약 PAI 모듈 내부에서 이 값을 `CLEAR ok_code.` 로 지워주지 않고 그냥 놔두면, 데이터베이스 저장이 끝난 뒤에도 `ok_code` 메모리 방에는 계속 `'SAVE'` 문자가 고여있게 됩니다.
이 상태에서 사용자가 다른 입력 필드에서 값을 수정하거나, 혹은 빈 공간에서 무심코 키보드 [엔터] 키를 탁 누르면, 화면 엔진은 "PAI 이벤트가 터졌다" 고 판단하여 백엔드 PAI 모듈을 재실행합니다. 
이때 `ok_code` 방에 여전히 고여있던 `'SAVE'` 문자열이 다시 `CASE` 문에 검출되어 **"저장 로직이 한 번 더 실행"** 되는 대참사가 터집니다. 
따라서 ucomm 수신 즉시 로컬 백업 변수 `save_ok` 에 값을 대피시키고, 원천 총알통인 `ok_code` 방은 즉각 **`CLEAR`** 로 하얗게 세척 소멸하여 다음 엔터 클릭 시 오작동하지 않게 안전 잠금을 거는 것이 아바 Dynpro 백엔드 설계의 철칙입니다.

### Q2. SET SCREEN <번호> 와 LEAVE TO SCREEN <번호> (또는 LEAVE SCREEN)는 물리적으로 무엇이 다르며, 딘프로 PAI 실행 흐름 상에서 어떤 차이를 만드나요?
**A**: **다음 화면 번호만 예약해 두고 PAI 뒷 코드를 끝까지 다 실행하는가(지연 이동), 혹은 PAI 뒷 코드를 즉시 참수하고 그 자리에서 즉각 순간이동하는가(즉시 이탈)의 차이입니다.**
- **`SET SCREEN 0200` (예약 지연 기어)**:
  이 지시어를 만나면, 시스템은 메모리 맵 상의 'Next Screen' 칸에 `0200` 이라는 표식만 조용히 꽂아둡니다. **즉시 화면을 떠나지 않습니다.**
  `SET SCREEN` 아랫줄에 기입된 데이터베이스 업데이트나 메시지 코드가 PAI 모듈 끝까지 100% 전부 실행되고 난 뒤, PAI가 완전히 종료되는 찰나에 비로소 0200번 화면으로 부드럽게 넘어갑니다.
- **`LEAVE TO SCREEN 0200` (또는 `LEAVE SCREEN` 즉시 이탈 기어)**:
  이 지시어를 만나면, 시스템은 **그 즉시 현재 돌고 있는 PAI 백엔드 연산과 Dynpro 흐름을 완전히 중단(참수)하고 즉각 화면을 파괴해 이탈**합니다. 아랫줄에 기입된 어떠한 코드도 실행되지 않고 즉시 0200번 화면으로 점프합니다.
만약 "데이터를 검증해서 에러가 났으니 다음 화면으로 넘기지 않고 여기서 즉시 멈추고 복귀하겠다" 고 할 때는 반드시 `LEAVE TO SCREEN 0` 이나 즉시 이탈 명령을 써서 뒷단의 잘못된 데이터 저장 로직이 수행되지 않도록 도중 차단해야 합니다.

### Q3. Custom Container (cl_gui_custom_container) 객체를 PBO 모듈 안에서 빚어낼 때 IF go_cont IS INITIAL. 가드 스위치를 생략하면 어떤 런타임 에러가 발생하며 수명주기가 파괴되는 원인은 무엇인가요?
**A**: **사용자가 화면을 클릭할 때마다 도킹 다리 객체가 중복 생성되어, 시스템이 "동일한 구역에 이미 인스턴스가 존재한다" 며 OBJECT_ALREADY_EXISTS 런타임 덤프를 내뿜고 서버가 다운됩니다.**
PBO 모듈은 화면이 사용자 모니터에 뿌려지기 직전마다 호출되는데, 사용자가 드롭다운을 펼칠 때, 엔터를 칠 때, 버튼을 클릭할 때마다 끊임없이 반복하여 재실행됩니다.
만약 `IF go_cont IS INITIAL.` 이라는 가드 장치 없이 `CREATE OBJECT go_cont` 를 PBO에 쌩으로 적어두면, 
1. 화면이 처음 켜질 때 `CONT100` 영역에 1번 컨테이너 다리가 정상 빌드됩니다. (성공).
2. 사용자가 화면에서 엔터를 쳐서 PAI를 탄 뒤 화면이 다시 그려지는 2번째 PBO 시점에, 또다시 `CREATE OBJECT go_cont` 가 가동됩니다.
3. 시스템 커널은 **"이미 CONT100 화면 구멍에 1번 도킹 다리가 물려있는데 왜 똑같은 영역에 2번 도킹 다리를 강제로 덧대어 생성하려 하는가?"** 라고 판단하여 메모리 충돌 에러(`OBJECT_ALREADY_EXISTS`)를 터트리고 즉시 프로그램을 덤프 종료시킵니다.
따라서 참조 변수가 최초로 텅 비어있을 때(`IS INITIAL`) 딱 한 번만 객체를 생성하도록 안전 스위치를 채워두는 것이 컨테이너 생명주기 관리의 절대 가드라인입니다.

### Q4. Dynpro 화면의 컴포넌트 이름과 아바 백엔드 소스의 전역 변수명을 100% 동일하게 맞춰주어야만 가동되는 데이터 수송(Data Transport)의 런타임 매핑 메커니즘을 설명해 주세요.
**A**: **SAP GUI 화면 프레임워크가 PAI 시작 시점과 PBO 종료 시점에, 화면 메모리 버퍼와 프로그램 변수 메모리 주소록을 스캔하여 이름이 같은 필드끼리만 값을 일괄 덤프 복사(Data Transport)해 주기 때문입니다.**
Dynpro 화면 시스템은 아바 프로그램과 완전히 독립된 별개의 메모리 세션 공간에서 돌아갑니다.
- **PBO가 끝나는 찰나 (화면 송출 직전)**:
  화면 엔진은 프로그램 전역 변수방의 명단을 촥 긁어옵니다. 그리고 화면 레이아웃 상에 이름이 정확히 일치하는 입력 필드가 있으면, 변수 방에 들어있던 값(예: `p_seats = 5`)을 화면 상자 안으로 복사해 렌더링해 줍니다.
- **PAI가 시작되는 찰나 (사용자 행동 직후)**:
  사용자가 화면에 기입한 글자들을 낚아챈 뒤, 프로그램 메모리 방에서 스펠링이 100% 똑같은 변수를 찾아 그 안에 값을 털어 넣어 기입해 줍니다.
만약 화면 필드 이름은 `P_SEATS` 인데 아바 변수명은 `p_seat` (S 누락) 로 다르게 지었다면, 이름 매칭 테이블에서 짝을 찾지 못하므로 수송 기차는 값을 옮기지 않고 그냥 출발합니다. 결국 화면에 아무리 백만석을 입력해도 아바 변수 `p_seat` 방은 평생 텅 빈 초기값(0) 상태로 머무는 버그가 나므로 이름 일치는 절대 명세입니다.

### Q5. Subscreen Area를 활용해 메인 화면 안에 자식 화면을 끼워 넣었을 때, 자식 화면 내부의 버튼 ucomm이 자식 딘프로 PAI가 아닌 메인 PAI로만 유입되는 아키텍처적 제약은 무엇이고 어떻게 대처해야 하나요?
**A**: **자식 서브스크린은 독자적인 OK_CODE(ucomm 전송구)를 소유할 수 없는 화면 종속성 제약이 있어, 모든 버튼 클릭 신호가 부모 메인 화면의 OK_CODE 통으로 강제 우회 집결하기 때문이며, 메인 PAI에서 ucomm을 수집해 분기 처리해야 합니다.**
SAP Dynpro 아키텍처 상, 한 화면 시퀀스 안에서 실제 활성화되어 메인 프레임을 지배하는 OK field는 오직 메인 화면(Screen 1000번 등) 단 하나뿐입니다.
액자 프레임처럼 끼워 들어간 자식 화면(`0110`번 서브스크린) 내부에 배치된 [확인] 버튼을 사용자가 클릭하면, 자식 화면은 ucomm을 수납할 변수 방이 없으므로, 이 클릭 신호는 부모의 PAI 흐름을 강제로 깨우며 **부모 메인 화면의 `ok_code` 변수방으로 자동 전송**됩니다.
- **대처 설계**:
  따라서 자식 화면 내부에서 사용하는 버튼 ucomm 명칭(예: `'SUB_OK'`)이라 할지라도, 이 코드를 처리하는 분기문(`CASE save_ok`)은 자식 아바 모듈이 아니라 **부모 메인 화면의 PAI 모듈인 `user_command_0100 INPUT` 내부**에 기입해 일괄 분기 통제해야 합니다. 자식 화면은 오직 레이아웃과 데이터 렌더링만 돕고, 실제 사용자 행동 통제권은 부모 모듈이 총괄 쥐는 부모-자식 위임 설계가 표준입니다.
