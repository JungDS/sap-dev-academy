  - 레이아웃 구조체(`LVC_S_LAYO`)에서 `smalltitle = 'X'` 또는 `no_toolbar = 'X'` 같은 클래식 시각 억제 제어의 옵션 활용도 확인.

## 4. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. LVC_FIELDCATALOG_MERGE 함수를 호출할 때 필드 이름(fieldname)에 소문자가 섞여 있으면 왜 설정이 무시되고 먹통이 되나요?
**A**: **SAP ABAP Dictionary의 커널 엔진은 모든 물리적 테이블 명세와 컬럼 정보를 메모리 맵 상에 오직 '대문자(Uppercase)'로만 저장하여 식별하기 때문입니다.**
`LVC_FIELDCATALOG_MERGE`는 데이터베이스 테이블인 `ZBOOKING`을 읽어 자동으로 필드 카탈로그 구조를 빚어냅니다. 이때 생성된 `lt_fcat` 내부 테이블의 `fieldname` 컬럼 값은 `'BOOKING_ID'`, `'SEATS'` 처럼 대문자로 채워져 있습니다.
만약 학습자가 이를 수정하겠다고 `IF ls_fcat-fieldname = 'seats'.` 또는 `fieldname = 'Seats'` 와 같이 소문자가 포함된 문자열로 대조 판정을 시도하면, 문자 대조기(String Comparator)는 `'SEATS'` 와 `'seats'` 를 완전히 다른 문자로 간주하여 분기 조건을 탈출(Bypass)해 버립니다. 
따라서 `coltext` 나 `outputlen` 을 수정하는 갱신 코드가 단 한 번도 실행되지 않아, 화면 상에는 원래의 딱딱한 영어 대문자가 그대로 인쇄되는 버그가 터집니다. 
필드 카탈로그의 `fieldname` 값을 대조하거나 조작할 때는 반드시 **대문자 `'SEATS'`** 로만 기입하는 것이 아바 사전 결합의 철칙입니다.

### Q2. 데이터가 갱신될 때 set_table_for_first_display 메서드를 매번 반복 호출하여 표를 갈아 끼우면 화면과 메모리 상에 어떤 치명적인 폐해가 발생하나요?
**A**: **화면에 극심한 깜박임(Flickering)과 데이터 로드 지연이 발생하고, Control Framework 내부의 화면 도킹 주소 매핑이 꼬여 결국 시스템이 다운되거나 빈 유령 격자판만 남는 버그가 터집니다.**
- **`set_table_for_first_display` (최초 조립 출시)**:
  이 메서드는 단순히 데이터만 밀어 넣는 것이 아니라, 화면 Custom Control 영역과 내부 그리드 조종판을 처음으로 정렬, 병합, 탭 스택 등을 묶어 물리적 뼈대를 **최초로 한 땀 한 땀 그리는 무거운 렌더링 작업**입니다.
- **반복 호출 시 발생하는 대참사**:
  이미 조립이 완료된 그리드 위에서 데이터만 바뀌었는데 이 함수를 또 부르면, 시스템은 기존에 그려져 있던 컬럼 틀과 정렬 지침서를 강제로 파괴(Destroy)하고 0부터 다시 뼈대를 세워 그리기 시작합니다. 이로 인해 사용자는 모니터가 사정없이 하얗게 번쩍이는 깜박임을 보게 되며, 2중으로 바인딩된 메모리 주소록이 충돌하여 결국 런타임 덤프를 터트립니다.
따라서 첫 출시는 단 1회 `FIRST_DISPLAY`로 고정하고, 이후 조회를 바꾸어 데이터를 갈아 끼울 때는 무조건 가볍고 부드러운 전용 새로고침 API인 **`refresh_table_display`** 만을 기동해야 합니다.

### Q3. ls_layout-info_fname 필드에 색 코드 값(예: 'C610')을 직접 대입하면 왜 행 전체에 색칠이 되지 않고 하얗게 방치되는 오동작이 나나요?
**A**: **info_fname 은 '색상 값'을 직접 수납하는 창고가 아니라, "어떤 컬럼 필드방(예: ROWCOLOR)에 각 행의 색상 코드가 들어있는지" 를 가리켜주는 '필드명 이정표' 역할만 하도록 설계되어 있기 때문입니다.**
- **잘못된 설계**: `ls_layout-info_fname = 'C610'.`
  이렇게 쓰면 ALV 그리드는 표를 그릴 때 "아하! 이 테이블 안에 'C610' 이라는 이름을 가진 칼럼 열이 있나 보구나. 그 열에 가서 색 코드를 꺼내야지" 하고 lt_booking의 컬럼 명세에서 `'C610'` 이라는 필드를 찾습니다. 하지만 그런 필드는 존재하지 않으므로 결국 아무 색도 칠하지 못하고 무채색으로 렌더링을 끝냅니다.
- **올바른 설계**: 
  1단계로 데이터 구조에 `rowcolor TYPE c LENGTH 4` 라는 전용 방 컬럼을 개설하고, 2단계로 레이아웃에 **`ls_layout-info_fname = 'ROWCOLOR'.`** 라고 방 이름을 알려주어야 합니다.
그래야 그리드가 `'ROWCOLOR'` 컬럼 방으로 찾아가 "첫 번째 줄에는 'C610' 이 들어있군. 그럼 이 줄은 빨간색으로 칠하자! 두 번째 줄은 비어있군. 기본색으로 두자!" 하고 각 레코드의 값을 유동적으로 낚아채어 색을 칠하게 됩니다.

### Q4. refresh_table_display 를 호출할 때 is_stable 의 row 와 col 속성은 각각 어떠한 물리 스크롤 위치를 사수해 주며, 이를 켰을 때의 UX적 이점은 무엇인가요?
**A**: **사용자가 보고 있던 세로 줄(Row - 행)의 눈금 고도와 가로 줄(Col - 열)의 슬라이더 위치를 접착제로 붙인 듯 고정하여, 새로고침 시 화면이 맨 꼭대기로 튕겨 올라가는 불쾌한 시선 이동을 방지해 줍니다.**
- **`ls_stable-row = abap_true` (세로 스크롤 고정)**:
  사용자가 300건의 데이터 중 스크롤을 중간까지 내려 150번째 행을 수정하고 새로고침을 누르더라도, 150번째 행이 모니터 화면의 같은 높이에 그대로 멈춰 서 있도록 세로축 스크롤바의 눈금을 단단히 사수합니다.
- **`ls_stable-col = abap_true` (가로 스크롤 고정)**:
  가로로 컬럼이 30개나 되는 넓은 표에서 사용자가 가로 스크롤을 맨 우측으로 밀어 금액 컬럼을 감시하는 도중 새로고침이 일어나더라도, 가로 스크롤바가 맨 왼쪽 첫 열로 순간이동하지 않고 그 자리를 유지하게 지켜줍니다.
이 stable 옵션을 켜두어야만 새로고침 시 사용자가 보던 행과 열의 작업 맥락이 파괴되지 않고 부드럽게 데이터만 깜박임 없이 업데이트되는 명품 화면 정합성을 선사할 수 있습니다.

### Q5. 더블클릭 이벤트로 생성한 팝업 다이얼로그나 탭 화면을 닫을 때, 단순히 참조 변수를 CLEAR해 주는 것 외에 왜 반드시 free( ) 메서드를 호출해야 하나요?
**A**: **아밥 참조 변수를 CLEAR로 지우더라도, 백엔드 화면 제어 엔진(Control Framework) 세션 내부에는 컨테이너와 그리드의 물리적 UI 인스턴스가 여전히 살아남아 메모리 누수(Memory Leak)를 일으키기 때문에, free( ) 로 강제 파괴해 주어야 합니다.**
아밥 프로그램에서 `CLEAR go_grid.` 나 `CLEAR go_cont.` 를 가동하면, 프로그램 소스 상에서는 참조 다리가 끊어집니다.
하지만 SAP 커널의 프레젠테이션 레이어(CFW)는 화면에 그렸던 컨트롤 객체를 메모리에 여전히 부여잡고 살려둡니다. 이 때문에 사용자가 팝업창을 10번 열고 닫으면, 겉보기엔 창이 닫힌 것 같아도 백엔드 메모리에는 10개의 보이지 않는 컨테이너 좀비 객체가 누적되어 서버 리소스를 사정없이 갉아먹게 됩니다.
이 좀비들을 방지하고 OS 메모리를 완벽하게 환원하려면, 팝업을 닫는 닫기(`CLOSE`) 이벤트 모듈에서 **`go_grid->free( ).`** 와 **`go_cont->free( ).`** 메서드를 명시적으로 먼저 호출해 주어야 합니다. 
이 메서드는 CFW 세션의 실물 컨트롤 구조를 완전히 폭파(Destroy)하여 소멸시키고, 이후 참조 변수를 `CLEAR` 해야 비로소 가비지 컬렉터가 잔여 찌꺼기 메모리를 100% 회수해 깨끗한 상태로 복구해 줍니다.
