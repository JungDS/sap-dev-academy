## 3. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. 아바 에디터(SE38)에서 열심히 코드를 타이핑하고 저장(Ctrl+S)까지 마쳤는데 왜 F8 실행을 눌러도 예전 소스 코드 결과만 렌더링되나요?
**A**: **아바 런타임 엔진은 오직 활성화(Ctrl+F3)를 단행해야만 DB 안방의 Inactive 소스를 읽어 컴파일을 실행하고 새로운 실행 파일인 LOAD-Program 을 메모리에 올려 구동시키기 때문이며, 저장(Ctrl+S)만 한 상태는 Inactive 임시 락 상태로 격리됩니다.**
- **동작 원리 (Inactive vs Active)**:
  아바 소스를 짜고 저장(`Ctrl+S`)을 누르면, 코드는 데이터베이스 `REPOSRC` 테이블에 **'Inactive (비활성)'** 징표를 단 채 임시 저장만 됩니다.
  아바 커널 런타임 엔진은 사용자가 `F8` 실행을 때릴 때, 이 Inactive 테이블을 긁지 않고, 이미 컴파일 완료되어 메모리에 올라가 있는 **'Active (활성) LOAD-Program'** 바이너리만 고속 가동합니다.
  따라서 내 변경 소스가 실행에 반영되게 하려면 반드시 **`Ctrl+F3 (활성화)`** 단추를 때려야 합니다.
  활성화를 누르는 순간 컴파일러가 깨어나 내 Inactive 소스에서 주석을 다 걷어내고 바이너리 기계어 코드로 새로 조립(Rebuild)해 LOAD-Program 에 덮어씌워 줌으로써, 비소로 변경 사항이 런타임에 찰딱 도킹 기동됩니다.

### Q2. 다른 트랜잭션 화면에 들어가 있는 상태에서, 명령창에 그냥 SE38 을 입력하면 왜 이동하지 않고 경보음만 나며, 왜 꼭 앞에 /n 이나 /o 를 얹어서 /nSE38 로 쳐야 하나요?
**A**: **접두어가 없는 쌩 T-code 명령어는 오직 로그인 첫 화면(Easy Access)의 텅 빈 상태에서만 컴파일 조회가 뚫리며, 다른 작업 화면 내부에서는 현재 세션 인스턴스를 소멸(/n)하거나 새 창을 복제 사출(/o)할지 제어 명령을 명시해야만 이동이 성립하기 때문입니다.**
- **세션 격리 제어**:
  내가 현재 `SE11` 화면에 상주해 있다는 것은, 아바 애플리케이션 서버 램 메모리 상에 'SE11 전용 윈도우 세션 인스턴스' 가 활성화되어 락을 쥐고 있다는 뜻입니다.
  여기서 명령창에 아무 조건 없이 쌩 `SE38` 을 치면, 시스템은 "이 켜져 있는 SE11 세션을 그냥 버리고 가라는 건지, 살려두라는 건지" 물리적 판단을 내리지 못해 명령을 거절하고 경보음을 붑니다.
  - **`/nSE38` (Session Kill)**: "현재 켜져 있는 SE11 세션을 램 메모리 상에서 깨끗이 삭제(Kill)해 버리고, 그 자리에 SE38 화면을 얹어라!" 는 지시입니다. (저장 안 한 데이터가 다 날아가니 각별히 조심해야 합니다.)
  - **`/oSE38` (Session Spawn)**: "현재 SE11 은 그대로 놔두고, 최대 6개 제한 내에서 완전히 새로운 신규 가상 윈도우 세션 창을 복제 사출해 거기에 SE38 을 띄워라!" 는 지시입니다.

### Q3. 연습용 프로그램을 만지다 Local Object 버튼을 툭 눌러 $TMP 패키지에 저장했습니다. 이게 왜 다른 QA 나 운영 서버로 이송이 차단되는 물리 락(Lock)에 걸리며, 어떻게 탈출해야 하나요?
**A**: **로컬 저장을 단행하는 순간, SAP 의 오브젝트 디렉토리 대장 테이블인 TADIR 의 DEVCLASS 필드에 $TMP 가 바인딩되어 CTS 배포 엔진이 이송을 강제 차단하기 때문이며, Object Directory Entry 창을 열어 패키지명을 ZPACKAGE 로 스왑 재배속해야 이송 락이 탁 풀립니다.**
- **TADIR 이송 락 메커니즘**:
  SAP 시스템은 내가 만드는 모든 프로그램 개체를 디렉토리 대장 테이블인 **`TADIR`** 에 기록해 관리합니다.
  Local Object 로 저장을 마치는 순간 `TADIR` 테이블의 패키지명 필드(`DEVCLASS`)에는 **`$TMP`** 가 박힙니다.
  CTS 이송 배송 엔진은 이 `DEVCLASS = '$TMP'` 인 레코드를 만나는 즉시 "이것은 개인 로컬 샌드박스 개체이므로 타 장비 배송(Export)을 법적으로 금지한다" 고 판정해 강제 이송 락을 겁니다.
- **해소 과정**:
  이 락을 풀려면 ZHELLO 프로그램 속성이나 `SE38 -> Go to -> Object Directory Entry` 로드 창을 켭니다. 
  거기서 Package 명칭 `$TMP` 를 도려내고 내가 정식으로 개설한 개발 패키지명(예: `ZBOOKING_PACK`)을 새로 채워 저장(`Ctrl+S`)합니다.
  이 재할당이 일어나는 순간 `TADIR-DEVCLASS` 에 정식 패키지명이 오버라이트 기입되며 이송 락이 해제되고, 시스템이 "이송 기차 번호표(TR)를 생성하십시오" 라며 팝업을 띄우고 길을 열어줍니다.

### Q4. CTS 이송요청서(TR)를 생성하면 왜 부모 Request 와 자식 Task 의 2단계 구조로 나뉘며, 배포 릴리즈(Release)를 때릴 때 왜 자식 Task 를 1순위로 밀봉해야 하나요?
**A**: **자식 Task 는 소스 개체의 수정 권한(Lock)을 쥐고 있는 작업자용 주머니고, 부모 Request 는 선로를 타고 떠날 최종 배송 화물 상자이므로, 작업 도구가 든 자식 주머니를 먼저 닫고 락을 풀어주어야만 부모 상자를 밀봉해 출발(Export)시킬 수 있기 때문입니다.**
- **2계층 선후 프로토콜**:
  아바 이송 요청서(TR)는 아래와 같이 철저한 2단 부모-자식 트리 구조로 결선됩니다.
  - **부모 Request (EK0K900182)**: Basis 팀이 받아서 운영기에 통째 임포트 설치할 진짜 우편 상자입니다.
  - **자식 Task (EK0K900183)**: 개발자 개개인이 소스를 수정할 때 락을 소유하는 작업 바구니입니다.
  만약 자식 Task 가 릴리즈되지 않고 열려(Open) 있는 상태에서 부모 Request 를 먼저 릴리즈하려 하면, 시스템은 **"자식 작업자가 아직 도구를 쥐고 소스를 고치고 있을지 모르는데, 상자를 먼저 밀봉해 우체통에 던질 순 없다"** 고 판단해 기각 락을 뿜습니다.
  따라서 반드시 자식 Task 를 먼저 마우스 우클릭해 **`Release (밀봉)`** 처리해 주어 Z오브젝트의 수정 락을 회수/해제한 뒤, 마지막으로 부모 Request 를 릴리즈해야 비소로 OS 디스크에 배포용 물리 파일이 생성 추출 완료됩니다.

### Q5. WRITE 'Hello' 문장으로 출력한 예매 화면 결과는 데이터베이스에서 실시간으로 긁어서 그리나요, 아니면 메모리 상의 임시 공간에 보관되어 나오나요?
**A**: **WRITE 의 결과는 데이터베이스와 직결되지 않고, 아바 런타임 램 메모리 상에 가설된 임시 인쇄 대기판인 List Buffer 스풀 공간에 1줄씩 스트리밍 적재되어 있다가 프로그램이 종결되는 찰나에 단 1회 SAP GUI 화면으로 통째로 전송되어 데칼코마니 렌더링을 끝마칩니다. **
- **List Buffer 메모리 스풀링**:
  우리가 `WRITE 'Hello'` 를 때릴 때마다 Presentation 화면이 실시간으로 번쩍이며 화소를 켜는 비효율을 자행하지 않습니다.
  아바 WAS 는 임시로 잡힌 메모리 스풀 공간인 **List Buffer** 에 대고 지정 좌표 폭(len)만큼 16진수 바이트 데이터를 한 줄씩 차곡차곡 기입해 둡니다.
  여기에 `WRITE / 'World'` 를 만나면 슬래시 개행을 해석해 Buffer 의 2번째 행 첫 열로 줄 바꿈 적재를 이어 나갑니다.
  최종적으로 프로그램 로직이 다 돌고 종료 신호(End of Program)가 떨어지는 그 찰나에, List Buffer 에 고여 있던 장표 판때기를 통째로 SAP GUI 클라이언트로 전송해 텍스트 화면을 렌더링합니다. 
  이로 인해 실시간 DB 조회 성능 오버헤드와 GUI 통신 지연을 극단적으로 방지할 수 있습니다.
