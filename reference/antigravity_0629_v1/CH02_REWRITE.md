# CH02_REWRITE - 변수·표준 타입·상수·Text Symbol v1

> 목적: `content/abap/CH02`의 6개 레슨을 "IT 비전공자 입문자가 왜 필요한지부터 실습 확인까지 직관적으로 이해하고 따라갈 수 있는 완성 강의자료" 수준으로 재작성한 기준 원고다. 이 문서는 아직 `content/abap` 원본을 덮어쓴 결과가 아니라, 재작업 판정 이후 실제 반영 전 검수 가능한 v1 기준안이다.

## CH02 전체 설계

CH02의 한 문장 목표는 "DATA 구문의 Roll Area 메모리 레이아웃 배치 원리, LIKE 의 컴파일러 선행 참조(Descriptor Object) 제약, Complete/Incomplete 타입 분류 및 Packed Decimal(P) 타입의 BCD(Binary Coded Decimal) 비트 공식, CONSTANTS 상수의 Write-Protected 격리 가드, 그리고 Text Symbol 의 다국어 번역 디스크 테이블 `REPOTEXT` 적재 기작을 수립하여 메모리 변수 통제력을 확립한다"이다.

IT 비전공자 입문자는 소수점 연산 시 오차를 유발하는 `F (부동소수점)` 타입을 돈 계산에 기용해 대형 정산 사고를 내고, `P` 타입의 압축 10진수 바이트 공식인 `(Length * 2) - 1` 의 용량 제한을 무시해 데이터 오버플로우 덤프를 터트린다.
또한, 선언하지 않은 주소의 변수를 아래에서 `LIKE` 로 묶는 컴파일 역전 실수를 가질 뿐 아니라, 상수를 선언하고 코드 중간에 수정하려 들다 빌드 에러를 내며, 하드코딩 다국어 누락을 숨기는 `'텍스트'(001)` 오용으로 전사 다국어 번역을 파괴한다.
따라서 본 챕터는 다음과 같은 단계적 해소 장치로 설계를 배치한다.

1. **DATA & Roll Area**: 런타임 제어 흐름과 관계없이 컴파일러가 기동 시 애플리케이션 세션의 Roll Area 메모리에 정적/동적으로 변수의 바이트 용량을 맵 배정하는 원리.
2. **LIKE Descriptor 의무**: 데이터 사전이 아닌 이미 활성화 완료된 쌩 변수 객체의 Descriptor(메모리 정보 기술자) 주소를 참조하므로 대상 변수보다 반드시 하단에 배치되어야 하는 순서 제약 명세.
3. **P BCD 압축 공식**: 오차 없는 정산 계산을 위해 1바이트당 2자리를 채우고 꼬리에 부호 비트(4비트)를 얹어 자릿수를 `(Length * 2) - 1` 로 산출하는 BCD 공식 수립.
4. **TYPES 순수 틀 설계**: 메모리 바이트를 점유하지 않고 오직 컴파일러의 신택스 검사용 메타 틀로만 작용하는 로컬 `TYPES` 선언.
5. **CONSTANTS Write-Protected**: 런타임 수정을 차단하기 위해 쓰기 방지(Write-Protected) 락이 걸린 램 주소지에 상수를 할당해 재수정 문법 에러를 유발하는 보안 기작.
6. **REPOTEXT 다국어 결선**: 소스 코드 하드코딩 번역 유실을 막기 위해 번역 테이블 **`REPOTEXT`** 에 메시지를 연동하고 `sy-langu` 로그인 언어로 캐싱하는 구조 주입.

## CH02-L01 - 변수 선언(DATA)

### 왜 필요한가

우리가 이전 CH01 에서 `WRITE 'Hello'.` 처럼 따옴표 리터럴(Literal - 코드에 박힌 날값)을 화면에 출력하는 데 성공했다.
그런데 이 출력 대상 텍스트를 프로그램 내에서 열 번 재사용해야 하거나, 콘서트 좌석 수(`100`)에서 예매 신청 좌석 수(`3`)를 차감한 연산 결과를 잠시 메모리에 담아두었다가 파일로 써야 한다면?
- "코드 안에 날값 따옴표를 계속 중복해서 써두면, 나중에 값이 `200` 으로 바뀔 때 흩어진 수십 개 라인을 다 찾아 손수 고쳐야 하고, 그 과정에서 오타가 들어가 시스템이 침몰한다."
한 번 타이핑한 값을 램(RAM) 메모리에 안전하게 박아두고, 필요할 때마다 이름표로 꺼내어 읽고 쓸 수 있는 가상의 보관 용기가 필요하다.
그 보관 용기가 바로 **변수 (Variable)** 다.

**아바 런타임 엔진이 기동될 때 애플리케이션 내부 세션의 Roll Area (사용자별 독립 메모리 영역) 공간에 바이트 크기만큼 주소를 할당받는 DATA 구문의 동작 원리와, 데이터 Descriptor(기술자)를 본떠 생성하는 LIKE 의 컴파일러 순서 제약을 수호하는 기술**이 필요하다. 그것이 **DATA 변수 선언**의 장악이다.

### 무엇인가

#### 1. DATA gv_msg TYPE string (변수 선언과 메모리 배정)
- **DATA**: "램(RAM) 메모리에 지정된 타입의 크기만큼 빈 방(주소)을 확보해 달라" 고 아바 커널에게 명령하는 핵심 키워드다.
- **Roll Area 할당**: **사용자가 F8 실행을 때리면, 프로그램 내의 `DATA` 선언부들은 로직의 순서와 상관없이 컴파일 타임에 즉각 애플리케이션 서버의 'Roll Area' 라는 메모리 구역에 변수의 바이트 폭만큼 물리 주소를 선제 매핑(Static Memory Allocation)받는다. 방이 만들어진 뒤에야 런타임 엔진이 `=` 대입문으로 방 안에 값을 넣을 수 있다.**

#### 2. TYPE vs LIKE (형식 참조와 메모리 객체 본뜨기)
- **TYPE**: `R3TR TABL` 이나 `Data Element` 처럼 전산 사전(DDIC)에 이미 박혀 있는 정적 형식(Data Type)의 형틀 명세를 그대로 적용해 빈 방을 조형한다.
- **LIKE (Descriptor 복제)**: **이미 내 프로그램 세션 안에 DATA 로 선언하여 메모리를 점유하고 있는 실제 '데이터 오브젝트(변수)' 의 구조와 크기 기술자(Descriptor Object)를 거울로 비추듯 똑같이 복제해 새 방을 조형한다.**

#### ⚠️ [ LIKE 사용 시 컴파일러의 선행 참조 상단 배치 제약 ]
- *컴파일러가 위에서 아래로 구문을 파싱해 갈 때 발생하는 물리 순서 제약이다.*
- **원인**: **`LIKE gv_price` 는 메모리에 실체화된 `gv_price` 의 기술 주소를 훔쳐 와야 빌드가 성립된다. 만약 코드 윗줄에 `LIKE gv_price` 를 먼저 기재하고, 정작 `DATA gv_price` 를 그 밑줄에 적으면 컴파일러는 "아직 태어나지 않은 대상의 메모리 구조를 본뜨려 했다" 며 빌드를 전면 거부하고 즉사 에러를 뿜는다. LIKE 대상 원본은 무조건 상단에 먼저 선언 완료되어 상주해야 한다.**

### 어떻게 확인하는가

DATA 체인 구조와 LIKE 의 선행 참조 배선이 체결된 코드를 검증한다.

```abap
REPORT zhello_variable.

" 1. [TYPE 으로 4바이트 정수형 변수를 Roll Area 에 정적 할당]
DATA gv_seats TYPE i VALUE 100. 

" 2. [★ LIKE 로 상단의 gv_seats 메모리 구조를 그대로 본뜸 - 선행 선언 완료!]
DATA gv_booked LIKE gv_seats. 

" 3. [체인 콜론(:)을 활용한 일괄 선언 및 VALUE 초기값 도킹]
DATA: gv_customer TYPE string VALUE '정훈영',
      gv_status   TYPE c LENGTH 1 VALUE 'N'.

" 4. [대입 = 구문으로 값을 램 방에 집어넣음]
gv_booked = 3. 

WRITE: / gv_customer, '님 예매 좌석:', gv_booked.
```

#### 체험/시뮬레이터 설계 (Roll Area 메모리 계판)
- **프로세스 플로우**:
  1. 학습자가 [Roll Area 램 바둑판] 과 [DATA 소스 칩] 을 본다.
  2. [DATA gv_seats TYPE i] 칩을 꽂자, 램 바둑판의 4칸(4바이트)이 파란색으로 촥 점유 마킹되며 `gv_seats` 라벨이 붙는다.
  3. 이때 상단에 [DATA gv_twin LIKE gv_seats] 칩을 꽂고 아래에 gv_seats 를 두자, "Target memory block undefined! Compile Error!" 경고등이 번쩍인다.
  4. gv_seats 칩을 위로 올리자, gv_seats 의 4바이트 파란색 정보가 gv_twin 에 그대로 전사 복제 배선되며 초록불이 켜지는 참조 피드백을 감상한다.
- **상태 및 데이터**:
  - `선언 안 한 gv_unknown 변수에 값을 대입하려 시도한 상태` -> 런타임 결과: `Syntactical check failed. Variable unknown` 하이라이트.
- **피드백**: 변수 선언은 Roll Area 메모리 주소를 할당받는 물리적 설계이며, LIKE 는 반드시 상단에 선행 선언된 타깃을 참조해야 함을 각인한다.

### 실수/주의

- **초기값 VALUE 옵션을 지정하지 않은 채 변수를 선언하고, 기본값이 NULL 일 것이라 오인해 비교 구문을 작성**:
  - 아바의 모든 변수는 NULL 이 없으며, 할당 시점에 타입별 표준 초기값(정수 `0`, 문자 `공백`)으로 즉각 초기화된다.
  - **초기값 비교 시에는 쌩 공백이나 `0` 과 비교해야 오류를 방지할 수 있음을 수호해야 한다.**

### 정리

- 변수 선언은 **`DATA`** 구문으로 램 **`Roll Area`** 메모리에 빈 영역을 할당받는 행위다.
- **`TYPE`** 은 정적 명세 형틀을 빌리며, **`LIKE`** 는 이미 선언된 실체 변수의 형상을 복제한다.
- **`LIKE`** 의 대상 변수는 컴파일러 파싱 정합성을 위해 반드시 **`상단에 먼저 선언`** 완료되어 있어야 한다.

## CH02-L02 - Complete 타입(STRING·I·F·D·T)

### 왜 필요한가

변수 방을 만드는 DATA 선포 방식을 익혔다.
이제 `TYPE` 뒤에 적어줄 'Complete 타입 (완전형 타입)' 의 분류와 크기 제어가 문제다.
- " 사용자의 이름처럼 들어올 글자의 크기가 가변적인 데이터, 혹은 금액과 나이 같은 정수 데이터를 담고 싶다.
이를 위해 I, STRING, D, T 같은 타입을 골라 적으려는데, 이 타입들이 내부적으로 차지하는 물리 바이트 용량과 런타임 나눗셈 연산의 한계는 무엇인가?"
나이를 담는 `I` 타입에 소수점 결산 나눗셈을 실행하면, 아바 엔진이 소수를 쌩 반올림 처리해 날려버림으로써 매월 정산 잔액이 맞지 않는 버그를 뿜는다.
또한 날짜 `D` 타입을 단순 8바이트 정수로 착각해 연산하려다 런타임 덤프를 터트린다.

**길이 지정을 컴파일러 단에서 따로 적지 않아도 되는 완전형 타입(STRING, I, F, D, T)의 물리 바이트 용량 범위와, 4바이트 정수(I) 나눗셈의 반올림 정합성 한계를 규명하는 기술**이 필요하다. 그것이 **Complete 타입의 활용**이다.

### 무엇인가

#### 1. Complete 타입 (완전형 타입)
- 선언 시점에 `LENGTH` 자릿수 크기를 개발자가 쌩으로 명시하지 않아도, 아바 엔진 내부에 "이 타입은 물리적으로 몇 바이트를 쓴다" 고 규격서 상에 이미 100% 설계 완료되어 작동하는 표준 데이터 타입이다.

#### 🧭 [ 5대 표준 Complete 타입 메모리 규격서 ]
- *각 완전형 타입은 메모리 속에서 아래와 같이 정교하게 정의된 바이트 영역을 점유한다.*

```text
[1] STRING (가변 문자열) : 
   내부적으로 dynamic 메모리를 사용해 글자가 들어오는 만큼 램 용량이 가변적으로 늘어난다.
   │
[2] I (4바이트 정수 - Integer) :
   정확히 4바이트(32비트)를 점유하며, 약 -21억 ~ +21억 범위의 순수 소수점 없는 정수를 저장한다.
   │
[3] F (8바이트 부동소수점 - Float) :
   8바이트(64비트)를 점유하며 소수 계산을 소화하나, 컴퓨터의 2진수 한계로 '근삿값(오차)' 이 발생한다.
   │
[4] D (8바이트 날짜 - Date) :
   무조건 8바이트 문자 뼈대를 가지며 `YYYYMMDD` (예: '20260630') 형태의 고정 텍스트로 저장된다.
   │
[5] T (6바이트 시간 - Time) :
   무조건 6바이트 문자 뼈대를 가지며 `HHMMSS` (예: '145200') 형태의 고정 텍스트로 저장된다.
```

#### ⚠️ [ 정수(I) 나눗셈 반올림 충돌 제약 ]
- **정수 `I` 타입 변수에 소수 연산 결과를 덮어씌우면, 아바 런타임은 수학적 버림이 아니라 반올림(Round)을 강제 기동한다.**
- **예**: `7 / 2` 를 실행하면 결과는 `3.5` 이다. 이를 정수 `I` 에 대입하면 자동으로 반올림 처리되어 결과가 `4` 로 저장된다. 만약 `3` 으로 버려질 것이라 착각해 마일리지 정산 코드를 짜면, 전표 금액이 눈덩이처럼 불어나 전산 사고를 유발한다. (소수는 Incomplete `P` 타입을 써야 안전하다.)

### 어떻게 확인하는가

Complete 타입들의 변수 선언과 날짜/정수 대입 시의 내부 연산 결과를 검증한다.

```abap
REPORT zhello_complete.

DATA: gv_name  TYPE string,
      gv_date  TYPE d,
      gv_r1    TYPE i,
      gv_r2    TYPE i.

gv_name = '정훈영 사원'. " 가변적으로 램 방 크기 증가!
gv_date = '20260630'.   " YYYYMMDD 날짜 형식 고정 기입!

gv_r1 = 7 / 2. " 3.5 -> 반올림 기동으로 4 안착!
gv_r2 = 5 / 2. " 2.5 -> 반올림 기동으로 3 안착!

WRITE: / gv_name, 
       / '접속일자:', gv_date,
       / '7/2 결과:', gv_r1,
       / '5/2 결과:', gv_r2.
```

#### 체험/시뮬레이터 설계 (LIKE 배선 참조기)
- **프로세스 플로우**:
  1. 학습자가 [I 정수 주머니] 와 [D 날짜 주머니] 를 본다.
  2. [7 / 2 연산 카드] 를 [I 정수 주머니] 에 넣는다. 주머니 위 스케일러가 3.5 를 가리키다가 찰딱 [반올림 기어] 가 맞물려 돌며 주머니 속에 숫자 `4` 가 채워지는 모습을 확인한다.
  3. 이번엔 날짜 주머니에 따옴표 없이 [20260630] 숫자를 던진다.
  4. 날짜 주머니가 덜컹거리며 "Date type D requires 'YYYYMMDD' text format!" 적색 경고를 뿜어 정합성 오용을 경고하는 연산 피드백을 감상한다.
- **상태 및 데이터**:
  - `소수가 필요한 정산 변수에 TYPE I 를 무단 지정해 꼬여버린 상태` -> 런타임 결과: `Loss of precision! Rounding occurred on settlement` 하이라이트.
- **피드백**: Complete 타입은 저마다 고유 바이트 크기와 날짜 문자열 제약 및 나눗셈 반올림 필터를 지님을 인지한다.

### 실수/주의

- **소수점 계산을 정확히 하겠답시고 8바이트 부동소수점 TYPE F 를 돈 계산(Financial calculation)에 무단 기용**:
  - 부동소수점 `F` 는 `10.00` 이 내부적으로 `9.9999999999...` 로 저장되는 2진수 오차 위험을 가지고 있어, 100만 건 정산 시 1원 단위 빵꾸를 수시로 터트린다.
  - **금융/돈 계산은 무조건 정확한 고정 소수점 TYPE P 를 수호해야 한다.**

### 정리

- **`Complete 타입`** 은 길이를 따로 선고하지 않는 내장 고정/가변형 표준 타입(STRING, I, F, D, T)이다.
- 정수 **`I`** 는 4바이트 크기로서 계산 결과 대입 시 수학적 **`반올림`** 을 단행한다.
- 날짜 **`D`** 는 내부적으로 숫자 데이터가 아닌 `YYYYMMDD` 의 **`8바이트 문자`** 형태로 적재 보존된다.

## CH02-L03 - Incomplete 타입(C·N·P)

### 왜 필요한가

Complete 타입 지정법을 배웠다.
이번에는 "사용자 코드는 무조건 4글자", "금액은 소수 2자리 포함해 8바이트" 와 같이 내 마음대로 크기를 조율해야 하는 'Incomplete 타입 (미완성형 타입)' 의 설계와 소수 압축 계산의 수학이 장벽이다.
- " 금액을 정확히 연산하기 위해 `P` 타입을 쓰려는데, `LENGTH 8` 로 선언하면 이 상자가 내부적으로 담을 수 있는 정수와 소수를 포함한 전체 숫자의 최대 길이는 자릿수로 몇 개인가?"
그리고, 주민번호나 일자 코드에서 특정 자릿수만 쏙 뽑아내어 가공하는 오프셋 연산을 잘못 지정하면, 엄뚱한 메모리 필드를 침범해 데이터를 파괴하는 레이아웃 침범 장애를 겪는다.

**미완성형 타입(C, N, P)의 자릿수 길이 지정법과, Packed Decimal(P) 타입의 BCD 비트 구조 공식인 `자릿수 = (Length * 2) - 1` 및 문자 오프셋 가공(pos(len)) 기술**이 필요하다. 그것이 **Incomplete 타입의 장악**이다.

### 무엇인가

#### 1. Incomplete 타입 (미완성형 타입)
- 선언할 때 뒤에 무조건 `LENGTH n` 을 얹어서 "몇 칸짜리 메모리 공간을 확보하라" 고 개발자가 쌩 자릿수를 명시해 주어야 비로소 컴파일러가 조형을 완수하는 타입이다.

#### 🧭 [ 3대 Incomplete 타입 자릿수 매칭 명세 ]
- *각 미완성형 타입은 선고된 자릿수 크기에 따라 물리 메모리를 조율한다.*

```text
[1] C (고정 문자 - Character) :
   LENGTH 로 지정한 글자 수만큼의 1바이트 텍스트 방을 확보한다. (기본값: 공백 채움)
   │
[2] N (숫자 문자 - Numeric Text) :
   주민번호나 우편번호처럼 '숫자 형태로만 구성된 문자열' 이다.
   정수 I 에 '00123' 을 넣으면 앞의 0이 잘려 '123' 이 되지만, N LENGTH 5 에 넣으면 '01234' 앞자리 0이 온전히 보존된다.
   │
[3] P (압축 소수 - Packed Decimal) :
   오차 없는 정밀 소수를 표현하기 위한 아바 최고의 실무용 돈 계산 타입이다.
   1바이트에 10진수 숫자 2자리를 쪼개어 압축해 담는 BCD(Binary Coded Decimal) 기법으로 데이터 크기를 절반 절약한다.
```

#### ⚠️ [ Packed Decimal P 타입 BCD 자릿수 산출 공식 ]
- **`LENGTH len` 으로 선언된 P 타입 변수가 담을 수 있는 순수 최대 10진수 자릿수(DECIMALS 포함) 공식은 `자릿수 = (len * 2) - 1` 이다.**
- **이유**: **마지막 1바이트의 반(4비트) 공간은 이 숫자가 양수(+)인지 음수(-)인지 판정하는 부호 비트로 할당되어 숫자를 담지 못하기 때문이다. 따라서 `LENGTH 8` 이면 `(8 * 2) - 1 = 15` 자리의 숫자를 정수/소수 총합해서 담을 수 있다.** (만약 `DECIMALS 2` 를 지정했다면, 정수 13자리 + 소수 2자리 = 총 15자리를 저장할 수 있어 한도 초과 시 오버플로우 덤프가 발생한다.)

#### 2. 오프셋/길이 구간 추출 (pos(len) 문자 조작)
- `C` 나 `N` 같은 고정 문자열 뒤에 `+시작위치(가져올길이)` 를 붙이면, 0번 인덱스부터 시작해 특정 구간의 문자 뼈대만 칼로 싹 도려내어 읽거나 오버라이트 갱신할 수 있다. (예: `gv_date+0(4)` -> 앞 4자리 연도만 절단).

### 어떻게 확인하는가

P BCD 공식과 오프셋 갱신이 결합된 코드를 검증한다.

```abap
REPORT zhello_incomplete.

" 1. [ P 타입 선언: Length 8 이므로 BCD 공식 (8*2)-1 = 최대 15자리 수용! ]
"    그중 소수 2자리이므로, 정수는 최대 13자리(15-2)까지 담을 수 있음!
DATA gv_salary TYPE p LENGTH 8 DECIMALS 2.

" 2. [ C 고정문자 선언: 지정 길이만큼 빈 방 셋업 ]
DATA gv_ymd TYPE c LENGTH 8 VALUE '20260630'.
DATA gv_year TYPE c LENGTH 4.

" 3. [ 오프셋 가공 기동: 0번째 자리부터 4글자 쏙 뽑아내기 ]
gv_year = gv_ymd+0(4). " '2026'

" 4. [ 오프셋 갱신: 끝 2자리를 '01' 일자로 갱신 ]
gv_ymd+6(2) = '01'. " '20260601' 로 변경!

gv_salary = '9999999999999.99'. " 최대 15자리 한도 정밀 기입 확인 (오차 없음!)

WRITE: / '연도:', gv_year,
       / '변경된 일자:', gv_ymd,
       / '연봉 한도:', gv_salary.
```

#### 체험/시뮬레이터 설계 (BCD 압축 바이트 연산기)
- **프로세스 플로우**:
  1. 학습자가 [P LENGTH 8 주머니] 와 [숫자 전광판] 을 본다.
  2. [9999999999999.99] (정수 13자리 + 소수 2자리 = 총 15자리) 를 기입하고 주머니에 넣는다. 주머니 바이트 눈금 8개가 파란색으로 100% 꽉 채워지고 안전 합격등이 켜진다.
  3. 자릿수를 1개 늘려 [10000000000000.00] (총 16자리) 으로 입력하고 밀어 넣는다.
  4. 주머니가 한도 초과로 부풀어 오르며 "BCD Bit overflow! Maximum 15 digits exceeded! Dump!" 적색 경보 사이렌이 요란하게 울리는 연산 한도 피드백을 감상한다.
- **상태 및 데이터**:
  - `금액 데이터를 BCD 한도 계산 없이 Length 3 으로 너무 짧게 지정한 상태` -> 런타임 결과: `CX_SY_CONVERSION_OVERFLOW. Arithmetic overflow` 하이라이트.
- **피드백**: Incomplete P 타입은 용도에 맞는 LENGTH 지정 시 반드시 BCD 공식 `(len*2)-1` 을 밟아 정밀 설계해야 함을 체득한다.

### 실수/주의

- **문자 C LENGTH n 선언 시 n 을 빠뜨리고 그냥 TYPE c 로만 쌩 선언**:
  - 이 실수 시 컴파일러는 디폴트 길이를 딱 `1` 바이트로 단절해 버리므로, 'ABCD' 를 대입해도 앞 대가리 'A' 만 남고 나머지는 영구 훼손되어 잘려 나간다.
  - **C 와 N 선언 시에는 무조건 `LENGTH` 자릿수 선고를 수호해야 한다.**

### 정리

- **`Incomplete 타입`** 은 뒤에 `LENGTH` 로 공간 폭을 선고해야 조형되는 타입(C, N, P)이다.
- **`N`** 은 숫자 형태의 코드(우편번호 등)를 담을 때 앞자리 0을 수호하기 위해 쓴다.
- **`P`** 타입의 최대 자릿수는 **`(Length * 2) - 1`** 공식이 적용되며 마지막 반 바이트는 부호에 귀속된다.
- 문자열 변수의 구간 절단 및 부분 갱신은 **`+시작(길이)`** 오프셋 구문으로 돌파한다.

## CH02-L04 - Local Type(TYPES) 재사용

### 왜 필요한가

Incomplete 타입의 폭 지정을 완수했다.
이제 금액이나 우편번호에 할당할 `p length 8 decimals 2` 같은 동일 선고를 프로그램 내에서 수십 번 반복 타이핑해야 하는 '관심사 중복' 이 장벽이다.
- " 같은 정산 금액 변수 `price`, `tax`, `total` 을 만드는데, 구문을 한 곳에서 정의하고 몽땅 가져다 쓰고 싶다. 
정의를 한 곳에서 빚어두는 TYPES 는 DATA 선언과 비교해 메모리 점유 시 어떤 구조적 차이를 보이는가?"
TYPES 로 정의한 틀 이름에 대고 직접 `=` 값을 대입하려 들다 "TYPES 에는 값을 담을 수 없다" 고 아바 컴파일 에러를 뿜고 폭사한다.

**TYPES 로 독자적인 로컬 형틀(Type)을 빚고, 이 틀은 램 메모리 바이트를 1자도 차지하지 않는 메타 데이터 규격이며, 오직 DATA 선언이 이 틀을 조형용 참조(TYPE)로 쓴다는 물리적 차이를 규명하는 기술**이 필요하다. 그것이 **Local Type (TYPES)** 의 완수다.

### 무엇인가

#### 1. TYPES ty_amount TYPE p LENGTH 8 DECIMALS 2 (타입의 뼈대 정의)
- **TYPES**: **메모리(RAM)의 바이트 공간을 단 1비트도 점유(할당)하지 않는 순수한 '개체 설계 도면(Type)' 을 프로그램 내에 생성하는 명령이다. 값이 들어올 방을 설계하지만, 실제 집을 짓지는 않는다.**
- **ty_ 접두어**: Types 의 약자로서, 변수(gv_)와 구별하기 위해 접두어 `ty_` 를 이름표에 장착하는 것이 실무 표준 아바 Conventions 이다.

#### 🧭 [ TYPES vs DATA 메모리 점유 메커니즘 명세 ]
- *두 명령어는 아바 세션 메모리(Internal Session) 상에서 아래와 같은 극단적인 역할 분담을 가진다.*

```text
[TYPES 선언] : ty_amount TYPE p ... 
   - 램 메모리 점유 : 0 바이트 (메모리 공간을 잡지 않는다.)
   - 역할 : 컴파일러의 뇌세포에 "앞으로 ty_amount 라는 명칭의 틀이 오면 소수 2자리의 Packed 형식을 가진 도면으로 해석하라" 고 명세 사전에 기록만 해둔다.
   │
   ▼ ( TYPE ty_amount 로 연결 결선 )
   │
[DATA 선언] : gv_price TYPE ty_amount.
   - 램 메모리 점유 : 8 바이트 (실제 Roll Area 메모리에 물리 주소를 뚫는다.)
   - 역할 : ty_amount 도면을 펴고, 그 스펙 그대로 8바이트 압축 10진수 메모리 방을 지어 올린다. (이 방에 값을 넣는다.)
```

### 어떻게 확인하는가

TYPES 로 형틀을 개설하고 DATA 가 이를 TYPE 지시어로 소비하는 소스 구조를 검증한다.

```abap
REPORT zhello_types.

" 1. [★ TYPES 로 금액 전용 ty_amount 로컬 도면 개설 - 0바이트 점유!]
TYPES ty_amount TYPE p LENGTH 8 DECIMALS 2.
TYPES ty_code   TYPE c LENGTH 4.

" 2. [DATA 변수 선언 시 TYPE 뒤에 방금 빚은 ty_amount 도면을 배선 매칭!]
DATA: gv_price TYPE ty_amount,
      gv_tax   TYPE ty_amount,
      gv_total TYPE ty_amount,
      gv_ccode TYPE ty_code.

" [가짜 대입 시도 검증]
" ty_amount = '100.00'. " <- 이 짓을 하면 "ty_amount is a TYPE, not a variable" 빌드 즉사 에러!

" 3. [진짜 데이터 방에 대입 집행]
gv_price = '100.00'.
gv_tax   = gv_price * '0.1'.
gv_total = gv_price + gv_tax.

WRITE: / '단가:', gv_price, 
       / '부가세:', gv_tax, 
       / '총합:', gv_total.
```

#### 체험/시뮬레이터 설계 (TYPES 조형기)
- **프로세스 플로우**:
  1. 학습자가 [TYPES 붕어빵 형틀 ty_amount] 과 [DATA 붕어빵 밀가루] 를 본다.
  2. [TYPES ty_amount] 레버를 당기자, 램 메모리 칸에 아무 불도 안 켜지고 [0 Bytes] 전광판이 유지된다. (껍데기 틀만 등록됨.)
  3. [DATA gv_price TYPE ty_amount] 칩을 붕어빵 틀에 넣자, 램 메모리 기판 8칸에 노란 불이 들어오며 `gv_price` 주소가 실물 생성된다.
  4. 이때 [ty_amount = 100] 대입 플러그를 붕어빵 틀에 꽂으려 들자, [Type Assignment Blocked] 적색 사이렌 경보가 울리는 정합성 피드백을 감상한다.
- **상태 및 데이터**:
  - `TYPES 의 ty_ 접두어를 빠뜨리고 DATA 변수와 동일하게 gv_ 로 지어 명칭이 꼬여버린 상태` -> 런타임 결과: `Name conflict. gv_amount already defined as type` 하이라이트.
- **피드백**: TYPES 는 메모리를 점유하지 않는 메타 형틀이며, DATA 가 집을 짓기 위한 설계 규격서임을 체득한다.

### 실수/주의

- **TYPES 를 작성해 둔 것이 다른 아바 프로그램에서도 불러와 쓸 수 있는 글로벌 사전 객체라고 착각**:
  - 이 챕터의 TYPES 은 오직 현재 소스 텍스트 파일 안에서만 통하는 '로컬 타입' 이며, 타 프로그램이 찌르면 그런 타입이 없다며 컴파일 폭사가 난다.
  - **전사 공유 타입은 CH03 에서 배울 DDIC (Data Element) 로 올려야 함을 수호해야 한다.**

### 정리

- **`TYPES`** 문은 변수가 아니며, 메모리를 점유하지 않는 순수 **`로컬 형틀`** 을 선포한다.
- 타입명은 **`ty_`** 접두어를 달아 실제 변수명(`gv_`)과 관심사를 분리 격리한다.
- 값을 대입하고 계산을 가하는 실체 메모리 방은 오직 **`DATA`** 가 구축한다.

## CH02-L05 - CONSTANTS — 값에 이름 붙이기

### 왜 필요한가

로컬 TYPES 형식까지 조립했다.
이번에는 프로그램 전반에서 사용되는 '원주율 3.14159' 나 '할인율 0.1' 과 같이 절대 수정되면 안 되는 고정 값들의 '변조 위협' 이 장벽이다.
- " 프로그램 연산에 쓰이는 매직 넘버(Magic Number) 값들을 변수 `gv_pi` 에 담아 썼다.
그런데 후임자 정훈영 사원이 실수로 루프 계산 도중 `gv_pi = 3` 이라고 오버라이트 수정 코딩을 해버려, 그 뒤로 계산된 모든 면적 결산 금액이 조용히 뒤틀려 빵꾸가 나는 장애가 났다."
중간에 덮어써져 코드가 파손되는 것을 원천 방어하기 위해, 값을 읽기 전용으로 단단히 굳히는 기술이 필요하다.
그것이 **상수 (CONSTANTS)** 의 완수다.

**CONSTANTS 를 선포하고 VALUE 로 값을 영구 고정하며, 이 상수는 램 상에서 Write-Protected (쓰기 방지) 메모리 영역에 주소가 격리 배정되어 재수정 시 런타임이 아닌 빌드 단계에서 컴파일러가 차단하는 기술**이 필요하다. 그것이 **CONSTANTS 의 활용**이다.

### 무엇인가

#### 1. CONSTANTS gc_pi TYPE f VALUE '3.14159' (상수 선언)
- **CONSTANTS**: **"이 주소지의 메모리는 읽을 수만 있고, 프로그램이 끝날 때까지 값을 변경하려 들면 강제로 에러를 내며 빌드를 막으라" 고 아바 컴파일러에게 지시하는 읽기 전용 락 선언문이다.**
- **gc_ 접두어**: Global Constant 의 약자로서, 접두어 `gc_` 를 이름표에 장착하는 것이 실무 표준 아바 Conventions 이다.

#### 🧭 [ 상수의 Write-Protected 락 메모리 기작 명세 ]
- *아바 커널은 상수가 선언되면 메모리 맵 상에서 아래와 같이 안전 격리막을 가동한다.*

```text
[DATA 변수 메모리 주소] : gv_pi
   - 메모리 영역 : Read/Write (읽고 쓰기 자유)
   - 런타임 동작 : 언제든 '=' 대입을 만나면 램의 비트 값이 오버라이트 갱신된다.
   │
[CONSTANTS 상수 메모리 주소] : gc_pi
   - 메모리 영역 : Write-Protected (쓰기 금지 잠금 영역 배정)
   - 컴파일 동작 : 컴파일러 파서가 gc_pi 를 대입 연산자 '=' 의 좌변(Left-hand side)에서 발견하는 즉시, 
                   "Constant cannot be changed" 적색 문법 에러를 뿜으며 컴파일 자체를 강제 기각 차단한다.
```

#### 2. abap_true 와 abap_false (SAP 표준 불리언 상수)
- 참(`'X'`)과 거짓(`공백`) 상태를 하드코딩 리터럴로 쓰는 해악을 막기 위해 SAP 가 공인 제공하는 시스템 표준 상수다. (코드 가독성을 수호한다.)

### 어떻게 확인하는가

상수를 셋업하고 수정 시도 시 컴파일러가 락을 기동해 에러를 뿜는 코드를 검증한다.

```abap
REPORT zhello_constant.

" 1. [★ CONSTANTS 로 원주율 상수 gc_pi 영구 고정 선포 - gc_ 접두어 장착]
CONSTANTS gc_pi TYPE p LENGTH 4 DECIMALS 3 VALUE '3.142'.
CONSTANTS gc_discount TYPE p LENGTH 3 DECIMALS 2 VALUE '0.10'.

DATA gv_price TYPE p LENGTH 8 DECIMALS 2 VALUE '10000'.
DATA gv_final TYPE p LENGTH 8 DECIMALS 2.

" [가짜 대입 시도 및 컴파일 게이트 검증]
" gc_pi = '3.000'. " <- 이 짓을 적는 즉시 아바 컴파일러가 "gc_pi is constant" 에러 빌드 차단!

" 2. [상수에서 값을 안전하게 읽어서 비즈니스 계산 처리]
gv_final = gv_price - ( gv_price * gc_discount ).

WRITE: / '원주율 상수:', gc_pi,
       / '할인율 상수:', gc_discount,
       / '최종 정산가:', gv_final.
```

#### 체험/시뮬레이터 설계 (Write-Protected 상자 락)
- **프로세스 플로우**:
  1. 학습자가 [Write-Protected 메모리 쇠상자 gc_discount] 와 [대입 플러그 = 0.20] 을 본다.
  2. 쇠상자 겉면에 [VALUE = 0.10] 칩이 단단히 용접되어 있다.
  3. [대입 플러그] 를 쇠상자에 강제로 꽂으려 대자, 상자 위에서 [컴파일 수문장] 이 삑- 호루라기를 불며 "Write protection violation! Compile blocked!" 적색 자물쇠 락을 찰딱 채우는 모션을 확인한다.
  4. 플러그를 뽑고, [읽기 케이블] 만 연결하자 상자 속 0.10 데이터가 안전하게 나와 계산기로 공급되는 정합성 피드백을 감상한다.
- **상태 및 데이터**:
  - `상수 CONSTANTS 선언 시 VALUE 초기값을 깜빡 빠뜨리고 쌩 빌드한 상태` -> 런타임 결과: `Syntax error. Value missing for constant` 하이라이트.
- **피드백**: 상수는 덮어쓰기 사고를 원천 차단하는 컴파일 가드레일 장치이며, 변경을 차단하기 위한 필수 통제임을 체득한다.

### 실수/주의

- **자주 바뀔 가능성이 있는 파라미터 값(예: 올해 부가세율 0.1)을 굳이 상수로 못 박아두고, 나중에 세법이 바뀌었을 때 소스 코드 전체를 다 뜯어고쳐 이송 트레인을 다시 날리는 실수**:
  - **진짜 변하지 않는 물리 상수(원주율, SAP 표준 상태값)만 CONSTANTS 로 박고, 세법이나 설정 값은 변수나 Configuration DB 테이블을 찌르게 설계하는 것이 유지보수의 안전 룰이다.**

### 정리

- 절대 변경 불가능한 고유 값은 **`CONSTANTS`** 상수로 굳힌다.
- 상수는 **`gc_`** 접두어를 달아 선언하며, 반드시 **`VALUE`** 초기화 지정을 완료해야 한다.
- 컴파일러가 **`Write-Protected`** 메모리 영역 제약을 기동하여, 상수의 임의 대입 시도를 빌드 단계에서 차단한다.
- 참/거짓 플래그는 날것의 'X' 대신 SAP 표준 상수 **`abap_true`** 와 **`abap_false`** 를 쓴다.

## CH02-L06 - Text Symbol — 화면 글자의 다국어

### 왜 필요한가

상수와 TYPES 셋업까지 완료했다.
이제 화면 리포트에 출력할 한글 텍스트 '예약번호' 나 '결제금액' 의 '다국어 번역' 이 마지막 장벽이다.
- " 화면에 뿌릴 한글 텍스트를 소스 코드에 `WRITE '예약번호'.` 라고 쌩으로 하드코딩해 두었다.
미국 지점 지사장 정훈영 대리가 영어 계정으로 로그인해 프로그램을 켰는데, 영어 번역 화면 대신 여전히 '예약번호' 라는 한글 날것이 삐져나와 지점 사용을 방해하는 장애가 났다."
다국어 글로벌 시스템인 SAP 에서 소스 코드 내에 특정 국가 언어를 하드코딩해 두는 행위는 형상 품질을 파괴하는 행위다.

**화면 텍스트를 코드에서 분리해 세 자리 번호(TEXT-001)로 빼서 관리하고, 이 번역 데이터가 DB 테이블 `REPOTEXT` 에 언어별 키로 적재되어 `sy-langu` 로그인 언어에 맞춰 동적 렌더링되게 만드는 기술**이 필요하다. 그것이 **Text Symbol** 의 완수다.

### 무엇인가

#### 1. Text Symbol (텍스트 심볼 리소스)
- **프로그램 소스 텍스트 파일 내에 글자 리터럴을 박아두지 않고, 세 자리 숫자 번호표(예: 001)를 달아 외부 텍스트 저장실에 다국어 사전 형태로 분리 수납하는 다국어 격리 기술이다.**

#### 🧭 [ 다국어 번역 데이터베이스 REPOTEXT 테이블 연동 구조 ]
- *텍스트 심볼을 적으면, SAP 백엔드 데이터베이스 상에서는 아래와 같은 번역 배선망이 작동한다.*

```text
[1] 번역 딕셔너리 테이블 : REPOTEXT
   - 키 필드 : PROGRAM (프로그램명), LANGU (언어 키 - KO, EN, JA), KEY (세 자리 번호 - 001)
   - 데이터 필드 : ENTRY (실제 다국어 번역 텍스트)
   │
[2] 텍스트 작성 시점 :
   - 한글 로그인 상태에서 001 번에 '예약 번호' 입력 -> REPOTEXT 테이블에 KO 언어로 적재된다.
   - 영어 로그인 상태에서 001 번에 'Booking No' 입력 -> REPOTEXT 테이블에 EN 언어로 적재된다.
   │
[3] F8 실행 런타임 격발 :
   - 아바 커널은 사용자의 세션 로그인 언어 시스템 변수인 sy-langu (예: 'E') 값을 낚아챈다.
   - REPOTEXT 테이블에서 LANGU = 'E' 이고 KEY = '001' 인 레코드의 'Booking No' 를 램 메모리로 퍼 올린다.
   - 모니터 화면에는 한글 대신 깔끔한 영어 라벨이 렌더링되어 다국어가 수호된다.
```

#### 2. TEXT-001 vs '금액'(001) (번역 누락 검출 게이트의 차이)
- **`TEXT-001` (클린 번호 선언)**: **번역실에 001 번에 대칭되는 텍스트가 존재하지 않으면, 화면에 쌩 공백을 출력한다. 이로 인해 개발/QA 단계에서 "번역이 누락되었다" 는 장애 사실이 눈에 띄어 즉각 교정된다.**
- **`'금액'(001)` (기본값 병기 선언)**: **번역이 누락되면 괄호 앞의 디폴트 글자('금액')를 화면에 우회 방출한다. 화면이 깨지지는 않지만, 번역 누락 상태가 숨겨져 영문 사용자 화면에 계속 한글이 노출되는 다국어 누락 버그를 방출한다. 실무에선 TEXT-nnn 이 권장된다.**

#### 3. 헝가리안 명명법 (Conventions)
- **`gv_` (Global Variable)**: 프로그램 전체에서 살아 움직이는 전역 변수 배지다.
- **`lv_` (Local Variable)**: 서브루틴이나 메서드 내부에서만 잠깐 살다 소멸하는 로컬 변수 배지다.

### 어떻게 확인하는가

텍스트 심볼을 활용해 번역 누락을 방지하고 다국어로 출력하는 코드를 검증한다.

```abap
REPORT zhello_textsymbol.

" 변수 명명 표준 준수 (gv_)
DATA gv_amt TYPE p LENGTH 8 DECIMALS 2 VALUE '50000.00'.

WRITE: / sy-uline.

" 1. [★ 다국어 수호: 하드코딩 대신 텍스트 심볼 TEXT-nnn 으로 라벨 출력!]
"    TEXT-001 은 번역실(REPOTEXT) 에 '예약 번호'/'Booking ID' 로 다국어 등록되어 있음!
WRITE: / TEXT-001, ':', 'T0001'. 

" 2. [ TEXT-002: 번역실에 '결제 금액'/'Amount' 로 매칭 등록된 심볼 ]
WRITE: / TEXT-002, ':', gv_amt.

" 3. [비추천 괄호형: 번역 누락 시 '기본글자' 가 출력되나 누락 검출을 방해함]
" WRITE: / '할인율'(003), ':', '10%'. 

WRITE: / sy-uline.
```

#### 체험/시뮬레이터 설계 (REPOTEXT 다국어 번역 디스플레이)
- **프로세스 플로우**:
  1. 학습자가 [REPOTEXT DB 테이블] 과 [Fiori Elements 텍스트 라벨창] 을 본다.
  2. [로그인 언어 = KO] 로 설정하고 조회를 때린다. REPOTEXT 에서 `LANGU = KO` 인 '예약 번호' 데이터를 가져와 화면에 초록 불로 한국어를 그린다.
  3. 이번엔 [로그인 언어 = EN] 으로 교체한다.
  4. DB 에 대치하는 영어 번역 `LANGU = EN` 이 존재하지 않자, [TEXT-001] 로 작성된 칸은 쌩 흰색 공백이 되어 "번역 누락 포착 경보" 가 울리고, 번역을 기입해 채우자 영어로 찰딱 렌더링을 끝마치는 다국어 피드백을 감상한다.
- **상태 및 데이터**:
  - `TEXT-001 과 '텍스트'(001) 의 괄호 번호를 세 자리가 아닌 1 두 자리로 오기해 빌드한 상태` -> 런타임 결과: `Syntax error. Text symbol key must be 3-digit numeric` 하이라이트.
- **피드백**: 화면의 다국어 수호는 텍스트 심볼과 REPOTEXT 테이블의 매칭이며, TEXT-nnn 규격이 누락 방지에 최선임을 이해한다.

### 실수/주의

- **프로그램 내에서 사용자에게 즉시 경고로 띄우는 알림 팝업 메시지(MESSAGE)까지 모두 Text Symbol 로 때려박아 남발**:
  - 텍스트 심볼은 화면에 박혀 움직이지 않는 '고정 라벨/제목' 용도다. 
  - **사용자 알림 경고는 번역과 메시지 클래스를 타는 전용 MESSAGE ID 로 빚어 쏴야 함을 수호해야 한다.**

### 정리

- 화면 고정 글자는 하드코딩을 배제하고 다국어를 위해 **`Text Symbol`** 로 격리한다.
- 텍스트 심볼은 번역 데이터베이스 테이블인 **`REPOTEXT`** 에 언어별로 적재된다.
- **`TEXT-nnn`** 은 번역 누락 시 공백을 내어 누락을 조기에 검출하고 방어한다.
- 전역 변수는 **`gv_`**, 로컬 변수는 **`lv_`**, 상수는 **`gc_`** 로 명명하는 표준 Conventions 을 수호한다.
