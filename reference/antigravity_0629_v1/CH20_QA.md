## 4. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. CLASS_CONSTRUCTOR 정적 생성자 내부에서 RAISING 절 예외 전파가 언어 명세상 전면 금지되는 아키텍처적 사유는 무엇인가요?
**A**: **클래스를 처음으로 호출하여 정적 생성자를 작동시키는 주체가 누구인지 컴파일 시점에 알 수 없어, 호출자에게 예외 처리의 의무 계약을 강제할 수 없기 때문입니다.**
인스턴스 생성자인 `constructor` 는 개발자가 직접 코드에 `NEW zcl_booking_manager( )` 라고 명시해 만드는 시점에 실행되므로, 바로 겉을 `TRY/CATCH` 로 감싸서 예외를 안전하게 낚아챌 수 있습니다.
반면 정적 생성자인 `class_constructor` 는 프로그램 흐름 중 언제 최초 실행될지 모릅니다. 
어떤 동료 개발자가 그저 `gv_ver = zcl_booking_manager=>gv_version.` 이라며 정적 변수 하나를 평화롭게 읽었을 뿐인데, 그 첫 액세스 때문에 뜬금없이 `class_constructor` 가 기동되고 그 안에서 예외가 뿜어 나온다면, 단지 정적 변수를 읽으려던 호출자는 예외가 터질 것을 전혀 알지도 못했고 `TRY/CATCH` 를 치지도 않아 결국 시스템이 덤프 폭사를 맞게 됩니다. 
이로 인해 아바 엔진은 정적 생성자 머리꼬리에 `RAISING` 을 적는 행위를 문법 차단하며, **"정적 영역의 초기화 에러는 반드시 내부 소스 코드 안에서 스스로 TRY/CATCH 로 완수해 소멸시키라"** 는 책임을 부과하는 물리 아키텍처입니다.

### Q2. CATCH 구문을 적을 때 하위 예외(zcx_fully_booked)보다 상위 예외(cx_root)를 위에 적으면 컴파일 오류가 나거나 하위 분기가 작동하지 않는 기하학적 매커니즘을 알려주세요.
**A**: **TRY-CATCH 문은 위에서 아래로 순차 스캔하며 예외 타입의 적합성을 검문하므로, 입구에 배치된 포괄적 부모 그물이 자식 예외까지 몽땅 낚아채어 아래 구체 핸들러를 무력화(Shadowing)시키기 때문입니다.**
예외 클래스의 상속 계보는 `zcx_fully_booked -> cx_static_check -> cx_root` 순으로 수립되어 있습니다. 즉, `zcx_fully_booked` 은 `cx_root` 의 한 종류(자식)입니다.
만약 `CATCH cx_root` 를 맨 위에 두면, `zcx_fully_booked` 라는 예외가 쿼리에서 튀어 올랐을 때, 아바 엔진은 1번 검문소에서 "너 `cx_root` 계열 맞니?" 하고 검사합니다. 
자식은 부모의 피를 이어받았으므로 당연히 "예, 맞습니다" 라고 판정되어 1번 `CATCH` 방으로 촥 흡수 수송됩니다. 
이로 인해 2번 라인에 열심히 적어둔 `CATCH zcx_fully_booked` 는 냄새도 맡아보지 못하고 지나가는 데드코드가 됩니다. 
따라서 문법 검사기나 컴파일러는 이 순서 오류를 감지하면 **"Hides catch block"** 이라는 syntax error를 띄워 빌드를 정지시킵니다. 그물은 무조건 **가장 촘촘하고 좁은 자식 그물(하위 예외)부터 상단에 차곡차곡 배치**해야 정밀 낚시가 성립합니다.

### Q3. lo_mgr 참조 변수를 선언만 해두고 NEW 객체 생성을 생략한 채 화살표(->)를 찔러 호출하면 발생하는 GET_REF_TO_INCONSISTENT 덤프의 원인과 구체적 방어 코드는 무엇인가요?
**A**: **명함 참조 변수방에 실제 메모리 주소가 들어있지 않은 빈 상태(INITIAL)인데, 가상의 유령 주소지를 해독(Dereference)하여 내부 메서드를 기동하려 들다 메모리 파서(Parser)가 길을 잃고 터지는 정밀 번지수 누수 장애이며, IS BOUND 로 원천 방어합니다.**
- **덤프의 물리 원인**:
  `DATA lo_mgr TYPE REF TO zcl_booking_manager.` 이 실행되면 메모리에는 32비트 혹은 64비트 크기의 **'메모리 주소 전용 수납함'** 하나만 달랑 개설되고, 안은 빈 문자열인 `INITIAL` (Null 주소) 상태가 됩니다.
  여기에 대고 `lo_mgr->book( )` 을 지르면, 컴퓨터는 "lo_mgr 수납함 안에 들어있는 번지수로 찾아가라" 는 지시를 해독하려는데, 안에 `'0x00000000'` (존재하지 않는 지하 공간) 주소가 적혀있어 갈 곳을 잃어버리고 시스템 전체를 셧다운시키는 **`GET_REF_TO_INCONSISTENT`** 덤프를 터트립니다.
- **구체적 방어 코드**:
  반드시 호출 전에 이 주소 수납함에 실제 메모리 주소가 바인딩되어 올바른 객체를 가리키고 있는지 검문하는 **`IS BOUND`** 논리 식을 물려 방어해야 합니다.
  ```abap
  " [ 안전 가드가 수립된 인스턴스 기동 정석 ]
  IF lo_mgr IS BOUND.
    lo_mgr->book( iv_seats = 2 ).
  ELSE.
    " 덤프 폭사 우회 후 긴급 생성 또는 안내 처리
    lo_mgr = NEW #( ).
    lo_mgr->book( iv_seats = 2 ).
  ENDIF.
  ```

### Q4. 자식 클래스에서 부모 클래스의 private 멤버를 상속받아 직접 호출할 수 없는 격리 한계는 무엇이고, protected 및 super-> 를 통한 해결법을 설명해 주세요.
**A**: **private 구역은 클래스의 소스 코드 내벽 밖으로의 데이터 유출을 차단하는 정적 캡슐화 방어선이므로 상속을 받은 자식이라도 침투가 금지되며, protected 로 상속권한을 개방하고 super-> 로 부모 코드를 빌려 써서 통제합니다.**
- **격리 한계**: 
  부모의 `private` 멤버 변수는 부모 본인의 `METHOD ... ENDMETHOD` 중괄호 안쪽 소스 코드에서만 메모리 쓰기/읽기 컴파일러 바인딩이 일어납니다. 자식은 뼈대를 상속받을 뿐, 부모 코드의 내벽 공간이 아니므로 문법적으로 private 접근이 가로막힙니다.
- **해결법**:
  1. **`PROTECTED SECTION` 개설**: 
     외부 범용 프로그램에는 숨겨두고 오직 상속받은 자식의 내벽 코드 안에서만 변수를 읽고 쓰게 허용하고 싶다면, 부모 클래스 설계 시 해당 `DATA` 속성을 private 이 아닌 **`PROTECTED`** 구역으로 지정 승격해야 자식이 `me->mv_capacity` 로 만질 수 있습니다.
  2. **`super->` 메서드 위임**:
     자식 내부에서 굳이 데이터를 직접 변경하지 않고, 부모가 빚어둔 검증 로직을 고스란히 태우고 싶다면, 자식의 재정의 메서드 안에서 **`super->book( )`** 을 명시 기동해 줍니다. 
     이로 인해 제어권이 부모 메서드로 우회 수송되어, 부모의 private 속성에 부모 메서드가 안전하게 대입 작업을 완수하고 다시 자식에게 바통을 넘겨주는 대리인 아키텍처가 성립해 모순이 말끔히 해결됩니다.

### Q5. CASE TYPE OF 분기 시 자식 구체 타입보다 부모 추상 타입을 위에 배치하면 상속 다형성 판정이 왜곡되는 물리적 스캔 순서 제약은 무엇인가요?
**A**: **CASE TYPE OF 문은 위에서 아래로 실시간 동적 타입(Dynamic Type) 호환 검사를 집행하므로, 상위에 포괄적인 부모 필터가 서식하면 자식 객체도 부모의 상속 규격에 통과되어 1번 분기에 먼저 함락 납치되기 때문입니다.**
`ZCL_VIP_BOOKING` (자식) 객체를 `lo_mgr` 변수에 업캐스팅하여 얹었습니다.
실제 런타임에 작동하는 스캔 순서는 다음과 같습니다.
1. `CASE TYPE OF lo_mgr.` 스캔 시작.
2. 1번 분기: `WHEN TYPE zcl_booking_manager (부모)` 검문소 작동.
3. 엔진 판정: "lo_mgr 안의 알맹이가 VIP 자식인데, 자식은 부모의 피를 이식받았으므로 zcl_booking_manager 계열이 맞는가?" -> **"예, 부모 클래스 규격에 부합합니다."** 판정 완료.
4. 결과: 자식 객체는 VIP 전용 할인 메서드를 켜보지도 못한 채, 일반 예약 부모 처리 분기(`1번 분기`)로 그냥 이송 완료되어 버립니다. 아래의 `WHEN TYPE zcl_vip_booking` 은 영원히 스킵되어 낙동강 오리알이 됩니다.
따라서 다형성 타입 분기를 배치할 때는 반드시 **상속 트리의 가장 밑바닥 말단에 서식하는 초구체적 자식 클래스 타입(VIP)을 가장 첫 번째 WHEN 절에 선 배치**하고, 점진적으로 상위 부모로 거슬러 올라가게 계단을 배치해야 물리 매칭 사양이 정상 작동합니다.
