## 4. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. JOIN 쿼리에 @DATA( ) 를 써서 인라인 타깃으로 수령했더니 컴포넌트가 wa-scarr-carrid 처럼 중첩 구조체(Nested Structure)로 꼬여서 변수를 찾지 못해 에러가 납니다. 왜 그런가요?
**A**: **SELECT list에 콤마 필드를 구체적으로 지목하지 않고 `SELECT *` 로 여러 테이블을 조인해 던져주면, 컴파일러가 중복 컬럼명의 충돌을 원천 방지하기 위해 각 부모 테이블 이름을 서브 구조체명(Sub-structure)으로 삼는 Nested 레이아웃으로 결과를 조립하기 때문입니다.**
예를 들어 `scarr` 테이블과 `spfli` 테이블을 조인했는데, 두 테이블 모두 `carrid` 라는 동일한 컬럼 명칭을 지니고 있습니다.
만약 이 상태에서 `SELECT * INTO TABLE @DATA(lt_join)` 을 날리면, 새로 태어난 `lt_join` 테이블의 한 행 구조 안에는 `carrid` 라는 기둥이 2개 들어가야 하므로 이름 충돌 장애가 납니다. 아바 엔진은 이를 회피하기 위해 행 내부를 `scarr` 구조체방과 `spfli` 구조체방으로 중첩 분할하여, 최종적으로 `ls_join-scarr-carrid` 와 `ls_join-spfli-carrid` 형식으로 데이터를 담아줍니다. 
이로 인해 일반 1층짜리 납작한 구조체인 줄 알고 `ls_join-carrid` 를 호출하면 "컴포넌트가 존재하지 않는다" 는 컴파일 거부 에러가 납니다.
**해결책**: 이를 1층짜리 납작한 구조로 안전하게 받으려면, 귀찮더라도 `SELECT c~carrid, c~carrname, p~connid` 처럼 **조인 테이블의 특정 필드를 콤마 리스트로 1:1 명시하여 쿼리를 작성**해야 1층짜리 정갈한 테이블 그릇이 인라인으로 생성됩니다.

### Q2. SQL의 1-기반 SUBSTRING과 ABAP의 0-기반 substring은 왜 시작점 숫자가 다르게 설계되었나요? 실행 주체의 차이가 있나요?
**A**: **두 함수의 연산이 실행되는 물리적 엔진과 서버 위치가 180도 다르기 때문이며, SQL 함수는 DB 표준 규격(1-기반)을 따르고 ABAP 함수는 프로그래밍 언어 배열 규격(0-기반)을 따릅니다.**
- **SQL `SUBSTRING( ... )`**: 
  이 함수는 아바 프로그램 서버가 아니라 **'데이터베이스(DB) 엔진'** 내부에서 데이터 가공 처리를 집행합니다. 데이터베이스 표준 언어 규격(ANSI SQL)에서 모든 문자열 및 배열 컬럼의 맨 첫 글자를 지칭하는 시작 주소 인덱스는 물리적으로 **`1`** 로 약속 정의되어 있습니다. 따라서 첫 번째 글자부터 떼려면 시작값을 **`1`** 로 적어야 합니다.
- **ABAP `substring( ... )`**:
  이 함수는 데이터베이스와 상관없이 **'아바 애플리케이션 서버 RAM 메모리'** 안에서 아바 독자 가공을 집행합니다. C언어, 자바, 파이썬 등 대다수 현대 프로그래밍 언어에서 메모리 내의 배열 데이터에 접근할 때 첫 번째 항목을 지칭하는 오프셋(Offset) 지표는 항상 **`0`** 부터 출발합니다. 
따라서 동일한 첫 글자를 슬라이스하고 싶더라도 SQL은 시작 번호 **`1`**, 아바는 시작 번호 **`0`** 으로 인덱싱 격리가 일어나는 것이며, 이 아키텍처적 실행 주체 차이를 의식하지 않으면 실무 작성 시 기둥 문자열이 1칸 밀려 잘리는 다발성 슬라이스 버그가 발생합니다.

### Q3. SELECT FROM @itab 가동 시 Deep Type(String/Reference) 컬럼을 소스로 쓸 수 없는 물리적 사유는 무엇이고, SSTRING 우회 원리는 어떻게 되나요?
**A**: **메모리 SQL 엔진이 RAM 상에 올라온 내부 테이블 데이터를 읽을 때, 메모리 번지수가 행 외부에 찢어져 적재되는 가변 딥 타입(Deep Type) 필드는 정적 바인딩 버퍼링을 처리할 수 없기 때문이며, SSTRING은 이를 DB의 고정 가변 규격으로 강제 묶어 해결합니다.**
- **물리적 사유**:
  일반 `c` 타입이나 `i` 타입은 한 행이 차지하는 바이트(Byte) 크기가 `20바이트`, `4바이트` 처럼 메모리 상에 딱딱 고정(Flat)되어 정렬 기입되어 있습니다. 반면 일반 `string` 이나 `REF TO` 클래스 변수는 행 내부에는 실제 데이터가 들어있지 않고 데이터가 들어있는 **외부 메모리의 포인터 주소값(Deep Reference)** 만 기재되어 있습니다. 
  이 상태에서 `SELECT FROM @itab` 가 가동되면, 메모리 SQL 엔진은 1:1로 평평하게 버퍼링 메모리를 복사하여 SQL 스택에 적재하려 드는데, 포인터 주소만 덩그러니 들어있으니 실제 텍스트 내용을 읽어 들이지 못해 물리적 바인딩 대참사가 일어납니다. 따라서 엔진 자체적으로 딥 타입의 개입을 사전에 강제 문법 차단하는 것입니다.
- **`SSTRING` 우회 원리**:
  **ABAP Dictionary의 가변 문자 타입인 `SSTRING` 은 최대 길이(예: 255자) 제한 정보가 정의에 포함된 특수한 문자 유형입니다.** 
  변수를 `string` 대신 `TYPE sstring` 으로 매핑 정의해 두면, 컴파일러는 "아, 이 문자열은 메모리 포인터 주소로 찢지 않고 최대 길이 255바이트 범위 안에서 정적으로 공간을 할당해 처리해도 좋구나" 라고 인지하여, 행 내부 버퍼에 물리 텍스트를 평평하게(Flat) 직접 적재해 줍니다. 이 덕분에 메모리 SQL 엔진이 포인터 꼬임 없이 안전하게 데이터를 수집할 수 있게 우회되는 물리 원리입니다.

### Q4. 단일 엘리멘트 테이블(TABLE OF i)을 SELECT FROM @itab 할 때, 왜 table_line AS 별칭 을 지정해 주지 않으면 인라인 타깃이 컴파일 에러를 내며 거부하나요?
**A**: **인라인 타깃 선언(@DATA)이 결과 내부 테이블의 행 구조를 조립할 때, 필드 이름표가 없는 단순 컬럼은 구조체 컴포넌트(Field Name)로 등재할 수 없어 조립 뼈대(Schema)가 성립되지 않기 때문입니다.**
만약 `TABLE OF i` 타입의 `gt_ints` 메모리 테이블이 실재한다고 가정해 봅시다.
여기서 `SELECT * FROM @gt_ints INTO TABLE @DATA(lt_res)` 를 실행하려 하면 아래와 같은 모순에 부딪힙니다.
1. 아바 컴파일러는 `gt_ints` 의 행 알맹이가 구조체가 아니라 단일 정수 `i` 열이라는 것을 감지합니다. 이 경우 이 열의 식별 컬럼명을 아바 시스템 예약어인 **`table_line`** 으로 강제 번역 지정합니다.
2. 이제 결과를 받아 수납할 `lt_res` 테이블의 한 행 구조(Structure)를 빚어야 하는데, 컬럼 이름이 `table_line` 그대로 기입되려 하면, 아바의 구조체 컴포넌트 작명 규칙 상 시스템 예약어와 겹쳐서 변수방 이름을 빚을 수 없는 컴파일 충돌이 일어납니다.
3. 따라서 "컬럼의 정식 칭호(FieldName)를 지어줄 수 없다" 며 에러를 내며 빌드를 차단하는 것입니다.
**해결책**: 이 단일 열 쿼리 우측에 **`SELECT table_line AS num`** 처럼 명시적으로 칭호 별칭(AS)을 꽂아 넘겨주면, 인라인 타깃은 안심하고 `num` 이라는 정식 컴포넌트 기둥을 가진 1열짜리 정갈한 결과 구조체 테이블을 빚어낼 수 있게 되어 컴파일 에러가 평화롭게 복구됩니다.

### Q5. Outer Join 시 취소 조건(status <> 'C')을 ON 절에 두는 것과 WHERE 절에 두는 것이 결과 테이블의 행 유실을 다르게 내는 관계형 대수의 매커니즘은 무엇인가요?
**A**: **ON 절은 두 테이블을 접착 결합하는 단계에서만 필터링하여 미매칭 행의 생존을 보존하는 반면, WHERE 절은 결합이 끝난 뒤 완성된 통합 테이블 전체에 대해 검문하여 결합 실패로 인해 Null로 메워진 우측 행들까지 통째로 쓸어내어 소멸시키기 때문입니다.**
예매 현황 리포트에서 `ZCONCERT` (공연 - C001, C002, C003) 와 `ZBOOKING` (예매 내역)을 `LEFT OUTER JOIN` 합니다. 여기서 신규 공연인 C003은 예매가 단 1건도 없는 순수한 상태입니다.
- **`ON` 절 배치 매커니즘 (`ON b~concert_id = c~concert_id AND b~status <> 'C'`)**:
  1. 결합 단계: `ZBOOKING` 테이블에서 status 가 `'C'` 인 취소 행을 제외한 알맹이 예매 데이터만 골라냅니다.
  2. 아우터 결합: `ZCONCERT` 기준으로 결합을 수행합니다. C003 공연은 예매가 0건이라 매칭되는 로우가 아예 없습니다.
  3. 결과: `LEFT OUTER` 규칙에 따라, C003 공연 행은 소멸하지 않고 살려두되, 우측 예매 열 정보만 `NULL` (또는 빈 값)로 메워진 채 **최종 합본 테이블에 C003 행이 무사히 안착 생존**합니다.
- **`WHERE` 절 배치 매커니즘 (`WHERE b~status <> 'C'`)**:
  1. 1차 아우터 결합: `ZCONCERT` 와 `ZBOOKING` 을 그냥 공연 코드로만 결합합니다. 이때 C003 공연도 매칭 예매가 없으므로 우측 열이 `NULL` 로 채워진 채 일단 결합 테이블에 얹힙니다.
  2. 2차 WHERE 검문: 결합이 다 끝난 완성 테이블 전체를 대상으로 `WHERE b~status <> 'C'` 조건 검문을 돌립니다.
  3. 참사: C003 공연 행은 우측 예매 정보가 `NULL` 상태입니다. 관계형 대수 규칙 상 **`NULL <> 'C'` 라는 연산은 참(True)이 아니라 알 수 없음(Unknown/False)으로 판정**됩니다. 이로 인해 WHERE 검문소는 C003 행을 "조건 불합격 탈락자" 로 판단하여 **테이블에서 통째로 소멸 삭제**시켜 버립니다.
이로 인해 예매 내역이 없는 신규 공연 C003은 보고서 목록에서 아예 실종되어 사라지는 심각한 보고서 데이터 정합성 파괴 장애를 초래하게 되는 물리 원리입니다. 따라서 우측 종속 테이블의 필터 조건은 무조건 **`ON` 절에 결합 고정**하는 것이 철칙입니다.
