# CH25_REWRITE - Lock Object와 동시성 제어 v1

> 목적: `content/abap/CH25`의 5개 레슨을 "IT 비전공자 입문자가 왜 필요한지부터 실습 확인까지 직관적으로 이해하고 따라갈 수 있는 완성 강의자료" 수준으로 재작성한 기준 원고다. 이 문서는 아직 `content/abap` 원본을 덮어쓴 결과가 아니라, 재작업 판정 이후 실제 반영 전 검수 가능한 v1 기준안이다.

## CH25 전체 설계

CH25의 한 문장 목표는 "다중 사용자 환경에서 먼저 저장한 변경이 나중 변경에 덮여 조용히 유실되는 잃어버린 갱신(Lost Update) 문제를 파악하고, SE11에서 Lock Object(`EZ_*`)와 Lock Argument를 설계해 ENQUEUE/DEQUEUE 함수를 확보하며, 비관적 락(Pessimistic: E/S/X 모드)과 낙관적 락(Optimistic: ETag/Timestamp 대조) 전략을 구분하고, `_SCOPE` 파라미터별 잠금 수명 매칭 및 `DEQUEUE` 시의 scope 불일치 락 방치 부작용을 통제하여 안전 변경 표준 패턴(ENQUEUE -> READ -> DML -> COMMIT -> DEQUEUE)을 정립한다"이다.

IT 비전공자 입문자는 `ENQUEUE` 시 `_scope = 2` 로 잠갔으나 `DEQUEUE` 시 아무 매칭 없이 `_scope = 1` 로 던져 락이 풀리지 않고 영구 좀비 상태로 방치되어 다른 동시 접속자를 무한 대기시키는 장애를 내고, SAP Lock 이 데이터베이스 로우를 알아서 물리 강제 통제하는 줄 착각하여 선행 잠금 체크 없이 쌩 SQL UPDATE 를 찔러 갱신 덮어쓰기 유실을 낸다.
또한, 낙관적 락 구현 시 변경시각 컬럼을 단순 `changed_at = sy-uzeit` (초단위)로만 기입했다가 동일 1초 이내 충돌을 잡지 못해 Lost Update 를 내고, 락을 조회 전이 아니라 변경 직전에 걸어 충돌을 허용한다.
따라서 본 챕터는 다음과 같은 단계적 해소 장치로 설계를 배치한다.

1. **Lost Update 규명**: 동시에 같은 예약 행을 수정할 때 발생하는 잃어버린 갱신 현상의 실체 및 `COMMIT/ROLLBACK` 의 한계 명세.
2. **Lock Object 설계**: SE11 에서 `E` + 테이블명 관례로 Lock Object 를 빚고, 기본키를 Lock Argument 로 지정하여 ENQUEUE/DEQUEUE 함수를 자동 획득하는 흐름 수립.
3. **E/S/X 잠금 모드 격리**: 변경용 배타 락 **`E` (Exclusive)**, 조회용 공유 락 **`S` (Shared)**, 동일 사용자 중첩 불가 강한 배타 락 **`X` (Exclusive 비누적)** 의 호환성 분석.
4. **DEQUEUE scope 매칭**: `DEQUEUE` 해제 시, **ENQUEUE 시의 _SCOPE 값보다 작게 주면 해제가 묵살(No-op)** 되어 좀비 락이 방치되는 scope 수명 매칭 철칙 수립.
5. **자발적 규약 명세**: SAP Lock 은 SM12 central lock table 의 논리적 징표일 뿐이므로, **모든 변경 프로그램이 DML 전 ENQUEUE 선행 체크를 자발적으로 물려야만 동시성이 사수됨**을 규명.
6. **sy-msgv1 미보장 가드**: `foreign_lock = 1` 충돌 시, 락 소유자명이 sy-msgv1 시스템 변수에 항상 보장되어 들어오는 것은 아니므로 SM12 조회 함수 활용 가이드 수립.
7. **비관 vs 낙관 격리**: 미리 잠그는 **비관적 락(Pessimistic)** 과 저장 직전 변경시각을 대조하는 **낙관적 락(Optimistic)** 전략의 비교.
8. **낙관적 락의 초단위 함정**: 단순 초단위(sy-uzeit) 비교는 동일 1초 충돌에 유실되므로 **밀리세컨드 Timestamp, Version Counter, ETag** 가 안전함을 교육.
9. **안전 변경 통합 패턴**: **ENQUEUE(잠금) -> READ(조회) -> UPDATE(변경 DML) -> COMMIT(확정 자동해제) -> DEQUEUE(명시완료)** 의 표준 순차 조립 및 캡스톤 완성.

## CH25-L01 - Lock Object 설계 기준

### 왜 필요한가

우리가 앞선 데이터 변경 챕터에서 배웠던 `UPDATE` 나 `MODIFY` DML 문장들은 데이터를 DB 에 쓰고 트랜잭션을 처리하는 데 훌륭하게 기여했다.
하지만 우리가 만드는 프로그램은 나 혼자 쓰는 것이 아니다. 수십 명의 예매 담당 직원이 동시에 시스템에 달라붙어 일한다.
- A 직원과 B 직원이 같은 예매번호 `1001`(기존 2석 예약) 내역을 각자의 화면에 조회했다.
- A 직원이 예약 인원을 3석으로 고쳐 [저장]을 눌러 성공했다.
- 1초 뒤, 아직 이전 화면(2석)을 켜두고 멍하니 있던 B 직원이 예약 인원을 1석으로 입력해 [저장]을 눌렀다.
그 순간, 먼저 저장한 A 직원의 3석 데이터는 아무 경고 에러도 없이 흔적도 없이 덮어씌워져 조용히 사라진다. 이것이 **[[Lost Update]] (잃어버린 갱신)** 대형 금융 사고다.
앞서 배운 `COMMIT WORK` 은 내 한 세션 안에서 전표의 원자성(All or Nothing)만 지켜줄 뿐, **"두 사람이 동시에 동일한 행을 조회해서 수정 중이다"** 라는 다중 사용자 변경 충돌은 전혀 물리적으로 막지 못한다.

**여러 사용자가 같은 예매를 동시에 편집하고 수정하여 데이터를 손상시키는 충돌을 막기 위해, "이 데이터 레코드는 현재 내가 편집권을 쥐고 있으니 다른 사람은 손대지 마라" 고 징표를 박아두는 물리 장벽 기술**이 필요하다. 그것이 **Lock Object (잠금 오브젝트)** 의 설계다.

### 무엇인가

#### 1. Lock Object (잠금 오브젝트)
- DB 테이블의 특정 행 데이터를 동시 변경 충돌로부터 보호하기 위해, SE11(Dictionary) 에서 생성하는 특수 논리 자원이다.
- **EZ_BOOKING**: Lock Object 의 이름은 관례상 **`E` (Enqueue) + 대상 테이블명** 으로 생성하여 관리하는 것이 원칙이다.

#### 2. Lock Argument (잠금 인수)
- *잠금을 거는 범위(Scope)를 정하는 결정적 지표다.*
- Lock Object 에 "이 테이블은 무엇을 기준으로 락을 잠글 것인가" 를 지목하는 필드 목록이다.
- **범위 규칙**: 보통 테이블의 **기본키(booking_id)** 를 Lock Argument 로 지정한다. 키를 빼고 넓게 지정하면, 1번 예약 1건을 고치는 동안 다른 사용자의 2번, 3번 예약 변경까지 연쇄적으로 차단되어 시스템이 통째로 굳어 멈추는 동시성 마비를 겪는다.

#### 3. 3대 잠금 모드 (Lock Mode) 명세

| 모드 | 뜻 | 다른 사용자 S(공유) 허용 여부 | 다른 사용자 E(배타) 허용 여부 | 실무 용도 |
| --- | --- | --- | --- | --- |
| **`E`** | **Exclusive** (배타적 쓰기 락) | **X** (차단) | **X** (차단) | 변경 화면 기동 시 99% 사용하는 표준 모드 |
| **`S`** | **Shared** (공유 읽기 락) | **O** (동시 S 수용) | **X** (차단) | 대량 분석 조회 시 변경 시도를 불허할 때 사용 |
| **`X`** | **eXclusive** (비누적 배타 락) | **X** (차단) | **X** (차단) | 동일 프로그램 내에서도 중첩 락을 허용하지 않을 때 |

### 어떻게 확인하는가

SE11 상에서 Lock Object 를 조회해 테이블과 키 아규먼트가 이쁘게 매핑되어 활성화된 상태를 검수한다.

```text
" 1. SE11 에디터에서 Lock Object 'EZ_BOOKING' 을 지명한다.
" 2. [Tables] 탭의 Primary Table 에 'ZBOOKING' 테이블이 잘 안착되어 있는지 확인한다.
" 3. [Lock parameter] 탭에 key field 인 'BOOKING_ID' 가 Lock Argument 로 
"    정확히 지정되어 활성화(Active) 상태를 지니고 있는지 확인한다.
```

#### 체험/시뮬레이터 설계 (잠금 모드 호환 매트릭스)
- **프로세스 플로우**:
  1. 학습자가 중앙의 [ZBOOKING 1001번 행]을 본다.
  2. A 직원이 [E (Exclusive)] 잠금 카드를 1001번에 꽂는다. 1001번 레코드 주변에 두터운 철벽 방어막이 렌더링된다.
  3. B 직원이 [S (Shared)] 카드로 접근을 시도한다. 철벽에 부딪혀 "Access Denied" 가 터지며 튕겨 나간다.
  4. A 의 카드를 [S (Shared)] 로 낮춘다. B 가 [S (Shared)] 카드를 들고 가자, 방어막이 투명하게 열리며 두 명의 S 공유 읽기가 나란히 도킹 성공하는 비주얼 피드백을 감상한다.
- **상태 및 데이터**:
  - `Lock Argument 필드를 누락해 텅 비운 채 활성화를 지르려 든 상태` -> SE11 컴파일 가이드라인: `Lock Object EZ_BOOKING must contain at least one Lock Parameter` 적색 가드 하이라이트.
- **피드백**: 변경 작업용 락은 배타 쓰기 락인 `E` 모드를 기본 축으로 엮어야 안전함을 학습한다.

### 실수/주의

- **SAP Lock 이 데이터베이스의 SQL UPDATE 변경 문장을 자동으로 감지해 물리적으로 가로막는 마법을 부린다고 오해**:
  - **SAP Lock 은 물리 DB 레벨의 하드웨어 락이 아니라, SM12 중앙 락 테이블에 징표를 적어두고 소통하는 '논리적 약속(자발적 규약)' 에 불과하다.**
  - 잠금을 걸어둔 행이라 할지라도, 어떤 나쁜 개발자가 짠 변경 프로그램이 "잠금이 걸려있는지 조회하는 선행 ENQUEUE 체크" 를 생략하고 그냥 쌩 `UPDATE zbooking SET ...` SQL 쿼리를 날려버리면 DB 는 락 유무와 전혀 무관하게 덮어써져 Lost Update 가 그대로 터진다.
  - **즉, 모든 데이터를 바꾸는 프로그램은 DML 실행 전 무조건 자발적으로 ENQUEUE 체크를 1순위 수행하도록 코딩 규율을 지켜야만 락이 효력을 발휘한다.**

### 정리

- **`Lost Update`** 는 안 잠그고 각자 고치다가 먼저 고친 사람의 내역이 덮여서 소멸하는 현상이다.
- Lock Object 는 관례상 **`EZ_테이블명`** 으로 SE11 에서 빚어낸다.
- Lock Argument 는 락의 범위를 뜻하며 보통 **기본키** 단위로 좁게 조준 지정한다.
- 변경 처리는 배타 락인 **`E`** 모드를 99% 기본 기어로 사용한다.

## CH25-L02 - ENQUEUE / DEQUEUE Function Module

### 왜 필요한가

Lock Object 를 SE11 에서 이쁘게 다 빚어냈다.
그런데 이 잠금 오브젝트 파일은 일반 테이블이나 클래스와 달리, 내부에 아바 소스 코드를 타이핑해 넣는 에디터 공간이 전혀 존재하지 않는다. 
어떻게 이 논리 자원을 가져다가 내 아바 소스 코드 안에서 잠금을 획득하고 다시 해제하는 물리 연산으로 기동할 수 있을지 막막하다.

**Lock Object 를 활성화하는 찰나, 백엔드 시스템이 백그라운드에서 잠그고(ENQUEUE) 푸는(DEQUEUE) 특수 펑션 모듈 함수 2개를 공짜로 자동 생성해 배포해 주는 아바 엔진 기술**이 필요하다. 그것이 **ENQUEUE / DEQUEUE Function Module** 의 호출이다.

### 무엇인가

#### 1. ENQUEUE_EZ_BOOKING (잠금 획득 함수)
- Lock Object 를 활성화하면 접두어 `ENQUEUE_` 가 붙어서 시스템에 자동 생성 등록되는 특수 함수다. 이 함수를 호출할 때 예매 Key(`booking_id`)를 패싱하면 SM12 락 테이블에 징표가 등록되어 잠금이 시작된다.
- **EXCEPTIONS foreign_lock = 1**: *잠금 성공 여부를 가리는 절대 검문 변수다.*
- **의무 분기**: 이미 다른 동시 사용자가 락을 쥐고 있어서 락 획득에 실패했을 때, 이 예외 코드로 떨어지게 된다. **이 충돌은 에러 덤프가 아니라 정상적인 동시 충돌 현상이므로, 개발자는 무조건 `sy-subrc = 1` 분기를 쳐서 "다른 사용자가 편집 중입니다" 고 사용자에게 정중히 거절하고 변경 로직을 긴급 정지해야 한다.**

#### 2. DEQUEUE_EZ_BOOKING (잠금 해제 함수)
- 접두어 `DEQUEUE_` 가 붙어 탄생하는 락 수거 함수다. 호출 시 동일한 Key 를 던지면 SM12 에 잠들어 있던 내 락 징표가 영구 소멸하며 다른 사용자에게 편집권이 해방된다.

#### 3. _SCOPE (잠금 수명 제어)
- 잠금이 언제까지 살아남아 기동될지 수명을 제어하는 기어다.
- **`_scope = '1'`**: 현재 구동 중인 아바 프로그램 세션 내에서만 락이 돕니다. 프로그램이 끝나거나 수동 `DEQUEUE` 시 풀린다.
- **`_scope = '2'`** (기본값): **SAP LUW 와 연동되어, 나중에 메인 프로그램이 `COMMIT WORK` 이나 `ROLLBACK WORK` 을 찌를 때, 그 시점에 비동기 업데이트 포스팅이 완수되는 찰나에 락이 자동으로 수거 해제되는 가장 똑똑하고 안전한 수명 방식이다.**

### 어떻게 확인하는가

ENQUEUE 함수를 지목 호출하고 foreign_lock 예외 코드를 수집 판정하는 소스를 검증한다.

```abap
REPORT zch25_l02_lock.

DATA: lv_booking_id TYPE zbooking-booking_id VALUE 'B00003'.

START-OF-SELECTION.
  " 1. [ENQUEUE 잠금 획득 격발]
  CALL FUNCTION 'ENQUEUE_EZ_BOOKING'
    EXPORTING
      mode_zbooking  = 'E'            " 배타 락 모드 지정!
      booking_id     = lv_booking_id  " 락을 잠글 기본키 타깃 지정!
      _scope         = '2'            " LUW 커밋 자동 해제 연동!
    EXCEPTIONS
      foreign_lock   = 1              " ★ 남이 잠갔을 때 분기 예외!
      system_failure = 2
      OTHERS         = 3.

  " 2. [★ 예외 분기 판정 철칙]
  IF sy-subrc = 1.
    " 덤프를 내지 않고, 편집 중임을 정중히 화면에 뿌리고 로직 긴급 셧다운!
    MESSAGE '현재 다른 사용자가 해당 예약 건을 변경 편집하고 있습니다.' TYPE 'I'.
    EXIT.
  ELSEIF sy-subrc <> 0.
    MESSAGE '잠금 모듈 가동 중 시스템 오류가 감지되었습니다.' TYPE 'A'.
  ENDIF.

  WRITE: / lv_booking_id, '예약 행의 독점 편집권을 안전하게 획득했습니다.'.
```

#### 체험/시뮬레이터 설계 (2세션 잠금 데모)
- **프로세스 플로우**:
  1. 학습자가 좌측 [User A 화면]과 우측 [User B 화면]을 동시에 띄운다.
  2. User A 가 [ENQUEUE B001] 단추를 누른다. 가운데 [SM12 Lock List 보드]에 User A 가 B001 을 배타 락으로 소유했음이 기재된다.
  3. 즉시 User B 가 [ENQUEUE B001] 을 찌른다. sy-subrc = 1 불이 켜지며 "외국인 락(foreign_lock) 감지! 진입 차단!" 정중히 거절 경고가 발생한다.
  4. User A 가 [DEQUEUE B001] 또는 [COMMIT] 을 치자, SM12 목록에서 락 징표가 날아가고 User B 가 비로소 진입 통과되는 모습을 감상한다.
- **상태 및 데이터**:
  - `foreign_lock = 1 예외 처리를 무시하고 바로 UPDATE DB 쿼리를 타도록 코딩한 상태` -> 런타임 결과: `Lock check bypassed. Lost Update occurred and Data corrupted` 경보 레이저 경보.
- **피드백**: ENQUEUE 의 핵심 가치는 `foreign_lock` 이 터졌을 때 즉각 연산을 포기하고 멈춰 서는 예외 분기에 있음을 배운다.

### 실수/주의

- **수동 DEQUEUE 를 지를 때 _SCOPE 값을 ENQUEUE 시점보다 작게 전달**:
  - **`DEQUEUE` 의 `_SCOPE` 값은 무조건 `ENQUEUE` 시 잡았던 값과 같거나 더 크게 지정해야만 해제가 통과된다.**
  - 예컨대 ENQUEUE 시 `_scope = '2'` 로 안전하게 묶어놓고, DEQUEUE 시 아무 생각 없이 `_scope = '1'` 로 호출해 버리면 아바 엔진은 해제 신호를 통째 묵살하고 락을 영구 방치한다. 
  - 결국 다른 동시 접속자들은 계속 '다른 사람이 편집 중' 이라는 창을 보며 일도 못 하고 대기하는 끔찍한 좀비 락 장애를 낸다.

### 정리

- Lock Object 를 활성화하면 **`ENQUEUE_*`** 와 **`DEQUEUE_*`** 함수가 자동 생성된다.
- **`foreign_lock`** 은 이미 남이 락을 선점했음을 뜻하므로 즉각 연산을 정지 거절해야 한다.
- **`_scope`** 파라미터는 락의 수명을 쥐며, 통상 **`'2'`** 를 주어 커밋 연동 자동해제를 유도한다.
- 수동 해제 시의 scope 레벨은 무조건 ENQUEUE 시와 **동일 매칭**해야 해제가 먹힌다.

## CH25-L03 - Lock 해제와 예외 처리

### 왜 필요한가

ENQUEUE/DEQUEUE 함수를 엮었다.
그런데 이번에는 락이 풀려나가는 정확한 소멸 지점에 관한 아바 런타임 수명 이해가 꼬인다.
" 한 직원이 예매 변경 화면을 켜고 [예약 좌석 수 수정] 버튼을 클릭해 정상적으로 `ENQUEUE` 를 걸고 독점 편집권을 땄다.
그런데 이 직원이 락을 쥔 채로 [저장] 단추를 누르지 않고, 화면을 켜둔 채 그대로 퇴근해 버렸다."
수십 명의 다른 예매 직원들은 이 퇴근한 직원의 예약 번호를 영구히 수정하지 못하고 대기하며 발을 동동 구르는 마비 현상을 겪는다.

**락이 자동으로 소멸해 해방되는 3대 소멸 트리거(DEQUEUE 명시, LUW 커밋/롤백 완수, 세션 종료)의 작동 원리를 파악하고, 불필요한 장시간 독점을 피하도록 락 점유 시간을 최소화하는 관리 기술**이 필요하다. 그것이 **Lock 해제 타이밍과 예외 수습**의 설계다.

### 무엇인가

#### 1. SAP Lock 이 자동으로 풀려 소멸하는 3대 타이밍 명세
- **`DEQUEUE_*` 명시 호출**: 프로그램 소스 코드 상에서 내가 필요가 끝나 명시적으로 해제 함수를 지르는 찰나 풀린다.
- **`COMMIT WORK` / `ROLLBACK WORK` 완수**: `_scope = '2'` 락의 위대한 가치다. 트랜잭션이 디스크에 영구 확정되거나 취소 완료되는 즉시, 백엔드 엔진이 SM13 큐 포스팅을 매듭짓고 락을 자동으로 청소 해제해 준다.
- **아바 세션(Session)의 영구 종료**: 웹 브라우저 창을 완전히 닫아버리거나, SAP GUI 프로세스를 강제 킬(Kill)하여 로그인 세션이 끊어지는 즉시, 엔진은 메모리에 방치된 그 로그인 ID 소유의 모든 좀비 락을 자동으로 싹 털어 청소 소멸시킨다. (따라서 퇴근 후에도 영구 잠기는 재앙은 다행히 세션 컷에 의해 방어된다.)

#### 2. DEQUEUE_ALL (일괄 락 청소기)
- 현재 실행 프로그램 세션이 쥐고 있던 모든 락 아규먼트 징표를 single 쿼리로 한 번에 영구 털어내는 일괄 청소 함수다. 주로 비정상 종료 시 복구 클린업용으로 사용한다.

#### ⚠️ [ sy-msgv1 시스템 변수의 락 소유자 ID 미보장 경계 명세]
- *초보 개발자들이 디버깅 시 락 소유자를 화면에 뿌리려다 엉뚱한 값을 출력하는 단골 에러 지점이다.*
- ENQUEUE 실패(`sy-subrc = 1`) 시, "현재 락을 쥔 주동자가 누구다" 라는 사용자 ID 가 `sy-msgv1` 시스템 변수에 무조건 들어올 것이라 가정하고 이를 화면 메시지에 엮어 짜는 것은 불완전하다.
- **이유**: 아바 시스템 커널 명세 상, 락 격발 상황에 따라 `sy-msgv1` 에 소유자 ID 정보를 채워주는 행위는 영구 의무 사양이 아니다. 값이 빈 값(`space`)으로 멍청하게 돌아오는 경우가 빈번하므로, **구체적 락 소유자를 반드시 추적해야 할 때는 `ENQUEUE_READ` 표준 조회 함수를 별도로 호출하거나 SM12 모니터 상에서 직접 대조해야 안전**하다.

### 어떻게 확인하는가

COMMIT WORK 이후 scope=2 잠금의 자동 소멸을 증명하고, 실패 시의 수거를 확인하는 코드를 검증한다.

```abap
REPORT zch24_l03_release.

DATA: lv_booking_id TYPE zbooking-booking_id VALUE 'B00004'.

START-OF-SELECTION.
  " 1. [잠금 가동]
  CALL FUNCTION 'ENQUEUE_EZ_BOOKING'
    EXPORTING
      booking_id = lv_booking_id
      _scope     = '2' " scope=2 로 커밋 자동 수거 지정!
    EXCEPTIONS
      foreign_lock = 1 OTHERS = 2.
      
  IF sy-subrc = 0.
    " 2. [변경 DML 예약]
    UPDATE zbooking SET status = 'C' WHERE booking_id = @lv_booking_id.
    
    " 3. [★ 확정 및 자동 해제 격발]
    " 이 COMMIT WORK 이 돌 때, DB 확정이 끝나고 zbooking-B00004 에 걸려있던 
    " EZ_BOOKING 락도 SM12 테이블에서 자동으로 싹 소멸 해제됨을 확인!
    COMMIT WORK.
    
    WRITE: / '저장 확정 및 잠금이 자동으로 해제 완료되었습니다.'.
  ENDIF.
```

#### 체험/시뮬레이터 설계 (잠금 해제 판별 퀴즈)
- **프로세스 플로우**:
  1. 학습자가 [EZ_BOOKING B001 번 락이 걸린 칠판]을 본다.
  2. 칠판 하단에 [SELECT SINGLE * FROM zbooking WHERE booking_id = 'B001'] 버튼이 뜬다. 이를 작동시키자 "조회 성공! 락은 여전히 유지!" 피드백이 들어온다.
  3. [COMMIT WORK] 단추를 누른다. 락 분필 징표가 파란 연기가 되어 스르륵 지워지는 "자동 소멸 완료" 모션을 감상한다.
- **상태 및 데이터**:
  - `다른 예매번호인 B002 번을 수정 변경 처리한 상태` -> 런타임 결과: `Lock on B001 is unaffected. Still Locked` 경보 발생.
- **피드백**: 단순 SELECT 읽기나 엉뚱한 다른 키 DML 은 현재 잡힌 락의 생명주기에 어떠한 해제 영향도 주지 못함을 판별한다.

### 실수/주의

- **락 징표를 변경 작업 시간 내내 몇 십 분씩 쥐고 있도록 방치하는 설계**:
  - 예매 상세 화면을 더블클릭하자마자 `ENQUEUE` 를 꽉 쥐게 짜두면, 해당 직원이 예약을 안 고치고 화면만 띄워둔 채 점심 식사를 하러 간 1시간 내내 다른 모든 직원의 변경이 완전히 굳어 막힌다.
  - **락은 무조건 데이터를 입력 고치기 직전에 짧게 잡고, 변경 DML 후 즉각 커밋하여 락 해제를 최단 시간 내에 완수하는 것이 동시성 성능의 철칙이다.**

### 정리

- 락은 **`DEQUEUE`**, **`COMMIT/ROLLBACK`**, **`세션 종료`** 3대 트리거 시 해제된다.
- **`DEQUEUE_ALL`** 은 내 현재 프로그램의 모든 잠금 징표를 일괄 반환 소멸시킨다.
- 락 소유자 추적 시 시스템 변수 `sy-msgv1` 은 미보장 락이므로, **`SM12`** 나 **`ENQUEUE_READ`** 를 쓴다.

## CH25-L04 - 다중 사용자 변경 충돌 시나리오

### 왜 필요한가

자동 해제와 수명 격리까지 마스터했다.
그런데 이번에는 업무 환경에서 비관적 잠금을 쓸지, 낙관적 잠금을 쓸지 동시성 전략 기획이 꼬인다.
"전 세계 콘서트 예매 목록 조회 화면이다. 매초 수천 명의 팬들이 달라붙어 공연 좌석 리스트를 단순 조회한다. 
비관적 락(Pessimistic)이 안전하다고 배워서, 사용자가 공연 조회 버튼만 눌러 화면에 띄우는 순간마다 `ENQUEUE` 배타 락을 쾅쾅 잠가버렸다."
그 순간, 1번 사용자가 리스트를 구경하는 동안 다른 999명의 대기자들은 조회조차 해보지 못하고 화면 버퍼링 모래시계를 보며 먹통 폭사를 겪는다. 
반대로, 락을 아예 아무것도 안 걸고 방치했더니 이번에는 Lost Update 가 사방에서 터져 고객 예약이 증발한다.

**충돌이 매초 격발되는 위험한 변경 전용 화면에는 미리 강하게 걸어 잠그는 '비관적 락' 을 씌우고, 단순 조회가 빈발하고 충돌이 드문 대량 조회용 화면에는 잠금 없이 가볍게 띄우되 저장 직전에 변경 정황을 최종 대조하는 '낙관적 락' 전략을 격리 설계하는 기술**이 필요하다. 그것이 **낙관적/비관적 락**의 장착이다.

### 무엇인가

#### 1. 비관적 잠금 (Pessimistic Lock)
- **전략**: "충돌이 일어날 게 분명하니, 조회를 시작하고 편집 화면을 여는 그 1단계 찰나부터 **`ENQUEUE_EZ_BOOKING` 으로 배타 락을 쾅 걸어 잠가** 다른 사람의 진입을 원천 봉쇄하겠다" 는 철저한 독점 주의다. (충돌이 빈번하고 수정 화면 비중이 높을 때 기본 장착한다.)

#### 2. 낙관적 잠금 (Optimistic Lock)
- **전략**: "다들 조회 위주로 구경하고 충돌은 거의 안 나니, 조회를 시작할 때는 락을 아예 안 잠가서 동시 조회 속도를 극대화하겠다. 다만, **사용자가 고치고 나서 최종 [저장] 단추를 누르는 그 0.1초의 짧은 찰나에만 '내가 처음 조회해 갔던 시점의 데이터 값' 과 '현재 DB 에 박혀있는 원본 값' 이 동일한지 변경 대조 검사**하여, 그사이 누군가 고쳐 어긋났다면 수정을 거부하겠다" 는 가볍고 스마트한 감시 주의다.

#### ⚠️ [ 낙관적 락 구현 시 changed_at 초단위(sy-uzeit) 비교의 데이터 유실 함정 명세]
- *낙관적 락 충돌 판정을 설계할 때 아바 컴파일러가 잡아주지 않는 최악의 시간 유실 버그다.*
- **오동작**: 낙관적 변경 추적용 데이터로 `changed_at = sy-uzeit` (초단위 시각)만 활용하게 코딩을 짜두는 경우.
- **이유**: **동일한 1초(예: 14시 10분 05초)라는 극도로 짧은 동시 찰나의 순간에 User A 와 User B 가 연속 저장 변경을 때리면, 둘 다 changed_at 이 '141005' 로 완벽히 동일해지므로, 저장 직전 찰나 대조 검문소가 충돌을 인지하지 못하고 그대로 문을 열어주어 Lost Update 갱신 파괴가 뚫려버리기 때문이다.**
- **대안 (ETag/Version)**: 낙관적 락 충돌 대조용 징표는 초단위 시각이 아니라, 변경할 때마다 무조건 1씩 더해 올라가는 정수 카운터 **`Version Counter` (1, 2, 3...)** 나, 밀리세컨드 단위 **`Timestamp`**, 혹은 고유 해시 값인 **`ETag`** 를 락 마스터로 채택해야 안전하다.

### 어떻게 확인하는가

낙관적 락을 위해 Version Counter 징표를 저장 직전 SELECT 로 재대조 판정하는 소스를 검증한다.

```abap
REPORT zch24_l04_optimistic.

DATA: lv_id         TYPE zbooking-booking_id VALUE 'B00005',
      " 1단계: 사용자가 처음 화면에 조회해 갈 때 들고 간 버전 값!
      lv_read_version TYPE i VALUE 2, 
      ls_db_booking   TYPE zbooking.

START-OF-SELECTION.
  " [ 2단계: 저장 직전 DB 원본 버전 벌크 찰나 대조 검사 ]
  SELECT SINGLE * FROM zbooking WHERE booking_id = @lv_id INTO @ls_db_booking.
  
  IF ls_db_booking-version <> lv_read_version.
    " 3단계: [낙관 락 검문소 불일치 격퇴]
    " 내가 가져간 버전은 2인데, 누군가 고쳐서 DB 버전은 3이 되었음!
    MESSAGE '내가 편집 중인 동안 다른 사용자가 이미 데이터를 고쳐 버전이 변경되었습니다. 다시 재조회하십시오.' TYPE 'E'.
  ELSE.
    " 버전이 완벽히 일치한다면 안심하고 변경 집행 후 버전을 1개 올려 3으로 적재!
    ls_db_booking-version += 1.
    ls_db_booking-seats = 5.
    
    UPDATE zbooking FROM @ls_db_booking.
    COMMIT WORK.
    WRITE: / '낙관적 락 검증 통과: 버전', ls_db_booking-version, '으로 안전 갱신 완료.'.
  ENDIF.
```

#### 체험/시뮬레이터 설계 (Lost Update 시뮬)
- **프로세스 플로우**:
  1. 학습자가 [User A] 와 [User B] 의 조회 시점 버전을 [Version = 5]로 나란히 든 것을 본다.
  2. [비보호 모드]로 실행한다. User A 가 업데이트하며 버전을 6으로 올리나, User B 가 곧이어 옛 버전 상태의 데이터로 DB 를 UPDATE 쳐버려 User A 의 내역이 통째 삭제 증발하는 Lost Update 대참사를 본다.
  3. [낙관적 락 모드 ON] 을 켠다. User B 가 저장을 누르는 찰나 "DB 버전 = 6 <> 내 버전 = 5 불일치! 거절!" 방어막이 가동되는 피드백을 감상한다.
- **상태 및 데이터**:
  - `동작 카운터나 버전 필드가 없이 changed_at = sy-uzeit 초단위로 락을 돌린 상태` -> 런타임 결과: `Double updates within same second bypassed. Conflict occurred` 사이렌 경보.
- **피드백**: 낙관적 락은 충돌이 드문 대량 조회 업무의 성능을 극대화하는 위대한 도구이며 ETag/버전 징표가 필수임을 깨닫는다.

### 실수/주의

- **동시성 충돌 빈도가 엄청나게 높은 주문 예약 생성 화면인데 귀찮다고 낙관적 락만 믿고 방치**:
  - 낙관적 락은 충돌이 터졌을 때 "그동안 사용자가 입력창에 공들여 타이핑한 20개 필드의 고생한 내용" 을 전면 취소 거부하고 "다시 재조회해서 처음부터 타이핑해라" 고 튕겨내 버린다.
  - **따라서 충돌 확률이 50%가 넘는 좁고 치열한 편집 화면에는 사용자의 입력 노고를 지켜주기 위해 애초에 진입부터 입구를 비관적 배타 락(`ENQUEUE`)으로 걸어 잠가 다른 사람을 차단하는 것이 정석이다.**

### 정리

- **`비관적 락`** 은 진입 즉시 잠그고, **`낙관적 락`** 은 락 없이 최종 저장 시점에만 대조한다.
- 낙관적 락은 초단위 시각 유실 함정을 피하기 위해 **Version Counter** 나 **ETag** 를 징표로 삼는다.
- 조회가 흔하면 **낙관적**, 편집 충돌이 격렬하면 **비관적** 전략을 격리 설계한다.

## CH25-L05 - Lock Object와 COMMIT/ROLLBACK 연결

### 왜 필요한가

비관/낙관 락의 충돌 성격과 낙관적 버전 카운터 대안, 그리고 DEQUEUE scope 불일치 락 방치 가이드라인까지 동시성 락의 3대 무기를 모두 마스터했다.

이제 이 파편들을 **DML DB 쓰기 연산과 트랜잭션 커밋/롤백 생명주기 벨트에 안전하게 순서대로 매핑 체결하여 '동시성과 원자성을 동시에 완벽 사수하는 최종 안전 변경 통합 패턴'** 으로 종합 완성할 순간이다.
어디서 잠그고, 어디서 읽고, 어디서 쓰며, 언제 커밋해서 락을 푸는지의 시퀀스 배선 순서가 1밀리초라도 어긋나면 락이 걸리지 않은 틈새 찰나에 갱신 손실이 침범해 들어와 데이터를 찢어발긴다.

### 무엇인가

#### 🧭 안전 변경 통합 아키텍처 패턴의 5단계 순차 명세
우리는 아래와 같이 5단계 철칙 배선 순서를 엄격히 밟아 변경 트랜잭션을 완성한다.

```text
[1단계] ENQUEUE (잠금 획득) : 반드시 DML 쿼리는 물론, SELECT 읽기조차 가동하기 '전' 에 입구에서 락 선점!
   │
[2단계] SELECT READ (데이터 조회) : 락을 안전 독점 획득한 완벽 무결 상태에서 DB 원본 레코드를 메모리에 로드!
   │
[3단계] UPDATE DML (변경 예약) : 메모리에서 필드 값을 고치고, 감사필드 스탬프를 찍어 DML 문장 실행 예약!
   │
[4단계] COMMIT / ROLLBACK (트랜잭션 확정) : sy-subrc 성공 결과를 취합해 영구 디스크 확정 격발!
   │
[5단계] DEQUEUE (잠금 해제) : scope=2 인 경우 커밋이 청소해 주나, 명시적으로 락을 완전히 해방 완수!
```

### 어떻게 확인하는가

5단계 순차 체계를 꼬임 없이 완벽하게 조립해 가동하는 종합 실습 소스 코드를 검증한다.

```abap
REPORT zch24_l05_pattern.

DATA: lv_id        TYPE zbooking-booking_id VALUE 'B00006',
      ls_booking   TYPE zbooking,
      lv_err_check TYPE abap_bool VALUE abap_false.

START-OF-SELECTION.
  " [ 1단계: 선행 ENQUEUE 잠금 철칙 ]
  CALL FUNCTION 'ENQUEUE_EZ_BOOKING'
    EXPORTING
      booking_id     = lv_id
      _scope         = '2' " scope=2 로 커밋 자동 수거 연결!
    EXCEPTIONS
      foreign_lock   = 1
      system_failure = 2
      OTHERS         = 3.

  IF sy-subrc = 1.
    MESSAGE '다른 사용자가 편집 중입니다. 잠시 후 다시 시도하십시오.' TYPE 'I'.
    EXIT. " 락 획득 실패 시 즉각 연산 포기 및 정지!
  ELSEIF sy-subrc <> 0.
    MESSAGE '시스템 락 덤프 감지.' TYPE 'A'.
  ENDIF.

  " [ 2단계: 안전 보장 상태에서의 SELECT READ ]
  SELECT SINGLE * FROM zbooking WHERE booking_id = @lv_id INTO @ls_booking.
  
  IF sy-subrc = 0.
    " [ 3단계: 변경 필드 가공 및 DML 예약 ]
    ls_booking-seats += 1. " 좌석 1석 증대!
    ls_booking-changed_by = sy-uname. " 감사 Stamp
    ls_booking-changed_on = sy-datum. " 감사 Stamp
    ls_booking-changed_at = sy-uzeit. " 감사 Stamp
    
    UPDATE zbooking FROM @ls_booking.
    IF sy-subrc <> 0.
      lv_err_check = abap_true.
    ENDIF.
  ELSE.
    lv_err_check = abap_true.
  ENDIF.

  " [ 4단계: 트랜잭션 확정 및 자동 해제 격발 ]
  IF lv_err_check = abap_false.
    COMMIT WORK.
    WRITE: / '저장 성공 및 락 자동 해제 완료.'.
  ELSE.
    ROLLBACK WORK.
    WRITE: / '오류 발생으로 롤백 복구 및 락 수거 완료.'.
  ENDIF.

  " [ 5단계: DEQUEUE 최종 명시 수거 완수 ]
  " (scope=2 이므로 커밋 단에서 소멸했겠으나, 코드 안전 위생을 위해 ENQUEUE 와 동일 scope 로 마무리!)
  CALL FUNCTION 'DEQUEUE_EZ_BOOKING'
    EXPORTING
      booking_id = lv_id
      _scope     = '2'.
```

#### 체험/시뮬레이터 설계 (안전 변경 패턴 흐름도)
- **프로세스 플로우**:
  1. 학습자가 [5단계 파이프라인 레일]을 본다.
  2. [READ 카드를 1단계 ENQUEUE 보다 위로 올려서 조립] 해 본다. A 와 B 가 동시에 똑같은 데이터를 읽어가 버린 뒤에 락을 잡으려 들어, 갱신 손실 방지선이 붕괴되는 오류 사이렌을 확인한다.
  3. 카드를 다시 원래의 [1. ENQUEUE -> 2. READ -> 3. UPDATE -> 4. COMMIT -> 5. DEQUEUE] 정상 순서로 배선 조립한다.
  4. 가동하자, 데이터 구슬이 레일을 따라 충돌 없이 촥 미끄러지며 안전 저장 완료 불빛이 들어오는 시퀀스 피드백을 감상한다.
- **상태 및 데이터**:
  - `UPDATE zbooking 을 친 뒤 COMMIT 도 치기 전에 DEQUEUE 를 명시 실행한 상태` -> 런타임 결과: `Lock released too early. Bypassed transaction update and Lost Update occurred` 사이렌 경보.
- **피드백**: 잠금은 내 연산의 처음부터 끝까지를 품어주는 안전 우산이므로, 변경 DML 은 물론 커밋 확정이 끝나기 직전까지 절대로 우산을 미리 접으면(DEQUEUE) 안 됨을 확인한다.

### 실수/주의

- **SELECT READ 조회를 먼저 다 해놓고, 값 변경 UPDATE 문을 찌르기 직전의 찰나에 쌩뚱맞게 ENQUEUE 를 걸어 잠그기**:
  - **이것은 문맥적으로 락이 없는 틈새(Gap)를 노출하는 최악의 아키텍처 붕괴 안티패턴이다.**
  - 락을 걸기 전에 이미 A 와 B 가 동시에 옛날 데이터를 select 해가 버렸기 때문에, 뒤늦게 업데이트 직전에 락을 잡는 것은 "소 잃고 외양간 문 고치는" 격이며 B 가 덮어써서 Lost Update 가 그대로 터진다.
  - **락은 무조건 조회 SELECT 를 찌르기 '전' 에 입구에서 가장 먼저 획득해야 함을 수호해야 한다.**

### 정리

- **`안전 변경 통합 패턴`** 은 ENQUEUE -> READ -> UPDATE -> COMMIT -> DEQUEUE 다.
- 락은 무조건 조회 SELECT 를 날리기 **`전`** 에 선행 확보한다.
- 해제 DEQUEUE 는 무조건 DML 은 물론 **`COMMIT`** 이 완수된 최후단에 조율해 작동한다.
- 이 5단계 시퀀스 벨트 배선을 수호해야 동시성과 원자성을 동시에 완벽 사수할 수 있다.
