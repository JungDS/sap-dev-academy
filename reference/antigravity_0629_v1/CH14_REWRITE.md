# CH14_REWRITE - Classic DDIC View·유지보수 객체 v1

> 목적: `content/abap/CH14`의 9개 레슨을 "IT 비전공자 입문자가 왜 필요한지부터 실습 확인까지 직관적으로 이해하고 따라갈 수 있는 완성 강의자료" 수준으로 재작성한 기준 원고다. 이 문서는 아직 `content/abap` 원본을 덮어쓴 결과가 아니라, 재작업 판정 이후 실제 반영 전 검수 가능한 v1 기준안이다.

## CH14 전체 설계

CH14의 한 문장 목표는 "조회용(Database/Projection) 및 F4/유지보수용(Help/Maintenance) 4대 클래식 뷰의 용도와 SQL 가부 한계를 정비하고, Foreign Key 무결성 검증, TMG/SM30의 5대 활성화 게이트 및 One/Two Step 화면 구조, SE54 뷰 클러스터 계층 묶음을 활용하여 공연 등록 및 조회 비즈니스 캡스톤을 조립한다"이다.

IT 비전공자 입문자는 SQL 조회가 원천 금지된 Help/Maintenance View에 `SELECT` 쿼리를 날려 빌드 오류를 겪고, TMG 화면 미생성 상태로 SM30을 켜서 게이트 차단 오류를 마주하며, SE16N 데이터 브라우저를 현업의 정식 화면으로 오인 배포하는 오류를 범한다.
따라서 본 챕터는 다음과 같은 불편 해결의 단계적 설계로 전개한다.

1. **조인의 반복 노고 해소**: 프로그램 소스 코드마다 구구절절 작성하던 무겁고 반복되는 JOIN 조건절 $\rightarrow$ SE11 데이터 사전에 조인 뼈대를 이름 붙여 영구 등록해 재사용하는 **Database View** 학습.
2. **필드 노출의 통제 (Projection View)**: 수십 개의 테이블 컬럼 중 사용자에게 불필요한 변경일자나 생성자 등의 열을 솎아내어 핵심 업무 필드만 가볍게 노출하는 **Projection View**의 설계 의도 정립.
3. **F4 도움말의 꼬표 (Help View)**: 공연 코드(`C001`) 옆에 가수 이름(`정훈영`) 설명 텍스트를 덧붙여 보여주는 Search Help 전용 뷰인 **Help View**와 누락 방지용 **Outer Join** 성격 이해.
4. **표준 유지보수 서류판 (Maintenance View)**: 여러 관련 테이블을 묶어 데이터 입력 다이얼로그의 뼈대로 쓰는 **Maintenance View**와 데이터 사전 수준의 강제 **Foreign Key** 입력 정합성 가드 원리.
5. **표준 화면의 출산 (Table Maintenance Generator)**: Utilities 도구를 가동해 CUD 화면을 자동 빌드하는 TMG의 **One Step**(1단계 화면)과 **Two Step**(2단계 상세 화면) 분리 기준 및 이송요청(Recording) 연동 규칙.
6. **SM30 5대 게이트웨이**: 테이블 활성화, 유지보수 허용 플래그, TMG 화면 빌드, 권한 그룹 매칭, 이송요청 스위치가 다 가동되어야 비로소 문이 열리는 **게이트 체크리스트** 학습.
7. **계층형 묶음 관리 (View Cluster)**: "공연 마스터 $\rightarrow$ 회차 정보 $\rightarrow$ 등급 상세" 처럼 계층으로 이어진 테이블 그룹을 SE54에서 묶어 상위 선택 값이 하위 화면에 자동 동기화되는 **뷰 클러스터** 조립.
8. **데이터 점검의 칼 (SE16N)**: 개발자가 테이블 데이터를 고속 모니터링하는 **SE16N** 브라우저와 현업 정식 배포 금지 지침.
9. **미래 CDS의 예고**: SE11 GUI 기반의 클래식 한계와 ADT 소스 기반의 현대적 **CDS View Entity**의 존재 목적 경계 쪼개기.
10. **공연 등록 캡스톤 실습**: ZCONCERT에 TMG 화면을 생성해 F4 Search Help를 장착하고 SM30으로 등록한 뒤, ZV_CONCERT DB View 및 SELECT 쿼리로 데이터를 수집 확인하는 종합 연동 실습.

### 범위와 금지선

- **R6 classic-first 철저 수호**:
  - `SELECT * FROM zv_concert INTO TABLE gt_concert.` 와 같이 classic SELECT 문 및 classic standard 참조 형식을 수호한다.
  - CDS 뷰의 식, 집계, Annotation 문법은 이 장에서 일절 다루지 않고 오직 **SE11 GUI 클래식 환경**만으로 조립을 수행한다.
- **변수 스코프 접두어**: 모든 스크립트는 전역 범위로 간주하며, 일반 변수는 `gv_`, 구조체는 `gs_`, 내부 테이블은 `gt_`, 클래스 참조 변수는 `lo_`/`go_` 접두어를 엄수한다.
- **이름 풀 준수**: 예제 이름은 지정된 이름 풀에서만 가져오며, 주인공은 항상 **정훈영**으로 통일한다.

## CH14-L01 - Database View와 Open SQL JOIN 비교

### 왜 필요한가

우리가 앞 장(CH13)에서 서로 흩어진 여러 테이블을 엮어 하나의 큰 결과로 합쳐 가져오는 `INNER JOIN` 쿼리 기술을 훌륭하게 섭렵했다. 

하지만 실무 현장으로 나가보면, "공연 마스터 정보와 공연장 이름 및 위치를 합치는 조인 연산" 은 비단 예매 리포트뿐만 아니라, 공연 정산 프로그램, 콘서트 마감 화면, 가수별 스케줄러 등 **회사 내 수십 개의 서로 다른 프로그램에서 매번 똑같은 조건으로 무한 반복해서 작성**된다. 
만약 어느 날 갑자기 공연장 테이블의 컬럼 이름이 바뀌거나 조인 기준 키가 하나 더 늘어나면, 개발자는 소스 사방에 작성해 둔 수십 개 프로그램의 JOIN 조건문을 일일이 다 뜯어고쳐야 하는 재앙 수준의 유지보수 노동 고통을 겪게 된다. 

프로그램 소스 코드에 조인을 구구절절 적어두지 말고, **데이터베이스 사전(DDIC)에 조인 연결 뼈대와 규칙을 고정해 두고 '하나의 가상 테이블 이름'으로 영구 등록하여 여러 프로그램이 테이블을 SELECT 하듯 이름만 불러 재사용하는 아키텍처**가 필요하다. 그것이 **[[Database View]]** (데이터베이스 뷰)이다.

### 무엇인가

#### 1. Database View (데이터베이스 뷰)의 정의
- **정의**: 하나 이상의 Basis Table(원천 투명 테이블)들을 공통 키로 엮어, SE11 데이터 사전 상에 가상의 단일 테이블 이름(예: `ZV_CONCERT`)으로 등록해 사용하는 조회용 뷰 객체다.
- **물리적 거동**: 뷰를 생성하고 활성화하면, SAP는 실제 데이터베이스 레벨에 SQL View 객체를 생성해 둔다.
- **[공식 뷰 필드 규칙]**: 뷰를 설계할 때, 클라이언트 격리(Client Dependency) 속성을 유지하기 위해 **반드시 뷰의 첫 번째 컬럼에 `CLNT` 형식 필드(예: `MANDT` 클라이언트 키)를 배치**해야 한다.

#### 2. SQL 쿼리 조회 가능 여부
- **조회 가능**: Database View는 테이블과 동격으로 취급되므로, **아밥 프로그램의 `SELECT * FROM ZV_CONCERT` 절의 FROM 뒤에 테이블명 대신 직접 대입하여 자유롭게 쿼리 조회를 수행**할 수 있다.
- **Read-Only**: 데이터를 저장하는 물리 디스크 장소가 아니므로 오직 조회만 가능하다. 여러 테이블을 엮은 Database View는 무조건 **INNER JOIN** 성격으로 작동하므로, 짝꿍이 없는 공연 데이터는 뷰에서 아예 누락되어 소실됨을 유념해야 한다.

#### 3. Database View vs 소스 코드 JOIN 의 설계적 책무 비교

| 비교 관점 | Database View (디렉토리 뷰) | 소스 코드 JOIN |
| --- | --- | --- |
| **정의하는 위치** | **SE11 데이터 사전 (DDIC)** | 아밥 프로그램 소스 파일 내부 |
| **유지보수 재사용성** | 뷰 한 곳만 고치면 사용하는 전체 프로그램 자동 동기화 | 프로그램마다 일일이 들어가 소스를 고쳐야 함 |
| **결합 자유도** | DDIC에 고정된 **INNER JOIN** 만 가동 | 상황에 따라 INNER, LEFT OUTER 자유 선택 |
| **적합한 비즈니스 상황** | 규칙이 변하지 않는 공통 마스터 조회 | 프로그램마다 조건이 계속 요동치는 가변 조회 |

### 어떻게 확인하는가

에디터에서 Database View인 ZV_CONCERT를 테이블처럼 직접 SELECT 하여 화면에 인쇄하는 구성을 검증한다.

```abap
REPORT zch14_l01_db_view.

" ZV_CONCERT 데이터베이스 뷰의 구조를 고스란히 TYPE 참조하여 내부 테이블 선언!
DATA: gt_concert TYPE STANDARD TABLE OF zv_concert,
      gs_concert TYPE zv_concert.

START-OF-SELECTION.
  " [뷰 조회 철칙] 조인의 복잡한 조건(ON 절 등)을 몰라도, 마치 테이블처럼 뷰 이름으로 SELECT!
  SELECT * FROM zv_concert
    INTO TABLE gt_concert.

  LOOP AT gt_concert INTO gs_concert.
    WRITE: / gs_concert-concert_id, 
             gs_concert-artist, 
             gs_concert-capacity.
  ENDLOOP.
```

#### 체험/시뮬레이터 설계 (코드 JOIN vs Database View 비교 보드)
- **프로세스 플로우**:
  1. 학습자가 [코드 JOIN] 탭을 클릭한다. 복잡하게 얽힌 `ON c~id = p~id` 소스 코드가 가득 출력된다.
  2. [Database View] 탭으로 전환한다. 단 한 줄의 `SELECT * FROM ZV_CONCERT` 클린 쿼리로 똑같은 데이터가 출력되는 화면을 본다.
  3. [ZCONCERT 테이블의 'C003' 공연 데이터 삭제] 단추를 누른다.
  4. 원천 데이터가 지워지자, 뷰 결과판에서도 즉시 `C003` 줄이 실시간 동기화되어 사라지는 거동을 확인한다.
- **상태 및 데이터**:
  - `zv_concert` 의 첫 컬럼에서 MANDT를 제거하고 활성화 시도 -> "Key field MANDT is missing in first position" 사전 에러 하이라이트.
- **피드백**: Database View는 조인의 규칙을 사전(DDIC)에 캡슐화해 묶어둔 훌륭한 재사용 껍데기 창임을 인지시킨다.

### 실수/주의

- **뷰 자체를 데이터 보관소로 착각**: "공연 정보를 ZV_CONCERT 뷰에 직접 INSERT 해서 등록해야지" 라고 뷰에 데이터를 직접 쓰려고 들면, 여러 테이블 뷰는 DB 수준에서 읽기 전용으로 가로막히므로 데이터 쓰기는 반드시 원본 테이블(`ZCONCERT`)에 행해야 한다.

### 정리

- **`Database View`** 는 반복되는 조인을 SE11 데이터 사전에 등록해 테이블처럼 불러 쓰는 조회용 뷰다.
- 클라이언트 종속성 수호를 위해 **반드시 첫 번째 컬럼에 MANDT**를 바인딩하여 설계한다.
- 여러 테이블 뷰는 철저히 **INNER JOIN** 성격으로 작동하여 짝이 없으면 행이 누락되므로 조회 껍데기로만 쓴다.

## CH14-L02 - Projection View 개념과 한계

### 왜 필요한가

테이블을 만들면, 그 테이블 안에 들어있는 모든 데이터 열(컬럼)을 화면에 그대로 노출해서 보여주는 게 가장 상식적으로 보인다. 
하지만 실무 현업의 관점은 다르다. 공연 마스터 테이블 속에는 비즈니스에 필요한 공연 코드, 정원 외에도 내부 승인 상태 코드, 시스템 생성자 ID, 최종 변경 시간 등 수많은 내부용 기술 필드 40개가 함께 적재되어 있다. 

이 테이블을 그대로 현업 사용자 검색창이나 입력창에 날것으로 노출하면, 화면이 극도로 번잡해져서 사용자는 어떤 필드를 만져야 할지 헷갈리고, 보지 말아야 할 대외비 단가나 내부 기술 정보가 무방비로 노출되는 설계상 취약성을 내포하게 된다. 

여러 테이블을 합치는 복잡한 계산 없이, **단일 원천 테이블에서 딱 핵심 업무 필드 3~4개만 보이도록 열 차단 필터(커튼)를 쳐서 좁고 가볍게 데이터를 투과시켜 보여주는 특수 뷰**가 필요하다. 그것이 **[[Projection View]]** (프로젝션 뷰)이다.

### 무엇인가

#### 1. Projection View (프로젝션 뷰)의 정의
- **정의**: **단 하나의 Basis Table(원천 투명 테이블)** 에서 원하는 특정 컬럼(열)만 쏙쏙 솎아내어 보여주도록 껍데기 라인을 조율하는 단일 테이블 전용 뷰 객체다.
- **물리적 거동**: 조인도 하지 않고 행을 거르는 검색 조건(WHERE)도 내포하지 않으므로, 데이터베이스 레벨에 SQL View 실물 객체를 생성하지 않고 SAP 어플리케이션 사전 수준에서만 필드를 걸러 매핑해 준다.

#### 2. SQL 쿼리 조회 가능 여부
- **조회 가능**: 1개 테이블만 모방하고 있으므로, Database View와 마찬가지로 **아밥 프로그램 소스의 `SELECT` 문 FROM 자리에 직접 써서 쿼리로 조회 수납하는 것이 완벽하게 허용**된다.

#### 3. Projection View 설계의 철칙과 한계
- **Key 필드 필수 장착**: 뷰 필드를 솎아낼 때, 귀찮다고 기본 키(Primary Key) 필드를 제외하고 설명 필드만 노출하면, 훗날 이 뷰를 통해 읽어 들인 1번 행이 데이터베이스 상의 어느 원천 레코드인지 컴퓨터가 식별할 도리가 없어 런타임 매핑 에러가 나므로, **테이블의 기본 키는 무조건 뷰에 포함**시켜야 한다.
- ** CDS View Entity 로의 진화 예고**:
  - *미래 이정표*: 이 클래식 Projection View는 계산이나 가공 컬럼을 넣지 못하는 심각한 기능적 한계가 있다. 현대 SAP 개발 표준에서는 이보다 100배 강력한 소스 코드 기반의 **CDS View Entity** (CH22에서 상세 배움)로 진화하여, 계산식(`CASE WHEN` 등)을 품은 최첨단 프로젝션을 사용하므로 방향의 연계 감각만 잡고 넘어간다.

### 어떻게 확인하는가

에디터에서 Projection View인 ZPV_CONCERT_BASIC을 통해 필요한 4개 필드만 클린 쿼리로 SELECT 하는 구문을 확인한다.

```abap
REPORT zch14_l02_projection_view.

" 원본 ZCONCERT 테이블 대신, 필드가 4개로 솎아진 ZPV_CONCERT_BASIC 뷰 구조를 참조!
DATA gt_basic TYPE STANDARD TABLE OF zpv_concert_basic.

START-OF-SELECTION.
  " 1개 테이블의 프로젝션 뷰이므로 SELECT 쿼리가 정상 작동함
  SELECT * FROM zpv_concert_basic
    INTO TABLE gt_basic.
    
  " gt_basic 구조에는 base_price나 created_by 같은 불필요한 열이 없어
  " 메모리 사용량도 절약되고 데이터 매핑도 깔끔해짐
```

#### 체험/시뮬레이터 설계 (Projection View 필드 선택)
- **프로세스 플로우**:
  1. 학습자가 ZCONCERT 원본의 10개 필드 명단을 본다.
  2. [기본 정보만 노출] 체크박스를 켠다.
  3. `base_price` 와 `created_by` 등의 불필요한 열 위로 투명한 회색 커튼이 촥 내려와 시야를 차단하고, `concert_id` 와 `artist` 열만 또렷하게 투과되어 보이는 연출을 감상한다.
- **상태 및 데이터**:
  - `concert_id` (기본 키) 체크 해제 시도 -> "Primary Key field must be included in Projection View" 붉은색 에러 경고등 켜짐.
- **피드백**: 프로젝션 뷰는 원본 데이터를 복제하지 않고, 오직 열(Column)의 시야각만 좁혀주는 스마트 가드망임을 인지시킨다.

### 실수/주의

- **행을 거르는 필터 용도로 오용**: "서울 공연장 정보만 걸러서 보여주는 프로젝션 뷰를 만들어야지" 하고 접근하면 안 된다. Projection View는 **열(Column)을 솎아내어 숨길 뿐**, 행(Row)을 거르는 selection condition 세팅은 불가능하므로, 행 필터는 `Database View`나 아밥 쿼리 `WHERE` 문으로 해결해야 한다.

### 정리

- **`Projection View`** 는 단 하나의 테이블에서 업무상 필요한 특정 열(Column)만 솎아내어 노출하는 단일 뷰다.
- 식별 정합성을 위해 **기본 키(Key) 필드는 무조건 뷰에 포함**시켜 설계해야 한다.
- 행을 필터링하는 조건절이나 복잡한 조인은 불가능하므로, 오직 "필드 노출 차단 가드" 용도로만 사용한다.

## CH14-L03 - Help View와 Search Help 연결

### 왜 필요한가

공연을 등록하는 화면에서 사용자에게 "이 콘서트의 공연장 코드는 `H01` 입니다" 라고 코드 번호를 키보드로 외워서 치라고 요구하면, 현업 담당자는 백이면 백 오타를 내서 에러를 터트리거나 선배가 적어둔 코드 종이 장부를 책상에서 뒤지는 업무 병목을 겪는다. 

사용자가 키보드 `F4` 키를 누르면 화면에 공연장 코드 정보와 옆에 한글 이름(**"올림픽홀"**), 그리고 위치(**"서울시 송파구"**)까지 보조 설명으로 친절하게 붙여서 리스트업해주고, 그중 하나를 마우스로 더블클릭하면 관련 코드 `H01` 이 화면 입력창에 자동으로 쏙 돌아와 기입되는 편리한 **F4 입력도움말**이 필수적이다. 
단일 테이블만 가지고는 코드 이름만 나올 뿐 이 풍부한 다국어 설명 정보를 엮어낼 도리가 없으므로, **도움말의 영양가 있는 소스 역할을 수행하기 위해 다중 텍스트 테이블을 안전하게 묶어 대기해 주는 F4 전용 뷰**가 필요하다. 그것이 **[[Help View]]** (헬프 뷰)이다.

### 무엇인가

#### 1. Help View (헬프 뷰)의 정의
- **정의**: 오직 **`Search Help` (F4 입력도움말)** 의 검색 리스트 데이터 소스(Selection Method)로 활약하기 위해, 여러 보조 설명 테이블들을 묶어서 개설하는 **F4 전용 특수 뷰 객체**다.
- **물리적 거동**: 데이터베이스 레벨에 실물 객체를 생성하지 않고, 오직 SAP GUI F4 팝업창 가동 시 동적 렌더링에만 사용된다.

#### ⚠️ [물리적 쿼리 조회 불가 철칙]
- *아바 개발자들이 시험이나 실무에서 100% 한 번씩 뒤통수 맞는 구문 한계선이다.*
- **Help View는 프로그램 소스 코드의 SELECT 문을 사용해 데이터를 조회하는 것이 절대(Absolutely) 불가능하다.**
- `SELECT * FROM ZHV_CONCERT` 라고 적는 즉시 컴파일러가 "ZHV_CONCERT is a Help View and cannot be accessed via ABAP SQL" 이라며 쿼리를 차단하므로, 프로그램 조회용 뷰(`Database View`)와 헬프용 뷰를 머릿속에서 엄격히 분리 설계해야 한다.

#### 2. Outer Join (외부 조인)의 보존성 거동
- **Outer Join**: Help View는 짝이 없으면 행이 소실되는 Database View의 INNER JOIN 방식과 달리, **`Outer Join` 성격으로 여러 테이블을 엮는다.**
- *이점*: 예를 들어 공연장 `H03` 번호의 한글 번역 명칭 텍스트 데이터가 보조 테이블에 실수로 누락되어 비어있더라도, `H03` 코드 자체는 헬프 창 리스트에 누락 없이 보존되어 나타나고 이름 칸만 곱게 빈칸으로 표출된다. 만약 inner join이었다면 `H03` 코드 자체가 F4 창에서 증발하여 사용자가 코드를 평생 고르지 못하게 되는 장애로 번지게 된다.

### 어떻게 확인하는가

F4 입력도움말(Search Help) 내부에서 Help View가 Selection Method로 도킹되어 화면과 값을 주고받는 파라미터 매핑 상태를 검증한다.

```text
[ Search Help 파라미터 유통 구조도 ]
사용자가 화면 입력창 s_hall 에서 F4 클릭
  ──> 1. Import Parameter: 화면의 값을 Help View의 검색 필터 인수로 전달 (조회 축소)
  ──> 2. Selection Method: Help View (Outer Join) 가동하여 코드 + 한글 설명 리스트업
  ──> 3. Export Parameter: 사용자가 더블클릭해 고른 'H01' 키 값을 화면 입력창 s_hall 로 안전 배달 복귀
```

#### 체험/시뮬레이터 설계 (F4 여행 경로)
- **프로세스 플로우**:
  1. 학습자가 가상 입력창 [공연장 코드] 우측의 [다중 검색(화살표)] 단추를 클릭한다.
  2. 실행 돋보기가 `Search Help ZSH_HALL` 로 달려간다.
  3. 그 안에서 데이터 원천인 `Help View ZHV_HALL` 을 노크해, 주 테이블의 코드 `H01` 과 보조 테이블의 이름 **"올림픽홀"** 을 Outer Join으로 한 줄로 묶어 표로 렌더링한다.
  4. 사용자가 "올림픽홀" 줄을 더블클릭하자, Export 필터를 거쳐 `H01` 문자가 화면 텍스트 상자에 쏙 복귀 안착되는 3단계 동적 이동 지도를 관찰한다.
- **상태 및 데이터**:
  - `SELECT * FROM zhv_hall` 소스 기입 후 컴파일 -> "ZHV_HALL is a Help View and SQL access is prohibited" 에러 빨간불 점등.
- **피드백**: Help View는 일반 조회용이 아니라 오직 F4 Search Help에 기름진 영양분(보조 설명)을 대주는 전용 도우미 뷰임을 선사한다.

### 실수/주의

- **Help View에 일반 SQL SELECT 쿼리 날리기**: 개발 도중 "F4 화면에 나오던 뷰 이름이 `ZHV_...` 니까 프로그램 조회 쿼리에서도 `FROM ZHV_...` 에서 읽어와서 출력해야지" 했다가 컴파일 차단을 마주하므로, 조회 쿼리는 무조건 `Database View` 나 `투명 테이블`을 타깃으로 삼아야 한다.

### 정리

- **`Help View`** 는 F4 Search Help 입력도움말 창에 코드와 보조 텍스트 설명을 함께 보여주는 **F4 전용 뷰**다.
- 데이터 누락 방지를 위해 **Outer Join** 성격으로 테이블을 엮어 동작한다.
- **ABAP SQL(SELECT)을 통한 조회가 원천 불가능**하므로, 프로그램 조회 쿼리에 도킹하지 않는다.

## CH14-L04 - Maintenance View와 Foreign Key 관계

### 왜 필요한가

우리가 SE11 데이터 사전에서 신나게 투명 테이블(`ZCONCERT`)을 빌드했다. 
하지만 이 테이블의 정원이나 가격 같은 마스터 데이터를 채워 넣기 위해 매번 개발자가 `SE11 -> Utilities -> Table Contents -> Create Entries` 화면을 켜서 한 건 한 건 노가다로 대신 타이핑해 넣어줄 수는 없는 노릇이다. 

실무 현업의 일반 업무 담당자가 웹 화면이나 SAP 표준 화면을 통해 "직접 신규 공연을 등록하고, 잘못 등록한 정원을 수정하며, 취소된 공연 행은 삭제할 수 있도록" 깔끔하고 정돈된 CUD(생성/변경/삭제) 가공용 화면을 열어주어야 한다. 
또한 이 과정에서 사용자가 실수로 존재하지도 않는 가짜 공연장 코드 `H999` 를 대충 입력하고 저장 버튼을 누르는 데이터 변조 사고를 치면 마스터 데이터가 오염된다. 
**업무 담당자가 데이터를 일괄 생성/유지보수할 수 있는 표준 서류판을 제공하고, 데이터 사전 수준에서 Foreign Key(외래 키) 장벽을 가동해 정합성이 검증된 진짜 알짜배기 마스터 데이터만 입력되도록 강제하는 유지보수용 특수 뷰**가 필요하다. 그것이 **[[Maintenance View]]** (메인터넌스 뷰)이다.

### 무엇인가

#### 1. Maintenance View (유지보수 뷰)의 정의
- **정의**: **`Table View Maintenance` (SM30/SM31 T-code)** 라는 SAP 표준 유지보수 화면 트랜잭션을 가동하기 위해, 관련 테이블들을 결합해 설계하는 **유지보수 화면 전용 뷰 객체**다.
- **물리적 거동**: Help View와 마찬가지로 데이터베이스 레벨에 실물 뷰 객체를 생성하지 않고, 오직 표준 유지보수 다이얼로그(`SE54`)의 입력 제어판 메타데이터 뼈대로만 가동된다.
- **[SQL 조회 절대 불가]**: **Help View와 마찬가지로 아밥 프로그램 소스에서 `SELECT * FROM ZMV_...` 라고 쿼리를 때려 읽어오는 것이 절대 불가능하다.**

#### 2. Foreign Key (외래 키)의 강력한 사전적 Input Check
- **입력 검증 장벽**: Maintenance View에 등록된 하위 테이블 필드(`zconcert-hall_id`)가 상위 마스터 테이블(`zhall-hall_id`)과 **`Foreign Key` 관계**로 단단히 체결되어 있으면,
- **동작**: 사용자가 SM30 화면에서 데이터를 입력하고 엔터를 치거나 저장하는 찰나에, SAP 화면 프레임워크는 사전(DDIC)에 정의된 외래 키 관계를 자동으로 평가하여 **"입력한 H999 공연장은 Hall 마스터 대장에 존재하지 않는 가짜 코드이므로 입력을 거부합니다"** 라고 화면 아래에 빨간 에러 메시지를 띄우며 저장을 완벽히 차단한다 (`Input Check`). (이로써 마스터 데이터 오염이 원천 예방된다).

#### 3. Database View vs Maintenance View 목적지 대조

| 구분 척도 | Database View (조회용 뷰) | Maintenance View (유지보수용 뷰) |
| --- | --- | --- |
| **태생 목적** | 아밥 프로그램의 **데이터 조회(SELECT) 효율화** | 현업 부서의 **데이터 입력/수정/삭제(SM30) 자동화** |
| **SQL 쿼리 조회** | **완벽 허용** (`FROM ZV_...`) | **절대 불가능** |
| **결합 테이블 조건** | 개발자가 `ON` 절에 직접 매치한 조인 조건 | 데이터 사전에 등록된 **Foreign Key 종속 관계**만 허용 |
| **가동 화면** | SELECT 조회 쿼리, SE16N 브라우저 | **SE54 화면 생성기, SM30/SM31 관리 화면** |

### 어떻게 확인하는가

에디터와 SM30 검문 가드를 대조하여 Foreign Key가 존재하지 않는 키 입력을 통제하는 과정을 확인한다.

```text
[ Foreign Key 사전 입력 검문 모션 ]
사용자가 SM30 공연 등록 화면에 [공연: C003 | 공연장: H999] 입력 후 [저장] 클릭
  ──> 1. 시스템 사전(DDIC)의 Foreign Key 종속 관계(ZCONCERT-HALL_ID -> ZHALL-HALL_ID) 확인
  ──> 2. ZHALL 테이블에서 'H999' 레코드가 존재하는지 SELECT SINGLE 자동 검증 수행
  ──> 3. 존재하지 않으므로 화면 입력 필드를 빨간색으로 잠그고 "Entry H999 does not exist in ZHALL" 에러 표출 가드!
```

#### 체험/시뮬레이터 설계 (Foreign Key로 문 열기)
- **프로세스 플로우**:
  1. 학습자가 SM30 가상 화면의 [공연장 코드] 입력 칸에 `H999` 를 강제로 써넣는다.
  2. [저장] 단추를 누른다.
  3. Foreign Key 검문소의 셔터 차단막이 철컥 내려앉으며 입력 상자가 붉게 충혈되고, 아래 상태 바에 "존재하지 않는 공연장 번호입니다" 에러 메시지가 출력되어 저장 버튼이 비활성화되는 가드를 확인한다.
- **상태 및 데이터**:
  - `ZCONCERT 테이블에 Foreign Key 지정을 해제한 상태` -> 검문소가 사라져 가짜 코드 `H999`가 그대로 통과해 DB 마스터 데이터를 훼손 오염시키는 위험 렌더링.
- **피드백**: Foreign Key는 단순한 문서용 장식 선이 아니라, SM30 입력 화면의 정합성을 사전 수준에서 자동으로 사수해 주는 강력한 보안 쉴드임을 인지시킨다.

### 실수/주의

- **조회 효율을 위해 Maintenance View 생성**: "프로그램 쿼리 조인을 편하게 하려고 `ZMV_...` 유지보수 뷰를 만들었습니다" 하면, 쿼리에서 `FROM ZMV_...` 를 부를 수 없어 무용지물이 되므로, 조회 목적은 철저히 `Database View` 를 개설해야 한다.

### 정리

- **`Maintenance View`** 는 조회 편의가 아니라 현업 담당자의 표준 CUD 입력 화면(**SM30**)을 제공하기 위해 만드는 전용 뷰다.
- 상하 테이블 관계는 반드시 사전의 **`Foreign Key`** 종속성을 기반으로 연결되어 설계된다.
- **SQL 쿼리 조회가 절대 불가**하므로 프로그램 소스 코드 내부에서 조회 타깃으로 삼지 않는다.

## CH14-L05 - Table Maintenance Generator / SM30

### 왜 필요한가

유지보수용 서류판인 Maintenance View(`ZMV_CONCERT`)의 설계도면 도면을 완성했다. 
하지만 도면만 그렸을 뿐, 현업 사용자가 실제로 마우스를 클릭해 행을 추가하고 삭제할 수 있는 표준 GUI 입력 팝업창 화면이 아직 메모리에 창조되어 존재하지 않는다. 

개발자가 한 땀 한 땀 화면 레이아웃을 그리고 저장 단추를 코딩해서 입력 폼을 짜려면 족히 3일은 걸리는 엄청난 수공업 노동이 필요하다. 
데이터 사전 도면 규격을 분석하여, **1초 만에 추가/수정/삭제 단추와 데이터 입력 격자가 깔끔히 장착된 표준 CUD 화면 폼을 전자동으로 출산해 주는 화면 자동 빌더**가 필요하다. 그것이 **[[Table Maintenance Generator]]** (테이블 유지보수 생성기, TMG) 이며, 이렇게 창조된 문을 열고 들어가 운영하는 트랜잭션 화면이 **`SM30`** 이다.

### 무엇인가

#### 1. Table Maintenance Generator (TMG - 화면 자동 빌더)
- **위치**: `SE11` 테이블/뷰 화면 $\rightarrow$ 상단 메뉴 `Utilities` $\rightarrow$ **`Table Maintenance Generator`** 선택.
- **동작**: 권한 그룹과 화면 단계 수, 화면 번호를 지정하고 [화면 생성] 버튼을 누르면 시스템은 내부적으로 다이얼로그 화면(Screen)과 입력 검증 제어 모듈을 단숨에 자동 코딩해 완성한다.
- **SM30 / SM31**: TMG로 탄생한 화면 폼을 최종 실행하여, 사용자가 데이터를 수납/가공하고 저장(`SAVE`)하는 표준 마스터데이터 관리 트랜잭션 화면이다.

#### 2. TMG 화면의 2대 유형 기어: One Step vs Two Step
화면을 자동으로 빚어낼 때, 테이블의 필드 개수 용량에 따라 단계를 전략적으로 결정한다.

- **One Step (1단계 유형 - 목록 화면 1개)**:
  - **구조**: 단 하나의 **목록(Table Control) 화면**에서 행 추가와 셀 편집을 즉석에서 처리한다.
  - **적합**: 공연장 코드와 이름처럼 필드가 2~3개밖에 없는 단순 마스터 테이블에 상책이다.
- **Two Step (2단계 유형 - 목록 + 상세 폼 화면 2개)**:
  - **구조**: 1번 화면은 전체 목록을 스캔하는 개요 판이고, 특정 행을 더블클릭하면 2번 화면인 **독립 단건 상세 입력 카드 폼(Form Screen)** 으로 뛰어들어 상세 편집을 수행한다.
  - **적합**: 공연 마스터처럼 필드가 15개가 넘어 한 화면에 가로 스크롤로 다 담을 수 없는 뚱뚱한 테이블에 상책이다.

#### 3. SM30 문이 열리기 위한 [5대 게이트 체크리스트]
*SM30을 켰을 때 "유지보수 화면이 존재하지 않는다"며 굳어버리는 에러를 방지하는 5대 물리 열쇠다.*

1. **1문 - 테이블 활성화 (`Active`)**: 테이블 자체가 SE11에서 오류 없이 활성화 완료 상태여야 한다.
2. **2문 - 유지보수 허용 플래그 (`Display/Maintenance Allowed`)**: 테이블 기술 속성(Technical Settings) 탭의 유지보수 여부 스위치가 `Allowed`(허용) 로 켜져 있어야 한다. (기본값인 Not Allowed 상태면 진입이 차단된다).
3. **3문 - TMG 화면 빌드 (`Generator Created`)**: Utilities 메뉴를 타고 들어가 TMG 화면 객체 생성을 성공적으로 완료해 두어야 한다.
4. **4문 - 권한 그룹 매칭 (`Authorization Group`)**: 사용자가 진입 가능한 권한 그룹(일반적으로 임시 테스트용인 `&NC&` - No Group 지정)이 TMG에 정확히 세팅되어 있어야 한다.
5. **5문 - 이송 Recording 등록 스위치**: Customizing 테이블일 경우 저장 시 **이송요청서(TR)** 를 자동으로 강제 팝업해 물어주는 Recording Routine이 켜져 있어야 시스템 간 데이터 이송이 안전하게 성립된다.

### 어떻게 확인하는가

SE11에서 TMG를 생성하고 SM30으로 진입하여 데이터를 입력/저장하는 수동 절차를 검수한다.

```text
[ SM30 화면 배포 및 운영 검수 절차 ]
1단계: SE11 ZCONCERT 테이블 진입 -> 유지보수 허용 옵션 'Allowed' 세팅 활성화
2단계: 메뉴 Utilities -> TMG 클릭 -> Authorization Group = '&NC&', 
       Maintenance Type = 'One Step', 화면 번호 = '9001' 입력 후 [화면 생성] 클릭
3단계: /nSM30 이동 -> 테이블명 'ZCONCERT' 입력 후 [Maintain] 클릭하여 5대 문 해제 확인!
4단계: [New Entries] 클릭 -> C001 공연 입력 후 [저장] -> 이송요청 팝업 물림 작동 점검
```

#### 체험/시뮬레이터 설계 (SM30이 열리기까지)
- **프로세스 플로우**:
  1. 학습자가 SM30 유지보수 콘솔 앞의 닫혀있는 거대한 철문을 본다. 철문에는 5개의 튼튼한 금속 자물쇠가 걸려있다.
  2. 학습자가 [1. 테이블 활성] 열쇠를 꽂자 1번 자물쇠가 풀리며 파란 불이 들어온다.
  3. 순서대로 [2. 허용 플래그 Allowed], [3. TMG 화면 생성], [4. NC 권한 매칭], [5. 이송 스위치] 열쇠를 다 꽂는다.
  4. 5개의 자물쇠가 전부 해제되는 찰나, 철문이 스르륵 열리며 깔끔한 그리드 테이블 SM30 입력 판이 찬란하게 드러나는 동적 해제 비주얼을 확인한다.
- **상태 및 데이터**:
  - `유지보수 허용 플래그를 Not Allowed로 둔 상태` -> 2번 게이트가 붉게 막히며 "Display/Maintenance not allowed for table" 에러 하이라이트.
- **피드백**: TMG 화면 생성은 5대 물리 열쇠 조건이 다 맞춰져야 비로소 가동되는 종합 시스템 조립 공정임을 체감시킨다.

### 실수/주의

- **테이블 필드 변경 후 TMG 미갱신**: 실무에서 정말 많이 저지르는 버그다. 테이블에 신규 컬럼을 하나 추가해 놓고 TMG를 재생성(Recreate/Change)해주지 않으면, SM30 화면에는 새로 추가한 필드가 나타나지 않거나 런타임에 화면 필드 불일치 덤프를 터트리므로, **테이블 필드 변경 시 TMG 갱신은 절대 세트 철칙**이다.

### 정리

- **`Table Maintenance Generator`** (TMG)는 테이블 규격을 뜯어 CUD 입력 GUI 화면을 즉석에서 자동 코딩 출산하는 부품이다.
- 필드가 적으면 목록형인 **`One Step`**, 필드가 많으면 목록+상세 카드형인 **`Two Step`** 화면으로 빌드한다.
- SM30 문이 열리려면 **테이블 활성, Allowed 플래그, TMG 생성, 권한 매칭, 이송 지정**의 5대 게이트를 다 뚫어야 한다.

## CH14-L06 - View Cluster — 관련 뷰를 묶어 유지보수

### 왜 필요한가

테이블 하나당 TMG를 가동해서 각각의 SM30 관리 화면을 엮어 제공하는 법을 정복했다. 
하지만 실제 현업의 마스터데이터 관리 프로세스는 그렇게 단순하게 흘러가지 않는다. 

"공연 마스터 정보를 하나 등록한 뒤, 그 하위에 딸린 1회차, 2회차, 3회차 상세 공연 정보도 넣어야 하고, 그 회차별로 존재하는 VIP석, R석, S석의 좌석 등급별 가격 정보도 함께 등록해야 합니다"
이 연관된 마스터-디테일 테이블 3개를 각자 따로 SM30 화면을 3번 켜서 수동으로 코드를 맞춰 기입하게 하면, 현업 담당자는 "어라? 내가 1회차에 가격을 적었나? 2회차에 적었나?" 헷갈리며 상하 관계 키가 완전히 뒤틀려 깨지는 데이터 부정합 사고를 무조건 겪게 된다. 

마스터 테이블과 하위 종속 테이블들의 유지보수 화면을 **단 하나의 계층형 통합 관리 화면으로 묶어두고, 상위 마스터에서 1번 공연을 콕 집어 선택하면 하위 탭에는 자동으로 그 공연에 종속된 회차와 가격 정보만 동기화되어 깔끔하게 물고 내려가는 계층식 통합 유지보수 장치**가 요구된다. 그것이 **[[View Cluster]]** (뷰 클러스터)이다.

### 무엇인가

#### 1. View Cluster (뷰 클러스터)의 정의
- **정의**: Foreign Key 계층 관계로 단단하게 엮여 있는 여러 유지보수 뷰와 테이블들을 **SE54** 트랜잭션을 통해 **단 하나의 통합 계층 관리 구조(Tree 구조)** 로 조립해 다루는 고급 유지보수 객체다.
- **물리적 거동**: 독자적인 화면 다이어그램(View Cluster Maintenance Dialog)을 개설하여, 좌측에는 계층 구조 트리 메뉴를 띄우고 우측에는 선택 행에 종속된 세부 내역을 동적 필터링해 노출한다.

#### 2. 상하 계층의 동적 동기화 원리
- **자동 필터 조건**: 사용자가 좌측 트리 메뉴의 상위인 '공연(Header)' 목록에서 특정 공연 `C001` 행을 마우스로 선택하고 하위인 '회차(Detail)' 메뉴를 더블클릭하면, 시스템은 내부적으로 부모 키 값을 상속하여 **`WHERE concert_id = 'C001'` 필터를 자동으로 걸어 하위 화면을 노출**한다. (따라서 사용자는 엉뚱한 부모 키를 잘못 입력하는 실수를 원천적으로 차단당한다).
- **Foreign Key 모델의 투명성**: 뷰 클러스터는 독자적으로 데이터 관계를 빚지 못한다. 반드시 테이블 간에 단단한 **`Foreign Key` 관계가 사전에 정의**되어 있어야만 계층 트리를 구성할 수 있으므로, 기초 데이터 모델의 정합성이 그대로 드러나게 된다.

### 어떻게 확인하는가

SE54에서 헤더-상세 계층 테이블들을 묶어 뷰 클러스터를 가동하고 계층 동기화를 검수한다.

```text
[ View Cluster 계층 작동 검수도 ]
좌측 메뉴 트리: [공연 마스터] -> [공연 회차 정보] -> [회차별 가격]
1단계: 우측 '공연 마스터' 목록에서 [C001 | 정훈영 갈라콘서트] 행 선택
2단계: 좌측 트리에서 '공연 회차 정보' 더블클릭
3단계: 우측 화면이 C001 공연에 속한 [01회차 | 6월30일], [02회차 | 7월1일] 로우만 
       자동 필터링되어 수납 노출되는 동기화 검수 완료!
```

#### 체험/시뮬레이터 설계 (계층 유지보수 지도)
- **프로세스 플로우**:
  1. 학습자가 좌측 트리의 [공연 헤더] 메뉴를 누른다. 우측에 공연 3건 목록이 뜬다.
  2. 학습자가 2번 공연 행을 클릭하고, 좌측 트리 아래의 [회차 상세] 단추를 더블클릭한다.
  3. 우측 테이블이 파란색 물결 효과를 내며 싹 물갈이되면서, 오직 2번 공연에 속한 상세 회차 2줄만 솎아져 동기화되는 동적 인터랙티브 화면을 체감한다.
- **상태 및 데이터**:
  - `Foreign Key 관계가 부재한 쌩뚱맞은 테이블을 엮어 클러스터 생성 시도` -> "No Foreign Key relationship found between tables" 에러 딱지 출력.
- **피드백**: 뷰 클러스터는 여러 SM30을 단순히 모아둔 바로가기 링크창이 아닌, 부모-자식의 피를 묶어 정합성을 보존하는 일체형 뷰어임을 알려준다.

### 실수/주의

- **단일 테이블 관리에 뷰 클러스터 남용**: 테이블 1개만 관리하면 되는데 멋있어 보인다고 뷰 클러스터를 정의하면, 불필요하게 좌측 트리 영역이 지분만 차지하고 SE54 내부 빌드가 복잡해져 유지보수 낭비를 초래하므로, **단건 테이블은 무조건 심플한 SM30** 단독 화면으로 회귀해야 한다.

### 정리

- **`View Cluster`** 는 마스터와 여러 종속 테이블들을 묶어 하나의 계층형 화면(Tree-Detail)에서 일관되게 관리하는 고급 도구다.
- 상위 행 선택 시 하위 목록이 **부모 키를 기준으로 자동 동적 필터링**되어 정합성을 수호한다.
- 반드시 테이블 간에 **Foreign Key** 종속성이 정교하게 체결되어 있어야만 빌드가 성립된다.

## CH14-L07 - SE16N 데이터 브라우저

### 왜 필요한가

우리가 데이터를 조인해서 읽는 뷰도 만들었고, SM30 유지보수 화면에 뷰 클러스터 계층 화면까지 완벽하게 빌드해 제공했다. 
하지만 개발자와 시스템 운영자의 본능적인 가려움은 해결되지 않았다. 

"지금 방금 프로그램 에러가 났는데, 테이블에 필터가 어떻게 박혀서 저장되었는지 TMG 화면 켜고 들어가서 찾기엔 너무 팝업이 많고 번잡합니다. 그냥 **아무 테이블이든 이름만 치면 1초 만에 내부 원천 레코드 전체를 엑셀 격자 모양으로 시원하게 들여다볼 수 있는 만능 즉석 돋보기 도구**가 갖고 싶습니다"
이 가려움을 해결해 주는 개발자용 데이터베이스 브라우저 돋보기가 필요하다. 그것이 **`SE16N`** (Data Browser) 이다.

### 무엇인가

#### 1. SE16N (데이터 브라우저)의 정의와 특징
- **정의**: 테이블이나 뷰의 이름만 기입하면, 별도의 화면 제작이나 TMG 생성 절차 없이 **테이블 내부 데이터를 즉시 그리드 표로 인쇄해 주는 개발자/운영자용 초고속 데이터 돋보기 트랜잭션**이다.
- **가동법**: `SE16N` 실행 $\rightarrow$ 테이블명 기입 $\rightarrow$ F8 실행 끝.
- *강력한 기능*: 화면 내에서 즉석 컬럼 필터, 정렬, 레이아웃 커스텀 저장, 엑셀 파일 로컬 저장이 GUI 기본 사양으로 즉시 제공된다.

#### ⚠️ [운영 환경 데이터 수동 강제 편집 금지령 (보안 철칙)]
- *초보 개발자들이 가장 쉽게 저지르는 보안 불감증 사고이자, 감사(Audit) 지적 1순위 함정이다.*
- SE16N은 테이블의 원천 데이터를 마음대로 고칠 수 있는 수동 편집창(디버깅 모드 우회 등)을 제공하곤 한다.
- **위험성**: 이 경로를 통해 운영 서버 DB 데이터를 직접 펜으로 슥슥 고쳐서 저장하면, 데이터 사전 및 프로그램 수준에서 열심히 설계해 둔 **Foreign Key 정합성 검증과 비즈니스 입력 체크 가드가 100% 무력화되어 완전히 우회(Bypass)** 된다.
- **결과**: 결국 겉으로는 성공한 것처럼 보여도 데이터가 찢어져 정합성이 파괴되는 심각한 DB 정합성 붕괴를 초래하므로, **운영 서버 데이터 임의 변경은 절대 금지**이며 오직 조회(Inspection) 목적으로만 활용해야 한다.

### 어떻게 확인하는가

SM30 화면에서 ZCONCERT 데이터를 수정한 뒤, SE16N 브라우저를 즉시 켜서 데이터가 창고에 물리적으로 안전하게 쓰여 수납되었는지 점검 절차를 확인한다.

```text
[ 데이터 보존 수동 추적 검수 절차 ]
1단계: SM30 화면 진입 -> ZCONCERT에 [C003 | 정훈영 갈라쇼 | 500석] 입력 후 [저장]
2단계: /nSE16N 트랜잭션 화면 실행
3단계: Table = 'ZCONCERT', concert_id = 'C003' 조건 입력 후 [실행 (F8)] 클릭
4단계: 즉석 돋보기 그리드 판에 [C003 | 정훈영 갈라쇼 | 500석] 로우 상자가 
       실시간 수납 완료되어 노출되는 것을 최종 확인!
```

#### 체험/시뮬레이터 설계 (저장 후 추적기)
- **프로세스 플로우**:
  1. 학습자가 SM30 화면에서 데이터를 고치고 [저장]을 누른다. 데이터 트랙터가 디스크 창고로 출발한다.
  2. [SE16N 돋보기 켜기]를 누른다. 돋보기 렌즈 속에 ZCONCERT 디스크의 18번 섹터가 줌인되어 비춰진다.
  3. 방금 저장한 따끈따끈한 ` 정훈영 갈라쇼` 로우 상자가 18번 방에 안전하게 안착해 정렬되어 있는 물리 저장 상태를 직접 모니터링한다.
- **상태 및 데이터**:
  - `SE16N 강제 편집 시도` -> "Bypassing business logic warning" 빨간색 보안 경보 계기판 활성화 피드백.
- **피드백**: "SE16N은 개발자의 진단 돋보기일 뿐, 현업의 정식 업무 화면으로 릴리즈해서는 절대 안 된다" 는 설계 경계선을 각인시킨다.

### 실수/주의

- **현업 사용자에게 조회용으로 SE16N 권한 배포**: "사용자가 테이블 데이터를 보고 싶어 하니 간편하게 SE16N 조회 권한을 쥐여 주자" 했다가, 사용자가 덤프 데이터나 인사 대외비 정보까지 싹 조회해가거나 테이블을 망가뜨리는 사고가 나므로, 현업 사용자에게는 무조건 **정식 조회 리포트 프로그램(SALV 등)** 을 짜서 서빙해야 한다.

### 정리

- **`SE16N`** 은 별도 코딩 없이 테이블 내부를 초고속 점검 모니터링할 수 있는 개발자용 만능 데이터 브라우저다.
- 데이터 사전 수준의 비즈니스 가드를 무력화하므로, **운영 서버 원천 데이터 직접 강제 수정은 절대 엄금**한다.
- 현업 사용자에게는 SE16N을 절대 배포하지 않으며, 오직 안전하게 설계된 **정식 SALV 리포트 화면**을 통해 데이터를 제공한다.

## CH14-L08 - Classic View와 CDS 비교

### 왜 필요한가

우리가 이 장에서 SE11 GUI 화면을 열심히 클릭해서 만드는 4대 클래식 뷰(`Database, Projection, Help, Maintenance View`)의 기본 감각을 아주 훌륭하고 정밀하게 몸에 이식했다. 

그런데 현업 프로젝트의 최신 아키텍처 문서를 들여다보거나 선배들의 모던 개발 화면을 훔쳐보면, SE11을 켜지 않고 이상한 소스 코드 창에서 뷰를 조립해 코딩하는 **`CDS View` (Core Data Services)** 라는 단어가 수없이 등장한다. 
"클래식 뷰는 이제 완전히 낡아 없어지는 골동품인가요? 굳이 왜 이걸 지금 공부한 거죠?" 라는 기초 무용론 회의감에 휩싸여 학습의 집중도가 흐려지기 십상이다. 
**우리가 배운 클래식 뷰가 해결한 아키텍처적 기초 감각과, 모던 CDS가 활약하는 현대적 기술 경계선의 관계 지도를 명확하게 쪼개어 정리**해 두어야 흔들림 없는 설계 눈빛을 갖게 된다.

### 무엇인가

#### 1. 클래식 DDIC 뷰의 아키텍처적 한계
- **SE11 GUI 방식**: 예전 클래식 뷰들은 SE11이라는 닫힌 마우스 드로잉 화면 안에서 마우스 클릭으로만 조인을 짜고 칼라를 솎아냈다.
- **식(Expression) 연산 부재**: 뷰 내부에서 `A 컬럼 + B 컬럼 = C 컬럼` 같은 수학적 사칙연산 가공이나 `CASE WHEN` 조건절 가공을 절대로 할 수 없는 극심한 기능적 제약이 있었다.
- **버전 관리 불가**: 소스 코드가 아니라 사전 객체라, 깃(Git) 등의 최신 버전 관리 도구로 코드 변경 이력을 추적할 수 없는 한계가 있었다.

#### 2. 모던 CDS View Entity (현대적 소스 코드 기반 뷰)
- **정의**: 현대 SAP 개발의 핵심 중추다. SE11을 켜지 않고 Eclipse ADT 툴에서 **순수 소스 DDL 코드 선언문으로 조인과 프로젝션을 코딩해 정의하는 현대적 뷰**다.
- **장점**: 뷰 내부에 풍부한 가공 식, 집계, 그리고 Fiori/OData 연계 어노테이션(`@`)을 가득 담아낼 수 있어 현대 SAP RAP 프레임워크의 절대적인 근간으로 활약한다.
- *가드레일*: 본격 CDS 코딩 실습은 한참 자란 뒤인 **CH22**에서 정식으로 다룬다. 지금은 "클래식 뷰 개념이 자란 곳 위에 CDS 코드가 얹어지는 타임라인 관계" 만 이식한다.

#### 3. Classic DDIC View vs CDS View Entity 경계선 대조

| 비교 지표 | Classic DDIC View | CDS View Entity (모던 CDS) |
| --- | --- | --- |
| **정의하는 환경** | **SE11 마우스 클릭 GUI 화면** | 이클립스 ADT 내의 **소스 코드 파일** |
| **가공 연산 자유도** | 거의 불가능 (단순 짝맞춤 수준) | 풍부한 식, 집계, `Association` 결합 지원 |
| **현대 도구 친화성** | SAP GUI 전용 (OData 연계 불리) | OData 및 모바일 Fiori 표준 화면 다이렉트 도킹 |
| **학습 타임라인 기어** | **지금 이 장 (기초 감각 적재)** | **나중 (CH22에서 본격 폭발)** |

### 어떻게 확인하는가

학습 타임라인 카드를 정석대로 2대 영역으로 분류하고 설계 의도를 검증한다.

```text
[ 지금 마스터해야 할 클래식 감각 ]
- 테이블 간의 Foreign Key 관계망 이해
- F4도움말과 Help View의 Export 매핑 동작
- SM30을 켜기 위한 Allowed 및 TMG 생성 조건
- 뷰 클러스터의 헤더-상세 계층 동기화

[ 나중에 CDS 챕터(CH22)에서 폭발시킬 모던 지식 ]
- DDL 소스 코드로 JOIN과 CASE WHEN 구현하기
- OData 발행을 위한 @OData.publish 어노테이션 이식
- RAP 프레임워크와의 연동
```

#### 체험/시뮬레이터 설계 (타임라인 경계 카드)
- **프로세스 플로우**:
  1. 학습자가 "Foreign Key 입력 체크" 카드를 드래그한다.
  2. [지금 CH14 클래식] 바구니에 골인시키자 "통과! 데이터 무결성의 기초 뼈대는 지금 다져야 합니다" 메시지가 출력된다.
  3. "OData Fiori 연계 어노테이션" 카드를 집어든다.
  4. [지금] 에 넣으려 하자 "차단! 현대 연계는 CH22 CDS 영역에서 꼼꼼히 배우게 됩니다" 라고 나중 표지판이 켜지는 모션을 관찰한다.
- **상태 및 데이터**:
  - `SM30 화면 생성` 카드 -> 지금 CH14 분류 통과.
  - `RAP 연동` 카드 -> 나중 CH22 분류 통과.
- **피드백**: "클래식 뷰에서 다지는 테이블 관계성 감각은, 훗날 CDS 코드를 작성할 때도 뼈대가 되어 그대로 이어지는 필수 주춧돌" 임을 나선형 나침반 그래픽으로 보여준다.

### 실수/주의

- **클래식을 무가치한 유물로 폄하하고 CDS만 쫓기**: "어차피 실무에선 CDS 쓴다니까 TMG나 SM30, Foreign Key는 대충 보고 넘어가야지" 했다가, 현업에 가서 여전히 90% 이상 가동되는 기존 자산 마스터데이터 SM30 화면에서 에러가 났을 때 TMG 갱신이나 권한 문제 하나를 못 풀어 개발이 막히는 수모를 겪으므로, 클래식 기초를 아주 꼼꼼하게 다져두어야 한다.

### 정리

- **`Classic View`** 는 SE11 GUI로 테이블 관계성과 SM30 유지보수 화면의 뼈대를 다지는 필수 기초다.
- **`CDS View Entity`** 는 뷰를 DDL 소스 코드로 조립해 가공 식을 얹어 현대 Fiori 화면으로 통하는 통로를 뚫는 현대적 주역이다 (CH22).
- 클래식에서 배우는 Foreign Key와 뷰 필드 감각은 CDS를 지배하는 뼈대로 상속되므로 지금 매우 꼼꼼히 연마한다.

## CH14-L09 - 실습 — 공연 등록 화면 (View · SM30)

### 왜 필요한가

우리가 배운 Database View의 조회 원리, Help View의 F4 입력도움말 매핑, Maintenance View의 Foreign Key 가드, TMG의 화면 출산 기술을 각각 파편적으로 공부했다. 

이제 이 모든 사전(DDIC) 객체들을 **🎫 하나의 완결된 실무 마스터데이터 관리 파이프라인**으로 꽁꽁 조립하여 묶어낼 시간이다. 개발자가 직접 테이블 로우를 건드는 초보 단계에서 탈퇴하여, **"업무 담당자는 정식 표준 화면(SM30)에서 F4를 눌러 존재성 검증을 거친 깨끗한 공연 정보를 다건 등록하고, 개발자는 이를 Database View 및 SELECT 쿼리로 안전하게 긁어와 리포트에 띄워 확인하는 정석 프로세스"** 의 풀 체인을 내 손으로 직접 구현해 내야 한다.

### 무엇인가

#### 🎫 마스터데이터 등록 & 조회 파이프라인 종합 아키텍처 명세
우리는 데이터 사전에 1개의 투명 테이블, 1개의 데이터베이스 뷰를 조립하고 화면을 생성한다.

1. **원천 테이블 조립 (`ZCONCERT`)**:
   - `concert_id` (PK), `perf_id` (Foreign Key $\rightarrow$ `ZPERF-PERF_ID`), `date`, `hall_id` 컬럼 구성.
   - `ZPERF` 테이블과의 Foreign Key 결합을 통해, 존재하지 않는 가짜 perf_id 입력 차단 가드 개설.
2. **F4 Search Help 체결**:
   - `perf_id` 필드에 Search Help를 장착하여, 입력 상자에서 F4를 누르면 공연 이름과 설명 설명 텍스트가 Outer Join으로 묶여 렌더링되게 엮음.
3. **TMG 화면 출산**:
   - `ZCONCERT` 에 Utilities TMG를 가동해 One Step 화면 번호 9001번으로 관리 창 출산.
4. **조회용 Database View 조립 (`ZV_CONCERT`)**:
   - `ZCONCERT` 와 `ZPERF` 를 INNER JOIN 결합하여 `concert_id`, `perf_id`, `perf_name`(공연명), `date`, `hall_id` 를 뷰 필드로 솎아내어 등록. (client MANDT를 1순위 첫 컬럼에 배치).
5. **조회 리포트 구동**:
   - 아밥 프로그램 내부에서 `ZV_CONCERT` 뷰를 FROM 자리에 직접 써서 SELECT 조회 실행.

### 어떻게 확인하는가

에디터와 SM30 및 SE16N 브라우저를 횡단하며 캡스톤 5단계 프로세스의 정합성을 종합 수동 검수한다.

1. `SM30` 을 실행하고 테이블 `ZCONCERT` 를 입력해 진입한다.
2. [New Entries]를 누르고 공연 ID `C003` 을 기입한다.
3. `perf_id` 칸에서 `F4` 키를 눌러 팝업창을 띄우고, 등록해 둔 `P001` (정훈영 갈라쇼)을 더블클릭해 선택한다. (Help View F4 복귀 작동 검수).
4. `perf_id` 에 가짜 코드 `P999` 를 손으로 입력하고 엔터를 친다. -> Foreign Key 검문소가 작동하여 화면 아래에 "P999 does not exist" 빨간 에러 메시지를 뿜으며 저장을 거부하는지 확인한다. (정합성 가드 검수).
5. 올바르게 `P001`로 입력한 뒤 [저장] 단추를 누른다.
6. `/nSE38` 로 건너가 아래의 검증 리포트 프로그램을 작성 후 실행하여, 조인 조건 없이도 Database View 를 경유해 공연 코드 `C003` 과 함께 짝꿍인 공연 이름 **"정훈영 갈라쇼"** 가 한 줄로 예쁘게 리포트에 인쇄되어 출력되는지 최종 확인한다.

```abap
REPORT zrep_concert_view_check.

" ZV_CONCERT 데이터베이스 뷰를 참조하는 뷰 수납용 내부 테이블 선언
DATA gt_concert TYPE STANDARD TABLE OF zv_concert.
DATA gs_concert TYPE zv_concert.

START-OF-SELECTION.
  " 5단계: Database View를 경유해 INNER JOIN 조립 결과를 고속 SELECT 수납!
  SELECT concert_id perf_id perf_name date hall_id
```

#### 체험/시뮬레이터 설계 (공연 등록 콘솔)
- **프로세스 플로우**:
  1. 학습자가 [F4 도움말 가동]을 누른다. ZPERF에서 P001 등의 코드 설명표가 F4 창에 출력된다.
  2. [P001 선택 후 SM30 저장]을 탭한다. 데이터가 ZCONCERT 디스크 원천 창고에 적재된다.
  3. [Database View로 확인] 버튼을 누르자, ZV_CONCERT의 투명 창을 통해 원천 정보와 ZPERF의 이름이 INNER JOIN으로 나란히 묶인 조인 데이터가 리포트에 깔끔하게 표출되는 연동 경로 지도를 추적한다.
- **상태 및 데이터**:
  - `ZPERF에서 P001 데이터를 지운 상태` -> ZV_CONCERT는 INNER JOIN 방식이므로, ZCONCERT에 데이터가 살아있음에도 불구하고 매칭이 깨졌기 때문에 최종 뷰 결과에서 해당 공연 행이 통째로 실종되는 뷰 소실 현상 데이터 피드백 표출.
- **피드백**: "Database View는 INNER JOIN 결합이므로, 기준 정보(P001)가 마스터에서 지워지면 뷰 조회 화면에서도 행이 완전히 증발하여 보이지 않게 된다" 는 데이터 유실 피드백을 빨간불로 경고한다.

### 실수/주의

- **TMG 생성 과정 누락**: "SM30을 켰는데 maintenance dialog가 부재하다며 실행이 막힙니다" 하는 것은 테이블만 활성화하고 Utilities 메뉴에서 TMG(Screen 9001번 등) 빌드를 진행하지 않았기 때문이므로 순서를 꼼꼼히 챙겨야 한다.

### 정리

- **`ZCONCERT` TMG 생성 $\rightarrow$ F4 Search Help 장착 $\rightarrow$ `SM30` 현업 등록 $\rightarrow$ `ZV_CONCERT` DB View 조회** 5단계를 거쳐 완벽한 마스터데이터 유통 체인을 완성한다.
- Foreign Key 결합을 통해 SM30 화면 상에서의 **비정합성 가짜 코드 입력을 차단**하여 정합성을 사수한다.
- Database View는 INNER JOIN으로 작동하므로, 마스터 데이터가 삭제되면 뷰 화면에서도 행이 실종되는 거동을 이해한다.
