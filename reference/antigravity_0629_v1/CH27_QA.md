## 3. 공식 문서 수동 근거 명세

`abapset_handler.htm` 을 대조하여 Event Handler 의 물리 메모리/안전 제약 요건을 철저히 규명했다.

- **SET HANDLER Null 참조 덤프 제약 규칙**:
  `SET HANDLER` 를 통해 이벤트를 바인딩할 때, 이벤트 발생자(Raiser: `mo_grid`) 또는 이벤트 수령자(Handler: `handler`) 참조 변수가 `INITIAL` (Null)인 채로 껍데기 기동하면 즉시 **`SET_HANDLER_FOR_NULL` (발생자 Null) 또는 `SET_HANDLER_HOBJ_NULL` (수령자 Null) 런타임 덤프**를 터트리고 시스템이 강제 다운된다.
- **Garbage Collector 참조 유지 사양**:
  `SET HANDLER` 에 의해 바인딩이 성공하면, 아바 메모리 관리 테이블(System Registration Table)이 발생자와 수령자 사이의 참조 관계를 유지하므로 가비지 컬렉터(Garbage Collector)의 수집 범위로부터 수령자 객체가 강제 삭제되지 않도록 보존(Keep-alive)되는 성질이 언어 명세상 공식 보장됨을 대조 및 수록 완료.

## 4. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. SET HANDLER 를 사용하여 이벤트를 잘 등록해 두었는데 왜 가끔 SET_HANDLER_FOR_NULL 이라는 무서운 런타임 덤프가 터지며 시스템이 폭사하나요? 방어 코딩은 무엇인가요?
**A**: **이벤트를 발생시킬 CL_GUI_ALV_GRID (go_grid) 인스턴스가 CREATE OBJECT (또는 NEW)로 메모리에 만들어지기 전인 Null (INITIAL) 상태에서 성급하게 SET HANDLER 배선부터 질렀기 때문이며, 등록 전 IS BOUND 검문소로 방어해야 합니다.**
- **덤프의 물리 원인**:
  아바 엔진은 `SET HANDLER ... FOR oref` 문장을 해석할 때 oref 가 가리키는 실제 메모리 주소를 뒤져서 이벤트 수신 테이블에 등록하려 듭니다.
  그런데 oref 가 아무 주소도 없는 껍데기 Null 상태이면, "존재하지도 않는 유령 grid 객체에 대고 어떻게 락을 얹으라는 말이냐" 며, 예외 catch 도 허용하지 않고 즉시 `SET_HANDLER_FOR_NULL` 치명적 덤프를 터트려 세션을 파괴합니다.
- **방어 코딩**:
  그리드가 정상 생성되어 메모리에 영구 안착해 있는지 아래와 같이 **`IS BOUND`** 꼬리표로 무조건 검문한 뒤에만 등록을 태워야 안전합니다.
  ```abap
  " [ oref 가 유효한 객체 주소를 쥐었을 때만 배선 등록 허용! ]
  IF go_grid IS BOUND AND go_handler IS BOUND.
    SET HANDLER go_handler->on_double_click FOR go_grid.
  ELSE.
    " 덤프 방어용 예외 메시지 출력
    MESSAGE 'ALV 컨테이너 객체가 활성화되지 않아 이벤트 바인딩을 보류합니다.' TYPE 'S' DISPLAY LIKE 'E'.
  ENDIF.
  ```

### Q2. ALV 더블클릭이나 핫스팟 이벤트에서 넘겨주는 e_row-index 행 번호를 그대로 내부 테이블 INDEX 에 대고 읽어서 처리했습니다. 그런데 왜 사용자가 컬럼 정렬이나 필터를 켜는 순간 엉뚱한 사람의 예약 데이터가 지워져 버리는 대형 금융 사고가 나나요?
**A**: **e_row-index 는 화면에 단지 줄지어 서 있는 시각적인 '껍데기 줄 번호' 일 뿐이며, 정렬/필터가 작동하는 순간 화면 상의 줄 번호와 실제 내부 테이블(mt_booking)의 인덱스 매핑이 완전히 뒤집어지기 때문입니다.**
- **사고 유시 시나리오**:
  1. 원본 예약 테이블: 1행(홍길동), 2행(임꺽정), 3행(정훈영).
  2. 사용자가 ALV 화면에서 '이름 내림차순' 정렬을 때렸습니다. -> 화면 뷰에는 1줄(정훈영), 2줄(임꺽정), 3줄(홍길동)로 바뀝니다.
  3. 사용자가 화면의 1줄(정훈영)을 더블클릭해 취소를 찌릅니다.
  4. 이벤트는 화면 1줄을 클릭했으므로 `e_row-index = 1` 을 줍니다.
  5. 프로그램은 이 1을 쌩으로 읽어 `READ TABLE mt_booking INDEX 1` 을 때려 원본 테이블의 1행인 '홍길동' 님 예약을 강제 취소 포스팅해 날려버립니다.
- **정석 방어선 (업무 키 추출)**:
  `e_row-index` 줄 번호는 화면 위치를 가져오기 위한 다리일 뿐입니다. 
  무조건 이 행 번호로 예약 테이블의 해당 행을 읽어 **`booking_id = 'B010'` 과 같은 고유 기본키(Key) 값만 먼저 쏙 빼낸 뒤, 그 booking_id 를 조건으로 `READ TABLE mt_booking INTO DATA(ls) WITH KEY booking_id = ...` 로 2차 재식별**해야 정렬/필터 꼬임 시에도 무결하게 정훈영 님의 예약이 조준 갱신됩니다.

### Q3. toolbar 이벤트 메서드 소스 본문 내벽 안에서 직접 UPDATE DB 쿼리를 치거나 MESSAGE ... TYPE 'I' 같은 경고 팝업을 적어 넣으면 왜 안 되나요?
**A**: **toolbar 이벤트는 사용자가 화면 리사이즈를 하거나, 마우스를 툴바 위로 쓱 굴리거나, 그리드가 리프레시될 때마다 시도 때도 없이 수십 번 연속 격발되는 렌더링 루틴이므로, 그 안에 업무 로직을 박아두면 성능 파쇄기 대형 덤프가 나기 때문입니다.**
- **작동 모순**:
  `toolbar` 이벤트는 버튼을 진짜 클릭했을 때 도는 이벤트가 아닙니다.
  "예약 취소라는 이쁜 아이콘 버튼을 화면 툴바 진열장에 어떻게 그려서 올릴 것인가" 의 **UI 껍데기 렌더링 셋업**입니다.
  사용자가 윈도우 창 크기를 늘릴 때마다, 마우스를 툴바에 올릴 때마다 이 toolbar 메서드는 수십 번 백그라운드에서 다시 실행됩니다. 
  그 안에 DB 수정 구문을 꽂아두면 화면 크기를 살짝 조절하는 찰나에 50번 연속 예약 취소 쿼리가 실행되어 락 테이블과 DB 버퍼가 과부하로 마비됩니다.
- **철칙**:
  `toolbar` 이벤트 안에서는 오직 `APPEND VALUE #( function = 'ZCANCEL' ... ) TO e_object->mt_toolbar.` 단추 배치 징표만 적고 빠져나가야 하며, 진짜 취소 비즈니스 로직은 버튼이 클릭되어 유저 커맨드 명령이 송출되는 **`user_command` 이벤트 단독 구역**으로 격리해야 정석입니다.

### Q4. 커스텀 버튼을 다 고르고 [예매 취소]를 클릭했습니다. 디버거로 찍어보니 내 내부 테이블 데이터 status = 'CANCELLED' 로 잘 고쳐진 것을 확인했는데, 왜 화면 표 상에는 여전히 '정상' 상태로 굳어 있나요?
**A**: **백엔드 메모리 테이블(mt_booking)의 값만 바꾼 것이지, 화면을 그리는 ALV 프레임워크에게 "메모리가 바뀌었으니 표를 리렌더링해서 새로 그려라" 고 지시하는 새로고침(Refresh) 전원을 안 켰기 때문입니다.**
- **해결책**:
  상태 값을 루프로 다 돌며 고친 뒤, 반드시 그 꼬리에 아래와 같이 **`refresh_table_display`** 격발 포트를 기입해야 합니다.
  ```abap
  " 내부 테이블 변조 완료 후!
  mo_grid->refresh_table_display(
    " 필요 시 is_stable (정렬/스크롤 위치 고정 옵션) 셋업 동반
  ).
  ```
이 한 줄이 누락되면 화면은 옛 데이터 상태로 굳어 멈추어 있게 되며, 사용자는 버튼이 뇌사 상태인 줄 오해하고 단추를 무한 클릭하여 중복 트랜잭션을 날리는 런타임 버그의 시발점이 됩니다.

### Q5. ALV 툴바 핸들러 클래스 변수를 로컬 서브루틴/메서드 안방에서 지역 변수(DATA)로만 대충 NEW 해서 SET HANDLER 하면 왜 툴바 버튼 클릭 시 작동을 안 하고 먹통이 되나요?
**A**: **지역 참조 변수(Local Reference)는 서브루틴/메서드가 실행 완료되어 닫히는 즉시 스택 메모리에서 소멸되므로, 바인딩 배선도 끊겨 가비지 컬렉터(GC)에 의해 핸들러 인스턴스 자체가 조기 수거 사형당하기 때문입니다.**
- **소멸 타임라인 (에러 코드)**:
  ```abap
  FORM init_alv.
    " DATA 로 선언해 지역 변수 처리!
    DATA lo_handler TYPE REF TO lcl_alv_handler.
    lo_handler = NEW #( io_grid = go_grid ).
  ENDFORM. " <- 이 ENDFORM 을 탈출하는 1밀리초의 찰나!
  ```
  `lo_handler` 는 지역 스택 변수이므로 PBO 화면을 다 그리고 서브루틴 `init_alv` 을 종료하고 탈출하는 순간 흔적도 없이 증발합니다.
  비록 `SET HANDLER` 로 묶어두었더라도, 참조를 쥐고 있는 주인 징표가 날아갔기 때문에 아바 가비지 컬렉터는 런타임에 이 핸들러를 "주인 없는 좀비 객체" 로 간주해 메모리 빗자루로 쓸어 지워버립니다. 
  결과적으로 사용자가 화면에서 버튼을 클릭해도 수신할 핸들러 자체가 이미 저승으로 갔기 때문에 버튼은 장식이 되어 먹통이 됩니다.
- **방어 규칙**:
  핸들러 변수는 절대로 서브루틴 안에 숨겨진 임시 로컬 변수로 두면 안 되며, 프로그램이 켜져 있는 내내 생존할 수 있도록 **프로그램 메인 GD(Global Data) 범위의 `DATA go_handler` 전역 슬롯**이나, 화면을 총괄 지휘하는 **Controller 클래스의 멤버 속성(Attribute)** 에 보관해야 GC 의 사형을 면하고 영구 수명(Keep-alive)을 보장받습니다.
