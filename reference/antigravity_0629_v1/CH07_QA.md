## 3. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. Transparent Table (투명 테이블) 이라는 명칭에서 '투명하다' 는 단어는 구체적으로 무슨 뜻이며, 구조체(Structure) 선언과 비교했을 때 물리적으로 어떤 생존력 차이가 있나요?
**A**: **'투명하다' 는 것은 아바 사전(DDIC)에 정의된 테이블 명세가 실제 DBMS 의 물리 테이블 구조와 1:1로 완전 일치해 내부가 똑같이 들여다보인다는 뜻이며, 프로그램 종료 즉시 공중 분해되는 램 휘발성의 구조체와 달리 디스크에 데이터가 영구 보존됩니다.**
- **투명 테이블의 본질과 물리 생존력**:
  - **투명(Transparent)의 실체**: 
    우리가 SE11 에서 `ZGUGUDAN` 이라는 테이블을 빚을 때 컬럼의 순서, 이름, 타입 정보는 실제 데이터베이스(DBMS - HANA DB 등)에 완전히 똑같은 모습의 물리 테이블 구조로 자동 투영되어 생성됩니다. 사전과 실제 DB 구조가 중간 매핑 변환 레이어 없이 투명하게 1:1 일치하므로 투명 테이블이라 부릅니다.
  - **생존력의 차이**:
    - **Structure / Internal Table (메모리 휘발성)**: 
      Application Layer 의 램(RAM) 공간 내에만 임시 가설된 풍선 그릇입니다. 프로그램 실행 창을 닫거나 컴퓨터가 강제 부팅되는 순간, Roll Area 세션이 파괴되어 데이터는 영구 소멸됩니다.
    - **Transparent Table (디스크 영속성)**: 
      SE11 활성화 격발 시, 아바 커널의 **`DDL Processor`** 가 물리 DDL 문(`CREATE TABLE`)을 DB 로 방출해 Database Server 의 하드디스크 디렉토리(Persistent Storage)에 테이블을 물리적으로 굴착 개설합니다. 프로그램이 수백 번 꺼지고 사용자가 로그아웃해도 데이터는 하드디스크 자성체에 영구 잠금되어 보존되며 전사적으로 공유됩니다.

### Q2. 테이블의 첫 번째 컬럼에 MANDT 를 키 필드로 선언해 두면, 프로그램에서 Open SQL 로 데이터를 조회할 때 클라이언트 격리 처리가 어떻게 자동으로 이루어지나요?
**A**: **개발자가 SQL 문에 클라이언트 조건을 누락하더라도, 아바 커널의 Open SQL 번역기가 DBMS 로 쿼리를 송출하기 직전에 `WHERE MANDT = '현재로그인한클라이언트번호'` 조건절을 묵시적 자동으로 강제 주입(Implicit Client Filtering)하여 물리 격리합니다.**:
- **Implicit Client Filtering (묵시적 격리) 작동 원리**:
  SAP ERP 는 단 하나의 DB 로 수많은 그룹 계열사(Client - 방)가 서버 자원을 공유해 쓰는 멀티테넌트(Multi-tenant) 아키텍처입니다. 
  100번 클라이언트 방 사용자가 조회 버튼을 눌렀을 때, `ZGUGUDAN` 테이블에서 200번 클라이언트 방의 데이터가 노출되면 중대 보안 사고가 터집니다.
  이를 막기 위해 테이블 첫 컬럼에 **`MANDT`** 를 키로 꽂아 둡니다. 
  개발자가 소스 상에 `SELECT * FROM zgugudan INTO TABLE gt_gugu.` 라고 쌩 쿼리를 짜서 격발하면, 아바 커널 런타임 엔진이 SQL 을 가로채어 **`WHERE MANDT = '100'`** (사용자 세션의 로그인 클라이언트 번호)을 SQL 꼬리에 묵시적 자동 조입하여 가공한 뒤 DB 에 쏩니다. 
  그 결과 DBMS 는 오직 100번 꼬표가 붙은 데이터 세그먼트만 디스크에서 긁어오므로 룸 간 데이터 침범이 완벽히 차단됩니다. (이를 클라이언트 독립성이라 합니다.)

### Q3. Technical Settings (기술 설정) 에서 지정하는 Data Class (APPL0, APPL1, APPL2) 와 Size Category 는 실제 데이터베이스(DBMS) 단에서 어떤 물리적 공간 격리를 유발하나요?
**A**: **Data Class 는 테이블 성격에 따라 마스터(APPL0), 트랜잭션(APPL1), 설정(APPL2)으로 나누어 DB 의 물리 테이블스페이스(Tablespace) 저장 디렉토리를 분리 격리하여 디스크 I/O 병목을 막고, Size Category 는 최초 획득할 물리 세그먼트(Initial Extent) 크기를 가이드해 단편화를 가드합니다.**
- **인프라 힌트 기작**:
  - **`Data Class` (물리 테이블스페이스 격리)**:
    - **APPL0 (Master Data)**: 데이터 입력은 가끔 발생하고 주로 고속 조회가 격발되는 마스터 성격의 디스크 영역에 테이블을 배치해 조회 캐시 효율을 극대화합니다.
    - **APPL1 (Transaction Data)**: 초당 수백 건의 영수증이나 결산 데이터가 인서트되어 요동치는 영역입니다. DB 가 쓰기 분산(I/O 분배)이 최적화된 독립 물리 드라이브 구역에 할당해 쓰기 병목을 예방합니다.
    - **APPL2 (Organization/Config)**: 회사 코드 설정 등 정적인 정보입니다. 메모리 적재율이 높은 전용 스페이스에 상주시켜 탐색 오버헤드를 0에 수렴시킵니다.
  - **`Size Category` (Initial Extent 가이드)**:
    테이블이 처음 램/디스크를 확보할 때 획득할 '초기 덩어리(Extent)' 의 바이트 용량을 힌트로 줍니다. 
    대용량 테이블인데 Category 를 최소인 `0` 으로 지정해 두면, 데이터가 10만 건 들어찰 때마다 디스크 세그먼트를 추가로 야금야금 쪼개서 획득하는 **단편화 (Fragmentation)** 가 발생해 디스크 읽기 헤더가 춤을 추며 I/O 속도가 지연됩니다. 
    미리 예상 규모에 맞는 Size 를 선고해 두어 단편화 공간을 넓게 원샷 확보(Extent Allocation)해야 성능이 사수됩니다.

### Q4. 데이터 사전(DDIC)에서 테이블 정의 명세와 필드 정보는 구체적으로 어떤 SAP 시스템 데이터베이스 테이블들에 보존되어 형상 관리되나요?
**A**: **테이블의 마스터 속성(헤더)은 DD02L 테이블에 메타 데이터로 저장되고, 테이블 내 각 컬럼의 구체적인 필드 속성 리스트는 DD03L 시스템 테이블에 행 단위로 적재되어 시스템에 의해 형상 관리됩니다.**
- **사전 시스템 메타 테이블 구조**:
  - **`DD02L` (Table Header 대장)**:
    SE11 에서 `ZGUGUDAN` 을 빚고 저장할 때, "ZGUGUDAN 은 application 데이터 클래스(A)이며, Transparent 종류이고, 클래스는 APPL1 이다" 라는 식의 테이블 껍데기 헤더 명세 속성 정보가 `DD02L` 테이블에 `TABNAME = ZGUGUDAN` 키를 물고 1행 인서트 기록됩니다.
  - **`DD03L` (Field Component 대장)**:
    ZTABLE 에 소속된 필드들의 리스트 정보입니다. `ZGUGUDAN` 의 `MANDT`, `DAN`, `MUL`, `RESULT` 의 4개 컴포넌트 정보가 `DD03L` 테이블에 4행으로 촥 적재됩니다. 각 행에는 필드의 순번(`POSITION`), 키 여부(`KEYFLAG`), 상속받은 Data Element 명 등이 물리 보존됩니다.
  - **의존성 추적**:
    `Where-Used List` (사용처 추적)를 쏠 때, 아바 엔진은 바로 이 **`DD03L`** 을 쿼리 검색하여 "ZDE_DAN 이라는 Data Element 를 타입으로 쓰고 있는 테이블이 무엇이냐?" 를 1초 만에 훑어내어 형상 정합성을 수호합니다.

### Q5. 활성화되어 10만 건의 실 데이터가 가득 차서 서비스 중인 운영 Transparent Table 의 특정 필드 길이나 타입을 SE11 에서 쌩으로 바꾸면 왜 치명적인 참사가 나나요?
**A**: **DB 레벨에서 테이블 락(Table Lock)이 걸려 전산이 마비되고, 컬럼의 물리적 바이트 변환 불일치(Conversion Conflict)가 발생해 기존 100만 건 데이터가 쌩으로 파괴 유실되는 대형 뇌사 장애를 유발하기 때문입니다.**
- **물리 테이블 변경의 위험성**:
  Structure 는 메모리 형상일 뿐이어서 타입을 마음껏 고쳐도 무방합니다. 
  그러나 Transparent Table 은 디스크 디렉토리에 고정 바이트 블록 단위로 데이터 레코드가 기계적으로 정렬 적재되어 있습니다. 
  예컨대 10자리 문자(`CHAR 10`) 컬럼을 갑자기 정수형(`INT4`)으로 바꾸거나 길이를 좁히면, DBMS 엔진은 저장된 기존 10진수 비트열을 정수로 해석하지 못해 데이터 변환 에러(Database Conversion Error)를 터트리며 테이블 자체를 깨뜨려 먹습니다. 
  또한 구조 변경 연산 중에는 테이블의 모든 데이터에 쓰기/읽기 락(Lock)이 강제 체결되므로, 현업에서 조회를 누르는 수천 명의 세션이 전원 락 대기에 걸려 업무 전체가 굳어 서버가 정지됩니다. 
  **철칙**: 실 데이터가 상주하는 테이블의 속성 수정은 데이터 백업 후 사전 데이터 마이그레이션 이송 절차(CTS - CH35)를 타야만 생존이 보증됩니다.
