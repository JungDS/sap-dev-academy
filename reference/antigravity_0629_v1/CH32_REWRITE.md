# CH32_REWRITE - 성능 분석과 튜닝 v1

> 목적: `content/abap/CH32`의 5개 레슨을 "IT 비전공자 입문자가 왜 필요한지부터 실습 확인까지 직관적으로 이해하고 따라갈 수 있는 완성 강의자료" 수준으로 재작성한 기준 원고다. 이 문서는 아직 `content/abap` 원본을 덮어쓴 결과가 아니라, 재작업 판정 이후 실제 반영 전 검수 가능한 v1 기준안이다.

## CH32 전체 설계

CH32의 한 문장 목표는 "성능 추측의 해악을 타파하고, DB 병목 추적기 `ST05 SQL Trace` 와 ABAP 코드 내부 시간 소모 히트 리스트인 `SAT Runtime Analysis`, 운영 전체의 무거운 SQL 을 수집하는 `SQLM`, 최다 빈발 1+N 성능 재앙인 `SELECT in LOOP` 퇴치(FOR ALL ENTRIES 및 JOIN 튜닝), FAE 드라이버 빈 테이블 가드, 그리고 HANA DB 엔진에 집계를 위임하는 Code Pushdown 아키텍처를 구축한다"이다.

IT 비전공자 입문자는 `FOR ALL ENTRIES IN @lt_driver` 쿼리를 격발할 때 드라이버 테이블 `lt_driver` 가 완전히 비어 있는(`INITIAL`) 상태에서 가동하여 WHERE 절 조건을 통째로 잃고 전체 테이블을 쌩으로 긁어 메모리 폭사 덤프를 내고, 성능 개선 요구 시 눈으로 트레이스를 뜨는 측정 시퀀스를 거치지 않은 채 "대충 이 SELECT 가 느리겠지" 라고 추측 튜닝을 때려 엉뚱한 부수 버그만 배가시킨다.
또한, S/4HANA 인메모리 장비 하에서 성능을 올린답시고 보조 인덱스(Secondary Index)를 남발해 테이블 CUD 수정 쓰기 연산 시 인덱스 재정렬 오버헤드를 극대화하는 실책을 범한다.
따라서 본 챕터는 다음과 같은 단계적 해소 장치로 설계를 배치한다.

1. **ST05 SQL Trace**: 추측 튜닝의 참상을 고발하고, Trace On/Off 시퀀스 작동법 및 중복/루프 내 SELECT 검출 가이드 장착.
2. **SAT Runtime Analysis**: Hit List 를 통해 ABAP 연산, DB 조회, System 시간 비중을 식별하고 소량 테스트 왜곡 방어를 위한 대표 데이터 바인딩.
3. **SQL Monitor (SQLM)**: 로컬 측정을 넘어 실 운영 서버의 누적 무거운 SQL 을 누계 순위화하고 SWLT 와 매칭하는 법 교육.
4. **Secondary Index 절제**: S/4HANA 에서 인덱스 설계의 제한성을 경고하고 쓰기 성능 오염을 막기 위한 측정 근거주의 주입.
5. **SELECT in LOOP 박멸 (1+N)**: 루프 내 SELECT 안티패턴을 FOR ALL ENTRIES 및 JOIN 으로 일괄 사전 조회해 내부 램 메모리 READ TABLE BINARY SEARCH 로 도킹하는 튜닝 개설.
6. **FAE 빈 테이블 가드**: 드라이버 테이블의 공백 유입 시 쿼리가 대오염되는 것을 막기 위한 **`IF lt_driver IS NOT INITIAL.`** 예약 선행 가드 의무 수립.
7. **HANA Code Pushdown**: 수백만 건 데이터를 램으로 굳이 다 퍼 올리지 않고, DB 단에서 **`SUM` 과 `GROUP BY`** 집계를 완료해 결과만 수송하는 대량 패키징 설계 확립.

## CH32-L01 - ST05 SQL Trace

### 왜 필요한가

우리가 이전 데이터베이스와 SQL 챕터에서 이쁜 테이블들을 빚고 멋지게 데이터들을 셀렉트해 왔다.
그런데 실제 현업 서비스 장비를 돌리기 시작하면 가슴이 답답해지는 고통을 호소한다.
- "예약 마감 리포트 화면을 켰는데, 모래시계 휠이 3분 동안 뱅글뱅글 돌다가 화면이 꺼져 버린다."
이때 튜닝하겠다고 소스 코드를 열어 "대충 내 육감으로 보아하니 이 20번째 줄 SELECT 쿼리가 범인일 것이다" 고 단정하고, 그 쿼리를 만지거나 인덱스를 마구 생성해 버린다.
하지만 실체 범인은 50번째 줄에 숨겨진 엉뚱한 쿼리였고, 내 추측성 튜닝으로 인해 멀쩡하던 다른 전표 로직만 파손되어 컴파일 에러를 내며 뻗는 부수 장애가 난무한다.

**뇌피셜이나 감각으로 튜닝하지 않고, DB 통신망에 스파이 도청 장치를 달듯 트레이스 전원(Trace On/Off)을 켜서 컴퓨터가 몇 밀리초 동안 어떤 쿼리를 몇 번 돌려 병목을 먹었는지 돋보기로 추적해 내는 기술**이 필요하다. 그것이 **[[ST05]] SQL Trace** 다.

### 무엇인가

#### 1. 성능 제1철칙: "추측하지 말고 측정하라"
- *성능 튜닝의 절대적인 골든 룰이다.*
- 병목의 위치를 정확한 수치 데이터로 기록하지 않은 채 진행하는 튜닝은, 성능을 개선시키지 못하고 엉뚱한 버그만 양산하는 무의미한 삽질이다.

#### 2. ST05 SQL Trace (SQL 추적기)
- 내 SAP 세션에서 발생하는 모든 데이터베이스 SQL 쿼리 송수신 로그를 1밀리초 단위로 수집해 기록하는 전용 계측 트랜잭션 화면이다.
- **Trace On** 과 **Trace Off**:
  - `Trace On` 단추를 누르는 순간부터 도청이 시작된다.
  - 내 예매 프로그램을 쌩 켜서 느린 동작을 1회 재현해 준다.
  - 즉시 `Trace Off` 를 누르고 `Display Trace` 로 로그 분석실에 들어간다.

#### 3. Identical/Redundant Selects (중복 쿼리 병목)
- *ST05 가 검출해 주는 가장 대표적인 병목 주범이다.*
- 똑같은 `SELECT SINGLE artist FROM zconcert WHERE concert_id = 'C001'` 쿼리를 내부 테이블 루프를 돌며 수천 번 똑같이 반복 조회해 네트워크 밴드위스를 폭사시키고 있는 현상이다. (한 번만 조회해 메모리에 올려두어 캐싱 처리해야 함을 경고한다.)

### 어떻게 확인하는가

ST05 Trace 를 통해 느린 병목 쿼리와 중복 쿼리를 찾아내어 개선하는 흐름을 검증한다.

```text
[1단계] ST05 계측 가동 :
   - ST05 실행 -> Trace On 클릭!
   - 다른 창에서 zbooking_report 실행 (30초 소요 후 완료됨)
   - ST05 복귀 -> Trace Off 클릭!
   - Display Trace 조회 격발!
   
[2단계] Trace 분석 목록 확인 :
   ---------------------------------------------------------------------------------
   | SQL 문장                               | 실행 횟수 | 소모 시간(ms) | 판정       |
   ---------------------------------------------------------------------------------
   | SELECT * FROM zbooking WHERE ...      | 1        | 120           | 정상       |
   | SELECT artist FROM zconcert ...        | 5,000    | 28,000        | 병목 (Redundant) |
   ---------------------------------------------------------------------------------
   -> 똑같은 zconcert 쿼리가 5,000번 반복 수행되며 28초를 다 갉아먹은 병목 원 흉 확인 완료!
```

#### 체험/시뮬레이터 설계 (ST05 트레이스 분석기)
- **프로세스 플로우**:
  1. 학습자가 [느려터진 예매 보고서 실행] 버튼을 본다.
  2. [ST05 Trace = OFF] 인 상태로 실행해 봤자, "30초 걸림" 이라는 답답한 메시지만 뜬다.
  3. [ST05 Trace = ON] 을 켜고 보고서를 구동한다.
  4. 로그 리스트 판넬이 촥 열리며 `SELECT ... FROM zconcert` 줄에 [빨간색 경고등]이 켜지고 "Identical Calls: 5,000 times! Duration: 28s" 병목 수치가 요란하게 노출되는 피드백을 감상한다.
- **상태 및 데이터**:
  - `운영 장비에서 Trace On 을 켜둔 채 퇴근하여 로그 파일이 하드 디스크를 꽉 채운 상태` -> 런타임 결과: `Server storage overflow warning. Trace file size exceed limit` 적색 사이렌 경보.
- **피드백**: 트레이스는 오직 내가 재현할 찰나에만 짧게 켜고 꺼서 격리 수집해야 안전함을 인지한다.

### 실수/주의

- **성능이 느리다고 측정 자료도 없이, 일단 Standard DB 테이블에 보조 인덱스(Secondary Index)부터 마구 생성**:
  - 인덱스를 만들면 조회는 아주 미세하게 빨라질지 몰라도, 그 테이블에 예매를 등록/수정(EML/DML)할 때마다 인덱스 나무 지도를 서버가 다시 그리느라 저장 성능이 10배 느려지는 치명적 장애를 유발한다.
  - **무조건 ST05 의 팩트 수치를 기반으로 튜닝을 기획해야 한다.**

### 정리

- 성능 개선은 뇌피셜을 거부하고 **`측정 우선 원칙`** 을 준수해야 성공한다.
- **`ST05`** 는 Trace On/Off 시퀀스를 밟아 DB 쿼리 소모 시간과 실행 횟수를 밀리초 단위로 수집한다.
- **`Identical Selects`** 는 동일 쿼리를 반복 조회하는 병목이므로 사전 일괄 조회로 개선한다.

## CH32-L02 - SAT Runtime Analysis

### 왜 필요한가

ST05 로 DB 튜닝은 완료했다.
그런데 이번에는 DB 조회가 아니라 내 아바 소스 코드 안방에서 발생하는 시간 소모가 병목이다.
- "디비 조회 쿼리는 전부 0.01초 만에 끝났는데, 정작 아바 프로그램 화면이 켜지기까지는 여전히 10초가 걸린다. 
어느 내부 테이블 정렬 로직이나, 어떤 중첩 LOOP 메서드가 시간을 몽땅 갉아먹고 있는지 눈으로 확인이 안 된다."
아바 프로그램 내벽에는 수백 개의 서브루틴과 로컬 메서드들이 얽혀 돌기 때문에, 소스 파일을 쌩 눈으로 쳐다봐서는 병목 지점을 특정할 수 없다.

**ABAP 프로그램 구동 런타임 프로파일링(SAT)을 기동하여, 어느 클래스, 어느 메서드 줄 번호에서 총 소모 시간의 80% 가 고착되어 소모되었는지 1순위 히트 리스트(Hit List)로 정밀 검출해 내는 기술**이 필요하다. 그것이 **SAT Runtime Analysis (런타임 분석)** 이다.

### 무엇인가

#### 1. SAT (Runtime Analysis)
- DB 의 병목을 넘어서, ABAP 프로그램 내부의 순수 알고리즘(Loop 연산, Internal Table 탐색, Type 형변환, SORT 등)이 소모한 런타임 시간을 백분율과 마이크로초 단위로 추적 프로파일링해 주는 고급 트랜잭션 화면이다. (구 SE30 의 진화형이다.)

#### 2. Hit List (시간 점유 히트 리스트)
- *SAT 분석의 나침반이다.*
- 실행 시간 동안 가동된 모든 서브루틴/메서드 명칭들을 "가장 오랜 누적 시간을 소모한 순서" 로 1위부터 100위까지 랭킹 리스트를 매겨주는 분석 창이다.

#### 3. ABAP / Database / System 시간 점유율 막대
- **내 프로그램이 소모한 총 시간 중 ABAP 연산 비율이 높은지, DB 조회 비율이 높은지 막대그래프로 시각화해 짚어준다. ABAP 비율이 90% 라면 내부 테이블 READ 구조를 SORTED 나 HASHED 로 스왑 튜닝해야 함을 즉각 가이드해 준다.**

### 어떻게 확인하는가

SAT 를 구동해 Hit List 1위 메서드를 추적하여 내부 테이블 READ 에 BINARY SEARCH 가 빠진 것을 잡아 튜닝하는 흐름을 검증한다.

```text
[1단계] SAT 계측 가동 :
   - SAT 실행 -> Measure 변형 지정 -> zbooking_report 실행 완료!
   - 런타임 프로파일 분석 창 개방!
   
[2단계] 시간 점유 비율 막대 확인 :
   - ABAP   : 85% (■■■■■■■■□□) -> ABAP 로직이 범인 확정!
   - DB     : 10% (■□□□□□□□□□)
   - System :  5% (□□□□□□□□□□)
   
[3단계] Hit List 랭킹 확인 :
   ---------------------------------------------------------------------------------
   | Rank | Class / Method                         | Hit Count | Net Time (ms) |
   ---------------------------------------------------------------------------------
   | 1    | LCL_REPORT=>SEARCH_BOOKING             | 10,000    | 9,800         |
   | 2    | CL_GUI_CFW=>FLUSH                      | 1         | 150           |
   ---------------------------------------------------------------------------------
   -> SEARCH_BOOKING 메서드가 10,000번 호출되며 9.8초를 독점 사용 중인 병목 확인!
   -> 소스 확인 결과, STANDARD TABLE 에 BINARY SEARCH 없이 쌩 LOOP READ 를 때린 것이 검출됨!
```

#### 체험/시뮬레이터 설계 (SAT Hit List)
- **프로세스 플로우**:
  1. 학습자가 [ABAP vs DB 비율 막대기]를 본다. 
  2. [ABAP = 90%] 에 불이 번쩍인다.
  3. [Hit List] 단추를 누르자, `READ_TABLE` 소모 시간이 압도적 1위로 노출된다.
  4. [STANDARD TABLE 쌩 검색] 소스를 [HASHED TABLE 검색] 카드로 촥 갈아 끼우고 리런한다.
  5. Hit List 1위 소모 시간이 9.8초에서 0.05초로 순식간에 수축 완료되어 전체 합격 초록등이 들어오는 피드백을 감상한다.
- **상태 및 데이터**:
  - `테스트 데이터를 단 3건만 넣고 SAT 프로파일을 가동한 상태` -> 런타임 결과: `Data size too small. Net time shown as 0ms. Bottleneck invisible` 하이라이트.
- **피드백**: 코드 프로파일링은 반드시 실무 규모의 대표 데이터(수천 건 이상)를 밀어 넣고 돌려야 병목 계측이 가능함을 각인한다.

### 실수/주의

- **SAT 자체가 컴파일 및 가동 오버헤드를 발생시켜 프로그램 속도를 평소보다 2~3배 느리게 억누름을 간과**:
  - SAT 분석 중에 "아니, 튜닝 전에도 10초 걸렸는데 측정기를 끼우니까 30초가 걸리네? 시스템 망했다" 고 착각해 비명을 지르는 입문자가 많다.
  - **수치의 '절대 시간' 은 무시하고, 각 메서드 간의 '상대적 소모 시간 백분율(%) 비중' 만 짚어보는 나침반으로 삼아야 한다.**

### 정리

- **`SAT`** 는 ABAP 소스 내부의 알고리즘과 메서드가 소모한 상대적 런타임 시간을 추적한다.
- **`Hit List`** 는 누적 소모 시간 순으로 호출 랭킹을 매겨 범인을 핀포인트로 알려준다.
- ABAP 비율이 높으면 내부 테이블 탐색을 **`SORTED/HASHED`** 나 **`BINARY SEARCH`** 로 튜닝한다.

## CH32-L03 - SQL Monitor / SQLM

### 왜 필요한가

ST05 와 SAT 로 한 개발자 세션의 튜닝은 끝마쳤다.
그런데 이번에는 "실제 운영 서버 전체 가동 시간" 동안 누적되는 진짜 실무 병목 탐색이 문제다.
- "내 개인 PC 개발 장비에서 테스트용 예약 10건으로 실행할 때는 0.01초 만에 돌아서 아주 만족스러웠다.
그런데 운영 서버에 배포되자, 전국의 수천 명 사용자가 매일 5만 번씩 이 쿼리를 호출하는 바람에, 0.01초짜리 쿼리가 하루 누적 500초의 서버 CPU 점유율을 차지해 운영 데이터베이스 램을 다 갉아먹고 기계를 기절시킨다."
내 로컬 1회성 트레이스(`ST05`)만으로는 실제 다중 유저(Multi-user)가 대량 호출할 때 발생하는 누적 오버헤드를 볼 방법이 없다.

**실제 가동 중인 운영 장비 백그라운드에 조용히 패킷 수집기(SQLM)를 상시 구동하여, 전체 트랜잭션 중 누적 호출 횟수와 누적 점유율이 가장 높은 최악의 1위 쿼리를 골라내어 secondary index 나 튜닝 순위를 획정하는 기술**이 필요하다. 그것이 **SQL Monitor / SQLM** 의 집행이다.

### 무엇인가

#### 1. SQLM (SQL Monitor)
- 개발자의 수동 1회성 측정을 넘어, 실제 가동 중인 Production(운영) 시스템 백엔드 전체에서 발생하는 모든 SQL 의 실행 빈도, 소모 시간, 처리 레코드 수를 누적 집계하여 데이터로 순위를 매겨주는 고성능 운영 모니터링 툴이다.

#### 2. SWLT (SQL Performance Tuning Worklist)
- SQLM 이 수집해 온 무거운 쿼리 정보와, 정적 코드 분석 정보를 믹스 결합하여 개발자가 즉각 튜닝해야 할 수정 코드 목록을 1:1 매칭해 보여주는 통합 작업 대장이다.

#### ⚠️ [ Secondary Index 남용 억제 및 S/4HANA 권장 제약 명세]
- *데이터베이스 쓰기 성능 폭사를 막는 인덱스 가이드라인이다.*
- SQLM 에서 특정 WHERE 쿼리가 자주 느리다고 검출되면 보조 인덱스(Secondary Index)를 달고 싶어진다.
- **오작동**: 테이블에 보조 인덱스를 5개 이상 마구 난사한다.
- **이유**: **테이블 조회(Select)는 10% 빨라질지 모르나, 예매를 등록/수정/삭제(CUD)할 때마다 5개 인덱스 트리를 백엔드가 매번 재구성하느라 쓰기 성능이 200% 폭사하는 역효과를 내기 때문이다. 특히 S/4HANA (HANA DB) 장비 하에서는 인메모리 열 압축 특성상 인덱스 없이도 고속 서치가 가능하므로 인덱스 설계를 철저히 통제 억제해야 한다.**
- **정석**: 보조 인덱스는 오직 **SQLM 데이터를 근거로 하루 실행 건수가 수십만 건 이상이며 비키 조회가 명백히 검증된 단독 1~2개 필드에만 칼로 자르듯 절제 지정**해야 안전하다.

### 어떻게 확인하는가

SQLM 모니터링 정보를 수집해 Secondary Index 추가 여부를 의사결정하는 흐름을 검증한다.

```text
[1단계] SQLM 운영 수집 데이터 스캔 :
   - SQLM 실행 -> 1주일 간의 누적 운영 집계 로드!
   ---------------------------------------------------------------------------------
   | SQL 문장 (타깃 Table)                  | 누적 호출 횟수 | 누적 소모 시간 | 판정 |
   ---------------------------------------------------------------------------------
   | SELECT * FROM zbooking WHERE cust_id   | 850,000        | 92,000s        | 최악 |
   ---------------------------------------------------------------------------------
   -> cust_id 필드에 인덱스가 없어 하루 85만 번 Full Table Scan 을 돌며 92,000초 누적 부하 발생 확인!

[2단계] 튜닝 집행 :
   - zbooking 테이블의 cust_id 필드에 Secondary Index 'Z01' 단 1개만 정교하게 절제 추가!
   - 1주일 후 SQLM 재조회 -> 누적 소모 시간이 92,000초에서 80초로 수축 완료! (DB 수명 수호)
```

#### 체험/시뮬레이터 설계 (SQLM 운영 집계판)
- **프로세스 플로우**:
  1. 학습자가 [운영 트랜잭션 집계 대시보드]를 본다.
  2. [SELECT FROM zbooking] 라인이 누적 85만 번 호출로 [CPU 과부하 1위 적색 바]로 솟구쳐 있다.
  3. [Secondary Index Z01 생성 칩] 을 테이블에 꽂는다.
  4. 조회 점유 바가 초록색 안전 영역으로 스르륵 내려앉는 모습을 확인한다.
  5. 이번엔 욕심내어 [Z02, Z03, Z04 인덱스 칩 3개 추가] 를 누른다. 조회는 불이 꺼지는데, 옆 전광판의 [예매 등록 수정 쓰기 처리 시간 바] 가 적색 경보로 폭발 상승하는 오작동 비주얼 피드백을 감상한다.
- **상태 및 데이터**:
  - `측정 근거 없이 기획 회의에서 감각으로만 인덱스를 5개 추가한 상태` -> 런타임 결과: `Database insert speed dropped by 300%. Secondary index lock contention` 하이라이트.
- **피드백**: 인덱스는 양날의 검이므로, 오직 SQLM 수치를 근거로 절제 설계해야 함을 체득한다.

### 실수/주의

- **보조 인덱스를 달았는데, 쿼리 WHERE 문에서 인덱스 필드 순서를 무시하고 임의로 배치**:
  - 보조 인덱스가 `(CONCERT_ID, BOOKING_DATE)` 복합 키 순서로 묶여 설계되었는데, 내 SELECT WHERE 절에는 `BOOKING_DATE` 만 단독으로 기재하면 인덱스가 타지 않고 무용지물이 된다.
  - **인덱스가 작동하려면 복합 키의 첫 필드(Leading Field)가 WHERE 절에 반드시 매칭 기입되어야 함을 수호해야 한다.**

### 정리

- **`SQLM`** 은 실 운영 서버 가동 시간 동안의 모든 SQL 누적 부하 랭킹을 집계한다.
- 인덱스는 쓰기 오버헤드를 동반하므로 **`HANA DB 에선 보조 인덱스 남발을 철저히 절제`** 한다.
- 보조 인덱스 추가 시에는 복합 키의 **`첫 필드 (Leading Field)`** 가 WHERE 절에 와야 작동한다.

## CH32-L04 - SELECT in LOOP 제거

### 왜 필요한가

SQLM 분석까지 완료했다.
이제 아바 성능 장애의 영원한 최악의 끝판왕이자, 초보 개발자들의 99% 가 작성해 배포하는 최악의 안티패턴인 'SELECT in LOOP (1+N)' 의 격퇴를 마주한다.
- " 10,000건의 예약 목록을 루프 돌며 출력한다.
그런데 루프 안방 내부에서 `SELECT SINGLE artist FROM zconcert WHERE concert_id = ...` 쿼리를 매 줄마다 쌩 격발시켰다."
예약이 10,000건이면 DB 와의 통신 왕복 횟수가 `1 + 10,000 = 10,001` 번 발생한다. 
디스크가 닳아 빠지고 네트워크 포트 트래픽이 폭사하며 0.1초 만에 끝날 리포트가 5분 동안 멈춰 서서 사용자를 울부짖게 만든다.

**루프 안에서 디비를 수천 번 쪼개 찌르는 안티패턴을 전면 차단하고, FOR ALL ENTRIES 나 JOIN 을 사용해 밖에서 1번에 일괄 DB 데이터를 긁어와 메모리에 올린 뒤, 2진 탐색(BINARY SEARCH)으로 결합해 0.01초 만에 연산을 종결하는 기술**이 필요하다. 그것이 **SELECT in LOOP 제거**다.

### 무엇인가

#### 1. SELECT in LOOP (1 + N 번 왕복 재앙)
- LOOP 문 내부에 SELECT 문을 기재하여, 내부 테이블의 행 수(N)만큼 데이터베이스 왕복을 무한 격발해 시스템 리소스를 파괴하는 최악의 아바 코딩 안티패턴이다.

#### 2. FOR ALL ENTRIES (FAE - 사전 일괄 수거)
- 드라이버 테이블의 모든 조건 키들을 믹스 결합해, 단 1번의 쿼리로 해당하는 타깃 DB 레코드들을 통째로 긁어 램(RAM) 내부 테이블에 올려주는 일괄 조회 구문이다.

#### ⚠️ [ FOR ALL ENTRIES 사용 시 빈 테이블 FAE 폭사 방어선 명세 ]
- *모든 아바 개발자가 입사 첫날 실수하여 서버를 뻗게 만드는 절대 마귀 영역이다.*
- `FOR ALL ENTRIES IN @lt_booking` 을 던질 때, 조건 드라이버 테이블인 `lt_booking` 이 완전히 INITIAL(비어있는) 상태로 쿼리가 날아가면, **아바 SQL 엔진은 WHERE 조건을 통째로 소멸 무시하고 대상 zconcert 테이블 전체 레코드 수백만 건을 쌩으로 다 긁어 퍼 올리는 대형 램 오버플로우 덤프**를 터트린다.
- **방어선 (빈 가드)**: **반드시 FAE SQL 문장 바로 윗줄에 `IF lt_booking IS NOT INITIAL.` 가드를 둘러 쳐서, 드라이버 테이블에 데이터가 최소 1건 이상 심어져 있을 때만 쿼리를 타도록 가로막는 것이 철칙 중의 절대 철칙이다.**

#### 3. READ TABLE ... BINARY SEARCH (2진 탐색 결합)
- FAE 로 긁어온 캐시 테이블(`lt_concert`)을 키 순서대로 **`SORT`** 정렬해 두고, 루프 안방에서는 디비 쿼리 대신 **`BINARY SEARCH`** 배지를 달아 램에서 초고속 2진 서치로 데이터를 매칭해 엮어내는 기법이다. (조회 횟수를 단 1번으로 단축한다.)

### 어떻게 확인하는가

SELECT in LOOP 안티패턴을 FAE 와 BINARY SEARCH 로 튜닝해 1만 번 조회를 1번으로 감축하는 코드를 검증한다.

```abap
" ❌ [ 안티패턴 - LOOP 내 SELECT 난사로 DB 10,000번 폭사 유발 ]
LOOP AT lt_booking INTO DATA(ls_book).
  SELECT SINGLE artist FROM zconcert 
    WHERE concert_id = @ls_book-concert_id INTO @DATA(lv_artist). " N 번 왕복!
ENDLOOP.
```

```abap
" ✅ [ 튜닝 완료 - 밖에서 단 1번으로 긁어오고 램에서 2진 탐색 매칭 ]

" 1. [★ 드라이버 테이블 빈 가드 철칙 장착!]
IF lt_booking IS NOT INITIAL.

  " 2. [FOR ALL ENTRIES 로 10,000건의 타깃을 단 1번에 일괄 긁어오기!]
  SELECT concert_id, artist FROM zconcert
    FOR ALL ENTRIES IN @lt_booking
    WHERE concert_id = @lt_booking-concert_id
    INTO TABLE @DATA(lt_concert_cache). " 램 캐시 테이블에 장착!
    
  " 3. [★ 2진 탐색을 위한 SORT 정렬 지정!]
  SORT lt_concert_cache BY concert_id.
  
ENDIF.

" 4. [메인 루프 가동 - 루프 안에는 SELECT 가 0줄!]
LOOP AT lt_booking INTO DATA(ls_book_real).
  " 5. [BINARY SEARCH 2진 탐색으로 0.0001초 만에 램에서 매칭 완료!]
  READ TABLE lt_concert_cache INTO DATA(ls_cache) 
       WITH KEY concert_id = ls_book_real-concert_id
       BINARY SEARCH.
       
  IF sy-subrc = 0.
    " 연산 수행...
  ENDIF.
ENDLOOP.
```

#### 체험/시뮬레이터 설계 (SELECT in LOOP 튜닝 시나리오)
- **프로세스 플로우**:
  1. 학습자가 [예매 건수 N = 10,000] 슬라이더를 본다.
  2. [안티패턴 모드] 로 구동한다. [DB 접근 게이지] 가 10,000번 깜빡깜빡 왕복하며 기계가 과열되어 "처리 속도: 28초" 빨간불이 터진다.
  3. [FAE + BINARY 튜닝 모드] 로 전환한다.
  4. [DB 접근 게이지] 가 딱 "1번" 지잉 긁어오고 멈춘 뒤, 아바 루프 안에서 2진 탐색 기어가 돌아가며 "처리 속도: 0.02초" 초록색 합격 전광판이 반짝이는 카타르시스 피드백을 감상한다.
- **상태 및 데이터**:
  - `FAE IN @lt_booking 을 때리기 전 IS NOT INITIAL 검사를 생략한 상태` -> 런타임 결과: `INITIAL driver table! Full table scan executed. Out of Memory Crash` 덤프 번개 작동.
- **피드백**: FAE 를 칠 때는 빈 테이블 가드를 치고, 캐시 테이블은 `SORT` 후 `BINARY SEARCH` 가 삼위일체로 맞물려야 튜닝이 성립함을 배운다.

### 실수/주의

- **FAE 로 캐시 테이블을 긁어왔는데, SORT 정렬을 생략하고 READ TABLE ... BINARY SEARCH 를 격발**:
  - 정렬이 안 된 테이블에 대고 2진 탐색(BINARY SEARCH)을 날리면, 아바 엔진은 엉뚱한 줄을 집어 올린 뒤 "값을 못 찾았다" 며 `sy-subrc = 4` 로 퉁치고 스킵하여 결제 데이터 전체가 누락되는 시각적 공백 버그를 야기한다.
  - **BINARY SEARCH 뒤에는 반드시 상응하는 SORT 정렬을 수호해야 한다.**

### 정리

- 루프 내에 SELECT 를 때리는 **`SELECT in LOOP`** 는 시스템을 파괴하는 주범이다.
- FAE 를 칠 때는 빈 테이블 오염 방지를 위해 무조건 **`IS NOT INITIAL`** 가드를 두른다.
- 긁어온 테이블은 반드시 **`SORT`** 정렬을 단행한 뒤 **`BINARY SEARCH`** 로 2진 탐색 매칭한다.

## CH32-L05 - 대량 데이터 처리와 Package 설계

### 왜 필요한가

SELECT in LOOP 제거까지 완료했다.
이제 성능 튜닝의 최종 종착역인 '대량 데이터 패키징 처리' 와 'HANA DB 엔진으로의 계산 위임(Code Pushdown)' 을 마주한다.
- " 100만 건의 예매 금액 합계를 구하려 한다.
기존 방식으로는 `SELECT * FROM zbooking INTO TABLE @lt_all.` 로 100만 건 레코드를 몽땅 아바 램(RAM)으로 퍼 올려 가져온 뒤, `LOOP AT lt_all` 을 돌려 한 줄씩 금액을 더해 전광판에 뿌렸다."
100만 건 바이너리가 램에 얹히는 순간 메모리 한도 초과(Page Allocation Failed)로 아바 WAS 가 즉사하고, 왕복 전송 트래픽으로 네트워크 허브에 불이 난다.
굳이 아바 램으로 무거운 데이터를 다 퍼 와서 LOOP 고생을 자행하지 않고, **"HANA DB 안방 엔진에게 SUM 과 GROUP BY 연산을 지시해 연산 결과 단 1줄만 수송받는 기술"** 이 필요하다. 그것이 **Code Pushdown (코드 푸시다운)** 이다.

### 무엇인가

#### 1. Code Pushdown (코드 푸시다운)
- *Modern S/4HANA 성능 설계의 핵심 철학이다.*
- **"데이터를 ABAP 램으로 가져와 계산하지 말고(Data to Code), 계산 연산 로직을 데이터가 잠들어 있는 DB 내부로 내려보내 집계가 끝난 정산 엑기스만 수송받아라(Code to Data)"** 는 원칙이다.

#### 2. SUM( seatsocc ) AS occ GROUP BY carrid
- *Code Pushdown 을 실현하는 표준 Open SQL 문법이다.*
- DB 가 원래 제일 잘하는 '집계(SUM)', '정렬(ORDER BY)', '필터(GROUP BY)' 를 DB 단에서 고속 실행 완료시켜 넘겨받음으로써 아바 Loop 부하를 0줄로 수축시킨다.

#### 3. Package Processing (패키징 분할 처리)
- 100만 건 데이터 DML(저장/수정)이 수반될 때, 이들을 한 번에 커밋하려다 락 테이블이 꽉 차 터지는 것을 막기 위해, `UP TO 5000 ROWS` 나 패키징 스트리밍을 통해 5,000건 단위로 쪼개어 가공하고 루프 꼬리마다 `COMMIT WORK` 로 락을 풀어 메모리를 소생시키는 분할 점령(Divide and Conquer) 설계 기술이다.

### 어떻게 확인하는가

ABAP Loop 집계 방식과 DB Pushdown GROUP BY 방식의 물리 성능 차이를 검증한다.

```abap
" ❌ [ 레거시 느린 방식 - 100만 건을 램으로 다 퍼 올려 Loop 연산 ]
SELECT seatsocc FROM sflight INTO TABLE @DATA(lt_flight).
LOOP AT lt_flight INTO DATA(ls_flg).
  gv_total = gv_total + ls_flg-seatsocc. " 100만 번 아바 CPU 낭비!
ENDLOOP.
```

```abap
" ✅ [ Modern 튜닝 방식 - DB 단에 집계를 위임하고 결과 1줄만 수령! ]
" 아바 Loop 가 단 0줄! 네트워크 트래픽 100만 배 감소!
SELECT SUM( seatsocc ) AS occ 
  FROM sflight 
  INTO @DATA(lv_total_db). " DB 에서 고속 집계가 완료된 엑기스 단 1줄만 획득!
```

#### 체험/시뮬레이터 설계 (Code Pushdown vs ABAP Loop)
- **프로세스 플로우**:
  1. 학습자가 [대량 데이터 N = 1,000,000건] 라디오 버튼을 본다.
  2. [ABAP Loop 집계 모드] 를 누른다. 100만 개 상자가 트럭에 실려 아바 서버로 덜덜덜덜 배달되어 와서, 아바 일꾼이 100만 번 상자를 까 더하느라 "소모 시간: 18초, 램 사용량: 80%" 경고등이 켜진다.
  3. [Code Pushdown 모드] 를 지른다. DB 공장 안에서 프레스 기계가 1초 만에 100만 개 상자를 단 1개의 팩으로 압축 완료하고, 아바 서버에는 그 압축 팩 1개만 쓱 배달되어 "소소 시간: 0.05초, 램 사용량: 0.1%" 청정 상태 피드백이 연출되는 모습을 감상한다.
- **상태 및 데이터**:
  - `수백만 건 벌크 데이터를 패키징 분할 없이 한 번에 MODIFY TABLE 하려 든 상태` -> 런타임 결과: `Lock table overflow. Rollback executed` 적색 덤프 작동.
- **피드백**: 대량 처리 시에는 DB GROUP BY 와 패키징 분할(`UP TO ROWS`)의 가치를 이해하고 배치해야 함을 확인한다.

### 실수/주의

- **HANA DB 가 인메모리 엔진이라 모든 걸 다 알아서 해줄 거라 믿고, 1억 건 벌크 테이블을 아무 분할 패키징 없이 쌩 SELECT SINGLE 난사**:
  - HANA DB 가 우주선 속도일지라도 1억 건 벌크 메모리가 아바 서버 통신 대역을 타는 순간 네트워크 대역폭 병목에 걸려 물리적으로 즉사한다.
  - **대량 연산 시에는 무조건 패키징 분할과 집계 위임을 소스에 기재해 수호해야 한다.**

### 정리

- 대량 연산 시에는 **`Code to Data`** 철학에 따라 로직 연산을 DB 단으로 내려보낸다 (Code Pushdown).
- DB 가 연산하는 **`SUM / GROUP BY`** SQL 문법을 적극 채택해 아바 Loop 점유율을 0줄로 지운다.
- 대량 DML 시에는 락 해제를 위해 **`Package 분할 처리`** 와 5,000건 커밋 셋업을 장착한다.
- *참고로 이 SQL 튜닝 기술은 classic/modern 공통이며, 다음 장에 배울 AMDP/CDS 에 이 원리가 탑재된다.*
