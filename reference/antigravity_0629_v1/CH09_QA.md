## 3. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. SE11 에서 테이블 필드에 Foreign Key 를 걸면, 실제 Oracle 이나 HANA 데이터베이스 레벨에 외래키(Referential Integrity Constraint) 물리 제약이 정말 생성되나요?
**A**: **아니오, 아바의 Foreign Key 는 DB 단에 물리적인 외래키 제약을 만들지 않으며, 오직 SAP GUI Dynpro PAI 화면이나 TMG(유지보수 SM30) 세션에서 아바 커널이 선제적으로 입력 값을 마스터 테이블과 조회 대조하는 Application-level Check 방식으로 작동합니다.**
- **외래키 물리 동작의 실체**:
  타 정통 전산 환경(Java, C# 등)에서는 DB 에 외래키를 걸면 DBMS 엔진이 디바이스 레벨에서 락을 채워 무결성을 지킵니다. 
  그러나 SAP 은 성능 극대화와 유연한 형상 관리를 위해, 데이터베이스 자체에 외래키 제약을 선포하지 않습니다. 
  외래키 검증은 오직 사용자 입력 화면인 **`프레젠테이션/애플리케이션 레이어`** 에서 아바 커널이 가볍게 작동시킵니다. 
  - **직접 쓰기 우회 함정**:
    이로 인해, 프로그램 소스 상에서 화면을 거치지 않고 Open SQL DML 구문인 **`INSERT zbooking FROM gs_booking.`** 을 다이렉트로 날려 저장할 때는 이 외래키 검문 필터가 100% 우회(Bypass)됩니다. 
    체크 테이블에 없는 유령 공연 ID `'C999'` 가 ZBOOKING DB 테이블 장부에 아무 경고 없이 인서트 적재되는 무결성 파손이 터집니다. 
    **철칙**: 프로그램 단에서 DB 직접 쓰기(DML)를 기동하기 직전에는, 반드시 소스 상에서 `SELECT SINGLE` 을 쏘아 마스터 테이블에 해당 키가 실재하는지 수동 존재 검증을 병행 수호해야 안전합니다.

### Q2. 도메인(Domain)을 빚을 때 적어주는 Value Table 과, 테이블 필드에 지정하는 Check Table 은 아바 컴파일러 사전 관점에서 구체적으로 어떤 기작 차이가 있나요?
**A**: **Value Table 은 도메인 사전 대장(DD01L)에 기록되는 단순한 선언적 '제안' 이어서 자체적으로는 입력 검증을 단 1% 도 하지 못하지만, Check Table 은 필드 대장(DD03L)에 외래키 관계가 최종 등록 활성화된 상태여서 PAI 입력 거부와 F4 도움말을 강제 기동합니다.**
- **DD01L vs DD03L 결선 기작**:
  - **`Value Table` (도메인 제안 - DD01L)**:
    도메인 `ZDO_CONCERT_ID` 를 빚으며 Value Table 단에 `ZCONCERT` 를 적어 올렸습니다. 이 정보는 도메인 명세서 테이블 **`DD01L`** 에 단순 텍스트로 박힙니다. 
    이 도메인을 상속받은 테이블 필드 `CONCERT_ID` 에 사용자가 아무 쓰레기 값이나 쳐 넣어도 시스템은 생 통과시킵니다. 
    Value Table 은 단지 설계도를 빚을 때 시스템이 "어? 이 필드 도메인에 ZCONCERT 가 Value Table 로 적혀 있네? 그럼 이 필드의 외래키 체크 테이블도 ZCONCERT 로 도킹할 거지?" 하고 팝업 창에 자동으로 기본 제안을 띄워주는 **설계 도우미**에 불과합니다.
  - **`Check Table` (필드 강제 - DD03L)**:
    SE11 테이블 편집기에서 외래키 버튼을 누르고 필드 간 1:1 선을 연결해 활성화한 상태입니다. 
    이 관계 정보는 필드 명세 테이블 **`DD03L`** 에 외래키 구조체 디스크립터(Descriptor)로 주입 적재됩니다. 
    이 단계가 체결되어야만 화면 PAI 단에서 없는 키 입력을 차단하고 F4 도움말 창을 띄우는 **실제 물리 락**이 기동됩니다.

### Q3. 사번이나 공연 ID 도메인에 적용되는 ALPHA 변환 루틴은 컴퓨터 백엔드에서 구체적으로 어떤 펑션 모듈들을 호출해 앞자리에 0 을 밀어 넣나요?
**A**: **화면 입력을 내부 저장용 10자리 0 채움 값으로 변환하는 `CONVERSION_EXIT_ALPHA_INPUT` 과, 내부의 0 채움 값을 화면 표시용 0 제거 값으로 변환하는 `CONVERSION_EXIT_ALPHA_OUTPUT` 펑션 모듈이 아바 VM 에 의해 자동 호출됩니다.**
- **Conversion Exit ALPHA 의 물리 기작**:
  사용자가 화면 입력 칸에 사번 `'42'` 를 치고 Enter 를 누르면, 아바 가상머신은 도메인의 ALPHA 링크를 읽어 즉각 **`CONVERSION_EXIT_ALPHA_INPUT`** 을 내부 격발합니다. 
  이 펑션은 왼쪽 빈칸 바이트를 루프 돌며 `'0'` 문자열을 채워 넣어 **`'0000000042'`** (10자리 고정폭) 내부 형식 값으로 램 세션 데이터를 패딩 치환해 DB 에 보냅니다.
  반대로 DB 에서 `'0000000042'` 를 읽어와 화면에 뿌릴 때는 **`CONVERSION_EXIT_ALPHA_OUTPUT`** 이 기동되어 좌측의 0 들을 싹 소거(Trim)하고 깔끔한 `'42'` 만 사용자 화면에 렌더링합니다.
  - **조회 실패 함정**: 
    개발자가 Open SQL 쿼리 조건식에 쌩 문자로 **`WHERE emp_id = '42'`** 라고 때리면, DB 장부에 박혀 있는 물리 비트열인 **`'0000000042'`** 와 매칭되지 않아 데이터가 0건 조회 누락되는 참사를 초래합니다. 
    따라서 쿼리 조건 우변 변수도 사전에 INPUT 펑션 모듈로 0 패딩을 쳐서 던져야 정합성이 수호됩니다.

### Q4. 텍스트 테이블 ZCONCERT_T 를 빚을 때 언어 키 SPRAS 를 빼먹으면 다국어 처리 화면에서 어떤 물리적 데이터 훼손이 터지며, 로그인 언어와 어떻게 조인되나요?
**A**: **언어 키가 빠지면 마스터 1건당 단 1개 언어의 명칭만 독점 저장되어 다국어 공존이 원천 붕괴되며, 언어 키가 존재할 때는 아바 커널이 로그인 세션 언어 레지스터 `sy-langu` 를 낚아채 `WHERE SPRAS = sy-langu` 조인을 묵시적으로 자동 처리해 줍니다.**
- **다국어 조인 정합성**:
  공연 ID `C001` 에 대고 한국어명 '클래식' 과 영어명 'Classic' 을 둘 다 보존하려면, 키 구조가 **`CONCERT_ID + SPRAS`** 복합 키여야만 중복 락에 걸리지 않고 2행 적재가 성립됩니다. 
  만약 `SPRAS` 키를 빼먹으면 C001 키가 중복되어 두 번째 언어 저장 시 Primary Key Constraint 락이 걸려 튕깁니다.
  - **sy-langu 자동 필터링**:
    사용자가 한국어(KO) 세션으로 로그인하면 전역 변수 `sy-langu` 에는 `'3'` (한국어)이 탑재됩니다. 
    F4 헬프 뷰어나 사전 엔진이 ZCONCERT-CONCERT_ID 를 F4 팝업으로 그릴 때, 커널은 ZCONCERT_T 테이블과 조인 쿼리를 날리며 **`AND SPRAS = sy-langu`** 조건절을 묵시적 자동으로 쿼리 끝에 붙여 쏩니다. 
    이로 인해 100% 모국어 텍스트만 엄선 상속되어 다국어가 구현됩니다.

### Q5. 화면의 한 입력 필드에 Search Help 도 걸려 있고, Domain 고정값도 있고, Check Table 도 매핑되어 있다면 F4 를 누를 때 어떤 도움말이 최종 당선되어 작동하나요?
**A**: **아바 커널의 6단계 호출 우선순위 사다리 게이팅 룰에 의해, 가장 상위 레벨인 'Search Help' 가 최하위인 'Check Table' 및 '고정값' 을 쌩 오버라이트(Override) 밀어내어 최종 팝업창으로 기동 당선됩니다.**
- **6단계 F4 Input Help 우선순위 사다리**:
  1. **1단계**: 프로그램 이벤트 단의 수동 코딩 (`AT SELECTION-SCREEN ON VALUE-REQUEST` / `PROCESS ON VALUE-REQUEST` 서브루틴). 존재 시 무조건 1순위 강제 독점.
  2. **2단계**: 입력 필드PARAMETERS 선언 우측에 직접 지정한 **`MATCHCODE OBJECT`** 검색 도움말.
  3. **3단계**: 데이터 엘리먼트(Data Element)나 테이블 필드 정의에 꽂아둔 **`Search Help`**.
  4. **4단계**: Foreign Key 릴레이션으로 맺어진 **`Check Table`** 마스터 키 리스트.
  5. **5단계**: 도메인(Domain) 정의단에 콕 박아둔 고정값 목록 (**`Fixed Values`**).
  6. **6단계**: 타입 자체 속성 기반 **`기본 도움말`** (D 타입이면 달력, T 면 시계 등).
  - **결론**: 
    3순위인 Search Help 가 걸려 있다면, 4순위인 Check Table 이나 5순위인 도메인 고정값은 순위 밀림 기각되어 F4 화면에 전혀 나타나지 않습니다. 
    도움말 설계 시에는 이 우선순위 사다리를 짚어, 원하는 단 하나의 결선 장치만 명확히 도킹해 두는 것이 충돌 방지의 정석입니다.
