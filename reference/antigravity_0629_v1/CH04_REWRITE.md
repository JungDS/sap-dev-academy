# CH04_REWRITE - 연산자와 흐름 제어 v1

> 목적: `content/abap/CH04`의 8개 레슨을 "IT 비전공자 입문자가 왜 필요한지부터 실습 확인까지 직관적으로 이해하고 따라갈 수 있는 완성 강의자료" 수준으로 재작성한 기준 원고다. 이 문서는 아직 `content/abap` 원본을 덮어쓴 결과가 아니라, 재작업 판정 이후 실제 반영 전 검수 가능한 v1 기준안이다.

## CH04 전체 설계

CH04의 한 문장 목표는 "Zero Division(`CX_SY_ZERODIVIDE`) 덤프 발생과 PAI/런타임 예외 필터링, CONCATENATE 와 `&&` 연산자의 메모리 버퍼 할당 및 오버헤드 차이, IF/CASE 조건식의 해시 점프 테이블(Jump Table) O(1) vs re-evaluation O(N) 기작, DO/WHILE 중첩 루프의 `sy-index` 덮어쓰기 메모리 백업 규칙, 그리고 `ST22` 덤프 관제 시스템의 `SNAP` 테이블 적재 구조를 명세하여 제어 조작 능력을 수립한다"이다.

IT 비전공자 입문자는 루프 내에서 소수점 `/` 이나 MOD 연산 시 분모가 `0` 이 되는 조건을 방치해 `CX_SY_ZERODIVIDE` 커널 덤프로 프로그램을 폭사시키고, 대량 정산 반복문 내에서 무분별한 `&&` 힙 임시객체 재할당 남발로 가비지 컬렉터 성능 저하를 자초한다.
또한, 중첩 DO 문 실행 시 `sy-index` 시스템 레지스터 전대전 덮어쓰기 특성을 유실해 바깥 루프의 인덱스를 영구 분실하고, 디버깅 시 BREAK-POINT 잔재를 운영으로 배포하는 형상 오염을 저지른다.
따라서 본 챕터는 다음과 같은 단계적 해소 장치로 설계를 배치한다.

1. **Zero Division 커널 가드**: 분모가 `0` 일 때 런타임 뇌사를 일으키는 연산을 차단하는 `IF 분모 <> 0` 선제 수문장 필터 제공.
2. **CONCATENATE 고속 복제**: 임시 String 힙 메모리 객체를 생성하지 않고 고정 C 런타임 카피로 동작해 루프 내 오버헤드가 없는 CONCATENATE 활용.
3. **CASE Hash Jump Table**: 단일 변수 분기 시 CPU 의 불리언 재평가를 배제하고 O(1) 해시 점프 테이블로 고속 점프하는 CASE WHEN 사용법 교육.
4. **sy-index 로컬 캐싱**: 중첩 루프에서 안쪽 인덱스 충돌로 뭉개지는 것을 막는 바깥 `gv_dan = sy-index` 로컬 변수 백업 규칙 장착.
5. **EXIT / CONTINUE / CHECK**: 즉시 완전 탈출(EXIT)과 이번 회차 건너뛰기(CONTINUE/CHECK)의 제어 흐름 분기를 명확히 도식화.
6. **ST22 SNAP 덤프 복구**: 프로그램 비정상 종료 시 시스템 테이블 `SNAP` 에 원인 코드가 적재되는 덤프 분석 및 Watch Point 실무 디버깅 활용법 명세.

## CH04-L01 - 산술 연산과 대입 · 날짜 산술

### 왜 필요한가

우리가 이전 CH03 에서 PARAMETERS 로 사용자로부터 숫자를 전역 입력받는 것까지 완료했다.
이제 입력받은 금액에 수수료율을 곱하거나, 10만 건 결산 데이터의 정수 몫과 나머지를 계산하는 '수학 연산' 의 통제가 다음 장벽이다.
- " 두 변수의 값을 사칙연산하거나 제곱 연산하는데, 계산 순서가 복잡할 때 어떻게 괄호 제어를 하는가? 
그리고 계산 도중, 나누는 분모가 `0` 이 되어 프로그램이 기절해 뻗어버리는 장애는 왜 터지고 어떻게 막는가?"
나눗셈 연산자 `/` 나 몫 `DIV`, 나머지 `MOD` 를 쏠 때, 계산 흐름 상 분모 값이 돌발적으로 `0` 이 되어 들어오면 아바 엔진은 구문 수행을 포기하고 ST22 덤프 화면을 뿜으며 프로그램을 뇌사 정지시킨다.

**분모 0 연산 시 발생하는 catchable 커널 예외 `CX_SY_ZERODIVIDE` 에러의 물리적 폭사 원인을 규명하고, 연산자 앞뒤 공백 의무화 및 날짜 D/T 타입의 특수 윤달 달력 산술 정합성을 수호하는 기술**이 필요하다. 그것이 **산술 연산과 대입**의 완수다.

### 무엇인가

#### 1. Zero Division (0 나눗셈 폭사 - CX_SY_ZERODIVIDE)
- **커널 덤프**: **나눗셈 연산 시 우변 분모 값(`gv_y`)에 `0` 이 들어오면, 아바 런타임 가상머신(VM)은 즉각 연산 장치를 정지시키고 `CX_SY_ZERODIVIDE` 라는 시스템 예외(Exception)를 방출한다. 이 예외를 가드로 감싸 거르지 않으면, 프로그램은 SNAP 시스템 테이블에 로그를 찍고 런타임 덤프(Short Dump)로 즉사 셧다운된다.**
- **방어**: 나눗셈을 실행하기 바로 앞줄에 반드시 `IF gv_y <> 0.` (0이 아닐 때만 나눈다) 조건식 게이트를 세워 안전을 사수해야 한다.

#### 2. 연산자 및 괄호 공백 선고 규칙
- *컴파일러 토큰 해석의 아바 고유 제약이다.*
- 아바는 모든 단어를 공백으로 찢어 토큰화한다. 
  `7+3` 이라고 적으면 컴파일러는 '+ 기호' 가 아니라 '7+3' 이라는 통째 이름의 변수를 찾으러 가다 에러를 낸다. 
  따라서 반드시 **`7 + 3`** 처럼 공백을 사방에 두어야 하며, 괄호 역시 **`( 2 + 3 )`** 으로 안팎을 띄워 기재해야 빌드가 통과된다.

#### 3. 날짜 D / 시간 T 타입의 달력 산술 (Calendar Arithmetic)
- 날짜 `D` 타입(8바이트 고정문자)에 `+ 1` 을 더하면, 16진수 연산으로 `'20260132'` 가 되는 것이 아니라, 윤달과 월말 일수를 아바 커널이 계산해 **`'20260201'` (2월 1일)** 로 윤년/윤달을 자동 월차 계산하여 다음 날짜로 정밀 이동시킨다.
  - **날짜 - 날짜**: 두 날짜 사이의 간격 일수(정수 `I`)를 뱉어낸다.

#### 4. CLEAR 변수 (메모리 버퍼 제로 소거)
- 변수 안의 램 메모리를 날려 초기 상태로 복구한다. 숫자는 `0`, 문자는 `공백` 으로 Roll Area 바이트가 클리어된다. (구식 `MOVE` 는 더 쓰지 않고 `=` 대입이 표준이다.)

### 어떻게 확인하는가

0 나눗셈 가드 IF 문과 uline 가로선, uline uline uline 연산자가 장착된 산술 코드를 검증한다.

```abap
REPORT zhello_arithmetic.

DATA: gv_x TYPE p LENGTH 8 DECIMALS 2,
      gv_y TYPE p LENGTH 8 DECIMALS 2 VALUE '0.00',
      gv_r TYPE p LENGTH 8 DECIMALS 2.

DATA: gv_date TYPE d VALUE '20260228', " 2026년 2월 28일 (평년)
      gv_next TYPE d.

" 1. [★ 중요: 0 나눗셈 런타임 폭사를 차단하기 위해, 분모가 0이 아닐 때만 나눔!]
IF gv_y <> 0.
  gv_r = 100 / gv_y.
ELSE.
  gv_r = 0. " 0 나눗셈 예외 우회 처리!
ENDIF.

" 2. [날짜 특수 달력 산술 기동]
gv_next = gv_date + 1. " 평년 2월 28일 뒤이므로 3월 1일인 '20260301' 로 윤달 이동 완료!

" 3. [DIV/MOD 정수 몫과 나머지 연산 격발 - 공백 필수 수호!]
DATA(gv_q) = 17 DIV 5. " 3
DATA(gv_mod) = 17 MOD 5. " 2

WRITE: / '나눗셈 결과:', gv_r,
       / '윤달 계산 다음날:', gv_next,
       / '17 DIV 5 몫:', gv_q,
       / '17 MOD 5 나머지:', gv_mod.
```

#### 체험/시뮬레이터 설계 (Zero Division 덤프 차단기)
- **프로세스 플로우**:
  1. 학습자가 [나눗셈 연산 기어] 와 [분모 y = 0 칩] 을 본다.
  2. [y = 0 칩] 을 꽂고 [0 검문 IF 가드 = OFF] 상태에서 가동을 누른다.
  3. 번개 소리와 함께 [CX_SY_ZERODIVIDE 덤프] 가 터지며 기판 전체가 빨간색으로 녹아내린다.
  4. [0 검문 IF 가드 = ON] 스위치를 켜자, y=0 임이 감지되어 연산 기어 작동이 멈추고 안전 우회 초록등이 들어오는 예외 복구 피드백을 감상한다.
- **상태 및 데이터**:
  - `연산자 7+3 양옆의 공백을 생략하고 쌩으로 붙여 쓴 상태` -> 런타임 결과: `Syntax error. The name "7+3" is unknown` 하이라이트.
- **피드백**: 나누기 분모 `0` 유입은 전산 셧다운을 내는 치명적 행위이며, 반드시 IF 차단막이 선행되어야 함을 체득한다.

### 실수/주의

- **구식 문장형 연산문인 ADD 10 TO gv_n 이나 DIVIDE gv_n BY 4 를 모던 계산식과 혼용해 남발**:
  - 문장형 연산은 읽기에는 직관적일지 모르나 복잡한 괄호 수식을 조율할 수 없고 코드 라인만 길어지므로, 실무에선 단순 누적(`+=` 대용) 이외에는 **`gv_n = gv_n + 10`** 과 같은 수식어 대입 방식을 표준 수호해 쓴다.

### 정리

- 분모가 `0` 일 때 나눗셈을 실행하면 **`CX_SY_ZERODIVIDE`** 커널 덤프로 폭사한다.
- 모든 산술 연산자(`+ - * / DIV MOD`)와 괄호 `( )` 안팎에는 반드시 **`공백을 띄워`** 선고해야 한다.
- 날짜 `D` 와 시간 `T` 는 10진 연산이 아닌 **`달력 윤달 특수 산술`** 이 적용되어 하루/1초를 계산한다.

## CH04-L02 - 문자열 다루기

### 왜 필요한가

산술 계산 제어를 완수했다.
이제 예매자 성명 결합이나, ID 코드 쪼개기, 민감 정보 REPLACE 같은 '문자열 연산' 의 성능 제어가 다음 장벽이다.
- " 문자열을 엮을 때 `CONCATENATE` 문을 쓰는 것과 모던 `&&` 연산자를 쓰는 것은 백엔드 메모리 힙(Heap) 오버헤드에서 어떤 물리적 성능 차이를 유발하는가?"
10만 건 대량 청산 데이터 루프 속에서 `&&` 연산자를 무분별하게 남발해 문자열을 엮으면, 매 루프마다 램 메모리 힙 영역에 임시 String 객체가 새로 할당되고 수거(Garbage Collection)되는 오버헤드가 누적되어, 연산 속도가 기어가다 타임아웃으로 뻗어버리는 성능 장애를 야기한다.

**CONCATENATE 의 C 런타임 고속 메모리 복제 원리와, `&&` 의 가비지 할당 비용 차이를 이해하고, SPLIT/FIND/REPLACE 및 strlen 내장함수를 기동하는 기술**이 필요하다. 그것이 **문자열 제어**의 완수다.

### 무엇인가

#### 1. CONCATENATE vs && (문자열 결합의 메모리 성능 분석)
- **CONCATENATE ... INTO ... SEPARATED BY (C 런타임 카피)**: **메모리 상에 별도의 임시 객체를 가설하지 않고, 결합할 문자 데이터의 바이트 크기를 계산해 타깃 변수 버퍼로 다이렉트 고속 전송(C-Level Memory Copy)한다. 루프 내에서 가비지 컬렉터 부하가 없어 압도적으로 고속 가동된다.**
- **`&&` (가변 String Chaining)**: **사용하기는 간편하나, 컴파일러가 내부적으로 힙(Heap) 메모리에 임시 String 인스턴스를 하나 빚어서 두 글자를 엮어 담은 뒤, 이를 최종 변수에 넘기고 임시 인스턴스를 파괴하는 가비지 회수 사이클을 유발한다. (루프 밖 가벼운 작업에 한해 절제 사용하는 것이 정석이다.)**

#### 2. SPLIT gv_text AT ':' INTO gv_a gv_b (문자열 쪼개기)
- 특정 구분자(`AT`)를 기준으로 글자 뼈대를 칼로 절단하여, 쪼개진 바이트 덩어리들을 준비된 복수의 수신 변수에 차례대로 쑤셔 넣어 주는 지시어다.

#### 3. FIND 와 REPLACE
- **FIND**: 대상 문자열 내에 내가 찾고자 하는 텍스트가 존재하는지 스캔하여, 찾으면 성공 징표인 **`sy-subrc = 0`** 을 세팅하고 못 찾으면 `4` 를 세팅해 준다.
- **REPLACE**: 특정 텍스트 구간을 도려내고 새로운 문자로 촥 갈아 끼운다. (개인정보 마스킹에 유용하다.)

#### 4. strlen( string ) (글자 수 내장함수)
- 괄호 안에 대상 문자를 던지면, 런타임 엔진이 글자 수를 세서 정수 결과값을 찰딱 반환해 주는 표준 내장함수다.

### 어떻게 확인하는가

CONCATENATE 와 && 성능 기작 및 SPLIT, strlen 내장함수가 결합된 코드를 검증한다.

```abap
REPORT zhello_string.

DATA: gv_1    TYPE string VALUE '정훈영',
      gv_2    TYPE string VALUE '대리',
      gv_dest TYPE string.

DATA: gv_csv  TYPE string VALUE 'KR:SEOUL:CONCERT',
      gv_c1   TYPE string,
      gv_c2   TYPE string,
      gv_c3   TYPE string.

" 1. [★ 성능 사수: 대량 루프를 탄다고 가정 시, 힙 오버헤드가 없는 CONCATENATE 기용!]
CONCATENATE gv_1 gv_2 INTO gv_dest SEPARATED BY space. " '정훈영 대리'

" 2. [가벼운 라벨 결합은 모던 && 연산자 기용 (공백 띄우기 필수!)]
DATA(gv_label) = '직급:' && gv_2. " '직급:대리'

" 3. [SPLIT 으로 CSV 문자 3조각 쪼개기]
SPLIT gv_csv AT ':' INTO gv_c1 gv_c2 gv_c3. " gv_c1='KR', gv_c2='SEOUL', gv_c3='CONCERT'

" 4. [strlen 내장함수로 글자 수 판정]
DATA(gv_len) = strlen( gv_c2 ). " 'SEOUL' 은 5글자!

WRITE: / '결합 결과:', gv_dest,
       / '라벨 결과:', gv_label,
       / '쪼개진 c2:', gv_c2,
       / '글자 길이:', gv_len.
```

#### 체험/시뮬레이터 설계 (CONCATENATE 메모리 수송선)
- **프로세스 플로우**:
  1. 학습자가 [힙 Heap 메모리 기지] 와 [변수 목적지 포트] 를 본다.
  2. [&& 연산자 칩] 을 구동하자, 힙 기지에서 임시 String 박스가 촥 생성되어 엮인 후 목적지 포트에 복사되고 박스가 붉게 터지며 가비지 폐기물 연기가 뿜어져 나오는 모션을 감상한다. (루프 시 부하 축적 비주얼).
  3. [CONCATENATE 칩] 으로 스위칭한다. 힙 기지에 임시 박스 생성 없이, 두 문자가 다이렉트 고속 수송선을 타고 목적지 포트로 즉시 찰딱 전송 완료되는 쾌속 피드백을 감상한다.
- **상태 및 데이터**:
  - `SPLIT AT 구분자를 텍스트에 없는 세미콜론(;)으로 지정해 쪼개기에 실패한 상태` -> 런타임 결과: `sy-subrc = 4. Destination variables unchanged` 하이라이트.
- **피드백**: 대량 처리망에서는 `&&` 의 가비지 비용을 억제하고 `CONCATENATE` 고속 카피를 물려야 성능이 사수됨을 배운다.

### 실수/주의

- **문자열 검색을 하겠답시고 옛 구식 아바 명령어인 SEARCH ztext 를 무단 사용**:
  - `SEARCH` 문은 최신 아바 릴리즈와 클라우드 Clean Core 스펙에서 전면 폐기(Obsolete)된 비권장 명령어이므로, 정적 검사 ATC 에서 가차 없이 릴리즈가 차단된다.
  - **문자열 검색은 무조건 현대 공인 표준인 `FIND` 문을 수호해 써야 한다.**

### 정리

- 대량 루프 내 문자열 결합 시 **`&&`** 는 힙 임시객체 생성 부하를 낳으므로 **`CONCATENATE`** 가 고속 복제에 유리하다.
- **`SPLIT`** 은 구분자(`AT`)를 기둥 삼아 데이터를 잘라 여러 변수에 분배 격리한다.
- 문자열 찾기는 **`FIND`**, 교체는 **`REPLACE`** 를 쓰며 직전 결과는 **`sy-subrc`** 에 부착된다.
- 문자의 실 글자 길이는 **`strlen( )`** 내장함수로 찰딱 리턴받는다.

## CH04-L03 - IF와 조건식

### 왜 필요한가

문자열 다듬기 제어까지 완수했다.
이제 "결제 액수가 1,000원을 넘으면 소액 결제로 분기하라" 는 분기 조건 판단의 '연산 비용' 이 장벽입니다.
- " IF, ELSEIF, ELSE 를 써서 다각도로 조건을 쪼갰다. 
그런데 IF 조건식 라인이 기동될 때, 컴퓨터 내부 CPU 단에서는 조건 참/거짓 판정을 매 라인마다 어떻게 처리하며 성능 낭비가 나는가?"
그리고 참/거짓(Boolean) 전용 물리 타입이 없는 아바에서, 쌩 문자 `'X'` 를 조건 플래그로 썼을 때 코드의 가독성이 파손되어 소스 리팩터링이 불가능해지는 위협에 직면한다.

**IF 조건식 가동 시 매 줄마다 조건 참/거짓 식을 CPU 가 새로 계산해 판정하는 Re-evaluation(재평가) 동작과, boolean 대용 표준 `abap_bool` 및 `abap_true/false` 상수 결선 기술**이 필요합니다. 그것이 **IF 와 조건식**의 장악입니다.

### 무엇인가

#### 1. IF / ELSEIF / ELSE / ENDIF (재평가 분기 기작)
- **Re-evaluation (재평가)**: **IF 문은 인터프리터가 매 `ELSEIF` 줄에 적힌 복합 비교식(`p_amt > 1000 AND p_vip = 'X'`)을 만날 때마다, CPU 가 레지스터 값을 직접 꺼내어 연산식을 매번 새로 계산해 평가한다. 따라서 ELSEIF 사다리가 20단씩 길어지면 맨 밑줄 비교를 탈 때 엄청난 연산 낭비를 수반한다.**

#### 2. 비교 연산자 영문/기호 매칭
- `=` (`EQ`), `<>` (`NE`), `>` (`GT`), `<` (`LT`), `>=` (`GE`), `<=` (`LE`) 의 쌍으로 작동하며, 가독성을 위해 기호를 수호해 쓰는 것이 현대 트렌드다.

#### 3. IS INITIAL (초기값 수문장)
- **변수 안방에 데이터가 아예 비어 있는 상태(정수 `0`, 문자 `공백`)인지를 단 한 줄로 깔끔하게 판정하는 특수 비교 지시어다.** (하드코딩 `IF var = 0 OR var = space` 를 묶어 대치하므로 성능과 가독성이 비약한다.)

#### 4. abap_bool 과 XSDBOOL / BOOLC (불리언 정합성)
- **abap_bool (표준 불리언 타입)**: 아바에는 쌩 `boolean` 타입이 없어, 한 글자 문자(`c length 1`) 규격을 포장해 만든 SAP 공인 참/거짓 전용 타입이다.
- **abap_true / abap_false (표준 불리언 상수)**: 참(`'X'`)과 거짓(`공백`) 값을 날코딩하지 않게 막는 표준 상수다.
- **XSDBOOL( 조건식 )**: **괄호 안에 조건식을 던지면, 참일 때 `'X'` (abap_true) 를, 거짓일 때 `공백` (abap_false) 을 자동으로 찰딱 리턴해 주는 스마트 불리언 대입 함수다.**

### 어떻게 확인하는가

abap_bool 과 XSDBOOL, 그리고 IS INITIAL 이 조립된 IF 분기 코드를 검증한다.

```abap
REPORT zhello_if.

PARAMETERS: p_amt TYPE zde_amount,
            p_vip TYPE abap_bool DEFAULT abap_false.

DATA gv_large_flag TYPE abap_bool.

" 1. [★ 중요: 빈값 입력을 선제 판정하는 IS INITIAL 가드 기동!]
IF p_amt IS INITIAL.
  WRITE: / '금액이 비어 있습니다. 입력이 거부됩니다.'.
ELSE.

  " 2. [IF 복합 조건 사다리 재평가 기동]
  IF p_amt > 100000 AND p_vip = abap_true.
    WRITE: / 'VIP 특별 고액 정산 대상자'.
  ELSEIF p_amt > 10000.
    WRITE: / '일반 고액 정산 대상자'.
  ELSE.
    WRITE: / '소액 정산 대상자'.
  ENDIF.

ENDIF.

" 3. [★ XSDBOOL 함수를 사용해 조건 결과를 불리언 플래그에 원샷 대입!]
gv_large_flag = xsdbool( p_amt > 50000 ). " 5만 초과 시 'X', 아니면 공백!

IF gv_large_flag = abap_true.
  WRITE: / '고액 플래그 ON!'.
ENDIF.
```

#### 체험/시뮬레이터 설계 (IF/CASE 분기 레일)
- **프로세스 플로우**:
  1. 학습자가 [IF 철도 선로] 와 [조건 검사기] 를 본다.
  2. [p_amt = 15000] 기차가 굴러온다.
  3. [1번 분기: > 100000] 조건 차단막이 내려선다. CPU 가 15000 > 100000 식을 계산(Re-evaluation)해 거짓 판정을 낸 뒤 차단막을 올린다. 기차가 아래로 흐른다.
  4. [2번 분기: > 10000] 차단막을 만나자 CPU 가 식을 재평가해 참 판정을 내고 선로 스위치를 찰딱 꺾어 '일반 고액' 역으로 기차가 진입하는 연산 제어 피드백을 감상한다.
- **상태 및 데이터**:
  - `abap_bool 플래그 비교 시 gv_flag = 'Y' 처럼 표준 abap_true(X) 가 아닌 다른 문자로 오인 기입한 상태` -> 런타임 결과: `Boolean match fail. Branch bypassed` 하이라이트.
- **피드백**: IF 분기는 매 ELSEIF 단계마다 조건식을 CPU 가 물리적으로 재연산해 평가하는 비용을 동반함을 인지한다.

### 실수/주의

- **IF 문 비교 시, 괄호 ( ) 안팎의 공백을 빼먹어 컴파일 에러 유발**:
  - `IF (p_amt > 5000)` 처럼 괄호 내벽 공백을 안 띄워 쓰면 아바 컴파일러는 `(p_amt` 라는 괴이한 명칭의 함수나 변수를 찾으러 가다가 컴파일 기각 에러를 때린다.
  - **무조건 괄호 양옆은 `( p_amt > 5000 )` 처럼 띄워 써야 작동함을 수호해야 한다.**

### 정리

- **`IF-ELSEIF`** 는 매 라인마다 복합 식을 CPU 가 새로 평가하는 **`Re-evaluation`** 분기 방식이다.
- 비어 있는 기본값 상태를 감지하는 수문장은 **`IS INITIAL`** 이다.
- 참/거짓 플래그는 C 1 대신 **`abap_bool`** 타입을 쓰며, 값은 **`abap_true`** / **`abap_false`** 상수를 장착한다.
- 조건식의 참/거짓 판정을 불리언 변수에 1초 만에 대입할 때는 **`XSDBOOL( )`** 함수를 쓴다.

## CH04-L04 - CASE 분기

### 왜 필요한가

IF 복합 사다리 제어까지 완수했다.
이번에는 "회원 등급이 A 일 때, B 일 때, C 일 때..." 처럼 '단일 변수의 고정 값 분기' 의 성능 최적화가 장벽입니다.
- " 한 개 변수 `gv_grade` 의 값을 10가지 등급 코드와 매칭 비교한다. 
IF ELSEIF 로 쓰자니 가독성도 나쁘고 연산 속도도 느린데, CASE WHEN 은 내부적으로 아바 인터프리터가 분기할 때 어떤 압도적인 속도 혜택을 제공하는가?"
이 CASE WHEN 구문 선포 시 비교 연산자 `<>` 나 범위 연산 `> 10` 을 쑤셔 넣으려다 컴파일러에 의해 기각당하는 문법 한계를 명세해야 한다.

**단일 변수 고정값 일치 비교 시 인터프리터가 내부적으로 Hash-like Jump Table (해시 구조의 점프 테이블)을 생성해 O(1) 초고속 분기를 단행하는 CASE 의 물리 메커니즘과, 범위 연산 제한 및 WHEN OTHERS 가드 기술**이 필요합니다. 그것이 **CASE 분기**의 장악입니다.

### 무엇인가

#### 1. CASE ... WHEN ... WHEN OTHERS ... ENDCASE (고속 점프 분기)
- **Hash-like Jump Table (해시 구조 점프 테이블)**: **아바 컴파일러는 `CASE gv_grade.` 를 만나는 순간, 내벽의 `WHEN 'A'`, `WHEN 'B'` 목록들을 해시 맵 구조의 분기 테이블로 메모리에 선제 등록해 둔다. 실행 시점에 CPU 는 10개의 WHEN 을 위에서부터 순차 재평가(Re-evaluation)하지 않고, gv_grade 의 해시 값으로 타깃 WHEN 블록의 주소를 0.0001초 만에 찾아 다이렉트 홉 점프(Direct Jump - O(1) 에 근접)를 단행한다. 분기 개수가 수십 개로 많아질수록 IF 사다리보다 백배 고속 작동한다.**

#### 2. CASE 분기의 Equality 제약
- **CASE WHEN 은 오직 `=` (EQ - 같냐?) 일치 비교만 소화한다.** 
- `WHEN > 10` 이라거나 `WHEN gv_y <> 0` 과 같은 범위 및 관계 비교를 억지로 기입하면, 아바 컴파일러가 신택스 에러를 뿜으며 즉각 빌드를 차단한다. (범위/관계 비교는 IF 의 전유물이다.)

#### 3. WHEN OTHERS (최종 예외 가드)
- *예상 밖의 값이 들어올 때의 백엔드 탈출구다.*
- 상기 지정한 WHEN 등급 목록에 단 1개도 해당되지 않는 이방인 값이 들어왔을 때, 조용히 씹히는 사고를 막기 위해 디폴트 에러나 예외 처리를 담아 방출하는 필수 안전 가드 영역이다.

### 어떻게 확인하는가

해시 분기가 성립되는 CASE WHEN 과 WHEN OTHERS 조립 코드를 검증한다.

```abap
REPORT zhello_case.

PARAMETERS p_grade TYPE c LENGTH 1 DEFAULT 'B'.

DATA gv_desc TYPE string.

" 1. [★ 단일 변수 p_grade 에 대고 해시 점프 테이블 가동!]
CASE p_grade.

  " 2. [WHEN 에 복수 일치 값을 콤마(,)로 묶어 OR 매핑!]
  WHEN 'A' OR 'S'.
    gv_desc = '우수 고객 예약 승인'.
    
  WHEN 'B'.
    gv_desc = '일반 고객 예약 승인'.
    
  " 3. [★ 어떤 WHEN 에도 걸리지 않은 쓰레기 예외 값을 최종 포착 가드!]
  WHEN OTHERS.
    gv_desc = '미인가 등급 - 승인 거절'.
    
ENDCASE.

WRITE: / '등급 판정 결과:', gv_desc.
```

#### 체험/시뮬레이터 설계 (IF/CASE 분기 레일)
- **프로세스 플로우**:
  1. 학습자가 [CASE 해시 점프 레일판] 과 [p_grade = B 칩] 을 본다.
  2. p_grade 기차가 진입한다. 
  3. [WHEN A], [WHEN S] 차단막을 거치지 않고, 기차가 진입과 동시에 [B 해시 슬롯 주소] 로 눈 깜짝할 사이에 촥 미끄러져 들어가 '일반 고객' 역으로 다이렉트 다이빙하는 O(1) 모션을 관찰한다.
  4. 이번엔 p_grade 에 듣도 보도 못한 [X] 를 넣자, [WHEN OTHERS 리다이렉트 통로] 가 찰딱 열려 거절 역으로 떨어지는 예외 방어 피드백을 감상한다.
- **상태 및 데이터**:
  - `WHEN 뒤에 범위 비교식인 p_amt > 1000 을 억지로 적어 컴파일한 상태` -> 런타임 결과: `Syntax error. Comparators not allowed in WHEN` 하이라이트.
- **피드백**: 단일 변수의 1:1 매칭 분기는 CPU 재연산이 없는 `CASE` 가 최선의 성능이며, `WHEN OTHERS` 가 안전 가드임을 인지한다.

### 실수/주의

- **CASE WHEN ENDCASE 를 적을 때 짝꿍 닫는 지시어를 ENDIF 로 혼동해 코딩**:
  - 런타임 컴파일러가 CASE 의 끝을 못 찾아 짝 안 맞음 에러를 뿜으며 즉각 빌드를 다운 차단한다.
  - **CASE 의 짝꿍은 무조건 `ENDCASE` 임을 수호해야 한다.**

### 정리

- 단일 변수의 1:1 고정값 분기 시 **`CASE`** 는 내부적으로 **`해시 점프 테이블`** 을 구동해 고속 O(1) 분기한다.
- CASE 는 범위/관계 비교가 불가능하며 오직 **`일치 (=)`** 비교만 전담한다.
- 상기 목록 외의 돌발 유입 값을 가드하기 위해 **`WHEN OTHERS`** 예외 절을 의무 동반한다.

## CH04-L05 - DO / WHILE · 루프 제어

### 왜 필요한가

CASE 분기 제어까지 완수했다.
이제 1~9단 구구단 출력이나 10만 건 대량 이자 적리 계산 같은 '반복(Loop) 연산' 의 스택 통제가 장벽입니다.
- " 중첩 반복문을 기동했는데, 안쪽 루프를 도는 순간 바깥 루프의 현재 회차 번호가 쌩 덮어씌워져 유실되어 루프가 엉망으로 깨진다.
왜 이 시스템 회차 필드 `sy-index` 가 유실되는지 물리 원인을 밝히고, 루프를 탈출/건너뛰는 지시어들의 세부 동작 차이는 무엇인가?"
DO 나 WHILE 내벽에서 안쪽 루프를 2차로 중첩 격발하면, 전역 단일 레지스터인 `sy-index` 가 안쪽의 F8 카운트로 오버라이트 갱신되어 바깥 회차를 영구 상실하는 대참사를 겪는다.

**`sy-index` 시스템 카운터 레지스터의 중첩 루프 덮어쓰기 유실 제약과, 3대 탈출 지시어(EXIT: 즉시 탈출, CONTINUE: 다음 회차 강제 점프, CHECK: 조건부 다음 회차 점프)의 메모리 분기 차이를 규명하는 기술**이 필요합니다. 그것이 **DO / WHILE 루프 제어**의 완수입니다.

### 무엇인가

#### 1. DO n TIMES ... ENDDO (정 횟수 반복)
- **sy-index 스택 유실**: **`sy-index` 는 아바 엔진 내에 단 1개만 존재하는 전역 시스템 카운터 필드다. `DO 9 TIMES.` 를 기동하면 1, 2, 3... 번호가 sy-index 에 들어온다. 그런데 내벽에 안쪽 `DO` 를 중첩해서 또 기동하는 순간, 안쪽 루프가 `sy-index` 값을 1로 강제 오버라이트 덮어쓰며 바깥 번호가 소멸 유실된다. 따라서 바깥 회차 값은 안쪽 루프에 가기 전 반드시 일반 로컬 변수(`gv_dan = sy-index`)에 백업 캐싱하는 복제 결선이 필수적이다.**

#### 2. WHILE 조건식 ... ENDWHILE (조건부 반복)
- 횟수가 고정되지 않고, 지정 조건식이 `abap_true` 인 참 상태를 유지하는 동안 무한히 반복을 누적 기동하는 엔진이다. (조건을 언젠가 거짓으로 돌리는 보정 코드가 누락되면 무한 루프 CPU 100% 락 뇌사를 유발하므로 각별히 조율해야 한다.)

#### 🧭 [ 3대 루프 제어 지시어 EXIT / CONTINUE / CHECK 비교 명세 ]
- *반복문 내부의 흐름을 쪼개는 3대 제어 필터 장치다.*

```text
[1] EXIT (루프 완전 즉시 즉사 탈출) :
   - 지시어를 만나는 즉시, 루프 아래의 ENDDO 를 뚫고 나와 반복문 자체를 영구 소멸 종결한다.
   │
[2] CONTINUE (이번 회차만 스킵 점프) :
   - 하단의 남은 아바 구문들을 일절 실행하지 않고 쌩 건너뛰어, 
     즉각 다음 sy-index 회차(예: 3회차 -> 4회차)로 머리를 꺾어 반복을 계속한다.
   │
[3] CHECK 조건식 (조건부 CONTINUE 기동) :
   - 조건식이 '참' 이면 아무 짓도 안 하고 아래 구문으로 흐르고, 
     조건식이 '거짓' 이면 즉시 CONTINUE 와 똑같이 이번 회차를 씹고 다음 회차로 점프한다.
     ( IF NOT 조건. CONTINUE. ENDIF. 구문을 한 줄로 압축한 특수 결선이다. )
```

### 어떻게 확인하는가

sy-index 백업과 3대 탈출 필터가 엮인 반복 제어 코드를 검증한다.

```abap
REPORT zhello_loop.

DATA: gv_dan TYPE i,
      gv_sum TYPE i.

" 1. [바깥 루프 기동: 2단부터 9단까지 총 8회 회전 설계]
DO 8 TIMES.
  " gv_dan = 2, 3, 4 ... 9
  " 2. [★ 중요: 안쪽 루프로 들어가기 전, sy-index 덮어쓰기 유실을 막기 위해 로컬 백업!]
  gv_dan = sy-index + 1. 
  
  " 3. [안쪽 루프 기동: 곱하는 수 1부터 9까지 회전]
  DO 9 TIMES.
    DATA(gv_mul) = sy-index.
    
    " [★ CHECK 가드 기동: 곱하는 수가 5일 때만 아래 연산을 씹고 다음 회차 점프!]
    CHECK gv_mul <> 5. 
    
    " [★ EXIT 가드 기동: 곱하는 수가 8을 넘으면 이 안쪽 루프를 강제 즉시 탈출!]
    IF gv_mul > 8.
      EXIT.
    ENDIF.
    
    DATA(gv_res) = gv_dan * gv_mul.
    WRITE: / gv_dan, '*', gv_mul, '=', gv_res.
  ENDDO.
ENDDO.
```

#### 체험/시뮬레이터 설계 (루프 스택 카운터)
- **프로세스 플로우**:
  1. 학습자가 [바깥 회전판 sy-index] 와 [안쪽 회전판 sy-index], [백업 상자 gv_dan] 을 본다.
  2. 바깥 판이 돌며 sy-index 가 1이 된다. 
  3. [백업 상자 = OFF] 인 채로 안쪽 판을 돌린다. 안쪽 판이 돌며 전역 sy-index 를 1, 2, 3... 으로 덮어써서, 바깥 판 번호 1 이 영구 소멸해 루프가 오작동으로 터진다.
  4. [백업 상자 = ON] 을 켜자, 바깥 sy-index 번호 1 이 gv_dan 에 복제되어 잠기고, 안쪽 판이 9번 돌아 sy-index 가 9가 되어도 바깥 번호가 gv_dan 에서 온전히 살아남아 정합성을 지키며 기어가 도는 피드백을 감상한다.
- **상태 및 데이터**:
  - `WHILE 문에서 탈출 조건을 제어하지 않아 무한 루프 락 CPU 100% 가 걸린 상태` -> 런타임 결과: `TIME_OUT. Maximum execution time exceeded` 하이라이트.
- **피드백**: 중첩 루프 진입 시 sy-index 백업은 선택이 아닌 커널 전역 변수 침범을 막는 절대 공식임을 배운다.

### 실수/주의

- **DO TIMES 문 안에 CHECK 구문을 넣으면서, CHECK 조건을 쌩 변수가 비어 있는지 판정하는 IS INITIAL 과 혼동해 거꾸로 지정**:
  - `CHECK gv_var IS INITIAL` 은 "비어 있어야만 아래 구문을 실행하고, 채워져 있으면 스킵 점프하라" 는 반대 동작을 낳아 데이터가 통째 스킵 훼손되는 참사를 부른다.
  - **CHECK 조건은 '통과 조건 (참일 때 통과)' 임을 헷갈리지 않고 수호해야 한다.**

### 정리

- 전역 단일 필드인 **`sy-index`** 는 중첩 루프 기동 시 덮어쓰기 유실되므로, 진입 전 **`로컬 캐시 변수`** 에 백업한다.
- 루프 완전 탈출은 **`EXIT`**, 현재 회차 스킵 점프는 **`CONTINUE`** 다.
- **`CHECK 조건`** 은 조건이 거짓일 때 `CONTINUE` 로 즉시 점프 필터링한다.

## CH04-L06 - 디버깅 입문

### 왜 필요한가

반복문 제어 구문까지 완수했다.
이제 프로그램이 돌다가 비정상 종료(뇌사)되는 사고 원인 관제와, 실행 중인 메모리 변수 스캔이 마지막 장벽입니다.
- " 결산 계산 중 갑자기 빨간 에러 창(Short Dump)이 뜨며 프로그램이 기절해 죽어버렸다. 
이 덤프가 터졌을 때 어떤 시스템 테이블에 사고 로그가 적재되며, `/h` 디버거 단축키 4형제(F5, F6, F7, F8)는 메모리를 어떻게 락 제어하는가?"
디버깅을 위해 소스 코드에 `BREAK-POINT.` 구문을 대충 적어두고 그대로 운영(PRD) 장비로 이송(TR)해 버리면, 실제 현업 사용자가 조회를 누르는 순간 결제 화면이 디버거로 멈춰 서서 전산 대란이 터지는 참변을 맞이합니다.

**프로그램 덤프 발생 시 사고 로그가 데이터베이스 `SNAP` 시스템 테이블에 적재되는 ST22 분석 구조와, 디버거 단축키 4형제의 물리 흐름 제어, 그리고 Watch Point 감시 기술**이 필요합니다. 그것이 **디버깅과 ST22 분석**의 완수입니다.

### 무엇인가

#### 1. Short Dump 와 ST22 Dump Analysis (SNAP 대장)
- **Short Dump**: 0 나누기나 타입 위배 등 치명적인 런타임 예외를 방어하지 못했을 때 커널이 기동하는 전산 비상 정지 장치다.
- **SNAP (덤프 적재 대장)**: **덤프가 터지면 아바 커널은 즉각 사고 당시의 소스 코드 줄 번호, 변수 값, 오류 코드, 유저 ID 를 DB 테이블인 `SNAP` 에 인서트 기록한다. 우리는 트랜잭션 `ST22` (Dump Analysis)를 켜서 이 `SNAP` 테이블 정보를 인덱스 검색하여, "운영계의 이 에러가 몇 번째 라인에서 무슨 변수 값 때문에 죽었는지" 를 원격으로 낱낱이 짚어내 해부할 수 있다.**

#### 2. /h 와 BREAK-POINT (디버거 강제 격발)
- **/h (T-code 명령창 디버그 스위치)**: 코드를 단 1글자도 수정하지 않고, 명령창에 `/h` 를 치는 즉시 다음 엔터 이벤트 시점부터 메모리 락을 걸어 디버거 세션을 강제 분리 기동하는 실무 최고의 클린 디버깅 스위치다.
- **BREAK-POINT. (소스 내장 멈춤)**: 소스 상에 지시어를 심어 강제 정지시킨다. (주의: 이송 전 제거가 엄격한 실무 CTS 통제 룰이다.)

#### 🧭 [ 디버거 제어 단축키 4형제 동작 명세 ]
- *디버거 세션 홀딩 중 CPU 제어권을 한 단계씩 이동시키는 방향타다.*

```text
[1] F5 (Step Into - 1줄 다이빙) :
   - 노란 화살표가 가리키는 라인을 딱 1줄만 실행한다. 만약 그 라인이 서브루틴이나 클래스 메서드 호출이면, 호출 내벽 깊숙이 다이빙해 들어가 첫 줄에 정지한다.
   │
[2] F6 (Execute - 호출 우회 통과) :
   - 1줄 실행하되, 함수나 클래스 호출을 만나면 내부로 들어가지 않고, 내부 연산을 백엔드에서 원샷 실행 완료하고 다음 아랫줄로 홉 점프해 대기한다.
   │
[3] F7 (Return - 호출 탈출 복귀) :
   - 현재 들어가 헤매고 있는 서브루틴이나 함수 내벽 영역을 즉시 탈출하여, 나를 호출해 준 부모 프로그램 아랫줄로 올라와 정지한다.
   │
[4] F8 (Continue - 런 프리 실행) :
   - 더 이상 한 줄씩 멈추지 않고, 다음 BREAK-POINT 나 Watch Point 를 만날 때까지 프로그램을 쌩 고속 구동 완료시킨다.
```

#### 3. Watch Point (변수 실시간 감시 포트)
- **"gv_sum 변수 값이 100 으로 변하는 순간 강제로 디버거를 정지시키라" 고 메모리 감시 필터를 거는 장치다. 수천 번의 루프를 손가락 아프게 F5 로 누르지 않고도, 버그가 터지는 변수 변동의 그 역사적 찰나 라인으로 한 번에 순간 이동하게 돕는다.**

### 어떻게 확인하는가

`/h` 로 디버거를 기동하고 Watch Point 를 가설해 덤프 원인을 추적하는 흐름을 검증한다.

```text
[1단계] SE38 구구단 프로그램 실행 :
   - 입력 화면 Parameter 단에서 명령창(좌상단)에 '/h' 입력 후 Enter!
   - 화면 하단에 "Debugging switched on" 팝업 점등! F8 실행 단추 클릭!
   
[2단계] 디버거 화면 로드 및 Watch Point 가설 :
   - 노란 화살표 1라인 대기 중.
   - 우측 Watch Point 단추 클릭 -> Variable : 'gv_res', Condition : 'gv_res > 50' 기입 완료!
   - [F8 (Continue)] 단추 클릭!
   
[3단계] Watch Point 격발 확인 :
   - 루프가 고속으로 돌다가, gv_res 가 54가 되는 그 찰나(gv_dan=6, gv_mul=9)에 디버거가 자동으로 촥 홀딩 정지!
   - 하단 변수 탭에서 gv_dan = 6, gv_mul = 9 실시간 램 값 검수 성료! (버그 탐색 완료!)
```

#### 체험/시뮬레이터 설계 (디버거 세션 락커)
- **프로세스 플로우**:
  1. 학습자가 [노란 화살표 CPU 기어] 와 [메모리 디스플레이 창] 을 본다.
  2. [F5 단추] 를 누르자 기어가 딱 1칸 톱니 회전하며 gv_sum 이 10에서 20으로 올라간다.
  3. [F8 단추] 를 누르자 기어가 쌩 회전해 끝까지 가려는데, [Watch Point 센서 (gv_res > 50)] 에 닿는 순간 [노란 화살표 쇠사슬] 이 내려와 기어를 찰딱 락 채워 세션을 세우는 디버깅 피드백을 감상한다.
- **상태 및 데이터**:
  - `BREAK-POINT 주석을 지우지 않고 운영(PRD) 이송 결재를 올린 상태` -> 런타임 결과: `ATC Check failed. Hardcoded Breakpoint detected in production track` 하이라이트.
- **피드백**: 디버깅은 쌩 텍스트 출력이 아니라, 램 레지스터를 세워 들여다보는 홀딩 세션이며, `/h` 와 Watch Point 가 최고의 무기임을 체득한다.

### 실수/주의

- **디버깅 도중, 루프가 1만 번 도는 구간에서 F5 (Step) 단추만 마우스로 무한 난사하며 손가락 노동을 단행**:
  - 디버거가 응답 없음으로 굳거나 개발자가 지쳐 쓰러진다. 
  - **대량 루프에서는 무조건 `Watch Point` 에 변수명과 탈출 값을 걸고 `F8` 로 다이렉트 점프해 포착해야 함을 수호해야 한다.**

### 정리

- 치명적 런타임 폭사 사고 시 변수와 위치 정보가 DB **`SNAP`** 테이블에 적재되며, **`ST22`** 에서 조회 분석한다.
- 실무 클린 디버깅은 소스 수정 없이 명령창에 **`/h`** 를 입력해 즉시 격발한다.
- **`F5`** 는 1줄 다이빙, **`F6`** 은 호출 우회 통과, **`F7`** 은 부모 복귀, **`F8`** 은 쌩 연속 가동이다.
- 특정 변수가 요동치는 찰나를 저격해 멈추는 메모리 센서는 **`Watch Point`** 다.

## CH04-L07 - 종합 실습: 구구단

### 왜 필요한가

구구단 3단계 범위 조절 스펙까지 완성했다.
이제 이 관통예제 조립 시 발생하기 쉬운 '중첩 루프 인덱스 훼손' 과 '범위 역전 오류' 가 마지막 장벽입니다.
- " 2단부터 9단까지, 안방에 곱하는 수 1~9를 돌리는 중첩 DO 문을 짰는데, 화면에 2단 결과가 1줄 나오고 프로그램이 그냥 끝나버리는 버그가 난다. 
이 훼손 원인과 3단계 범위(p_from ~ p_to) 조절 시 횟수를 영리하게 지정하는 수학은 무엇인가?"
안쪽 루프가 도는 순간 전역 `sy-index` 가 강제 오버라이트 갱신되어 바깥 회차를 훼손하는 정합성 장벽을 이해해야 한다.

**바깥 sy-index 가 덮어씌워져 파손되는 것을 방지하기 위해 로컬 변수 `gv_dan = sy-index` 로 백업 격리하고, 3단계 범위 횟수 공식 `p_to - p_from + 1` 을 엮어 구구단을 조립 완료하는 기술**이 필요합니다. 그것이 **구구단 종합 실습**의 완수입니다.

### 무엇인가

#### 1. 중첩 DO 루프와 sy-index 덮어쓰기 복구
- 바깥 루프 `DO 8 TIMES` 와 안쪽 루프 `DO 9 TIMES` 가 결선될 때, 안쪽 루프의 `DO` 문이 기동되자마자 전역 `sy-index` 시스템 카운터는 안쪽 회차 수인 1, 2, 3... 으로 강제 리셋 갱신된다. 
- 이 리셋이 발생하면 바깥 루프는 자신이 몇 번째 단을 돌고 있었는지 망각하여 단 1회전 만에 프로그램을 끝마친다.
- **해소**: 안쪽 루프로 머리를 꺾기 직전에 반드시 **`gv_dan = sy-index + 1`** (1회차는 2단이므로 +1)을 수행하여, 단수 정보인 `gv_dan` 을 안전한 로컬 변수 주소에 백업 보존해야만 안쪽 루프가 9번 도는 내내 단수가 수호된다.

#### 2. 3단계 범위 횟수 연산 공식
- `p_from` (시작 단 - 예: 3단)부터 `p_to` (끝 단 - 예: 5단)까지 범위만 돌리기 위해 전체 1~9단 루프를 다 켜고 `IF` 로 범위 밖을 필터링하는 구조는 쓸데없는 CPU 리소스를 소모한다.
- **효율 루프**: 애초에 루프 회전수 자체를 **`p_to - p_from + 1`** 회 (5 - 3 + 1 = 총 3회) 로 지정해 `DO` 문을 돌리고, 루프 최종 꼬리 라인에서 시작 단수(`gv_dan`)를 1씩 누적 가산(`gv_dan = gv_dan + 1`)해 나가는 방식이 실무 고속 연산의 핵심 공식이다.

### 어떻게 확인하는가

중첩 DO 백업과 3단계 효율 루프 공식이 결합된 완제 구구단 코드를 검증한다.

```abap
REPORT zhello_gugudan.

PARAMETERS: p_from TYPE i DEFAULT 3 OBLIGATORY, " 필수 가드!
            p_to   TYPE i DEFAULT 5 OBLIGATORY.

DATA: gv_dan TYPE i,
      gv_mul TYPE i,
      gv_res TYPE i,
      gv_cnt TYPE i.

START-OF-SELECTION.

  " 1. [입력 범위 역전 검증 가드 올리기]
  IF p_from > p_to.
    WRITE: / '시작 단이 끝 단보다 클 수 없습니다. 실행이 기각됩니다.'.
    EXIT.
  ENDIF.

  " 2. [★ 효율 공식 기동: 루프 회전수 = 끝단 - 시작단 + 1]
  gv_cnt = p_to - p_from + 1.
  gv_dan = p_from. " 단수 초기화

  DO gv_cnt TIMES. " 지정된 3회만 촥 회전!

    WRITE: / sy-uline, / '★', gv_dan, '단 가동', / sy-uline.

    " 3. [안쪽 루프 기동: sy-index 는 안쪽 회전수로 충전 사용!]
    DO 9 TIMES.
      gv_mul = sy-index.
      gv_res = gv_dan * gv_mul.
      
      WRITE: / gv_dan, '*', gv_mul, '=', gv_res.
    ENDDO.

    " 4. [★ 다음 단수를 위해 1 누적 가산]
    gv_dan = gv_dan + 1.

  ENDDO.
```

#### 체험/시뮬레이터 설계 (구구단 실행기)
- **프로세스 플로우**:
  1. 학습자가 [시작 단 = 3] 과 [끝 단 = 5] 인풋 박스를 본다.
  2. [F8] 을 때려 가동하자, [루프 회전 횟수 계산기] 가 작동해 `5 - 3 + 1 = 3` 회전 칩을 DO 문에 꽂는다.
  3. 3단이 출력되고, 단수 기어가 1 가산되어 4단, 또 가산되어 5단까지 회전 완료 후 루프가 종결된다.
  4. 이때 역전 값 [시작 단 = 9], [끝 단 = 2] 를 기입하자, [역전 IF 차단바] 가 내려와 "시작 단이 끝 단보다 큽니다!" 경보를 뿜고 즉시 멈추는 정합성 피드백을 감상한다.
- **상태 및 데이터**:
  - `안쪽 루프에서 gv_mul 에 sy-index 복사를 누락해 항상 1단만 계산되는 상태` -> 런타임 결과: `Logic bug. gv_mul remains static value` 하이라이트.
- **피드백**: 중첩 루프는 sy-index 덮어쓰기 유실 락을 로컬 변수 백업으로 돌파하고, 횟수 산출은 수학 공식을 타야 함을 깨닫는다.

### 실수/주의

- **중첩 DO 문 바깥 루프의 sy-index 를 백업하지 않고, 안쪽 루프 안에서 sy-index 를 두 번 직접 호출해 연산에 참여**:
  - 바깥 단수가 안쪽 곱하는 수와 1:1 동기화 오작동을 일으켜 2*1=2, 3*2=6 과 같이 대각선 대재앙 연산만 출력되고 구구단이 꼬여 폭사한다.
  - **안쪽 진입 전 바깥 인덱스 백업은 아바 반복문 코딩의 구국 철칙임을 수호해야 한다.**

### 정리

- 중첩 루프에서 안쪽 `DO` 가 돌면 **`sy-index` 시스템 필드는 안쪽 카운트로 강제 덮어씌워진다**.
- 따라서 바깥 회차 `sy-index` 는 안쪽 루프 개시 전에 **`로컬 캐시 변수(gv_dan)`** 에 안전하게 대입 복제해 둔다.
- 구간 범위 루프 기동 시 효율 회전수는 **`p_to - p_from + 1`** 공식으로 빚어 돌린다.
- 입력 값 역전 발생 시에는 최상단에서 **`IF 가드`** 로 걸러 즉시 실행을 차단 기각한다.
