## 3. 공식 문서 수동 근거 명세

`abapopen_dataset.htm` 을 대조하여 파일 핸들 및 OS 보안 제약 요건을 철저히 규명했다.

- **최대 동시 파일 오픈 제한 규명**:
  한 내부 세션(Internal Session)당 동시에 열려 상주할 수 있는 파일 개수는 **최대 100개(Up to 100 files)** 로 AS 엔진 단에서 하드코딩 제약되어 있다.
  만약 이 개수를 초과해 `OPEN DATASET` 을 날리면 catchable exception 인 **`CX_SY_TOO_MANY_FILES` 캐시 덤프**가 터져 장비가 마비된다.
- **중복 오픈 및 OS 에러 예외 명세**:
  동일 파일이 닫히기 전에 재오픈 시 **`CX_SY_FILE_OPEN`** 이 터지며, `MESSAGE` 구문을 엮어야만 OS 단의 물리 에러 원인을 `sy-subrc = 8` 과 함께 수집할 수 있음을 규명했다.
- **Directory Traversal 보안 권고 대조**:
  외부 파라미터가 파일명으로 그대로 유입되면 시스템 전체 Unix 패스워드 등 상위 디스크 경로가 탈취될 수 있으므로 화이트리스트 경로를 정규화(`FILE_VALIDATE_NAME` 등)하여 검증하는 보안 가이드라인 준수를 확인 및 수록 완료.

## 4. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. BAPI 를 다 실행하고 나서 그냥 COMMIT WORK 을 쓰면 왜 안 되고, 반드시 BAPI_TRANSACTION_COMMIT EXPORTING wait = abap_true 를 때려야 하나요?
**A**: **BAPI 는 표준 버퍼 클리어 등의 내부 후처리가 묶인 전용 커밋 함수를 타야 정합성이 맞으며, wait = abap_true 를 주지 않으면 DB 저장 완료 전 다음 아바 코드가 기격발되어 데이터 유실을 유발하는 동기 지연(Sync delay)이 나기 때문입니다.**
- **동기(Synchronous) 대기의 중요성**:
  그냥 쌩 `COMMIT WORK` 를 때리면, 아바 엔진은 DB 서버에게 "저장해라" 고 신호만 던지고, 그 저장이 실제로 디스크에 완전히 기입되어 완료되었는지 확인도 안 한 채 곧바로 다음 소스 코드 줄로 튕겨 넘어가 실행해 버립니다.
  이로 인해, 바로 꼬리에 "방금 BAPI 로 생성한 오더 번호를 2차 SELECT 해와서 화면에 뿌려라" 고 조회를 날리면, DB 디스크에는 아직 오더가 이송 완료되지 않아 `sy-subrc = 4` 빈 값을 읽어 화면에 구멍이 뻥 뚫리는 버그를 초래합니다.
- **해결**:
  `wait = abap_true` 옵션을 전송하면, DB 가 "오더 디스크 적재 100% 완료함!" 이라고 응답 신호를 퉤 뱉어줄 때까지 아바가 0.1초 동안 딱 버티고 기다려주어 정합성 불일치를 수호합니다. 
  또한 `BAPI_TRANSACTION_COMMIT` 내부에는 표준 버퍼와 표준 락(Standard Lock)을 깔끔하게 해제하는 청소 코드가 내장되어 있어 락 고착을 방방해 줍니다.

### Q2. OPEN DATASET 만 수십 번 돌리고 CLOSE DATASET 을 빼먹었습니다. 왜 101번째 파일을 열 때 갑자기 CX_SY_TOO_MANY_FILES 예외 덤프가 터지나요?
**A**: **한 아바 세션당 동시에 메모리에 열어둘 수 있는 파일 핸들(File Handle)의 최대 개수는 100개로 언어 명세상 못 박혀 있기 때문이며, 루프 사용 완료 시점 즉시 CLOSE DATASET 으로 뚜껑을 닫아 리소스를 OS 에 반환해야 덤프를 피합니다.**
- **덤프 메커니즘**:
  컴퓨터 운영체제(OS)는 프로세스가 파일을 열 때마다 파일 지시자(File Descriptor)를 할당해 메모리를 격리 통제합니다.
  아바 커널 엔진은 무한 루프 파일 오픈으로 서버 자원이 고갈되는 것을 막기 위해, 한 번에 열어둘 수 있는 동시 개수를 **100개**로 강력히 통제합니다.
  사용이 끝난 파일에 대해 `CLOSE DATASET lv_path.` 호출을 누락해 두면 메모리 상에 파이프라인 핸들이 그대로 살아남아 누수(Leak)가 나고, 101번째에 도달하는 찰나 `CX_SY_TOO_MANY_FILES` 치명적 덤프를 터트려 프로그램을 셧다운시킵니다.
  **철칙**: 파일 처리가 끝나는 ENDDO 나 LOOP 꼬리에는 반드시 잊지 말고 **`CLOSE DATASET`** 자물쇠를 채워 주어야 안전합니다.

### Q3. 사용자가 화면 파일명 입력창에 ../../etc/passwd 라고 쳐서 업로드를 태웠습니다. 이게 왜 Directory Traversal 보안 해킹이 되며, 어떻게 방어해야 하나요?
**A**: **입력된 상대 경로가 시스템 루트 상위 폴더로 탈출하여 리눅스/윈도우의 OS 핵심 보안 파일인 패스워드 대장을 쌩으로 열어 전송해 주는 무서운 취약점으로 작동하기 때문이며, logical filename 정규화 함수로 방어합니다.**
- **해킹 메커니즘**:
  프로그램이 지정한 내부 기본 작업 디렉토리가 `/usr/sap/interfaces/` 라고 칩시다.
  여기에 외부 사용자가 텍스트 창에 파일명으로 `../../etc/passwd` 를 주입했습니다.
  아바가 검증 없이 `OPEN DATASET '/usr/sap/interfaces/../../etc/passwd'` 를 실행하면, OS 는 상대 경로를 해석해 상위 폴더로 두 칸 껑충 뛰어 결국 `/etc/passwd` 에 접근해 파일 뚜껑을 열어버립니다. 
  해커는 단 몇 초 만에 SAP 서버가 상주하는 OS 의 모든 계정 정보와 패스워드 해시 데이터를 털어가서 서버를 장악합니다.
- **정석 방어선**:
  물리 파일 경로를 열기 전에 반드시 **`FILE_VALIDATE_NAME`** 표준 함수에 대고 검증을 거쳐야 합니다. 
  이 함수는 사전에 약속해 둔 안전 폴더 논리 명칭(`logical_filename`) 범위를 벗어나 상위 디렉토리로 기어오르는 탈출 시도를 사전에 감지해 즉시 에러 예외(`name_not_valid = 1`)를 뿜고 차단해 줍니다.

### Q4. RFC 함수 호출 시 DESTINATION 을 SM59 에 연결하고 쏠 때, 왜 communication_failure 예외 뒤에 굳이 MESSAGE lv_msg 꼬리표를 얹어야 하나요?
**A**: **이 꼬리표를 얹지 않으면 통신 장애 시 그냥 sy-subrc = 1 만 퉤 뱉을 뿐이라 "망이 끊긴 건지, 상대 서버 주소가 틀린 건지, 아이디 패스워드가 만료된 건지" 상세 원인을 알 수 없어 디버깅이 먹통이 되기 때문이며, OS 가 알려준 상세 원인 텍스트를 lv_msg 에 캡처하기 위함입니다.**
- **디버깅 단서 획득**:
  네트워크 통신 오류는 내 아바 소스 코드가 틀려서가 아니라, 상대방 물리 서버 스위치가 내려갔거나 방화벽이 차단되어 터지는 외부 장애가 90% 이상입니다.
  `MESSAGE lv_msg` 구문을 엮어두면, 에러가 터졌을 때 OS 소켓 단에서 보내온 **"Connection refused by peer (상대가 연결 거부)"** 나 **"User account locked (로그인 잠김)"** 같은 진짜 장애 물리적 단서 메시지가 `lv_msg` 문자열 변수에 고스란히 저장됩니다. 
  개발자와 인프라 팀은 이 원인 문구 한 줄을 보고 5분 만에 네트워크 장비를 고칠 수 있지만, 이게 없으면 며칠 동안 애꿎은 아바 함수 소스만 뜯어보며 삽질을 겪게 됩니다.

### Q5. BAPI 가 없는 트랜잭션을 BDC 로 자동 등록할 때, Call Transaction 방식 대신 Session Method (SM35) 방식을 야간 대량 배치에 채택하는 이유는 무엇인가요?
**A**: **Call Transaction 은 에러 발생 시 그 자리에서 롤백되고 데이터가 유실되지만, Session 방식은 실패한 예약 건들만 SM35 큐 상에 따로 고이 보관해 두어 나중에 에러가 난 화면만 띄워 수동 보정 저장할 수 있는 견고한 사후 재처리 구조를 보장하기 때문입니다.**
- **SM35 세션의 가치**:
  - **`Call Transaction` (실시간형)**: 1,000건을 올리다 500번째 행에서 수량 포맷 에러가 나면, 프로그램은 그냥 에러 로그 한 줄 뿜고 통과하지만, 이미 날아간 500번째 데이터는 다시 올릴 길이 막막해 현업이 엑셀을 돋보기로 대조하며 다시 쳐야 합니다.
  - **`Session (SM35)` (배치 큐형)**: 1,000건을 `BDC_INSERT` 함수로 세션 큐에 쑤셔 넣으면, SAP 배치 큐가 이를 접수합니다. 
    새벽에 배치를 돌리다 에러가 나도, 실패한 500번째 건은 삭제되지 않고 `SM35` 트랜잭션 큐에 **'에러 대기'** 상태로 안전하게 보관됩니다.
    다음 날 아침 출근한 현업 직원이 SM35 에 들어가 [오류 행 화면 가동] 단추를 누르면, 에러가 난 그 입력창 화면이 딱 켜지며 "아, 수량이 비었구나" 하고 빈칸만 타자로 메워 [저장] 을 누름으로써 데이터 유실 한 건 없이 100% 안전하게 구출/재처리를 완료할 수 있습니다.
  - **결론**: 실시간 즉각 피드백이 필요한 온라인 화면은 Call Transaction 을 쓰고, 대량 결산이나 정산 배치 데이터 인터페이스는 **SM35 Session** 방식을 채택하는 것이 실무의 아키텍처 정석입니다.
