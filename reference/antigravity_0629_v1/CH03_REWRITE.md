# CH03_REWRITE - DDIC Domain·Data Element + PARAMETERS v1

> 목적: `content/abap/CH03`의 3개 레슨을 "IT 비전공자 입문자가 왜 필요한지부터 실습 확인까지 직관적으로 이해하고 따라갈 수 있는 완성 강의자료" 수준으로 재작성한 기준 원고다. 이 문서는 아직 `content/abap` 원본을 덮어쓴 결과가 아니라, 재작업 판정 이후 실제 반영 전 검수 가능한 v1 기준안이다.

## CH03 전체 설계

CH03의 한 문장 목표는 "DDIC Domain 과 Data Element 의 1:N 상속 관심사 분리, DD01L/DD04L/DD07L 데이터베이스 사전 테이블 적재 기작, PARAMETERS 가 유발하는 1000번 Selection Screen PAI 입력 버퍼, OBLIGATORY 가동 시점, 그리고 LOWER CASE 의 대문자 강제 변환 필터링 물리 구조를 규명하여 전역 데이터 사전 모델링의 주춧돌을 세운다"이다.

IT 비전공자 입문자는 `DATA` 변수 선언 시 `TYPE zdomain_name` 처럼 도메인 명칭을 직접 연결했다가 컴파일 에러를 맞닥뜨리고, 선택 화면 `PARAMETERS` 선언 시 `LOWER CASE` 옵션을 빠뜨려 소문자 입력값이 PAI 변환 버퍼 단에서 대문자로 무단 강제 오버라이트되어 저장되는 정합성 훼손을 겪는다.
또한, `OBLIGATORY` 필수 제약이 일반 아바 비즈니스 로직 시점이 아닌 선택 화면 1000번의 PAI 수문장 게이트에서 기동되는 메커니즘을 알지 못해 제어 시점을 혼동한다.
따라서 본 챕터는 다음과 같은 단계적 해소 장치로 설계를 배치한다.

1. **도메인(DD01L) 물리 적재**: 순수 데이터 타입, 길이, 고정값(`DD07L`) 등 물리적 도면 스펙만 격리하여 데이터 사전 `DD01L` 에 저장 활성화하는 도메인 레이어 수립.
2. **데이터 엘리먼트(DD04L) 의미 결선**: 도메인의 물리 속성을 상속받고 그 위에 비즈니스 의미, 다국어 4단 필드 라벨(`DD04L`), Parameter ID를 얹는 데이터 엘리먼트 설계.
3. **1:N 공항코드 상속 실례**: 동일한 공항 3자리 도메인(`ZDO_AIRPORT`)을 출발 공항(`ZDE_FROM_AIRP`)과 도착 공항(`ZDE_TO_AIRP`) 데이터 엘리먼트가 상속해 다르게 사용하는 1:N 설계 명세.
4. **1000번 Selection Screen PAI**: `PARAMETERS` 선포 시 아바가 가설하는 1000번 선택 화면의 PAI(Process After Input) 입력 버퍼 기작 규명.
5. **OBLIGATORY 수문장**: 빈값일 때 PAI 버퍼 검문 시점에서 즉각 "Fill in all required fields" 메시지를 뿜고 실행을 강제 차단 롤백하는 게이트 수립.
6. **LOWER CASE 대문자 필터**: 소문자 입력이 대문자로 자동 훼손 오용되는 것을 막는 `LOWER CASE` 옵션 장착 가이드라인 수립.

## CH03-L01 - Domain — 기술 속성 정의

### 왜 필요한가

우리가 이전 CH02 에서 로컬 타입 선언인 `TYPES ty_amount TYPE p LENGTH 8 DECIMALS 2.` 를 통해 금액 변수들을 빚어냈다.
그런데 이 금액 타입을 1번 예매 확인 프로그램뿐만 아니라, 2번 정산 배치 프로그램, 3번 재무 회계 전표 프로그램에서도 똑같이 정의해서 쓰고 싶다면?
- "프로그램마다 똑같은 금액 타입을 소스 파일에 손수 중복 타이핑해 두면, 소수점 자리를 2자리에서 3자리로 변경하라는 본사 세법 지침이 내려올 때 수백 개의 소스 파일을 다 찾아 뜯어내다 정합성이 깨져 전산 마비가 온다."
프로그램 밖의 중앙 통제실에 단 한 번만 물리 규격 도면을 활성화해 박아두고, 모든 프로그램이 빨대를 꽂아 전역적으로 가져다 쓰게 만들어야 한다.
그 중앙 통제실이 **DDIC (ABAP Dictionary)** 이며, 그 안에서 순수 물리 규격만을 굳히는 도면이 **Domain (도메인)** 이다.

**데이터 사전 Domain 이 1바이트 형식 및 고정값(Fixed Values) 정보를 데이터베이스 시스템 테이블 `DD01L` 및 `DD07L` 에 적재하는 활성화 기작과, Domain 명칭을 DATA 변수에 직접 TYPE 으로 기입할 수 없는 컴파일 차단 제약을 수호하는 기술**이 필요하다. 그것이 **Domain 의 설계**다.

### 무엇인가

#### 1. DDIC (ABAP Dictionary - 데이터 사전)
- 서버 내부 데이터베이스 상에 프로그램과 독립적으로 전역 데이터 규격(도메인, 데이터 엘리먼트, 테이블)을 선포하고 보존하는 중앙 집중식 형상 정의 제어실이다. (트랜잭션 `SE11` 에서 기동한다.)

#### 2. Domain (도메인 - 물리 속성 도면)
- **비즈니스적 의미는 일절 배제한 채, 오직 데이터 타입(CHAR, NUMC, DEC 등)과 물리 길이(Length), 그리고 소수 자릿수(Decimals) 및 값의 한계를 규정하는 고정값(Fixed Values)만을 맑게 수호하는 물리 형틀이다.**

#### 🧭 [ Domain 데이터베이스 적재 및 런타임 캐시 아키텍처 명세 ]
- *도메인을 생성하고 활성화하는 찰나, SAP 백엔드에서는 아래와 같이 시스템 물리 테이블에 데이터가 인서트 적재된다.*

```text
[1] 도메인 생성 및 속성 정의 : 
   - ZDO_STATUS 라는 도메인을 CHAR 1글자, 고정값 'O'(Open) / 'C'(Closed) 로 빚는다.
   │
[2] 물리 테이블 적재 (SE11 저장 및 활성화) :
   - 도메인의 헤더 속성(CHAR, 1) -> DB 테이블 DD01L (Domain 헤더 테이블) 에 인서트 적재 완료!
   - 도메인의 고정값 목록 ('O', 'C') -> DB 테이블 DD07L (Domain 고정값 테이블) 에 인서트 적재 완료!
   │
[3] 런타임 컴파일 캐시 로딩 (Active Status) :
   - F3 활성화를 누르면, 아바 커널이 DD01L/DD07L 을 긁어서 고속 실행을 위한 런타임 캐시를 구축한다.
   - 이 활성 캐시 상태가 상주해야만, 다른 테이블이나 프로그램이 ZDO_STATUS 도면을 런타임에 찌를 수 있다.
```

#### ⚠️ [ Domain 의 DATA TYPE 직접 대입 금지 제약 ]
- *입문자가 100% 겪는 컴파일 에러다.*
- **도메인은 순수 물리 규격서일 뿐, 변수로 직접 기용할 수 없다.** 
  `DATA gv_status TYPE zdo_status.` 라고 적으면 아바 컴파일러는 **"ZDO_STATUS 는 도메인이므로 직접 데이터 오브젝트의 타입으로 사용할 수 없다"** 며 빌드를 거부한다. 
  변수를 빚을 때는 반드시 이 도메인 위에 비즈니스 의미와 텍스트 라벨을 한 겹 씌운 **Data Element (데이터 엘리먼트)** 를 경유해 `TYPE zde_status` 형태로 선언해야 함이 아바 DDIC 의 철칙 제약이다.

#### 3. CHAR ↔ c, DEC ↔ p (데이터 사전과 아바 프로그램의 이름 쌍)
- **도메인(DDIC)** 에서는 `CHAR`, `NUMC`, `DEC` 라는 영문 대문자 약속 명칭을 쓰고, 이 도메인을 상속받아 **아바 프로그램 변수**를 빚으면 커널에 의해 각각 `c`, `n`, `p` 타입으로 매핑 변환되어 램 메모리에 적재된다. (둘은 같은 물리 뿌리를 가진다.)

### 어떻게 확인하는가

SE11 에서 도메인을 생성하고 활성화 상태를 검증하는 단계를 수행한다.

```text
[1단계] SE11 도메인 생성 진입 :
   - Domain : ZDO_BOOK_STATUS (Z/Y 네임스페이스 규칙 준수!) -> Create 클릭!
   
[2단계] 물리 스펙 기입 :
   - DataType : CHAR (1바이트 고정문자 도면 지정)
   - No. Characters : 1 (1글자 크기 한도 제한)
   - Value Range 탭 진입 -> 고정값(Fixed Value) 셋업 :
     └- 'O' (설명: Open - 예매 대기)
     └- 'C' (설명: Cancel - 예매 취소)
     
[3단계] 저장 및 활성화 실행 :
   - [저장] 클릭 -> DD01L, DD07L 테이블에 Inactive 징표로 적재 완료!
   - [활성화 (Ctrl+F3)] 격발 -> 배지 상태가 'Active' 로 점등 성공! 
   - 런타임 캐시 배선 가동 완료! (Data Element 가 상속할 준비 성료!)
```

#### 체험/시뮬레이터 설계 (DDIC 도메인 활성화 기판)
- **프로세스 플로우**:
  1. 학습자가 [SE11 도메인 캔버스] 와 [DD01L DB 판넬], [DD07L DB 판넬] 을 본다.
  2. 캔버스에 [CHAR 1] 과 [고정값 O/C] 를 적고 [저장] 을 누르자, 두 판넬에 [비활성 회색 데이터] 가 인서트된다.
  3. 아바 프로그램에서 `ZDO_BOOK_STATUS` 를 찌르자 "Domain not active" 빨간불이 켜진다.
  4. [활성화] 단추를 누르자, 판넬이 초록색으로 활활 타오르며 런타임 캐시 기어가 회전하고 프로그램 참조 라인이 초록불로 연결되는 활성화 피드백을 감상한다.
- **상태 및 데이터**:
  - `Z/Y 로 시작하지 않는 AMOUNT 라는 이름으로 도메인을 생성하려 시도한 상태` -> 런타임 결과: `Name range violation. Z/Y namespace required` 하이라이트.
- **피드백**: 전역 도메인은 물리 속성(DD01L)과 고정값(DD07L) 테이블에 적재되며, 활성화를 단행해야 런타임에 뚫림을 명심한다.

### 실수/주의

- **자주 데이터 목록이 늘어날 소지가 다분한 값들(예: 콘서트 좌석 등급 S/A/B/C/D)을 도메인 내부 고정값(Fixed Value)으로 적어 하드코딩해 배포**:
  - 나중에 VIP 석이나 특석이 신설될 때마다 도메인을 매번 수정하고 전체 패키지를 재활성화하는 대대적인 형상 릴리즈를 단행해야 한다.
  - **목록이 유동적으로 바뀔 값은 고정값이 아닌, 일반 DB 테이블(`ZTABLE`)을 파서 값 테이블(Value Table) 외래키 결선으로 설계해야 안전함을 수호해야 한다.**

### 정리

- 전역 공유 데이터 규격을 선포하는 통제실은 **`DDIC`** (트랜잭션 SE11) 이다.
- **`Domain`** 은 데이터 타입, 길이, 고정값 등 오직 **`물리 기술 속성`** 만을 담아 `DD01L`/`DD07L` 테이블에 보존한다.
- 도메인 명칭은 프로그램 변수 선언 시 **`TYPE 뒤에 직접 기재할 수 없으며`**, 반드시 Data Element 를 타고 들어와야 한다.

## CH03-L02 - Data Element — 의미·라벨 입히기

### 왜 필요한가

물리 속성 도메인(`ZI_` 격인 ZDO)은 활성화했다.
이제 이 도면 위에 "이것은 예매자명이다", "출발 공항이다" 라는 비즈니스 의미와 텍스트 라벨을 물려주어야 한다.
- " 공항 3자리 코드라는 물리 규격(`CHAR 3`)은 출발지 공항이나 도착지 공항이나 똑같다.
만약 이 3자리 도메인을 하나 만들고, 그 위에 비즈니스 의미 구분 없이 프로그램에 가져다 쓰면 어떻게 될까?"
화면에 라벨을 그릴 때 출발 공항도 '공항코드', 도착 공항도 '공항코드' 라고 똑같이 렌더링되어 사용자가 어느 칸에 출발지를 적어야 할지 대혼란을 겪는 레이아웃 파손 장애를 초래한다.

**도메인의 물리 속성을 상속받으면서, 비즈니스 의미와 다국어 Field Label(짧음/중간/길게/헤더)을 격리 결선하여 `DD04L` 테이블에 적재하고, 1:N 공항코드 사례처럼 도메인을 고도로 재사용하는 기술**이 필요하다. 그것이 **Data Element 의 설계**다.

### 무엇인가

#### 1. Data Element (데이터 엘리먼트 - 비즈니스 의미)
- **도메인(Domain)이 제공하는 물리 데이터 속성 위에, 필드 라벨(Label)과 F1 도움말 문서 등 인간이 인지할 수 있는 '의미적 정보(Semantic Context)' 를 한 겹 입혀서, 실제 테이블 필드와 프로그램 변수 타입으로 바로 활용할 수 있게 완성해 주는 전역 타입 객체다. (네이밍 규칙 zde_ 또는 ZDE_ 로 시작한다.)**

#### 🧭 [ Domain 1 : Data Element N 상속 아키텍처 명세 ]
- *동일한 기술 도면을 여러 비즈니스가 물려받아 가칭 이름을 다르게 달아 쓰는 DDIC 의 정석 설계다.*

```text
       ┌─────────── [ Domain (물리 도면) ] ───────────┐
       │ Name : ZDO_AIRPORT                           │
       │ Spec : CHAR 3 (공항 3자리 코드 기술 정보)     │
       └──────────────────────┬───────────────────────┘
                              │
               ┌──────────────┴──────────────┐ ( 1 : N 상속 배선 결선 )
               ▼                             ▼
   [ Data Element A ]            [ Data Element B ]
   - Name : ZDE_FROM_AIRP        - Name : ZDE_TO_AIRP
   - Label : 출발 공항 (From)     - Label : 도착 공항 (To)
   - 역할 : 출발지 의미 주입       - 역할 : 도착지 의미 주입
```

#### 2. Field Label (길이별 4단 필드 라벨)
- *사용자 화면 UI 크기에 유연하게 대응하는 라벨 텍스트다.*
- **Short(10자)**, **Medium(20자)**, **Long(40자)**, **Heading(55자)** 의 4단 폭 텍스트를 기입해 둔다.
- **동작**: Fiori Element 표 컬럼 너비가 좁아지면 짧은 라벨('출발')을, 넓어지면 긴 라벨('출발 공항 코드')을 화면 엔진이 `DD04L` 번역 명세에서 알아서 선택해 유연 렌더링한다.

#### 3. Parameter ID (메모리 기억 보관 단추)
- 사용자 PC 세션 내에 글로벌 SAP 메모리 버퍼 영역(SET/GET Parameter)에 값을 박아둔다.
- **예**: 예약 화면을 닫고 다른 여객 화면을 켜도 마지막에 입력했던 공항코드 `SEL` 이 인풋 박스에 자동으로 물려 나오는 사용자 경험 편의 장치다.

### 어떻게 확인하는가

ZDO_AIRPORT 도메인을 상속받아 ZDE_FROM_AIRP 데이터 엘리먼트를 빚는 SE11 단계를 검증한다.

```text
[1단계] SE11 Data Element 생성 진입 :
   - Data Type : ZDE_FROM_AIRP -> Create 클릭!
   
[2단계] 도메인 배선 및 라벨 기입 :
   - Elementary Type 라디오 단추 선택 -> Domain 칸에 'ZDO_AIRPORT' 입력!
   - ( ZDO_AIRPORT 의 CHAR 3 속성이 ZDE_FROM_AIRP 로 자동 상속 매핑 완료! )
   - Field Label 탭 진입 -> 다국어 폭 기입 :
     └- Short (10)  : 출발
     └- Medium (20) : 출발지
     └- Long (40)   : 출발 공항 코드
     └- Heading(55) : 출발 공항 코드 명칭
     
[3단계] 저장 및 활성화 :
   - [활성화] 클릭 -> DB 테이블 DD04L (Data Element 테이블) 에 적재 완료!
   - 아바 에디터(SE38) 소스에서 `DATA gv_airport TYPE zde_from_airp.` 선언 가능 확인!
```

#### 체험/시뮬레이터 설계 (1:N 공항 상속 배선도)
- **프로세스 플로우**:
  1. 학습자가 [ZDO_AIRPORT 도메인 판넬 (CHAR 3)] 을 본다.
  2. 그 아래에 [ZDE_FROM_AIRP] 와 [ZDE_TO_AIRP] 데이터 엘리먼트 칩이 있다.
  3. 두 칩에서 도메인 판넬로 [상속 케이블] 을 쭉 꽂자, 두 칩 모두 3자리 문자 속성으로 충전 점등된다.
  4. 이때 ZDE_FROM_AIRP 에 [라벨 = 출발 공항] 을 치자, 화면 UI 판넬의 출발지 인풋 박스 라벨이 '출발 공항' 으로 이쁘게 변하는 다국어 매칭 피드백을 감상한다.
- **상태 및 데이터**:
  - `Field Label 데이터 텍스트 입력을 쌩으로 비워둔 채 활성화한 상태` -> 런타임 결과: `Validation warning. Blank label will cause empty column header in ALV` 하이라이트.
- **피드백**: Data Element 는 물리 속성 도메인을 상속하고, 그 위에 Field Label 이라는 비즈니스 외피를 덮어씌워 변수 타입으로 조형됨을 깨닫는다.

### 실수/주의

- **Data Element 를 새로 빚으면서 Field Label Field 의 '길이(Max Length)' 값 지정을 한글 자릿수와 맞지 않게 너무 짧게 적어 렌더링 시 뒤에가 잘려 나오는 오타 유발**:
  - Max Length 에 `2` 라고 적고 텍스트에 `출발공항` (4글자)을 적으면, 화면 상에서 글자가 싹 잘려 '출' 만 보이는 레이아웃 파손을 맞이한다.
  - **Max Length 숫자는 우측에 타이핑한 한글 문자 길이보다 넉넉하게 지정해 수호해야 한다.**

### 정리

- **`Data Element`** 는 전역 도메인의 물리 스펙을 참조 상속받아 비즈니스 의미와 라벨을 가미한 타입 객체다.
- **`Field Label`** 4단 텍스트(Short, Medium, Long, Heading)를 채워두면 화면 엔진이 너비에 맞춰 유연 렌더링한다.
- 동일 도메인을 여러 Data Element 가 상속해 다각도로 쓰는 **`1:N 재사용성`** 이 두 계층 분리 아키텍처의 기둥이다.

## CH03-L03 - PARAMETERS — 라벨·F4 자동 적용

### 왜 필요한가

Data Element 와 Domain 의 전역 형상 셋업을 완수했다.
이제 내 아바 프로그램에 대고 사용자가 값을 타이핑해 넘길 수 있는 '입력 인터페이스 화면(Selection Screen)' 의 설계가 마지막 장벽이다.
- " 프로그램이 시작되기 전에, 사용자로부터 '예약할 좌석 수' 나 '고객 코드' 를 키보드로 입력받을 인풋 필드 화면을 간결하게 구성하고 싶다.
이를 위해 `PARAMETERS` 구문을 쏠 때, 전역 데이터 엘리먼트(`ZDE_SEATS`)를 바인딩하면 어떤 이점들이 공짜로 제공되는가?"
입문자가 이 PARAMETERS 구문을 선포할 때 필수 입력 옵션인 `OBLIGATORY` 검문이 기동되는 시점을 몰라 엉뚱한 비즈니스 로직 단에서 빈값 에러 처리를 하드코딩하려 해 코드가 파손된다.
또한 소문자 보존 옵션(`LOWER CASE`)을 빠뜨려, 사용자가 소문자로 쳐서 넘긴 패스워드나 코드가 대문자로 자동 훼손 오버라이트 갱신되는 위협에 대책 없이 노출된다.

**PARAMETERS 선언을 통해 1000번 Selection Screen PAI 입력 버퍼 게이트를 가설하고, OBLIGATORY 수문장 필터를 PAI 단에 도킹하며, LOWER CASE 가드로 대문자 오용을 원천 차단하는 기술**이 필요하다. 그것이 **PARAMETERS 의 화면 제어**다.

### 무엇인가

#### 1. PARAMETERS p_seats TYPE zde_seats (입력 선택 화면 개설)
- **PARAMETERS**: **아바 런타임이 기동될 때 `START-OF-SELECTION` 본 로직으로 진입하기 직전, 화면 중앙에 입력칸과 실행(F8) 버튼이 있는 '1000번 선택 화면 (Selection Screen)' 을 자동으로 가설해 주는 화면 인터페이스 지시어다.**
- **p_ 접두어**: Parameters 의 약자로서, 접두어 `p_` 를 매겨 변수명과 구분 짓는 것이 실무 표준 Conventions 이다. (최대 8글자 제약이 있다.)

#### 2. Data Element 바인딩의 3대 공짜 보상
- **① 라벨 자동 조치**: 입력 박스 왼쪽에 `ZDE_SEATS` Field Label 에 작성해 둔 한글 명칭('예약 좌석')이 쌩 코딩 없이 자동으로 렌더링된다.
- **② F4 Search Help 자동 조치**: 입력 박스 옆에 돋보기(선택) 단추가 공짜로 붙으며, 마우스 클릭이나 F4 클릭 시 도메인 `DD07L` 에 용접된 고정값('O'/'C')이 모달 팝업으로 촥 렌더링된다.
- **③ 자동 입력 검증**: 고정값 이외의 쓰레기 값을 치고 엔터를 때리면, 아바가 "Entry ... not valid" 경고를 뿜어 애초에 프로그램 내로 쓰레기 값이 침투하는 것을 차단한다.

#### 🧭 [ Selection Screen PAI 이벤트 버퍼 아키텍처 명세 ]
- *사용자가 F8 실행을 누르는 그 순간, 백엔드 PAI 게이트에서는 아래와 같이 커널 입력 검문이 기동된다.*

```text
사용자가 p_seats 에 '10' 입력 후 [F8 (실행)] 클릭!
   │
   ▼ [ 1000번 Selection Screen PAI (Process After Input) 검문소 진입 ]
   │
[OBLIGATORY 검문소] : p_seats OBLIGATORY 선언 여부 판정!
   - p_seats 가 비어 있으면, 메시지 'Fill in all required fields' 를 방출하며 
     본문 START-OF-SELECTION 로직 진입을 강제 차단 롤백한다.
   │
[LOWER CASE 검문소] : p_name LOWER CASE 선언 여부 판정!
   - ( 선언 누락 시 ) : 소문자 'sel' -> 아바 대문자 치환 테이블을 통과하여 'SEL' 로 오버라이트 갱신!
   - ( 선언 장착 시 ) : 소문자 'sel' -> 대문자 치환 필터를 바이패스하여 'sel' 날것 그대로 보존 저장!
   │
   ▼ [ 검문 통과 ] -> 비로소 START-OF-SELECTION (본 로직 기동!)
```

### 어떻게 확인하는가

OBLIGATORY 와 LOWER CASE 가 포함된 PARAMETERS 선택 화면을 격발해 동작을 검증한다.

```abap
REPORT zhello_parameters.

" 1. [ Data Element zde_amount 를 바인딩하여 라벨/F4 자동 획득! ]
"    필수 값 입력 지정 (OBLIGATORY)
PARAMETERS p_amt TYPE zde_amount OBLIGATORY.

" 2. [ 기본값 DEFAULT 지정 및 대문자 강제 치환 필터를 바이패스할 LOWER CASE 장착! ]
PARAMETERS p_user TYPE c LENGTH 12 DEFAULT 'JeongHy' LOWER CASE.

START-OF-SELECTION.
  " 3. [ 1000번 PAI 검문을 무사 통과한 무결한 입력값 출력 ]
  WRITE: / '입력된 결제 금액:', p_amt,
         / '입력된 소문자 유저:', p_user.
```

#### 체험/시뮬레이터 설계 (Selection Screen PAI 버퍼)
- **프로세스 플로우**:
  1. 학습자가 [1000번 선택 화면 UI] 와 [PAI 입력 검문 라인] 을 본다.
  2. `p_amt` 에 값을 비워두고 [F8 실행] 을 누른다.
  3. [OBLIGATORY 센서] 가 삑 경보를 울리며 "OBLIGATORY check failed! Entry required!" 적색 차단바가 내려오고 본문 로직 실행이 원천 멈춘다.
  4. 값을 채우고, `p_user` 란에 소문자 `jeong` 을 친 뒤, [LOWER CASE = OFF] 면 검문소 통과 시 글자가 `JEONG` 대문자로 오용 변조되는 모션을 감상한다.
  5. [LOWER CASE = ON] 스위치를 켜자, 소문자 jeong 그대로 통과해 본문에 초록불이 켜지는 정합성 피드백을 감상한다.
- **상태 및 데이터**:
  - `PARAMETERS 이름을 8글자를 초과하여 ZBOOKING_ID 로 너무 길게 선언한 상태` -> 런타임 결과: `Syntax error. Parameters name must not exceed 8 characters` 하이라이트.
- **피드백**: PARAMETERS 는 1000번 선택 화면 PAI 버퍼 검문을 생성하는 강력한 이중 입력 가드이며, 소문자 보존을 위해선 LOWER CASE 가 배선되어야 함을 체득한다.

### 실수/주의

- **선택 화면 입력칸 옆에 나오는 라벨 텍스트를 수동으로 바꾼답시고 쌩 Text Element (Selection Texts) 화면에 들어가 PARAMETERS 라벨 명칭을 억지로 오버라이트 하드코딩해 저장**:
  - 이 실수 시 나중에 전역 Data Element 의 Field Label 이 개정되더라도, 이 프로그램의 입력칸 라벨만 홀로 옛날 글자로 굳어 안 바뀌는 형상 불일치가 터진다.
  - **특별한 경우가 아니라면, Selection Texts 에서 [Dictionary Reference] 체크박스를 켜서 Data Element 의 라벨이 자동으로 밀려 내려오게 연동 수호해야 한다.**

### 정리

- **`PARAMETERS`** 는 본문 실행 전 입력 필드 한 칸을 1000번 **`Selection Screen`** 에 가설하는 지시어다.
- Data Element 바인딩 시 **`Field Label`**, **`F4 Search Help`**, **`입력값 검증`** 의 3대 혜택이 공짜로 장착된다.
- **`OBLIGATORY`** 는 1000번 PAI 시점에서 필수 값 누락을 선제 차단한다.
- 소문자 알파벳 날것의 입력을 보존하기 위해선 반드시 **`LOWER CASE`** 옵션을 수반한다.
- PARAMETERS 변수명은 문법 제약 상 **`최대 8글자 이내`** 로 개설해야 덤프를 피한다.
