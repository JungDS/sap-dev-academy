## 4. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. Help View 와 Maintenance View 는 왜 프로그램 소스 코드의 SELECT(SQL) 조회 문장의 FROM 뒤에 써서 데이터를 읽어올 수 없나요?
**A**: **두 뷰는 데이터를 조회하는 SQL 소스(Data Source)가 아니라, 오직 F4 도움말의 화면 레이아웃과 SM30 입력 폼의 '메타데이터 뼈대 계약서'로만 활약하도록 설계되었기 때문**입니다.
이것은 SAP의 철저한 역할 분리 정책입니다.
- **`Database View` / `Projection View`**: 활성화 시 실제 데이터베이스 레벨에 SQL View 객체가 생성되어 DB가 직접 조인을 처리해 결과를 던져줍니다. 따라서 SELECT 조회가 가능합니다.
- **`Help View` / `Maintenance View`**: 데이터베이스에 실물 뷰 객체로 생성되지 않고, 오직 SAP NetWeaver Application Server의 사전(Dictionary) 안에만 가상의 뼈대로 존재합니다. 이들은 각각 Search Help 팝업창을 켤 때나, SM30 화면을 켤 때 시스템 프레임워크가 내부적으로 외래 키를 체크하고 화면 레이아웃을 엮어주기 위한 **화면 매핑 도구**일 뿐입니다. 데이터베이스가 그 실체를 모르기 때문에, SQL SELECT 문으로 읽으려고 시도하면 아예 컴파일러 단에서 조회를 차단당하게 됩니다.

### Q2. Database View는 INNER JOIN으로 엮이고, Help View는 Outer Join으로 엮인다고 배웠습니다. F4 도움말 화면을 구축할 때 이 조인 방식 차이가 초래하는 치명적인 유실 버그 시나리오는 무엇인가요?
**A**: **번역 설명 텍스트가 누락되었다는 이유로, F4 팝업창 리스트에서 '공연 코드 자체가 완전히 실종되어 골라지지 않는' 업무 마비 버그를 초래합니다.**
공연 회차 코드 `P001` (주 테이블)과 공연 한글 설명 **"정훈영 갈라콘서트"** (보조 테이블)를 엮어 도움말 뷰를 짠다고 해보자.
어느 날 바쁜 업무 중에 신입 사원이 보조 테이블에 `P001` 의 설명 텍스트를 기입하지 않고 누락시켰습니다.
- **INNER JOIN (`Database View` 등)**: 두 테이블에 공통 짝이 다 존재해야 하므로, 텍스트가 없는 `P001` 은 결과에서 통째로 지워집니다. 그에 따라 사용자가 화면에서 F4를 누르면 **`P001` 이라는 코드 자체가 F4 검색 창에 나타나지 않게 됩니다.** 사용자는 코드가 존재함에도 F4에 없으니 코드가 삭제된 줄 오해하는 장벽을 겪습니다.
- **Outer Join (`Help View` 특성)**: 짝이 없더라도 주 테이블의 `P001` 은 무조건 보존되어 F4 검색 목록에 잔류합니다. 비록 설명 이름 칸은 빈칸(초기값)으로 허전하게 보일지언정, 사용자는 `P001` 코드를 마우스로 안전하게 더블클릭해 선택하여 화면에 수납할 수 있습니다. 
따라서 도움말을 제공할 때는 무조건 **Help View**를 소스로 삼아야 데이터 증발 대참사를 방어할 수 있습니다.

### Q3. Table Maintenance Generator (TMG)를 빚어낼 때 One Step과 Two Step 화면 유형을 현업에서 결정하는 명확한 업무적 공식 기준이 무엇인가요?
**A**: **"한 행의 가로 너비(필드 개수)가 표준 모니터 화면 안에 스크롤 없이 한눈에 들어오는가" 에 따라 결정합니다.**
- **`One Step` (1단계 - 목록 화면 1개)**:
  - **결정 공식**: 테이블의 필드 개수가 2~5개 내외로 아주 2~3칸 수준에 불과할 때 고릅니다.
  - **작동**: 엑셀 시트처럼 격자 표 화면 1개만 덜렁 나옵니다. 사용자는 엑셀 채우듯 그 목록 격자 칸에 직접 데이터를 타이핑해 저장합니다.
- **`Two Step` (2단계 - 목록 + 상세 폼 화면 2개)**:
  - **결정 공식**: 테이블의 필드가 10~20개가 넘어, 가로 스크롤바를 한참 당겨야만 컬럼을 볼 수 있을 정도로 필드가 많을 때 고릅니다.
  - **작동**: 1번 화면에서는 목록 개요만 요약하여 좁게 보여주고, 사용자가 수정할 행을 마우스로 더블클릭하면 2번 화면인 **독립 상세 카드 폼(Form Screen)** 이 열려, 위에서 아래로 정렬된 넓은 입력 필드 카드를 보며 꼼꼼하게 편집을 수행합니다.
필드가 엄청나게 많은데 욕심내어 One Step으로 만들면 현업 사용자가 가로 스크롤을 하다가 옆 필드를 누락해 저장하는 데이터 유실 실수가 빈발하므로, 필드 수량을 쳐다보고 단계를 나누어야 합니다.

### Q4. 존재하지 않는 가짜 외래 키 코드 입력을 데이터베이스(DB)단이 아닌 데이터 사전(DDIC) 수준에서 완벽하게 차단할 수 있는 내부 작동 원리(Foreign Key 와 Input Check)는 무엇인가요?
**A**: **SAP GUI 화면 프레임워크가 SM30 화면의 PAI(Process After Input) 제어 단계에서, 필드의 Foreign Key 관계 사전을 확인하여 자동으로 상위 테이블에 SELECT SINGLE 조회를 실시간 대조해 튕겨 내주기 때문입니다.**
이것은 DB 엔진이 체크 제약을 수행하는 것이 아닙니다. 
우리가 `ZCONCERT-HALL_ID` 필드에 `ZHALL` 테이블을 Foreign Key로 걸어두면, 시스템 사전에는 **"ZCONCERT-HALL_ID 필드에 값이 입력되는 순간, ZHALL-HALL_ID 에서 해당 값이 있는지 확인하라"** 는 엄격한 메타데이터 조약서가 기입됩니다.
사용자가 SM30 화면에서 `H999` 를 치고 엔터를 눌러 백엔드로 데이터를 유통하려고 할 때, 화면 제어판 모듈은 이 조약서를 자동 스캔하여 데이터베이스의 `ZHALL` 테이블로 **`SELECT SINGLE hall_id FROM zhall WHERE hall_id = 'H999'`** 조회를 물밑에서 강제로 전송합니다. 
여기서 데이터가 검색되지 않고 `sy-subrc = 4` 실패가 감지되면, 화면은 즉시 다음 연산 단계를 차단하고 화면을 붉은색 에러 락으로 잠근 채 오류를 띄웁니다. 이 메타데이터 기반 자동 검증 체인을 **Input Check** 라고 부릅니다.

### Q5. 개발자들이 테이블 상태를 볼 때 가장 애용하는 SE16N 데이터 브라우저를 현업 사용자의 정식 조회 화면 대안으로 릴리즈하여 사용하게 하면 안 되는 치명적 보안/시스템적 위험성이 무엇인가요?
**A**: **비즈니스 프로그램 수준에 심어둔 모든 '권한 체크(Authority Check)' 와 '데이터 입력 정합성 검증 가드'가 100% 무력하게 우회(Bypass)되어, 인사/재무 데이터 유출 및 DB 데이터 깨짐 대참사가 터지기 때문입니다.**
SE16N은 테이블의 생살을 날것으로 노출하는 개발자용 긴급 진단 메스(Scalpel)입니다.
1. **정합성 검증 우회**: 현업 사용자가 SE16N 편집 기능이나 조건 우회를 작동하여 데이터를 건드리면, 우리가 프로그램 소스 상에 정교하게 짜놓은 "잔여석 계산 후 정원 초과 시 예매 거절" 과 같은 비즈니스 로직 가드가 단 한 줄도 실행되지 않고 DB에 다이렉트로 가짜 좌석 수가 쓰여 저장됩니다. (결국 공연 데이터 정합성이 찢어져 파괴됩니다).
2. **보안 권한 체크 우회**: 정식 프로그램(SALV 등)은 백엔드에서 "이 사용자는 서울 공연장 데이터만 볼 수 있는 권한(`AUTHORITY-CHECK`)이 있다" 고 걸러서 안전하게 서빙하지만, SE16N은 테이블을 쌩으로 열어버리므로 사용자가 부산, 광주 등 전국 공연 데이터와 심지어 동료의 급여 원장 정보까지 싹 엑셀로 유출해 가는 보안 사고가 유발됩니다.
따라서 SE16N은 철저히 개발자 전용 돋보기로 숨겨두고, 사용자에겐 안전한 전용 프로그램 화면만 배포하는 것이 아키텍처적 금지선입니다.
