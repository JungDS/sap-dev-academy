## 3. 공식 문서 수동 근거 명세

`CL_SALV_TABLE` 및 `CL_GUI_ALV_GRID` 와 관련하여 SAP 표준 컨트롤의 시각 조종 사양을 철저히 검증했다.

- **헤더 텍스트 개칭 규칙**:
  SALV의 `get_column( )` 은 런타임의 컬럼 폭 픽셀(Pixel) 값의 동적 변화에 대응하기 위해 내부적으로 short(10자), medium(20자), long(40자) 세 영역의 버퍼를 가동한다. 이 중 단 1개만 개칭하면 화면 좁아짐/넓어짐 시 원래 DDIC 데이터 딕셔너리의 이름표 텍스트로 롤백 복원되는 물리 사양 확인 및 해결 가이드 반영 완료.
- **`LVC_T_SCOL` 과 `LVC_T_STYL` 의 스펠링 불일치 트랩**:
  셀 색상을 엮는 `LVC_T_SCOL` 구조체 행의 컴포넌트는 타깃을 지목할 때 **`fname`** 필드를 가동하며, 셀 스타일(동작 락, 버튼화)을 엮는 `LVC_T_STYL` 구조체 행의 컴포넌트는 **`fieldname`** 필드를 가동한다. 이 물리 필드명의 스펠링 불일치 한계를 기재해 개발자 오작동 예방 완료.
- **Layout ctab_fname 및 stylefname 연동선 매핑 의무**:
  행 내부(outtab)에 아무리 deep 컴포넌트로 데이터를 주입했어도, Layout 구조체의 식별 속성 문자열에 해당 자식 필드 이름(`'CELLCOLORS'`, `'CELLSTYLES'`)을 문자열 대문자로 연동해 주지 않으면 ALV가 이를 무시하는 사양 검증 및 소스화 완료.
- **Stable Refresh 와 soft refresh 의 2단계 공정**:
  `refresh_table_display` 는 DB를 재조회(Re-select)하지 않으며, 오직 아바 메모리 내부 테이블(outtab)의 값의 물리 변화만 감지해 화면을 튕김 없이(`is_stable-row/col = X`) 고쳐 그리는 메모리 동기화 제어 구조임을 명세 완료.

## 4. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. get_column( ) 에서 set_long_text 로 한글 제목을 '정원' 으로 고쳤는데, 왜 화면 컬럼 폭을 마우스로 좁히면 원래 영어 이름인 'CAPACITY' 로 롤백 복원되나요?
**A**: **SALV 엔진이 컬럼 너비 축소에 대응하기 위해 헤더 텍스트를 자동으로 교체 투사하는데, 좁아진 공간용 텍스트(short/medium)가 여전히 옛날 영문 버퍼로 방치되어 있어 이를 강제 호출하기 때문입니다.**
SALV는 컬럼 너비가 좁아지거나 넓어질 때 글자가 잘려 시각 공해가 일어나는 것을 방지하기 위해 내부적으로 3단 헤더 버퍼를 굴립니다.
- 너비가 40자 이상으로 넉넉할 때: **`long_text`** 투사
- 너비가 20자 전후일 때: **`medium_text`** 투사
- 너비가 10자 이하로 극단적으로 좁을 때: **`short_text`** 투사
개발자가 `set_long_text( '정원' )` 만 고치면, 화면이 넓을 때는 이쁘게 '정원' 으로 나오지만, 사용자가 마우스 드래그로 너비를 좁혀 `medium` 이나 `short` 임계값 안으로 컬럼 폭이 들어오는 순간, 미처 고쳐두지 않은 `CAPACITY` 영문 이름표가 화면에 교체 렌더링됩니다.
**해결책**: 따라서 헤더명을 커스텀 작명해 줄 때는 귀찮더라도 **`set_short_text`**, **`set_medium_text`**, **`set_long_text`** 의 3가지 메서드를 동시에 한 세트로 묶어서 동일한 칭호로 싹 수정 코딩해 두어야 어떤 폭 변화에도 한글 칭호가 견고히 사수됩니다.

### Q2. 셀 색상의 LVC_T_SCOL 은 컴포넌트명이 fname(5글자)이고, 셀 스타일의 LVC_T_STYL 은 fieldname(9글자)인 이유는 무엇이며, 이를 헷갈리면 어떻게 되나요?
**A**: **SAP 가 역사적으로 셀 색상 구조체와 셀 스타일 구조체를 각기 다른 버전과 개발 부서에서 설계하여 표준 필드명을 통일하지 않고 출시했기 때문이며, 헷갈리면 컴파일 에러를 맞거나 시각 효과가 묵살됩니다.**
- **`LVC_T_SCOL` (셀 색상)**:
  행 내부의 자식 행 타입인 `LVC_S_SCOL` 은 타깃 기둥 이름을 **`fname`** 이라는 필드에 적기로 약속되었습니다.
- **`LVC_T_STYL` (셀 스타일)**:
  반면 자식 행 타입인 `LVC_S_STYL` 은 타깃 기둥 이름을 **`fieldname`** 이라는 필드에 기재하기로 규정되었습니다.
이 철자 불일치를 모르는 채 `LVC_T_STYL` 을 조립하면서 `fname = 'SEATS'` 라고 적으면, 컴파일러는 "스타일 구조체에는 fname 이라는 컬럼이 존재하지 않는다" 며 문법 에러를 터트려 빌드를 멈춥니다. 
반대로 색상에 `fieldname` 을 기입하면 문법 에러는 안 나지만 ALV 엔진이 타깃 컴포넌트를 식별하지 못해 런타임에 색상이 투사되지 않고 회색으로 묵살되는 오동작을 낳습니다. 따라서 이 철자 차이는 두뇌 속에서 명확히 격리 기재해 두어야 합니다.

### Q3. refresh_table_display 를 가동했는데도 왜 화면의 잔여석 숫자가 0.001% 도 바뀌지 않고 그대로 멈춰 있나요? 제가 무엇을 빼먹었나요?
**A**: **리프레시 메서드는 스스로 데이터베이스(DB) 테이블을 재조회해 오는 자동 조회 장치가 아니므로, 갱신 전에 내 프로그램 메모리의 내부 테이블(Outtab) 알맹이 값을 먼저 코드로 수정해 두는 1단계 과정을 누락했기 때문입니다.**
`go_grid->refresh_table_display( )` 가 실행될 때, ALV 엔진은 단지 **"내가 화면에 띄우고 있는 gt_outtab 이라는 내부 테이블의 현재 값을 보고 화면 모니터의 픽셀을 다시 그려라"** 는 그리기 동기화만 집행합니다.
따라서 DB 단에서 예약을 추가했더라도, 내 아바 프로그램 내부 테이블(`gt_outtab`)의 잔여석 값을 코드로 고치지 않은 채 리프레시만 지르면, ALV 는 변경되지 않은 옛날 램(RAM) 데이터를 다시 복제해 그릴 뿐이므로 화면은 미동도 하지 않습니다.
**해결 철칙 (2단계 순서)**:
- **1단계**: 먼저 `READ TABLE gt_outtab ASSIGNING ...` 이나 루프를 통해 내 프로그램 내부 테이블의 잔여석 값을 DB 최신 수량으로 계산 수정해 바꿉니다.
- **2단계**: 그 아랫줄에 비소로 `refresh_table_display` 를 호출해야 화면의 픽셀이 갱신되어 반영됩니다. 이 순서를 머릿속에서 절대로 뒤섞지 말아야 합니다.

### Q4. 내부 테이블 행 구조에 cellcolors 필드를 품고 열심히 색 코드를 APPEND 했는데, 왜 화면에는 여전히 흰색/회색으로만 나오고 색이 전혀 안 나오나요?
**A**: **아무리 행 내부에 색상 방을 이쁘게 꾸며두었어도, Layout 구조체의 ctab_fname 속성에 "CELLCOLORS 라는 변수명이 색상 정보방이다" 라고 알려주는 매핑 연동선을 연결하지 않아 연결 누수가 났기 때문입니다.**
행 내부에 자식 테이블로 품은 `cellcolors` 필드는, Layout 연결선이 엮이기 전까지는 ALV 입장에서 그저 '화면에 노출해서는 안 될 이상한 입체 데이터 열' 일 뿐입니다. ALV 가 이 필드가 색상 정보가 담긴 특수방임을 해독하게 하려면,
**`ls_layout-ctab_fname = 'CELLCOLORS'.`** 
라는 도킹 선로를 명시적으로 연결해 주어야 합니다. 
이 선로 연결이 완료되어야만, ALV 엔진이 화면을 그리기 직전에 "아, 이 행의 cellcolors 구역으로 들어가서 fname 에 적힌 컬럼과 색 코드를 꺼내와 픽셀을 색칠해야겠구나" 라고 해독 가동을 시작합니다. 이 매핑은 양방향 체결 선로이므로 절대 빠뜨려서는 안 됩니다.

### Q5. no_out = abap_true 로 설정해 화면에서 컬럼을 숨겼는데, 왜 이 컬럼을 민감한 예매 금액 정보 등을 가리는 보안(Security) 가드로 사용하면 위험한가요?
**A**: **no_out 은 단지 '최초 화면 기동 시 얌전하게 숨겨두는 비주얼적 미적 기능' 일 뿐이며, 사용자가 ALV 상단 툴바의 레이아웃 설정 단추를 클릭해 언제든 그 기둥을 다시 화면에 노출해 볼 수 있기 때문입니다.**
필드 카탈로그의 `no_out` 속성은 컬럼을 물리적으로 파괴해 소멸시키는 것이 아닙니다. 
사용자는 프로그램 실행 중 상단 툴바의 `Layout 변경` 단추를 누르고, `숨김 컬럼 목록` 에서 언제든 해당 금액 기둥을 오른쪽 `표시 컬럼 목록` 으로 마우스 드래그해 넘겨 화면에 띄울 수 있고, 이를 Variant로 저장해 두고두고 볼 수 있습니다.
따라서 사용자가 봐서는 안 될 극비 보안 데이터를 `no_out` 에 기대어 숨겨두면 심각한 정보 노출 보안 사고를 초래합니다.
**보안 가이드**: 사용자 권한 상 가려야 할 데이터라면, 필드 카탈로그에서 숨기는 꼼수를 쓰지 말고, **최초 데이터베이스 `SELECT` 쿼리문 단계에서 해당 컬럼 기둥을 아예 목록에서 누락시켜 아바 서버 램 메모리로 가져오지 않는 것만이 유일한 보안 격리 방어선**입니다.
