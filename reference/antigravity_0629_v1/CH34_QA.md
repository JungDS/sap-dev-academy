## 3. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. Smart Forms 디자인을 다 마치고 활성화했습니다. 그런데 왜 아바 소스 프로그램에서 /1BCDWB/SF00000032 와 같은 함수 이름을 직접 선언해 호출(하드코딩)하면 안 되나요?
**A**: **이 함수명 꼬리의 숫자 일련번호는 각 서버 장비(개발, QA, 운영)에 업로드된 오브젝트 개수와 컴파일 활성화 순서에 따라 무작위로 계속 뒤바뀌기 때문이며, 하드코딩 시 운영 배포 시 100% 함수 부재 덤프를 터트리고 사망합니다.**
- **동적 명명 메커니즘**:
  Smart Forms 는 레이아웃 텍스트와 노드를 컴파일할 때, 뒤쪽에서 임시 Function Module 을 동적으로 자동 생성합니다.
  개발 장비에선 이 폼의 이름이 `/1BCDWB/SF00000032` 였을지 몰라도, QA 장비로 넘어가면 이전에 활성화된 다른 폼들이 많아 `/1BCDWB/SF00000087` 로 다른 번호표를 쥐게 됩니다.
  아바 소스에 32번을 박아둔 채 운영에 릴리즈하면, 시스템은 32번 함수를 찾지 못해 "Call of non-existent function" 치명적 컴파일 덤프를 터트립니다.
- **해결**:
  반드시 `SSF_FUNCTION_MODULE_NAME` API 를 경유해 디자인한 폼 명칭(`ZSF_TICKET`)을 파라미터로 넘겨주고, 현재 기계가 실시간으로 발급해 둔 진짜 동적 함수명(`lv_fm`)을 구득하여 **`CALL FUNCTION lv_fm`** 의 동적 지시어로 깨워야 서버 환경과 무관하게 100% 안전하게 동작합니다.

### Q2. Adobe Forms 를 빌드해서 호출을 돌렸는데, PDF 장표가 전혀 나오지 않고 Connection Refused 통신 에러가 뿜어집니다. 이 현상의 근본 원인이 무엇이며 무엇을 점검해야 하나요?
**A**: **Adobe Forms 는 아바 WAS 가 직접 렌더링하지 않고 외부 자바 ADS (Adobe Document Services) 데몬 서버와의 실시간 네트워크 통신망을 타고 변환해 오기 때문이며, SM59 Destination ADS 연결 상태를 점검해야 합니다.**
- **렌더링 메커니즘**:
  Smart Forms 와 달리, Adobe Forms 는 PDF 바이트 파일을 구워내기 위해 자바 웹 애플리케이션인 **ADS** 에 데이터를 XML 로 던져 렌더링을 집행합니다.
  따라서 이 자바 ADS 서버의 전원이 꺼져 있거나, SAP 가 ADS 와 통신할 때 사용하는 RFC Destination인 `ADS` (`SM59` HTTP 연결)의 인증 ID/PW 정보가 만료되었거나, 방화벽이 차단되면 렌더링 통로가 즉사합니다.
- **조치 시퀀스**:
  즉시 `SM59` 트랜잭션을 켜고, Connection Type 'G' 에 대고 **`ADS`** 연결 테스트(Connection Test)를 찔러 자바 데몬의 응답 신호(HTTP 200 OK 등)가 무사히 도킹해 오는지 네트워크 통신 전산선부터 진단해야 수습을 시작할 수 있습니다.

### Q3. Adobe Forms 를 호출하는 아바 프로그램을 코딩하고 있습니다. 왜 FP_JOB_OPEN 과 FP_JOB_CLOSE 함수가 소스 상에서 무조건 1:1 세트로 열리고 닫혀야 하나요?
**A**: **이 1:1 쌍을 닫아주지 않고 누수(Leak) 시키면, 외부 ADS 서버와의 통신 세션 커넥션 풀이 반환되지 않고 고착되어 100건 발송을 넘기는 찰나 사내 ADS 엔진 서버 전체가 다운되는 대재앙을 유발하기 때문입니다.**
- **세션 누수의 공포**:
  `FP_JOB_OPEN` 은 자바 ADS 서버에게 "지금부터 PDF 를 구울 테니 전용 통신 채널을 1개 확보해 열어두라" 는 신호탄입니다.
  프로그램이 끝나거나 에러가 났을 때 `FP_JOB_CLOSE` 로 "사용이 완료되었으니 채널 뚜껑을 닫고 자원을 반환한다" 고 신고를 안 해주면, DB 파일 핸들 누수처럼 통신 소켓이 램 상에 좀비 상태로 누적 고착됩니다.
  동시 결제량이 조금만 몰려도 커넥션 풀(Connection Pool)이 100% 꽉 차서 정산 인쇄망이 멎어 버려 온 회사의 인쇄기와 전송 메일 발급이 중단됩니다.
  **철칙**: TRY-CATCH 문이나 예외 탈출로 구문 꼬리에는 반드시 무슨 일이 있어도 **`FP_JOB_CLOSE`** 가 격발되도록 감싸 매치해야 안전이 수호됩니다.

### Q4. PDF XSTRING 2진 바이트 데이터를 gui_download 로 로컬 PC 폴더에 내려받아 저장했는데, PDF 뷰어를 열었더니 '손상된 파일 포맷이라 열 수 없다' 며 하얗게 깨진 파일이 나옵니다. 무엇이 잘못되었나요?
**A**: **파일 다운로드 시 filetype 을 ASC 나 DAT 같은 텍스트 모드로 때리면, OS 송수신기가 파일 내의 개행 바이트 문자열을 텍스트로 오역 변조하여 바이너리 오프셋을 영구 파손하기 때문이며, 반드시 filetype = 'BIN' 과 bin_filesize = xstrlen 지정을 장착해야 합니다.**
- **이진 바이트 파손 원인**:
  PDF 파일은 압축된 바이너리(`XSTRING`) 구조체입니다. 
  이를 텍스트 파일 복사하듯 `filetype = 'ASC'` 로 쏘면, 윈도우나 리눅스 OS 단의 파일 라이터가 "아, 텍스트 데이터구나" 하고 개행 바이트(`CRLF`)를 만날 때마다 Unicode 인코딩 변환을 자기 멋대로 먹이거나 뒤를 자르는 만행을 저지릅니다.
  파일의 16진수 암호 키가 뒤틀려 결국 PDF 뷰어가 헤더 포맷을 읽지 못하고 손상 판정을 내버립니다.
- **정석 방어선**:
  바이트 원본이 다치지 않게 무조건 **`filetype = 'BIN'`** (바이너리 모드)를 장착하여 가공 없는 순수 이진 비트 이송을 수호하고, **`bin_filesize`** 에 `xstrlen( lv_pdf_data )` 로 정확한 물리 바이트 길이를 대어 꼬리표를 채워 보내야 온전한 PDF 가 디스크에 안착됩니다.

### Q5. 고객이 예매를 저장했는데 티켓 자동 인쇄물이 안 나온다고 긴급 에러가 들어왔습니다. 폼 레이아웃 ZADOBE 디자인 소스를 디버깅해 뜯어보기 전에 무엇을 먼저 점검해 책임을 분리해야 하나요?
**A**: **인쇄 대기실 모니터인 SP01 (스풀 모니터)을 켜서 스풀 문서가 발행 완료(Complete) 상태로 대기 중인지 조회해야 하며, 스풀 상태를 통해 아바 로직 문제인지 프린터 물리 장비 문제인지를 단 3초 만에 시각적으로 칼 격리할 수 있기 때문입니다.**
- **책임 격리 시퀀스**:
  1. 즉시 **`SP01`** 에 들어가 내 사용자 ID 와 전표 저장 시간으로 인쇄 스풀 목록을 조회합니다.
  - **상황 A (스풀 목록 자체가 0건)**:
    아바 소스 코드나 Output Control(`BRF+`/`NAST`) 조건 배선이 먹통이 되어 폼 자체를 깨우지 못했거나 에러 덤프가 난 상태입니다. (아바 개발팀이 소스 코드를 열고 디버깅을 시작해야 합니다.)
  - **상황 B (스풀 목록은 Status = 'Compl.' 완료 상태로 이쁘게 떠 있음)**:
    아바와 ADS 서버는 100% 무결하게 PDF 를 구워 프린터 큐 전송까지 정상 종결했습니다. 
    단지 프린터 기기에 인쇄 종이가 다 떨어졌거나, IP 선이 빠졌거나, 현장 물리 프린터 하드웨어 고장이 100% 확실합니다. (아바 소스는 죄가 없으므로 현장 인프라팀에 프린터 수리 헬프를 던지고 롤백 탈출하면 됩니다.)
  - **결론**: 이 **스풀 징검다리 판독** 하나로 수십 시간의 아바 소스 삽질 분석 낭비를 사전에 철통 방어할 수 있습니다.
