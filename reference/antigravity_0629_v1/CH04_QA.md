## 3. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. 아바에서 숫자를 0 으로 나눴을 때 발생하는 런타임 에러(Short Dump)의 정체는 무엇이며, 개발할 때 어떻게 완벽히 예방하나요?
**A**: **분모가 0 일 때 나눗셈을 실행하면 아바 커널이 CX_SY_ZERODIVIDE 라는 예외를 터트려 프로그램을 강제 셧다운(Short Dump)시키므로, 나눗셈 직전 라인에 `IF 분모 <> 0.` 검문소 가드를 도킹해 사전에 우회 처리해야 합니다.**
- **동작 원인 및 예방 (Zero Division)**:
  컴퓨터 연산 장치는 수학적으로 정의되지 않는 0 으로 나누기(Division by Zero)를 만나는 즉시 동작을 정지합니다.
  아바 커널은 이 찰나에 **`CX_SY_ZERODIVIDE`** 라는 치명적 예외를 던집니다.
  이를 방어하는 코드를 소스 상에 짜두지 않았다면, 프로그램은 강제 격리 종료(Short Dump)되어 데이터 유실을 낳습니다.
  - **예방 규칙**:
    나눗셈 `/`, `DIV`, `MOD` 를 쏘기 바로 윗줄에, `IF gv_y <> 0.` 이라는 0 배제 조건식 필터를 무조건 기입합니다.
    분모 변수 값이 0 이 아닐 때만 나누기를 타고, 0 일 때는 덤프를 내는 대신 디폴트 값 0 을 강제 대입하거나 경고를 뱉게 함으로써, 런타임 폭사 사고를 100% 철통 방어할 수 있습니다.

### Q2. 대량 데이터 루프(Loop) 속에서 문자열들을 조각조각 엮을 때, CONCATENATE 를 쓰는 것과 && 연산자를 쓰는 것은 백엔드 메모리 힙(Heap)과 CPU 부하에서 어떤 엄청난 성능 차이를 유발하나요?
**A**: **CONCATENATE 는 임시 객체 없이 타깃 변수 버퍼로 데이터를 다이렉트 고속 전송(C-Level Memory Copy)하지만, && 연산자는 매 엮음마다 힙(Heap) 메모리에 임시 가변 String 객체를 재할당하고 폐기하는 가비지 컬렉터 부하를 유발해 속도를 저하시킵니다.**
- **문자 엮기 성능의 실체**:
  - **`CONCATENATE`**: 결합할 텍스트들의 총 바이트 길이를 먼저 잰 뒤, 목적지 메모리 주소 영역으로 쌩 데이터를 고속 다이렉트 복사해 붙여 넣습니다. 램 상에 찌꺼기 텍스트 객체를 남기지 않아 루프를 수만 번 돌려도 속도가 지체되지 않습니다.
  - **`&&` (String Chaining)**: 사용하기는 아주 편하고 세련되어 보이지만, 컴파일러가 내부적으로 힙(Heap) 공간에 임시 String 객체 버퍼를 지어 올린 뒤 글자를 엮어 담고, 이를 최종 변수에 넘긴 뒤 임시 버퍼를 파괴하는 무거운 메모리 자원 사이클을 돕니다. 
  10만 건 대량 루프 안에서 `&&` 를 남발하면 힙 세션이 쓰레기 객체로 가득 차, 가비지 컬렉터(Garbage Collector)가 이 찌꺼기들을 치우느라 CPU 자원을 다 점유해 시스템이 기어가는 성능 병목을 초래합니다. 
  **철칙**: 대량 루프에서는 **`CONCATENATE`** 를, 루프 밖 1회성 가벼운 조립에는 **`&&`** 를 사용해야 안전합니다.

### Q3. IF-ELSEIF 사다리를 길게 늘여 쓰는 것에 비해, CASE-WHEN 을 사용하면 백엔드 컴파일러가 분기할 때 성능적으로 어떤 압도적인 혜택을 제공하나요?
**A**: **IF 는 매 ELSEIF 조건식을 CPU 가 매번 새로 계산해 평가(Re-evaluation)해야 해서 최하단 분기 시 연산 낭비가 심하지만, CASE 는 컴파일 시점에 해시 구조의 점프 테이블(Hash-like Jump Table)을 메모리에 등록해 0.0001초 만에 타깃 블록으로 홉 점프(O(1))하기 때문입니다.**
- **해시 점프 테이블의 동작 기작**:
  - **`IF - ELSEIF` (순차 재평가)**:
    컴퓨터는 1번 조건이 거짓이면 2번 조건을, 2번이 거짓이면 3번 조건을 CPU 가 레지스터 값을 가져와 매번 직접 참/거짓 판단을 연산(`Re-evaluation`)합니다. 비교 대상 분기가 20개면 최악의 경우 20번의 CPU 계산을 다 밟아야 해 느려집니다.
  - **`CASE - WHEN` (해시 고속 점프)**:
    컴파일러는 `CASE gv_var.` 를 빚는 순간 내부적으로 해시 맵 구조의 분기 테이블을 램에 자동 개설합니다. 
    실행할 때 CPU 는 WHEN 의 위아래 순서를 훑지 않고, `gv_var` 변수 값의 해시 메모리 주소로 **다이렉트 홉 점프(Direct Jump - O(1))** 를 실행해 맞춤 WHEN 블록을 격발합니다. 
    분기 개수가 많아질수록 `CASE` 가 압도적인 성능적 우위를 점하며 소스 가독성도 극대화됩니다.

### Q4. 반복문(DO/WHILE) 속에서 사용되는 EXIT, CONTINUE, CHECK 지시어는 루프 실행 흐름 제어에서 구체적으로 어떻게 다르게 작동하나요?
**A**: **EXIT 는 루프를 즉시 완전 종결 파괴해 빠져나가고, CONTINUE 는 이번 회차 하단 소스를 씹고 즉각 다음 회차로 점프하며, CHECK 조건은 조건이 '거짓' 일 때 CONTINUE 와 똑같이 스킵 점프를 기동시킵니다.**
- **3대 루프 제어 스위치**:
  1. **`EXIT`**: "더 이상 볼일 없다. 루프 완전 해체!"
     - 지시어를 스치는 찰나, 루프 밑바닥 `ENDDO`/`ENDWHILE` 뒷줄로 프로그램 제어 포인터를 순간 이동시켜 반복을 즉사 종결합니다.
  2. **`CONTINUE`**: "이번 회차는 여기까지만 하고, 다음 바퀴 돌리자!"
     - 하단에 적힌 남은 WRITE 나 계산 코드를 일절 무시하고 쌩 건너뛰어, 즉각 다음 `sy-index` 회차(예: 3회차 실행 중 만났다면 즉시 4회차로)의 머리 라인으로 복귀해 반복을 계속합니다.
  3. **`CHECK 조건식`**: "조건이 참이면 아래로 통과하고, 거짓이면 이번 바퀴를 건너뛰어라!"
     - 조건식 판정이 참이면 검문 통과판처럼 아래 줄 코드로 유연하게 흐릅니다. 
     - 반대로 조건식이 **거짓**이면 즉각 `CONTINUE` 가 기동되어 다음 회차 바퀴로 스킵 점프합니다. (즉, `IF NOT 조건. CONTINUE. ENDIF.` 구문을 한 단어로 압축한 특수 스위치입니다.)

### Q5. 구구단 2단계 중첩 DO 문을 돌릴 때, 왜 바깥 sy-index 값을 gv_dan = sy-index + 1 과 같이 로컬 변수에 무조건 백업해 두고 안쪽 루프로 들어가야 하나요?
**A**: **sy-index 는 아바 시스템 내부에서 단 1개만 존재하는 공용 카운터 레지스터 필드이므로, 안쪽 루프가 개시되는 순간 바깥 인덱스 번호를 강제로 덮어씌워(Overwrite) 오염 소멸시키기 때문입니다.**
- **sy-index 덮어쓰기 유실**:
  바깥 `DO 8 TIMES` 루프를 돌 때, 첫 바퀴 회차 번호인 `1` (2단을 위한 값)이 `sy-index` 에 담깁니다.
  이 상태에서 백업 없이 곧장 안쪽 `DO 9 TIMES` 루프로 진입하는 순간, 안쪽 루프의 1회차가 격발되면서 전역 `sy-index` 값은 안쪽 카운트 번호인 **`1`** 로 강제 덮어쓰기 오버라이트 갱신됩니다.
  그 결과 바깥 루프는 자신이 몇 번째 단을 돌고 있었는지(원래 1회차였던 사실)를 망각하게 되어, 안쪽 루프가 9번 돌고 뻗는 찰나에 바깥 루프도 1회전 만에 통째 종료되는 오작동 대참사를 초래합니다.
  - **해결**:
    안쪽 루프 괄호를 열기 바로 전선에서 **`gv_dan = sy-index + 1`** 을 쏘아, 현재의 소중한 바깥 인덱스 1 값을 나만의 일반 변수 주소인 `gv_dan` 램 방에 대입 결선해 안전하게 복제 잠금해 두어야만 안쪽 루프가 sy-index 를 난사하며 요동치더라도 내 단수 데이터가 온전히 보존됩니다.
