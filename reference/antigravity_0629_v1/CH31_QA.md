## 3. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. OData Gateway 실무 조회를 위해 concertset_get_entityset 메서드 내부를 구현하고 있습니다. 페이징 쿼리인 UP TO @iv_top ROWS 와 OFFSET @iv_skip 을 SELECT 문장에 기재했는데 왜 계속 컴파일 에러가 나나요?
**A**: **OFFSET 을 사용하는 아바 Open SQL 쿼리문은 정렬 기준을 명시하는 ORDER BY 구문이 의무적으로 선행 기입되어야만 문법 정합성을 충족하기 때문이며, 정렬 구문을 1순위로 채워야 컴파일이 뚫립니다.**
- **컴파일 차단 원인**:
  데이터베이스 엔진은 "어디서부터 몇 건을 건너뛰어(OFFSET) 리스트를 뿜어라" 는 명령을 수행할 때, 기준이 되는 정렬 순서가 보증되어 있지 않으면 페이지 이동을 누를 때마다 가져오는 데이터가 뒤바뀌는 정합성 파손 위험을 안게 됩니다. 
  따라서 아바 컴파일러는 `OFFSET` 지표가 존재할 때 `ORDER BY` 가 소스 줄에 없으면 즉각 컴파일 에러를 때려 빌드를 막아버립니다.
- **문법 통과 규칙**:
  ```abap
  " [ OFFSET 페이징 셋업 시 반드시 ORDER BY 가 선행되어야 함! ]
  SELECT concert_id, artist FROM zconcert
    INTO CORRESPONDING FIELDS OF TABLE @et_entityset
    UP TO @iv_top ROWS
    ORDER BY concert_id " <- 이 정렬 지정이 반드시 앞에 와야만 함!
    OFFSET @iv_skip.
  ```

### Q2. 수신 측 시스템에서 IDoc 목록을 WE02 에서 확인해 보니 상태 코드가 51 번 빨간 불로 되어 멈춰 있습니다. 이 상태가 무엇을 뜻하며, 왜 방치하면 안 되고 BD87 로 어떻게 구출하는지 비유로 알려주세요.
**A**: **상태 51 은 편지(IDoc)가 본관 우체국에는 무사히 도착했으나, 내벽 데이터(마스터 부재 등)가 꼬여 DB 테이블 적재를 단호히 거절당한 '오류 대기' 상태를 뜻하며, 백엔드 원인을 보정한 뒤 BD87 구출 센터에서 Process(재처리)를 때려 53번 성공으로 유인 승격시켜야 예약 누락을 막습니다.**
- **방치의 위험성**:
  상태 51 로 굳어버린 IDoc 은 자동으로 DB 에 써지지 않고 큐에 영구 동결됩니다. 
  이를 방치하면 협력사 시스템은 "주문서를 쐈다" 고 생각하는데, 우리 장부에는 거래가 누락되어 매출 빵꾸 및 정산 분쟁을 초래합니다.
- **BD87 구출 과정**:
  1. `WE02` 상태 51 에러 세부 로그를 열어 "공연ID = C999 가 zconcert 마스터에 없습니다" 오류 단서를 찾습니다.
  2. 기준 마스터 테이블에 C999 공연 코드를 수동 등록해 줍니다.
  3. `BD87` (재처리 전산실)로 들어가 오류 IDoc 번호를 찍고 **`Process`** (재처리) 기어를 돌립니다.
  4. 원본 편지 내용물 그대로 Inbound 함수가 다시 돌며, 이번엔 C999 가 마스터에 존재하므로 53번(Inbound 성공) 마크로 기분 좋게 승격되어 DB 에 예매 장부가 기록 완료됩니다.

### Q3. 비동기 ALE 통신망을 통해 IDoc 주문서를 유입받는 Inbound 수신 모듈을 짜고 있습니다. 왜 수신부 1순위 코드 줄에 멱등성(Idempotency) 가드를 올리고 EDIDC-DOCNUM 키를 스캔해야 하나요?
**A**: **비동기 통신망은 네트워크 패킷 유실 지연 시 동일한 IDoc 편지를 2번 연속 발송하는 오작동이 다분하므로, 중복 유입 시 이미 처리된 편지(DOCNUM)인 것을 SELECT 검문하여 쌩 까고 돌려보내지 않으면 중복 결제 및 중복 저장을 유발하기 때문입니다.**
- **중복 유입 시나리오**:
  1. 협력사가 1번 주문 편지를 보냈습니다.
  2. 우리 우체국이 편지를 받아 성공적으로 DB 에 결제 적재(상태 53)를 마쳤습니다.
  3. 그런데 마침 통신 응답망이 끊겨 협력사에게 "잘 받았다" 고 회신 신호를 못 쏴주었습니다.
  4. 협력사는 배달 사고가 난 줄 알고 똑같은 내용의 1번 주문 편지를 또 쏴버립니다.
  5. 2번째 들어온 1번 편지를 그냥 받아 적재하면, 동일한 결제가 2번 중복 적재되어 고객 계좌가 털리는 정합성 대재앙이 납니다.
- **멱등성 방어선**:
  Inbound 메서드 첫머리에 **`EDIDC-DOCNUM` (고유 우편물 일련번호)** 이 내 성공 이력 로그 테이블에 이미 잠자고 있는 녀석인지 검문하여, 존재한다면 DML 을 격발하지 않고 즉시 Status = 53 성공 마크만 퉤 뱉고 복귀하도록 차단막을 쳐야 중복 결제를 원천 수호할 수 있습니다.

### Q4. IDoc 3층 구조인 EDIDC, EDIDD, EDIDS 각각의 물리적 역할과, 에러 원인을 디버깅 추적할 때 왜 EDIDS 레코드 테이블을 뒤져서 분석해야 하는지 알려주세요.
**A**: **EDIDC 는 수신처 주소가 적힌 편지 봉투 헤더(Control)이고, EDIDD 는 내용물 세그먼트 데이터(Data)이며, EDIDS 는 배송 이력 로그(Status)이므로, 어디서 배달 사고가 왜 났는지는 3층인 EDIDS 상태 이력을 돋보기로 파헤쳐야 상세 OS/어플리케이션 에러 단서가 수집되기 때문입니다.**
- **3층의 물리적 역할 분할**:
  - **`EDIDC`**: 편지를 보낸 IP, 받는 논리 시스템닉네임, 메시지 유형(예: ORDERS)을 쥐어 라우팅을 통제합니다.
  - **`EDIDD`**: 실제 고객명, 예약수 등 sdata 1,000자 문자 쪼가리들이 세그먼트 코드에 맞춰 줄지어 타는 램 컨테이너입니다.
  - **`EDIDS`**: 배달 기사가 시간대별로 "2시 접수, 3시 출발, 4시 오류 발생" 등을 남긴 다이어리 로그 테이블입니다.
- **디버깅 추적**:
  IDoc 이 빨간 불이 들어왔다면, 데이터(EDIDD)는 죄가 없습니다. 
  어떤 가드에 걸려 어떤 예외 덤프를 먹고 주저앉았는지는 배송 기사가 3층 **`EDIDS`** 에 대고 "줄 3번 seats 필드의 'ABC' 는 숫자로 변환 불가" 라고 상세 사유 텍스트를 남겨두기 때문에, `WE02` 모니터 상에서 Status 레코드를 정밀 추적해 읽어야 10초 만에 에러를 고칠 수 있습니다.

### Q5. BD64 에서 ALE Distribution Model (분배 모델)을 완벽하게 그리고 연결망 SM59 도 뚫어두었습니다. 그런데 왜 양쪽 서버 간 IDoc 송수신을 날리면 파트너 프로파일 에러를 내며 먹통이 되나요?
**A**: **BD64 분배 지도는 라우팅 방향의 기획서일 뿐이며, 실제 각 기계의 최종 관문 계약 서류인 WE20 Partner Profile (파트너 프로파일)을 클라이언트마다 수동 셋업하지 않았거나 배포를 빼먹었기 때문입니다.**
- **먹통의 핵심 원인**:
  아무리 BD64 에 "이 기계로 ORDERS 메시지를 보내겠다" 고 지도를 잘 그려두었어도, 수신 서버 관문 헌병인 **`WE20`** 파트너 프로필에 대고 **"ORDERS 메시지가 유입되는 것을 공식 수락하고, 이를 받으면 Process Code ZORDE(ZIDOC_INPUT_BOOKING 함수와 도킹)를 작동시켜 처리해라"** 고 수신 계약서 서명을 셋업해 두지 않으면, 수신처 헌병은 침입으로 규정해 패킷을 폐기해 버립니다.
  특히 `WE20` 은 개발에서 운영으로 커스터마이징 TR(전송)이 안 타고 날아가는 **'각 클라이언트별 독립 수동 마스터 셋업 데이터'** 이므로, 운영 서버 배포 즉시 운영기 WE20 창을 열고 ORDERS 매핑 서류 작성을 수동으로 낙찰해 주어야만 통신망 전원이 켜집니다.
