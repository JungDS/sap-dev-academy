## 3. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. DATA 변수 선언 시 TYPE 과 LIKE 는 무엇이 다르며, 왜 LIKE 대상 변수는 항상 코드 위쪽에 적혀 있어야 컴파일이 되나요?
**A**: **TYPE 은 정적 설계도 명세를 참조해 메모리를 조형하지만, LIKE 는 실제 램에 할당된 쌩 변수 객체의 메모리 서술자(Descriptor) 주소를 직접 조인 복사하므로, 본뜨려는 변수가 컴파일러 파싱 방향상 상단에 먼저 상주하고 있어야 빌드가 성립되기 때문입니다.**
- **동작 원인**:
  - `TYPE ty_amount`: 데이터 사전이나 로컬 `TYPES` 에 선언된 정적 껍데기 형식(Type)을 참조합니다. 실체 메모리가 없어도 이름 정의만 존재하면 컴파일러가 바로 방을 지을 수 있습니다.
  - `LIKE gv_price`: 이미 프로그램 세션 안에 실물로 가설되어 램 용량을 쥐고 있는 변수 `gv_price` 의 기술 데이터(Descriptor)를 훔쳐 옵니다. 
  만약 컴파일러가 위에서 아래로 한 줄씩 소스를 조립 파싱해 가는데, 윗줄에 `LIKE gv_price` 를 먼저 만나고 정작 `DATA gv_price` 가 그 아랫줄에 선언되어 있다면, 컴파일러는 "아직 메모리에 실체가 없는 가상의 존재를 본뜨려 했다" 며 참조 오류를 내뿜고 즉시 빌드를 기각 차단합니다. 

### Q2. 결제 금액이나 부가세 등 돈 데이터를 다룰 때 F (부동소수점) 타입을 쓰면 왜 언젠가 대형 정산 사고가 터지며, 왜 P (Packed Decimal) 타입을 써야만 안전한가요?
**A**: **부동소수점 F 는 컴퓨터의 2진수 연산 특성상 10.00 을 9.99999999... 로 오차(근삿값) 저장해 1원 단위 불일치를 유발하지만, Packed Decimal P 는 BCD 압축 기법을 써서 사람이 적은 10진수 숫자를 소수점 아래까지 한 치의 오차도 없이 고스란히 저장하기 때문입니다.**
- **근삿값 F vs 고정소수점 P**:
  - **부동소수점 `F`**: 아주 광활한 범위(약 $10^{308}$)의 수학 과학 계산용 과학 소수점입니다. 빠르게 계산하되 2진수로 값을 표현하다 보니 `3.14` 를 넣으면 끝자리가 미세하게 늘어지거나 잘리는 컴퓨터 공학적 태생 한계를 갖습니다. 100만 건 전표를 합산하면 오차가 누적되어 최종 시재가 털리는 정산 장애가 100% 터집니다.
  - **압축십진수 `P`**: 우리가 적은 숫자 `123.45` 를 BCD 방식으로 메모리에 그대로 박아 10진수 연산을 실행하므로, 단 0.0000001 원의 연산 오차도 허용하지 않고 칼같이 정확한 연산 결과를 반환해 금융/돈 계산의 정합성을 수호합니다.

### Q3. DATA gv_num TYPE p LENGTH 8 DECIMALS 2 라고 변수를 선언했을 때, 이 변수가 내부적으로 담을 수 있는 정수와 소수를 포함한 전체 숫자의 최대 길이는 자릿수로 몇 개인가요?
**A**: **P 타입은 1바이트당 10진수 2자리를 저장하는 BCD 압축 공식을 따르며, 마지막 반 바이트(4비트)는 부호 영역으로 회수되므로 `자릿수 = (Length * 2) - 1` 에 의해 총 15자리를 저장할 수 있습니다.**
- **BCD 자릿수 공식 산출 과정**:
  `LENGTH 8` 로 선언되면 이 변수는 메모리에서 정확히 **8바이트 (64비트)** 를 확보합니다.
  BCD(Binary Coded Decimal) 기법은 4비트(=반 바이트) 공간 안에 10진수 숫자 1개(`0~9`)를 찰딱 매핑해 넣습니다.
  따라서 8바이트는 원래 `8 * 2 = 16` 자리를 담을 수 있어야 합니다.
  하지만 컴퓨터가 이 숫자가 플러스(+)인지 마이너스(-)인지 구별해야 하므로, **맨 꼬리 반 바이트(4비트)는 숫자를 담지 못하고 '부호 마크 비트' 로 강제 헌납**됩니다.
  그 결과 실제 숫자가 들어설 수 있는 최대 공간은 `16 - 1 = 15` 자리가 됩니다.
  여기에 `DECIMALS 2` 를 빼면, 정수를 최대 13자리까지 적고 소수점 이하 2자리를 박을 수 있는 임계 용량이 최종 셋업됩니다.

### Q4. CONSTANTS 상수를 선언하고 코드 중간에 실수로 gc_pi = '3.14' 처럼 값을 대입하여 덮어쓰려 시도하면 런타임에 프로그램이 폭사하나요, 아니면 빌드 전에 차단되나요?
**A**: **상수 gc_pi 는 아바 컴파일러에 의해 쓰기 방지(Write-Protected) 락이 걸린 램 주소지에 배정되므로, 런타임이 기동되기도 전인 SE38 빌드(활성화) 단계에서 문법 에러를 뿜으며 즉각 안전 기각됩니다.**
- **Write-Protected 가드**:
  만약 상수가 런타임 오류로 처리된다면, 사용자가 실제 운영 중에 프로그램 계산 버튼을 누르는 순간 덤프가 터져 서버가 뻗는 사고를 맞이합니다.
  이를 방지하기 위해 아바 컴파일러는 구문 분석 단계에서 `gc_pi` 라는 상수 명칭이 대입 연산자 `=` 의 왼쪽에 적히는 순간, **"gc_pi는 상수이므로 값을 재지정할 수 없다"** 고 빨간 경고창을 띄우며 컴파일(활성화) 자체를 전면 잠가버립니다.
  이로 인해 개발자가 코드를 빌드해 배포 기차에 싣기 전에, 실수로 상수를 덮어쓰려던 코딩 버그가 100% 원천 예방됩니다.

### Q5. 텍스트 심볼을 적을 때 TEXT-001 방식으로 적는 것과 '금액'(001) 방식으로 기본값을 괄호 병기하여 적는 것은 실무 배포 시 다국어 번역 정합성에서 어떤 치명적인 차이를 야기하나요?
**A**: **TEXT-001 은 번역이 누락되면 화면에 쌩 공백을 내보내 누락을 조기에 감지하도록 게이팅하지만, '금액'(001) 은 누락 시 임시 디폴트 글자('금액')를 화면에 우회 방출하여 다국어 번역이 누락된 버그를 숨기고 배포를 오염시키기 때문입니다.**
- **다국어 누락 방치 리스크 (REPOTEXT)**:
  모든 번역 텍스트는 데이터베이스의 **`REPOTEXT`** 테이블에 번호표(`001`)를 키로 해 격리 보관됩니다.
  - **`TEXT-001`**: 영문 번역이 `REPOTEXT` 에 존재하지 않는 상태에서 영어 로그온 시, 화면에 공백이 뜹니다. 개발자와 테스터가 화면을 보자마자 "어? 왜 제목 라벨이 비어 있지?" 하고 즉각 인지해 다국어 등록을 마저 채울 수 있습니다.
  - **`'금액'(001)`**: 영문 번역이 누락되면 화면에 괄호 앞의 한글 기본값인 **'금액'** 이 그대로 방출됩니다. 화면이 깨지지는 않아 임시로 편해 보이지만, 미국 지사장이 접속했을 때 화면 구석구석에 한글 라벨이 그대로 노출되는 '다국어 누락 참사' 가 조용히 숨겨진 채 실무 운영 서버까지 이송 배포되는 형상 통제 파탄을 초래합니다. 
  따라서 번역 품질 관리를 위해 실무에서는 **`TEXT-001`** 형식을 칼 수호해 사용합니다.
