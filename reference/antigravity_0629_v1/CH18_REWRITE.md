# CH18_REWRITE - Modern ABAP Syntax v1

> 목적: `content/abap/CH18`의 7개 레슨을 "IT 비전공자 입문자가 왜 필요한지부터 실습 확인까지 직관적으로 이해하고 따라갈 수 있는 완성 강의자료" 수준으로 재작성한 기준 원고다. 이 문서는 아직 `content/abap` 원본을 덮어쓴 결과가 아니라, 재작업 판정 이후 실제 반영 전 검수 가능한 v1 기준안이다.

## CH18 전체 설계

CH18의 한 문장 목표는 "장황한 절차 중심의 고전 문법에서 벗어나, 선언과 동시에 타입을 알아채는 Inline Declaration(DATA/FINAL)과 결과 형상대로 데이터를 짜내는 생성 연산자(VALUE/CORRESPONDING), 대괄호 콕 집기 검색(Table Expression) 및 안전 가드(line_exists/line_index), 그리고 문자열 템플릿(|...|)과 복합 대입(+=)을 활용하여, 기존 레거시 비즈니스 판정 코드를 또렷하고 간결하게 현대식 모던 아밥 구조로 전면 리팩터링한다"이다.

IT 비전공자 입문자는 선언문 `data` 라는 변수를 먼저 개설해 두어 `DATA(data) = ...` 가 서브스트링 추출로 꼬이는 컴파일 트랩에 걸리고, `VALUE` 대입 시 `BASE` 누락으로 기존 화물을 싹 포맷시키며, Table Expression `tab[ ]` 사용 시 없는 행 조회 예외 덤프를 터트리고, 인라인 선언과 동시에 `+=` 누적 대입을 붙여 쓰다가 문법 거부를 겪는다.
따라서 본 챕터는 다음과 같은 불편 해결의 단계적 설계로 전개한다.

1. **의도와 선언의 밀착**: 80줄 위에 변수를 적어두어 위아래로 눈을 굴리며 추적하던 고통 $\rightarrow$ 값이 탄생하는 바로 그 자리(LOOP, READ, 계산식)에서 변수 주머니를 즉시 빚어내는 **`DATA( )`** 인라인 선언 수립.
2. **수정 불가 방어선 (FINAL)**: 한 번 수납하면 절대로 변경되어선 안 되는 상수형 값을 박는 **`FINAL( )`** 읽기 전용 상수 가드.
3. **엑셀 틀 복사 (VALUE)**: 한 필드씩 주소를 적고 APPEND를 연타하던 절차 나열 $\rightarrow$ 행과 열의 엑셀 리터럴 모양 그대로 값을 부어 조립하는 **`VALUE #( )`** 생성자와 중복 키 에러 경고.
4. **기존 화물 지키기 (BASE)**: VALUE 연산 시 기존 내부 테이블의 데이터를 소멸시키지 않고 꽁꽁 유지하며 덧붙이는 **`BASE`** 누적 지시어 명세.
5. **Comprehension 루프 (FOR)**: 테이블 데이터를 순회하며 다른 형태의 새로운 내부 테이블로 초고속 둔갑 생성해 내는 **`FOR ... IN`** 기법.
6. **이름표 자동 매핑 (CORRESPONDING)**: 서로 다른 두 구조체나 테이블 사이에서 이름이 같은 필드는 자동으로 이동시키고, 이름이 다르면 **`MAPPING`**, 특정 열은 **`EXCEPT`** 로 제외하는 통합 이송 생성자.
7. **대괄호 한 놈 집기 (Table Expression)**: READ TABLE 문법 없이 대괄호 `lt_booking[ 1 ]` 기호로 행을 콕 집어 읽는 **Table Expression** 사양.
8. **예외 방어막 (line_exists)**: 없는 행을 대괄호로 바로 집었을 때 덤프(`CX_SY_ITAB_LINE_NOT_FOUND`)가 나지 않도록, 사전에 존재 여부를 판정해 주는 **`line_exists( )`** 와 인덱스 번호를 솎아내는 **`line_index( )`** 방어 장치.
9. **눈에 보이는 문장 (|...|)**: 복잡한 CONCATENATE 쪼개기 없이, 중괄호 `{ }` 임베딩과 `DATE = USER` 사용자 서식 옵션으로 최종 인쇄 문장 모양 그대로 코딩하는 **String Template** 장착.
10. **축약 연산 (+=)**: 누적 합계를 낼 때 자기 변수명을 양쪽에 적지 않고 한 번에 연산 대입하는 **복합 대입 연산자(`+=`, `-=`, `*=`, `/=`)** 의 문법 명세.
11. **레거시 모던 리팩터링**: 앞서 빌드했던 classic 예매 좌석 합계, 새 예매 행 추가, Table Expression 안전 조회 판정 코드를 동작 보존(동일 결과) 원칙 하에 또렷하게 리팩터링하는 콘서트앱 모던 완성.

### 범위와 금지선

- **SQL 경계선 철저 격리**:
  - `SELECT` 쿼리문 내부의 호스트 변수 탈출 기호 `@` 나 콤마 필드 분리 등 New Open SQL 문법은 **CH19**의 고유 영역이므로 이 장에서는 철저히 격리 차단하며, 오직 **SQL 바깥의 일반 ABAP 문법 리팩터링에만 완벽하게 집중**한다.
- **변수 스코프 접두어**: 모든 스크립트는 전역 범위로 간주하며, 일반 변수는 `gv_`, 구조체는 `gs_`, 내부 테이블은 `gt_` 접두어를 기본으로 삼으나, `DATA(lv_total)` 과 같이 인라인 선언된 로컬 성격 변수는 `lv_`, `ls_`, `lt_` 접두어를 사용하는 예외를 부여하여 modern 감각의 실무 코드를 반영한다.
- **이름 풀 준수**: 예제 이름은 지정된 이름 풀에서만 가져오며, 주인공은 항상 **정훈영**으로 통일한다.

## CH18-L01 - Inline Declaration

### 왜 필요한가

우리가 지금까지 짠 classic ABAP 코드는 프로그램을 시작하기 전에 반드시 에디터 맨 위 구역에 `DATA: gv_total TYPE i, gs_book TYPE zbooking.` 형태로 변수 방을 미리 선언해 두어야 했다. 
변수 선언부가 위에 가지런히 모여 있으면 단선적으로 무엇을 쓰는지 한눈에 들어오는 장점은 있다. 하지만 코드가 100줄, 200줄로 길어지게 되면, 80줄 아래에서 `READ TABLE lt_booking INTO gs_book` 이라는 코드를 읽는 개발자는 "이 `gs_book` 이라는 녀석의 구조체 방 타입이 도대체 뭐였지?" 하며 매번 마우스 스크롤을 맨 위로 올려 선언부를 확인하고 다시 내려오는 극심한 인지 스크롤 피로와 가독성 저하를 겪게 된다. 

**변수가 생성되고 값이 최초로 수납되는 바로 그 물리적 자리(LOOP, READ, 대입식)에서, 별도의 타입 선언문 작성 없이 변수방을 즉시 즉흥적으로 개설하여 가독성을 극대화하는 선언 기술**이 필요하다. 그것이 **인라인 선언(Inline Declaration)** 의 장착이다.

### 무엇인가

#### 1. Inline Declaration: DATA( ) 와 FINAL( ) 의 정의
- **`DATA(변수명)`**: 값이 할당되거나 반환되는 쓰기 위치(Write Position)에서 컴파일러가 문맥상 타입을 자동으로 유추하여 즉시 변수를 탄생시키는 선언 연산자다.
- **`FINAL(변수명)`**: `DATA()` 와 동일하게 인라인으로 변수를 만들지만, 최초로 대입된 값에서 **이후 절대 값을 변경하거나 재대입할 수 없도록 강제 락(Lock)을 걸어 상수로 고정**시키는 읽기 전용 변수 선언 연산자다.

#### 2. DATA( ) 의 물리적 컴파일 수명 규칙
- *가장 오해하기 쉬운 런타임 함정이다.*
- 인라인 변수는 런타임에 동적으로 생겼다 사라지는 휘발성 임시 방이 아니다. **컴파일러가 코드를 기계어로 번역하는 시점에 해당 처리 블록(Subroutine 이나 Method) 전체의 로컬 변수로 정식 등록**해 준다.
- 따라서, 실행 시점에 `IF` 조건 분기가 작동하지 않아 해당 인라인 선언 라인을 런타임에 밟지 않고 지나갔다 하더라도, 그 아래 코드에서 해당 변수를 읽거나 쓰는 것은 아주 정상적으로 작동한다. 컴파일 시점에 변수방은 이미 만들어져 대기 중이기 때문이다.

#### ⚠️ [변수명 'data' 사용 시 substring 변환 오류 컴파일 트랩]
- *실무 모던 아밥 작성 시 뒤통수를 때리는 치명적인 함정 명세다.*
- 만약 프로그램 전역이나 로컬 상위 구역에 이미 **`data`** 라는 이름의 변수가 선언되어 실재하고 있는 상태에서, 아랫줄에 **`DATA(data) = '1'.`** 이나 `DATA(var) = ...` 를 사용하면 컴파일러는 이를 인라인 선언이 아니라, **`data` 변수의 일부 영역을 잘라내어 접근하는 오프셋/길이 substring 연산(`data(len)`)으로 해석하여 꼬여버린다.**
- **방어선**: 따라서 모던 아밥을 가동할 때는 변수 이름을 절대로 `data` 라는 4글자 표준 약어와 겹치게 명명하지 말아야 한다.

### 어떻게 확인하는가

에디터에서 READ TABLE과 LOOP AT 식 안에 인라인 선언을 꽂아 넣고 타입을 유추하는 구조를 검증한다.

```abap
REPORT zch18_l01_inline.

TYPES: BEGIN OF ty_book,
         booking_id TYPE zbooking-booking_id,
         customer   TYPE zbooking-customer,
       END OF ty_book.
DATA gt_booking TYPE STANDARD TABLE OF ty_book.

START-OF-SELECTION.
  " [classic 선언] DATA ls_book TYPE ty_book. (이 선언을 과감히 생략!)

  " 1. [인라인 READ] READ 문맥을 통해 DATA(ls_book)의 타입을 ty_book으로 자동 유추 선언!
  READ TABLE gt_booking INTO DATA(ls_book) WITH KEY booking_id = '0001'.
  
  " 2. [인라인 LOOP] 루프 내부 데이터 타입이 ls_row 로 자동 복제 선언!
  LOOP AT gt_booking INTO DATA(ls_row).
    WRITE: / ls_row-customer.
  ENDLOOP.

  " 3. [상수 가드] 최초 대입 이후 변경이 불가능한 읽기 전용 상수방 개설!
  FINAL(lv_max_limit) = 100.
  " lv_max_limit = 200. " <-- 만약 이 줄을 켜면 컴파일러가 "Left-hand side is not writable" 문법 거부!
```

#### 체험/시뮬레이터 설계 (Inline 선언 위치 판정기)
- **프로세스 플로우**:
  1. 학습자가 [DATA(data) = 'A'.] 카드 칩을 고른다.
  2. 시스템이 "상위에 DATA data TYPE c 변수가 실재하므로 substring 오동작이 납니다!" 붉은 경고등을 켠다.
  3. [DATA(lv_total) = 5 + 3.] 카드를 집는다.
  4. 우측 런타임 컴파일러가 대입식 오른쪽 타입(`i`)을 잃어 `lv_total` 방을 정적 인트 정수형으로 즉시 수립하고 녹색 통과 배지를 띄운다.
  5. 학습자가 `lv_total` 변수를 10줄 아래 `IF` 문 밖에서 호출해 본다. 컴파일러가 수명 주기 상단부터 정적으로 감지하여 정상 바인딩해 주는 라인 추적 모션을 감상한다.
- **상태 및 데이터**:
  - `FINAL(lv_max) = 10` 선언 후 `lv_max += 1` 가동 상태 -> 컴파일 에러 코드 `Immutable variable` 하이라이트.
- **피드백**: 인라인 선언은 짧게 쓰기 편법이 아닌, 의도 근처로 선언을 전진 배치시키는 정밀 컴파일러 타입 추론 기술임을 깨닫게 한다.

### 실수/주의

- **동일 Context 안에서 같은 인라인 변수명을 2번 이상 중복 DATA( ) 선언**: "이 LOOP에서도 `DATA(ls_row)`를 쓰고, 바로 아랫줄 다른 LOOP에서도 `DATA(ls_row)`를 새로 선언해서 써야지" 했다가는, 동일 처리 방(Subroutine/Method) 내에 똑같은 변수를 2번 선언한 꼴이 되어 중복 선언 문법 거부 에러가 나므로, 한 번 선언한 인라인 변수는 아래에서 재선언 없이 **그냥 변수 이름(`ls_row = ...`)만 적어 재활용**해야 한다.

### 정리

- **`DATA( )`** 는 값을 받아내는 시점에 정적으로 타입을 알아채어 변수를 인라인으로 빚어내는 장치다.
- **`FINAL( )`** 은 최초 대입 이후 변경이 절대 금지되는 강력한 상수형 읽기 전용 가드다.
- 변수명 **`data`** 가 이미 선언된 방에서는 인라인이 substring으로 오해되므로 충돌명을 피한다.

## CH18-L02 - VALUE Constructor Expression

### 왜 필요한가

인라인 선언으로 코드를 가까이 당겼다. 
그런데 이번에는 구조체나 내부 테이블에 초기 예시 데이터를 가득 채우는 코드가 불만이다. 
"구구단 행이나 예매 테스트 데이터를 3건 만들고 싶다" 
classic 룰을 따르면 `gs_line-dan = 2. gs_line-mul = 1. gs_line-result = 2. APPEND gs_line TO gt_gugu.` 이 4줄의 작업을 3번 반복해야 하므로 총 12줄의 장황한 대입 연산이 이어진다. 코드를 읽는 동료는 "이게 구구단 2단을 만들겠다는 소리구나" 라는 의도보다, `dan`, `mul` 필드에 일일이 값을 집어넣고 옮겨 담는 지루한 절차적 코딩 노가다에 짓눌리게 된다. 

**변수 선언과 대입 절차를 일일이 나열하지 않고, 내가 만들고자 하는 최종 엑셀 표(테이블)의 모양 그대로 행과 열의 값들을 한 식에 부어서 즉시 빌드해 내는 생성 기술**이 필요하다. 그것이 **[[VALUE]] 생성 연산자**의 장착이다.

### 무엇인가

#### 1. VALUE Constructor (생성 연산자) 의 구조
- **구조**: **`VALUE 타입명( (필드=값 필드=값) (필드=값) )`** 형태로 기입한다.
- **`#` (타입 추론)**: 만약 대입식 좌변 변수의 자료형(예: `ty_line_tab`)이 명확하게 지정되어 선언되어 있다면, 굳이 타입명을 다 적지 않고 **`VALUE #( (dan=2) )`** 처럼 `#` 샵 기호로 대체하여 타입을 자동 상속받아 빚을 수 있다.

#### 2. 기존 화물을 지켜내는 BASE 의 가드 역할
- *초보 개발자들이 모던 아바에 입문하자마자 기존 테이블 데이터를 통째로 날려 먹는 초대형 함정이다.*
- 아바 시스템에서 `lt_gugu = VALUE #( ( dan = 3 ) )` 구문을 실행하면, 기존에 2단 구구단이 들어있던 `lt_gugu` 테이블은 **기존 데이터가 싹 초기화되어 날아가고 오직 3단 행 1건으로 덮어써 진다.** VALUE는 기본적으로 "새 판을 짠다" 는 리터럴 성격이 지배적이기 때문이다.
- **해결책**: 기존 데이터를 보존하면서 꼬리에 행을 이어 붙이고(Append) 싶다면, 반드시 **`BASE`** 지시어를 가동하여 **`VALUE #( BASE lt_gugu ( dan = 3 ) )`** 형태로 부모 기반(Base)을 이정표로 꽂아주어야 기존 화물이 무사히 생포 보존된다.

#### 3. FOR 반복 테이블 생성 (Table Comprehension)
- 루프를 돌며 데이터를 가공 수집할 때, 장황한 `LOOP AT ... APPEND` 를 칠 필요가 없다.
- VALUE 문법 안에 **`FOR 임시변수 IN 원천테이블`** 지시어를 물려주면, 원천 테이블을 돌면서 계산된 신규 행들을 한 식 안에서 순식간에 루프 생성해 낸다. (파이썬의 List Comprehension과 동일한 물리 아키텍처).

### 어떻게 확인하는가

VALUE 연산자와 BASE, FOR 구문을 가동하여 내부 테이블을 한 식으로 채우는 코드를 검증한다.

```abap
REPORT zch18_l02_value.

TYPES: BEGIN OF ty_gugu,
         dan    TYPE i,
         mul    TYPE i,
         result TYPE i,
       END OF ty_gugu.
TYPES ty_gugu_tab TYPE STANDARD TABLE OF ty_gugu WITH EMPTY KEY.

START-OF-SELECTION.
  " 1. [구조체 한 식] VALUE 에 구체적 타입명 ty_gugu를 꽂아 한 덩어리로 조립!
  DATA(ls_one) = VALUE ty_gugu( dan = 2 mul = 3 result = 6 ).

  " 2. [테이블 한 식] 괄호 ( ) 하나가 한 행(Row)을 의미! 좌변 타입 상속 # 기호 활용!
  DATA(lt_table) = VALUE ty_gugu_tab(
    ( dan = 2 mul = 1 result = 2 )
    ( dan = 2 mul = 2 result = 4 )
  ).

  " 3. [기존 보존 BASE] 기존 2개 행 데이터를 소멸시키지 않고 꼬리에 3단을 추가!
  lt_table = VALUE #( BASE lt_table ( dan = 3 mul = 1 result = 3 ) ).

  " 4. [FOR 컴프리헨션] 1부터 9까지 루프를 돌며 2단 9개 로우를 즉시 생성!
  DATA(lt_gugu_2dan) = VALUE ty_gugu_tab(
    FOR i = 1 WHILE i <= 9
    ( dan = 2 mul = i result = 2 * i )
  ).
```

#### 체험/시뮬레이터 설계 (VALUE Builder)
- **프로세스 플로우**:
  1. 학습자가 2개 행이 들어있는 가상 테이블 보드를 쳐다본다.
  2. [BASE 없는 VALUE 실행] 버튼을 클릭한다.
  3. 기존 2개 행이 순식간에 하얗게 소멸하며 증발하고 신규 1개 행만 덩그러니 남는 데이터 손실 대참사 연출을 본다.
  4. [되돌리기]를 하고 [BASE 탑재 VALUE 실행] 버튼을 탭한다. 기존 2개 행이 밑으로 내려가고 그 뒤에 자연스럽게 3번째 행이 추가 결합되어 3개 행으로 완성되는 안전한 조립 애니메이션을 감상한다.
- **상태 및 데이터**:
  - `Hashed Unique Key 테이블에 VALUE #( (key=1) (key=1) ) 으로 중복 키를 주입한 상태` -> 실행 즉시 "Duplicate Key Exception" 빨간불 경고와 함께 `CX_SY_ITAB_DUPLICATE_KEY` 예외 유발 시각화.
- **피드백**: VALUE는 새 값을 만드는 생성기이므로, 누적 시에는 BASE가 절대적인 방패막이 됨을 입증한다.

### 실수/주의

- **FOR 문 내부에서 사용하는 임시 루프 변수를 외부로 유출**: `FOR i = 1 WHILE i <= 9` 구문에서 사용한 임시 인덱스 변수 `i` 나 `FOR line IN lt_table` 의 `line` 은 **오직 VALUE 괄호 내부식 안에서만 살아 숨 쉬는 로컬 스코프**를 가진다. 괄호가 닫힌 바깥 코드 영역에서 `gv_val = i.` 형태로 해당 변수를 호출하려 들면 컴파일 에러("i is unknown")로 거부당하므로 수명 범위를 지켜야 한다.

### 정리

- **`VALUE`** 생성자는 구조체나 테이블을 한 덩어리의 리터럴 모양 그대로 빚어내는 도구다.
- 기존 데이터를 잃지 않고 꼬리에 덧붙이려면 무조건 **`BASE`** 추가어를 가동한다.
- unique 제약을 먹은 Sorted/Hashed 테이블에 중복 키를 VALUE로 밀어 넣으면 덤프가 나므로 주의한다.

## CH18-L03 - CORRESPONDING과 구조 매핑

### 왜 필요한가

VALUE와 인라인을 정복했다. 
그런데 이번에는 다른 두 테이블 간의 데이터 이송이 번거롭다. 
"데이터베이스에서 가져온 원본 예매 구조체(`ZBOOKING`)를, 화면 SALV 출력에 알맞게 다듬은 커스텀 구조체(`ty_booking_ui`)로 데이터를 옮겨 싣고 싶다" 
하지만 두 구조체는 필드명이 미세하게 다르다. 화면용은 예매 상태 필드가 `display_status` 로 되어 있고 원본 DB는 `status` 로 되어 있다. 
이걸 수작업으로 옮기려면 `ls_ui-booking_id = ls_db-booking_id. ls_ui-display_status = ls_db-status.` 처럼 필드를 한 줄씩 짝 맞춰 매칭해야 하므로 장황함이 극에 달한다. classic의 `MOVE-CORRESPONDING`은 이름이 같은 필드만 옮길 수 있어, 이름이 다른 필드는 결국 수동으로 개별 라인 대입을 쳐야 했다. 

**이름이 같은 필드는 아밥 엔진이 알아서 고속 자동 매핑해 주고, 이름이 어긋난 필드만 핀포인트로 연결 고리를 엮어주며, 필요 없는 필드는 제외하고 한 식의 흐름 안에서 새 타입으로 탄생시키는 이송 기술**이 필요하다. 그것이 **[[CORRESPONDING]]** 생성 연산자이다.

### 무엇인가

#### 1. CORRESPONDING 연산자의 정의와 매핑 규칙
- **정의**: 두 구조체나 내부 테이블 간에 서로 같은 이름을 가진 컬럼들을 1:1로 자동 수색해 데이터를 복사 적재해 주는 모던 생성 연산자다.
- **`MAPPING` (이름표 변경 결합)**: **`MAPPING 대상필드 = 원본필드`** 문법 체계를 가동하여, 이름이 다르게 찢어져 있는 두 컬럼 사이의 데이터 수송 파이프라인을 수동 연결해 준다.
- **`EXCEPT` (수송 제외)**: 대상 구조체 필드 중, 원본에서 가져오지 않고 텅 빈 초기값(Initial) 상태로 방치해 두고 싶은 필드를 명시적으로 지정해 배제한다.

#### ⚠️ [원본 기준이 아닌 대상(Target) 기준의 복사 방향성]
- *CORRESPONDING 아키텍처의 가장 중요한 지점이다.*
- 이 생성자는 **좌변(Target Type)에 정의되어 있는 필드들만을 기준**으로 동작한다.
- 원본(Source) 구조체에 컬럼이 100개가 있더라도, 내가 만들고자 하는 대상 구조체에 컬럼이 5개뿐이라면, 이름이 맞는 5개 필드만 안전 수송되고 원본의 나머지 95개 필드는 어떠한 에러 메시지도 없이 **조용히 소멸 버려진다(Ignore).**

### 어떻게 확인하는가

원본 예매 구조체를 화면용 UI 구조체로 MAPPING 과 EXCEPT를 적용해 복사하는 소스를 검증한다.

```abap
REPORT zch18_l03_corresponding.

TYPES: BEGIN OF ty_booking_ui,
         booking_id     TYPE zbooking-booking_id,
         customer       TYPE zbooking-customer,
         display_status TYPE zbooking-status,   " 원본과 이름 다름 (status)
         audit_user     TYPE syuname,           " 원본에 없는 신규 필드
       END OF ty_booking_ui.

DATA gs_db TYPE zbooking. " 원본 DB 테이블 버퍼

START-OF-SELECTION.
  gs_db = VALUE #( booking_id = '1001' customer = '정훈영' status = 'R' ).

  " [이송 집행 철칙] MAPPING은 대상=원본, EXCEPT는 대상필드 기준 지정!
  DATA(ls_ui) = CORRESPONDING ty_booking_ui(
    gs_db
    MAPPING display_status = status      " 대상의 display_status 에 원본 status 수송!
    EXCEPT  audit_user                    " 대상의 audit_user 필드는 복사 제외!
  ).
```

#### 체험/시뮬레이터 설계 (Field Mapping Board)
- **프로세스 플로우**:
  1. 학습자가 좌측 [원본 DB: status] 노드와 우측 [대상 UI: display_status] 노드를 본다. 선이 엇갈려 끊겨있다.
  2. [MAPPING display_status = status 코드 기입]을 선택한다. 두 노드 사이에 녹색 파이프라인이 촥 연결된다.
  3. 이어서 [EXCEPT audit_user] 칩을 누르자, 대상의 `audit_user` 필드 위로 차단막이 내려앉으며 초기값(공백)으로 굳는다.
  4. [CORRESPONDING 가동]을 클릭하자, 원본의 값 `'R'` 이 파이프라인을 타고 우측 대상의 `display_status` 로 흐르고, 나머지 같은 이름 필드들은 자동으로 매칭 통과되어 완성 UI 구조체 카드가 번쩍 렌더링되는 모션을 확인한다.
- **상태 및 데이터**:
  - `MAPPING 방향을 status = display_status (원본=대상) 로 거꾸로 적은 상태` -> "Field status not found in target structure" 컴파일 거부 에러 하이라이트.
- **피드백**: 매핑 선언의 방향은 무조건 "바깥에 살고 있는 대상 필드가 안쪽 원본 필드를 지목하는 대상 = 원본" 공식임을 각인시킨다.

### 실수/주의

- **이름은 우연히 같으나 의미가 완전히 다른 필드의 무분별한 복사**: 원본 테이블의 `status`(예매의 예약 상태)와 대상 구조체의 `status`(결재 진행 상태)처럼, 이름은 스펠링이 `'STATUS'` 로 우연히 똑같지만 실질 업무적 의미가 180도 다른 경우, 필터 없이 CORRESPONDING을 때리면 엉뚱한 결재 상태 필드에 예약 문자 `'R'` 이 쑤셔 넣어지는 데이터 오염 장애를 내므로, **의미가 다른 동명 필드는 반드시 `EXCEPT` 로 제외한 뒤 수동 조율**해야 한다.

### 정리

- **`CORRESPONDING`** 은 이름이 일치하는 구성원을 낚아채 복사하는 매핑 연산자다.
- 다른 이름의 짝짓기는 **`MAPPING 대상 = 원본`**, 수송 제외는 **`EXCEPT 대상`** 을 지정한다.
- 대상 타입(Target) 기준으로만 작동하며, 원본에만 존재하는 필드들은 조용히 삭제 누락된다.

## CH18-L04 - Table Expression

### 왜 필요한가

매핑 연산자까지 정복했다. 
이제 내부 테이블(Internal Table) 안에 담긴 특정 레코드 한 줄을 찾아내어 접근하는 코드가 장황하다. 
"15번째 행을 읽어오고 싶다" 거나 "예매 번호가 `B001`인 행의 고객명을 출력하고 싶다" 
classic의 유일한 도구는 `READ TABLE gt_booking INTO ls_book WITH KEY booking_id = 'B001'. WRITE ls_book-customer.` 이었다. 
변수방을 또 만들고, read table 문장을 길게 치고, 성공 여부를 굳이 아래에 `IF sy-subrc = 0` 으로 체크해야 하니, 단순 필드 한 개를 꺼내 보고 싶은 1회성 의도에 비해 코드가 무겁고 지저분해진다. 

**별도의 INTO 변수 선언과 sy-subrc 검문 없이, 파이썬이나 자바의 배열 대괄호 기호인 `lt_booking[ 1 ]` 처럼 테이블 명 뒤에 조건 괄호를 붙여 즉시 한 행의 값을 값 표현식처럼 꺼내 쓰는 초단축 기술**이 필요하다. 그것이 **[[Table Expression]]** (테이블 대괄호 식)이다.

### 무엇인가

#### 1. Table Expression (대괄호 콕 집기)의 구조와 스펙
- **구조**: **`내부테이블[ 키조건 = 값 ]`** 또는 인덱스로 집을 때는 **`내부테이블[ index번호 ]`** 형태로 내부 테이블 이름 뒤에 공백 없이 대괄호를 붙여 기입한다.
- **스펙**: 이 대괄호 식은 그 자체로 하나의 **완전한 값 표현식(Expression)**이다. 즉, 값을 다른 변수에 대입할 수도 있고, `GV_NAME = lt_booking[ 1 ]-customer.` 처럼 내부 필드 경로를 물결표나 하이픈으로 낚아채 한 줄로 출력할 수도 있다.

#### ⚠️ [없는 행 조회 시 CX_SY_ITAB_LINE_NOT_FOUND 런타임 덤프 폭발 위험]
- *모던 아바를 쓰면서 가장 많은 신입 사원들이 리얼 프로덕션 서버를 터트려 먹는 위험천만한 함정이다.*
- `READ TABLE` 은 행을 찾지 못하면 조용히 `sy-subrc = 4` 로 신호등만 바꾸고 다음 줄로 통과한다.
- **반면, Table Expression `lt_booking[ booking_id = 'B999' ]` 은 없는 행을 직접 대괄호로 읽는 순간 `CX_SY_ITAB_LINE_NOT_FOUND` 라는 수복 불가능한 강한 런타임 에러 덤프를 터트리며 시스템을 강제 폭사시킨다.**
- **방어선 (line_exists)**: 따라서 언제든 값이 없을 수 있는 가변 조건(사용자 입력값 대조 등)인 경우, 절대로 대괄호를 직접 들이밀지 말고, 사전에 존재 여부를 판정해 주는 **`line_exists( )`** 논리 참/거짓 판단 함수로 감싸서 방어벽을 세운 뒤 통과한 경우에만 대괄호를 들이밀어야 안전하다.

### 어떻게 확인하는가

대괄호 식으로 행을 검색하고 line_exists 와 line_index 로 방어하는 소스 규격을 검증한다.

```abap
REPORT zch18_l04_table_expr.

DATA: lt_booking TYPE STANDARD TABLE OF zbooking,
      gv_search  TYPE zbooking-booking_id.

START-OF-SELECTION.
  gv_search = 'B999'. " DB에 없는 유령 예매 코드

  " 1. [대형 참사 코드] line_exists 가드 없이 없는 키를 대괄호로 직접 호출!
  " DATA(ls_dump) = lt_booking[ booking_id = gv_search ]. " <-- 실행 즉시 LINE_NOT_FOUND 덤프 폭발!

  " 2. [안전 방어선 철칙] line_exists로 먼저 검문 후 참일 때만 대괄호 진입!
  IF line_exists( lt_booking[ booking_id = gv_search ] ).
    DATA(ls_hit) = lt_booking[ booking_id = gv_search ].
    WRITE: / '고객명:', ls_hit-customer.
  ELSE.
    WRITE: / '검색 결과가 존재하지 않습니다.'.
  ENDIF.

  " 3. [인덱스 추출] 해당 조건 행이 테이블의 몇 번째 줄에 숨어있는지 번호만 수집 (없으면 0 반환)
  DATA(lv_row_index) = line_index( lt_booking[ booking_id = gv_search ] ).
```

#### 체험/시뮬레이터 설계 (READ TABLE vs Table Expression)
- **프로세스 플로우**:
  1. 학습자가 조회 키에 존재하는 값 `B001` 을 입력하고 [Table Expression 실행]을 누른다. 대괄호 `lt_book[ B001 ]` 이 1번 방을 콕 집어 고객명 '정훈영' 을 깔끔하게 한 줄로 출력한다.
  2. 이어서 존재하지 않는 유령 키 `B999` 를 기입하고 [가드 없이 실행]을 클릭한다.
  3. 번개 기호와 함께 "CX_SY_ITAB_LINE_NOT_FOUND 발생! 프로그램 비정상 셧다운!" 붉은 사이렌 경보 팝업이 화면을 뒤덮는 파괴 연출을 목격한다.
  4. [line_exists 방어선 활성화] 스위치를 켠다. 검문소 바가 내려오고, B999는 가드문에서 거짓(False) 판정을 받아 덤프 우회로로 통과해 "찾지 못했습니다" 안전한 실패 메시지를 띄우는 정상 피드백을 감상한다.
- **상태 및 데이터**:
  - `없는 행을 line_exists 없이 대괄호로 읽은 상태` -> 런타임 상태: `DUMP Crash`.
- **피드백**: 대괄호 조획식은 매우 간결하지만 폭사 제약이 따르므로, 존재 확신이 없는 필드는 line_exists 검문이 필수 의무임을 증명한다.

### 실수/주의

- **대괄호 식 실행 직후에 IF sy-subrc = 0 으로 성공 판정 시도**: Table Expression은 독자적인 값 대입식 흐름이라 **시스템 상태 변수인 `sy-subrc` 를 절대로 건드리지 않고 세팅하지 않는다.** 즉, 직전 대괄호 조회가 성공했든 실패했든 `sy-subrc` 는 이전 쌩뚱맞은 다른 코드의 신호등 값을 그대로 붙잡고 있으므로, 성공 판정은 무조건 **`line_exists( )`** 함수의 논리 결과를 조건 분기문으로 태워 판정해야 한다.

### 정리

- **`Table Expression`** (`tab[ ]`) 은 테이블의 한 행을 대괄호 기호로 즉시 낚아채는 식이다.
- 없는 행을 직접 조회 시 시스템 덤프가 나므로, 반드시 **`line_exists( )`** 로 가드를 친다.
- 대괄호 조회는 **`sy-subrc` 를 세우지 않으므로**, 성공 여부 판정 시 상태 변수 비교를 배제한다.

## CH18-L05 - String Template과 내장 함수

### 왜 필요한가

대괄호 조회식까지 마스터했다. 
이제 화면 딘프로나 툴바 메시지에 뿌려줄 문장을 조립하는 코드가 장황하다. 
"예매번호 B001, 고객 정훈영, 예약 좌석 2석 입니다" 라는 정갈한 최종 안내 문구를 완성하고 싶다. 
classic의 유일한 도구는 `CONCATENATE '예매번호' lv_id '고객' lv_name '예약 좌석' lv_seats '석 입니다' INTO lv_msg.` 였다. 
따옴표와 쉼표, 변수명이 정신없이 뒤섞여 있으니 코드를 읽는 개발자는 최종 사용자가 모니터에서 보게 될 메시지의 정갈한 모양새를 머릿속으로 어지럽게 다시 짜 맞추어야 한다. 날짜가 `20260630` 으로 빽빽하게 출력되어 하이픈을 넣기 위해 조각을 내는 노가다도 추가된다. 

**메시지가 출력될 최종적인 정갈한 문자열 문장의 형상 그대로 코딩하되, 변수가 들어갈 구멍만 중괄호 `{ }` 로 뚫어 라이브 투사하고, 날짜와 숫자를 한 번에 사용자 포맷으로 교체하는 템플릿 기술**이 필요하다. 그것이 **String Template** (`|...|`) 의 장착이다.

### 무엇인가

#### 1. String Template (|...|) 의 문법 규칙
- **문법**: 파이프 기호 **`| 문자열 내용 |`** 형식으로 감싸 선언한다.
- **`{ }` 임베딩 (변수 투사)**: 템플릿 문자열 내부에서 변수나 연산식을 꽂아 넣고 싶을 때 중괄호를 사용한다. (예: `|고객 { gv_name }님|`).
- **⚠️ [중괄호 안쪽 좌우 공백 엄수 가드 규칙]**:
  - *String Template에서 컴파일러가 빨간 줄을 긋는 다발성 문법 실수다.*
  - `{gv_name}` 처럼 중괄호에 바짝 붙여 쓰면 문법 에러가 난다.
  - **반드시 `{ gv_name }` 처럼 중괄호 괄호 안쪽으로 좌우 최소 1칸의 공백 스페이스를 비워주어야** 템플릿 파서가 이를 변수 지표로 정상 인식한다.

#### 2. DATE = USER / NUMBER = USER 서식 변환 칩
- 중괄호 안에 날짜 변수인 `sy-datum` 이나 금액 `10000` 을 그냥 쌩으로 넣으면 화면에 `20260630` 이나 `10000.00` 으로 투박하게 비쳐 독독성이 떨어진다.
- **원리**: 중괄호 변수명 우측에 **`{ sy-datum DATE = USER }`** 또는 **`{ lv_amt NUMBER = USER }`** 와 같이 사용자 포맷 변환 옵션을 추가해 주면, 아바 엔진이 로그인 계정의 개인 프로필 설정(예: YYYY-MM-DD, 10,000)을 알아채어 친숙한 기호로 런타임에 싹 포맷 치환해 준다.

#### 3. 함수형 문자열 내장 API
- classic의 문자 조작 명령(`TRANSLATE`, `SUBSTRING` 등)을 대체하는 간결한 반환형 내장 함수들이 제공된다.
- **`to_upper( )`**: 대문자로 강제 승격 반환.
- **`strlen( )`**: 문자열의 실제 길이 값 반환.
- **`substring( val = ... off = ... len = ... )`**: 특정 오프셋부터 길이만큼 잘라낸 문자열 반환.

### 어떻게 확인하는가

String Template과 서식 옵션, 내장 함수형 API를 가동해 메시지를 조립하는 소스를 검증한다.

```abap
REPORT zch18_l05_string_temp.

DATA: gv_name  TYPE c LENGTH 10,
      gv_price TYPE p LENGTH 5 DECIMALS 2.

START-OF-SELECTION.
  gv_name  = '정훈영'.
  gv_price = '150000.00'.

  " 1. [템플릿 철칙] 파이프 기호 감싸기 및 중괄호 안쪽 공백 좌우 1칸 준수!
  " 2. [서식 철칙] NUMBER = USER 를 탑재해 3자리 쉼표 150,000원 자동 포맷 가동!
  DATA(gv_message) = |예매 고객: { gv_name } / 금액: { gv_price NUMBER = USER }원|.
  
  " 3. [내장 함수] 값 반환형 substring API를 가동해 앞 3글자만 추출!
  DATA(gv_short_name) = substring( val = gv_name off = 0 len = 3 ).
```

#### 체험/시뮬레이터 설계 (Message Composer)
- **프로세스 플로우**:
  1. 학습자가 [DATE = USER OFF] 상태에서 템플릿 `{ sy-datum }` 의 결과 문자열을 본다. `'20260630'` 으로 뻑뻑하게 붙어 출력된다.
  2. [DATE = USER ON] 칩 스위치를 켠다. 결과 텍스트가 순식간에 하이픈을 먹은 `'2026-06-30'` 으로 정갈하게 변환 렌더링되는 변화를 본다.
  3. [substring off=0 len=99] 로 입력 길이보다 길게 범위를 줘서 돌려본다.
  4. 가상 컴파일 보드에 "Substring length out of bounds" 적색 경고 팝업이 뜨며 런타임 덤프 예고를 터트리는 모션을 감상한다.
- **상태 및 데이터**:
  - `{ gv_name }` -> 정상 컴파일 통과. `{gv_name}` -> 컴파일 오류: `Syntax error near '}'`.
- **피드백**: String Template은 문자 이음새의 가독성을 비약적으로 높이며, 중괄호 안의 공백과 범위 한계는 안전선 안에서 제어해야 함을 보여준다.

### 실수/주의

- **String Template의 중괄호 { } 내부에 너무 길고 복잡한 연산식이나 분기 로직 코딩**: "템플릿 안에서 계산이 되니까 `{ lt_book[ status = 'R' ]-seats * 150000 - gv_discount }` 같은 3층짜리 복잡한 계산식을 괄호 안에 다 때려 넣어야지" 했다가는, 템플릿의 장점인 문장 가독성이 다 파괴되고 디버깅이 불가능해지므로, **복잡한 계산은 윗단에서 인라인 변수로 먼저 도출해 이름을 붙이고 중괄호 안에는 깔끔하게 변수명 단독 임베딩**만 남겨야 한다.

### 정리

- **`String Template`** (`|...|`) 은 최종 인쇄될 문장의 형상 그대로 조립하는 문자열 포맷터다.
- 중괄호 `{ }` 내에 변수를 심을 때는 **괄호 안쪽으로 1칸 이상의 좌우 공백**을 필수 제공한다.
- **`DATE = USER`** 와 **`NUMBER = USER`** 옵션은 사용자 포맷으로 자동 번역 치환해 주는 장치다.

## CH18-L06 - Legacy 코드의 Modern ABAP 리팩터링

### 왜 필요한가

모든 개별 신규 모던 아밥 조각들을 정복했다. 
이제 이 조각들을 흩어둔 채 두는 것이 아니라, 실무에서 마주하는 지저분하고 낡은 레거시(Legacy - 고전) 아밥 코드를 만나도 당황하지 않고 **"동일한 실행 결과를 보장하면서 가독성과 실수의 위험을 획기적으로 낮추는 깔끔한 모던 코드로 세련되게 다듬는 수술(Refactoring)"** 능력이 절대 필요하다. 

줄 수 줄이기 시합이 아니라, **동작 보존(동일 결과) 원칙 아래에서 코드를 또렷하게 다듬고, 복합 누적 연산자를 물려 가독성을 완성하는 리팩터링 정석**이 필요하다. 그것이 **Modern ABAP 리팩터링 공정**이다.

### 무엇인가

#### 1. 리팩터링 (Refactoring) 의 3대 철칙
1. **동작 보존 (Behavior Preservation)**: 코드의 생김새와 키워드는 화려하게 바뀌더라도, 사용자가 입력하고 받아보는 최종 연산 결과 데이터는 **이전 구식 코드의 결과와 100% 한 치의 오차도 없이 동일**해야 한다.
2. **가독성 확보 (Clear Intent)**: "이 사람이 여기서 누적 합을 구하려 하는구나" 라는 업무 의도가 코드 줄기에서 명확하게 드러나야 한다.
3. **가드라인 수립 (Safety Guard)**: 대괄호 식을 쓰더라도 `line_exists` 방패를 세워 덤프의 위험을 원천 방어하는 안전선이 도킹되어야 한다.

#### 2. 복합 대입 연산자 (Compound Assignment)
누적 연산을 처리할 때 변수명을 좌우 중복해 적던 지루한 타이핑을 축약한다.
- **`+=`**: 더하기 대입 (예: `gv_total += ls_book-seats.` 은 `gv_total = gv_total + ls_book-seats.` 와 완전히 일치).
- **`-=`**: 빼기 대입.
- **`*=`**: 곱하기 대입.
- **`/=`**: 나누기 대입.
- **⚠️ [선행 선언 필수 규칙]**: `+=` 는 기존에 존재하는 주머니 방에 값을 누적해 얹는 연산이므로, 선언과 동시에는 쓸 수 없다. 반드시 **`DATA(gv_sum) = 0.` 으로 변수를 먼저 탄생시켜 둔 뒤에 `gv_sum += 10.` 을 가동**해야 컴파일 거부를 피한다.

### 어떻게 확인하는가

구식 대입 및 누적 코드를 모던 복합 대입 및 VALUE 연산으로 교체 리팩터링하는 Before-After 규격을 검증한다.

```abap
REPORT zch18_l06_refactor.

TYPES: BEGIN OF ty_line,
         dan    TYPE i,
         mul    TYPE i,
         result TYPE i,
       END OF ty_line.
TYPES ty_line_tab TYPE STANDARD TABLE OF ty_line WITH EMPTY KEY.

START-OF-SELECTION.
  " --------------------------------------------------
  " [ Before - 구식 레거시 코드 ]
  " --------------------------------------------------
  DATA ls_line_old  TYPE ty_line.
  DATA lt_table_old TYPE ty_line_tab.
  DATA lv_sum_old   TYPE i.

  ls_line_old-dan    = 2.
  ls_line_old-mul    = 1.
  ls_line_old-result = 2.
  APPEND ls_line_old TO lt_table_old.

  lv_sum_old = lv_sum_old + ls_line_old-result.

  " --------------------------------------------------
  " [ After - 모던 리팩터링 완료 코드 ]
  " --------------------------------------------------
  " 1. 구조체 대입 및 APPEND를 VALUE 테이블 생성으로 한 식에 묶기!
  DATA(lt_table_new) = VALUE ty_line_tab(
    ( dan = 2 mul = 1 result = 2 )
  ).

  " 2. [선행 선언 필수] 누적할 변수를 먼저 0으로 인라인 선언!
  DATA(lv_sum_new) = 0.
  
  " 3. [복합 대입] += 와 Table Expression 1번방 콕 집기 결합!
  " (이전 구식 코드와 실행 결과 데이터 100% 동일 보증 검수)
  lv_sum_new += lt_table_new[ 1 ]-result.
```

#### 체험/시뮬레이터 설계 (classic ↔ modern 코드를 짝지어 비교)
- **프로세스 플로우**:
  1. 학습자가 좌측 [구식 코드] 와 우측 [모던 코드] 패널을 대조한다.
  2. [리팩터링 실행]을 누르자, 구식 코드가 쪼개지며 변수 선언이 사라지고 우측 모던의 인라인과 복합 연산자로 정돈되는 흐름 지도를 감상한다.
  3. 하단 [결과 무결성 검수기]에 두 코드의 출력 행 수(1건)와 최종 합계(2)가 완벽히 일치하여 녹색 판정 배지가 부여되는 피드백을 감상한다.
- **상태 및 데이터**:
  - `DATA(lv_sum) += 5` 와 같이 선언과 누적을 동시에 짠 코드 상태 -> 컴파일 에러 코드 `Variable lv_sum not yet declared` 하이라이트.
- **피드백**: 리팩터링은 동작 무결성을 절대 수호하면서 코드의 어지러운 스크롤을 또렷하게 정렬하는 정제 공정임을 인지시킨다.

### 실수/주의

- **짧은 한 줄 압축에 미쳐 가독성을 역으로 파괴**: "모던 아밥 구문들을 한 라인으로 다 때려 박을 수 있으니, 9행짜리 구구단 테이블을 생성하고 그중 3번째 줄의 결과를 끄집어내는 코드를 한 줄짜리 체인 코드로 축약해 버려야지" 했다가는, 코드 해독력이 극도로 떨어져 역효과가 나므로, **쓸 수 있다고 기술을 남용하지 말고 가독성을 해치는 구간은 과감하게 줄바꿈과 인라인 변수로 쪼개어 가독성을 사수**해야 한다.

### 정리

- 리팩터링의 핵심 1순위는 코드 외양이 변해도 **실행 결과 데이터는 100% 동일함을 보증**하는 것이다.
- 누적 합 연산은 자기 이름을 중복하지 않는 복합 대입 연산자 **`+=`** 로 대체한다.
- SQL문 자체의 모던화는 다음 장(CH19)의 고유 영역이므로, 이번 장 리팩터링은 **SQL 바깥만 정비**한다.

## CH18-L07 - 실습 — 콘서트앱 모던 리팩터

### 왜 필요한가

우리가 공부한 인라인 선언, VALUE 테이블 빌드, CORRESPONDING 이름 매핑, Table Expression 안전 조회 가드, String Template 메시지 제작, 그리고 복합 대입 누적식을 모두 섭렵했다. 

이제 이 파편들을 **🎫 하나의 살아있는 실무 관통 관문인 '콘서트 예매 프로그램'** 에 투사하여 리팩터링을 완성할 순간이다. 
단순 성공 시나리오만 돌아가는 반쪽짜리 코드가 아니라, 존재하지 않는 예매 번호 `B999` 로 조회 폭탄을 떨어뜨렸을 때도 `line_exists( )` 방패가 정상 발동해 런타임 덤프 없이 안전하게 "조회 실패" 메시지를 뿜고 종료되는 **강건하고 우아한 실무 캡스톤 예제**를 완성해 내야 한다.

### 무엇인가

#### 🎫 콘서트 예매 모던 리팩터링 핵심 3대 타깃 및 방어선 명세
우리는 기존에 작성했던 예매 모듈 코드의 핵심 영역을 다음과 같이 모던 구조로 전환 튜닝한다.

1. **좌석 수 누적 LOOP 리팩터 (`LOOP AT INTO DATA( )` & `+=`)**:
   - `DATA ls_book TYPE zbooking.` 선언을 지워버리고, 루프 선언부 안으로 선언을 전진 배치: `LOOP AT lt_book INTO DATA(ls_book).`
   - 누적 합을 구하는 공식을 모던 복합 대입으로 축약: `lv_total += ls_book-seats.`
2. **신규 예매 테스트 레코드 리팩터 (`VALUE zbooking( )`)**:
   - 한 필드씩 노가다로 5줄 대입하던 코드를, 새 예매 행의 모양 리터럴 그대로 VALUE 연산자에 실어 한 덩어리의 식으로 즉시 생성해 빌드.
3. **예매 번호 안전 조회 리팩터 (`line_exists` & Table Expression & Template)**:
   - 입력받은 예매 코드(`lv_id`)의 실재 여부를 **`line_exists( lt_book[ booking_id = lv_id ] )`** 로 1차 안전 검문.
   - 합격한 경우에만 대괄호 식으로 행을 즉시 꺼내고(`DATA(ls_hit) = lt_book[ ... ]`), 불합격 시 예외 덤프 없이 우아하게 "찾지 못했습니다" 템플릿 문장을 출력하며 탈출.

### 어떻게 확인하는가

리팩터링을 마친 후 3대 테스트 케이스를 수동 실행하여 동작 무결성을 검수한다.

1. `/nSE38` 을 켜고 리팩터링 프로그램을 실행한다.
2. 좌석 수 누적 결과를 수동 검수한다. -> `2석 + 1석 + 4석` 이 정상 산출되어 총합 `7석` 데이터가 100% 동일하게 렌더링되어 나오는지 확인한다. (1문 루프 누적 검수).
3. 존재하는 예매 번호 `B001` 을 조회한다. -> 템플릿에 물려 "예매자 정훈영님의 좌석 수는 2석입니다" 메시지가 최종 포맷에 맞게 또렷하게 잘 조립되어 출력되는지 확인한다. (2문 템플릿 검수).
4. **존재하지 않는 가상의 예매 번호 `B999` 를 조회한다.** -> 대괄호 식이 `LINE_NOT_FOUND` 덤프를 터트리지 않고, `line_exists` 방어벽에서 안전하게 포착되어 "예매번호 B999를 찾지 못했습니다" 라고 평화롭게 안내 메시지를 뿜고 종료되는지 확인한다. (3문 대형 덤프 방어선 검수 - 최고 중요).

```abap
" [ 콘서트 예매 프로그램 모던 리팩터링 완성본 ]
REPORT zch18_l07_concert_modern.

" 원천 예매 내역 테이블이라 가정!
TYPES ty_book_tab TYPE STANDARD TABLE OF zbooking WITH EMPTY KEY.
DATA(gt_book) = VALUE ty_book_tab(
  ( booking_id = 'B001' customer = '정훈영' seats = 2 status = 'N' )
  ( booking_id = 'B002' customer = '손흥민' seats = 1 status = 'N' )
  ( booking_id = 'B003' customer = '아이유' seats = 4 status = 'N' )
).

DATA(gv_search_id) = 'B999'. " 조회할 입력 코드

START-OF-SELECTION.
  " --------------------------------------------------
  " ① 좌석 합계 계산 - 인라인 선언과 += 복합 대입으로 리팩터링!
  " --------------------------------------------------
  DATA(lv_total_seats) = 0.
  LOOP AT gt_book INTO DATA(ls_row).
    lv_total_seats += ls_row-seats.
  ENDLOOP.
  
  WRITE: / |총 예약 좌석 수: { lv_total_seats }석|.

  " --------------------------------------------------
  " ② 신규 예매 추가 - VALUE constructor 리터럴로 리팩터링!
  " --------------------------------------------------
  DATA(ls_new_booking) = VALUE zbooking(
    concert_id = 'C001'
    perf_no    = '01'
    customer   = '정훈영'
    seats      = 2
    status     = 'N'
  ).

  " --------------------------------------------------
  " ③ 개별 예매 조회 - line_exists 방어 가드와 Table Expression 리팩터링!
  " --------------------------------------------------
  " [방어선 철칙] 존재 여부를 검문하여 덤프 폭발을 원천 가드!
  IF line_exists( gt_book[ booking_id = gv_search_id ] ).
    " 참일 때만 안전하게 대괄호 검색 기어 가동!
    DATA(ls_hit) = gt_book[ booking_id = gv_search_id ].
    
    " String Template과 안쪽 공백을 준수해 최종 인쇄 메시지 렌더링!
    WRITE: / |예매자 { ls_hit-customer }님의 좌석 수는 { ls_hit-seats }석입니다.|.
  ELSE.
    " 실패 시에도 덤프 없이 평화롭게 우회 실패 안내문 출력!
    WRITE: / |예매번호 { gv_search_id }를 찾지 못했습니다.|.
  ENDIF.
```

#### 체험/시뮬레이터 설계 (종합 캡스톤 렌더러)
- **프로세스 플로우**:
  1. 학습자가 조회 키에 `B999` 를 입력하고 [모던 리팩터 가동]을 누른다.
  2. PAI 수송선이 출발하고, 1차 검문소 `line_exists` 지점에 불이 번쩍 들어온다.
  3. B999는 테이블에 없으므로 불합격 빨간불이 들어오고, 대괄호 연산 구역을 우회 통과(Bypass)하여 안전 지대로 내려앉는다.
  4. 템플릿 가동 공장에서 "예매번호 B999를 찾지 못했습니다" 스킨 메시지를 조립 인쇄 완료하여, 프로그램 비정상 종료 없이 100% 정상 평화 종료 배지를 띄우는 시각 애니메이션을 감상한다.
- **상태 및 데이터**:
  - `B999 입력 후 line_exists 가드를 끈 채 실행한 상태` -> 런타임 Crash 발생 경보음과 함께 프로그램 흐름도가 박살 나는 폭사 화면 표출.
- **피드백**: 모던 아밥의 참된 가치는 코드의 단순한 축약이 아닌, 강건한 안전 가드(`line_exists`) 위에서 비로소 완성됨을 시각화한다.

### 실수/주의

- **String Template 내부에 복잡한 예매 가능 여부 등의 IF/CASE 비즈니스 판정 조건식을 직접 구겨 넣기**: 템플릿의 중괄호 안에는 오직 최종 출력할 텍스트 변수명만 콤팩트하게 담아야 한다. 텍스트를 한글로 바꿀지 영문으로 바꿀지, 혹은 성공/실패 여부를 판정하는 비즈니스 조건 논리는 **반드시 템플릿 실행 윗단에서 일반 IF/CASE 제어문으로 명확히 도출해 둔 뒤, 템플릿은 출력 조립만 전담**시켜야 가독성의 대원칙을 수호할 수 있다.

### 정리

- **`DATA( )`** 와 **`+=`** 의 이식을 통해 반복 루프의 쓸데없는 선언 스크롤을 전면 정비한다.
- **`line_exists( )`** 는 Table Expression 도입 시 덤프 폭발을 방어하는 필수 대문 게이트다.
- **`String Template`** 은 문장의 최종 레이아웃을 코드와 1:1로 매핑해 가독성을 높인다.
