# CH01_REWRITE - 개발 환경과 첫 프로그램 v1

> 목적: `content/abap/CH01`의 7개 레슨을 "IT 비전공자 입문자가 왜 필요한지부터 실습 확인까지 직관적으로 이해하고 따라갈 수 있는 완성 강의자료" 수준으로 재작성한 기준 원고다. 이 문서는 아직 `content/abap` 원본을 덮어쓴 결과가 아니라, 재작업 판정 이후 실제 반영 전 검수 가능한 v1 기준안이다.

## CH01 전체 설계

CH01의 한 문장 목표는 "단순 따라하기식 입문 장벽을 혁파하고, Presentaion-Application-DB 3-Tier 아키텍처와 SAP GUI RFC, T-code 명령창 세션 제어 세션 명령어(`/n`, `/o`), 프로그램 소스코드 컴파일 엔진(`REPOSRC`, `REPOTEXT` 적재 및 LOAD-Program), WRITE 구문의 List Buffer 메모리 스풀 구조, 로컬 오브젝트 `$TMP` 의 `TADIR` 이송 락(Lock) 제약 및 CTS 이송 요청(TR) Task-Request 부모 자식 소유권 정합성을 수립해 개발 첫걸음을 장악한다"이다.

IT 비전공자 입문자는 아바 에디터(`SE38`)에서 작성한 코드가 데이터베이스 `REPOSRC` 테이블에 Inactive 상태로 고착되어 있음에도 활성화(Ctrl+F3)를 누락하여 옛 런타임 `LOAD-Program` 결과물만 반복 실행하며 헤매고, 다른 화면으로 강제 전환할 때 현재 윈도우 인스턴스를 소거(Kill Session)하는 `/n` 명령의 램 유실 특성을 무시해 작성 중이던 코드를 영구 분실한다.
또한, 저장 시 Local Object 버튼을 눌렀을 때 시스템 데이터 디렉토리 테이블인 **`TADIR`** 의 `DEVCLASS` 필드에 `$TMP` 패키지명이 영구 바인딩되어 강제 이송 락이 걸려 타 장비에 릴리즈가 차단되는 물리적 메커니즘을 알지 못한다.
따라서 본 챕터는 다음과 같은 단계적 해소 장치로 설계를 배치한다.

1. **3-Tier & Diag Protocol**: 사용자 PC(Presentation), 아바 애플리케이션(Application), 데이터베이스(DB)로 3단계 분리된 아키텍처와 SAP GUI 전용 Diag RFC 통신망 규명.
2. **명령창 세션 제어 (/n vs /o)**: 윈도우 세션을 킬(Kill)하고 넘어가서 램 유실을 야기하는 `/n` 과, 6개 세션 한도 내에서 새 창을 사출 복제하는 `/o` 스위치의 차이 교육.
3. **REPOSRC & LOAD-Program**: 저장 시 소스 코드가 적재되는 `REPOSRC` 와 주석 제거 컴파일을 단행해 런타임 실행 바이너리를 메모리에 올리는 활성화 기작 해부.
4. **List Buffer 메모리 스풀**: `WRITE` 로 출력한 화면 결과가 DB 실시간 조회가 아닌, 램 메모리 상의 **List Buffer** 라는 스풀 공간에 1줄씩 렌더링된 텍스트 장표 상태임을 명세.
5. **TADIR DEVCLASS $TMP 가드**: 로컬 저장 시 `TADIR` 테이블에 `$TMP` 가 바인딩되어 이송 트레인 락이 잠기는 물리 제약과 Z패키지 전환 해결책 제공.
6. **Task-Request CTS 부모 자식 트리**: CTS 이송 시 작업자용 자식 Task 가 먼저 릴리즈 풀림(Release)을 통과해야만 부모 Request 요청서가 최종 락 해제되어 이송 기차에 실리는 2계층 결선 수립.

## CH01-L01 - SAP GUI 로그온과 화면 구성

### 왜 필요한가

우리가 보통 아바 코드를 한 줄 쓰기도 전에 마주하는 것은 일반 인터넷 웹 브라우저나 Z파일 실행기가 아닌, 다소 투박해 보이는 'SAP GUI' 라는 전용 로그인 창이다.
IT 비전공자 입문자는 "왜 내 PC 에 개발 프로그램을 깔아서 쓰지 않고, 매번 이 무거운 전용 접속기를 켜서 연결 정보를 치고 들어가야 하는가" 라는 근본적인 의구심을 품는다.
- "ABAP 코드는 내 컴퓨터의 CPU 나 메모리 하드웨어 위에서 실행되지 않는다.
모든 프로그램의 실행, DB 연산, 연산 버퍼 메모리는 전산실에 설치된 '대형 중앙 애플리케이션 서버' 안에서 통째로 굴러간다."
사용자 PC 는 그저 서버가 가공해서 보내준 화면 픽셀 데이터를 모니터에 비춰주고, 내 키보드 타이핑 값을 서버로 실어 날라주는 '디스플레이 중계 단말기' 에 불과하다. 

** Presentation(사용자 모니터), Application(아바 연산 서버), Database(데이터 저장 디렉토리)로 삼분할된 3-Tier 아키텍처의 물리적 협업 망과, 전용 통신 규격(Diag/RFC)을 수호하며 중앙 서버 공간(Client)에 정식 로그온하는 기술**이 필요하다. 그것이 **SAP GUI 로그온**의 완수다.

### 무엇인가

#### 🧭 [ SAP 3-Tier 물리 아키텍처 명세 ]
- *모든 SAP 연산은 사용자 PC 의 자원을 탐하지 않고, 아래의 철저한 3대 분할 영역에 걸쳐 중앙 집권식으로 작동한다.*

```text
[1층] Presentation Layer (표현 계층 - SAP GUI) :
   사용자의 PC 에서 돌아가는 그래픽 뷰어 프로그램이다. 
   서버가 가공 완료해 텍스트와 그래픽으로 쏴준 화면 전송 패킷을 그려주고 키보드 입력을 회수해 간다.
   │ ( SAP 전용 고속 Diag / RFC 프로토콜 통신망 작동 )
   ▼
[2층] Application Layer (애플리케이션 계층 - ABAP Instance Server) :
   실제 아바 프로그램의 로직 연산, 루프, WRITE 버퍼 계산을 단행하는 연산 두뇌 영역이다.
   컴파일러와 WAS 메모리가 상주하여 연산을 종결한다.
   │ ( Open SQL 데이터베이스 통신 )
   ▼
[3층] Database Layer (데이터베이스 계층 - R/3 DB 또는 HANA DB) :
   물리 디스크 위에 테이블 장부를 격리해 보관하고, 데이터 트랜잭션 롤백과 커밋을 소화하는 저장소다.
```

#### 1. System ID (SID - 시스템 식별 삼형제 키)
- 전산실에 가동 중인 수많은 SAP 서버 중 내가 접속하려는 인스턴스를 구분하는 영어 대문자 3글자 고유 징표 명칭이다. (예: 개발 서버 `DEV`, 검수 서버 `QAS`, 운영 라이브 서버 `PRD`)

#### 2. Client (고객사 격리 테넌트 방)
- **하나의 물리 서버와 동일한 데이터베이스 디렉토리 환경 내에서, 데이터를 철저히 칼 격리하여 서로 보이지 않게 막아주는 3자리 숫자(000~999) 테넌트 독립 방이다.**
- **격리벽**: 동일한 `zbooking` 테이블일지라도 100번 Client 방에서 넣은 예매는 200번 Client 방 로그온 시 절대로 보이지 않는다. (MANDT 라는 첫 키 필드 구분을 통해 아바 커널이 자동으로 격리해 주기 때문이다.)

#### 3. SAP Easy Access
- 로그온 후 마주하는 첫 종합 메뉴판 화면이다. 이곳의 왼쪽 위 명령창(Command Field)이 모든 화면으로 점프하는 지름길 게이트웨이 역할을 전담한다.

### 어떻게 확인하는가

Presentation GUI 에서 지정 SID 와 Client 번호를 타고 Application 서버로 로그온이 도킹하는 연결 구도를 검증한다.

```text
[1단계] SAP Logon 클라이언트 기동 :
   - SID : DEV (개발 인스턴스 서버 선택)
   - Connection 설정 완료 -> 더블 클릭 격발!
   
[2단계] 로그온 크레덴셜 기입 :
   - Client : 100 (100번 개발 환경 격리 방 입장 선언!)
   - User   : JEONGHY (정훈영 계정)
   - PW     : ********
   - Lang   : KO (한국어 텍스트 패키지 도킹 지시!)
   - Enter 격발 -> Application 서버가 비밀번호 및 Client 인가 여부를 DB 와 대조 판정!
   
[3단계] 도킹 성공 :
   - SAP Easy Access 첫 화면 로드 완료! (3-Tier 배선 정식 결선 및 세션 램 활성화 완료!)
```

#### 체험/시뮬레이터 설계 (3-Tier 데이터 수송선)
- **프로세스 플로우**:
  1. 학습자가 [Presentation User PC] 와 [Application Server], [Database] 삼각 기지를 본다.
  2. [로그온] 을 클릭하자, ID/PW 가 실린 [Diag 프로토콜 수송선] 이 날아가 Application 기지에 찰딱 도킹한다.
  3. [Client = 100] 번 스위치를 켜자, DB 기지 속의 100번 격리 구역 창만 초록색으로 밝혀지며 데이터 배송 케이블이 연결된다.
  4. 이때 [Language = FAKE] 로 치자, 지원하지 않는 언어 패키지라며 수송선이 거부당해 튕겨 나가는 도킹 실패 피드백을 감상한다.
- **상태 및 데이터**:
  - `서버 IP 설정이 끊겨 연결 지연 타임아웃이 난 상태` -> 런타임 결과: `WSAETIMEDOUT. Connection to Application Server failed` 하이라이트.
- **피드백**: SAP GUI 는 쌩 PC 독립 툴이 아니라, 3-Tier 아키텍처 파이프라인의 입구 단말기임을 이해한다.

### 실수/주의

- **Client 번호 입력을 대충 넘어가 기본값 000 번 방으로 로그온을 시도**:
  - 000번 방은 SAP 시스템 설치 시 제공되는 '공장 초기 표준 격리 방' 이어서 일반 개발물 생성이 엄격히 차단되며, 락 에러를 뿜고 쫓겨난다.
  - **반드시 아카데미가 지정해 준 개발사 전용 방(예: 100번 또는 800번) 번호를 명시 기입하고 들어가는 습관을 수호해야 한다.**

### 정리

- 아바는 **`Presentation - Application - DB`** 3-Tier 분산 서버 아키텍처로 가동된다.
- **`Client`** 3자리 숫자는 동일 디스크 데이터베이스 내에서 비즈니스를 격리하는 독립 격리 방이다.
- 첫 화면 **`SAP Easy Access`** 명령창을 통해 모든 점프 처리를 시작한다.

## CH01-L02 - T-code와 SE38 첫 실행

### 왜 필요한가

SAP Easy Access 첫 화면에 입장했다.
이제 개발 에디터 창을 켜서 코딩을 준비해야 한다.
- " 아바 코딩 편집기를 켜려는데, 메뉴 트리를 `도구 -> ABAP Workbench -> 개발 -> ...` 로 한참을 마우스로 펼쳐 들어가야 해서 시간이 너무 오래 걸리고 번거롭다."
실무 아바 개발자는 이 길고 느린 메뉴 트리를 절대 사용하지 않는다. 
각 화면마다 할당된 짧은 주소 배지인 **트랜잭션 코드 (T-code)** 를 명령창에 쳐서 1초 만에 점프한다.
그런데 이 점프 명령어 작동 시, 현재 켜져 있는 윈도우 인스턴스를 소멸하고 넘어가는지, 아니면 새 창을 복제하여 여는지에 대한 세션 제어 원리를 꿰차지 못하면, 기껏 에디터에서 열심히 코딩하던 램 버퍼 소스 코드를 다른 화면 점프 시 통째로 날려 먹는 참변을 겪는다.

**명령창 점프 접두어(/n: 세션 소멸 이동, /o: 신규 세션 사출 복제)의 세션 램 작동 제어 원리를 터득하고, Z/Y 네임스페이스 예약 제약 준수 하에 개발 에디터(SE38)를 기동하는 기술**이 필요하다. 그것이 **T-code 와 SE38 에디터의 장악**이다.

### 무엇인가

#### 1. T-code (Transaction Code - 트랜잭션 코드)
- SAP 내부에 존재하는 수천 개의 개별 전산 화면(SE38, SE11 등)에 가설해 둔 고유 주소 배지 번호다. (웹 브라우저 주소창에 URL 주소를 쳐서 직행하는 원리와 같다.)

#### 🧭 [ 명령창 세션 제어 명령어 /n 과 /o 아키텍처 명세 ]
- *명령창에 쌩 T-code 를 치는 것과 접두어를 얹는 것은, 서버 내부 램 세션 인스턴스를 다루는 기작이 판이하게 다르다.*

```text
[1] 쌩 T-code 입력 (예: SE38) :
   오직 로그인 직후 아무 창도 안 열린 'SAP Easy Access' 상태에서만 직행 이동을 허용한다. 
   만약 SE11 같은 다른 작업 화면 내에서 쌩 SE38 을 치면 "Unknown Command" 에러가 뜨며 씹힌다.
   │
[2] /n 접두어 입력 (예: /nSE38 - Session Kill & Move) :
   "현재 켜져 있는 윈도우 세션과 연산용 램 메모리 버퍼를 통째로 즉시 삭제(Kill)하고, 
   그 자리에 SE38 화면 인스턴스를 새로 얹어 기동하라" 는 강제 덮어쓰기 명령이다. 
   ( 저장 안 한 수정 코드가 있다면 경고 없이 영구 유실될 수 있으므로 주의해야 한다. )
   │
[3] /o 접두어 입력 (예: /oSE38 - Session Spawn & Clone) :
   현재 작업 중인 화면은 건드리지 않고 그대로 놔둔 채, 
   완전히 새로운 신규 가상 윈도우 창(GUI Session - 최대 6개 제한)을 복제 사출하여 
   그 새 창에 SE38 을 켜라는 명령이다. (멀티태스킹의 핵심이다.)
```

#### 2. SE38 (ABAP Editor - 아바 에디터)
- 아바 실행형 프로그램(Report)의 소스를 키보드로 타이핑하고, 구문 검사, 저장, 활성화, 테스트 실행을 단행하는 아카데미의 메인 개발 전산실이다.
- **Z 또는 Y 네임스페이스 규칙**: **프로그램 명칭 생성 시 무조건 첫 글자는 `Z` 나 `Y` 로 시작해야 한다. 그 외 `A~X` 로 시작하는 단어는 SAP 사가 전 세계 공통으로 배포하는 'SAP 표준 오브젝트' 용 영역이므로, 입문자가 침범해 만들면 시스템이 네임스페이스 권한 위배 에러를 내며 저장을 거절한다.**

#### 3. SE11 (ABAP Dictionary)
- 테이블이나 데이터 Element 등 데이터 설계도를 그리는 전산실이다.
#### 4. SE80 (Object Navigator)
- 프로그램, 클래스, 패키지 등 내 모든 관련 유산들을 한눈에 트리 구조로 훑어보는 종합 탐색기 화면이다.

### 어떻게 확인하는가

/o 접두어로 기존 에디터 버퍼를 유지한 채 세션을 추가 사출해 SE38 로 점프하는 흐름을 검증한다.

```text
[1단계] SE11 에디터 가동 중 :
   - 명령창에 쌩 SE38 입력 -> Enter 격발! -> 딩- 경고음 작동! ("이 화면에선 쌩 코드 점프 불가")
   
[2단계] /o 접두어로 새 세션 사출 실행 :
   - 명령창에 /oSE38 입력 -> Enter 격발!
   - 램 분석: 기존 SE11 화면 세션은 메모리에 그대로 유지!
   - 새로운 2번째 윈도우 창이 복제 사출되며 SE38 Initial Screen 기동 완료! (멀티태스킹 돌입)
   
[3단계] ZHELLO 프로그램 창조 :
   - Program 칸에 'ZHELLO_WORLD' 기입 (Z 네임스페이스 규칙 준수!)
   - Create 단추 클릭 -> Type = Executable Program 선택 및 Local Object 저장 낙찰!
```

#### 체험/시뮬레이터 설계 (T-code 세션 킬/사출 모션)
- **프로세스 플로우**:
  1. 학습자가 [SE11 켜져 있는 화면 윈도우 #1] 을 본다.
  2. 명령창에 [/nSE38] 을 치고 쏜다. 윈도우 #1 이 번쩍하더니 SE38 로 덮어씌워지고, [메모리 버퍼] 상자 속의 ZTABLE 소스가 싹 증발 소거되는 참상을 목격한다.
  3. 다시 SE11 을 켜고, 이번엔 [/oSE38] 을 친다.
  4. 윈도우 #1 은 가만히 있고, 우측에 [윈도우 #2] 가 촥 복제 분열하여 사출되어 그 자리에 SE38 이 기동하는 세션 분리 피드백을 감상한다.
- **상태 및 데이터**:
  - `세션 사출 한도 6개를 초과해 /o 를 친 상태` -> 런타임 결과: `Maximum number of GUI sessions reached` 적색 경고 작동.
- **피드백**: `/n` 은 현재 세션을 킬하고 세션 램을 날리며, `/o` 는 신규 복제 사출하는 세션 제어임과 Z/Y 네임스페이스 장벽을 인지한다.

### 실수/주의

- **다른 프로그램 개발 화면에서 에디터를 저장하지 않은 채 성급하게 명령창에 /nSE38 을 기격발**:
  - 이 실수 시 아바 에디터는 PAI/PBO 데이터 플러시 없이 세션을 죽여버리므로, 지난 30분 동안 타이핑한 소스 코드가 메모리 소멸과 함께 증발해 흔적도 없이 사라진다.
  - **화면 점프 전에는 무조건 Ctrl+S 로 저장하는 습관을 수호해야 한다.**

### 정리

- **`T-code`** 는 복잡한 메뉴 대신 명령창을 통해 다이렉트로 화면 주소로 점프하는 단축키다.
- 접두어 **`/n`** 은 세션을 킬하고 넘어가며, **`/o`** 는 최대 6개 한도 내에서 새 세션 창을 복제 사출한다.
- 커스텀 프로그램은 반드시 **`Z`** 나 **`Y`** 로 시작해야 네임스페이스 충돌을 피한다.

## CH01-L03 - REPORT와 주석

### 왜 필요한가

SE38 에디터에서 첫 ZHELLO 프로그램을 개설했다.
이제 백지에 아바 코드의 뼈대를 갖춰야 한다.
- " 프로그램 맨 첫 줄에 `REPORT zhello.` 라고 적혀 있는데 이게 무슨 뜻인가? 
그리고 코드를 실행했을 때, 컴퓨터가 읽어서 계산해야 할 진짜 소스 코드와, 우리 개발자가 이해하기 위해 남겨두는 텍스트 설명글(주석)을 컴파일러가 어떻게 구분하는가?"
이 프로그램 시작 선언을 빠뜨리거나 문법 꼬리표 마침표(`.`)를 생략하면 빌드 조차 거부당한다. 
또한 주석 기호를 잘못 적어 설명 텍스트를 컴파일러가 진짜 명령 코드로 읽으려 들다가 정합성이 깨져 구문 에러를 뿜는다.

**프로그램의 첫 선포인 REPORT 구문과 문장 종결자 마침표(.)를 수호하고, 한 줄 주석(*) 및 부분 주석(\") 기호를 적어 컴파일 엔진이 텍스트 설명을 스킵하게 만드는 기초 구조 기술**이 필요하다. 그것이 **REPORT 와 주석**의 완수다.

### 무엇인가

#### 1. REPORT zhello. (프로그램 시작 신고)
- **모든 실행형 아바 프로그램(Report Program)의 1순위 대가리 줄에 단 한 번 선포해야 하는 마침표 문장이다. 아바 컴파일러에게 "이 프로그램은 사용자가 실행 버튼(F8)을 누르면 첫머리부터 즉각 순차 기동될 독립 실행형 소스다" 라고 정식 신고하는 필수 선언문이다.**

#### 2. 마침표 (.) - Sentence Terminator
- *아바 언어의 가장 삼엄한 문법 종결 도장이다.*
- 자바나 C언어가 문장 끝에 세미콜론(`;`)을 찍듯, 아바는 반드시 마침표(`.`)를 찍어주어야 한 줄 문장이 종결된다. (마침표를 빠뜨리고 다음 줄을 적으면 두 문장이 한 단어로 결합되어 문법 덤프를 터트린다.)

#### 3. 주석 (Comment - 컴파일러 패스 카드)
- *개발자가 소스 코드 틈새에 남겨두는 인간용 메모장이다.*
- **한 줄 전체 주석 (`*` - 별표)**: **무조건 라인의 '맨 첫 칸(1열 Column)' 에 별표를 기재해야 작동한다. 2열이나 중간에 쓰면 주석으로 인식되지 않고 에러가 난다.**
- **부분/꼬리 주석 (`"` - 따옴표)**: 코드 코딩 문장이 도는 중간 줄이나 꼬리 칸 어디서든 따옴표를 심고 뒤에 메모를 적으면, 따옴표 우측 영역만 컴파일러가 투명하게 스킵해 통과시킨다.

### 어떻게 확인하는가

REPORT 선언과 두 가지 주석 기호가 혼합 조립된 에디터 뼈대 구조를 검증한다.

```abap
*----------------------------------------------------------------------*
*  1. [한 줄 전체 주석] : 반드시 1열 라인 맨 앞에 별표(*)를 얹어 기술!
*  작성자: 정훈영 사원
*  목적: 첫 아바 뼈대 프로그램 및 주석 스킵 확인
*----------------------------------------------------------------------*
REPORT zhello_world. " 2. [★ REPORT 프로그램 선포 및 종결 마침표 장착!]

" 3. [부분 주석] : 따옴표 뒷영역은 컴파일러가 완전히 무시함!
WRITE 'Hello'. " 화면에 텍스트를 출력하라 (문법 마침표 필수!)
```

#### 체험/시뮬레이터 설계 (컴파일러 REPOSRC 로드 인스턴스)
- **프로세스 플로우**:
  1. 학습자가 [REPORT zhello.] 소스를 칩니다.
  2. 첫 칸에 별표(*) 없이 [이 프로그램은 정훈영이 짬] 이라고 메모를 남깁니다.
  3. [컴파일러 전원] 을 켜자, 메모 글자를 해석 불가능한 명령어로 오역해 [적색 컴파일 에러등] 이 들어오는 충돌을 본다.
  4. 메모 맨 앞에 [별표(*)] 카드를 꽂아 첫 칸을 메운다.
  5. 컴파일러가 메모 줄을 가볍게 패스 스킵하고 `REPORT` 선언만 인식해 초록색 정상 합격등을 점등하는 빌드 피드백을 감상한다.
- **상태 및 데이터**:
  - `REPORT 문장 끝에 마침표(.)를 누락한 상태` -> 런타임 결과: `Report statement not terminated. Syntactical parsing error` 하이라이트.
- **피드백**: 아바 컴파일러는 문장 마침표(`.`)로 문맥을 절단하고, 첫 열 별표(`*`)와 따옴표(`"`)를 디텍션해 주석을 스킵함을 깨닫는다.

### 실수/주의

- **코드 문장 중간 빈칸 뒤에 따옴표(") 대신 별표(*)를 적어서 주석을 달려고 시도**:
  - `WRITE 'Hello' * 주석달기` 라고 적으면 컴파일러는 곱하기(*) 연산자로 오역하여 "Hello 와 주석달기 변수 간 곱셈 연산이 불가하다" 고 문법 에러를 터트린다.
  - **줄 중간 주석은 무조건 따옴표(")를 수호해야 한다.**

### 정리

- 실행형 아바의 첫 줄은 반드시 **`REPORT 프로그램명.`** 으로 개시한다.
- 아바의 모든 문법 문장은 **`마침표 (.)`** 로 맺어야 빌드 덤프를 피한다.
- **`맨 첫 칸 별표 (*)`** 와 **`중간 따옴표 (")`** 는 아바가 인식하는 양대 주석 약속이다.

## CH01-L04 - WRITE와 첫 출력

### 왜 필요한가

프로그램 구조 뼈대는 완성했다.
이제 내 아바가 화면에 드디어 첫 글자를 뱉어내게 만드는 출력(Output) 명령을 격발해야 한다.
- " 프로그램에게 'Hello' 라는 글자를 모니터 화면에 뿌리라고 명령하고 싶다.
그런데 이 출력한 텍스트가 어떤 원리로 렌더링되며, 줄 바꿈이나 수직 구분선 같은 제어는 어떻게 단행하는가?"
아바 런타임이 작동해 화면을 그릴 때, 데이터베이스 테이블을 실시간으로 계속 긁어서 화면에 렌더링하는 구조로 설계하면 데이터 트래픽이 감당되지 않는다.
- "화면에 뿌릴 데이터를 램 메모리 상의 '임시 인쇄 대기판(List Buffer)' 에 차곡차곡 1줄씩 스트리밍으로 얹어 조립해 둔 뒤, 최종 조립본만 화면에 한 번에 쏴주는 기술이 필요하다."
그것이 **WRITE 명령어와 List Buffer** 의 가동이다.

### 무엇인가

#### 1. WRITE '텍스트' (화면 출력 명령)
- 싱글 쿼테이션(`'`)으로 감싼 텍스트나 변수 데이터를 화면에 출력하도록 지시하는 아바의 가장 고전적이고 확실한 출력 지시어다.

#### 🧭 [ WRITE 와 List Buffer 메모리 스풀 아키텍처 명세 ]
- *우리가 WRITE 를 때릴 때, 백엔드 메모리 단에서는 즉각 모니터 픽셀을 켜지 않고, 아래와 같은 List Buffer 임시 창고를 거친다.*

```text
[1] WRITE 'Hello' 격발 :
   아바 엔진은 이 'Hello' 라는 텍스트 데이터를 Presentation 화면으로 바로 쏘지 않는다.
   임시 램 메모리 스풀 공간인 List Buffer 의 1번째 행, 1번째 열 위치에 텍스트 바이트를 살포시 얹어 적재한다.
   │
[2] 1줄 단위 스트리밍 적재 :
   이어서 WRITE / 'World' 를 만나면, List Buffer 의 다음 2번째 행으로 개행(/)해 
   'World' 바이트를 누적 징검다리 기입해 둔다.
   │
[3] 프로그램 종결 시 (End of Program) :
   로직 연산이 모두 끝나고 프로그램이 닫히는 순간, 
   List Buffer 에 완성되어 상주하던 텍스트 판때기(스풀)를 SAP GUI 화면으로 통째로 단 1회 전송해 
   사용자 모니터에 데칼코마니처럼 찍어 렌더링을 끝마친다.
```

#### 2. 슬래시 (/) - 개행 제어기
- `WRITE / '텍스트'` 와 같이 슬래시를 기재하면, List Buffer 에서 현재 쓰던 행을 닫고 **'다음 줄 첫 번째 열'** 로 강제 줄 바꿈 이동해 쓰기를 시작하라는 제어 기호다.

#### 3. sy-vline 과 sy-uline (시스템 가로/세로선)
- **sy-vline (Vertical Line)**: List Buffer 에 수직 기둥 벽(`|`)을 그려주는 시스템 상수다.
- **sy-uline (Under Line)**: 1줄 전체를 가로지르는 수평 밑선(Underline)을 쫙 그어 영역을 쪼개 주는 시스템 상수다.

### 어떻게 확인하는가

슬래시 개행과 vline, uline 을 믹스 결합해 메모리 버퍼에 장표를 그리는 코드를 검증한다.

```abap
REPORT zhello_output.

" 1. [1행 1열에 첫 글자 기입]
WRITE '콘서트 예매 현황'.

" 2. [sy-uline 가로선 쫙 긋기 - 다음 줄에 그리기 위해 슬래시 개행 병행!]
WRITE / sy-uline. 

" 3. [개행 후 세로선 기둥 vline 과 텍스트 조립]
WRITE: / sy-vline, '예약번호', 
         sy-vline, '예약자명', 
         sy-vline.
         
" 4. [가로 닫는 선]
WRITE / sy-uline.
```

#### 체험/시뮬레이터 설계 (List Buffer 메모리 조작기)
- **프로세스 플로우**:
  1. 학습자가 [List Buffer 램 메모리 판넬 (Grid)] 을 본다.
  2. [WRITE 'Hello'] 코드를 넣자, Grid 1행 1열 칸에 H-e-l-l-o 데이터 비트가 심어진다.
  3. [개행 슬래시 (/) 없이 WRITE 'World'] 를 꽂자, 1행 7열 뒤에 World 가 이어 붙는다.
  4. [개행 슬래시 (/) ON] 을 켜자, 2행 1열 칸으로 포커스가 툭 떨어지며 World 데이터가 안착되고, 프로그램이 끝나자 완성된 판넬이 [SAP GUI 모니터] 로 데칼코마니 복사 전송되어 렌더링되는 시각 피드백을 감상한다.
- **상태 및 데이터**:
  - `개행 제어 없이 가로선 uline 만 난사한 상태` -> 런타임 결과: `List Buffer Width Exceeded. Character overlap` 하이라이트.
- **피드백**: WRITE 의 실체는 모니터 화면이 아니라 메모리 List Buffer 스풀 공간의 좌표 기입 연산임을 깨닫는다.

### 실수/주의

- **WRITE 'Hello' 문장 끝 마침표(.) 뒤에 쌩뚱맞게 슬래시(/)를 적어서 개행을 시도**:
  - `WRITE 'Hello'. / 'World'.` 라고 적으면 컴파일러는 마침표 뒤의 슬래시(/)를 문장의 시작으로 읽어 "슬래시라는 명령어가 없다" 고 컴파일 즉사 에러를 낸다.
  - **슬래시 개행 기호는 무조건 WRITE 명령어 몸체 안에(마침표 찍기 전에) 심어주어야 작동함을 수호해야 한다.**

### 정리

- **`WRITE`** 문장은 화면 출력을 위해 임시 램 버퍼인 **`List Buffer`** 에 데이터를 1줄씩 누적 기입한다.
- **`슬래시 (/)`** 제어자는 다음 행 첫 열로 개행 줄 바꿈을 명령한다.
- 수직선은 **`sy-vline`**, 수평선은 **`sy-uline`** 시스템 상수를 활용해 표 칸막이를 조립한다.

## CH01-L05 - WRITE 서식과 다듬기

### 왜 필요한가

기본 WRITE 출력은 완성했다.
이제 출력 장표의 품질을 높이기 위한 정밀 서식(위치, 정렬, 색상) 다듬기가 문제다.
- " 티켓 장표의 좌측 여백을 10칸 띄우고 싶고, 글자는 딱 20글자 크기로 잘라 우측 정렬하고 싶다.
그리고 실패한 예매 내역은 빨간색으로, 성공한 건은 초록색으로 화려한 테마 색을 칠하고 싶다."
IT 비전공자 입문자는 "HTML 처럼 복잡한 CSS 도 없는데, 아바 쌩 텍스트 창에서 어떻게 폭과 컬러 스타일을 제어하는가" 의구심을 가진다.
만약 이 텍스트 칸막이 폭 위치 지정을 정교하게 매칭해 두지 않으면, 한글(2바이트)과 영문(1바이트)의 물리 글자 크기 차이로 인해 표 칸이 지그재그로 어긋나 뭉개지는 레이아웃 파손 버그를 겪는다.

**WRITE 구문 뒤에 위치 좌표와 출력 폭을 기재하고(pos(len)), RIGHT-JUSTIFIED 정렬 배지를 얹으며, COLOR 옵션을 통해 List Buffer 의 가변 텍스트 바이트 마킹을 정밀 제어하는 기술**이 필요하다. 그것이 **WRITE 서식 제어**다.

### 무엇인가

#### 1. WRITE pos(len) '텍스트' (위치와 폭 강제 지정)
- **pos(Position)**: List Buffer 의 몇 번째 열(Column) 자리에 시작 좌표를 찍을지 지정하는 숫자다.
- **len(Length)**: 출력할 데이터가 아무리 길거나 짧아도 딱 '몇 바이트 크기 칸' 만큼만 공간을 확보해 표현하라고 제한하는 폭 크기 지정이다. (예: `WRITE 10(20) 'Hello'` -> 10번째 칸부터 시작해 20바이트 폭 크기로 렌더링).

#### 2. LEFT-JUSTIFIED / CENTERED / RIGHT-JUSTIFIED (정렬)
- pos(len)으로 확보해 둔 칸막이 상자 내부에서 글자를 왼쪽, 가운데, 오른쪽 어디로 밀착시켜 정렬할지 지시하는 정합성 옵션이다. (금액은 RIGHT, 글자는 LEFT 가 실무 철칙이다.)

#### 3. COLOR col_id (배경 색상 입히기)
- *텍스트 장표의 비주얼을WOW 하게 만드는 핵심 스타일이다.*
- 1번(Blue)부터 7번(Orange)까지 지정된 시스템 색상 코드를 텍스트 뒷배경에 촥 마킹해 칠해준다.
  - **`COLOR COL_POSITIVE` (초록색)**: 성공, 정상 완료 상태 등을 통보할 때의 안전 지표다.
  - **`COLOR COL_NEGATIVE` (빨간색)**: 에러, 정원 초과, 결제 거부 등을 강력 경고할 때의 위험 지표다.

### 어떻게 확인하는가

좌표 폭과 RIGHT 정렬, 그리고 NEGATIVE 빨간 색상이 가미된 WRITE 조립 소스를 검증한다.

```abap
REPORT zhello_style.

WRITE: / sy-uline.

" 1. [1열부터 10바이트 폭으로 '예약번호' 출력 & 정렬 지정]
WRITE: / 1(10) '예약번호' LEFT-JUSTIFIED,
         12(20) '예약자명' CENTERED,
         " 2. [★ 금액 정보는 35열부터 15바이트 폭으로 우측 정렬 지정!]
         33(15) '결제금액' RIGHT-JUSTIFIED.

WRITE: / sy-uline.

" 3. [실제 데이터 기입 및 에러 행 빨간색 마킹!]
WRITE: / 1(10) 'T0001',
         12(20) '정훈영',
         33(15) '50,000' RIGHT-JUSTIFIED COLOR COL_POSITIVE. " 성공 초록등!

WRITE: / 1(10) 'T0002',
         12(20) '임꺽정',
         " 4. [★ 잔액 부족 실패 행은 빨간색 Negative 마킹!]
         33(15) '0'      RIGHT-JUSTIFIED COLOR COL_NEGATIVE.
```

#### 체험/시뮬레이터 설계 (List Buffer 스타일링)
- **프로세스 플로우**:
  1. 학습자가 [결제금액 열] 과 [COLOR COL_NEGATIVE] 카드를 본다.
  2. [금액 정렬 = LEFT] 로 지정하면, 문자 폭 끝이 안 맞아서 표가 지그재그로 어긋나는 오작동 레이아웃을 확인한다.
  3. [금액 정렬 = RIGHT] 로 칩을 교체하자 소수점 끝자리가 찰딱 줄을 서 정렬된다.
  4. [COL_NEGATIVE] 를 2번째 행 금액에 드롭하자, 배경이 촥 빨간색으로 타오르며 렌더링되어 실패 상태를 한눈에 식별하게 해주는 비주얼 피드백을 감상한다.
- **상태 및 데이터**:
  - `한글 2바이트를 고려하지 않고 폭(len)을 2글자 크기로 억지로 우겨넣어 한글이 깨진 상태` -> 런타임 결과: `Character truncation. Layout alignment broken` 하이라이트.
- **피드백**: 텍스트 장표 서식은 바바 단위의 좌표 폭 매칭과 비주얼 마킹(COLOR)의 조화가 생명줄임을 배운다.

### 실수/주의

- **COLOR COL_NEGATIVE 를 주고 텍스트를 출력하면서, INTENSIFIED OFF (배경 색조 약화) 옵션 누락으로 눈부심 유발**:
  - 배경이 너무 눈부신 쌩 빨강으로 칠해져 화면 가독성을 오염시키고 현업의 피로도를 올린다.
  - **배경색 지정 시에는 상황에 따라 `INTENSIFIED OFF` 나 `INVERSE ON` (글자 색상 반전) 옵션을 조율해야 함을 수호해야 한다.**

### 정리

- **`pos(len)`** 문법은 List Buffer 내의 좌표 시작 칸(pos)과 할당 폭(len)을 규정한다.
- 숫자는 **`RIGHT-JUSTIFIED`** (우측), 문자는 **`LEFT-JUSTIFIED`** (좌측) 정렬이 실무 표준이다.
- 정상은 **`COL_POSITIVE`** (초록), 오류는 **`COL_NEGATIVE`** (빨강) 시스템 색상으로 가시성을 높인다.

## CH01-L06 - $TMP와 개발 패키지

### 왜 필요한가

WRITE 스타일 장표까지 이쁜 프로그램을 빚어냈다.
이제 이 프로그램 ZHELLO 의 저장 공간 분류가 문제다.
- " 어제 열심히 짠 첫 프로그램을 Local Object 에 저장했다.
오늘 동료 개발자 정훈영 대리가 '내가 고쳐볼 테니 그 ZHELLO 소스 코드 좀 내 QA 나 내 PC 화면으로 이송해 달라' 고 요청해 왔다."
Local Object 버튼을 누르는 것은 "내 컴퓨터 쌩 바탕화면에 텍스트 파일 하나 저장한 것" 과 다를 바 없는 격리 락 상태다. 
- "다른 기기로 이송(Transport) 가능한 '공식 전산 패키지 폴더(ZPACKAGE)' 를 시스템에 개설하고, 내 프로그램을 그 폴더 내부 영역으로 정식 이송 배속하는 기술이 필요하다."
그것이 **$TMP 로컬 탈출과 개발 패키지 이송**의 완수다.

### 무엇인가

#### 1. $TMP (로컬 전용 임시 패키지)
- **어떠한 이송 트레인(이송 요청서)에도 엮이지 않는 SAP 내의 가상 임시 창고다. 다른 동료에게 공유되지 않고 다른 QA/운영 시스템으로 배포가 원천 락 걸린 격리 지대다.**

#### 2. 개발 패키지 (Z-Package)
- ZHELLO 같은 내 프로그램 개체들을 정식으로 묶어 담을 수 있는 '공식 전산 관리 폴더' 다. (네이밍 규칙 `Z` 로 시작하여 개설한다.)
- 이 폴더에 담긴 오브젝트들은 비로소 **이송 요청서(TR)** 라는 배송 열차 티켓에 실려 바다(QA, 운영)를 건널 권한을 얻는다.

#### ⚠️ [ $TMP 개발물의 TADIR 시스템 이송 락 제약 명세 ]
- *개발물이 개발기에 영구 격리 구금되는 최악의 배포 차단 장벽이다.*
- **TADIR (개발 오브젝트 디렉토리 대장)**: SAP 시스템 내부에는 모든 프로그램의 소속 패키지명을 관리하는 `TADIR` 이라는 대형 테이블이 상주하고 있다.
- **물리 락 작동**: **Local Object 로 저장하는 순간 `TADIR` 테이블의 `DEVCLASS` (패키지명) 필드에 `$TMP` 가 기입된다. 이 DEVCLASS 가 `$TMP` 인 레코드는 CTS 배포 엔진이 "이송 절대 불가 로컬 개체" 로 규정해 강제 락을 걸기 때문에, 어떠한 배포 기차(TR)를 생성해 쑤셔 넣으려 해도 "이송 불가 오브젝트" 라며 배송 자체를 전면 기각 기각한다.**
- **해소 (패키지 재할당)**: 이를 풀기 위해서는 반드시 **SE80 이나 SE38 속성 창에 들어가 [Change Package] 버튼을 클릭하고 `$TMP` 를 정식 `ZPACKAGE` 로 스왑 재할당**해 주어야만 비소로 `TADIR-DEVCLASS` 에 정식 패키지명이 적히며 이송 락이 탁 풀린다.

### 어떻게 확인하는가

$TMP 에 잠겨 이송 락이 걸린 오브젝트를 정식 Z 패키지로 이전해 락을 해소하는 흐름을 검증한다.

```text
[1단계] TADIR 이송 락 상황 확인 :
   - SE80 에서 zhello 프로그램 조회.
   - 소속 패키지 : $TMP (TADIR-DEVCLASS = '$TMP') -> 이송 락 잠금 상태!
   - 이송 요청서(TR)에 ZHELLO 를 담으려 시도 -> 삑- "Object cannot be transported" 경고 발생!
   
[2단계] 패키지 재배속 단행 :
   - SE38 -> zhello 입력 -> Go to -> Object Directory Entry 클릭!
   - Package 입력칸의 '$TMP' 를 도려내고, 정식 개설한 'ZBOOKING_PACK' 입력!
   - Save (저장) 클릭!
   
[3단계] 이송 락 해소 및 TR 티켓 발급 :
   - 시스템이 락 해제를 감지하고 "이송 요청서(TR) 번호를 입력하십시오" 팝업창을 즉각 띄움!
   - 신규 TR 'EK0K900182' 발급 및 조립 완료! (TADIR 이송 락 완벽 탈출 성료!)
```

#### 체험/시뮬레이터 설계 (TADIR 이송 락 해제 기어)
- **프로세스 플로우**:
  1. 학습자가 [$TMP 로컬 상자] 속에 갇힌 [ZHELLO 프로그램] 을 본다.
  2. 우측에 [운영계로 가는 이송 트레인] 이 대기 중이다.
  3. ZHELLO 를 열차에 실으려 하자, [$TMP 이송 락 쇠사슬] 이 ZHELLO 를 잡아당기며 "TADIR-DEVCLASS is Local! Transport Denied!" 빨간불이 켜진다.
  4. [Change Package] 레버를 당겨 `ZBOOKING_PACK` 으로 상자 라벨을 교체한다.
  5. 쇠사슬 락이 찰딱 풀리고 ZHELLO 가 이송 트랙 기차 속으로 안착하는 락 해소 피드백을 감상한다.
- **상태 및 데이터**:
  - `패키지를 쌩 Z로 교체하지 않고 강제 DB SQL 로 TADIR 을 임의 수정하려 든 상태` -> 런타임 결과: `System inconsistency! TADIR checksum mismatch. Dictionary corrupted` 적색 사이렌 경보.
- **피드백**: $TMP 탈출은 Object Directory Entry 속성 창을 통해 정식으로 패키지를 교체 배속해야 이송 락이 풀리는 정합성을 지닌다.

### 실수/주의

- **공식 ZPACKAGE 로 이전하면서, 여전히 로컬 테스트용 가짜 데이터나 Z임시 파일을 실수로 묶어 이송 요청서에 삽입**:
  - 이 실수 시 운영계에 테스트용 쓰레기 Z오브젝트가 다 넘어가 정산 전산망을 엉망으로 흩뜨린다.
  - **이송 릴리즈 전에는 반드시 내 패키지 내벽에 맑은 실무용 오브젝트만 남겨두고 임시는 철저히 $TMP 에 격리 소거해야 함을 수호해야 한다.**

### 정리

- **`$TMP`** 로컬 패키지는 이송 요청서(TR)를 탈 수 없는 영구 격리 락 창고다.
- 타 시스템 이송 배포를 위해서는 반드시 **`Z`** 로 시작하는 정식 **`개발 패키지`** 를 개설해 담는다.
- SE38 의 **`Object Directory Entry`** 에 들어가 패키지명을 이전 교체해야 **`TADIR`** 이송 락이 해제된다.

## CH01-L07 - 이송요청(TR)의 첫걸음

### 왜 필요한가

TADIR 이송 락을 풀고 정식 패키지 이송 기차에 소스를 태웠다.
이제 배송 열차의 발차표이자 티켓인 '이송 요청서(TR)' 의 릴리즈(Release) 제어 구조가 마지막 장벽이다.
- " 패키지를 바꾸자 이송 요청서 `EK0K900182` 가 생성되어 프로그램 ZHELLO 가 그 안에 담겼다.
Basis 운영 팀에게 '운영계로 배포해 달라' 고 요청했더니, '요청서가 아직 락이 걸려 릴리즈(Release)되지 않아 기차를 출발시킬 수 없다' 며 거절당했다."
아바 이송 요청서(TR)는 1덩어리의 쌩 상자가 아니다. 
개발자가 직접 변경을 빚는 '자식 Task' 와, 배포 최종 티켓인 '부모 Request' 의 2층 구조로 상호 락킹 제어된다.
이 자식 Task 와 부모 Request 의 릴리즈 선후 구조를 이해하지 못해 부모부터 릴리즈하려 버둥거리다 에러 락에 막혀 배포 기한을 놓친다.

**이송 요청서(TR)의 2계층 구조(부모 Request - 자식 Task)를 꿰차고, 자식 Task 의 개별 락을 먼저 Release 시켜 부모에게 소유권을 양도한 뒤, 최종 부모 Request 를 릴리즈해 배송 트랙에 얹는 기술**이 필요하다. 그것이 **이송요청(TR)의 첫걸음**이다.

### 무엇인가

#### 1. TR (Transport Request - 이송 요청서)
- 개발기에서 수정한 소스 코드와 테이블 조각들을 안전하게 담아 포장한 뒤, 품질(QAS)과 운영(PRD) 서버로 날려 보내는 배송 컨테이너 박스 티켓이다. (트랜잭션 `SE09` / `SE10` 에서 총괄 제어한다.)

#### 🧭 [ CTS 이송 요청서(TR) 부모-자식 2계층 락킹 구조 명세 ]
- *요청서는 형상 충돌을 방지하기 위해, 아래와 같은 삼엄한 자식->부모 락 해제 프로토콜을 따른다.*

```text
[부모] Transport Request (부모 요청서 - 예: EK0K900182) :
   최종 기차 선로에 얹어 발송할 진짜 컨테이너 봉투다. 
   자식 Task 들이 모두 락이 풀려 안으로 귀속되기 전에는, 부모 혼자서는 절대로 릴리즈(발차)할 수 없다.
   │
   └── [자식] Task (자식 작업서 - 예: EK0K900183) :
          실제 개발자 개개인(정훈영)에게 소유권이 배정되어 Z오브젝트를 직접 담아 잠그는 작업 바구니디.
          이 자식 바구니가 열려 있는 동안에는 ZHELLO 프로그램은 수정 락(Lock)이 걸려 다른 사람이 고칠 수 없다.
```

#### 2. Release (릴리즈 - 이송 기적 소리)
- 요청서 상자를 완전히 밀봉 락을 채워, 개발기 디스크 밖으로 추출(Export)하여 배송 대기 큐에 얹어주는 동작이다.

#### ⚠️ [ 자식 Task 선행 릴리즈 의무 명세 ]
- *이송 승인 시 입문자 백이면 백 다 헤매는 선후 정합성 제약이다.*
- **부모 Request 를 먼저 마우스 클릭하고 Release (트럭 그림 아이콘)를 누른다.**
- **기각 에러**: **"Task 가 아직 열려 있어 Request 를 릴리즈할 수 없다" 며 시스템이 릴리즈를 강제 기각한다.**
- **이유**: **자식 작업자(Task)가 도구를 쥐고 소스를 수정 중일 가능성(Task Open)이 남아 있는 상태에서는, 부모 컨테이너를 먼저 밀봉해 출발시키는 하극상 배송이 원천 불가능하기 때문이다. 무조건 자식 Task 를 먼저 마우스 우클릭해 Release 풀기를 완료(밀봉)한 뒤, 부모 Request 를 마지막으로 최종 Release 해야만 기차가 출발 통과된다.**

### 어떻게 확인하는가

SE09 에서 자식 Task 를 먼저 릴리즈하고 부모 Request 를 릴리즈해 발송하는 프로세스를 검증한다.

```text
[1단계] SE09 이송 관리판 진입 :
   - 요청서 트리 구조 확인:
     └- 부모 Request : EK0K900182 (정훈영) [수정 중 락 상태]
          └- 자식 Task : EK0K900183 (정훈영) [ZHELLO 프로그램 담김]
          
[2단계] 자식 Task 선행 릴리즈 격발 :
   - 자식 Task (EK0K900183) 선택 -> [Release (Ctrl+F9)] 클릭!
   - 상태: 자식 Task 아이콘 옆에 초록색 완료 체크등 점등! (바구니 밀봉 성공!)
   
[3단계] 부모 Request 최종 릴리즈 격발 :
   - 이제 부모 Request (EK0K900182) 선택 -> [Release] 클릭!
   - 시스템이 락 테이블을 해제하고 OS 디스크에 물리 이송 파일(Data/Co-file) 배출 성공!
   - 기차 선로에 적재 완료! (CTS 이송 발송 최종 완료 및 락 프리 성료!)
```

#### 체험/시뮬레이터 설계 (CTS 이송 트레인 릴리즈)
- **프로세스 플로우**:
  1. 학습자가 [부모 Request 상자] 와 그 밑에 매달린 [자식 Task 주머니] 를 본다.
  2. [부모 상자] 에 대고 [Release] 단추를 누른다. 주머니가 열려 있다며 "Task EK0K900183 is open! Blocked!" 빨간 불이 번쩍인다.
  3. [자식 Task 주머니] 에 대고 [Release] 를 때린다. 주머니 입구가 찰딱 닫히며 초록 불로 변한다.
  4. 다시 [부모 상자] 에 대고 [Release] 를 때린다. 상자가 촥 밀봉되어 기차 화물칸에 실려 덜컹덜컹 출발하는 배포 피드백을 감상한다.
- **상태 및 데이터**:
  - `Inactive 상태의 프로그램이 담긴 채로 릴리즈를 강제 격발한 상태` -> 런타임 결과: `Release failed. Program ZHELLO is inactive in DB` 하이라이트.
- **피드백**: 이송 릴리즈는 무조건 자식 Task 선행 밀봉, 그리고 프로그램 Active 상태 합치가 맞물려야 배송망이 격발됨을 배운다.

### 실수/주의

- **자식 Task 를 릴리즈해 락을 다 풀어두고서, 부모 Request 릴리즈를 깜빡 잊고 퇴근**:
  - 이 실수 시 자식만 닫혔을 뿐 부모 기차가 출발하지 않아, 새벽 Basis 배포 배치 작업 시 내 프로그램만 쏙 빠져 운영계 배포가 누락되는 불상사를 초래한다.
  - **자식을 릴리즈했다면 잊지 말고 즉시 부모 Request 도 세트로 끝까지 릴리즈를 완수해야 함을 수호해야 한다.**

### 정리

- 이송요청서(TR)는 개발물이 담기는 배송 컨테이너 기차 티켓이다.
- TR 은 **`부모 Request`** 와 실무 변경을 담는 **`자식 Task`** 의 2층 락킹 구조다.
- 반드시 **`자식 Task`** 를 1순위로 릴리즈 밀봉한 뒤, 최종 **`부모 Request`** 를 릴리즈해야 발송된다.
- *이 CTS 이송 원리는 SAP 의 근간이며, 최신 gCTS 나 Cloud 이송에서도 이 객체 락과 릴리즈 정합성 원리가 그대로 계승 적용된다.*
