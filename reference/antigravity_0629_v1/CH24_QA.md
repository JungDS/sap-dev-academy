## 3. 공식 문서 수동 근거 명세

`abapinsert_dbtab.htm` 을 대조하여 DB Write DML 의 물리 작동 제약을 철저히 규명했다.

- **대량 INSERT 와 ACCEPTING DUPLICATE KEYS 규칙**:
  단건 `INSERT` 와 달리, 내부 테이블을 일괄 전송하는 `INSERT FROM TABLE @itab` 은 중복 키가 단 1건이라도 충돌하면 `sy-subrc = 4` 실패 분기로 떨어지는 대신, 즉시 **`DBSQL_DUPLICATE_KEY_ERROR` 런타임 덤프를 터트려 시스템을 강제 중단**시킨다. 이 런타임 덤프 폭사를 방지하고 실패 행을 스킵하며 정상적으로 `sy-subrc = 4` 로 수습하려면 구문 끝에 **`ACCEPTING DUPLICATE KEYS`** 옵션을 의무 명세해야 한다는 절대 문법 제약 수집 및 수록 완료.

## 4. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. INSERT FROM TABLE 을 사용하여 100건을 한 번에 넣다가 중복 키 에러로 런타임 덤프가 터졌습니다. 왜 단건 INSERT 처럼 sy-subrc = 4 로 안 떨어지고 프로그램이 폭사하며, 구체적인 방어 문법은 무엇인가요?
**A**: **단건과 달리 대량 일괄 적재(Bulk Insert)는 DB 엔진 레벨에서 고속 처리를 감행하므로 정합성 충돌 시 커널이 즉각 DBSQL_DUPLICATE_KEY_ERROR 치명적 예외를 터트려 셧다운시키며, ACCEPTING DUPLICATE KEYS 옵션을 붙여 덤프를 회피해야 합니다.**
- **덤프의 물리 원인**:
  단건 인서트는 아바 런타임이 "이 한 건이 실패했구나" 라고 감지해 `sy-subrc` 변수에 실패 코드를 담아 반환해 주기 쉽습니다.
  하지만 `FROM TABLE` 대량 인서트는 아바 엔진이 수천 건의 패킷을 DB 세션에 통째로 쏟아부어 고속 물리 적재를 단행합니다. 이 과정에서 중복 키 충돌이 나면 DB 커널 수준에서 정합성 붕괴 비상벨을 울리고 세션을 강제 파괴해 덤프를 내뿜습니다.
- **구체적 방어 문법**:
  반드시 쿼리 꼬리에 **`ACCEPTING DUPLICATE KEYS`** 를 박아야 합니다.
  ```abap
  " [ 중복 키 덤프 에러를 우회하는 정석 대량 인서트 ]
  INSERT zbooking FROM TABLE @lt_booking ACCEPTING DUPLICATE KEYS.
  
  IF sy-subrc = 4.
    " 중복된 일부 찌꺼기 행은 무사히 스킵되었고, 성공한 수량만 sy-dbcnt 에 찍혀 올라옴!
    WRITE: / '일부 예약 중복 건은 스킵 완료. 성공 행수:', sy-dbcnt.
  ENDIF.
  ```

### Q2. 업데이트 태스크 모듈(IN UPDATE TASK 로 불려가는 함수) 본문 소스 코드 안에서 COMMIT WORK 이나 ROLLBACK WORK 을 적어 빌드하면 런타임에 왜 대형 덤프가 터지나요?
**A**: **이미 메인의 COMMIT WORK 에 의해 격발되어 비동기 세션을 돌리고 있는 갱신 포스팅(Posting) 구간 안방에서 또다시 커밋 경계를 흔드는 모순을 자행하면, 엔진이 무한 커밋 루프에 직면하여 COMMIT_IN_POSTING 덤프를 내뿜고 폭사하기 때문입니다.**
`CALL FUNCTION ... IN UPDATE TASK` 로 예약된 함수는 백엔드 SM13 대기 큐에 잠들어 있다가, 메인 프로그램이 `COMMIT WORK` 을 날려 락을 푸는 찰나에 비로소 호출되어 일어납니다.
즉, 업데이트 모듈은 이미 **'커밋 작업이 집행 중인 물리적 세션 내벽 내부'** 에서 돌고 있습니다.
그런데 이 함수 안방에 또 `COMMIT WORK` 이나 `ROLLBACK WORK` 문장을 집어넣어 실행시키면, 아바 엔진은 "커밋 처리 중에 또 새로운 커밋 요청이 들어왔으니, 꼬리를 물고 무한 루프를 돌아야 한다" 는 복합 모순을 인지하고, DB 전체 락 데드락 장애를 방어하기 위해 즉각 실행 세션을 강제 셧다운시켜 버립니다.
**철칙**: 업데이트 모듈 내부에는 순수한 DML 연산만 적어야 하며, **트랜잭션 확정/취소(COMMIT/ROLLBACK) 제어는 오직 이 함수를 지연 예약 호출한 메인 프로그램 최외곽 단독 영역에서만 엄숙하게 단 1번 기재**해야 합니다.

### Q3. SELECT ... ENDSELECT 데이터베이스 커서(Cursor) 루프 중간에 COMMIT WORK 을 실행하면 왜 후속 루프가 기동될 때 런타임 덤프가 터져 폭사하나요?
**A**: **커밋이나 롤백 지시가 작동하는 순간, 현재 사용 중이던 데이터베이스 연결 커서 세션 포트가 강제로 단선 폐쇄(Closed) 처리되므로, 루프가 다음 행을 읽기 위해 손을 뻗을 때 대상 커서가 소멸해 CURSOR_ALREADY_CLOSED 덤프를 터트리기 때문입니다.**
`SELECT ... ENDSELECT` 는 DB 로부터 100건의 데이터를 한 번에 가져오지 않고, DB 커서를 열어둔 채 루프를 돌 때마다 한 행씩 쪼개어 실시간으로 네트워크를 타며 가져옵니다.
이 루프 안에서 "한 건 저장 완료했으니 즉각 커밋해야지" 하고 `COMMIT WORK.` 을 때리는 순간, DB 는 현재의 논리 트랜잭션 단위가 종료되었다고 판단하고, 열려있던 모든 리소스 포트와 임시 커서 통로를 강제로 끊어 닫아 버립니다.
그 직후 아바 루프가 `ENDSELECT` 를 만나 "다음 행 하나 더 가져와라" 고 DB 포트를 노크하면, 포트가 이미 단선되어 증발해 있으므로 시스템은 즉각 폭사합니다.
**해결책**: 따라서 커서 루프 안에서는 절대로 커밋을 난사할 수 없으며, **루프가 완전히 다 돌아서 `ENDSELECT` 문장 밖으로 빠져나온 외곽 안전지대에서 단 1번 COMMIT WORK 을 실행**해야 아키텍처적으로 무결합니다.

### Q4. SAP 표준 테이블을 변경하는 BAPI 함수를 쓸 때는, 왜 제가 직접 COMMIT WORK 을 쓰지 않고 BAPI_TRANSACTION_COMMIT 이라는 표준 커밋 함수를 불러야 하나요?
**A**: **표준 BAPI 는 내부적으로 독자적인 버퍼 정합성 검증과 지연 업데이트 큐 등록을 복잡하게 조립해 두었으므로, 단순 쌩 COMMIT WORK 을 때리면 표준 버퍼가 동기화되지 않고 어긋나 깨진 전표가 포스팅되기 때문이며, BAPI 래퍼 커밋으로 수습해야 합니다.**
SAP 사가 개발해 개방한 표준 `BAPI_CREATE_SALES_ORDER` 등은 DML 을 직접 치지 않고, 내부 독자 버퍼에 임시 적재만 마친 채 실행을 끝냅니다.
이때 그냥 일반 아바 `COMMIT WORK.` 을 때리면, BAPI 가 요구하는 후속 버퍼 정합성 유효 검문소 가드를 패스하지 않고 그냥 억지로 저장 대기 큐만 털어버려 반쪽짜리 오류 전표가 저장됩니다.
반면 표준 **`BAPI_TRANSACTION_COMMIT`** 함수는 내벽 내부에서 **"BAPI 가 셋업한 모든 전표 버퍼의 정합성을 2차 검증 완수한 뒤, 안전함이 보증되면 COMMIT WORK AND WAIT 을 가동하라"** 고 래핑(Wrapping) 코딩이 완비되어 있습니다. 
따라서 표준 BAPI 로 변경을 때렸다면, 내 쌩 커밋을 쓰지 말고 무조건 동반 커밋 함수인 `BAPI_TRANSACTION_COMMIT` 을 세트로 호출해 주어야 안전합니다.

### Q5. 대량 변경 패키지 커밋(Package Commit) 루프를 다 돌고 빠져나온 맨 아랫줄 외곽에 최종 잔량 커밋(Final Commit)을 깜빡 빼먹고 생략하면 데이터가 구체적으로 어떻게 유실되나요?
**A**: **5,000건 주기로 잘라 커밋하고 남은 최종 나머지 잔량 데이터(예: 4,900건)는 루프 내부의 커밋 분기 조건(cnt >= 5000)을 충족하지 못해 대기 큐에만 방치되어 있다가, 프로그램 세션이 종료되는 순간 강제로 증발 롤백 소멸해 버립니다.**
데이터 404,900건을 5,000건씩 패키지 커밋 튜닝했습니다.
루프 내부에서는 5,000건, 10,000건, ... 400,000건이 되는 순간마다 정상적으로 `COMMIT WORK` 이 80번 기동되어 40만 건은 DB 디스크에 안전하게 박제 확정되었습니다.
그리고 남은 마지막 4,900건이 루프를 돌 때는, 카운터 `lv_cnt` 가 4900 에 머물러 `IF lv_cnt >= 5000` 분기를 타지 못하고 루프를 탈출해 버립니다.
이때 루프 바깥 하단 외곽에 잔량 최종 수거 커밋 코드가 누락되어 있으면, 이 4,900건은 확정되지 않은 임시 대기 상태로 허공에 떠 있습니다. 
프로그램이 실행 완료되어 화면이 꺼지는 그 찰나, SAP 런타임 커널은 **"미처 커밋되지 않고 방치된 찌꺼기 임시 데이터 4,900건을 안전을 위해 자동 ROLLBACK 소멸 처리"** 하여 지워버립니다. 
결과적으로 전체 404,900건 중 4,900건의 고객 예약 정보가 매일 밤 대량 유실되는 대형 참사가 발생합니다.
**방어선**: 루프 밖 하단 아랫줄에 반드시 대기 잔량이 남아있음을 체크해 수거하는 **`IF lv_pending = abap_true. COMMIT WORK. ENDIF.`** 최종 잔량 방어선을 물려두어야 무결성이 사수됩니다.
