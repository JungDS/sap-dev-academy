## 4. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. LEFT OUTER JOIN을 썼는데도 왜 예매 내역이 없는 신규 공연이 리포트에서 통째로 실종되나요?
**A**: **오른쪽 테이블의 조건 필터를 ON 절이 아닌 WHERE 절에 잘못 기입하여, LEFT JOIN의 보존 뼈대를 스스로 파괴(INNER JOIN으로 수축)했기 때문**입니다.
이것은 실무 현업에서 가장 많이 내는 치명적인 버그입니다.
- **`ON` 절 작동**: 조인이 끈으로 짝을 맺을 때 `C003` 공연은 예매가 없어 오른쪽 예매 번호와 상태 필드가 전부 **`NULL` (빈칸)** 상태로 우선 통과합니다.
- **`WHERE` 절 작동 (오염)**: 조인이 끝난 후 `WHERE b~status <> 'C'` 조건절 검문이 발동합니다. `C003` 공연의 상태는 짝이 없어 'NULL(빈칸)'인데, 검문 엔진은 **"빈칸은 'C'가 아니긴 하지만, NULL 상태의 깡통은 아예 비교 대상이 아니므로 버린다"** 거나 **"특정 상태값 비교 조건을 걸었으니 매칭이 있는 행만 남겨라"** 고 판정해 탈락시킵니다.
따라서 오른쪽 테이블의 조건은 절대로 WHERE에 쓰지 말고, **`ON c~concert_id = b~concert_id AND b~status <> 'C'`** 처럼 ON 절 우측 결합부에 AND로 도킹시켜야 짝이 없는 NULL 행이 검문을 우회하여 안전하게 보존 생존합니다.

### Q2. FOR ALL ENTRIES를 쓸 때 FAE 대상 테이블이 비어있는지 검증하는 IF IS NOT INITIAL 가드를 단 한 줄이라도 누락하면 훗날 실무 운영 서버에 어떤 재앙이 터지나요?
**A**: **검색 대상 데이터베이스 테이블 전체를 풀스캔(Full Table Scan)하여, 시스템 메모리가 고갈되거나 배치 백그라운드 작업이 타임아웃 덤프를 내며 서버가 정지되는 장애를 유발합니다.**
이것은 데이터베이스 엔진의 해석 약속 때문입니다. 
`FOR ALL ENTRIES IN gt_dept` 구문이 있을 때, 만약 `gt_dept` 테이블이 0행인 깡통 상태라면 DB 엔진은 **"VIP 부서 목록 조건이 비었으니, 조건이 아예 없는 것으로 간주하고 전체 사원 데이터를 다 긁어와라"** 고 해석하여 `WHERE` 절 전체를 무시하고 쿼리를 전체 조회로 풀어서 실행합니다.
운영 서버의 사원 테이블이나 전표 테이블 데이터가 1억 건이 넘는 상황에서 이 쿼리가 돌면, DB 서버 메모리를 다 잡아먹고 동시 접속한 다른 현업 직원의 화면까지 멈추게 만드는 대형 장애를 겪게 됩니다. FAE 직전 **`IF gt_dept IS NOT INITIAL.`** 가드는 선택이 아니라 생명선입니다.

### Q3. FAE(FOR ALL ENTRIES)로 데이터를 퍼 올릴 때, 실제 데이터베이스에 든 행 개수보다 결과 내부 테이블에 담긴 행 수가 왜 줄어들어 실종되나요?
**A**: **FAE 가동 시 시스템이 결과물에서 완전히 똑같이 생긴 중복 행을 자동 압축해 지워버리는 DISTINCT(중복 제거) 거동을 기본 사양으로 집행하기 때문입니다.**
예를 들어 `ZBOOKING` 테이블에 동일한 고객 `정훈영`이 예약한 내역 5줄이 들어있다고 해보자. 이 데이터를 FAE로 읽을 때, SELECT 목록에 예매 번호(`booking_id`)나 회차 번호(`perf_no`) 같은 유니크한 구분자 키 필드를 빼먹고, 오직 고객 이름(`name`)과 공연 ID(`concert_id`)만 적어서 가져오면 어떻게 될까요?
가져온 5줄의 데이터가 `[정훈영 | C001]` 형태로 전부 똑같이 생겼기 때문에, FAE 엔진은 이를 중복으로 인지하고 **4줄을 쓰레기통에 버리고 단 1줄만 남겨서 반환**합니다. 이 무서운 증발 현상을 막으려면, 중복 판정을 깨부술 수 있는 **유니크한 기본 키 필드(예: `booking_id`)를 SELECT 목록에 반드시 구체적으로 매핑해 포함**시켜야 행 유실이 방지됩니다.

### Q4. GROUP BY를 쓸 때 SELECT 목록에 있는 일반 컬럼(평필드) 중 GROUP BY 절에 없는 필드를 집계 함수(SUM/COUNT 등)로 감싸주지 않으면 왜 컴파일조차 막히나요?
**A**: **"한 주머니에 여러 다른 값이 어지럽게 엉켜 있을 때, 화면 1줄에 무슨 값을 대표로 인쇄해야 할지" 수학적으로 판정할 수 없는 모호성 오류가 발생하기 때문입니다.**
예를 들어, 공연 ID(`concert_id`)별로 묶어서 요약 리포트를 그리려고 `GROUP BY concert_id` 를 걸었습니다. 그런데 SELECT 목록에 예약석 합계(`SUM(seats)`) 외에 일반 고객명(`name`) 필드를 쌩으로 같이 적었다고 해보자.
`C001` 공연 그룹 주머니 안을 들여다보니, 예약한 고객이 `정훈영`, `김민수`, `이영희` 3명이 들어있습니다. 
이 3명이 합쳐져 단 1줄의 `C001` 요약행으로 화면에 인쇄되어야 하는데, 고객명 칸에는 3명의 이름 중 누구 이름을 올려야 할까요? DB 엔진은 이 모호함을 처리할 방법이 없습니다. 
따라서 그룹 키가 아닌 일반 필드는 반드시 **`COUNT(name)`** (예약한 사람 수 요약)나 **`MAX(name)`** 처럼 집계 함수로 감싸주어 다중 값을 단 1개의 요약 값으로 변환시켜 주거나, 아니면 **GROUP BY 절 뒤에 `name`을 같이 묶어** 고객 이름별로도 쪼개어 그룹화되도록 명시해 주어야만 문법 에러가 나지 않습니다.

### Q5. 여러 테이블 데이터를 하나로 짝지어 가져오는 3대 조회 전략(JOIN vs FAE vs ABAP Loop)을 현업 실무에서 결정하는 우선순위 의사결정 골든 룰을 명쾌하게 요약해 주세요.
**A**: **"조합할 데이터의 실물 위치와 가공의 복잡성"에 따라 3단계 기어로 짚고 결정합니다.**
- **1순위 (DB JOIN)**: 조인할 장부들이 **둘 다 데이터베이스 디스크 창고 안에 보관되어 있을 때**는 무조건 조인(`JOIN`)을 건다. DB가 짝을 맞춰 앱 서버로 결과물만 배달해주므로 네트워크 이동이 최소화되어 성능이 가장 우수합니다.
- **2순위 (FOR ALL ENTRIES)**: 짝지을 기준 리스트 조건이 DB가 아닌 **내 프로그램 메모리에 내부 테이블(`gt_itab`)로 이미 실려 올라와 존재할 때**는 조인을 못 하므로 FAE를 사용해 DB를 노크합니다.
- **3순위 (ABAP 직접 루프)**: 대조할 데이터가 아주 적은 소량이거나, JOIN/FAE 쿼리식으로는 구현하기 힘들 정도로 **아주 복잡한 아밥 수식 가공이 필요할 때**만 어쩔 수 없이 각자 읽어와 프로그램 내에서 `LOOP AT` 과 `READ TABLE` 조합으로 직접 짝을 버무립니다.
- **절대 금지**: 루프 문을 돌면서 그 바퀴 수만큼 SELECT를 때리는 중첩 조회(`LOOP 안의 SELECT`) 코드는 성능 파멸의 안티패턴이므로 무조건 JOIN이나 FAE로 통합 개량해야 합니다.
