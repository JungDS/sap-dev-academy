# CH12_REWRITE - SELECT-OPTIONS와 Range Table v1

> 목적: `content/abap/CH12`의 7개 레슨을 "IT 비전공자 입문자가 왜 필요한지부터 실습 확인까지 직관적으로 이해하고 따라갈 수 있는 완성 강의자료" 수준으로 재작성한 기준 원고다. 이 문서는 아직 `content/abap` 원본을 덮어쓴 결과가 아니라, 재작업 판정 이후 실제 반영 전 검수 가능한 v1 기준안이다.

## CH12 전체 설계

CH12의 한 문장 목표는 "단일 값 입력(PARAMETERS)의 한계를 넘어, 조건들을 행으로 쌓아두는 범위 테이블(Range Table)의 4대 물리 구조(SIGN, OPTION, LOW, HIGH)를 파악하고, 화면과 Range Table을 동시에 구축하는 SELECT-OPTIONS 기본 문법 및 WHERE IN classic 결합을 통해 다중 조건 예매 필터 리포트를 완성한다"이다.

IT 비전공자 입문자는 빈 Range Table이 조건 무제한(참, True)으로 풀려 대량 DB 데이터를 긁어와 성능을 마비시키는 함정을 모른 채 배포하고, CP 옵션과 SQL LIKE의 와일드카드 기호(`*`/`+` vs `%`/`_`)를 혼동해 검색 오류를 내며, `CLEAR` 누락으로 이전 범위 잔상이 흘러 들어가는 찌꺼기 버그를 유발한다.
따라서 본 챕터는 다음과 같은 불편 해결의 단계적 설계로 전개한다.

1. **단일 값 매칭의 한계**: PARAMETERS는 값 1개만 받아 "여러 개 동시 선택", "A부터 B까지 범위 지정", "특정 데이터만 쏙 빼고 제외" 라는 다중 조건 요건을 수용하지 못함 $\rightarrow$ 조건을 표 양식으로 수납하는 **Range Table** 도입.
2. **조건 수납 표의 4대 규격**: 포함/제외 방향을 결정하는 **SIGN**, 비교 방식을 잡는 **OPTION**, 값의 시작과 끝인 **LOW**와 **HIGH**의 물리 필드 분석.
3. **1타 2피의 문법 (SELECT-OPTIONS)**: 화면에는 다중 입력 박스를 열고 프로그램 내에는 동명의 Range Table을 개설하는 **SELECT-OPTIONS FOR** 결합과 **TABLES** 사전 연동.
4. **Range 쿼리 꽂기 (WHERE ... IN)**: Range Table에 실린 복잡한 다중 행 조건들을 DB 조회 쿼리에 다이렉트로 전달하는 **`WHERE 필드 IN 테이블`** classic 연동.
5. **빈 조건의 전체 조회 함정**: *데이터베이스 대장 정합성 철칙.* Range Table이 텅 비어있으면 "0건 조회"가 아니라 **"조건 무제한 전체 통과"** 가 발동되어 DB 풀스캔 성능 부하가 초래됨을 경고.
6. **다중 팝업의 2색 판정 (Include/Exclude)**: 다중선택 팝업창에서 생성되는 **녹색 포함(I)** 행(OR로 범위를 넓힘)과 **빨간 제외(E)** 행(최종 컷트)의 집계 연산 메커니즘 규명.
7. **문자 매치 옵션 (CP)**: 문자열 유사 비교인 `CP` 옵션의 와일드카드 기호인 **`*`** (무한 자릿수)와 **`+`** (정확히 1자)의 규칙 장착 및 `LIKE` 기호와의 격리.
8. **화면 없는 수동 조립**: 코드로 직접 Range Table을 개설해 조작하는 **`TYPE RANGE OF`** 선언과 `CLEAR`, `APPEND` 파이프라인.
9. **예매 필터 캡스톤**: 공연ID와 예약 상태 조건을 결합해 필터링된 예매 목록을 SALV로 화면에 띄우는 종합 실습.

### 범위와 금지선

- **R6 classic-first 철저 수호**: 
  - `TABLES` 선언 및 `SELECT-OPTIONS s_conc FOR ...` 명명 규칙(s_ 접두어 및 8자 이내 제약)을 엄수한다.
  - WHERE 조건절에 Range Table을 대입할 때 골뱅이 `@`를 절대 붙이지 않는 classic **`WHERE concert_id IN s_conc`** 구문만 사용한다. (modern `@` 대입 및 inline Range 선언은 **CH19**에서 전면 다룬다.)
  - Range Table의 구식 obsolete 문법인 `RANGES r_stat FOR ...` 은 classic 레거시 소스 분석용으로만 언급하고, 신규 코드에는 **`TYPE RANGE OF`** 표준 참조만 사용하도록 통제한다.
- **변수 스코프 접두어**: 모든 스크립트는 전역 범위로 간주하며, 선택 화면 Range Table은 **`s_`** (Selection Option), 코드로 직접 만드는 Range Table은 **`lr_`** (Local Range), 일반 변수는 `gv_`, 구조체는 `gs_`, 내부 테이블은 `gt_` 접두어를 엄수한다.
- **이름 풀 준수**: 예제 이름은 지정된 이름 풀에서만 가져오며, 주인공은 항상 **정훈영**으로 통일한다.

## CH12-L01 - Range Table 구조

### 왜 필요한가

앞 챕터에서 우리는 `ZBOOKING` 예매 목록 전체를 엑셀 표 같은 그리드화면인 SALV로 띄우는 리포트를 마스터했다. 

하지만 실제 회사 업무에서 직원이 리포트를 볼 때 "매번 전체 데이터를 다 가져와서 눈으로 찾는" 비효율을 가만둘 리 없다. 
당장 정훈영 예매 담당자로부터 "공연 `C001` 번호 건만 골라서 보고 싶어요", "상태가 취소(`C`)된 예매 건은 싹 빼고 예약(`N`) 상태만 나오게 해주세요", "공연 `C001`부터 `C003` 범위만 뽑아주세요"와 같은 **정교한 범위 및 다중 검색 필터 요구사항**이 빗발친다. 

우리가 기존에 배운 `PARAMETERS` (단일 입력창) 변수는 오직 단 하나의 값만 담을 수 있는 좁은 그릇이라 이 복잡한 범위 지정과 제외 처리를 한꺼번에 수행할 도리가 없다. 
비즈니스에서 발생하는 다중 검색 필터 조건들을 **메모리 상에 표처럼 한 줄씩 조건 레코드로 쌓아두고 한꺼번에 DB에 전달하는 범위용 특수 내부 테이블**이 필요하다. 그것이 **[[Range Table]]** (레인지 테이블)이다.

### 무엇인가

#### 1. Range Table (범위 테이블)의 정의
**[[Range Table]]** 은 범위와 다중 선택 조건을 담아두기 위해 SAP가 시스템 사전에 규격화해둔 **네 개의 컬럼(SIGN, OPTION, LOW, HIGH)을 가진 특수한 내부 테이블**이다.
- **조건을 표로 적는 규칙**: 복잡한 조건문 여러 개를 이 표에 한 행씩 조건 레코드(Row)로 기입하여 보관한다.

#### 2. Range Table을 지배하는 4대 컬럼
1. **[[SIGN]] (포함/제외 방향 - 1자리 CHAR)**:
   - 이 행의 조건을 최종 결과에 집어넣을 것인가 빼버릴 것인가 방향을 결정한다.
   - **`I`** (Include): 포함한다 (녹색).
   - **`E`** (Exclude): 제외한다 (빨강).
2. **[[OPTION]] (비교 연산 방식 - 2자리 CHAR)**:
   - 어떤 수학적 연산으로 데이터를 대조할 것인지 결정한다.
   - **`EQ`** (Equal): `LOW` 값과 정확히 같은 것.
   - **`BT`** (Between): `LOW` 값부터 `HIGH` 값 사이 영역에 있는 것.
   - **`CP`** (Contains Pattern): 와일드카드 패턴과 매칭되는 것.
   - **`NE`** (Not Equal): `LOW` 값과 다른 것.
3. **LOW (조건의 시작 값)**:
   - 비교할 기준값 혹은 범위의 시작점(Start Value)을 담는다. (가장 핵심 필드).
4. **HIGH (조건의 끝 값)**:
   - `OPTION`이 사이값 범위인 **`BT`** (Between) 또는 **`NB`** (Not Between) 일 때만 범위의 종착역 끝 값을 기입한다. (`EQ` 등 단건 비교 시에는 항상 빈칸으로 둔다).

#### 3. 사람 말 $\leftrightarrow$ Range Table 행 변환 규칙 대조

| 사람의 요구사항 (말) | SIGN | OPTION | LOW | HIGH | 해석 (ABAP 내부 판단) |
| --- | --- | --- | --- | --- | --- |
| 1. 공연 `C001`만 포함해줘 | **`I`** | **`EQ`** | `C001` | *(비움)* | `C001`과 정확히 일치하는 행을 포함한다. |
| 2. 공연 `C001`부터 `C003` 범위 포함 | **`I`** | **`BT`** | `C001` | `C003` | `C001` ~ `C003` 사이의 행을 포함한다. |
| 3. 취소 상태 `C`는 제외해줘 | **`E`** | **`EQ`** | `C` | *(비움)* | 상태가 `C`인 데이터는 리포트에서 뺀다. |
| 4. 가수 이름이 '정'으로 시작 | **`I`** | **`CP`** | `정*` | *(비움)* | '정'으로 시작하는 모든 패턴을 포함한다. |

### 어떻게 확인하는가

에디터와 교안 다이어그램을 대조하여 조건 데이터가 4대 컬럼에 수납되는 동작 규칙을 검증한다.

```abap
" Range Table 구조의 한 행은 항상 SIGN, OPTION, LOW, HIGH의 형태를 가진다.
" ZBOOKING-CONCERT_ID 필드의 타입을 본뜬 조건 한 행용 구조체 매핑 예시
DATA: BEGIN OF ls_range_line,
        sign   TYPE ddoption-sign,   " I 또는 E
        option TYPE ddoption-option, " EQ, BT, CP 등
        low    TYPE zbooking-concert_id,
        high   TYPE zbooking-concert_id,
      END OF ls_range_line.
```

#### 체험/시뮬레이터 설계 (Range Table 행 빌더)
- **프로세스 플로우**:
  1. 학습자가 [조건 카드] 중에서 "공연 C001부터 C003까지 포함" 카드를 탭한다.
  2. 중앙의 4칸짜리 빈 테이블에 `SIGN = I`, `OPTION = BT`, `LOW = C001`, `HIGH = C003` 값이 각각 포커싱되어 쏙쏙 날아가 채워지는 수납 과정을 목격한다.
- **상태 및 데이터**:
  - `SIGN`에 실수로 `BT` 입력 시도 -> "SIGN field length is 1, BT is not allowed" 칸 규격 위반 컴파일 오류 피드백.
  - `OPTION`에 `EQ`를 세팅하고 `HIGH`에 `C003`을 채운 상태 -> "HIGH value is ignored for EQ option" 흐름 설명 노출.
- **피드백**: Range Table은 4대 컬럼 규격이 칼같이 정형화된 조건 보관함임을 기호 다이어그램으로 선사한다.

### 실수/주의

- **SIGN과 OPTION의 값 혼동**: 입문자들이 처음 코드로 Range Table을 조작할 때 자릿수 규격을 혼동해서 `SIGN`에 `BT`를 넣거나 `OPTION`에 `I`를 억지로 대입해 대입 오류 구문 덤프를 겪는 실수를 범하므로, 포함/제외인 `I/E`는 오직 `SIGN`에만 들어가야 함을 유념해야 한다.

### 정리

- **`Range Table`** 은 다중 검색 필터 조건을 행으로 차곡차곡 적재하는 특수 내부 테이블이다.
- 한 행은 방향인 **`SIGN`**, 방식인 **`OPTION`**, 값인 **`LOW`** 와 **`HIGH`** 4요소로 지배된다.
- 단건 조건은 `LOW`만 사용하고, Between 범위 조건(`BT`)에서만 `LOW`와 `HIGH`를 묶어 작동시킨다.

## CH12-L02 - SELECT-OPTIONS 기본 문법

### 왜 필요한가

Range Table의 4대 필드(`SIGN, OPTION, LOW, HIGH`)를 마스터했다. 
하지만 실제 회사에서 시스템을 쓰는 일반 현업 직원에게 "Range Table에 조건 행을 추가할 테니 SIGN은 I를 쓰시고 OPTION은 BT로 적으세요" 라고 개발자용 쿼리 작성을 요구할 수는 없다. 

사용자는 단지 화면 모니터의 From/To 박스에 값을 편안하게 적거나, 오른쪽에 있는 작은 다중 선택(화살표) 버튼을 마우스로 클릭해서 입력하길 바랄 뿐이다. 
화면에 **사용자가 친숙하게 다룰 수 있는 다중 입력 도구를 자동으로 렌더링**하고, 프로그램 내부에는 이와 **동기화되는 Range Table 변수 그릇을 한 번에 동시에 창조해 주는 1타 2피의 영리한 결합 문법**이 요구된다. 그것이 **[[SELECT-OPTIONS]]** (셀렉션 옵션즈)이다.

### 무엇인가

#### 1. SELECT-OPTIONS의 1타 2피 동작 원리
- **1타 2피**: 프로그램 소스 코드에 `SELECT-OPTIONS` 한 줄을 선언해 올리면, SAP 컴파일러는 다음 두 가지를 물리적으로 동시에 생성해 매핑한다.
  - **화면(Selection Screen)**: From 입력 상자, To 입력 상자, 그리고 다중선택(화살표) 버튼이라는 복합 검색 필드 자동 렌더링.
  - **메모리(RAM)**: 화면에서 입력된 값을 그대로 수납하는, 동일한 이름의 **Range Table 내부 변수** 자동 개설.

#### 2. SELECT-OPTIONS FOR 선언 문법
- **기본 구문**:
  ```abap
  TABLES 테이블명.
  SELECT-OPTIONS s_이름 FOR 테이블명-필드명.
  ```
- **`TABLES` 의 필수 선언 지침**:
  - `FOR` 뒤에 매핑할 데이터베이스 테이블의 실제 컬럼 주소(`zbooking-concert_id`)를 적어 넣으려면, 아밥 컴파일러가 해당 테이블의 구조 규격을 인지할 수 있도록 그 윗줄에 반드시 **`TABLES 테이블명.`** 이라는 참조 명세서를 선언해 주어야 한다. (DB 데이터 조회가 아니라 순수 필드 뼈대 상속용 선언이다).
- **변수 명명 규칙**:
  - 선택 화면 범위 조건 변수임을 식별하기 위해 접두어 **`s_`** (Selection Option)를 붙인다.
  - *가드레일*: SELECT-OPTIONS로 만드는 조건 변수명은 SAP 스크린 시스템의 내부 규격 한계로 인해 **반드시 최대 8자 이내**로 지어야 한다. 8자를 초과하면 구문 에러가 난다 (예: `s_concert` -> 9글자이므로 에러, `s_conc` -> 6글자이므로 통과).

### 어떻게 확인하는가

에디터에서 TABLES와 SELECT-OPTIONS를 활용해 화면과 레인지 변수를 개설하고, 루프를 돌려 Range Table 데이터를 수동 검증한다.

```abap
REPORT zch12_l02_select_options.

" 1. [공식 확인 지침] FOR 필드 참조 상속을 위해 TABLES 명세 선언 필수!
TABLES zbooking.

" 2. s_ 접두어 장착 및 8글자 이내의 이름 s_conc로 화면/변수 1타 2피 개설!
SELECT-OPTIONS s_conc FOR zbooking-concert_id.

" 3. s_conc의 한 행 모양을 본뜬 Work Area를 LIKE LINE OF로 매핑 선언
DATA ls_conc LIKE LINE OF s_conc.

START-OF-SELECTION.
  " 4. 사용자가 화면에 입력한 다중 조건 표를 한 줄씩 꺼내서 WRITE 인쇄 검증
  LOOP AT s_conc INTO ls_conc.
    WRITE: / 'SIGN:',   ls_conc-sign, 
             'OPTION:', ls_conc-option, 
             'LOW:',    ls_conc-low, 
             'HIGH:',   ls_conc-high.
  ENDLOOP.
```

#### 체험/시뮬레이터 설계 (SELECT-OPTIONS 화면 변환기)
- **프로세스 플로우**:
  1. 학습자가 가상 Selection Screen 화면의 From 칸에 `C001`, To 칸에 `C003`을 입력하고 [실행]을 클릭한다.
  2. 화면의 입력값이 슥 아래로 붕괴되며, RAM 메모리 영역에 개설된 `s_conc` 내부 테이블의 1번 행으로 이동한다.
  3. `[I | BT | C001 | C003]` 데이터로 조립되어 안착하는 물리 동기화 과정을 관찰한다.
- **상태 및 데이터**:
  - `SELECT-OPTIONS s_concert_id FOR zbooking-concert_id.` (9글자 변수명) -> 컴파일 시 "s_concert_id is too long. Maximum 8 characters" 에러 하이라이트.
  - `TABLES` 선언을 생략하고 저장 -> "TABLES zbooking is missing for reference" 사전 검사 실패 피드백.
- **피드백**: SELECT-OPTIONS는 화면의 물리 박스와 프로그램 내부의 Range Table을 다리로 이어주는 1타 2피의 도구임을 가이드한다.

### 실수/주의

- **8글자 변수명 초과**: 현업에 나가서 실무 코딩을 할 때 가장 짜증나고 자주 겪는 에러다. 무심코 변수명을 `s_concert`로 길게 적었다가 활성화가 막히므로, 셀렉션 옵션의 이름은 무조건 **`s_`** 를 포함해 **8자리 컷**을 지켜야 한다.

### 정리

- **`SELECT-OPTIONS`** 는 화면 입력창과 내부 Range Table을 동시에 창조하는 1타 2피의 명령 구문이다.
- 필드 참조 상속을 위해 윗줄에 반드시 **`TABLES 테이블명.`** 선언을 선행해야 한다.
- 변수 명칭은 **`s_`** 접두어를 사용하며, 스크린 한계로 인해 **8자 이내**로 제한하여 작명한다.

## CH12-L03 - WHERE … IN (classic range)

### 왜 필요한가

사용자가 화면에서 입력한 필터 조건 데이터들을 `s_conc` Range Table 표 안에 이쁘게 차곡차곡 쌓는 데 완벽하게 성공했다. 

하지만 기차가 화물을 가득 싣고 대기만 하고 있을 뿐, 이 화물을 DB에서 데이터를 퍼 오는 조회 트랙과 연결하지 않으면 예매 목록 리포트는 여전히 전체 1,000만 건 데이터를 무식하게 다 긁어와서 보여주는 미완성 상태에 머문다. 
Range Table에 담긴 복잡다단한 다중 조건들을 **데이터베이스 쿼리 조회의 핵심 조건절에 핀포인트로 장착하여, 실제 검색 대상을 타깃 영역으로 정밀 축소시키는 쿼리 체결 기술**이 요구된다.

### 무엇인가

#### 1. 범위 조건의 핵심 열쇠: WHERE 필드 IN 레인지테이블
- **구문**:
  ```abap
  SELECT * FROM zbooking INTO TABLE lt_book WHERE concert_id IN s_conc.
  ```
- **동작**: 단 1줄의 `IN` 키워드 체결만으로, DB 엔진은 `s_conc` 레인지 테이블 내부에 쌓여있는 복잡한 포함 범위, 제외 목록, 패턴 조건들을 한눈에 읽어 들여 SQL 문맥에 맞는 타깃 필터를 동적으로 구성해 수행한다.
- **classic 가이드**: 우리는 classic 구문을 수호하므로 호스트 변수 탈출용 골뱅이 `@`를 절대 붙이지 않고 쌩으로 **`IN s_conc`** 라고 적는다.

#### ⚠️ [물리 DB 엔진의 비정합성 함정: 빈 Range는 무조건 참(True)]
- *이 장에서 가장 중요하게 인지해야 할 시스템 거동 법칙이다.*
- 만약 사용자가 화면의 `s_conc` 입력 칸에 From/To 값을 아무것도 입력하지 않고 깨끗하게 비워둔 채 [실행]을 누르면, `s_conc` Range Table은 0행 텅 빈 initial 상태가 된다.
- 이때 DB 엔진은 "조건이 비었으니 0건을 조회해라" 라고 처리하지 않고, **"공연 ID 필터 조건은 무제한 전면 해제(참, True)한다"** 고 판정한다.
- **결과**: 조건이 비어 있으면 **테이블 전체 데이터를 고스란히 다 긁어서** 화면에 올려버리므로, 데이터가 1억 건인 운영 서버에서 검색어 입력을 깜빡하고 조회를 누르면 시스템 메모리가 폭사하는 심각한 병목을 유발한다.

### 어떻게 확인하는가

에디터에서 ZBOOKING 조회의 WHERE IN classic 연동 구문을 빌드하고, 빈 조건 시의 전체 조회 거동을 검수한다.

```abap
REPORT zch12_l03_where_in.

TABLES zbooking.
SELECT-OPTIONS s_conc FOR zbooking-concert_id.

DATA gt_book TYPE TABLE OF zbooking.

START-OF-SELECTION.
  " 1. classic WHERE ... IN 구문 결합 (@ 기호 금지)
  SELECT * FROM zbooking 
    INTO TABLE gt_book
    WHERE concert_id IN s_conc.

  " 2. sy-subrc 및 sy-dbcnt 상태 검수
  IF sy-subrc = 0.
    " s_conc를 비우고 실행 시 18건 전체가 들어오며 성공 판정(sy-subrc = 0)이 뜸!
    WRITE: / '조회 성공, 건수:', sy-dbcnt. 
  ELSE.
    WRITE: / '검색 결과가 창고에 존재하지 않습니다.'.
  ENDIF.
```

#### 체험/시뮬레이터 설계 (WHERE … IN 필터 엔진)
- **프로세스 플로우**:
  1. 학습자가 `s_conc` 입력 창을 완전히 비운다.
  2. [SELECT 실행]을 클릭한다.
  3. DB 필터 게이트의 가이드라인 가시철망이 슥 내려앉아 완전 개방되고, DB 내부의 18행 데이터가 무방비로 통과해 내부 테이블로 다 쏟아져 들어와 `sy-dbcnt = 18`이 찍히는 현상을 본다.
- **상태 및 데이터**:
  - `s_conc = C001만` 입력 상태 -> 필터 게이트 가동으로 C001에 속한 9건만 솎아져 통과하고 `sy-dbcnt = 9` 리포트 상태 확인.
  - `WHERE concert_id IN @s_conc` -> "@ is not allowed in classic SQL" 컴파일 에러 상태 피드백.
- **피드백**: 빈 레인지 테이블 조건은 "제한 없음(전체 조회)" 으로 뚫린다는 시스템 보안 법칙을 경고등으로 알린다.

### 실수/주의

- **빈 검색어로 인한 운영 서버 풀스캔 방치**: 쿼리 개발 시 사용자가 아무 조건도 안 주었을 때의 방어막(필수 입력 검증 `OBLIGATORY` 등, CH15)을 세워두지 않으면, 실수로 조회 버튼을 누른 사용자에 의해 매번 전체 테이블 스캔 부하가 발생해 서버가 정지되는 장애를 초래하므로 주의해야 한다.

### 정리

- **`WHERE 필드 IN 레인지테이블`** 한 줄로 다중 필터 조건을 DB 쿼리에 동적으로 체결한다.
- classic Open SQL 구문이므로 Range 변수 앞에 **골뱅이(@)를 쓰지 않는다.**
- **빈 Range Table은 항상 무조건 참(True)** 으로 해석되므로 전체 데이터를 긁어가 성능 부하를 내지 않도록 조건 필수 검증을 염두에 둔다.

## CH12-L04 - Multiple Selection과 Include/Exclude

### 왜 필요한가

사용자가 화면에서 From/To에 값 하나를 넣어서 단순 범위 조회를 돌리는 법을 정비했다. 
하지만 실제 현업의 회계 처리나 전표 정산 업무에서는 훨씬 까다로운 조합 요건을 들이민다. 

"공연 `C001`부터 `C005`까지 범위 전체를 다 보고 싶은데, 그중에서 세무상 문제가 발생한 `C004` 번호 한 녀석만 콕 집어서 최종 출력물에서 빼고(제외) 서빙해 주세요"
이 복잡한 다중 선택(Multiple Selection) 요건을 화면에서 수용하고, 그것이 Range Table 내부적으로 어떻게 포함(Include)과 제외(Exclude) 행들의 사슬로 얽혀 **어느 필터가 더 힘이 세게 최종 합격 레코드를 골라내는지 집합 연산 원리**를 꿰뚫어야 비정합성 버그 없는 견고한 리포트가 탄생한다.

### 무엇인가

#### 1. 다중 선택 화살표 버튼 (Multiple Selection)
- **팝업 창**: SELECT-OPTIONS 입력 박스 맨 오른쪽을 보면 작은 녹색 화살표 모양 단추가 달려있다. 
- **동작**: 이 화살표를 마우스로 클릭하면 단일 값 추가, 범위 추가, 단건 제외, 범위 제외를 여러 줄로 무한 등록할 수 있는 다중 조건 입력 팝업창이 개방된다. 이 팝업에 입력된 수많은 행들은 그대로 프로그램 내부 **Range Table의 다중 레코드 행**으로 변환 수납된다.

#### 🟢 녹색 포함(SIGN = I) 과 🔴 빨강 제외(SIGN = E)의 집합 판정 룰
팝업창에 등록된 다중 행 조건들은 다음 2단계의 수학적 집합 연산을 거쳐 최종 데이터 통과를 심사한다.

1. **1단계: 포함(SIGN = I, Include) 범위의 OR 확장**:
   - `SIGN = I` 인 행이 여러 개 있으면, 시스템은 **"이 조건들 중 단 하나라도(OR) 걸려들면 일단 합격 후보군으로 수용한다"** 고 판정한다. (후보망을 넓게 넓게 펼치는 1차 거름망).
2. **2단계: 제외(SIGN = E, Exclude) 범위의 AND 컷트**:
   - `SIGN = E` 인 행은 포함 범위보다 훨씬 강력한 힘을 발휘한다. **"포함 조건 망에 걸려들어 합격했더라도, 제외 조건에 단 하나라도 걸리면 최종 칼날로 무참히 잘라내 탈락시킨다"** 고 판정한다. (가장 힘이 센 최종 문지기).

```text
  [DB 원천 로우] 
        │
        ▼
   [ 1단계 가드: Include (I) ]  ───> 조건 불일치 시 즉각 탈락!
        │ (OR 결합으로 통과)
        ▼
   [ 2단계 가드: Exclude (E) ]  ───> 제외 필터에 단 하나라도 걸리면 탈락!
        │ (AND 차감)
        ▼
  [ 최종 합격 데이터 출력 ]
```

### 어떻게 확인하는가

에디터와 팝업 조건 대조표를 통해 포함과 제외가 결합하여 2번이 걸러지고 탈락하는 논리 과정을 검증한다.

```text
Range Table 적재 상태:
행 1: SIGN = 'I', OPTION = 'BT', LOW = 'C001', HIGH = 'C003' (C001~C003 후보 포함)
행 2: SIGN = 'E', OPTION = 'EQ', LOW = 'C002', HIGH = ''     (C002 단건 차단 제외)

대조 데이터 판정 모션:
- C001 데이터 -> 1단계 통과 (Include), 2단계 통과 -> 최종 합격 노출!
- C002 데이터 -> 1단계 통과 (Include), 2단계 걸림 (Exclude) -> 불합격 즉시 탈락!
- C003 데이터 -> 1단계 통과 (Include), 2단계 통과 -> 최종 합격 노출!
- C004 데이터 -> 1단계 걸림 (Include 불합격) -> 즉시 탈락!
```

#### 체험/시뮬레이터 설계 (Include/Exclude 판정기)
- **프로세스 플로우**:
  1. 학습자가 ZBOOKING 예매 대장에 `C001`, `C002`, `C003` 데이터를 채워둔다.
  2. 다중 선택창에 `C001~C003 포함(🟢)` 칩과 `C002 제외(🔴)` 칩을 마우스로 조립한다.
  3. [판정 돌리기]를 누른다.
  4. 데이터들이 떨어질 때, `C002` 로우 상자가 🟢 가림막은 통과하지만 아래에 설치된 🔴 제외 바리케이드 장벽에 꽝 부딪혀 부서져서 탈락하고, `C001/C003`만 통과 박스에 안착하는 물리 애니메이션을 확인한다.
- **상태 및 데이터**:
  - `Include 'C001'` & `Include 'C003'` 세팅 상태 -> C001 또는 C003 중 하나만 맞아도 합격하는 OR 결합 상태 데이터 렌더링.
- **피드백**: 제외(Exclude, E)가 포함(Include, I)보다 나중에 적용되어 최종 삭감하는 더 힘이 강력한 파괴 필터임을 알려준다.

### 실수/주의

- **포함(I) 여러 행을 교집합(AND)으로 오해**: 초보 설계자들이 가장 많이 착각하는 로직이다. "공연 ID 필드에 `C001 포함` 과 `C003 포함` 두 행을 등록했으니, 공연 ID가 C001이면서 동시에 C003인 행을 찾겠지?" 라고 오해한다. 하나의 필드에 포함 `I` 조건이 여러 줄 수납되면 합집합인 **OR**로 넓어지므로, 당연히 두 공연 데이터가 모두 출력됨을 기억해야 한다.

### 정리

- 셀렉션 옵션 오른쪽의 **다중 선택 화살표**를 누르면 여러 값의 포함/제외 다중 행 레코드가 Range Table에 수납된다.
- **포함(I)은 녹색** 아이콘으로 후보를 넓게 합집합(OR)하고, **제외(E)는 빨강** 아이콘으로 최종 차집합(AND 차감)한다.
- 제외(Exclude) 컷트 필터는 포함(Include) 필터보다 나중에 작동하는 더 강력한 최종 검문 규칙이다.

## CH12-L05 - EQ / BT / CP 옵션 이해

### 왜 필요한가

Range Table의 비교 연산 방식인 `OPTION` 들을 공부하고 있다. 
단순 대조인 `EQ`나 사이 범위인 `BT`는 쉽게 이해하고 다룬다. 

하지만 사용자가 "공연 번호가 `C`로 시작하는 모든 공연만 리스트업해주세요" 라고 할 때 쓰는 패턴 매칭인 **`CP` (Contains Pattern)** 옵션이 기어가는 내부 연산 기호 방식을 모르면 에러를 유발한다. 
아밥의 `CP` 옵션은 우리가 흔히 SQL 조건절이나 코딩에서 사용하던 `%` 나 `_` 기호를 쓰면 매칭이 성립하지 않는다. **Range Table 전용 와일드카드 기호와 패턴 판정 룰**을 머리에 정확히 박아두어 버그를 미연에 차단해야 한다.

### 무엇인가

#### 1. 자주 쓰는 3대 OPTION 대표 주자
- **`EQ` (Equal - 같음)**: `LOW` 값 하나와 완전히 일치하는 것. (`HIGH`는 비워둠).
- **`BT` (Between - 사이)**: `LOW` 값 이상부터 `HIGH` 값 이하 범위에 드는 것. (반드시 두 필드 다 채워야 함).
- **`CP` (Contains Pattern - 패턴 매치)**: 문자열의 와일드카드 형태를 만족하는 것. (`LOW`에만 와일드카드 기호를 엮어 쓰고 `HIGH`는 비워둠).

#### 2. CP (Contains Pattern) 의 절대적 와일드카드 2대 기호
*SQL `LIKE` 와 반드시 기호를 쪼개어 구별해 써야 하는 철칙이다.*
- **`*` (스타 - 가변 문자열)**: 글자 수 제한이 없는 무한 자유도를 점유한다. (예: `C*` -> C로 시작하는 모든 글자 수의 코드).
- **`+` (플러스 - 단 1글자 점유)**: 정확히 **딱 한 글자**의 자릿수 점유만 허용한다. 
  - *예시*: `C+01` 은 `C` 뒤에 딱 1자가 붙고 그 뒤에 `01`이 붙은 **4자리 코드만 정확히 매칭**한다. `C001`은 매치 성공이나, 자릿수가 다른 `C0010`은 매치 실패로 탈락시킨다.
- **LIKE와의 대조**: SQL의 `LIKE`는 `%` 와 `_`를 쓰지만, Range Table의 `CP`는 **`*` 와 `+`** 를 표준으로 삼는다.
- **[공식 사전 작동 규칙]**: `OPTION = CP` 나 `NP`를 Range Table에 강제 선언해 주었을 때는, 우측 `LOW` 값 내부에 반드시 최소 1개 이상의 와일드카드 문자(`*` 또는 `+`)가 섞여 들어가 있어야 정상 구동되며, 와일드카드 기호가 전혀 없는데 CP를 지정해 던지면 프로그램이 비정상 종료된다.

### 어떻게 확인하는가

에디터와 패턴 매칭 보드를 통해 CP 옵션의 와일드카드 자릿수 판정 거동을 대조 확인한다.

```abap
" CP 패턴 매칭을 위한 Range Table 한 행 수동 기입 정석 예시
DATA: lr_conc TYPE RANGE OF zbooking-concert_id,
      ls_conc LIKE LINE OF lr_conc.

ls_conc-sign   = 'I'.
ls_stat-option = 'CP'.  " 패턴 비교 지정
ls_stat-low    = 'C*'.  " [공식 스펙] 반드시 와일드카드 * 또는 + 기호를 엮어야 함!
APPEND ls_conc TO lr_conc.
```

#### 체험/시뮬레이터 설계 (OPTION 비교 실험실)
- **프로세스 플로우**:
  1. 학습자가 드롭다운에서 비교 옵션을 `CP`로 고른다.
  2. `LOW` 입력 창에 `C+01` 패턴을 기입하고 [판정 구동]을 누른다.
  3. `C001` (4자리) 상자가 굴러떨어질 때는 초록불이 켜지며 매칭 통과되나, `C0010` (5자리) 상자가 접근할 때는 "자릿수 불일치 탈락" 상태로 튕겨 나가는 시각 피드백을 본다.
- **상태 및 데이터**:
  - `LOW`에 `C%` 기입 상태로 실행 -> 와일드카드 인식 불가로 인해 ZCONCERT 데이터가 단 1줄도 검색되지 않는 무반응 버그 상태 연출.
- **피드백**: SQL LIKE와 Range CP의 기호 편차 매퍼(`% / _` vs `* / +`)를 뚜렷한 대비표로 렌더링한다.

### 실수/주의

- **CP 옵션에 SQL 와일드카드 기호 기입**: 초보 개발자들이 가장 흔히 틀리는 지점이다. 쿼리 짜던 타성으로 `LOW = 'C%'` 를 기입하고 조회하면 아두런 데이터도 솎아내지 못하는 먹통 버그를 유발하므로, 레인지 테이블 CP 절에는 무조건 **`*`** 을 적어 넣어야 한다.

### 정리

- **`EQ`** 는 정확히 같음, **`BT`** 는 닫힌 사이 범위, **`CP`** 는 와일드카드 패턴 대조를 뜻한다.
- `CP` 옵션 사용 시에는 반드시 값 내부에 **`*`** (가변 길이) 또는 **`+`** (정확히 1자) 와일드카드 기호를 탑재해 주어야 한다.
- SQL `LIKE`의 기호들과 섞어 쓰지 않도록 기호의 물리적 편차를 명확히 격리한다.

## CH12-L06 - Selection Table 직접 조작 기초

### 왜 필요한가

지금까지 우리는 사용자가 화면에서 마우스로 입력창을 조작해 Range Table을 자동으로 채워주는 SELECT-OPTIONS 문법을 공부했다. 

하지만 모든 검색 필터 조건이 사용자의 모니터 입력창에서만 전달되지는 않는다. 
"매일 새벽 2시에 아무도 로그인하지 않은 상태에서 서버가 혼자 가동되는 대용량 정산 배치 프로그램", "사용자가 로그인하면 그 사용자의 사번 권한 정보를 뒤져서 조회 범위를 코드로 자동 차단 가드해 채워야 하는 보안 프로그램" 등. 
**화면 입력창이 아예 존재하지 않는 어둠의 백그라운드 환경에서, 순수 아바 코딩만으로 Range Table을 수동으로 조립해 채워 넣는 직접 제어 기술**이 필수적으로 요구된다.

### 무엇인가

#### 1. RANGES 구식 레거시 경고 인지 지침
- **경고**: 옛날 구형 소스를 뜯어보면 `RANGES r_stat FOR zbooking-status.` 라고 선언되어 있는 구문을 보게 된다. 이 방식은 `WITH HEADER LINE` 처럼 1인 2역 테이블을 만드는 **obsolete(폐기대상 구식)** 문법이다. 신규 개발 시에는 절대로 새로 만들지 않는다.

#### 2. 현대 표준 선언: TYPE RANGE OF
화면 없이 순수 프로그램 메모리 내부에서 돌릴 범위 테이블을 개설할 때는 전용 지시어인 **`TYPE RANGE OF`** 를 활용해 클린 선언한다.
```abap
DATA lr_stat TYPE RANGE OF zbooking-status. " lr_stat 범위 테이블 선언
```

#### 3. 코드로 직접 조건 1행을 조립해 채우는 3단계 수송
1. **Work Area 비우기 (`CLEAR`)**: 
   - 전 단계 연산의 찌꺼기 데이터(특히 범위용 `HIGH` 값 잔상)가 방안에 남아 데이터 오염 사고를 일으키는 것을 막기 위해 우선 상자 방을 깨끗이 비운다.
2. **필드 값 기입**: 
   - `sign`, `option`, `low`, `high` 필드에 우리가 배운 수식을 코드로 대입한다.
3. **기차 적재 (`APPEND`)**:
   - `APPEND ls_stat TO lr_stat.` 명령을 쳐서, 다 조립된 조건 행을 `lr_stat` 레인지 기차 꼬리에 수납한다.

### 어떻게 확인하는가

에디터에서 TYPE RANGE OF와 CLEAR, APPEND를 활용해 화면 없는 조건 필터를 수동 조립하고 조회를 실행한다.

```abap
REPORT zch12_l06_range_direct.

" 1. [구식 RANGES 우회] 현대 표준 TYPE RANGE OF 구문으로 lr_stat Range Table 선언!
DATA: lr_stat TYPE RANGE OF zbooking-status,
      ls_stat LIKE LINE OF lr_stat, " 한 행을 담을 작업 영역
      gt_book TYPE TABLE OF zbooking.

START-OF-SELECTION.
  " 2. 1회차 조건 조립: 상태 N(예약) 포함 등록
  CLEAR ls_stat. " 이전 방 청소 필수!
  ls_stat-sign   = 'I'.
  ls_stat-option = 'EQ'.
  ls_stat-low    = 'N'.
  APPEND ls_stat TO lr_stat. " 레인지 기차 1번 칸 탑승 완료

  " 3. 2회차 조건 조립: 상태 C(취소) 제외 등록
  CLEAR ls_stat. " 방 청소 누락하면 이전 'N' 상태가 고여있게 됨!
  ls_stat-sign   = 'E'.
  ls_stat-option = 'EQ'.
  ls_stat-low    = 'C'.
  APPEND ls_stat TO lr_stat. " 레인지 기차 2번 칸 탑승 완료

  " 4. 수동 조립 완료된 lr_stat 레인지 테이블을 classic WHERE IN 절에 꽂아 조회!
  SELECT * FROM zbooking 
    INTO TABLE gt_book
    WHERE status IN lr_stat.

  IF sy-subrc = 0.
    WRITE: / '필터링 조회 성공 건수:', sy-dbcnt.
  ENDIF.
```

#### 체험/시뮬레이터 설계 (코드로 Range 행 추가 스텝퍼)
- **프로세스 플로우**:
  1. 학습자가 `CLEAR ls_stat` 단계를 실행한다. 가상 `ls_stat` 방의 변수들이 회색으로 비워진다.
  2. 필드 대입 단계를 지나자 `[I | EQ | N | ]` 데이터가 방에 찬다.
  3. `[APPEND 실행]` 버튼을 탭한다. `ls_stat` 방의 데이터가 복사되어 하단의 `lr_stat` 레인지 테이블 레일의 1번 칸으로 덜컥 추가 수납되는 모션을 본다.
- **상태 및 데이터**:
  - `CLEAR` 단계를 고의로 건너뛰고 2회차 대입 가동 -> 이전의 SIGN, OPTION 잔상이 그대로 상속되어 덮어쓰기 데이터가 꼬이는 위험 상태바 점등.
- **피드백**: RANGES 구문은 폐기대상이므로 새 개발 시에는 반드시 `TYPE RANGE OF` 정석 선언을 활용하라는 Clean 지침을 환기시킨다.

### 실수/주의

- **수동 적재 시 CLEAR 누락**: 특히 `BT` 옵션으로 첫 행을 넣어 `HIGH` 에 값을 채운 뒤, 두 번째 행으로 `EQ` 단건을 추가할 때 `CLEAR` 를 누락하면, 이전 행의 `HIGH` 잔상이 그대로 고여있는 상태로 `APPEND` 되어 엉뚱한 범위 조회가 나가는 치명적인 데이터 누수 버그를 내게 되므로 **매 `APPEND` 전 `CLEAR` 선언**을 엄수해야 한다.

### 정리

- 화면 입력창이 부재한 환경에서는 **`TYPE RANGE OF 필드`** 로 독립 범위 테이블을 개설해 조작한다.
- 1행 조건 조립 시에는 **`CLEAR` $\rightarrow$ 값 할당 $\rightarrow$ `APPEND`** 3단계 파이프라인을 지킨다.
- 구식 레거시인 **`RANGES`** 선언은 사용하지 않고 최신 현대적 참조 형식을 표준으로 삼는다.

## CH12-L07 - 실습 — 공연·상태로 예매 필터

### 왜 필요한가

우리가 화면과 레인지를 동시에 개설하는 SELECT-OPTIONS 문법과, 다중 선택의 SIGN 포함/제외 판정, 콤마와 골뱅이를 배제해 대입하는 `WHERE ... IN` classic 결합 원리를 정교하게 섭렵했다. 

이제 우리가 콘서트 예매 앱 1단계(DB 설계)와 3단계(SALV 출력)의 대교단을 관통해 쌓아 올렸던 기술들을 유기적으로 총결합할 시간이다. 사용자가 화면에서 "공연 ID 범위는 C001부터 C003까지로 하되, 취소(C) 상태인 예매 건은 쏙 빼고 엑셀 표로 출력해주세요" 라고 입력했을 때, **정밀하게 필터링된 타깃 데이터만 ZBOOKING에서 퍼 올려 화면에 깔끔한 SALV 표로 인쇄해 내는 검색 조건형 예매 목록 리포트의 완성본**을 내 손으로 직접 실현해 보아야 한다.

### 무엇인가

#### 🎫 검색 조건 연동형 예매 목록 SALV 리포트 아키텍처
우리는 TABLES 명세를 세팅하고 두 개의 셀렉션 옵션 화면 필드를 배치한다.

1. **사전 및 화면 요소 빌드**:
   - `TABLES zbooking.`: zbooking 컬럼 구조 상속 선언.
   - `s_conc FOR zbooking-concert_id`: 공연 ID 범위 입력창 및 s_conc Range Table 개설.
   - `s_stat FOR zbooking-status`: 예매 상태 범위 입력창 및 s_stat Range Table 개설.
2. **데이터 조회 파이프라인**:
   - `WHERE concert_id IN s_conc AND status IN s_stat`: 두 레인지 테이블의 다중 포함/제외 필터 조합을 모두 만족(`AND`)하는 행만 ZBOOKING DB 창고에서 긁어와 `lt_book` 기차에 수납.
3. **그리드 화면 표출**:
   - `lo_alv` 포인터를 팩토리에 덤프 예외(`cx_salv_msg`) 처리해 조립하고, 툴바 버튼을 다 개방한 뒤 display로 스크린 화면에 띄움.

### 어떻게 확인하는가

에디터에서 검색 필터형 SALV 리포트 실습 코드를 작성해 활성화하고, 조건 입력에 따른 그리드 행 차감 상태를 종합 검수한다.

1. `/nSE38`로 이동해 프로그램 `ZREP_CONCERT_FILTER`를 생성한다.
2. 아래의 완성형 소스 코드를 기입하고 활성화(Ctrl+F3)한다.
3. 프로그램을 실행(`F8`)하여 From/To 입력창이 예쁘게 배치된 선택 화면(Selection Screen)을 확인한다.
4. **`공연ID From: C001, To: C001`** 세팅 후 실행을 눌러 C001 공연 건만 골라져 ALV 표에 서빙되는지 확인한다.
5. 뒤로 돌아와 오른쪽 다중 선택 화살표를 누르고 **`제외할 단일 값에 C (취소) 등록`** 한 뒤 실행하여 취소 상태 행이 완벽하게 필터 탈락되어 정산 집계에서 배제되는 무결성 작동을 최종 검수 통과한다.

```abap
REPORT zrep_concert_filter.

" 1. FOR 참조를 위한 TABLES 선언 및 1타 2피 SELECT-OPTIONS 개설
TABLES zbooking.

SELECT-OPTIONS: s_conc FOR zbooking-concert_id,
                s_stat FOR zbooking-status.

" 2. 데이터 기차 및 ALV 포인터 전역 선언
DATA: gt_book TYPE TABLE OF zbooking,
      go_alv  TYPE REF TO cl_salv_table.

START-OF-SELECTION.
  " 3. 파이프라인 1단계: classic WHERE IN 구문을 AND로 결합해 정교한 필터 조회!
  SELECT * FROM zbooking 
    INTO TABLE gt_book
    WHERE concert_id IN s_conc
      AND status     IN s_stat.

  " 4. 파이프라인 2단계: cx_salv_msg 덤프 방어하며 기계 조립
  TRY.
      cl_salv_table=>factory(
        IMPORTING r_salv_table = go_alv
        CHANGING  t_table      = gt_book
      ).

      " 5. 파이프라인 3단계: 툴바 버튼 일괄 개방 및 display 빔프로젝터 투사!
      go_alv->get_functions( )->set_all( abap_true ).
      go_alv->display( ).

    CATCH cx_salv_msg.
      MESSAGE '필터 리포트 화면 조립 중 예외가 발생했습니다.' TYPE 'I'.
  ENDTRY.
```

#### 체험/시뮬레이터 설계 (조건 입력형 예매 목록 실습)
- **프로세스 플로우**:
  1. 학습자가 선택화면 조건창에서 `s_conc = C001` 단건을 주고, `s_stat = C 제외(🔴)` 칩을 엮어 [실행]을 누른다.
  2. ZBOOKING DB의 18행 원본 레코드 중 C001에 속하면서 예약(`N`) 상태인 9행 상자들만 필터링 벨트를 통과해 살아남는다.
  3. go_alv 팩토리 렌즈를 타고 ALV 격자판 위에 총 9줄만 이쁘게 안착 정렬되는 모션을 최종 검수 확인한다.
- **상태 및 데이터**:
  - `모든 조건을 빈칸으로 둔 채 실행` -> 빈 range = true 자동 해제에 따라 18건 전체가 조인 없이 표출되는 풀스캔 상태 데이터 출력.
- **피드백**: SELECT-OPTIONS의 화면 조작이 Range Table의 SIGN, OPTION 다중 행으로 둔갑해 DB 필터를 정교하게 축소 지배함을 체감한다.

### 실수/주의

- **SELECT-OPTIONS FOR zbooking 선언 시 TABLES 누락**: "TABLES zbooking 선언을 빼먹어서 zbooking-concert_id 필드 타입을 인지할 수 없다는 컴파일 빨간 에러가 납니다" 하는 질문은 classic 참조 규칙을 빠뜨렸기 때문이므로 반드시 TABLES 세트를 동반 기입해야 한다.

### 정리

- **`SELECT-OPTIONS`** 와 **`WHERE ... IN`** 절의 classic 결합을 통해 다중 필터형 콘서트 예매 목록 리포트의 완성 뼈대를 창조한다.
- 빈 범위 조건은 무제한 조회가 되므로 성능적 병목 요인을 이해하고, 다중선택 팝업의 포함(I)과 제외(E) 집합 연산 법칙을 코드로 통제한다.
- **`SALV`** 격자 렌더링 화면과 결합하여 실무 현업에서 당장 쓸 수 있는 **조건 입력형 조회 리포트의 기본 포맷**을 완수한다.
