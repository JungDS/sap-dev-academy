## 4. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. LDB(Logical Database)가 없는 현대의 일반적인 SELECT 방식 신규 리포트 프로그램에서 END-OF-SELECTION을 구현할 필요가 전혀 없는 구체적인 아키텍처적 이유는 무엇인가요?
**A**: **데이터 공급이 끝났음을 마감 신호로 수신할 옛 Logical Database(LDB) 프레임워크가 아예 존재하지 않아, 이벤트를 개설해도 `START-OF-SELECTION` 꽁무니와 물리적으로 완전히 똑같이 실행되어 흘러가기 때문입니다.**
과거 SAP가 애용하던 LDB는 선택화면 띄우기와 데이터 SELECT 쿼리 작업을 시스템이 사전 정의된 채로 알아서 가동하고, 데이터를 한 줄씩 아밥 메모리로 퍼 올려주었습니다. 
이 LDB 공급 작업이 마침내 완결(End)되었을 때 바통을 이어받아 최종 보고서 출력을 뽑기 위해 존재했던 이벤트가 `END-OF-SELECTION` 이었습니다.
하지만 현대의 모든 리포트 프로그램은 LDB를 쳐다보지 않고, 우리가 직접 `START-OF-SELECTION` 구획 안에 `SELECT ... INTO TABLE gt_booking` 문장으로 다이렉트 쿼리를 던져 한 번에 수납합니다. 
이 구조에서는 `START-OF-SELECTION` 내부에서 조회가 이미 단번에 끝나기 때문에, 굳이 아래에 `END-OF-SELECTION` 방을 따로 파서 출력 코드를 쪼개 적어둘 하등의 이유가 없습니다. 적어두어도 그냥 순차적으로 이어서 도는 것에 불과하므로, 현대 공식 명세에서도 LDB가 없는 비-LDB 리포트에서는 이 이벤트를 아예 코딩할 필요가 없다며 **Obsolete(낡은 문법)** 로 가두어 놓은 것입니다.

### Q2. AT SELECTION-SCREEN ON <field>를 통해 에러를 트리거(MESSAGE TYPE 'E')했을 때, 화면에서 오직 그 필드만 입력 가능하게 하이라이트 활성화되고 나머지 모든 필드는 회색으로 잠기는(Lock) 내부 작동 원리가 무엇인가요?
**A**: **SAP 화면 처리 엔진이 에러가 난 특정 필드의 '체인 구조(Chain/Field)' 범위만 입력 재허용 대기로 묶고, 체인 바깥에 있는 나머지 모든 화면 컴포넌트들의 input 속성을 임시로 '0' (읽기전용)으로 얼려 잠가버리기 때문입니다.**
- **`AT SELECTION-SCREEN` (전체 검증)**:
  특정 필드를 지정하지 않고 단독으로 사용하면, 화면 전체 필드가 단일 체인으로 통째로 묶입니다. 여기서 에러(E)를 터트리면 화면의 모든 필드가 "검사 대상"으로 분류되어 전부 다 활성화된 채로 다시 복귀합니다.
- **`AT SELECTION-SCREEN ON p_conc` (개별 필드 검증)**:
  아밥 시스템은 화면 PAI 단계에서 이 지시어를 만나면, **오직 `p_conc` 필드 하나만 별도의 검증 체인(Chain)으로 독립 격리**시킵니다. 이 격리된 방 안에서 `MESSAGE ... TYPE E`가 폭발하면, 엔진은 **"아하, 오직 p_conc 필드에만 유효성 문제가 있구나. 사용자가 딴청 피우지 못하게 이 필드만 빨갛게 열어두고, 멀쩡한 나머지 필드들은 회색으로 고정 잠금해 시선을 이 칸에 집중시키자"** 라고 제어합니다.
여러 필드의 선후 관계가 아니라 단일 코드의 공백이나 마스터 존재성 검사는 반드시 `ON <필드>` 를 써주어야 현업 담당자가 수정할 곳을 단번에 눈으로 찾아 집중할 수 있는 수려한 UX가 완성됩니다.

### Q3. 선택화면 안에 개설한 본문 PUSHBUTTON이나 툴바 FUNCTION KEY를 사용자가 마우스로 클릭했을 때, 왜 일반 화면과 달리 sy-ucomm 변수가 아닌 sscrfields-ucomm 시스템 변수를 사용하여 제어해야 하나요?
**A**: **선택화면(Screen 1000번 등)은 SAP 내부적으로 커널이 자동 관리하는 '특수 화면 폼'이라, 일반 개발자가 커스텀하는 ucomm 변수 버퍼인 sy-ucomm 방으로 단축 코드가 유입되지 않고 차단 튕겨 나가도록 시스템 코어가 제약되어 있기 때문입니다.**
일반적으로 우리가 그리는 Dynpro 화면(CH16)에서는 버튼을 누르면 단축 코드가 `sy-ucomm` 변수 방에 즉시 기입되어 누구나 읽을 수 있습니다.
그러나 선택화면은 개발자가 화면 레이아웃을 수동으로 디자인한 화면이 아니라, `PARAMETERS` 등의 선언적 키워드를 보고 SAP NetWeaver 시스템이 뒤에서 자동으로 화면을 빌드한 특수 영역입니다. 
따라서 SAP는 선택화면 내에서 발생하는 보조 버튼 액션(푸시버튼, 툴바 단축아이콘)의 ucomm은 일반 화면용 `sy-ucomm` 과 엄격히 충돌을 분리하기 위해, **`sscrfields` 라는 전용 사전(DDIC) 구조체를 따로 준비하고 오직 `sscrfields-ucomm` 방으로만 단축 코드가 적재되도록 전송 차단벽**을 세워두었습니다. 
이 궤도를 모른 채 `CASE sy-ucomm` 만 백날 비교 코딩하면 버튼을 아무리 세게 클릭해도 프로그램이 묵묵부답으로 굳어버리는 컴파일 덤프 미감지 버그를 겪게 되므로, **`TABLES sscrfields.` 도킹 선언과 `sscrfields-ucomm` 대조**는 선택화면 버튼 제어의 단 하나의 정석 길입니다.

### Q4. PARAMETERS의 LOWER CASE 옵션을 날짜형(d) 이나 숫자형(i, p) 변수 뒤에 붙였을 때 왜 컴파일 빌드 오류가 발생하나요? 문자형에 한정되는 아키텍처적 이유를 설명해 주세요.
**A**: **대소문자(Uppercase/Lowercase)라는 개념의 실체 자체가 오직 문자 코드(ASCII/Unicode 등)를 다루는 문자형(Character-like) 데이터 타입에만 존재할 수 있는 문법 사양이기 때문입니다.**
- **문자형 필드 (`TYPE C`, `String` 등)**:
  사용자가 키보드로 영어를 치면 소문자 `a` (Unicode `U+0061`) 와 대문자 `A` (Unicode `U+0041`) 가 엄연히 다른 문자 코드로 존재합니다. SAP는 검색 조건 대조 효율을 극대화하기 위해 기본적으로 소문자 `a`를 전부 대문자 `A`로 치환해 변수 방에 전송하지만, `LOWER CASE` 옵션을 켜두면 치환을 bypass 하고 `a` 그대로 문자 메모리에 수납을 허용합니다.
- **숫자/날짜형 필드 (`TYPE I`, `P`, `D` 등)**:
  숫자 100(정수)이나 오늘 날짜 `20260630`(내부 날짜형)은 컴퓨터 내부적으로 대소문자 구분이 아예 성립하지 않는 이진수(Binary) 정수 값이거나 고정 포맷 숫자열일 뿐입니다. 여기에 대소문자 보존 옵션인 `LOWER CASE` 스위치를 결합하려 들면, 컴파일러는 **"숫자 물리 메모리 공간에는 대소문자라는 속성 자체가 애초에 정의되어 있지 않는데 무엇을 보존하겠다는 것인가?"** 라고 판단하여 사전 구문 차단 에러를 일으키는 것입니다.
따라서 옵션 카탈로그를 설계할 때 LOWER CASE는 철저히 문자형에만 타깃 도킹하도록 격리해야 합니다.

### Q5. F1 도움말 및 F4 입력 도움말 시점(POH/POV)에, 사용자가 화면의 다른 입력창들에 타이핑해 둔 값들이 아밥 프로그램 변수 공간으로 자동으로 복사 전송(Transport)되지 않는 이유는 무엇이며, 이를 우회 해결하기 위한 기법은 무엇인가요?
**A**: **F1/F4 도움말 구획은 화면 제어의 최종 입력 승인 단계(PAI)에 도달하기 전인 '임시 상담 모션'이라, 아직 사용자가 화면 타이핑을 다 마치지도 않았는데 중간 단계에서 함부로 화면 전체 데이터를 백엔드 변수 버퍼로 무단 이송 복사(Transport)하지 않도록 설계된 SAP GUI 화면 보호 규격 때문입니다.**
사용자가 한참 날짜를 치다가 중간에 F4를 눌러 팝업창을 켰다고 해보자. 이 시점은 아직 [실행]을 누른 최종 PAI 시점이 아닙니다. 
따라서 SAP GUI 화면 엔진은 아직 변수 방에 값을 복사해주지 않아, 프로그램 변수 공간은 여전히 텅 비어있게 됩니다.
만약 이 상태에서 "사용자가 위에 적어둔 공연장 ID 값을 조건으로 쿼리해서 하위 F4 목록을 걸러서 보여줘야지" 하고 `SELECT ... WHERE hall_id = p_hall` 로 단순 쿼리를 짜면, `p_hall` 변수가 텅 비어있으므로 F4 팝업창이 텅 비어버리는 오류가 발생합니다.
- **해결 기법**:
  이 화면 값 단절 장벽을 뚫기 위해, 도움말 이벤트 구획(`ON VALUE-REQUEST`) 내부에서 현재 화면 메모리에 임시로 떠 있는 날것의 타이핑 입력 값을 직접 강제로 훔쳐 오는 API 함수인 **`DYNP_VALUES_READ`** 나 **`RS_SELECTIONSCREEN_READ`** 를 호출해야 합니다. 이 함수를 호출하며 화면 필드 이름 `'P_HALL'` 을 넘겨주면, 임시 화면 버퍼를 수동으로 노크해 값을 수혈받아 오게 되며, 비로소 이 수혈받은 값으로 Dynamic SELECT 쿼리를 굴려 완벽하고 동적인 지능형 F4 필터링 팝업창을 구현할 수 있게 됩니다.
