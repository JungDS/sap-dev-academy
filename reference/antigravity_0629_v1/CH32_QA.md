## 3. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. FOR ALL ENTRIES 쿼리를 날릴 때, 드라이버 테이블이 비어 있는지(IS NOT INITIAL)를 왜 귀찮게 매번 IF 문으로 가두어 확인해 주어야만 덤프 폭사를 막을 수 있나요?
**A**: **드라이버 테이블이 비어 있는(0건) 상태로 FAE 를 쏘면, 아바 SQL 엔진은 WHERE 절 조건을 아예 소멸 무시하고 대상 테이블 전체를 Full Scan 조회하여 수백만 건 데이터를 램에 부어 덤프를 터트리기 때문입니다.**
- **동작적 기괴함**:
  아바 컴파일러의 역사적 제약 중 하나로, `FOR ALL ENTRIES IN @lt_booking` 은 "lt_booking 에 들어있는 각 행들의 키 조건들만 다 긁어와라" 는 구문입니다.
  그런데 `lt_booking` 이 비어 있다면 키 조건이 '0개' 가 됩니다. 
  이때 엔진은 "조건이 0개이니 아무것도 조회하지 않겠다" 고 기동하는 게 아니라, **"조건이 아예 없으니 WHERE 절을 몽땅 생략(Bypass)하고, zconcert 테이블에 상주한 전체 1,000만 건 데이터를 모조리 긁어서 아바 램으로 배송하라"** 는 쌩뚱맞은 방식으로 돌변합니다.
  결과적으로 개발 서버에선 3줄짜리 예제라 잘 넘어가다가, 운영 서버의 1,000만 건 데이터 앞에서 즉사해 램 오버플로우 덤프(Page Allocation Failed)를 뿜고 전산실이 마비됩니다.
- **철칙 방어선**:
  ```abap
  " [ FAE 기동 바로 윗줄에 안전 가드를 필사적으로 둘러야 함! ]
  IF lt_booking IS NOT INITIAL.
    SELECT concert_id FROM zconcert
      FOR ALL ENTRIES IN @lt_booking
      WHERE concert_id = @lt_booking-concert_id
      INTO TABLE @DATA(lt_cache).
  ENDIF.
  ```

### Q2. 현업 부서에서 프로그램이 너무 느리다고 속도를 개선해 달라고 요청해 왔습니다. 왜 소스 코드를 열어서 튜닝을 짜기 전에 ST05 와 SAT 데이터부터 뽑아야만 하나요?
**A**: **병목 지점을 정확한 수치 팩트로 측정하지 않은 채 '육감(추측)' 만으로 소스를 도려내 고치기 시작하면, 성능은 단 1% 도 나아지지 않은 채 멀쩡하던 다른 전표 정합성만 박살 내어 부수 버그를 양산하기 때문입니다.**
- **성능 측정의 중요성**:
  초보 개발자는 프로그램이 느리면 그냥 "SELECT 문이 많아서 그렇겠지" 혹은 "인덱스가 없어서 그렇겠지" 라고 추측하며, 만만한 20번째 줄 SELECT 를 만지거나 테이블에 무단으로 보조 인덱스를 달아버립니다.
  하지만 정작 3분 지연의 범인은 150번째 줄 내부 테이블 SORT 를 루프 안에서 1만 번 반복 격발시킨 아바 연산 오동작이었습니다. 
  엄한 20번째 줄 SELECT 만 튜닝하느라 전표 결제 정합성 로직을 건드려 버그를 추가로 심고 인덱스 쓰기 오버헤드만 늘리는 참극을 겪습니다.
  **철칙**: 튜닝은 의사의 청진기 진찰과 같습니다. 무조건 **`ST05`** 와 **`SAT`** 를 On/Off 하여 **"DB 조회에 28초 소요(범인은 50행 SELECT), ABAP 로직에 1초 소요"** 라는 수치 팩트 이정표를 쥐고, 점유율 1위인 50행 SELECT 만 조준 사격해야 최소의 소스 수정으로 최대의 성능 정화를 달성합니다.

### Q3. SELECT in LOOP 안티패턴(1+N 번 왕복)이 왜 소량 테스트 데이터를 넣고 돌릴 때는 눈치채지 못할 정도로 아주 빠르게 잘 기동하다가, 운영 서버에 배포되는 순간 시스템 셧다운의 원흉이 되나요?
**A**: **10건 루프일 때는 DB 왕복 10번이 0.01초에 끝나 티가 안 나지만, 실무 운영 데이터 10,000건이 유입되는 순간 DB 왕복 통신망이 10,000번 왕복하며 네트워크 패킷과 포트 트래픽이 기하급수적으로 폭사하기 때문입니다.**
- **1 + N 의 수리적 재앙**:
  `SELECT SINGLE` 은 한 번 격발할 때마다 네트워크 망을 타고 DB 서버로 넘어가서, 디스크 캐시를 뒤진 뒤, 다시 아바 WAS 로 패킷을 돌려보내는 '통신 오버헤드' 가 수반됩니다. 이 왕복 1회가 0.003초 걸린다고 가정해 봅시다.
  - **테스트 (N = 10건)**: `10 * 0.003s = 0.03초`. 눈이 깜빡할 새에 끝나므로 개발자는 내 로직이 무결하다고 착각해 배포 도장을 찍습니다.
  - **운영 (N = 10,000건)**: `10,000 * 0.003s = 30초`. 순수 네트워크 통신 대기 시간만으로 30초를 소모하며 프로그램이 멎어 섭니다.
- **해결**:
  `FOR ALL ENTRIES` 나 `JOIN` 을 사용하여 밖에서 1번만 쿼리를 쏴서 1만 건 데이터를 램 캐시에 촥 실어 나르고, 루프 내부에서는 램에 얹힌 캐시 테이블만 **`BINARY SEARCH`** 로 찾아 엮으면 통신 왕복이 단 **`1회 (0.003초)`** 로 수축 종결됩니다.

### Q4. 프로그램 속도를 고속으로 만들겠다고 SAP 표준 마스터 테이블(예: MARA, KNA1)에 보조 인덱스(Secondary Index)를 남발하면 왜 저장/수정 연산 시 시스템 성능이 통째로 썩어 들어가나요?
**A**: **인덱스는 조회(Select) 속도를 올리기 위해 가설해 두는 '기억용 나무 지도 주소록' 이므로, 인덱스가 늘어날수록 예매를 등록/수정/삭제(CUD)할 때마다 그 무거운 인덱스 지도 수십 개를 DB 엔진이 매번 실시간으로 재정렬하고 다시 그리느라 쓰기 성능이 폭사하기 때문입니다.**
- **인덱스의 쓰기 오버헤드**:
  보조 인덱스를 5개 달아두었다면, `INSERT` 한 줄을 쏠 때마다 DB 엔진은 내부적으로 5개 인덱스 지도를 다 찾아다니며 신규 주소를 추가 기입하느라 CUD 성능이 기하급수적으로 다운됩니다.
  특히 S/4HANA (HANA 인메모리 DB) 장비 하에서는 인메모리 컬럼 기반 압축 스캔 속도가 워낙 빨라, 굳이 보조 인덱스 지도를 만들지 않아도 고속 조회가 가능합니다.
  **철칙**: 보조 인덱스 추가는 절대 최종 수단이어야 하며, **반드시 SQLM(SQL 모니터)운영 데이터를 1주일 이상 긁어보아 '매일 수십만 번 호출되나 극도로 지연되는 필수 조건 필드' 1~2개로 엄선 검증된 경우에만 절제하여 개설**해야 안전이 보장됩니다.

### Q5. 10만 건의 예매 금액을 다 합산하여 전광판에 뿌리고 싶습니다. 왜 ABAP 내부 테이블로 10만 건을 쌩으로 SELECT 해와서 루프를 돌며 더하는 대신 SELECT SUM( ... ) GROUP BY 로 DB 에 집계를 떠넘겨야 하나요?
**A**: **10만 건 로우 데이터가 네트워크 선을 타고 아바 램으로 수송되는 트래픽 오버헤드와 램 소모량을 차단하고, DB 공장 내부에서 1초 만에 합산을 끝내고 정산 '결과 딱 1행' 만 가볍게 넘겨받는 것이 Code Pushdown (코드를 데이터로 위임)의 성능 정석이기 때문입니다.**
- **비교 분석 (Data to Code vs Code to Data)**:
  - **레거시 (Data to Code)**: 10만 건의 seatsocc 데이터를 아바 WAS 램으로 꾸역꾸역 수송합니다. (램 용량 100MB 사용, 전송 속도 10초, 아바 Loop 연산 10만 번 격발).
  - **Modern (Code to Data / Pushdown)**: `SELECT SUM( seatsocc ) FROM sflight INTO @DATA(lv_sum).` DB 단에서 10만 건을 고속 플래시 연산하여 `12,800` 이라는 결과물 단 8바이트 정수 1개만 아바 WAS 로 보냅니다. (램 용량 0.001MB 사용, 전송 속도 0.01초, 아바 Loop 연산 0줄 종결).
  **철칙**: DB 가 정렬, 집계, 필터를 제일 잘하므로, 이 연산들은 아바 Loop 로 가져오지 말고 SQL 의 **`SUM, GROUP BY, AVG`** 등을 활용해 DB 단으로 전적으로 위임해 내려보내야 S/4HANA 인메모리 아키텍처가 활성화됩니다.
