## 3. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. 배치 잡 등록을 위해 SUBMIT zprog VIA JOB jobname NUMBER jobnum 구문을 작성하고 빌드했더니 계속 문법 에러(Syntax Error)가 납니다. 왜 AND RETURN 을 의무적으로 함께 기입해 주어야만 하나요?
**A**: **백그라운드 예약 큐에 작업을 전달하는 주체 프로그램은 예약을 마치자마자 즉시 제어 세션을 돌려받아 자기 다음 줄 소스를 실행(Return)해야만 정상 작동하도록 컴파일러가 문법적으로 통제 제약하기 때문입니다.**
- **동작 원인**:
  `SUBMIT ... VIA JOB` 은 "지금 바로 실행하라" 가 아니라 "새벽 2시 스케줄러에 이 프로그램을 한 단계로 올려두라" 는 예약 전송입니다.
  만약 `AND RETURN` 옵션을 빼버리면, 아바 엔진은 호출한 프로그램을 거기서 통째 강제 중단시키고, 예약된 zprog 가 돌 때까지 영원히 WAS 대기 상태로 고착되는 기괴한 동기적 오작동을 유발하게 됩니다.
  따라서 아바 컴파일러는 예약과 세션 복귀가 온전히 맞물리도록, `VIA JOB` 선언 꼬리에는 반드시 **`AND RETURN`** 옵션을 강제 세트로 엮어 기입하도록 구문 명세서 상에서 차단 제약하고 있습니다.

### Q2. SUBMIT zprog VIA JOB jobname NUMBER jobnum 으로 야간 배치 스케줄러를 등록할 때, 굳이 USER sy-uname 이라는 옵션을 적지 말고 빼버리라고 표준에서 권고하는 진짜 사유는 무엇인가요?
**A**: **USER sy-uname 명시는 불필요한 중복 기입(Redundant)이며, 디버거 상에서 sy-uname 시스템 필드가 오염 변조되거나 릴리즈 결재 과정에서 세션 계정이 뒤틀려 엉뚱한 사람으로 권한이 꼬이는 위험을 낳기 때문입니다.**
- **생략의 원칙**:
  `VIA JOB` 은 기본적으로 옵션을 안 주면 "이 배치를 지금 예약하고 있는 현재 사용자 계정" 으로 백그라운드 구동 권한을 자동 할당해 줍니다. 
  여기에 굳이 `USER sy-uname` 이라고 명시하면, 첫째로 아무런 실익 없는 중복 코드가 되고, 둘째로 아바 디버거를 타고 넘어가거나 다른 TR 릴리즈 잡이 기동될 때 `sy-uname` 변수 값이 꼬여 엉뚱한 Basis 계정이나 임시 백그라운드 관리자 계정으로 권한이 세탁 기입되는 꼬임 버그를 낳습니다.
  **철칙**: 특별히 다른 부서 직원 계정(예: WMS_ADMIN) 권한을 빌려 실행해야 하는 특별한 경우가 아니라면, 내 권한으로 태우는 잡에는 **`USER` 옵션 지정을 통째로 지우고 생략**하는 것이 표준 아바 가이드라인입니다.

### Q3. 신규로 개발한 예매 확인서 프로그램과 예매 DB 테이블을 운영(PRD) 서버로 이송했습니다. 그런데 왜 "ZTABLE 이 존재하지 않습니다" 라는 이송 컴파일 실패 오류가 터지며 시스템이 마비되나요?
**A**: **선행 의존성이 걸려 있는 예매 테이블을 이송 요청서(TR)에 누락한 채 프로그램만 릴리즈했거나, 테이블보다 프로그램을 먼저 임포트하는 이송 역전(Transport Reversal)이 터졌기 때문입니다.**
- **이송의 역전/의존 덤프**:
  운영계 데이터베이스 입장에서는 `ZTICKET` 이라는 테이블 껍데기(물리 공간)가 먼저 생겨나 존재하고 있어야만, 이 테이블을 조회하는 `ZREPORT` 프로그램이 이송될 때 빌드 컴파일을 정상 통과합니다.
  - **원인 A (누락)**: 테이블 ZTICKET 이 내가 올린 이송 기차(TR)에 안 담겨 개발기에 버려진 상태입니다.
  - **원인 B (역전)**: TR 기차는 정상적으로 올렸으나, Basis 팀의 이송 제어실(`STMS`)에서 프로그램 임포트 단추를 1순위로 누르고 테이블 임포트를 2순위로 누르는 바람에, 선행 의존성이 뒤틀려 프로그램 컴파일이 벼락 덤프를 먹고 사망한 것입니다.
  **철칙**: 이송 전 `SE09` 에서 의존 개체 누락 여부를 정밀 체크하고, 임포트 시에는 반드시 테이블 구조체 등 **선행 의존 물리가 먼저 운영을 밟도록** 순서를 칼 수호해야 안전합니다.

### Q4. 야간 배치 잡 settle 프로그램 가동 시 발생하는 오류/성공 로그를 저장하려고 합니다. 왜 임시 ZLOG 테이블을 만드는 것보다 SAP 표준 Application Log (BAL) API 와 SLG1 을 쓰는 것이 백배 안전한가요?
**A**: **임시 Z테이블은 검색, 기간별 자동 삭제, 권한 격리가 안 되어 디스크를 폭사시키지만, 표준 BAL 은 기간별 인덱스 검색(SLG1)과 보관이 자동 지원되며, 개인 식별 정보(PII) 누출을 마스킹하는 보안 필터를 쉽게 심을 수 있기 때문입니다.**
- **Z로그 테이블의 해악**:
  나만의 ZLOG 테이블을 파서 인서트를 때리기 시작하면, 검색 화면도 매번 따로 개발해야 하고, 1년에 수백만 건씩 쌓이는 좀비 로그 데이터를 지워주는 청소 배치 잡도 따로 짜야 합니다. 
  가장 큰 위협은, 에러 원인 문자열에 고객의 "비밀번호, 주민번호" 같은 민감 정보가 마스킹 없이 날것으로 찍혀 데이터베이스에 상주함으로써 보안 감사의 형사 처벌 대상이 되는 것입니다.
- **표준 BAL 의 강점**:
  `BAL_DB_SAVE` 로 기입하면 SAP 표준 `SLG1` 로그 조회실에서 기간별, 객체 카테고리별, 유저별로 즉시 정교하게 인덱스 필터링 조회가 공짜로 지원됩니다. 
  또한, 오래된 로그는 표준 삭제 툴(`BAL_DELETE` / `SARA` 아카이빙)로 3초 만에 청소할 수 있으며, 로그 생성 전 사용자 개인정보를 마스킹(`XXXX`) 필터링하는 아키텍처를 통일되게 구축할 수 있어 보안 규정을 완벽 수호할 수 있습니다.

### Q5. 운영 이송 전 통과해야 하는 정적 품질 검사 도구인 ATC (ABAP Test Cockpit) 가 배포를 강제 기각하고 락을 채우는 치명적인 3대 위협 Findings 는 무엇인가요?
**A**: **1+N 번 DB 왕복으로 서버 리소스를 점유하는 'SELECT in LOOP' (성능), ? 바인딩 없이 문자를 쌩 결합해 덤프/데이터 파괴를 유발하는 'SQL Injection' (보안), 그리고 클라우드 전환을 가로막는 'Non-released API 접근' (클라우드 표준)의 3가지가 핵심 적색 에러 감지 대상입니다.**
- **3대 적색 경보의 의미**:
  1. **SELECT in LOOP**: 루프 내 쿼리를 쏴서 DB 와 네트워크를 수천 번 왕복하며 기계를 기절시키는 최다 빈발 성능 폭사 주범입니다. (FAE 나 JOIN 으로 빼라는 Finding 을 뿜어 차단합니다.)
  2. **SQL Injection**: `ADBC` Native SQL 사용 시 문자열을 덧셈 결합하여 사용자가 입력한 DROP TABLE 등 해킹 폭탄 코드가 DB 에 직접 주사되어 테이블을 파쇄하는 보안 틈새입니다. (? 플레이스홀더 바인딩을 강제합니다.)
  3. **Non-released API Access**: ABAP Cloud 아키텍처 하에서 표준 Released API 가 아닌 내부 Z오브젝트를 무단 직접 찌르는 행위입니다. (Clean Core 정책에 위배되므로 배포를 거절당합니다.)
  이 **3대 ATC Finding 게이트**를 철저히 통과해야 비소로 라이브 운영 서버에 영구 무결한 클린 아바 코드가 안착될 수 있습니다.
