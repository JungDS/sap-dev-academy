## 3. 공식 문서 수동 근거 명세

`abapclass_for_testing.htm` 을 대조하여 ABAP Unit 의 엔진 작동 제약 요건을 철저히 규명했다.

- **ABAP Unit Fixture 메서드 선언 철칙 및 FOR TESTING 배제**:
  단위 테스트 프레임워크가 격리 구동 전후에 인지하는 Fixture 메서드는 반드시 아바 엔진 고유 철자인 **`setup`**, **`teardown`**, **`class_setup`**, **`class_teardown`** 의 소문자/대문자 스펠링을 100% 동일하게 맞춰 구현해야 한다. 또한 이 Fixture 메서드 정의부 뒤에는 **`FOR TESTING` 구문을 절대 덧붙여 선언해서는 안 되며**, 순수 검증 테스트용 메서드에만 `FOR TESTING` 을 붙여야 정상 컴파일 및 실행을 수락받는다는 절대 문법 제약 확인 및 수록 완료.

## 4. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. ABAP Unit 테스트용 클래스 안방에 setup 과 teardown Fixture 메서드를 이쁘게 구현했는데 왜 계속 빌드 컴파일 에러가 나나요? 정의부 끝에 FOR TESTING 을 붙이면 안 되나요?
**A**: **setup 과 teardown 은 값을 검증하는 실제 단위 테스트 코드가 아니라 테스트 환경을 셋팅하는 Fixture 이므로 FOR TESTING 구문을 붙이면 문법 오류가 터지며, 반드시 setup 과 teardown 이라는 고정 스펠링으로 선언해야 합니다.**
- **컴파일 오류 원인**:
  아바 컴파일러는 `FOR TESTING` 지표가 붙은 메서드를 "단위 테스트 검증기" 로 인식하고 독립적으로 깨워 가동하려 듭니다.
  그런데 setup 이나 teardown 은 값을 assert(판정)하는 메서드가 아니라, 테스트 케이스가 돌기 직전과 직후에 사전/사후 조치를 대행해 주는 보조 Fixture 입니다. 여기에 `FOR TESTING` 을 덧붙여 선언해 두면 "보조 장치와 검증기를 혼동했다" 고 판단해 컴파일러가 강하게 빌드를 차단합니다.
- **Fixture 메서드 선언 규칙**:
  ```abap
  " 1. [Fixture 메서드 정의 - FOR TESTING 절대 기입 금지!]
  METHODS setup. 
  METHODS teardown.
  
  " 2. [실제 검증 단위 테스트 메서드 정의 - FOR TESTING 필수 기입!]
  METHODS remaining_calc FOR TESTING.
  ```
또한, 철자를 임의로 `my_setup` 등으로 바꿔 적으면 아바 엔진이 이 녀석이 Fixture 인지 인지하지 못하고 스킵하므로 반드시 **`setup`**, **`teardown`** 정해진 영문 철자를 사수해야 합니다.

### Q2. Singleton 클래스를 설계할 때 CREATE PRIVATE 옵션을 클래스 정의부에 빼먹으면 get_instance 를 이쁘게 만들어 두어도 왜 단일 인스턴스 보장이 박살 나나요?
**A**: **CREATE PRIVATE 이 누락되면 클래스의 인스턴스화 권한이 PUBLIC(기본값)으로 개방되어 외부 프로그램들이 get_instance 를 무시하고 쌩 NEW zcl_config( ) 로 인스턴스를 무한 복제 생성할 수 있게 되기 때문입니다.**
싱글톤의 대전제는 "이 클래스는 시스템 상에 오직 단 1개의 인스턴스만 존재한다" 입니다.
이 규칙을 보장하려면 get_instance 메서드를 만드는 것만으로는 절대 부족합니다.
반드시 클래스 대가리에 **`CREATE PRIVATE`** 옵션을 박아주어야 외부에서 `NEW` 나 `CREATE OBJECT` 로 이 클래스를 임의로 새로 만들려는 시도 자체를 컴파일러가 불법으로 규정해 에러로 때려막아 줍니다.
이 옵션을 빼먹어버리면, get_instance 도 작동하지만 다른 개발자들이 쌩 `NEW` 로 여러 개의 주소 객체들을 메모리에 난발하는 것을 컴파일러가 방어해 주지 않아 싱글톤의 단일성 성벽이 허무하게 무너집니다.

### Q3. MVC 기반 리포트 구조를 짤 때, Model 클래스 내벽 안에 ALV field catalog 속성 셋업이나 MESSAGE '에러' TYPE 'E' 문장을 살짝 적어두면 왜 아키텍처 결합도 붕괴라고 비판받나요?
**A**: **ALV 필드카탈로그와 화면 메시지 팝업은 시각 영역(View)과 흐름 제어(Controller)에 묶인 전용 레이아웃 정보이므로, 데이터/로직 전담 클래스인 Model 안으로 화면 코드가 틈새 오염되어 유입되면 '화면 변경 요구 시 Model 까지 통째 뜯어고쳐야 하는' 결합도 참사가 발생하기 때문입니다.**
Model 의 가장 아름다운 가치는 **"화면이 Fiori Elements 웹 브라우저 화면이든, 클래식 ALV 화면이든, 모바일 앱 화면이든 전혀 알 바 없이, 날것의 비즈니스 계산 결과 데이터만 100% 깔끔하게 돌려주는 순수성"** 에 있습니다.
그런데 Model 클래스 안에 `MESSAGE '...' TYPE 'E'` 와 같은 구문을 하드코딩해 박아두면, 훗날 이 모델을 화면 없는 백엔드 야간 배치 프로그램이나 API 통신용 백엔드로 재사용하려 할 때 화면 에러 팝업 세션이 격발되어 배치가 뻗어버리거나 API 통신 도중 강제 셧다운 덤프가 터지는 장애를 유발합니다.
**철칙**: Model 은 순수한 SQL 과 연산만 리턴하고, 화면을 그리는 속성이나 사용자 경고 격발은 무조건 **View** 와 **Controller** 로 물리 분리해 격리 기재해야 정석입니다.

### Q4. 단위 테스트 Mock(가짜 리포지토리)을 짤 때, 왜 select 쿼리가 들어가는 DB 접근부를 Interface 로 분리하고 생성자 주입(DI)을 해놓아야만 테스트가 가동되는지 직관적으로 설명해 주세요.
**A**: **업무 클래스 내부에 DB 직접 SELECT 가 고정 결합되어 있으면 테스트 실행 시 무조건 실제 데이터베이스 디스크를 조회해야만 하므로, 테스트용 가짜 데이터(Mock)로 갈아 끼워 넣을 틈새 통로가 차단되어 테스트가 불가능해지기 때문입니다.**
- **의존성 주입(DI)이 없을 때 (테스트 불가)**:
  `zcl_booking_manager` 내부에서 DB 쌩 쿼리를 날려 예약을 읽습니다. 
  테스트 1초 전에 동료 개발자가 DB 테이블의 1001번 예약 데이터를 지워버리면, 내 계산 로직은 멀쩡한데 테스트 코드는 "B001 데이터가 DB 에 없습니다" 고 에러를 내며 실패합니다.
- **인터페이스 분리와 생성자 주입 DI 장착 (테스트 가능)**:
  DB 찌르는 행동을 `zif_booking_repo` 인터페이스로 싹둑 잘라내고, 매니저를 생성할 때 `io_repo` 라는 부품 통로를 뚫어 밖에서 넣어주게(DI) 설계합니다.
  이러면 테스트할 때 진짜 DB 를 안 찌르고, **"DB 조회 없이 무조건 메모리 상에서 가짜 100석과 60석 정보만 퉤 뱉어주는 가짜 Mock 칩(ltd_booking_repo)"** 을 팩토리처럼 생성자에 주입해 줄 수 있습니다.
  진짜 DB 가 비어있든, 동료가 테이블을 밀어버리든 무관하게 내 계산 알고리즘(`100 - 60 = 40`)만 순식간에 독립적으로 안심 테스트할 수 있게 해방됩니다.

### Q5. 요금 계산 분기를 정리하기 위해 Strategy Pattern 을 짰는데, Price Strategy 클래스 파일 내벽 안에서 또 CASE type. WHEN VIP. WHEN 일반. 으로 쪼개 요금을 계산하면 왜 전략 패턴의 가치가 완전히 소멸하나요?
**A**: **각 요금 유형별 알고리즘 자체를 독립된 개별 클래스로 캡슐화해 떼어내는 것이 전략 패턴의 정석인데, 전략 클래스 내부에서 또 Case 분기를 치는 것은 전략의 실체 안에 또 다른 전략을 중첩하여 복잡한 사다리 Case 문을 원점으로 롤백시키는 행위이기 때문입니다.**
전략 패턴을 적용하는 이유는 새 요금이 추가될 때 기존 소스 코드를 칼로 비집어 수정하는 위험(OCP 위반)을 없애기 위함입니다.
이를 위해 **`zcl_vip_strategy`** (VIP 요금만 순수 계산), **`zcl_group_strategy`** (단체 요금만 순수 계산) 처럼 하나의 전략 클래스가 오직 단 하나의 알고리즘 방식만 100% 담백하게 캡슐화해야 합니다.
만약 전략 클래스 하나를 만들어두고 그 안에서 또 VIP 인지 단체인지 Case 분기 연산을 자행해 버리면, 얼리버드나 군인 요금이 추가될 때 그 전략 클래스를 비집고 들어가 소스를 또 뜯어고쳐야 하므로 OCP 가 붕괴되어 전략 패턴을 적용한 노력이 헛수고로 돌아갑니다.
**철칙**: **1개 전략 클래스 = 1개 순수 가격 알고리즘** 공식에 맞게 독립 분할해 빚어야 정석입니다.
