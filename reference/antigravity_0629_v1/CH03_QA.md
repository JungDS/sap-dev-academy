## 3. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. DATA 변수를 선언할 때 TYPE zdomain_name 처럼 도메인 명칭을 직접 적으면 왜 컴파일 에러를 뿜으며 빌드가 거절되나요?
**A**: **Domain(도메인)은 단지 물리 스펙만 기술한 가상의 설계 도면일 뿐, 아바 프로그램이 램 메모리에 방을 짓기 위한 '의미 정보(Data Type 명세)' 가 결여되어 있어 컴파일러가 직접 참조를 원천 차단하기 때문입니다.**
- **동작 원인 (Domain 은 실체가 없음)**:
  도메인 `ZDO_STATUS` 는 단순히 "1바이트 문자다" 라는 물리 사양만 담고 있습니다. 
  이것을 변수 타입으로 바로 쓰려 들면, 아바 컴파일러는 **"ZDO_STATUS 는 도메인이지 프로그램 변수를 위한 Data Type 이 아니다"** 라며 빌드를 뱉어냅니다.
  프로그램과 테이블 필드가 바라보고 변수를 지을 수 있는 정식 규격은, 도메인의 기술 사양을 물려받고 그 위에 비즈니스 이름표와 한글/다국어 라벨을 씌워 활성화한 **Data Element (데이터 엘리먼트)** 가 유일합니다. 
  따라서 `DATA gv_status TYPE zde_status.` 형태로 데이터 엘리먼트 명칭을 기입해 주어야만 정식 빌드가 뚫립니다.

### Q2. 도메인(Domain)과 데이터 엘리먼트(Data Element)는 왜 귀찮게 굳이 두 개의 개별 객체 레이어로 분리해서 설계하도록 아바 아키텍처가 제약하나요?
**A**: **동일한 물리 스펙(예: 공항 3자리 코드)을 출발지 공항, 도착지 공항처럼 전혀 다른 비즈니스 의미와 이름표(라벨)를 가진 여러 데이터 엘리먼트들이 상속하여 기술 도면을 100% 재사용(1:N)할 수 있게 격리하기 위함입니다.**
- **관심사 분리 (1:N 상속)**:
  출발 공항(`FROM`)과 도착 공항(`TO`)은 둘 다 똑같은 공항 코드 규격인 `CHAR 3 (3자리 문자)` 를 사용합니다.
  만약 두 계층이 나뉘어 있지 않다면, 우리는 출발 공항용 3자리 문자 타입 하나, 도착 공항용 3자리 문자 타입 하나씩 물리 설계를 중복해서 빚어야 합니다.
  아바 DDIC 는 이를 제거하기 위해 물리 규격은 도메인 `ZDO_AIRPORT` 에 단 한 번만 정의해 굳힙니다. 
  그 위에 비즈니스 의미와 화면에 띄울 한글 라벨만 다르게 칠한 데이터 엘리먼트 `ZDE_FROM_AIRP` (라벨: 출발 공항)와 `ZDE_TO_AIRP` (라벨: 도착 공항)를 개설해 이 도메인을 1:N 으로 공동 상속 배선시킵니다.
  이로 인해 물리 데이터 아키텍처는 단단히 한 곳에서 재사용 제어되면서, 화면의 비즈니스적 텍스트 표현은 유연하게 분리 격리될 수 있는 대단한 유지보수 정합성을 달성합니다.

### Q3. PARAMETERS 에 필수값 입력 옵션인 OBLIGATORY 를 선언했을 때, 사용자가 빈칸으로 실행을 누르면 이를 차단하는 검문 처리는 아바 프로그램 코드의 어느 시점에서 격발되나요?
**A**: **내 아바 프로그램의 비즈니스 로직(START-OF-SELECTION)이 기동되기 전에, 아바 커널이 가설해 둔 1000번 선택 화면(Selection Screen)의 PAI(Process After Input) 이벤트 버퍼 검문 시점에서 즉각 기동되어 차단합니다.**
- **PAI 수문장 기작**:
  `PARAMETERS` 로 인해 만들어진 화면은 일반 화면이 아닌 1000번 표준 선택 화면입니다.
  사용자가 [F8 실행] 단추를 누르는 찰나, 아바 런타임 엔진은 내부 PAI (입력 후 처리) 이벤트를 깨워 화면 버퍼를 스캔합니다.
  `OBLIGATORY` 꼬리표를 쥐고 있는 변수가 비어 있음이 감지되면, 커널은 내 프로그램의 메인 비즈니스 연산 줄을 1자도 기동하지 않고, 즉시 1000번 화면으로 실행 제어권을 롤백 튕겨내며 화면 하단에 **"모든 필수 필드를 입력하십시오 (메시지 00 055)"** 빨간 경고창을 띄워 트랜잭션을 전면 차단 사수합니다.

### Q4. 선택 화면 PARAMETERS 필드에 소문자로 아이디를 입력하고 실행했는데, 프로그램 내부 변수로 값을 읽어보면 왜 자꾸 대문자로 강제 치환되어 들어와 있나요?
**A**: **PARAMETERS 선언 시 LOWER CASE 옵션을 누락하면, 1000번 선택 화면 PAI 버퍼 검문소에서 아바 커널의 입력 필터가 모든 영문 소문자 입력을 강제로 대문자로 치환 오버라이트(Upper-case Conversion)하여 변수에 전달하기 때문입니다.**
- **소문자 훼손 예방 (LOWER CASE)**:
  아바의 선택 화면은 기본적으로 소문자 유입을 차단하고 대문자로 치환 처리하는 구식 필터가 기본 활성화되어 작동합니다. 
  대소문자를 엄격히 구별해야 하는 패스워드나 파일 경로, 소문자가 섞인 사용자 계정을 입력받기 위해서는, 반드시 구문 선언 꼬리에 **`LOWER CASE`** 옵션을 명시 배선해 주어야 합니다. 
  이 옵션이 도킹되어 있어야만, PAI 대문자 치환 필터가 작동하지 않고 우회(Bypass)하여 사용자가 타이핑한 소문자 날것 그대로 변수 공간에 안전하게 적재 보존됩니다.

### Q5. DDIC 도메인에 등록해 둔 고정값(Fixed Values)들과 값 테이블(Value Table) 정보는 실제 DB 테이블 어디에 적재되어 관리되나요?
**A**: **도메인의 헤더 정보(데이터 타입, 길이 등)는 DD01L 테이블에, 도메인 내부 고정값 목록('O'/'C' 등)은 DD07L 테이블에 물리 적재되며, 데이터 엘리먼트 정보는 DD04L 테이블에 분리 저장되어 시스템에 보존됩니다.**
- **도메인 관련 물리 시스템 대장**:
  - **`DD01L` (Domain Header)**: 도메인의 이름, 물리 타입(`DEC`, `CHAR`), 길이, 소수점 자리 등의 메타 속성을 기록하는 헤더 장부 테이블입니다.
  - **`DD07L` (Domain Values)**: 도메인에 빚어둔 고정값 목록과 각 고정값의 번역 설명 텍스트가 인서트 적재되는 값 사전 테이블입니다.
  - **`DD04L` (Data Element Header)**: 데이터 엘리먼트가 참조하는 도메인 배선 정보와 Field Label 길이 정보가 보존되는 엘리먼트 대장 테이블입니다.
  아바 커널은 우리가 SE11 에서 F3 활성화를 완료하는 순간, 이 시스템 테이블들을 긁어서 컴파일용 런타임 캐시 메모리에 적재해 둠으로써 프로그램들이 0.001초 만에 데이터 형식을 찾아 결선될 수 있게 보장합니다.
