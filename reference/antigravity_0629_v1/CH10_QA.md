## 3. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. 서브루틴 파라미터 전달 시 `CHANGING value(cv_result)` 처럼 value 수식어를 얹어 선언하는 것과, value 없이 쌩으로 쓰는 것은 RAM 메모리 정합성 관점에서 어떤 물리적 차이를 낳나요?
**A**: **value 수식어 없는 기본 CHANGING 은 호출자의 원본 메모리 주소를 8바이트 포인터로 직접 결선(Call by Reference)하여 서브루틴 안벽에서 값을 바꾸면 원본이 즉시 동시 수정되지만, `CHANGING value(cv_result)` 는 진입 시 원본을 로컬 복제 버퍼에 복사한 뒤 ENDFORM 에 정상 도달하는 찰나에만 원본으로 역복사(Copy-back)하므로, 중간에 비정상 종료 시 원본이 보호되는 트랜잭션 안전성을 확보합니다.**
- **RAM 물리 기작 차이의 실체**:
  - **`CHANGING cv_result` (쌩 Reference)**: 
    호출자 프로그램 메모리 영역의 원본 변수 주소(예: `0x7FFE1234`)를 FORM 파라미터 cv_result 에 곧바로 결선합니다. cv_result 는 별명 이름표에 불과하며, 서브루틴 안벽에서 `cv_result = 999.` 를 기입하는 그 순간 원본 메모리 셀 `0x7FFE1234` 의 비트열이 즉시 999 로 덮어 써집니다. 
    만약 연산 도중 런타임 에러가 터지면, 원본에는 **연산 중간값(partial update)** 이 박힌 채로 오염되어 데이터 정합성이 파괴됩니다.
  - **`CHANGING value(cv_result)` (Value and Result)**: 
    PERFORM 진입 시, 아바 커널은 원본 값의 전체 바이트를 Call Stack 프레임의 로컬 복제 버퍼에 1:1 사본(Copy-in)으로 복사합니다. FORM 안벽에서는 이 복제 버퍼만 조작합니다. 
    **ENDFORM 에 정상 도달하는 바로 그 찰나**, 로컬 버퍼의 최종 값을 원본 주소로 역복사(Copy-back)합니다. 
    중간에 런타임 에러나 RETURN 에 의한 비정상 복귀가 발생하면, **역복사가 기각되어 원본은 호출 전 값 그대로 보존**됩니다.

### Q2. 아바 프로그램에서 CALL FUNCTION 으로 함수 모듈을 호출할 때, 왜 해당 함수가 소속된 Function Group 전체가 메모리에 통째로 로드되며 어떤 위험이 수반되나요?
**A**: **아바 런타임은 함수 모듈 1개를 호출하는 즉시, 해당 함수가 소속된 Function Group 전체(같은 그룹의 모든 함수 + TOP 인클루드 글로벌 변수 선언부)를 RAM 세션 메모리에 일제히 선제 적재(Full Load)하며, TOP 인클루드에 선언된 글로벌 변수 버퍼는 프로그램 세션 종료 전까지 해제되지 않고 같은 그룹 내 모든 함수 간에 공유 접근이 가능하므로, 이전 함수 호출의 찌꺼기 데이터가 잔존하여 다음 함수 호출 결과를 오염시킬 수 있습니다.**
- **Function Group RAM 적재의 물리 기작**:
  함수 모듈은 독립 실행 파일이 아니라, Function Group 이라는 묶음 패키지(프로그램 유형 F)의 하위 인클루드입니다. 
  CALL FUNCTION 을 격발하면, 아바 커널은 해당 함수의 소속 그룹 전체 소스(TOP 글로벌 선언부 + 모든 함수 코드)를 Load Pool 이라는 메모리 영역에 통째로 적재합니다. 
  이로 인해 같은 그룹의 다른 함수를 이후에 호출할 때는 재적재가 불필요해 고속 작동하지만, TOP 인클루드의 글로벌 변수 `gv_cache` 에 첫 번째 함수 호출 결과가 남아 있는 채로 두 번째 함수가 이를 읽어 사용하면 의도치 않은 **교차 오염(Cross-contamination)** 이 발생합니다. 
  **방어**: 함수 진입부에서 `CLEAR gv_cache.` 를 선제 실행하여 이전 호출의 찌꺼기를 소독합니다.

### Q3. FORM ... ENDFORM (서브루틴)은 옛날 아바 코드에 엄청나게 많이 쓰여 있는데, 왜 최신 S/4HANA 신규 개발이나 Clean ABAP 표준 스펙에서는 이를 전면 폐기 지양하라고 하나요?
**A**: **FORM/PERFORM 은 타입 안전성 결여(Untyped 전달 허용), 프로그램 안벽 격리로 인한 전역 재사용 불가, 전역 변수 무차별 침범 유도, 그리고 현대 OO 설계 패러다임(캡슐화·상속·인터페이스)과의 호환 불가 때문에 Clean ABAP 표준에서 전면 폐기(Deprecated) 통제된 레거시 문법입니다.**
- **폐기 사유의 기술적 실체**:
  1. **타입 안전성 결여**: FORM 파라미터에 TYPE 을 생략하면 어떤 타입의 데이터든 넘어와 런타임에서야 터지는 Type Mismatch 에러를 유발합니다. 클래스 메서드는 파라미터 타입을 컴파일 시점에 강제 체크합니다.
  2. **프로그램 안벽 격리**: FORM 은 정의된 프로그램 안에서만 호출 가능합니다. 다른 프로그램에서 PERFORM ... IN PROGRAM 으로 외부 호출할 수는 있으나, 이는 비표준 안티패턴이며 런타임 에러 위험이 높습니다.
  3. **전역 변수 침범**: FORM 안벽에서 전역 변수를 직접 읽고 쓸 수 있어, 암시적 부작용(Side Effect)이 디버깅을 극도로 어렵게 만듭니다.
  4. **OO 비호환**: 캡슐화(PUBLIC/PRIVATE), 상속(INHERITING), 인터페이스(INTERFACE) 등 현대 설계 패턴을 전혀 지원하지 못합니다.

### Q4. 함수 모듈(SE37)의 EXPORTING 과 IMPORTING 파라미터의 방향이, 우리가 함수 호출 소스를 짤 때의 방향과 완전히 뒤바뀐 것처럼 느껴져 헷갈리는데 어떻게 1초 만에 방향을 잡나요?
**A**: **"방향은 항상 나(호출자)의 시선이다" 라는 한 문장으로 1초 만에 잡습니다. 내가 함수한테 값을 보낸다 = EXPORTING(나 → 함수). 내가 함수에게서 결과를 받는다 = IMPORTING(함수 → 나).**
- **방향 혼동 해소의 기억법**:
  SE37 함수 정의 화면에서 보면, 함수 입장의 IMPORTING (함수가 받는 값)이 호출자 입장에서는 EXPORTING (내가 보내는 값)이 됩니다. 
  이 뒤바뀜이 100% 혼동의 원인인데, **아바의 CALL FUNCTION 구문은 호출자(내 프로그램)의 시선으로 적도록 설계되어 있습니다.**
  - **`EXPORTING iv_amount = 1000`**: "나"가 함수에게 1000 을 "내보낸다(Export)".
  - **`IMPORTING ev_result = lv_out`**: "나"가 함수로부터 결과를 "가져온다(Import)".
  - **비유**: 택배를 보내는 사람(나 = EXPORTING)과 받는 사람(나 = IMPORTING). 항상 **나** 중심으로 생각합니다.

### Q5. 서브루틴이나 함수 내벽에서 지역변수로 선언한 DATA 변수와 STATICS 변수는, 호출이 끝나고 복귀할 때 RAM 상에서 수명 주기(Life-cycle)가 각각 어떻게 다른가요?
**A**: **DATA 지역변수는 FORM/METHOD 진입 시 Call Stack 프레임에 할당되었다가 ENDFORM/ENDMETHOD 복귀 시 프레임과 함께 즉시 파괴(Deallocate)되어 다음 호출 시 초기값(0, 공백)으로 재탄생하지만, STATICS 변수는 프로그램 글로벌 힙 메모리의 특별한 슬롯에 배치되어 호출이 끝나도 파괴되지 않고 프로그램 세션 종료 시까지 값이 영속 보존됩니다.**
- **수명 주기 RAM 기작의 실체**:
  - **`DATA lv_tmp TYPE i.` (일반 지역 변수)**:
    PERFORM/METHOD 호출 시 Call Stack 프레임이 새로 푸시(Push)되며, 이 프레임 안에 lv_tmp 의 메모리 슬롯이 할당됩니다. ENDFORM 에서 프레임이 팝(Pop)되면 슬롯이 통째로 해제됩니다. 
    **다음 호출 시에는 완전히 새로운 프레임이 생성되므로 lv_tmp 는 항상 초기값 0 으로 시작합니다.** 이전 호출에서 넣어둔 값은 소멸되어 접근 불가합니다.
  - **`STATICS sv_counter TYPE i.` (정적 지역 변수)**:
    최초 호출 시 프로그램 글로벌 힙 메모리에 별도 슬롯이 한 번 할당됩니다. 이 슬롯은 Call Stack 프레임 바깥에 위치하므로, ENDFORM 복귀 시에도 파괴되지 않습니다. 
    **두 번째, 세 번째 호출 시에도 이전에 저장해 둔 값이 그대로 남아 있어**, 호출 횟수 카운터나 누적 합산기 등에 활용됩니다. 프로그램 세션이 종료되어야 비로소 해제됩니다.
