## 3. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. Open SQL 이 특정 데이터베이스 제품군(Oracle, HANA 등)의 전용 SQL 문과 다른 점은 무엇이며, 커널 내부에서 어떻게 실시간 번역되나요?
**A**: **Open SQL 은 특정 DB 에 종속되지 않는 아바 표준 공용 쿼리 인터페이스이며, Application Server 내에 상주하는 Database Interface (DBI) 가 현재 장착된 DB 벤더의 전용 구문(Native SQL)으로 런타임에 실시간 번역(SQL Translation)하여 송출합니다.**
- **벤더 독립과 DBI 의 작동 기작**:
  각 데이터베이스 제조사마다 SQL 의 방언(Native SQL)이 조금씩 다릅니다. 
  만약 소스 코드에 Oracle 전용 쿼리를 그냥 짜뒀다면, 나중에 회사가 HANA DB 로 인프라를 바꿨을 때 수천 개의 소스 코드가 먹통이 되어 폭사합니다.
  아바는 이를 막기 위해 데이터베이스 독립 인터페이스인 **`Open SQL`** 규격을 빚었습니다.
  개발자가 작성한 Open SQL 은 실행되는 찰나에 **`Database Interface (DBI)`** 라는 커널 번역기를 통과합니다. 
  번역기는 뒤에서 도킹 스펙을 읽어, "지금 연결된 녀석은 HANA DB 군" 하고 `SELECT * FROM sflight` 를 HANA 용 Native SQL 로 번역 벼려내어 최종 송출합니다. 
  개발자는 데이터베이스 종류가 바뀌어도 단 1줄의 소스 수정 없이 동일한 비즈니스 로직을 사수할 수 있어 이식성이 극대화됩니다.

### Q2. SELECT ... ENDSELECT 를 사용하여 데이터를 조회하는 것과 INTO TABLE 을 쓰는 것은 백엔드 네트워크 패킷 왕복(Network Round-trip) 관점에서 어떤 치명적인 성능 차이를 만드나요?
**A**: **SELECT...ENDSELECT 는 DB 로부터 레코드를 1행씩 뗏목으로 나르는 Cursor-based Fetch 방식이어서 매 회차 네트워크 왕복 지연 오버헤드를 내지만, INTO TABLE 은 수천 행을 대형 바지선에 한꺼번에 싣고 오는 Array Fetch 방식이어서 왕복 횟수를 단 1회로 통제하기 때문입니다.**
- **네트워크 패킷 왕복의 오버헤드 공식**:
  - **`SELECT ... ENDSELECT` (Cursor-based Fetch)**:
    데이터베이스와 Application Server 는 서로 다른 물리 컴퓨터(서버 장비)로 케이블 네트워크 배선되어 있습니다. 
    1만 건의 데이터를 쌩 `SELECT ... ENDSELECT` 로 조회하면, 시스템은 첫 번째 행을 싣고 바다를 건너 App Server 에 던져 준 뒤, 대기합니다. WRITE 가 끝나면 다시 배를 돌려 DB 로 가 2번째 행을 싣고 옵니다. 
    이 짓을 1만 번 되풀이하며 1만 번의 **네트워크 패킷 왕복 지연 (Network Round-trip Latency - 평균 5ms 내외)** 이 누적되어, 연산 시간이 50초 이상 치솟으며 네트워크 파이프라인을 독차지해 전체 전산망을 굳게 만듭니다.
  - **`INTO TABLE` (Array Fetch / Block Fetch)**:
    DB 엔진이 매칭되는 1만 건 데이터를 메모리에 한 통 패키지로 크게 묶은 뒤, 대형 컨테이너 바지선에 한꺼번에 싣고 바다를 단 **1회**만 건너와 램 내부 테이블에 통째 도킹시킵니다. 
    네트워크 왕복 횟수가 단 1회로 통제되어, 1만 건 연산이 0.1초 만에 찰딱 격발 완료됩니다. 
    **실무 철칙**: 조회 시 `SELECT ... ENDSELECT` 루프 가설은 절대 금기령입니다.

### Q3. INTO CORRESPONDING FIELDS OF TABLE 을 쓰면 컬럼 순서가 달라도 알아서 매핑되어 편리한데, 왜 10만 건 이상 대량 배치 프로그램에서는 사용을 지양해야 하나요?
**A**: **DB 에서 들고 온 10만 건 데이터 컬럼 명세표와, 목적지 구조체의 필드 명세표를 런타임에 1:1로 실시간 대조 스캔(Run-time Name Mapping)하는 무거운 탐색 연산 비용이 추가되어 CPU 리소스를 낭비하기 때문입니다.**
- **이름 대조 스캔 비용의 실체**:
  `INTO CORRESPONDING` 은 아주 스마트하고 편리해 보이지만, 컴퓨터 내부에서는 런타임 엔진이 "DB 에서 긁어온 1번 필드 `CARRID` 와 동일한 이름의 컴포넌트가 내 램 서랍 구조체의 몇 번째 인덱스에 있는지" 실시간 문자 대조(Name Matching Search)를 매번 밟습니다. 
  10만 건의 행을 돌며 매 컬럼마다 이 이름 매치 비교 루프가 기동되므로, CPU 연산 비용이 기하급수적으로 폭증합니다.
  - **대안**: 
    대량 배치 정산 쿼리를 짤 때는, 귀찮더라도 목적지 내부 테이블의 필드 순서(`carrid, connid, price`)와 쌩 `SELECT` 의 Projection 컬럼 순서를 정확히 일치 결선해 두고, CORRESPONDING 옵션을 싹 빼버린 **쌩 `INTO TABLE`** 로 쏘아 올립니다. 
    이 방식은 이름 매칭 스캔이 완전히 배제되어, DB 버퍼의 바이트 뭉치가 램 내부 테이블 Body 로 지연 없이 순간 이동하므로 극적으로 고속 수송됩니다.

### Q4. Open SQL WHERE 절 우변에 ABAP 로컬 변수(Host Variable)를 그대로 적어 조회를 쏠 때, 왜 SQL Injection 같은 전산 공격이 불가능하며 속도도 빨라지나요?
**A**: **DBI 가 로컬 변수 값을 SQL 구문에 직접 결합하지 않고, 물음표 플레이스홀더(? - Bind Variable)로 파싱해 뼈대만 DB 에 먼저 보낸 뒤 실제 값은 메모리 매핑 장치로 격리 쑤셔 넣기 때문입니다.**
- **바인드 변수 (? 플레이스홀더) 배선의 보안/성능 이점**:
  - **SQL Injection 방어**:
    사용자가 웹 인풋 단에 `'KE' OR 1=1` 이라는 해킹 구문을 쑤셔 넣었습니다. 
    DBI 번역기는 쿼리를 `SELECT * FROM sflight WHERE carrid = ?` 라는 투명 뼈대로 변조 격리해 DB 로 쏩니다. 
    이어서 `?` 자리에 얹을 실제 값 `'KE' OR 1=1` 을 단순 데이터 비트로만 던집니다. 
    DB 엔진은 이를 SQL 명령어로 인식하지 못하고, 오직 "carrid 의 값이 `'KE' OR 1=1` 인 글자 장부만 찾아라" 고 안전 인식하여 해킹 명령 실행 자체가 원천 기각 봉쇄됩니다.
  - **Soft Parsing (소프트 파싱) 유도**:
    DB 엔진은 쿼리를 받으면 실행 계획(디스크 스캔 경로)을 생성하는 무거운 'Hard Parsing' 연산을 합니다.
    물음표 `?` 뼈대 쿼리로 날아오면, 다음번 로그인에 값이 'LH' 로 들어와도 뼈대가 같으므로 기존에 메모리(Shared Pool)에 짜둔 최적의 실행 계획 장부를 그대로 집어와 재사용(**`Soft Parsing`**)합니다. 
    매번 쿼리를 번역 해석하는 부하가 사라져 대량의 동시 접속자가 몰려도 DB 서버가 굳지 않고 유연하게 가동됩니다.

### Q5. SELECT * 를 쏘아 전체 컬럼을 가져오는 행위가, 대역폭(Bandwidth) 정체와 메모리 버퍼 낭비 관점에서 왜 비권장되나요?
**A**: **사용하지도 않을 거대한 XML, 가변 텍스트, 이송 로그 등 수십 개의 불필요한 대형 컬럼들이 데이터베이스 디스크 세그먼트로부터 조회되어 네트워크 대역폭(Bandwidth)을 낭비하고 램의 Roll Area 버퍼 공간을 불필요하게 가득 점유하기 때문입니다.**
- **Selective Projection (필드 지정)의 성능 차이**:
  `SELECT *` 는 테이블 안방의 모든 데이터를 싹 쓸어 모읍니다. 
  비즈니스 소스 상에서는 오직 `carrid` (항공사코드) 하나만 대조하려 하는데, 옆방의 좌석 현황, 운항 메모, 기종 코드 등 쓰지 않는 40개 필드까지 다 램으로 패키징 배송됩니다.
  100만 건 조회 시 `SELECT *` 를 쏘면, 쓸데없는 데이터 수백 메가바이트(MB)가 네트워크 라인을 점유해 수송 트래픽 정체(Bandwidth Jam)를 유발하고, WAS 의 메모리 캐시를 가득 갉아먹어 JVM/ABAP 세션 아웃 뇌사로 동시 접속자를 길바닥으로 내쫓습니다.
  **철칙**: 조회 시에는 오직 내가 비즈니스 계산에 사용할 알짜 컬럼들만 명시적으로 나열하는 **`Selective Projection`** 을 수호해야 시스템이 영구 무결 생존합니다.
