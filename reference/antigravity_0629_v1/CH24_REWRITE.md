# CH24_REWRITE - 실무 데이터 변경과 트랜잭션 제어 v1

> 목적: `content/abap/CH24`의 5개 레슨을 "IT 비전공자 입문자가 왜 필요한지부터 실습 확인까지 직관적으로 이해하고 따라갈 수 있는 완성 강의자료" 수준으로 재작성한 기준 원고다. 이 문서는 아직 `content/abap` 원본을 덮어쓴 결과가 아니라, 재작업 판정 이후 실제 반영 전 검수 가능한 v1 기준안이다.

## CH24 전체 설계

CH24의 한 문장 목표는 "ABAP 데이터베이스 영구 쓰기를 관장하는 DML 4형제(INSERT/UPDATE/MODIFY/DELETE)의 용도 기준과 감사 스탬프, SAP 표준 테이블의 BAPI 호출 의무화 규율, 트랜잭션의 원자성을 보존하는 COMMIT/ROLLBACK WORK, DB/SAP LUW 차이와 비동기/동기(COMMIT WORK AND WAIT) 업데이트 태스크(IN UPDATE TASK) 및 SM13 큐 연동, 일부 변경 실패 시의 재처리 멱등성 수립, 그리고 대량 연산 시의 리소스 폭사 방지용 패키지(Package) 단위 커밋까지 실무 트랜잭션의 정교한 안정성을 안전하게 구축한다"이다.

IT 비전공자 입문자는 대량 `INSERT FROM TABLE @itab` 사용 시 중복 키에 의해 터지는 `DBSQL_DUPLICATE_KEY_ERROR` 런타임 덤프 폭사를 방지하는 `ACCEPTING DUPLICATE KEYS` 옵션을 빼먹어 장애를 내고, `CALL FUNCTION ... IN UPDATE TASK` 지연 업데이트 모듈 본문 소스 안에서 `COMMIT WORK` 이나 `ROLLBACK WORK` 을 적어 `COMMIT_IN_POSTING` 대형 덤프를 터트린다.
또한, `SELECT ... ENDSELECT` 와 같이 데이터베이스 커서(Cursor)를 열고 루프를 도는 중간에 `COMMIT WORK` 을 때려 커서 단선 폐쇄 덤프로 시스템을 폭사시키며, 대량 패키지 커밋 루프를 다 돌고 마지막 줄 외곽에 최종 잔량 커밋 처리를 생략하여 4999건의 미커밋 데이터 유실 사고를 낸다.
따라서 본 챕터는 다음과 같은 단계적 해소 장치로 설계를 배치한다.

1. **DML 4형제 확립**: INSERT(신규), UPDATE(수정), MODIFY(upsert), DELETE(삭제) 의 물리적 작동 조건 및 감사 스탬프(`sy-uname`/`sy-datum`/`sy-uzeit`) Stamp 가이드 장착.
2. **표준 테이블 BAPI 규율**: `SFLIGHT` 같은 표준 테이블에 직접 DML 찌르기를 엄금하고, 데이터 무결성 보호를 위해 무조건 표준 BAPI 및 `BAPI_TRANSACTION_COMMIT` 래퍼를 타야 하는 아키텍처 규칙 고수.
3. **대량 INSERT 덤프 방어**: `FROM TABLE` 대량 삽입 시 중복 키로 인한 폭사를 우회하고 에러 핸들링을 접수하기 위해 **`ACCEPTING DUPLICATE KEYS`** 옵션을 필수로 덧붙이는 안전 가이드 탑재.
4. **원자성(Atomicity) 사수**: 헤더와 항목 중 하나만 저장되는 반쪽 저장을 격퇴하기 위해, 모든 DML 뒤의 `sy-subrc` 결과를 1:1 수집 누적하여 최종 `COMMIT WORK` 또는 `ROLLBACK WORK` 분기를 치는 안전 펜스 구축.
5. **동기/비동기 저장 격리**: 백엔드 지연 비동기 처리와, 저장이 디스크에 완수될 때까지 다음 아바 코드 진행을 묶어두는 **`COMMIT WORK AND WAIT`** 동기 가이드 격리.
6. **SAP LUW 지연 등록**: DB 에 즉시 쓰지 않고 SM13 대기 큐에 쌓았다가 커밋 시점에 기동하는 **`CALL FUNCTION ... IN UPDATE TASK`** 타이밍 규명.
7. **업데이트 모듈 내 커밋 절대 금지**: UPDATE TASK 모듈 내벽 내부에서는 독자적인 트랜잭션 경계를 흔드는 `COMMIT/ROLLBACK` 기재가 100% 금지되는 물리 제약 수립.
8. **Cursor 단선 방어**: 데이터베이스 커서가 열려 상주하는 `SELECT ... ENDSELECT` 내벽 루프 안에서는 `COMMIT WORK` 을 절대 불어넣을 수 없음을 규명.
9. **패키지 커밋 (1천~1만)**: 수십만 건 연산 시 롤백 세그먼트와 락 폭사를 예방하기 위해 카운터를 물려 패키지 단위로 쪼개어 커밋하고, **루프 아랫줄 외곽에 최종 잔량 커밋 플래그 가드**를 보존하는 일괄 튜닝.

## CH24-L01 - INSERT / UPDATE / MODIFY / DELETE 실무 기준

### 왜 필요한가

우리가 이전 셀렉트 쿼리 챕터에서 배웠던 `SELECT` 문장들은 데이터베이스 디스크에 저장되어 있는 데이터를 메모리에 로드해 **"읽기(Read)"** 로 보여주는 역할만 수행했다.
하지만 실제 현업에서 가동되는 비즈니스 프로그램의 50% 이상은 결국 데이터를 변경하는 작업이다.
- 신규 예약 고객이 돈을 내어 예약 행을 **새로 저장(Insert)** 한다.
- 회차 잔여석 현황 정원이나 예약 상태를 **수정(Update)** 한다.
- 취소된 예약 내역이나 테스트 데이터를 **삭제(Delete)** 한다.
만약 이 4가지 데이터 쓰기 행위의 차이를 명확히 구분하지 않고 맘대로 쿼리를 섞어 쓰게 되면, 이미 존재하는 기존 데이터의 기본키 레코드를 무단으로 덮어써서 데이터를 파괴하거나, 없는 데이터를 수정하려다 조용히 무시되는 쿼리 누수 장애를 맞이한다.

**데이터베이스 테이블에 대고 직접 영구적인 행 변경을 가하는 쓰기 4형제 DML 의 용도 경계를 명확히 획득하고, 누가 언제 데이터를 바꿨는지 감사 로그를 박으며, 대량 중복 덤프를 안전하게 우회하는 기술**이 필요하다. 그것이 **DML 실무 기준** 의 장착이다.

### 무엇인가

#### 1. DML 4형제의 물리 작동 공식 명세
- **`INSERT`**: 완전히 새로운 데이터를 DB 테이블에 최초 삽입한다. 만약 테이블 내에 **이미 동일한 기본키(Key)를 가진 행이 존재하면 삽입을 거부하고 실패(`sy-subrc = 4`) 처리**한다.
- **`UPDATE`**: 이미 존재하는 행의 특정 컬럼 값을 수정한다. 만약 테이블 내에 **해당 키를 지닌 대상이 없으면 아무 일도 하지 않고 실패(`sy-subrc = 4`) 처리**한다.
- **`MODIFY` (Upsert)**: 키가 존재하지 않으면 새로 `INSERT` 하고, 이미 존재하면 알아서 `UPDATE` 덮어쓰기를 가동하는 만능 퓨전 명령어다.
- **`DELETE`**: 키와 일치하는 대상 행을 DB 테이블에서 영구 삭제한다.

#### ⚠️ [ 대량 INSERT FROM TABLE 시 중복 키로 터지는 런타임 덤프 예방 명세]
- *실무 일괄 적재 프로그램에서 개발자들이 가장 빈번하게 겪는 비정상 셧다운 함정이다.*
- 단건 작업인 `INSERT zbooking FROM @ls_booking.` 은 중복 키 충돌 시 덤프 없이 `sy-subrc = 4` 로 떨어지며 프로그램이 정상 구동을 유지한다.
- **오동작**: 반면 내부 테이블을 통째로 쏟아 붓는 **`INSERT zbooking FROM TABLE @lt_booking.` 대량 DML 은, 100건 중 단 1건이라도 중복 키가 섞여 충돌하면 `sy-subrc` 를 세우지도 않고 즉각 `DBSQL_DUPLICATE_KEY_ERROR` 런타임 에러 덤프를 터트려 장비를 강제 다운시킨다.**
- **방어선 (ACCEPTING DUPLICATE KEYS)**: 대량 인서트 쿼리를 칠 때는 중복 키 충돌에 대비해, 반드시 구문 끝에 **`ACCEPTING DUPLICATE KEYS`** 옵션을 덧붙여야 한다. 이 옵션이 기재되면 충돌 행은 삽입하지 않고 안전하게 스킵 처리하며, 최종 성공한 행 개수만큼 `sy-dbcnt` 를 올리고 `sy-subrc = 4` 에러 분기를 반환하여 덤프 폭사를 완벽히 우회해 준다.

#### 2. 감사 필드 Stamp (누가, 언제)
- "이 데이터 레코드를 도대체 어떤 로그인 사용자가 언제 썼는지" 영구 추적하기 위해, `sy-uname`(사용자 ID), `sy-datum`(오늘 날짜), `sy-uzeit`(현재 시각) 시스템 변수를 구조체에 밀착 적재해 DML 을 실행하는 감사 로깅 룰이다.

### 어떻게 확인하는가

대량 INSERT 문에 DUPLICATE KEYS 가드를 올리고 sy- 변수로 스탬프를 찍는 소스를 검증한다.

```abap
REPORT zch24_l01_dml.

DATA: ls_booking TYPE zbooking,
      lt_booking TYPE STANDARD TABLE OF zbooking.

START-OF-SELECTION.
  " 1. [감사 필드 Stamp 적재]
  ls_booking-booking_id = 'B00001'.
  ls_booking-created_by = sy-uname. " 로그인 ID 스탬프
  ls_booking-created_on = sy-datum. " 오늘 날짜 스탬프
  ls_booking-created_at = sy-uzeit. " 현재 시각 스탬프
  
  APPEND ls_booking TO lt_booking.

  " 2. [★ 대량 인서트 덤프 방어선 작동]
  INSERT zbooking FROM TABLE @lt_booking ACCEPTING DUPLICATE KEYS.
  
  " 3. [영향 행수 검문]
  IF sy-subrc = 0.
    WRITE: / sy-dbcnt, '건의 예약이 무사히 적재되었습니다.'.
  ELSEIF sy-subrc = 4.
    WRITE: / '일부 중복된 키를 지닌 예약 행은 누락 스킵되었습니다. 영향 행 수:', sy-dbcnt.
  ENDIF.
```

#### 체험/시뮬레이터 설계 (DML 플레이그라운드)
- **프로세스 플로우**:
  1. 학습자가 [이미 booking_id = 'B001' 이 들어있는 ZBOOKING DB]를 본다.
  2. [INSERT B001] 단일 연산 카드를 꽂아 기동한다. sy-subrc 가 4로 조용히 떨어지는 것을 대조한다.
  3. 이번에는 [INSERT TABLE (B001 1건 포함 100건)] 카드를 꽂는다. [ACCEPTING DUPLICATE KEYS OFF] 인 상태로 실행하자, 벼락 번개가 치며 "DBSQL_DUPLICATE_KEY_ERROR 덤프 폭사!" 경보등이 울린다.
  4. [ACCEPTING DUPLICATE KEYS ON] 스위치를 올린다. 번개가 걷히고, 충돌 행은 걸러진 채 99건만 안전하게 들어가며 sy-subrc = 4 분기로 수습되는 모습을 감상한다.
- **상태 및 데이터**:
  - `UPDATE zbooking SET status = 'C' WHERE 조건절을 빼고 기동한 상태` -> 런타임 결과: `Warning! All rows in zbooking modified to status 'C'` 적색 경고 하이라이트.
- **피드백**: 대량 INSERT 는 덤프 방어 옵션이 무조건 수반되어야 하며, UPDATE/DELETE 시 WHERE 조건 누락은 대참사를 낳음을 이해한다.

### 실수/주의

- **SCARR, SFLIGHT 같은 SAP 표준 테이블에 INSERT, UPDATE 직접 DML 찌르기**:
  - **이것은 SAP 의 코어 데이터 정합성을 통째로 조용히 망가뜨리는 최악의 파괴 행위다.**
  - 표준 테이블은 뒷단에 복잡한 전표 동기화와 번호 채번 로직이 묶여 있어 직접 쓰면 데이터가 꼬이므로, **표준 테이블 변경은 무조건 BAPI 표준 API 를 경유해야 하며, 직접 DML 은 내가 직접 빚은 CBO 테이블(`ZBOOKING` 등)에만 좁게 기입**해야 한다.

### 정리

- **`INSERT`** 는 중복 키 존재 시 실패(`sy-subrc = 4`)하고, **`MODIFY`** 는 덮어쓴다.
- 대량 **`INSERT FROM TABLE`** 은 중복 키 충돌 시 덤프가 나므로 **`ACCEPTING DUPLICATE KEYS`** 로 방어한다.
- 모든 쓰기 레코드에는 **`sy-uname`**, **`sy-datum`** 감사 변수를 도장 찍듯 stamp 박는다.
- 표준 테이블 직접 DML 은 절대 금지하며 오직 **BAPI** 만 통과시킨다.

## CH24-L02 - COMMIT WORK / ROLLBACK WORK

### 왜 필요한가

DML 쓰기 문장들을 잘 박아두었다.
그런데 이번에는 업무 트랜잭션의 '전부 아니면 전무(All or Nothing)' 규칙이 장벽이다.
" 예약을 새로 추가하면서, 예매 헤더(`ZBOOKING`) 테이블에 1건 삽입하고, 예매 상세 항목(`ZBOOKING_ITEM`) 테이블에 3건의 좌석 상세 로우를 삽입하는 소스를 실행했다.
그런데 불행히도 헤더 삽입은 성공했는데, 상세 항목 삽입은 기본키 에러로 실패해 멈춰 섰다."
DML 은 실행 직후 DB 디스크에 즉시 확정되는 성격을 띠게 기재하면, **헤더만 존재하고 항목 상세 데이터는 공중 분해되어 실종된 '깨진 좀비 데이터(Orphan Record)'** 가 DB 에 그대로 흉하게 남아 데이터 정합성이 붕괴된다.

**헤더와 항목이 모두 100% 무사 성공했을 때만 DB 디스크에 영구 도장을 찍어 확정(COMMIT)하고, 단 1건이라도 실패해 어긋나면 실행한 모든 삽입 예약을 통째 취소해 직전 상태로 백지화(ROLLBACK)하는 트랜잭션 안전 제어 기술**이 필요하다. 그것이 **COMMIT WORK 와 ROLLBACK WORK** 의 장착이다.

### 무엇인가

#### 1. 원자성 (Atomicity) 의 정의
- 업무적으로 한 몸으로 움직여야 하는 연산 묶음(예매 헤더 + 상세 항목)은 **전체가 모두 완벽하게 성공하거나, 아니면 애초에 아무것도 안 한 것처럼 깨끗이 취소**되어야만 데이터 일관성이 사수된다는 트랜잭션의 철칙이다.

#### 2. 트랜잭션 제어 명령어 (DML 예약과 확정)
- **DML 문장(INSERT 등)은 실행 즉시 DB 디스크에 영구 기록되는 것이 아니라, 임시 트랜잭션 메모리 공간에 '예약 대기' 상태로 얹혀진다.**
- **`COMMIT WORK`**: 예약 대기 중인 모든 DML 가공 내역을 DB 디스크에 영구 확정 저장하고 락을 푼다.
- **`ROLLBACK WORK`**: 예약 대기 중인 변경을 싹 지우고 깨끗이 백지화 원점으로 되돌린다.

#### ⚠️ [ SELECT ... ENDSELECT 커서 루프 내 COMMIT/ROLLBACK 전면 금지 명세]
- *아바 코딩 시 가장 생각 없이 날렸다가 런타임 덤프 폭사를 맞는 중요 제약 한계다.*
- DB 커서(Cursor)를 열어두고 한 행씩 순회 처리하는 `SELECT ... ENDSELECT` 루프 안방에서는 절대로 `COMMIT WORK` 이나 `ROLLBACK WORK` 을 실행해선 안 된다.
- **이유**: **커밋이나 롤백 지시가 격발되는 순간, 데이터베이스 커서 세션 포트가 강제로 단선 폐쇄(Closed) 처리되므로, 루프가 다음 행을 읽으려 손을 뻗는 찰나에 갈 곳을 잃어 `CURSOR_ALREADY_CLOSED` 런타임 덤프를 터트리고 폭사하기 때문이다.**

### 어떻게 확인하는가

헤더와 항목의 sy-subrc 결과를 끝까지 모아 누적한 뒤, 분기 판정해 커밋/롤백을 집행하는 소스를 검증한다.

```abap
REPORT zch24_l02_commit.

DATA: ls_head TYPE zbooking,
      lt_item TYPE STANDARD TABLE OF zbooking_item,
      lv_err  TYPE abap_bool VALUE abap_false.

START-OF-SELECTION.
  " 1. [헤더 DML 예약 가동]
  INSERT zbooking FROM @ls_head.
  IF sy-subrc <> 0.
    lv_err = abap_true. " 실패 시 에러 플래그 적재!
  ENDIF.

  " 2. [항목 DML 예약 가동]
  INSERT zbooking_item FROM TABLE @lt_item.
  IF sy-subrc <> 0.
    lv_err = abap_true. " 실패 시 에러 플래그 적재!
  ENDIF.

  " 3. [원자성 분기 판정 철칙]
  IF lv_err = abap_false.
    " 2대 연산이 모두 완벽히 성공했을 때만 디스크 영구 확정!
    COMMIT WORK.
    WRITE: / '예매 정보가 영구 저장되었습니다.'.
  ELSE.
    " 단 1개라도 삐끗했다면 예약 변경을 통째 백지화 취소!
    ROLLBACK WORK.
    WRITE: / '오류 감지: 데이터가 롤백 취소되었습니다 (반쪽 저장 격퇴).'.
  ENDIF.
```

#### 체험/시뮬레이터 설계 (원자성 시뮬레이터)
- **프로세스 플로우**:
  1. 학습자가 [1단계: 헤더 저장 성공 (초록불)]과 [2단계: 항목 저장 실패 (빨간불)]가 어긋난 상황판을 본다.
  2. [COMMIT WORK] 단추를 누른다. 헤더만 DB 상자에 들어가 박히고 상세 항목은 텅 비어 있는 흉측한 [반쪽 좀비 데이터]가 생성되는 비주얼을 확인한다.
  3. [뒤로 가기]를 누르고, 이번에는 [ROLLBACK WORK] 단추를 탭한다. 헤더 데이터 조각마저 파랗게 증발해 버리며 DB 가 깨끗한 원점으로 복구 완료되는 원자성 모션을 감상한다.
- **상태 및 데이터**:
  - `SELECT ... ENDSELECT 루프 내부에 COMMIT WORK 을 강제 기입해 둔 상태` -> 런타임 상태: `CURSOR_ALREADY_CLOSED Crash Dump` 하이라이트.
- **피드백**: 트랜잭션 제어는 성공 확률의 누적이 완료된 최외곽 안전지대에서 한 번만 엄숙하게 집행해야 함을 배운다.

### 실수/주의

- **매 쿼리 한 문장 INSERT 뒤에 바로 COMMIT WORK 을 습관적으로 난사**:
  - `INSERT 헤더. COMMIT. INSERT 항목. COMMIT.` 
  - 이따위 파편 코딩을 자행하면 헤더는 이미 커밋 확정되어 통제 불능인데 뒤이어 돌린 항목 인서트가 실패했을 때, 앞의 헤더를 도로 롤백해 빼낼 수 없어 원자성이 완전히 파괴된다.
  - **커밋은 반드시 업무 묶음(LUW) 단위 전체의 결과가 수집된 뒤에만 딱 1번 기재해야 한다.**

### 정리

- DML 은 데이터 변경 **예약**일 뿐이며, **`COMMIT WORK`** 이 박혀야 디스크에 써진다.
- **`ROLLBACK WORK`** 은 임시 트랜잭션 메모리에 걸린 대기 변경을 통째 백지화 소멸시킨다.
- 데이터베이스 커서 루프(`SELECT ... ENDSELECT`) 내의 커밋은 **커서 단선 덤프**를 낳으므로 엄금한다.

## CH24-L03 - DB LUW와 SAP LUW 차이

### 왜 필요한가

원자성 커밋 제어까지 마스터했다.
그런데 이번에는 사용자가 여러 단계 화면(공연 선택 -> 회차 선택 -> 좌석 번호 기입 -> 최종 결제 저장)을 거치며 거래를 진행하는 다단계 업무의 트랜잭션 보존이 문제다.
- 데이터베이스 고유 트랜잭션인 'DB LUW' 는 화면이 깜빡이거나 사용자가 3초만 자리를 비우면 세션 타임아웃 락이 풀려 자동으로 커밋/롤백되어 변경 데이터가 유실된다.
- 사용자가 1번 화면부터 3번 화면까지 입력하는 긴 업무 세션 전체(예약 헤더 생성부터 최종 승인까지)를 하나의 거대한 원자성 트랜잭션으로 통째 묶어서 가동하고 싶다.
이 긴 시간 동안 DB 단독 락(Database Lock)을 물고 대기하면, 다른 수백 명의 동시 접속 고객들이 전부 락 대기에 걸려 화면이 굳어버리고 전체 시스템 마비 대란이 난다.

**긴 업무 화면의 동선 동안에는 데이터를 DB 디스크에 직접 쓰지 않고 메모리 '지연 대기 큐(Update Task)' 에 안전하게 예약 보관해 두었다가, 최종 단계에서 [저장]을 누르는 그 딱 1초의 찰나에만 DB 세션을 가동해 한꺼번에 일괄 처리하고 락을 풀어내는 기법**이 필요하다. 그것이 **DB LUW 와 SAP LUW 의 격리**이다.

### 무엇인가

#### 1. DB LUW 와 SAP LUW 의 아키텍처적 격리 명세
- **DB LUW (Logical Unit of Work)**: 데이터베이스 엔진 레벨에서 보장하는 매우 짧고 기술적인 트랜잭션 단위다. DML 문 기동부터 `COMMIT WORK` 이 찌르는 순간 락이 풀리고 닫힌다.
- **SAP LUW**: 사용자가 여러 화면과 다단계 입력 동선을 밟아 기동하는 **길고 비즈니스적인 업무 트랜잭션 단위**다.

#### 2. Update Module (IN UPDATE TASK 지연 등록기)
- **`CALL FUNCTION '함수명' IN UPDATE TASK`**: 함수를 지금 즉시 가동해 DB 에 DML 을 쏘는 것이 아니다.
- **원리**: **"나중에 사용자가 최종 COMMIT WORK 단추를 찌를 때, 그 시점에 비동기로 한꺼번에 이 함수를 꺼내어 DB LUW 로 묶어 실행해 달라" 고 지연 예약(Queueing)만 걸어두는 도킹 기법이다.** (따라서 이 구문 직후 DB 테이블을 아무리 select 해봤자 아직 데이터는 존재하지 않으며, `sy-subrc` 도 성공 여부를 가리키지 못한다.)
- **SM13 (업데이트 대기 큐 모니터)**: 예약 등록된 지연 업데이트 타스크들이 커밋을 기다리며 잠들어 있는 대기 관리자 트랜잭션이다.

#### ⚠️ [ UPDATE TASK 함수 내벽 내부에서 COMMIT WORK 절대 금지 제약]
- *업데이트 지연 실행 시 시스템을 즉각 폭사시키는 최악의 문법 충돌 함정이다.*
- **지연 가동을 예약받아 실행 중인 Update Module (`IN UPDATE TASK`) 코드 내벽 내부에서는 절대로 `COMMIT WORK` 이나 `ROLLBACK WORK` 문장을 적어 넣으면 안 된다.**
- **이유**: 업데이트 세션 자체가 이미 `COMMIT WORK` 에 의해 격발되어 돌고 있는 특수 세션 포스팅 구간이므로, 이 안에서 또 커밋 경계를 흔들면 무한 커밋 루프 모순에 빠져 시스템은 즉각 **`COMMIT_IN_POSTING`** 런타임 덤프를 터트리고 폭사하기 때문이다.

### 어떻게 확인하는가

IN UPDATE TASK 로 함수를 지연 등록하고 외부 메인 세션에서 커밋을 때려 격발하는 소스 규격을 검증한다.

```abap
REPORT zch24_l03_luw.

START-OF-SELECTION.
  DATA(ls_booking) = VALUE zbooking( booking_id = 'B00009' customer = '정훈영' ).

  " 1. [지연 업데이트 예약 등록]
  " (이 단계에서는 zbooking 테이블에 데이터가 아직 적재되지 않음!)
  " (호출 직후 sy-subrc 값은 무의미함!)
  CALL FUNCTION 'Z_SAVE_BOOKING_TASK' IN UPDATE TASK
    EXPORTING
      is_booking = ls_booking.

  WRITE: / '업데이트 태스크에 예약 등록 완료 (SM13 대기 상태).'.

  " 2. [최종 격발 스위치]
  " 이 문장이 실행되는 찰나에 비소로 Z_SAVE_BOOKING_TASK 함수가 비동기로 깨어나 
  " DB에 물리 INSERT 를 박고 1개의 짧은 DB LUW 로 묶어 커밋 완수!
  COMMIT WORK.
  
  WRITE: / '최종 확정 완료.'.
```

#### 체험/시뮬레이터 설계 (SAP LUW 타임라인)
- **프로세스 플로우**:
  1. 학습자가 [1번 화면: 정보 입력] -> [2번 화면: 결제] 동선을 밟는다.
  2. [IN UPDATE TASK 호출] 레버를 내린다. 데이터가 [SM13 대기 큐 상자]로 촥 빨려 들어가 잠들고, 실제 DB 테이블 상자는 여전히 비어있는 상태를 대조한다.
  3. [COMMIT WORK] 스위치를 올린다. 대기 큐 상자의 뚜껑이 열리며 데이터가 DB 테이블로 쿵 떨어져 영구 안착되는 타임라인 흐름을 감상한다.
- **상태 및 데이터**:
  - `Update 함수 내부 소스에 COMMIT WORK 을 적어 빌드한 상태` -> 런타임 결과: `COMMIT_IN_POSTING Crash Dump` 적색 전광판 발생.
- **피드백**: SAP LUW 는 긴 업무의 일관성을 사수하기 위한 지연 예약 메커니즘이며, 업데이트 함수 내 커밋은 덤프 폭사 지대임을 인지시킨다.

### 실수/주의

- **IN UPDATE TASK 지연 호출을 남발하고 최종 COMMIT WORK 호출을 누락**:
  - 함수를 예약만 등록해 두고 메인 프로그램 끝자락에서 `COMMIT WORK` 전원을 켜주지 않고 세션을 종료해 버리면, SM13 대기 큐에 잠들어 있던 모든 예약 내역은 DB 구경도 못 해보고 그대로 증발 소멸한다.
  - **지연 등록을 얹었다면 반드시 세션 마침표인 `COMMIT WORK` 이 격발되도록 보장해야 한다.**

### 정리

- **`DB LUW`** 는 DB 엔진 레벨의 짧은 락 단위이고, **`SAP LUW`** 는 긴 다단계 업무 단위다.
- **`IN UPDATE TASK`** 는 지금 실행하지 않고 커밋 시점에 비동기로 돌릴 함수를 예약한다.
- 지연 실행되는 업데이트 함수 내부에서는 **`COMMIT WORK` 기재가 문법적으로 전면 금지**된다.

## CH24-L04 - 오류 로그와 재처리 구조

### 왜 필요한가

LUW 제어까지 마스터했다.
그런데 이번에는 대량 변경 연산 시 마주하는 일부 예외 실패 처리가 엉망이다.
"엑셀 예매 내역 100건을 루프를 돌며 일괄 `INSERT` 쳤는데, 그중 98건은 성공했고 단 2건만 기존 예매 번호와 중복되어 `sy-subrc = 4` 로 튕겨 나갔다.
그런데 이 일부 실패한 2건의 구체적인 예매 번호와 실패 사유를 어디에도 기록하지 않고 그냥 프로그램을 흘려보냈다."
돈을 낸 고객의 예약 2건이 증발했는데, 시스템 로그 어디에도 "누가 왜 누락되었는지" 흔적이 없어 현업 사용자는 데이터 실종을 호소하고 개발자는 원인 추적 불가능의 미궁에 빠진다.

**루프 순회 중 일부 DML 실패가 터지는 즉시, 해당 실패 레코드의 Key 값과 구체적인 에러 정황(sy-subrc 값 등)을 '전용 오류 테이블' 에 안전하게 격리 수집해 두고, 원인을 보정한 뒤 '실패한 2건만 골라서' 안전하게 재실행(Re-process) 처리하는 기술**이 필요하다. 그것이 **오류 로그와 재처리 멱등성** 의 설계이다.

### 무엇인가

#### 1. 변경 오류 로깅 (Error Logging) 의 규칙
- 대량 변경 중 `sy-subrc` 가 0이 아닌 실패 로우를 만나면, 프로그램을 멈추거나 덤프내지 않고, **`실패 Key + 실패 원인 텍스트`** 를 구조체로 결합해 오류 수집 internal table에 모은 뒤, 오류 테이블에 인서트해 보존하는 추적 가이드다.

#### 2. 재처리 패턴 (Re-process Pattern)
- 오류가 터졌을 때 처음부터 100건을 무식하게 다시 돌리는 것이 아니라, **오류 테이블에 박제된 그 실패한 2건만 끄집어내어 기동**하는 스마트 시작점 튜닝이다.

#### ⚠️ [ 재처리 시 중복 키 에러를 재방어하는 멱등성(Idempotency) 설계]
- *재처리 로직을 설계할 때 가장 놓치기 쉬운 트랜잭션 롤백 루프 함정이다.*
- 오류 2건 중 1건은 고쳤는데, 1건은 여전히 중복 키 상태다. 
- **오동작**: 아무 생각 없이 재처리 프로그램을 `INSERT` 로 짜두면, 이미 성공해서 들어가 있는 98건 영역이나 덜 고친 1건 영역에 대고 다시 인서트를 때리다가 또다시 중복 키로 튕겨 나가는 도돌이표 오류 루프를 겪는다.
- **방어선 (멱등성 확보)**: 같은 처리를 두 번, 백 번 실행해도 언제나 안전하게 동일한 최종 상태를 보장하도록, **재처리 쿼리는 `INSERT` 대신 기존 행을 알아서 덮어쓰는 `MODIFY` (Upsert) 문으로 설계하거나, 실행 전 중복 키 존재 여부를 사전에 검문하는 가드 로직을 물려두어야 물리적으로 안전**하다.

### 어떻게 확인하는가

실패 로우를 오류 테이블에 수집하고 MODIFY 로 멱등성을 확보해 재처리하는 소스를 검증한다.

```abap
REPORT zch24_l04_reprocess.

DATA: gt_booking TYPE STANDARD TABLE OF zbooking,
      " 오류 수집용 전용 테이블 선언 (Key + 사유)
      BEGIN OF ls_err,
        booking_id TYPE zbooking-booking_id,
        err_msg    TYPE string,
      END OF ls_err,
      gt_err_log LIKE STANDARD TABLE OF ls_err.

START-OF-SELECTION.
  " [ 1단계: 일괄 예매 가동 및 일부 실패 수집 ]
  LOOP AT gt_booking INTO DATA(ls_book).
    INSERT zbooking FROM @ls_book.
    
    IF sy-subrc <> 0.
      " 실패 발생 시 조용히 흘려보내지 말고 키와 사유를 박제 수집!
      APPEND VALUE #( 
        booking_id = ls_book-booking_id
        err_msg    = |INSERT 실패: sy-subrc={ sy-subrc } / 시간: { sy-uzeit }|
      ) TO gt_err_log.
    ENDIF.
  ENDLOOP.
  
  " 2단계: [재처리 모듈 가동 가정 - 멱등성 확보 철칙]
  " 재처리 쿼리는 중복 키로 폭사하지 않도록 MODIFY 로 안전 덮어쓰기 가동!
  LOOP AT gt_err_log INTO ls_err.
    READ TABLE gt_booking INTO DATA(ls_retry) WITH KEY booking_id = ls_err-booking_id.
    IF sy-subrc = 0.
      " MODIFY 는 이미 키가 있으면 UPDATE, 없으면 INSERT 치므로 두 번 돌려도 안전!
      MODIFY zbooking FROM @ls_retry.
    ENDIF.
  ENDLOOP.
```

#### 체험/시뮬레이터 설계 (재처리 시뮬레이터)
- **프로세스 플로우**:
  1. 학습자가 [5건의 예매 요청]을 보낸다. 3건 성공(초록), 2건 중복키 실패(빨강)가 뜬다.
  2. [오류 테이블 수집 단추]를 누른다. 빨간 실패 레코드 2건만 촥 복사되어 [오류 로그 상자]로 격리 포장 적재된다.
  3. 학습자가 오류 값을 이쁘게 수정 보정한다.
  4. [재처리 실행]을 누르자, 이미 성공한 3건 영역은 건드리지 않고, 수정된 2건만 MODIFY 쿼리를 타고 DB 로 부드럽게 들어가 합류 완료되는 모습을 감상한다.
- **상태 및 데이터**:
  - `오류 로그에 key 값을 누락한 채 메시지만 덩그러니 남긴 상태` -> 런타임 결과: `Cannot identify which row failed. Reprocess impossible` 하이라이트.
- **피드백**: 실패 로깅은 재처리의 길잡이이며, 멱등성을 지닌 쿼리(`MODIFY` 등)로 엮어야 안전함을 확인한다.

### 실수/주의

- **일부 실패가 터졌는데도 sys-subrc 검문 없이 그냥 COMMIT WORK 을 과감하게 찌르기**:
  - 변경이 실패해 레코드가 날아갔는데도 에러 수집 없이 커밋을 때려버리면 시스템은 "모두 다 성공해서 정상 포스팅 완료되었다" 고 화면에 구라를 쳐버려 나중에 실종 데이터를 찾아 헤매는 재앙이 난다.
  - **반드시 실패가 누적 수집되었을 때는 커밋을 전면 차단하고 롤백을 때린 뒤, 오류 로그를 남겨 재처리를 도모해야 한다.**

### 정리

- 대량 변경 중의 DML 실패는 반드시 **키와 구체적 실패 사유**를 오류 로그에 남긴다.
- **`멱등성`** 은 몇 번을 재실행해도 최종 결과가 한결같이 안전하게 떨어지는 안전 성질이다.
- 재처리 쿼리는 중복 키 덤프 안전지대인 **`MODIFY`** 문이나 사전 중복 검사를 쓴다.

## CH24-L05 - 대량 변경 시 Package 처리

### 왜 필요한가

일부 실패의 재처리까지 마스터했다.
그런데 이번에는 수십만 건에 이르는 대량 적재 연산 시 백엔드 하드웨어 메모리가 터져 나가는 임계값에 도달한다.
" 매일 밤 연동되는 콘서트 예매 히스토리 50만 건 데이터를 internal table에 담아서 한 번에 DB 로 인서트(`MODIFY FROM TABLE`)했다.
그 순간, 아바 세션이 5초 동안 굳어버리더니 **'TSV_TNEW_PAGE_ALLOC_FAILED'** 메모리 할당 폭사 덤프나 DB 롤백 세그먼트 로그 풀(Full) 에러를 뿜으며 서버 전체가 마비 다운되었다."
커밋을 때리기 전까지 수십만 건의 변경 대기 내역이 메모리 롤백 공간과 DB 락 테이블에 빽빽하게 쌓여 해제되지 못해, 시스템의 숨통을 조이고 락 용량을 초과해 터져버린 것이다.

**수십만 건의 대량 데이터를 5천 건이나 1만 건 정도의 적절한 '한 봉지 패키지(Package)' 단위로 나누어 루프를 돌리고, 봉지가 찰 때마다 주기적으로 커밋(COMMIT WORK)을 때려 쌓인 락 메모리 부하를 시원하게 비워내는 분할 트랜잭션 기술**이 필요하다. 그것이 **Package 처리** (패키지 분할 커밋) 이다.

### 무엇인가

#### 1. 패키지 단위 처리 (Package Commit) 의 기하학적 메커니즘
- 수십만 건 전체를 하나의 트랜잭션 버퍼에 다 우겨넣지 않고, 5,000건 루프 카운터가 충족될 때마다 즉시 **`COMMIT WORK`** 을 기동해 DB 디스크에 확정 이송을 끝마치고, 점유 중이던 메모리와 락 테이블을 깨끗이 반환해 비우는 분할 세션 관리다.

#### ⚠️ [ 루프 아랫줄 외곽에 최종 잔량 커밋 처리 누락 트랩 명세]
- *대량 패키지 코딩 시 100% 저지르는 치명적 데이터 증발 유실 함정이다.*
- "5,000건 단위로 카운터를 쳐서 루프 안방에서 커밋을 때렸다. 전체 데이터가 404,900건이다."
- **오동작**: 5000의 배수인 400,000건까지는 루프 내부에서 정상 커밋되어 들어간다. 하지만 **루프를 빠져나온 직후, 마지막 12번 라인 외곽에 최종 잔량 커밋을 누락해 두면, 나머지 잔량인 4,900건은 커밋 스위치를 만나지 못해 메모리 공중에 붕 떠 있다가 프로그램 세션이 종료되는 순간 몽땅 증발 유실**되어 사라져 버린다.
- **방어선**: 루프 밖 하단에 반드시 **`COMMIT WORK` 문장을 1번 더 적거나**, 대기 잔량이 있음을 추적하는 플래그(`lv_pending`)를 두어 찌꺼기 1건까지 완전히 털어 내보내야 무결하다.

#### 2. 원자성 과 재시작성의 트레이드오프 (Trade-off)
- 패키지 커밋은 부분 성공을 허용하고 오류 난 패키지만 재처리하면 되는 **'대량 마스터 적재 업무'** 에 매우 적합하다.
- 반면, **"1건이라도 실패하면 50만 건 전체가 삽입 취소되어야 한다"** 는 철저한 원자성 업무에는 패키지 커밋을 엮으면 안 된다. 이미 앞단 40만 건이 커밋 완료되어 디스크에 못 박혔으므로 뒤에서 실패해도 전체 롤백이 원천 불가능해지기 때문이다.

### 어떻게 확인하는가

루프 내 5천 건 분기 커밋을 셋업하고, 루프 외곽에 잔량 최종 커밋 가드를 심는 소스를 검증한다.

```abap
REPORT zch24_l05_package.

DATA: gt_huge_data TYPE STANDARD TABLE OF zbooking,
      lv_cnt       TYPE i VALUE 0,
      lv_pending   TYPE abap_bool VALUE abap_false.

START-OF-SELECTION.
  LOOP AT gt_huge_data INTO DATA(ls_data).
    MODIFY zbooking FROM @ls_data.
    
    lv_cnt += 1.
    lv_pending = abap_true. " 저장 대기 변경이 존재함을 마크!

    " 1. [루프 내 패키지 커밋 분기] 5,000건마다 버퍼 비우기!
    IF lv_cnt >= 5000.
      COMMIT WORK.
      CLEAR lv_cnt.
      lv_pending = abap_false. " 커밋 완수로 대기 변경 해제!
    ENDIF.
  ENDLOOP.

  " 2. [★ 최종 잔량 커밋 가드 철칙]
  " 루프 밖 외곽에서 커밋되지 않고 남은 찌꺼기 4,900건을 털어내 확정!
  IF lv_pending = abap_true.
    COMMIT WORK.
    WRITE: / '나머지 최종 잔량 분량까지 완전 커밋 확정되었습니다.'.
  ENDIF.
```

#### 체험/시뮬레이터 설계 (패키지 커밋 시각화)
- **프로세스 플로우**:
  1. 학습자가 [50만 건 대량 연산]을 앞두고 [패키지 크기 = 단일(전체)]을 세팅해 가동한다.
  2. 빨간색 [미커밋 롤백 버퍼 게이지]가 임계치 끝까지 쭉쭉 차오르며 "자원 폭발! 덤프 쾅!" 폭사 애니메이션을 본다.
  3. [패키지 크기 = 5,000건] 으로 변경한다.
  4. 가동을 누르자, 게이지가 5000까지 차오르면 핑~ 하며 COMMIT 이 울리고 바닥으로 부드럽게 초기화되어 다시 비워지는 톱니바퀴 주기 패턴 시각 피드백을 감상한다.
- **상태 및 데이터**:
  - `루프 밖 외곽 최종 COMMIT WORK 을 생략한 상태` -> 런타임 결과: `4,900 rows of remainder data lost` 적색 유실 경보 발생.
- **피드백**: 대량 처리는 자원 톱니 안정성(주기적 커밋)과 외곽 잔량 수거가 핵심임을 확인한다.

### 실수/주의

- **패키지 커밋 단위를 너무 잘게(예: 10건 단위) 쪼개서 커밋을 난발**:
  - 패키지 크기가 너무 작으면 디스크 쓰기 헤더 세션이 수만 번 오락가락하는 **'커밋 오버헤드(Overhead)'** 가 극에 달해, 5분이면 끝날 배치 처리가 5시간이 지나도 끝나지 않고 멈춰 굳어버린다.
  - **패키지 크기는 메모리 스펙과 커밋 비용 속도의 밸런스를 고려해 통상 '1,000건 ~ 10,000건' 사이에서 절충 설계해야 한다.**

### 정리

- 대량 DML 은 메모리 폭사를 막기 위해 **`패키지 단위 COMMIT`** 으로 잘라 집행한다.
- 루프 내부 패키지 분기 후에, **루프 바깥 최하단에 최종 잔량 커밋**을 빼먹지 않는다.
- 원자성이 영구 사수되어야 하는 업무에는 패키지 커밋을 엮지 않고 사전 검증을 활용한다.
