## 3. 입문자용 단골 질문 Q&A (Glossary 보완용)

### Q1. 내부 테이블 변수 gt_data 에 대고 CLEAR 를 치는 것과 FREE 를 치는 것은 램 메모리 반환 성능 관점에서 구체적으로 무엇이 다른가요?
**A**: **CLEAR 는 테이블 내의 모든 행 데이터만 비우고 기존 확보한 메모리 예약 크기(Capacity)는 유지하여 향후 빠른 재사용을 꾀하지만, FREE 는 행을 비움과 동시에 램 점유 영역 자체를 완전히 해제하여 운영체제에 전원 반환합니다.**
- **메모리 소거 동작의 실체**:
  - **`CLEAR gt_data / REFRESH gt_data`**:
    내부 테이블 Body 에 적재되어 있던 수만 건의 레코드는 싹 삭제됩니다. 
    하지만 아바 가상머신은 "이 테이블에 또 데이터가 차오르겠지" 라고 판단해, 그동안 할당해 둔 램의 물리적 예약 용량(Capacity) 자체는 강제로 쥐고(Holding) 있습니다. 
    이로 인해 나중에 다시 `APPEND` 를 칠 때 램을 새로 뜯어내는 오버헤드 없이 고속 적재가 가능해집니다.
  - **`FREE gt_data`**:
    테이블 행 데이터를 소거할 뿐만 아니라, 이 테이블에 배정되어 예약 점유 중이던 **Roll Area 램 메모리 공간 전체를 완전히 해제(De-allocation)** 해 아바 VM 엔진에 환수시킵니다. 
    대용량 테이블 연산(예: 10만 건 정산 마감)이 완전히 끝나서 다시는 해당 테이블을 쓸 일이 없을 때 `FREE` 를 갈겨주어야만, 램 릭(Memory Leak)과 세션 용량 초과로 서버가 뻗는 사고를 원천 방지할 수 있습니다.

### Q2. HASHED TABLE 이 단건 탐색에서 최강의 초고속 O(1) 성능을 내는데도, 왜 INDEX 5 나 sy-tabix 를 써서 번호로 접근하려 하면 런타임 에러나 컴파일 에러가 나나요?
**A**: **Hashed 테이블은 내부 키 값을 기둥 삼아 해시 함수가 즉각 특정 주소(Bucket)를 가리켜 찾아가는 구조여서, 물리적으로 1번 행, 2번 행과 같은 '순서(인덱스 번호)' 개념이 램 상에 아예 존재하지 않기 때문입니다.**
- **인덱스 미배정의 물리적 한계**:
  Standard 와 Sorted 테이블은 메모리에 차례대로 일렬 적재되므로 `INDEX 5` 라고 치면 5번째 칸 주소를 바로 찾을 수 있습니다. 
  반면 Hashed 테이블은 키 데이터(예: `USER01`)를 해시 함수에 집어넣어 튀어나온 난수 주소(예: `0x7FFF32`)에 데이터를 박아 넣습니다. 
  데이터들이 램 곳곳의 해시 주소지에 난립하여 매달려 있으므로 "5번째로 들어온 녀석" 이 누구인지 메모리 포인터 구조상 역추적할 수 없습니다. 
  이진 트리나 해시 맵은 태생적으로 순서가 없는 구조입니다. 
  따라서 `INDEX n` 이나 `sy-tabix` 를 지칭해 억지로 쑤시려 들면, 아바 컴파일러는 "Hashed table cannot be accessed by index" 라며 문법 에러로 빌드를 전면 중단시킵니다.

### Q3. STANDARD 테이블에 BINARY SEARCH 옵션을 얹어 READ TABLE 을 쐈는데, 왜 테이블 내에 데이터가 버젓이 존재함에도 sy-subrc = 4 (못 찾음) 로 누락되나요?
**A**: **BINARY SEARCH (이진 탐색)는 중간 행 값을 찔러 범위 절반씩 날려가며 찾는데, 테이블이 해당 키로 SORT(정렬)되어 있지 않다면 엉뚱한 절반 영역을 버려 데이터가 묻혀 있는 칸을 생 지나쳐 버리기 때문입니다.**
- **이진 분기 쪼개기의 정렬 전제 조건**:
  테이블에 `[5, 1, 9, 3, 7]` 순서대로 정렬 없이 들어 있습니다. 여기서 `3` 을 찾으려 `BINARY SEARCH` 를 기동합니다.
  이진 엔진은 중간값인 3번째 칸의 `9` 를 찌릅니다. 그리고 판정합니다: "내가 찾는 3은 9보다 작네? 그렇다면 9의 오른쪽에 있는 녀석들은 다 9보다 클 테니 안 봐도 되겠다. 왼쪽 절반만 뒤지자!" 하고 9의 우측 영역(`[3, 7]`)을 통째 버립니다. 
  하지만 실재 데이터 `3` 은 우측에 버려진 구역에 묻혀 있었습니다. 결국 3을 찾지 못하고 **`sy-subrc = 4`** 를 뿜어냅니다. 
  이처럼 이진 탐색이 정합성을 발휘하려면 무조건 오름차순으로 완벽 정렬되어 있어야 "3은 9보다 작으니 왼쪽에 무조건 있다" 는 수학적 논리가 성립합니다. 
  **철칙**: Standard 테이블에 `BINARY SEARCH` 를 박을 때는, 바로 윗선에 동일 조회 키로 **`SORT gt_gugu BY dan.`** 을 반드시 도킹 완료해야 합니다.

### Q4. 10만 건 대량 데이터 루프를 돌며 구조체 값을 수정할 때, INTO gs + MODIFY 를 쓰는 것보다 ASSIGNING <fs> 필드 심볼을 쓰는 게 왜 성능적으로 압도적으로 빠른가요?
**A**: **INTO gs 는 매 루프 회차마다 행 데이터를 gs 버퍼로 무겁게 메모리 복사(Copy-by-Value)하고 수정 후 다시 테이블 본체로 카피해야 하지만, ASSIGNING <fs> 는 실제 행 주소 번지수(Pointer Reference)만 찰딱 낚아채 직접 갱신하므로 복사 비용이 완전히 0원이기 때문입니다.**
- **Call-by-Reference 튜닝의 실체**:
  - **`INTO gs_sales` + `MODIFY`**: 
    매 루프마다 테이블 Body 램 영역에서 `gs_sales` 라는 로컬 변수 방으로 100바이트 크기의 데이터를 쌩 복사해 복사본을 채웁니다. 값을 바꾼 뒤, `MODIFY` 문을 치면 `gs_sales` 의 100바이트를 다시 테이블 Body 의 해당 줄에 대고 덮어쓰기 복사합니다. 한 바퀴당 200바이트의 복사 오버헤드가 발생해 10만 건이면 20MB 의 메모리 복사 트래픽이 쏟아져 연산이 지체됩니다.
  - **`ASSIGNING <fs_sales>`**: 
    어떠한 데이터 복사도 일어나지 않습니다. 필드 심볼 `<fs_sales>` 는 단지 **"테이블 Body 의 1번째 행 주소지 자체를 직접 가리키는 포인터 이름표"** 일 뿐입니다. 
    `<fs_sales>-amount = 500.` 을 때리는 그 즉시, 복사본을 거치지 않고 테이블 본체의 메모리 비트가 **다이렉트로 즉각 갱신(Direct In-place Modification)** 됩니다. 
    `MODIFY` 구문을 쏠 필요도 없고, 메모리 복사 트랜잭션이 전혀 없어 루프 실행 속도가 개별 복사보다 최소 5배에서 10배 이상 비약적으로 향상됩니다.

### Q5. AT NEW gv_field 블록 내부에서 디버거를 켜보면, 왜 gv_field 오른쪽의 다른 모든 컴포넌트 필드 값들이 쌩뚱맞은 별표(*)나 공백으로 뭉개져 보이며 읽을 수 없나요?
**A**: **아바 런타임 인터프리터가 컨트롤레벨의 그룹 경계 판단 정합성을 엄격히 수호하기 위해, 검사 기준 필드보다 오른쪽에 위치한 하위 상세 데이터 영역을 임시로 강제 마스킹(Masking)해 가리기 때문입니다.**
- **컨트롤레벨의 메모리 마스킹 규칙**:
  `AT NEW region` 은 region 값이 바뀔 때 딱 1번만 격발됩니다. 
  이때 런타임 엔진은 내부의 상세 아이템명(`item`)이나 금액(`amount`) 같은 하위 상세 정보는 그룹 수준(Header)의 연산에 참여해서는 안 된다는 철학을 가지고 있습니다. 
  그래서 region 우측의 모든 필드를 쌩 **별표(`*` 또는 타입별 초기값)** 로 가로막아 조회를 차단시킵니다. 
  이 제약을 모르고 블록 안에서 `gv_val = gs_sales-amount.` 와 같이 대입을 때리면 쓰레기 값(별표)이 변수에 들어가 정산 계산이 통째 폭사합니다.
  - **해결책**:
    `AT NEW` 문을 열기 바로 윗줄에서, 현재 필드 심볼이나 구조체 데이터를 **`gs_backup = gs_sales.`** 로 복제 백업해 둔 뒤, 블록 내부에서는 이 백업본인 `gs_backup-amount` 를 꺼내 쓰는 우회 선로를 장착해야 안전합니다.
