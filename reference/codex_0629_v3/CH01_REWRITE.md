## 챕터 설계

CH01의 목표는 "ABAP을 설명으로만 아는 상태"에서 "내 손으로 접속하고, 프로그램을 만들고, 활성화하고, 결과를 보고, 보관 위치와 실행 바로가기를 이해한 상태"로 이동시키는 것이다. 아직 변수, 타입, 조건문, 데이터베이스는 배우지 않는다. 학습자는 프로그램이 무엇인지보다 먼저 "내가 쓴 것이 어디에서 실행되고, 화면에는 무엇이 바뀌는지"를 몸으로 알아야 한다.

레슨 흐름은 다음과 같이 유지한다.

1. SAPGUI 로그온과 화면 구성
2. T-code와 SE38 첫 실행
3. 프로그램 구조와 주석
4. WRITE로 문자열 출력
5. WRITE 심화 - 정렬, 폭, 색, 구분선
6. 개발 패키지와 이송요청 입문
7. T-code 생성 기초(SE93)

신규 레슨은 추가하지 않는다. CH01은 이미 "접속 -> 에디터 -> 코드 뼈대 -> 출력 -> 출력 다듬기 -> 보관 -> 실행 바로가기"로 닫힌 한 바퀴를 갖고 있다. 새 레슨을 넣기보다 각 레슨 내부를 완성 강의자료 수준으로 보강하는 것이 맞다.

## CH01-L01 - SAPGUI 로그온과 화면 구성

### 왜 필요한가

개발을 처음 배우는 사람은 보통 "코드를 어디에 쓰는지"부터 막힌다. ABAP은 내 PC의 메모장에 적고 바로 실행하는 언어처럼 느끼면 안 된다. 실제 코드는 SAP 애플리케이션 서버에 저장되고, 실행도 그 서버에서 일어난다. 학습자의 PC에 있는 SAPGUI는 서버 안으로 들어가는 조작 창이다.

그래서 첫 레슨의 목표는 코딩이 아니다. 목표는 "내 PC의 SAPGUI, 접속할 SAP 시스템, 로그인한 Client, 실행할 화면"을 구분하는 것이다. 이 구분이 잡혀야 나중에 "왜 내 프로그램이 저 시스템에는 없지?", "왜 같은 사용자로 로그인했는데 데이터가 다르지?", "왜 한국어 화면이 안 뜨지?" 같은 문제를 스스로 점검할 수 있다.

### 무엇인가

SAPGUI는 SAP 서버에 접속하는 데스크톱 클라이언트다. 웹 브라우저가 웹 서버에 접속하듯, SAPGUI는 SAP 시스템의 화면을 열고 사용자의 키보드·마우스 입력을 서버로 보낸다. 중요한 점은 SAPGUI 자체가 업무 데이터를 가진 것이 아니라는 점이다. 데이터와 프로그램은 서버 쪽에 있다.

로그온 화면의 핵심 입력값은 네 가지다.

- `Client`: 같은 SAP 시스템 안에서 분리된 작업 공간이다. 교육 시스템에서 `100`과 `200`은 서로 다른 방처럼 동작할 수 있다.
- `User`: 사용자 ID다. 권한, 기본 설정, 언어, 개발 가능 여부가 이 ID에 연결된다.
- `Password`: 인증 정보다. 여러 번 틀리면 잠길 수 있다.
- `Language`: 화면 언어다. `KO` 또는 `EN`처럼 입력한다. 시스템에 설치되지 않은 언어를 넣으면 원하는 언어로 뜨지 않을 수 있다.

로그온 후 보이는 첫 화면은 SAP Easy Access다. 여기서 학습자가 반드시 기억할 곳은 메뉴 트리와 명령창이다. 메뉴 트리는 기능을 폴더처럼 찾아가는 길이고, 명령창은 T-code를 직접 입력해 화면으로 바로 이동하는 길이다. 입문자는 메뉴 트리를 보며 SAP이 많은 기능을 가진 시스템이라는 감을 얻고, 실제 개발 실습은 명령창을 중심으로 진행한다.

### 어떻게 확인하는가

로그온이 성공하면 SAP Easy Access 화면이 열리고 화면 상단에 명령창이 보인다. 하단 상태 표시줄이나 시스템 메뉴에서 현재 시스템, Client, 사용자 정보를 확인할 수 있다. 수업에서는 학습자에게 다음 세 가지를 직접 확인하게 한다.

- 내가 입력한 `Client`가 현재 화면의 Client와 일치하는가?
- 명령창에 커서를 둘 수 있는가?
- 화면 언어가 의도한 언어인가?

이 확인이 끝나야 다음 레슨에서 명령창에 `SE38`을 입력하는 행동이 의미를 가진다. 접속 위치를 모르는 상태에서 T-code만 외우면, 나중에 다른 시스템이나 다른 Client에 접속했을 때 같은 실습을 재현하지 못한다.

### 실수와 주의

가장 흔한 실수는 Client를 잘못 입력하는 것이다. 사용자와 비밀번호가 맞는데도 데이터가 다르거나 개발 객체가 보이지 않는다면 먼저 Client를 확인한다. 두 번째 실수는 언어 코드다. 한국어 자료를 보며 실습하는데 SAPGUI가 영어로 뜬다면 `Language`를 `KO`로 넣어 다시 로그인한다. 세 번째는 비밀번호 잠금이다. 교육 시스템에서는 잠금 해제를 직접 할 수 없는 경우가 많으므로 관리자나 강사에게 초기화를 요청해야 한다.

학습자에게 "SAPGUI에 로그인했다"와 "ABAP 프로그램을 실행했다"를 섞지 않게 해야 한다. 지금은 건물 입구를 통과한 것뿐이다. 개발실에 들어가려면 다음 레슨의 T-code가 필요하다.

### 체험형 학습 설계

이 레슨은 코드가 없으므로 화면 구조 체험을 설계한다.

- 다이어그램: `학습자 PC(SAPGUI) -> SAP 애플리케이션 서버 -> Client 100/200` 흐름을 그린다.
- 입력 패널: `Client`, `User`, `Language` 세 칸을 둔다.
- 버튼: `로그온`, `Client 오류로 시도`, `언어 변경`, `처음으로`를 둔다.
- 상태: `입력 전`, `접속 성공`, `Client 오류`, `비밀번호 잠김`, `언어 불일치`를 표시한다.
- 피드백: 성공 시 "명령창을 찾으세요", 실패 시 "가장 먼저 Client와 Language를 확인하세요"처럼 다음 행동을 안내한다.

이 체험의 핵심은 정답 맞히기가 아니라 "로그온 실패도 원인이 여러 가지"라는 감각을 주는 것이다. 비전공자는 오류 메시지를 보면 전체가 망가졌다고 느끼기 쉽다. 여기서는 오류를 분해해서 보는 습관을 만든다.

### 정리

SAPGUI는 서버에 접속하는 창이고, ABAP 개발은 서버 안에서 일어난다. Client는 같은 시스템 안의 분리된 작업 공간이며, 로그온 후에는 SAP Easy Access 화면의 명령창을 찾아야 한다. 다음 레슨에서는 이 명령창에 T-code를 입력해 ABAP Editor로 바로 이동한다.

## CH01-L02 - T-code와 SE38 첫 실행

### 왜 필요한가

SAP Easy Access 메뉴는 방대하다. 메뉴를 매번 펼쳐서 개발 도구를 찾게 하면 학습자는 코딩을 시작하기 전에 길 찾기에서 지친다. SAP 개발자는 화면마다 붙은 짧은 주소인 T-code를 사용한다. 개발자는 "어디에 있는 메뉴인가"보다 "어떤 T-code로 들어가는가"를 먼저 기억한다.

이 레슨의 목적은 `SE38`이라는 ABAP Editor에 들어가 첫 실행형 프로그램을 만들고, 저장·활성화·실행의 기본 루프를 한 번 완료하는 것이다. 이 루프를 모르면 이후 모든 코드 설명이 허공에 뜬다. ABAP에서는 저장만으로 끝나지 않고 활성화해야 실행본이 바뀐다.

### 무엇인가

T-code는 SAP 화면으로 이동하는 짧은 코드다. 명령창에 `SE38`을 입력하면 ABAP Editor로 이동한다. `/nSE38`은 현재 화면을 닫고 이동하고, `/oSE38`은 새 세션으로 연다. `/nex`는 확인 없이 로그오프할 수 있으므로 입문 단계에서는 실수로 입력하지 않도록 주의시킨다.

SE38은 실행형 ABAP 프로그램을 만들고 실행해 보는 가장 단순한 시작점이다. 프로그램 이름은 고객 개발 영역인 `Z` 또는 `Y`로 시작해야 한다. 예제 이름은 `ZHELLO`로 둔다. 이름 규칙을 여기서 엄격히 잡아야 이후 DDIC 객체, T-code, 패키지 이름에서도 SAP 표준 객체와 충돌하지 않는다.

저장할 때는 우선 Local Object를 선택한다. 그러면 객체는 `$TMP` 패키지에 들어간다. `$TMP`는 학습과 개인 실험에는 빠르지만, 다른 시스템으로 이송할 수 없고 팀 공유에도 맞지 않는다. 지금은 속도를 위해 `$TMP`를 쓰고, CH01-L06에서 왜 패키지와 이송요청이 필요한지 다시 회수한다.

### 어떻게 확인하는가

실습은 다음 순서로 확인한다.

1. 명령창에 `SE38`을 입력했을 때 ABAP Editor가 열리는가?
2. Program 칸에 `ZHELLO`를 입력하고 Create를 눌렀을 때 생성 화면으로 넘어가는가?
3. Type을 Executable Program으로 선택했는가?
4. 저장 시 Local Object를 선택했는가?
5. 저장 후 활성화 버튼 또는 `Ctrl+F3`을 눌러 활성 상태가 되었는가?
6. 실행 버튼 또는 `F8`을 눌렀을 때 프로그램이 실행되는가?

아직 본문 코드가 비어 있으면 출력은 없을 수 있다. 그래도 "에디터에 들어가고, 저장하고, 활성화하고, 실행 버튼을 누르는 루프" 자체를 먼저 확인한다. 또한 `F1` 키워드 도움말과 `Ctrl+Space` 자동완성은 여기서 맛보기로 보여준다. 문법을 외우기 전에 도구가 도움을 준다는 사실을 알려야 한다.

### 실수와 주의

프로그램 이름을 `Z`나 `Y`로 시작하지 않으면 이름 범위 오류가 난다. 저장했는데 실행 결과가 바뀌지 않으면 활성화를 빠뜨린 것이다. Local Object 대신 패키지를 선택하면 이송요청 화면이 뜰 수 있는데, 이 레슨에서는 당황하지 말고 Local Object를 선택한다.

초보자에게 특히 중요한 구분은 "저장"과 "활성화"다. 저장은 초안을 보관하는 행동이고, 활성화는 실행 가능한 버전을 갱신하는 행동이다. ABAP 개발 루프는 `작성 -> 저장 -> 활성화 -> 실행`이다.

### 체험형 학습 설계

기존 위젯 `CH01-L02-S01`을 중심 장치로 사용한다.

- 프로세스 플로우: `명령창 -> SE38 -> 프로그램 생성 -> Local Object -> 저장 -> 활성화 -> 실행`.
- 버튼: `엔터`, `생성`, `저장`, `활성화`, `실행`, `처음부터`.
- 상태: `명령창 대기`, `신규 프로그램`, `저장됨`, `비활성`, `활성`, `실행 완료`.
- 데이터: 명령창 입력값 `SE38`, 프로그램명 `ZHELLO`, 저장 위치 `$TMP`.
- 피드백: `Z`/`Y`가 아닌 이름이면 "고객 네임스페이스가 아니다", 활성화 전 실행이면 "저장본과 실행본이 다르다"를 보여준다.

위젯 뒤에는 학습자가 직접 말로 설명하게 한다. "`ZHELLO`를 만들었다"가 아니라 "`SE38`에서 실행형 프로그램을 만들고 `$TMP`에 저장한 뒤 활성화했다"라고 말할 수 있어야 한다.

### 정리

T-code는 SAP 화면의 짧은 주소이고, `SE38`은 첫 ABAP 프로그램을 만들기 좋은 출발점이다. 프로그램 이름은 `Z` 또는 `Y`로 시작하고, 처음에는 `$TMP`에 Local Object로 저장한다. 저장만으로는 부족하며 활성화해야 실행본이 바뀐다. 다음 레슨에서는 프로그램 안에 들어가는 가장 작은 뼈대와 주석 규칙을 배운다.

## CH01-L03 - 프로그램 구조와 주석

### 왜 필요한가

빈 에디터가 열리면 초보자는 "첫 줄에 무엇을 써야 하는가"에서 멈춘다. ABAP 프로그램은 아무 글이나 위에서부터 적는 메모장이 아니다. 실행형 프로그램은 자신이 어떤 종류의 프로그램인지 알려 주는 시작 문장이 필요하고, 각 문장은 끝을 명확히 표시해야 한다. 또한 사람이 읽기 위한 설명은 코드와 구분되어야 한다.

이 레슨은 실행형 프로그램의 표지인 `REPORT`, 문장 끝의 마침표, 주석 두 가지를 다룬다. 이 세 가지는 작은 규칙처럼 보이지만, 초보자의 첫 문법 오류 대부분이 여기에서 나온다.

### 무엇인가

실행형 프로그램은 `REPORT` 문으로 시작한다.

```abap
REPORT zhello.
```

공식 문서 기준으로 `REPORT`는 executable program을 도입하는 프로그램 시작 문장이다. 독립 실행형 프로그램에서는 첫 문장이어야 한다. 여기서는 "이 프로그램은 실행형 리포트이고 이름은 `zhello`다"라고 선언한다고 설명한다.

ABAP 문장은 마침표 `.`로 끝난다. 줄바꿈이 문장의 끝이 아니다. 그래서 다음 코드는 한 문장이다.

```abap
REPORT
  zhello.
```

실제 수업에서는 위처럼 일부러 줄을 나누어 보여 준 뒤, "ABAP은 줄이 아니라 마침표로 문장을 닫는다"를 강조한다.

주석은 컴파일러가 실행하지 않는 설명이다. 공식 문서 기준으로 줄 첫 위치의 `*`는 전체 줄 주석을 만들고, 큰따옴표 `"`는 그 뒤의 내용을 주석으로 만든다.

```abap
* This line explains the program.
REPORT zhello.  " Program name
```

입문 자료는 한국어로 설명하지만, 실제 프로그램 주석은 가능하면 영어로 쓰는 습관을 소개한다. 다국적 운영과 협업에서는 영어 주석이 더 잘 통한다. 단, 이 레슨의 목표는 영어 작문이 아니라 "실행되는 코드와 실행되지 않는 설명을 구분한다"이다.

### 어떻게 확인하는가

확인은 의도적으로 작은 실험으로 진행한다.

1. `REPORT zhello.`만 둔 상태에서 문법 검사를 해 본다.
2. 마침표를 지우고 문법 검사를 해 본다.
3. `*`를 첫 칸에 둔 주석과 한 칸 들여쓴 ` *`를 비교한다.
4. `REPORT zhello. " Program name`처럼 큰따옴표 뒤 설명을 붙여 본다.
5. 저장 후 활성화 상태가 되는지 확인한다.

여기서 출력 결과를 기대하지 않는다. 이 레슨의 확인 대상은 리스트 화면이 아니라 문법 검사와 활성화 상태다. "오류 없이 활성화된다"가 이 레슨의 결과다.

### 실수와 주의

마침표 누락은 가장 흔한 오류다. 한 문장이 닫히지 않으면 다음 줄까지 한 문장으로 해석되어 엉뚱한 위치에서 오류가 날 수 있다. `*` 주석은 반드시 줄의 첫 위치에서 시작해야 한다. 한 칸이라도 앞에 공백이 있으면 전체 줄 주석이 되지 않는다.

키워드는 대소문자를 가리지 않지만, 예제에서는 `REPORT`처럼 키워드를 대문자로 쓰고 프로그램 이름은 소문자로 보여준다. 이는 공식 문서의 표기 관례와도 맞고, 코드 읽기에도 좋다.

활성화는 기존 활성 버전을 덮어쓴다. 입문 단계에서는 "저장은 초안, 활성화는 실행본 갱신"으로 기억시키고, 실제 버전 관리와 릴리즈 롤백은 운영 단계에서 다룬다.

### 체험형 학습 설계

이 레슨에는 코드가 있으므로 별도 "문법 상태 실험기"를 설계한다.

- 에디터 영역: `REPORT zhello.` 예제를 보여준다.
- 토글: `마침표 제거`, `별표를 첫 칸에서 한 칸 뒤로 이동`, `큰따옴표 주석 추가`, `키워드 소문자 변환`.
- 버튼: `문법 검사`, `저장`, `활성화`.
- 상태: `문법 정상`, `마침표 누락`, `전체 줄 주석 아님`, `활성화 완료`.
- 피드백: 오류 위치만 보여주지 말고 "문장 끝을 찾지 못했다", "`*`는 첫 칸일 때만 전체 줄 주석이다"처럼 원인을 설명한다.

이 체험은 코드를 많이 쓰게 하지 않는다. 초보자가 첫 오류에서 좌절하지 않도록 "오류를 만들어 보고 원인을 읽는" 쪽이 목적이다.

### 정리

`REPORT`는 실행형 프로그램의 시작 문장이다. ABAP 문장은 마침표로 끝나며, 줄바꿈은 문장의 끝이 아니다. `*`는 첫 칸에서 전체 줄 주석을 만들고, `"`는 그 뒤를 줄 끝 주석으로 만든다. 다음 레슨에서는 드디어 `WRITE`로 화면에 결과를 출력한다.

## CH01-L04 - WRITE로 문자열 출력

### 왜 필요한가

지금까지는 접속하고, 에디터를 열고, 프로그램의 뼈대를 이해했다. 하지만 학습자는 아직 "내 프로그램이 진짜 돌았다"는 감각을 갖지 못했다. 화면에 결과가 보이지 않으면 코드는 추상적인 글자에 머문다. `WRITE`는 ABAP에서 가장 빨리 결과를 눈으로 확인하게 해 주는 출력문이다.

이 레슨의 목표는 멋진 화면을 만드는 것이 아니다. 목표는 프로그램 실행의 피드백을 얻는 것이다. 내가 쓴 문장이 리스트 화면에 찍히는 순간, 학습자는 작성·활성화·실행 루프의 의미를 몸으로 이해한다.

### 무엇인가

`WRITE`는 데이터 객체나 리터럴의 내용을 현재 리스트에 출력한다. 첫 예제는 변수를 아직 배우지 않았으므로 작은따옴표로 감싼 문자 리터럴을 사용한다.

```abap
REPORT zhello.

WRITE 'Hello, ABAP!'.
```

문자 리터럴은 코드에 직접 적은 문자 값이다. 공식 문서 기준으로 작은따옴표로 감싼 텍스트 필드 리터럴은 문자 리터럴의 한 종류다. 지금은 `'Hello, ABAP!'`처럼 따옴표 안 내용이 그대로 출력된다고 이해하면 충분하다.

줄바꿈은 `/`로 만든다.

```abap
WRITE 'first'.
WRITE / 'second'.
```

`WRITE`는 기본적으로 이전 출력 뒤에 이어 쓴다. `/`를 붙이면 다음 줄 첫 칸부터 출력한다. 별도의 문장으로 줄을 넘기고 싶으면 `NEW-LINE.`도 사용할 수 있다. 다만 CH01에서는 `/`를 주된 줄바꿈 방식으로 훈련하고, `NEW-LINE`은 "다음 출력 위치를 새 줄로 옮기는 문장" 정도로만 둔다.

같은 시작 부분을 반복하는 문장은 체인으로 묶을 수 있다.

```abap
WRITE: / 'A', 'B', 'C'.
```

체인 문법은 콜론 왼쪽의 공통 부분을 한 번 쓰고, 나머지를 콤마로 나열한 뒤 마침표로 닫는 방식이다. 특히 `WRITE /: 'A', 'B'.`는 `WRITE / 'A'.`와 `WRITE / 'B'.`처럼 `/`까지 반복된다는 점을 명확히 보여 준다.

### 어떻게 확인하는가

확인은 기존 위젯 `CH01-L04-S01`로 진행한다.

1. "한 줄 출력" 예제를 실행해 `Hello, ABAP!`이 리스트 영역에 찍히는지 본다.
2. "이어 붙기"와 "줄바꿈" 예제를 번갈아 눌러 `/`의 차이를 본다.
3. 콜론 체인 예제를 실행해 `WRITE:` 뒤의 콤마 구분이 어떻게 여러 출력으로 풀리는지 본다.
4. 따옴표 하나를 지우거나 마침표를 지워 오류 메시지를 확인한다.

강의자는 실행 전 질문을 던진다. "이 코드는 한 줄에 나올까요, 두 줄에 나올까요?" 예측을 먼저 하게 한 뒤 실행 결과를 보게 해야 문법이 기억에 남는다.

### 실수와 주의

작은따옴표를 닫지 않으면 문자열이 끝나지 않는다. 마침표를 빠뜨리면 문장이 닫히지 않는다. `/`를 빠뜨리면 다음 출력이 같은 줄에 붙는다. 콜론 체인에서 `/`의 위치를 잘못 이해하면 줄바꿈 결과가 예상과 달라진다.

또 하나의 주의점은 리스트 화면의 한계다. `WRITE`는 학습과 단순 리포트에는 좋지만, 사용자가 정렬·필터·다운로드를 편하게 하는 표 화면은 아니다. 이 불편은 나중에 ALV를 배울 동기가 된다. CH01에서는 "출력의 첫 도구"로 충분하다.

### 체험형 학습 설계

기존 위젯 `CH01-L04-S01`을 다음 구조로 사용한다.

- 프로세스 플로우: `코드 선택 -> 예측 -> 실행(F8) -> 리스트 확인 -> 오류 실험`.
- 버튼: `한 줄 출력`, `이어 붙기`, `줄바꿈`, `콜론 체인`, `실행`.
- 상태: `실행 전`, `출력 성공`, `문자열 오류`, `마침표 오류`.
- 데이터: 코드 에디터의 현재 소스, 출력 리스트의 문자열 배열, 상태 메시지.
- 피드백: `/`가 있을 때와 없을 때의 차이를 결과 화면에서 줄 단위로 강조한다.

학습 과제는 "같은 결과를 두 방식으로 만들기"다. 예를 들어 `WRITE / 'A'. WRITE / 'B'.`와 `WRITE /: 'A', 'B'.`가 같은 결과를 만드는지 비교하게 한다.

### 정리

`WRITE`는 값을 리스트 화면에 출력한다. 작은따옴표 안의 문자 리터럴은 그대로 출력된다. 줄을 바꾸려면 `/`를 사용하고, 같은 시작 부분은 콜론과 콤마로 체인할 수 있다. 다음 레슨에서는 출력 폭, 정렬, 색, 구분선을 사용해 리스트를 읽기 좋게 다듬는다.

## CH01-L05 - WRITE 심화: 정렬, 폭, 색, 구분선

### 왜 필요한가

출력은 되었지만 보기 좋지는 않다. 숫자는 자릿수가 맞지 않고, 제목과 합계는 본문에 묻히며, 구분선이 없으면 어디부터 어디까지가 한 묶음인지 알아보기 어렵다. 실무 리포트의 첫 요구는 "값이 보인다"가 아니라 "사람이 빠르게 읽을 수 있다"이다.

이 레슨은 `WRITE`를 화면 확인 도구에서 간단한 리스트 구성 도구로 확장한다. 단, CH01은 classic-first 구간이므로 modern syntax를 새로 끌어오지 않는다. 변수 선언과 날짜 예제는 서식 확인을 위한 복사 실행용 선행 사용으로만 둔다.

### 무엇인가

출력 폭과 정렬은 칸을 맞추는 도구다.

```abap
WRITE: /(16) '정훈영'.
WRITE: /(16) 1250000 RIGHT-JUSTIFIED.
WRITE: /(16) 'TOTAL' CENTERED.
```

`(16)`은 출력에 사용할 폭이다. 텍스트는 보통 왼쪽, 숫자는 오른쪽, 제목은 가운데 정렬한다. 공식 문서의 `WRITE` 내부 포맷 옵션에는 `LEFT-JUSTIFIED`, `CENTERED`, `RIGHT-JUSTIFIED`가 포함된다.

색과 강조는 중요한 값을 눈에 띄게 한다.

```abap
WRITE: / '정상' COLOR 5.
WRITE: / '경고' COLOR 6 INTENSIFIED.
WRITE: / '반전' COLOR 4 INVERSE.
```

`COLOR`, `INTENSIFIED`, `INVERSE`는 현재 `WRITE` 출력에 적용할 수 있다. 여러 줄에 같은 형식을 유지하려면 `FORMAT`을 사용할 수 있다.

```abap
FORMAT COLOR 6 INTENSIFIED ON.
WRITE: / '문제 항목'.
WRITE: / '확인 필요'.
FORMAT RESET.
WRITE: / '일반 항목'.
```

중요한 것은 `FORMAT RESET`이다. 색을 켠 뒤 되돌리지 않으면 이후 출력도 계속 영향을 받을 수 있다.

여백과 구분선은 `SKIP`과 `ULINE`으로 만든다.

```abap
WRITE: / 'HEADER'.
ULINE.
SKIP.
WRITE: / 'BODY'.
```

`SKIP`은 리스트 커서를 아래 줄로 이동시켜 빈 줄 효과를 만들고, `ULINE`은 수평선을 출력한다.

숫자와 날짜 서식은 다음 예제를 복사 실행하며 관찰한다. 여기서 `DATA`, `TYPE`, `VALUE`, `sy-datum`은 CH02와 CH04에서 정식으로 배운다. 지금은 "서식 옵션의 결과를 보기 위한 준비 코드"로만 다룬다.

```abap
DATA lv_qty TYPE n LENGTH 4 VALUE '0001'.
DATA lv_amt TYPE p LENGTH 8 DECIMALS 2 VALUE '100.00'.

WRITE: / lv_qty NO-ZERO.
WRITE: / lv_amt DECIMALS 4.
WRITE: / lv_amt CURRENCY 'KRW'.
WRITE: / sy-datum DD/MM/YY.
```

`NO-ZERO`는 앞쪽의 의미 없는 0을 숨긴다. `DECIMALS`는 출력할 소수 자릿수를 조정한다. `CURRENCY`는 통화별 소수 자리 규칙을 반영한다. 날짜 서식은 시스템의 국가별 포맷 설정 영향을 받을 수 있다.

### 어떻게 확인하는가

기존 위젯 `CH01-L05-S01`로 확인한다.

1. 값을 숫자로 바꾸고 폭 슬라이더를 줄여 자릿수 정렬을 본다.
2. `LEFT`, `CENTERED`, `RIGHT`를 번갈아 누르며 같은 값의 위치를 비교한다.
3. 색을 바꾸고 `INTENSIFIED`, `INVERSE`를 켜서 강조 효과를 본다.
4. `ULINE`, `SKIP` 토글로 여백과 구분선의 차이를 본다.
5. 코드 패널과 출력 패널을 나란히 보고 "어떤 옵션이 어느 결과를 만들었는가"를 말로 설명한다.

수업에서는 "예쁘게 만들기"보다 "읽기 좋은 보고서의 기준"을 질문한다. 숫자는 왜 오른쪽 정렬해야 하는가? 합계는 왜 색만이 아니라 위치와 구분선도 필요한가? 이 질문이 CH11의 ALV 필요성으로 이어진다.

### 실수와 주의

폭보다 값이 길면 잘리거나 예상과 다른 표시가 될 수 있다. 숫자를 왼쪽 정렬하면 자릿수 비교가 어렵다. 색을 남발하면 강조의 의미가 사라진다. `FORMAT`을 사용한 뒤 `FORMAT RESET`을 빼먹으면 뒤 출력까지 색이 이어질 수 있다.

`EDIT MASK`와 `WRITE TO`는 이 레슨에서 다루지 않는다. `EDIT MASK`는 정확한 구문과 활용 범위가 CH01의 핵심을 넘어가고, `WRITE TO`는 문자열 처리 맥락에서 CH04-L02가 더 적절한 소유 위치다. CH01-L05의 정식 범위는 `FORMAT`, `COLOR`, `INTENSIFIED`, `INVERSE`, `ULINE`, `SKIP`, `NO-ZERO`, `DECIMALS`, `CURRENCY`, 날짜/시간의 기본 출력 포맷 맛보기로 제한한다.

### 체험형 학습 설계

기존 위젯 `CH01-L05-S01`을 다음 구조로 사용한다.

- 프로세스 플로우: `값 선택 -> 폭 선택 -> 정렬 선택 -> 색/강조 선택 -> 구분선/여백 선택 -> 리스트 확인`.
- 버튼과 입력: 예제 칩, 정렬 세그먼트, 폭 슬라이더, 색 선택, `INTENSIFIED`, `INVERSE`, `ULINE`, `SKIP` 체크.
- 상태: `기본 출력`, `폭 부족`, `강조 적용`, `구분선 적용`, `RESET 필요`.
- 데이터: 선택 값, 폭, 정렬, 색, 토글 상태, 생성된 ABAP 코드, 리스트 결과.
- 피드백: 숫자 정렬이 바뀔 때 열 기준선을 표시하고, `FORMAT`이 이어질 수 있음을 상태 배지로 알려 준다.

과제는 "금액 리스트 세 줄 만들기"다. 이름은 왼쪽, 금액은 오른쪽, 합계는 구분선 아래 강조 색으로 표시하게 한다. 변수와 내부 테이블은 아직 배우지 않았으므로 값은 리터럴과 단순 숫자로 직접 적는다.

### 정리

`WRITE`는 단순 출력뿐 아니라 폭, 정렬, 색, 강조, 구분선, 빈 줄을 통해 classic 리스트를 읽기 좋게 만들 수 있다. `FORMAT`은 여러 출력에 영향을 줄 수 있으므로 `FORMAT RESET`을 함께 기억한다. 숫자와 날짜 서식 예제의 변수 선언은 아직 정식 학습 대상이 아니며, 지금은 출력 결과 관찰용이다. 다음 레슨에서는 지금 만든 프로그램이 어디에 보관되고 어떻게 다른 시스템으로 이동하는지 배운다.

## CH01-L06 - 개발 패키지와 이송요청 입문

### 왜 필요한가

처음 만든 `ZHELLO`는 `$TMP`에 저장했다. 개인 실험이라면 충분하지만, 실무 개발은 개인 실험으로 끝나지 않는다. 다른 개발자가 보고, 품질 시스템에서 테스트하고, 운영 시스템으로 옮겨야 한다. `$TMP`에 둔 객체는 이 흐름을 탈 수 없다.

이 레슨은 "내 프로그램이 실행된다"에서 "내 프로그램이 관리 가능한 개발 객체가 된다"로 시야를 넓힌다. 초보자는 코드를 쓰는 것만 개발이라고 생각하기 쉽지만, SAP에서는 객체를 어디에 보관하고 어떤 이송요청에 묶는지도 개발의 일부다.

### 무엇인가

개발 패키지는 ABAP Repository 객체를 묶는 단위다. 공식 문서의 package glossary도 패키지가 repository objects를 self-contained units로 캡슐화한다고 설명한다. 입문자에게는 "개발 객체의 폴더"라고 시작하되, 단순 폴더와 다르게 이송 계층과 연결될 수 있다는 점을 강조한다.

`$TMP`는 Local Object를 담는 로컬 패키지로 이해시킨다. 학습과 임시 실험에는 편하지만 이송 대상이 아니다. 반대로 `Z` 또는 `Y`로 시작하는 개발 패키지에 객체를 저장하면 이송요청을 요구받을 수 있다.

이송요청은 변경분을 묶어 다음 시스템으로 보낼 수 있게 하는 관리 단위다. 개발자가 프로그램을 고치면 "어떤 변경이 어떤 요청에 들어갔는가"가 남아야 한다. 그래야 DEV에서 QAS, PRD로 이동할 때 빠진 객체나 순서 문제를 추적할 수 있다.

### 어떻게 확인하는가

확인은 실제 화면 체크리스트로 진행한다.

1. SE38 또는 SE80에서 프로그램의 속성을 열어 Package가 `$TMP`인지 확인한다.
2. `$TMP` 객체는 저장할 때 이송요청을 묻지 않았음을 회상한다.
3. 패키지 객체를 저장하는 화면 예시를 보여 주고, 이송요청 선택 또는 생성 단계가 뜨는 이유를 설명한다.
4. `SE09` 또는 `SE10`에서 내 이송요청 목록을 열면 요청 아래에 객체들이 묶여 보인다는 것을 시연한다.

이 레슨에서는 요청 릴리스, STMS 임포트, 의존성 정리까지 깊게 들어가지 않는다. 그것은 CH35의 운영·배포 관리 범위다. 지금은 "$TMP는 실험, 패키지는 공유와 이송의 출발점"을 확실히 잡는다.

### 실수와 주의

첫 번째 실수는 학습용 실험까지 무조건 패키지에 넣어 이송요청 관리로 복잡하게 만드는 것이다. 교육 초반에는 `$TMP`가 적절하다. 두 번째 실수는 반대로, 팀에 공유해야 할 객체를 계속 `$TMP`에 방치하는 것이다. 세 번째는 Workbench Request와 Customizing Request를 혼동하는 것이다. 프로그램과 DDIC 객체는 보통 Workbench Request에 들어간다.

패키지로 옮기는 것은 단순 저장 위치 변경이 아니라 관리 체계에 들어가는 행동이다. 객체 이름, 패키지 이름, 요청 설명을 대충 쓰면 나중에 찾기 어렵다.

### 체험형 학습 설계

이 레슨은 코드가 없으므로 의사결정 시뮬레이터를 설계한다.

- 상황 카드: `혼자 하루 실험`, `팀 동료가 사용할 공통 프로그램`, `QAS로 테스트 이송`, `운영 반영 예정`.
- 선택 버튼: `$TMP에 저장`, `개발 패키지 선택`, `새 이송요청 생성`, `기존 요청에 추가`.
- 상태: `로컬 실험`, `공유 가능`, `이송 대기`, `운영 배포 범위 아님`.
- 데이터: 객체명 `ZHELLO`, 패키지 `$TMP` 또는 `ZABAP_STUDY`, 요청 번호 예시.
- 피드백: 선택 후 "이 객체는 다른 시스템으로 이동 가능한가?", "누가 변경 이력을 추적할 수 있는가?"를 표시한다.

다이어그램은 `DEV -> QAS -> PRD` 세 칸을 그리고, `$TMP` 객체는 DEV 안에 머무르게 표시한다. 패키지와 이송요청이 붙은 객체만 화살표를 탈 수 있게 표현하면 입문자가 즉시 이해한다.

### 정리

`$TMP`는 빠른 실험에 좋지만 이송과 공유에는 맞지 않는다. 개발 패키지는 Repository 객체를 관리 가능한 단위로 묶고, 이송요청은 변경분을 다음 시스템으로 보낼 수 있게 기록한다. CH01에서는 원리와 판단 기준만 익히고, 릴리스와 운영 배포는 CH35에서 다룬다. 다음 레슨에서는 사용자가 프로그램을 더 쉽게 실행하도록 T-code를 붙인다.

## CH01-L07 - T-code 생성 기초(SE93)

### 왜 필요한가

개발자는 SE38에서 프로그램 이름을 입력하고 실행할 수 있다. 하지만 실제 사용자는 그렇게 일하지 않는다. 사용자는 짧은 코드 하나를 입력했을 때 바로 필요한 기능이 열리기를 기대한다. "SE38을 열고 프로그램명을 입력한 뒤 F8을 누르세요"는 개발자에게는 가능하지만 사용자에게는 불편하고 위험하다.

이 레슨의 목적은 내가 만든 실행형 프로그램에 T-code를 붙이는 것이다. 그러면 명령창에 `ZHELLO` 같은 코드만 입력해 프로그램을 실행할 수 있다.

### 무엇인가

T-code는 프로그램 실행 또는 화면 호출에 연결되는 짧은 이름이다. 공식 문서의 transaction code glossary는 트랜잭션 코드가 ABAP 프로그램의 dynpro, 다른 트랜잭션 코드, 또는 메서드 등에 연결되어 프로그램 실행에 사용된다고 설명한다. 또한 SE93에서 트랜잭션 코드를 유지·관리한다고 명시한다.

CH01에서는 report transaction만 다룬다. SE93의 유형 선택 화면에서 "Program and selection screen (report transaction)"을 고르고 실행형 프로그램 `ZHELLO`에 연결한다. 여기서 "selection screen"은 SAP 화면의 선택 항목 명칭으로만 노출한다. 선택 화면 자체는 CH03과 CH15에서 본격적으로 배운다.

실행형 프로그램은 T-code가 없어도 SE38에서 실행할 수 있다. 그래서 여기서 만드는 T-code는 필수가 아니라 사용자 편의 장치다. 반면 Module Pool처럼 화면 흐름이 있는 프로그램은 시작 화면을 지정한 T-code가 필요하다. 그 내용은 CH16의 Dynpro 범위이므로 여기서는 L1 예고로만 둔다.

### 어떻게 확인하는가

확인은 다음 순서로 진행한다.

1. 명령창에 `SE93`을 입력한다.
2. 새 T-code 이름을 `ZHELLO`처럼 `Z` 또는 `Y`로 시작하게 입력한다.
3. Create를 누르고 report transaction 유형을 선택한다.
4. 연결할 프로그램 이름에 `ZHELLO`를 입력한다.
5. 저장하고 활성화한다.
6. SAP 명령창에 `ZHELLO`를 입력해 프로그램이 실행되는지 확인한다.

확인 포인트는 세 가지다. T-code 이름이 고객 네임스페이스인가? 연결 프로그램명이 정확한가? 저장 후 활성화했는가?

### 실수와 주의

T-code와 프로그램 이름을 같은 값으로 맞추면 입문 단계에서는 헷갈림이 줄지만, 둘은 같은 객체가 아니다. T-code는 실행 바로가기이고, 프로그램은 실행될 소스 객체다. 프로그램명을 잘못 연결하면 T-code는 존재하지만 실행 시 엉뚱한 오류가 난다.

SAP 표준 T-code와 충돌하는 이름을 쓰면 안 된다. 고객용 T-code도 `Z` 또는 `Y`로 시작하게 한다. 만든 뒤 활성화하지 않으면 명령창 실행이 되지 않을 수 있다. 권한이 없는 사용자에게는 T-code가 있어도 실행이 제한될 수 있지만, 권한 설계는 이후 운영·보안 범위에서 다룬다.

### 체험형 학습 설계

기존 위젯 `CH01-L07-S01`을 중심으로 사용한다.

- 프로세스 플로우: `SE93 -> T-code 입력 -> 유형 선택 -> 프로그램 연결 -> 저장·활성화 -> 명령창 실행`.
- 버튼: `저장·활성화`, `실행(Enter)`, `초기화`.
- 상태: `신규`, `저장됨`, `활성`, `명령창 실행 성공`, `프로그램명 오류`.
- 데이터: T-code `ZHELLO`, 유형 `report transaction`, 프로그램 `ZHELLO`.
- 피드백: T-code는 만들었지만 프로그램명이 틀린 경우 "바로가기는 있는데 목적지가 틀렸다"라고 설명한다.

마지막 과제는 학습자가 "SE38 실행"과 "T-code 실행"의 차이를 설명하는 것이다. 전자는 개발자가 에디터에서 직접 실행하는 방식이고, 후자는 사용자가 명령창에서 기능으로 들어가는 방식이다.

### 정리

SE93은 T-code를 만들고 관리하는 도구다. 실행형 프로그램은 T-code 없이도 실행할 수 있지만, 사용자 편의를 위해 T-code를 붙일 수 있다. CH01의 전체 루프는 이제 완성되었다. 시스템에 접속했고, 프로그램을 만들었고, 활성화하고 출력했고, 보관 위치와 실행 바로가기를 이해했다. 다음 챕터에서는 값을 코드에 직접 박아 넣는 방식의 한계를 넘어 변수와 타입을 배운다.

## CH01 전체 마무리

CH01의 핵심은 "문법 지식"보다 "개발 루프의 감각"이다. 입문자는 다음 문장을 스스로 말할 수 있어야 한다.

- SAPGUI는 서버에 접속하는 창이고, Client는 같은 시스템 안의 분리된 작업 공간이다.
- T-code로 화면에 빠르게 이동하고, SE38에서 실행형 프로그램을 만든다.
- ABAP 문장은 마침표로 끝나고, 실행형 프로그램은 `REPORT`로 시작한다.
- `WRITE`는 리스트 화면에 결과를 보여 주는 첫 출력 도구다.
- 출력은 폭, 정렬, 색, 구분선으로 읽기 좋게 만들 수 있다.
- `$TMP`는 실험용 로컬 저장 위치이고, 패키지와 이송요청은 공유·이송을 위한 관리 단위다.
- SE93으로 T-code를 만들면 사용자가 프로그램을 더 쉽게 실행할 수 있다.

CH02로 넘어가는 자연스러운 불편은 분명하다. 지금까지는 값을 `'Hello, ABAP!'`처럼 코드에 직접 적었다. 같은 값을 재사용하거나 계산 결과를 담으려면 이름 붙은 보관 장소가 필요하다. 그것이 변수와 타입이다.
