Classic DDIC View·유지보수 객체

실습 — 공연 등록 화면 (View · SM30)

콘서트앱 — 챕터의 도구를 '공연 등록과 확인' 한 흐름으로 묶기.

Track 1Chapter 14Lesson 9중급

🎫 등록과 확인을 하나로

도구를 따로따로 알면 실무 감각이 안 생긴다. Database View는 조회에, Projection View는 필드 노출 축소에, Help View는 F4에, Maintenance View·TMG는 유지보수 화면에 쓴다. 이 실습은 이 도구들을 "공연 등록과 확인" 이라는 하나의 흐름으로 묶는다.

핵심 전환은 이것이다 — 개발자가 직접 데이터를 만지던 방식에서 벗어나, 업무 담당자가 표준 화면(SM30)에서 공연을 등록하고, 개발자는 View·조회 프로그램으로 결과를 확인한다.

도구 조립 — 무엇을 어디에

  1. 등록: ZCONCERT에 TMG로 유지보수 화면을 만들고 (Chapter 14 · TMG/SM30) SM30으로 다건 등록한다.
  2. 입력 도움: perf_id 입력에는 F4를(Chapter 9 · 입력도움말) 연결해 코드 직접 입력 실수를 줄인다.
  3. 조회: ZV_CONCERT Database View로(Chapter 14 · Database View) 공연과 공연명을 함께 조회한다.

확인 리포트는 챕터 경계에 맞게 단순 조회만 한다(데이터 생성·유지보수는 SM30이 담당).

공연 등록 콘솔

F4로 공연을 고르고 저장한 뒤, 원본 테이블(ZCONCERT)과 View(ZV_CONCERT)에서 각각 어떻게 보이는지 확인하자. 기준(ZPERF)을 지우면 View에서 행이 어떻게 사라지는지도 보자.

직접 해 볼 과제

성공 경로 — F4로 P001을 골라 공연을 등록하고, SE16N(원본)과 ZV_CONCERT(View)에서 perf_name이 함께 보이는지 확인한다. 실패 경로 — 존재하지 않는 perf_id는 F4/foreign key가 막는지 확인하고, 기준 정보가 누락되면 inner join View에서 행이 사라지는지 관찰한다.

흔한 실수와 점검

  • TMG 미생성 → SM30이 "테이블 없음" 오류. Table Maintenance Generator (Chapter 14 · TMG/SM30)를 먼저 만든다.
  • 뷰 조건 → perf_id로 정확히 이어야 중복·누락이 없다.
  • 원본 ≠ View → SE16N(원본)과 View 결과는 같은 질문이 아니다. View는 inner join이라 기준이 없으면 행이 빠진다.

정리

  • 이 챕터의 도구는 따로가 아니라 등록(SM30)·도움(F4)·조회(View) 한 흐름으로 묶인다.
  • 업무 담당자는 표준 화면으로 등록하고, 개발자는 View·리포트로 확인한다.
  • 원본 테이블 확인과 View 확인을 분리해서 본다.
