Modern ABAP Syntax

Legacy 코드의 Modern ABAP 리팩터링

classic 코드를 모던으로 — "짧게"가 아니라 "또렷하게", 결과는 그대로. += 까지.

Track 1Chapter 18Lesson 10중급

왜 바꾸나 — "짧게"가 아니라 "또렷하게" 🧭

여기까지 인라인 DATA( )·VALUE·CORRESPONDING·Table Expression·String Template· CONV/EXACT·COND/SWITCH·REDUCE/FILTER·LET을 하나씩 배웠다. 이제 이 조각들을 모아, 앞에서 쓰던 classic 코드를 다시 써 본다. 이걸 리팩터링이라 한다.

🎯 리팩터링(Refactoring, 동작은 그대로 두고 코드 모양만 다듬기)의 목표는 줄 수를 줄이는 게 아니다. ① 의도가 더 잘 보이고 ② 실수할 구석이 줄고 ③ 실행 결과는 똑같이 나오는 것. 이 셋이다.

그래서 이 레슨은 새 문법을 더 배우는 자리가 아니라, 이미 배운 걸로 "어떤 줄을 바꾸고 어떤 줄은 그냥 둘지" 판단하는 중간 점검이다.

같은 코드, 두 가지 표기 ✍️

먼저 흔한 classic 코드다. 구구단 한 줄을 만들어 테이블에 담고, 합계를 누적한다.

" --- classic (개선 전) ---
DATA ls_line  TYPE ts_line.
DATA lt_gugu  TYPE tt_line.
DATA lv_total TYPE i.

ls_line-dan    = 2.
ls_line-mul    = 1.
ls_line-result = 2.
APPEND ls_line TO lt_gugu.

lv_total = lv_total + ls_line-result.

동작은 한다. 그런데 한 줄을 만들려고 네 줄을 썼고, 누적도 lv_total을 양쪽에 두 번 적었다. 이 챕터에서 배운 문법으로 다듬으면 이렇게 짧아진다.

" --- modern (권장) ---
DATA(lt_gugu) = VALUE tt_line(
  ( dan = 2 mul = 1 result = 2 ) ).

DATA(lv_total) = 0.
lv_total += lt_gugu[ 1 ]-result.

여기서 내가 내린 판단은 세 가지다.

  1. Structure 채우고 바로 APPEND → VALUE로 묶는다. 결과 테이블의 "모양"이 코드에 그대로 보인다.
  2. 누적 대입 lv_total = lv_total + x → +=로 줄인다(바로 아래에서 정식 도입).
  3. Table Expression lt_gugu[ 1 ] → 바로 앞에서 한 행을 만들었으니 1번 행이 있는 게 확실할 때만 안심하고 쓴다. 행이 있을지 불확실하면 line_exists( )로 먼저 막는다(Table Expression 안전하게 읽기에서 배운 방어선).

+= -= *= /= — 누적을 짧게 ➕

산술 연산과 대입에서 배운 lv_total = lv_total + x를, 같은 변수를 한 번만 쓰고 끝내는 짧은 표기가 있다. 이것을 복합 대입(compound assignment, 계산하면서 동시에 대입)이라고 한다.

ABAP

lv_total += ls_line-result.   " lv_total = lv_total + ls_line-result 와 같다
lv_count -= 1.                " lv_count = lv_count - 1
lv_price *= 2.                " lv_price = lv_price * 2
lv_share /= 3.                " lv_share = lv_share / 3

⚠️ +=의 왼쪽 변수는 이미 존재해야 한다. +=는 "기존 값에 더하기"라서, 선언과 동시에는 못 쓴다. DATA(lv_total) += 10. ← 이건 안 된다. 먼저 DATA(lv_total) = 0.으로 만든 뒤에 lv_total += 10..

결과가 같은지 꼭 확인 ✅

리팩터링은 "컴파일되네"에서 끝나면 안 된다. 바꾸기 전과 후의 실행 결과가 같아야 진짜 성공이다. 순서는 이렇게 잡는다.

  1. classic 코드의 결과를 먼저 적어 둔다(합계·테이블 행 수·첫 행 값).
  2. modern으로 바꾼다.
  3. 같은 입력으로 다시 실행한다.
  4. 출력·행 수·합계가 모두 같은지 비교한다.
  5. 아직 안 배운 영역(모던 SQL·OO·예외 처리)을 실수로 끌어오지 않았는지 확인한다.
확인 항목 classic 기대 modern 기대
lt_gugu 행 수 1 1
첫 행 dan / mul / result 2 / 1 / 2 2 / 1 / 2
lv_total 2 2

세 값이 모두 같으면 — 표현만 바뀌고 결과는 그대로. 그게 좋은 리팩터링이다.

흔한 실수와 점검 🚧

  • BASE를 빠뜨림 → VALUE로 다시 쓰면서 기존 테이블에 행을 더하려던 거라면 VALUE ...( BASE lt_old ( … ) )처럼 BASE가 필요하다. 빼면 기존 행이 통째로 사라진다(VALUE Constructor Expression에서 다룬 함정).
  • += 왼쪽을 새로 선언 → 위에서 본 함정. 누적 변수는 먼저 있어야 한다.
  • 한 줄에 너무 많이 → 짧다고 다 좋은 게 아니다. 아래는 9줄짜리 테이블을 통째로 만들어 단 한 값만 꺼낸다.

" 😵 과한 표현 — 9행을 다 만들고 그중 하나만 쓴다
DATA(lt_all) = VALUE tt_line(
  FOR i = 1 WHILE i <= 9 ( dan = 2 mul = i result = 2 * i ) ).
DATA(lv_third) = lt_all[ 3 ]-result.

필요한 게 값 하나면 테이블을 통째로 만들 이유가 없다. 처음부터 그 값만 쓴다: DATA(lv_third) = 2 * 3. VALUE·FOR·Table Expression은 쓸 수 있다고 다 쓰는 게 아니라, 의도에 맞는 만큼만 쓴다. 헷갈리면 끊어 쓴다.

  • SQL은 그대로 둔다 → SELECT 문장을 모던 SQL로 바꾸는 건 이 레슨 범위 밖이다. SQL의 @·콤마는 Classic Open SQL과 Modern ABAP SQL 비교에서 따로 연다. 여기선 SQL 바깥의 ABAP 코드만 다듬는다.
