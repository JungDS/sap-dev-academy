CDS View Entity 기초

Interface View와 Projection View 구분

재사용용 기반 뷰와 소비용 뷰를 나눈다 — ZI_ / ZC_.

Track 1Chapter 23Lesson 2고급

왜 뷰를 또 나눌까

CDS View Entity 기본 구조에서 ZI_Concert 하나만 만들어도 프로그램은 조회할 수 있다. 그런데 실무에서 CDS가 한 장으로 끝나는 경우는 드물다. 같은 공연 데이터라도 소비처마다 원하는 모양이 다르기 때문이다.

  • 운영 리포트는 내부 관리 필드까지 필요할 수 있다.
  • 화면 목록은 사용자에게 보여 줄 필드만 있으면 된다.
  • 나중에 서비스가 붙으면, 외부에 노출해도 되는 필드와 내부에만 둬야 하는 필드가 갈린다.

이때 기반 모델 하나에 UI 주석·필드 제한·내부 계산을 전부 섞으면 CDS가 금세 지저분해진다. 그래서 CDS는 보통 계층을 나눈다 — 데이터와 관계를 안정적으로 담는 계층, 그리고 특정 소비 목적에 맞게 필드를 고르는 계층.

무엇인가 — 기반(ZI_)과 소비(ZC_)

이 프로젝트는 다음 이름 관습을 쓴다.

계층 이름 관습 역할
Interface View ZI_* 테이블과 가까운 재사용 기반 모델. 필드·기본 계산·관계를 안정적으로 정의.
Projection View ZC_* 특정 소비처에 맞춘 노출 모델. 필요한 필드를 고르고 주석을 붙이는 출구.

여기서 가장 중요한 한마디 — ZI_·ZC_는 ABAP 문법이 아니라 명명 관습이다. 시스템이 ZI_라서 자동으로 interface view라고 판단하는 게 아니다. 이름으로 역할을 드러내 팀이 같은 구조를 유지하게 돕는 장치일 뿐이다(이름을 ABC_Concert로 지어도 활성화는 된다 — 다만 팀 규칙을 잃는다).

@AccessControl.authorizationCheck: #NOT_REQUIRED
define view entity ZC_Concert
  as projection on ZI_Concert

as projection on ZI_Concert는 "ZC_Concert가 ZI_Concert를 기반으로 한다"는 뜻이다. 여기서 모든 필드를 다시 가져올 필요는 없다 — 소비처에 필요한 필드만 고른다.

@AccessControl.authorizationCheck: #NOT_REQUIRED
define view entity ZC_ConcertList
  as projection on ZI_Concert
{
  key concert_id,
      artist,
      venue
}

이 예제는 capacity를 뺐다. 테이블에도 ZI_Concert에도 capacity는 여전히 있지만, ZC_ConcertList를 보는 소비자는 그 필드를 보지 않는다. 이것이 projection의 가장 단순하고 중요한 효과다 — 삭제가 아니라 숨김.

직접 만들어 보기

아래 실험실에서 기반 ZI_Concert의 필드를 projection이 골라 노출해 보자. 두 동작의 차이가 핵심이다.

  • ZC_에서 필드를 숨김 → 기반 ZI_는 그대로다(비파괴). 소비자만 안 볼 뿐.
  • ZI_에서 필드를 제거 → projection도 더는 그 필드를 노출할 수 없다(연쇄). 기반에 없으니까.

어떻게 확인하는가

확인은 두 방향으로 한다.

  1. Data Preview 비교 — ZI_Concert와 ZC_Concert를 각각 미리 보면, projection에서 노출하지 않은 필드는 ZC_ 쪽에 안 보인다. "필드를 삭제했다"가 아니라 소비 모델에서 숨긴 것임을 눈으로 확인한다.
  2. ABAP SQL로 읽기 — 소비 프로그램은 ZC_만 알면 된다.
ABAP
1
2
3
SELECT concert_id, artist, venue
  FROM ZC_Concert
  INTO TABLE @DATA(lt_concerts).

이렇게 해 두면 기반 ZI_Concert나 테이블 zconcert가 어느 정도 바뀌어도, ZC_Concert가 같은 계약을 유지하는 한 소비 코드는 덜 흔들린다. 점검 기준은 간단하다.

질문 정상 답
이 뷰는 테이블과 가까운가? 그렇다면 ZI_에 둔다.
이 뷰는 화면·서비스·리포트 같은 소비 출구인가? 그렇다면 ZC_에 둔다.
UI 주석이 기반 뷰에 잔뜩 들어갔는가? 가능하면 ZC_나 Metadata Extension으로 옮긴다.

흔한 실수와 점검 🚧

  • ZI_·ZC_를 문법으로 착각 → 둘 다 그냥 이름 관습이다. 활성화는 다른 이름이어도 되지만, 관습을 잃으면 "어떤 뷰를 재사용해야 하는지"가 모호해진다.
  • 모든 뷰를 projection으로만 만들기 → projection은 기반이 있어야 의미가 있다. 테이블에서 직접 읽어 관계·의미를 정리하는 ZI_가 먼저고, 그 위에 소비용 ZC_가 올라간다.
  • 다음 장(RAP)의 projection 개념을 미리 끌어오기 → 공식 문서에는 provider contract 같은 내용도 있지만, 이 챕터에선 읽기 모델의 계층 분리까지만 한다. RAP 트랜잭션 모델·service·behavior는 RAP 아키텍처 개요에서 다룬다.

기반과 소비를 나눴다. 그런데 콘서트 앱에서 공연·회차·예매는 늘 함께 등장한다 — 이 관계를 매번 JOIN으로 반복하는 대신 모델에 이름 붙여 두는 방법 → Association 기초.
