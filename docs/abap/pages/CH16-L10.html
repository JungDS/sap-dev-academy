🎫 종합 — 직접 설계한 예매 입력 화면

앞 레슨을 따로 보면 Dynpro가 조각처럼 느껴진다. 마지막 실습에서 예매 입력 화면 0100을 만든다고 가정하고, Screen Painter 요소·PBO·PAI·OK_CODE·PF-STATUS·검증을 하나의 흐름으로 묶는다.

이 실습의 성공 기준은 DB 저장이 아니다. 실제 저장(INSERT·COMMIT WORK)과 잠금은 실무 데이터 변경과 트랜잭션 제어 이후에서 다룬다. 여기 목표는 값 입력 → 버튼 → 검증 메시지 → 안전한 화면 종료의 흐름을 만드는 것이다.

요구사항

  • Module Pool을 만들고 T-code로 화면 0100을 실행한다(Module Pool 프로그램 구조).
  • 화면에 GV_CONC(F4)·GV_PERF·GV_SEATS·GV_CUST·GV_STAT·OK_CODE를 둔다.
  • PBO에서 PF-STATUS·TITLEBAR를 켜고 상태 드롭다운을 채운다.
  • PAI에서 OK_CODE를 save_ok로 옮기고 CLEAR한 뒤, SAVE면 잔여석 계산·예매 판정의 can_book으로 검증한다.
  • 공연 정보는 DDIC ZCONCERT에서 오므로 TABLES zconcert로 잇고, 공연에 따라 달라지는 회차 F4는 직접 만든다(Dictionary 운반·F1·F4 직접 만들기).

PROGRAM sapmzconcert.

DATA: gv_conc  TYPE zconcert-concert_id,
      gv_perf  TYPE c LENGTH 3,
      gv_seats TYPE i,
      gv_cust  TYPE c LENGTH 30,
      gv_stat  TYPE c LENGTH 1,
      ok_code TYPE sy-ucomm,
      save_ok TYPE sy-ucomm.

DATA: gt_values TYPE vrm_values,
      gs_value  LIKE LINE OF gt_values.

PROCESS BEFORE OUTPUT.            " Flow Logic
  MODULE status_0100.
  MODULE fill_status_list.
PROCESS AFTER INPUT.
  MODULE user_command_0100.

MODULE status_0100 OUTPUT.
  SET PF-STATUS 'ST100'.
  SET TITLEBAR  'TB100' WITH gv_conc.   " 제목 "공연 &1" → gv_conc
ENDMODULE.

MODULE fill_status_list OUTPUT.        " 상태 드롭다운 채우기
  CLEAR gt_values.
  gs_value-key = 'R'. gs_value-text = '예약'. APPEND gs_value TO gt_values.
  gs_value-key = 'W'. gs_value-text = '대기'. APPEND gs_value TO gt_values.
  CALL FUNCTION 'VRM_SET_VALUES'
    EXPORTING id = 'GV_STAT' values = gt_values.
ENDMODULE.

MODULE user_command_0100 INPUT.
  DATA lv_ok TYPE abap_bool.

  save_ok = ok_code.                   " 보조 변수로 복사
  CLEAR ok_code.                       " 즉시 비움(반복 실행 방지)

  CASE save_ok.
    WHEN 'SAVE'.
      PERFORM can_book USING gv_conc gv_perf gv_seats CHANGING lv_ok.
      IF lv_ok = abap_true.
        MESSAGE '예매 가능 — 저장은 이후 단계에서' TYPE 'S'.
      ELSE.
        MESSAGE '좌석이 부족합니다' TYPE 'E'.   " 화면 유지(정상 검증)
      ENDIF.
    WHEN 'BACK' OR 'CANCEL'.
      LEAVE TO SCREEN 0.               " 화면 종료
    WHEN 'EXIT'.
      LEAVE PROGRAM.                   " 프로그램 종료
  ENDCASE.
ENDMODULE.

💡 위 예제는 한 코드 블록에 Flow Logic과 ABAP module을 함께 보여 준 것이다. 실제로는 Flow Logic은 Screen Painter에, MODULE ... ENDMODULE은 program source에 따로 있다.

화면 필드 연결·입력 도움 반영

공연 정보는 DDIC ZCONCERT에서 오므로 로 잇고, 공연에 따라 달라지는 회차 F4는 로 직접 만든다.

TABLES zconcert.                 " 공연ID·공연명을 DDIC에서 (L04)

PROCESS ON VALUE-REQUEST.        " Flow Logic — 회차 F4 (L05)
  FIELD gv_perf MODULE value_perf.
  • TABLES zconcert → 화면의 ZCONCERT-TITLE에 공연명이 라벨·F4까지 함께 온다.
  • value_perf 모듈은 DYNP_VALUES_READ로 공연ID를 읽어 그 공연의 회차만 후보로 띄운다.
  • 원리는 Dictionary 화면 필드와 TABLES 운반·화면에서 F1·F4 직접 만들기.

직접 실행해 보기 — 예매 입력 Dynpro 통합 시뮬레이터

좌석 수를 바꾸고 버튼을 눌러, PBO 준비 → PAI 검증 → 메시지/종료의 한 흐름을 따라가 보자. 좌석 과다 입력 시 MESSAGE E로 화면이 유지되는지, SAVE 뒤 Enter가 재실행되지 않는지 확인하자.

흔한 실수와 주의

  • Flow Logic과 ABAP source를 같은 곳에 → 설명상 합쳤을 뿐, 실제 위치는 다르다.
  • MESSAGE E 후 화면 유지를 실패로 오해 → 정상 검증 흐름이다(잘못 입력을 막고 다시 받음).
  • 저장까지 기대 → 화면·검증까지가 이 챕터. INSERT·COMMIT WORK은 이후.
  • EXIT를 LEAVE TO SCREEN 0으로 → EXIT는 LEAVE PROGRAM. 화면 종료와 프로그램 종료를 구분.

정리

직접 만든 화면의 생명주기를 한 번에 묶었다 — PBO가 준비하고, 사용자가 입력하고, PAI가 OK_CODE를 읽어 검증·종료한다.
