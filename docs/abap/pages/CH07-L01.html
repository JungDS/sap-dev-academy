Transparent Table 생성 (SE11)

값을 영구히 — DB에 1:1로 대응하는 투명 테이블을 만든다.

Track 1Chapter 7Lesson 1초급

끝나면 사라진다

CH06에서 구구단 81행을 lt_gugu에 담았지만, 프로그램이 끝나면 째 사라졌다. 다시 실행하면 또 처음부터다. 값을 디스크에 영구 저장하고 다른 사용자와도 공유하고 싶다 — 그게 (투명 테이블)다.

투명 테이블이란

은 DDIC에 정의하는 테이블로, DB의 실제 물리 테이블과 1:1로 대응한다. "투명(transparent)"은 DDIC 정의와 DB 테이블이 같은 모양으로 들여다보인다는 뜻.

Technical Settings 데이터 클래스·크기 지정 후 활성화

활성화하는 순간 실제 DB 테이블이 생성된다.

필드 타입 — Data Element vs Built-In

각 컬럼의 타입을 주는 방법은 두 가지다.

  • 참조(권장) — DAN에 ZDE_DAN 같은 Data Element를 주면, CH03에서 봤듯 라벨·F4·도움말까지 따라온다. 화면·ALV가 똑똑해진다.
  • Built-In 타입 직접 — INT4(정수) 같은 내장 타입을 바로 지정. 간단하지만 라벨·도움말이 없다.

업무 테이블은 거의 항상 Data Element로 준다. 그래야 전사적으로 의미·라벨이 일관된다.

MANDT — 클라이언트 분리

대부분 업무 테이블의 첫 키는 (클라이언트)다. CH01에서 본 "방(Client)"별로 데이터가 분리되는 장치다. 보통 직접 다루지 않아도 시스템이 챙긴다.

Technical Settings — 저장 방식 지정

활성화 전, Technical Settings(기술 설정)에서 DB에 어떻게 저장할지를 정한다(앞 표의 그 항목).

  • Data Class(데이터 클래스) — 테이블의 성격(예: 마스터 데이터·트랜잭션 데이터). DB가 비슷한 성격끼리 같은 저장 영역에 모은다.
  • Size Category(크기 범주) — 예상 행 수 규모(0~4 등). DB가 처음 잡을 공간을 가늠하는 힌트다(나중에 늘어나도 됨).

💡 입문 단계에선 기본값으로 두고 활성화해도 된다. "성격(Data Class)과 규모(Size Category)를 미리 알려 주는 것" 정도면 충분. 참고로 데이터가 많이 쌓이고 지워지길 반복하면 테이블에 빈틈이 생기는 Fragmentation(단편화)이 생길 수 있는데, 이건 DBA(운영 DB 관리자) 영역이다 — 지금은 이름만.

🔎 Where-Used List — SE11에서 어떤 ·Domain·테이블을 어디서 쓰는지 한 번에 찾아 준다. 객체를 고치기 전 "이걸 바꾸면 어디가 영향받지?"를 이걸로 먼저 확인한다.

흔한 실수와 점검

  • 키 미지정 → 최소 하나의 키 필드 필요. 보통 + 업무 키.
  • 활성화 안 함 → 활성화해야 DB 테이블이 실제로 생긴다.
  • 필드를 Built-In으로만 → 라벨·도움말을 잃는다. 가능하면 .
  • 활성 후 타입·길이 변경 ⚠️ → 이미 데이터가 든 테이블의 필드 타입·길이를 바꾸면 데이터 변환·손실·긴 잠금 위험이 있다. 운영 테이블 구조 변경은 신중히·이송 절차로(CH35).

그릇(ZGUGUDAN)은 만들었다. 이제 값을 직접 넣어 보자 → Create Entries로 구구단 입력(다음 레슨).
