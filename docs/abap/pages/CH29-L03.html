가장 권장되는 확장 — BAdI

Customer Exit·Enhancement Point는 "표준 어디에 코드를 꽂나" 가 초점이었다. 그런데 같은 시점이라도 국가·회사코드·사업부별로 구현이 다를 수 있고, 여러 구현을 켜고 끄거나 조건에 맞는 것만 골라야 한다. (Business Add-In)는 이걸 지향으로 푼다.

표준은 "어떤 구현을 부를지" 를 직접 모른다. 대신 (계약) 의 를 부르고, 고객이 그 인터페이스를 구현한다. Interface 기본 설계에서 배운 인터페이스가 여기서 표준–고객 사이의 계약으로 산다.

" 표준 쪽: 구현을 직접 모르고 인터페이스만 호출
GET BADI lo_badi
  FILTERS country = lv_country.

CALL BADI lo_badi->check_booking
  EXPORTING is_booking = ls_booking.

고객은 그 인터페이스 메서드를 구현한다(SE19로 생성).

  • SE18(정의 조회)·SE19(구현 생성). 인터페이스 이름은 보통 IF_EX_... 꼴.
  • 필터 BAdI — FILTERS로 국가·회사코드 등 조건별 구현을 자동 선택.
  • multiple-use는 여러 구현이 차례로, single-use는 하나만(참조가 비었는데 호출하면 가능).

구현을 꽂아 보기

구현(플러그)을 켜고 끈 뒤 표준을 실행해, 표준이 내 를 부르는지(없으면 기본 동작) 확인해 보라.

흔한 실수와 점검

  • 정의(SE18)와 구현(SE19) 혼동 → 정의만 보고 "작동한다" 할 수 없고, 구현만 만들어도 filter·activation이 안 맞으면 호출 안 된다.
  • filter를 가볍게 → filter는 부가 조건이 아니라 구현 선택 조건이다. 표준이 country='KR'로 부르는데 구현 filter를 US로 두면 로직이 완벽해도 호출 안 된다.
  • multiple-use 순서·중복 무시 → 여러 구현이 같은 메시지를 더하면 결과가 중복된다. 구현은 멱등하게, 같은 메시지 반복 방지 guard를 둔다.
  • 에 화면·외부연동 직접 → BAdI는 표준 중간에 실행된다. 느린 I/O· 제어·화면 호출은 표준 전체를 불안정하게 한다. 짧게 호출·판단·반환만, 복잡한 건 별도 서비스 로 분리한다.

정리

  • = 표준과 고객 구현 사이를 계약으로 분리하는 OO 확장.
  • 표준은 GET BADI로 구현을 준비하고 CALL BADI로 호출, 고객은 SE19 구현 로 응답한다.
  • Customer Exit보다 선호되는 이유 = 구조가 명확(signature·filter·use 방식·activation으로 관리).
