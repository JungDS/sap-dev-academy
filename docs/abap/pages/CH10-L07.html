🎫 2단계 — 핵심 로직 모듈화

CH09에서 만든 테이블에, "이 회차 잔여석이 몇인가?", "이 예매가 가능한가?"를 계산하는 로직을 더한다. 여기저기 흩지 말고 모듈(CH10)로 묶는다.

잔여석 = 정원 − 예매 합계

집계 SQL(SUM)은 CH13에서 배우니, 지금은 예매를 읽어 LOOP로 합산한다(CH06·CH08).

FORM get_remaining
  USING    iv_concert TYPE zbooking-concert_id
           iv_perf    TYPE zbooking-perf_no
  CHANGING cv_left    TYPE i.

  DATA: lt_book TYPE TABLE OF zbooking,
        ls_book TYPE zbooking,
        lv_sum  TYPE i,
        lv_cap  TYPE zconcert-capacity.

  SELECT * FROM zbooking INTO TABLE lt_book
    WHERE concert_id = iv_concert AND perf_no = iv_perf
      AND status <> 'C'.                 " 취소 제외

  LOOP AT lt_book INTO ls_book.
    lv_sum = lv_sum + ls_book-seats.
  ENDLOOP.

  SELECT SINGLE capacity FROM zconcert INTO lv_cap
    WHERE concert_id = iv_concert.

  cv_left = lv_cap  lv_sum.
ENDFORM.

FORM can_book
  USING    iv_concert TYPE zbooking-concert_id
           iv_perf    TYPE zbooking-perf_no
           iv_want    TYPE i
  CHANGING cv_ok      TYPE abap_bool.

  DATA lv_left TYPE i.
  PERFORM get_remaining USING iv_concert iv_perf CHANGING lv_left.
  cv_ok = boolc( iv_want <= lv_left ).    " 원하는 좌석 <= 잔여석
ENDFORM.

잔여석이 나왔으면, 요청 좌석과 비교하는 can_book 판정도 만져 보자. 취소 건을 합산하면 왜 틀리는지도 확인할 수 있다.

직접 해 볼 과제

  1. get_remaining을 호출해 한 회차의 잔여석을 출력한다.
  2. 정원보다 많은 좌석을 요청해 can_book이 거짓을 돌려주는지 확인.
  3. (도전) 같은 로직을 CH10-L04의 static method(zcl_booking=>get_remaining)로 바꿔 본다.

흔한 실수와 점검

  • 취소 건 합산 → status <> 'C'로 취소 예매는 제외.
  • 전역변수 의존 → 입력은 USING, 결과는 CHANGING으로(CH10-L02).

핵심 로직을 모듈로 묶었다. 이제 예매 목록을 표로 보여 주자 → CH11.
