Internal Table

Deep Structure 개념

구조 안에 Internal Table·문자열이 든 'Deep' Structure — 개념과 분류.

Track 1Chapter 6Lesson 5초급

행 안에 또 목록이 있다면

주문 한 건에는 "주문 항목 여러 줄"이 딸린다. 즉 한 행 안에 또 다른 이 들어가야 한다. 이렇게 Structure 안에 테이블이나 문자열 같은 가변 요소가 든 것을 (깊은 구조)라 한다. 여기서는 개념과 분류만 잡는다(구현은 뒤에서).

Flat vs Deep

  • Flat Structure — 모든 Component가 고정 길이 타입(c LENGTH n·n·d·t·i·p 등)이고, 아래의 deep 요소가 하나도 없다.
  • — Component 중에 가변 요소가 하나라도 있다: string, , 참조(reference) 등.

TYPES: BEGIN OF ts_item,              " 주문 항목 한 줄
         item TYPE c LENGTH 20,
         qty  TYPE i,
       END OF ts_item.

TYPES tt_item TYPE TABLE OF ts_item.  " 항목 여러 줄(테이블 타입)

TYPES: BEGIN OF ts_order,             " 주문 한 건
         order_id TYPE c LENGTH 10,
         customer TYPE c LENGTH 20,
         items    TYPE tt_item,       " 행 안에 테이블! → Deep
       END OF ts_order.

ts_order는 items Component가 Internal Table이라 Deep Structure다.

사실, 우리는 이미 Deep을 쓰고 있었다

이 챕터 내내 쓴 ts_person도 알고 보면 이미 Deep이다.

TYPES: BEGIN OF ts_person,
         name TYPE string,   " string은 길이 가변 → deep 요소!
         age  TYPE i,
       END OF ts_person.

string은 길이가 고정돼 있지 않은 가변 타입이라 deep 요소로 분류된다. 문자열 하나만 있어도 그 Structure 전체가 Deep이 된다. 지금까지 아무 문제 없이 썼듯 Deep이라서 못 쓰는 건 아니다. 다만 아래 상황들에서 Flat과 다르게 취급되므로 구분할 줄 알아야 한다.

중첩(Nested)과는 다른 개념

Structure 안에 Structure가 든 것은 중첩(nested)이다. 중첩이라도 안까지 전부 고정 길이 타입이면 여전히 Flat이다. Deep 여부는 "구조가 몇 겹인가"가 아니라 가변 요소(string·테이블·참조)가 있는가로만 정해진다.

왜 미리 알아두나

  • DB 저장: DB 테이블의 컬럼은 단일(elementary) 타입 하나여야 해서, · 참조·하위 Structure를 통째로 컬럼에 넣을 수 없다.
  • 화면 출력: 결과를 표 형태 화면으로 보여 주는 도구(ALV, 뒤에서 배운다)도 기본적으로 평평한 행을 전제한다.
  • 지금은 "구조 안에 테이블이 들어갈 수 있다"는 가능성과 분류 기준만 기억하면 된다. 실제 활용과 제약은 나선형으로 뒤에서 깊어진다.

흔한 오해와 점검

  • Deep을 DB 테이블 컬럼으로 → 안 된다. DB 테이블(Chapter 7 · Transparent Table)의 한 행은 단일 타입 컬럼들로만 구성한다.
  • 중첩 = Deep이라는 착각 → 중첩은 "겹"의 문제, Deep은 "가변 요소"의 문제. 별개다.
  • 지금 깊이 파려는 욕심 → 개념·분류만. 구현은 뒤에서.

분류 연습

구조 정의를 보고 Flat인지 Deep인지 판정해 보자.

이제 구조와 테이블을 깊이까지 안다. 배운 걸 모아 구구단 전체를 로 완성해 보자 → 구구단 전체 = Internal Table.
