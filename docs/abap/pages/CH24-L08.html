RAP / ABAP Cloud 입문

ABAP Cloud와 Released API 원칙

클라우드 준비된 개발 — Released API·restricted scope·Clean Core.

Track 1Chapter 24Lesson 8고급

왜 제한이 필요할까

이 과정에서 classic ABAP도 많이 배웠다. Dynpro, SALV, classic table maintenance, SAP GUI 기반 도구는 온프레미스 유지보수와 기존 시스템에서 여전히 중요하다. 그러나 신규 S/4HANA Cloud나 cloud-ready 확장을 만들 때는 아무 나 직접 호출하고, 표준을 직접 수정하고, SAP GUI 전용 흐름에 의존하는 방식이 맞지 않는다.

****는 업그레이드에 강한 cloud-ready 개발을 위해 언어 범위와 API 접근을 제한한다. 이 제한은 불편하게 하려는 규칙이 아니라, 표준 코어를 깨지 않고 확장하기 위한 안전장치다.

무엇인가 — cloud-ready 개발의 원칙

는 cloud-ready·upgrade-stable 개발을 위한 ABAP development model이다. 여기서는 ABAP for Cloud Development라는 restricted ABAP language version을 쓰고, 다른 repository object에 접근할 때 released API만 사용할 수 있다. 개발 도구도 ADT 중심이다.

원칙 뜻
ABAP Cloud에서 접근하도록 release된 repository object(또는 그 일부)만 쓴다.
Restricted language scope 모든 classic ABAP 구문·도구가 허용되는 것은 아니다.
ADT 중심 개발 ABAP Cloud 개발은 Eclipse 기반 ADT 흐름을 중심으로 한다.
RAP transactional model ABAP Cloud의 트랜잭션 프로그래밍 모델은 RAP다.
SAP 표준 코어를 직접 바꾸지 않고 공개된 확장 지점·API를 사용한다.

중요한 균형이 있다 — classic ABAP을 배운 것은 낭비가 아니다. 기존 시스템을 읽고 유지보수하려면 classic 지식이 필요하다. 다만 cloud-ready 신규 개발에서는 RAP·CDS·released API 원칙을 따른다.

직접 판정해 보기

아래에서 개발 시나리오가 에서 가능한지 판정해 보자. 제한되는 경우의 대안까지 보면, "무엇이 왜 막히는가"가 또렷해진다. released 여부는 이름으로 추측하면 안 된다.

어떻게 확인하는가

  1. language version — ABAP for Cloud Development로 설정된 는 classic 환경보다 엄격한 syntax/access check를 받는다.
  2. released 여부 — SE24에서 보이는 class, SE11에서 보이는 table이라고 cloud에서 쓸 수 있는 게 아니다. release contract와 visibility를 본다.
  3. GUI/unreleased 의존 — SAP GUI 전용 흐름이나 unreleased table 직접 접근에 의존하는지 본다. RAP 서비스 모델은 ADT와 OData/Fiori 소비를 전제로 한다.
  4. 표준 직접 수정 여부 — 관점에서 표준 수정을 피하고 공개 확장 지점·released API를 쓴다.

흔한 실수와 점검 🚧

  • = "클라우드 서버 ABAP"으로 이해 → 핵심은 서버 위치가 아니라 restricted language version·released API·RAP 중심 개발 모델이다.
  • released 여부를 이름으로 추측 → CL_*·BAPI_*·테이블 이름이 익숙하다고 released인 것은 아니다. 시스템의 API release 정보를 확인한다.
  • classic 지식을 부정 → 기존 온프레미스 시스템에는 classic 지식이 여전히 필요하다. cloud-ready 개발에서만 허용 경계를 다르게 본다.
  • = "아무 확장도 하지 말라"로 오해 → Clean Core는 확장 금지가 아니라, 표준을 훼손하지 않는 방식으로 확장하라는 원칙이다. 확장은 Customer Exit / User Exit 개념부터 이어지는 enhancement·BAdI로 한다.
