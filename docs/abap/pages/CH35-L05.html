운영의 블랙박스 — Application Log

운영에서 무슨 일이 있었는지(성공·경고·오류) 남기는 표준이 (BAL)다. WRITE·임시 로그 대신, 검색·보관·권한이 되는 표준 로그를 쓴다.

  • SLG1 — Application Log 조회(객체·기간·사용자별).
  • 메시지(타입 I/S/W/E, CH15)를 컨텍스트(키·단계)와 함께 남긴다.
  • 배치(CH35-L04)·인터페이스(CH30)·DML(CH24)의 결과를 여기 기록 → 재처리·감사.

운영 마인드셋

  • 무엇이·왜·언제 실패했는지 남겨야 고친다.
  • 성공도 적당히 남겨(추적성), 너무 많으면 노이즈 — 균형.

흔한 실수와 점검

  • 로그 없음 → 운영 사고의 원인을 못 찾는다. 핵심 단계엔 BAL.
  • 민감정보 로깅 → 개인정보·비밀번호는 남기지 않는다.

🧩 도전 과제 (서브퀘스트)

선택 과제 — 예매 앱을 운영 품질로 끌어올린다.

  1. 예매 프로그램에 ATC를 돌려 findings를 0(또는 정당한 예외)으로 만든다.
  2. 핵심 로직(remaining·검증)에 ABAP Unit 테스트를 붙여 회귀를 막는다.
  3. (심화) 야간 예매 정산을 Background Job(SM36)으로 스케줄하고 Application Log(BAL)로 결과를 남긴다.

힌트: L01(ATC)·L02(ABAP Unit)·L03(Transport)·L04(Background Job)·L05(BAL).

품질·테스트·이송·배치·로그까지 — 운영의 한 바퀴를 돌았다. 이제 배운 모든 걸 모아 실전 앱을 완성하는 Capstone → CH36.
