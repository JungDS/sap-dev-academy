데이터(속성)와 행동(메서드) 🧱

좋은 는 "무엇을 알고 있는가"(속성)와 "무엇을 할 수 있는가"()를 함께 설계한다. 예약 관리자라면 공연 ID·회차·정원 같은 상태를 알고, 잔여석 계산·예약 처리 같은 행동을 한다.

CLASS zcl_booking_manager DEFINITION PUBLIC CREATE PUBLIC.
  PUBLIC SECTION.
    METHODS book      IMPORTING iv_seats TYPE i.
    METHODS remaining RETURNING VALUE(rv_remaining) TYPE i.
  PRIVATE SECTION.
    DATA mv_concert TYPE zconcert-concert_id.   " 인스턴스 속성(객체마다 따로)
    DATA mv_perf    TYPE zbooking-perf_no.
    DATA mv_cap     TYPE i.
ENDCLASS.
  • : DATA는 속성(마다 따로). CLASS-DATA는 정적 속성(클래스에 하나, 공용).
  • 메서드: METHODS는 인스턴스 메서드, CLASS-METHODS는 정적 메서드.
  • mv_는 멤버 변수(member variable) 접두어 관례. 정적 속성(CLASS-DATA)은 객체가 아니라 클래스에 하나뿐이라 gv_(클래스 공용)로 구분해 쓴다.

공개 범위 — Visibility 🚪

ABAP
1
2

CLASS ... DEFINITION ...
  PUBLIC SECTION.       " 외부가 호출해야 하는 행동
  PROTECTED SECTION.    " 상속 자식에게 열어 둘 내부 계약
  PRIVATE SECTION.      " 외부가 직접 바꾸면 안 되는 상태
섹션 접근 가능 설계 기준
PUBLIC SECTION 누구나(외부·자신·자식) 외부가 호출할
PROTECTED SECTION 자신 + 자식 상속 확장용 내부 계약
PRIVATE SECTION 자신만 외부가 못 건드릴 상태

🛡️ 입문 기본 원칙 — "데이터는 보통 private, 행동은 필요한 만큼 public." 는 데이터를 숨기는 장식이 아니라, 잘못된 상태 변경을 막는 방어선이다.

DATA(lo_mgr) = NEW zcl_booking_manager( ).

DATA(lv_remaining) = lo_mgr->remaining( ).   " PUBLIC 메서드 → OK
" lo_mgr->mv_cap = 100.                       " PRIVATE 속성 직접 접근 → 오류

private 접근 오류는 런타임이 아니라 컴파일(syntax check) 단계에서 난다. 의 경계는 실행 중 뒤늦게 드러나는 약속이 아니라, 컴파일 시점에 검증되는 계약이다. ( vs 정적도 두 개를 만들어 비교해 보면, DATA는 객체마다 다르고 CLASS-DATA는 공용임이 드러난다.)

흔한 실수와 점검 🚧

  • 데이터를 PUBLIC → 외부가 정원을 마음대로 바꿔 book( )·remaining( )의 규칙을 우회. 데이터는 PRIVATE, 접근은 로.
  • 모든 걸 PRIVATE → 외부가 부를 메서드까지 닫으면 가 쓸모없다. 공개할 것과 숨길 것을 나누는 게 설계.
  • PROTECTED 남용 → "자식이 이 내부 계약에 의존해도 된다"는 뜻. 확장 의도가 분명치 않으면 private으로 시작.
  • vs 정적 혼동 → 마다 다른 값 = DATA, 클래스 공용 = CLASS-DATA.

속성·메서드·범위를 정했다. 객체가 태어날 때 반드시 필요한 값을 받는 생성자 → Constructor와 객체 초기화.
