Database View와 Open SQL JOIN 비교

반복되는 JOIN을 DDIC에 등록해 재사용 — Database View vs 코드 JOIN.

Track 1Chapter 14Lesson 1중급

같은 JOIN을 또 짜기 싫다

Chapter 13 · JOIN에서 여러 테이블을 코드의 JOIN으로 붙였다. 즉시 이해하기는 좋지만, 같은 결합을 여러 프로그램이 반복하면 문제가 생긴다. 공연 조회· 예매 확인·정산 리포트가 모두 zconcert와 zperf를 같은 조건으로 붙인다면, 필드명이 바뀌거나 조건을 보강할 때 프로그램마다 고쳐야 한다.

는 이 반복 조회 구조를 DDIC에 이름 붙여 등록하는 방법이다.

Database View — JOIN을 DDIC에 등록

는 하나 이상의 basis table에서 필요한 컬럼을 골라 만드는 DDIC 객체다. 여러 테이블이면 inner join으로 결합한다. 활성화하면 DB 쪽에도 SQL View가 만들어져 ABAP SQL의 FROM 자리에서 테이블처럼 읽을 수 있다.

DATA: gt_concert TYPE STANDARD TABLE OF zv_concert,
      gs_concert TYPE zv_concert.

SELECT * FROM zv_concert
  INTO TABLE gt_concert.

LOOP AT gt_concert INTO gs_concert.
  WRITE: / gs_concert-concert_id, gs_concert-perf_name,
           gs_concert-date,       gs_concert-hall_id.
ENDLOOP.

프로그램은 JOIN 조건을 모르는 채로도 결합된 결과를 읽는다. 정의가 View 안에 들어 있기 때문이다.

📌 중요한 제한 둘. ① Database View는 데이터를 새로 저장하지 않는다(결과는 basis table에서 읽힌다). ② 여러 테이블 View는 inner join이라, 공연 코드는 있는데 공연 마스터가 없으면 그 행은 결과에서 사라진다.

View vs 코드 JOIN — 책임 위치가 다르다

비교 코드 JOIN
정의 위치 SE11의 DDIC 객체 ABAP 프로그램 소스
재사용 여러 프로그램이 같은 View 프로그램마다 작성
결합 DDIC에 고정된 inner join inner·left outer 등 선택
변경 영향 View 쓰는 프로그램 전체 해당 프로그램 중심
적합 안정적인 공통 조회 관점 조건·결합이 매번 다른 조회

아래에서 같은 결과가 "코드 JOIN"과 "Database View" 두 정의에서 나오는 걸 확인하고, 마스터가 빠졌을 때 inner join이 행을 어떻게 떨어뜨리는지 직접 보자.

흔한 실수와 점검

  • 테이블 복사본으로 오해 → View엔 데이터가 없다. basis table이 바뀌면 결과도 바뀐다.
  • 모든 JOIN을 View로 → 조건이 매번 다르거나, 누락 행도 보존해야 하거나, 계산 컬럼이 필요하면 View가 답이 아니다. 한쪽만 있어도 남기려면 코드의 LEFT OUTER JOIN(Chapter 13 · LEFT OUTER JOIN)이나 별도 검증 리포트를 쓴다.
  • View Field 과다 노출 → View Field는 외부 프로그램의 구조 계약이다. 실제 읽는 필드만 노출한다(나중에 제거하면 쓰던 프로그램이 깨진다).

정리

  • 는 반복 JOIN을 DDIC의 공통 조회 관점으로 옮기는 도구다.
  • 여러 테이블이면 inner join 성격이다. 짝 없는 행은 빠진다.
  • 저장소가 아니며, 누락 보존·복잡 계산의 만능 도구도 아니다.

여러 테이블을 합치는 View는 봤다. 그럼 한 테이블에서 필요한 필드만 좁게 보여 주는 View는? → 다음 레슨(Projection View).
