Editable Grid ALV와 입력 검증

Editable Field Catalog 설정

ALV에서 직접 입력받게 — 편집 가능 컬럼 설정.

Track 2Chapter 31Lesson 1고급

보여주기에서 입력받기로

지금까지 ALV는 조회 결과를 보여 주거나 에 반응하는 표였다. 값을 바꾸려면 별도 상세 화면이나 팝업을 열어야 했다. 하지만 좌석 수·수량·간단한 상태값처럼 표에서 바로 고치면 편한 업무가 있다 — ALV 셀을 입력칸으로 열어 준다.

핵심은 "전체를 입력 가능"이 아니라 "정말 입력받아도 되는 컬럼만" 연다는 것이다. 예매번호·고객번호·콘서트ID 같은 키 컬럼을 사용자가 바꾸면 DB 행과 화면 행의 연결이 끊긴다. 반대로 SEATS처럼 업무상 조정 가능한 값만 편집 가능하게 한다.

편집할 컬럼만 열기 — Field Catalog

Field Catalog(컬럼별 표시·동작 설정표)에서 특정 컬럼의 edit를 켜면, 그 컬럼은 표시 셀이 아니라 입력 가능한 셀로 그려진다.

LOOP AT lt_fcat REFERENCE INTO DATA(lr_fcat).
  IF lr_fcat->fieldname = 'SEATS'.
    lr_fcat->edit = abap_true.        " 이 컬럼만 편집 가능
  ENDIF.
ENDLOOP.

입력 이벤트 등록

edit만 켜면 "입력할 수 있는 모양"만 생긴다. 프로그램이 변경 사실을 받으려면 그리드에 편집 를 등록해야 한다.

ABAP
1
2
go_grid->register_edit_event(
  i_event_id = cl_gui_alv_grid=>mc_evt_modified ).
  • mc_evt_modified — 셀을 고치는 순간 변경 이벤트 발생.
  • mc_evt_enter — Enter로 확정할 때 발생. 어느 쪽을 쓸지는 화면 UX에 따라 고른다.

즉 edit(입력칸 만들기) 와 register_edit_event(변경 받을 준비) 는 한 쌍이다.

편집 모드 켜 보기

편집 모드를 토글하면 SEATS만 입력칸으로 바뀌고 키·표시 컬럼은 그대로인 걸 확인해 보라.

흔한 실수와 점검

  • 전체 컬럼에 edit 일괄 적용 → ⚠️ 사용자가 키 값을 바꾸면 DB 행과 화면 행의 관계가 끊겨, 저장 시 엉뚱한 행을 덮어쓰거나 insert처럼 동작한다. 키 컬럼은 보호, 업무상 허용된 값만 연다.
  • edit만 켜고 등록 누락 → 값은 보이는데 프로그램이 변경 흐름을 놓친다. edit와 register_edit_event를 한 쌍으로 기억한다.
  • 입력칸 보인다 ≠ 저장된다 → 여긴 입력 가능한 화면을 만드는 단계다. 변경 감지·검증은 DATA_CHANGED Event부터, 저장 전 검증은 이 챕터 뒤에서 다룬다.

정리

  • Editable ALV의 출발 = Field Catalog에서 필요한 컬럼만 edit로 연다.
  • 입력칸 만들기(edit)와 변경 감지(register_edit_event)는 다른 일 — 둘 다 필요하다.
  • 키 컬럼은 보호하고, 업무상 수정 가능한 컬럼만 연다.

편집 가능하게 했다. 이제 값이 바뀌는 순간을 잡아 검증하는 DATA_CHANGED → DATA_CHANGED Event(다음 레슨).
