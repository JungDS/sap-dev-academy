권한과 존재를 일찍 확인

본 처리에 들어가기 전, "이 사용자가 볼 권한이 있나", "입력한 키가 실제 존재하나" 를 선택화면 단계에서 확인하면 안전하다.

DATA lv_dummy TYPE ztdept-dept_id.

AT SELECTION-SCREEN.
  SELECT SINGLE dept_id FROM ztdept
    INTO lv_dummy
    WHERE dept_id = p_dept.        " classic (@ 없이)
  IF sy-subrc <> 0.
    MESSAGE '존재하지 않는 부서' TYPE 'E'.
  ENDIF.

권한 검사 (AUTHORITY-CHECK)

AUTHORITY-CHECK OBJECT 'Z_DEPT'
  ID 'DEPT'  FIELD p_dept
  ID 'ACTVT' FIELD '03'.          " 03 = 조회
IF sy-subrc <> 0.
  MESSAGE '권한이 없습니다' TYPE 'E'.
ENDIF.
  • AUTHORITY-CHECK는 권한 객체로 사용자 권한을 확인하고 sy-subrc로 결과를 준다.
  • 깊은 권한 설계는 실무 트랙(TRACK-02)에서. 지금은 "입력 단계에서 막는다"는 패턴만.

흔한 실수와 점검

  • 권한 검사 생략 → 민감 데이터는 반드시 AUTHORITY-CHECK.
  • modern 혼입 → classic 구간이라 @DATA/@ 자제(→ CH18).
