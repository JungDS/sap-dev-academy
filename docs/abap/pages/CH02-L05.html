변수·표준 타입·상수·Text Symbol

CONSTANTS — 값에 이름 붙이기

코드에 박힌 숫자·문자에 이름을 붙여, 뜻은 분명하게 변경은 안전하게.

Track 1Chapter 2Lesson 5입문

코드에 박힌 정체불명의 값들

코드 곳곳에 3.1415926, 'X', 10 같은 값을 직접 적으면 괴롭다. ① 이 값이 뭔지 한눈에 안 보이고(이런 값을 매직 넘버라 부른다), ② 바꿀 때 흩어진 걸 빠짐없이 찾아 고쳐야 하고, ③ 여러 번 적다 오타가 난다(3.1415926을 어디선 3.1415927로). 값에 이름을 붙여 한 곳에서 못 박으면 다 해결된다 — 그게 (상수)다.

CONSTANTS로 선언하기

와 거의 같지만 VALUE로 값을 못 박는다. 변수는 값을 받을 수 있지만, 상수는 한번 정하면 다시 대입할 수 없다.

DATA      gv_my_var TYPE i.
CONSTANTS gc_10     TYPE i VALUE 10.   " 동작만 보려는 이름(실제론 gc_pi처럼 뜻을 담는다)

gv_my_var = 10.                 " 오른쪽 값(10)을 왼쪽 변수에 넣는다
gv_my_var = gc_10.              " 상수에서 값을 읽어 변수에 넣는다 → 10
gv_my_var = gc_10 + gc_10.      " 상수로 계산한 결과를 변수에 넣는다 → 20

" gc_10 = 20.                   " ✗ 문법 오류 — 상수는 다시 대입할 수 없다
  • 대입은 오른쪽 → 왼쪽. 상수는 오른쪽에서 읽기만 된다.
  • gc_ 접두어가 관례(global constant), 변수는 gv_.
  • 상수를 왼쪽에 두고 재대입(gc_10 = 20.)하면 문법 오류 — 실행 전 단계에서 막힌다.

변수로도 되는데, 왜 상수일까

위 계산은 변수로도 할 수 있다. 하지만 변수는 코드 어딘가에서 실수로 덮어써질 수 있다. 원주율(π)처럼 절대 바뀌면 안 되는 값을 변수에 뒀다가 누군가 실수로 바꾸면, 그 뒤의 모든 계산이 조용히 틀어진다. 상수로 못 박으면 그런 사고가 원천 차단된다 (재대입 자체가 문법 오류). 아래에서 pi를 줄마다 직접 적은 코드(왼쪽)와 상수 하나로 묶은 코드(오른쪽)를 나란히 비교해 보자.

SAP가 미리 만들어 둔 상수도 있다

매번 'X'/공백을 직접 쓰는 대신, SAP가 제공하는 표준 상수 abap_true(='X')· abap_false(=공백)를 쓰면 뜻이 더 분명해진다(참/거짓 플래그용). 이 약속은 CH04(흐름 제어) 에서 조건문과 함께 다시 만난다.

💡 옛 코드에서 — 클래스가 흔해지기 전에는 전역 상수·타입을 Type Group(TYPE-POOLS로 불러 씀)에 모아 뒀다. 지금은 **클래스의 CONSTANTS**가 그 자리를 대신한다(Type Group은 구식). 옛 코드에서 TYPE-POOLS abap. 같은 줄을 보면 "전역 상수 묶음이구나" 정도로 읽으면 된다.

흔한 실수와 점검

  • VALUE 누락 → CONSTANTS gc_x TYPE i.는 문법 오류다. 변수()와 달리 상수는 선언할 때 VALUE로 값을 반드시 확정해야 한다.
  • 상수에 재대입 → 상수는 못 바꾼다. 바뀔 값이면 DATA(변수)로.
  • 여전히 매직 넘버 → "이 숫자가 뭐지?" 싶으면 상수로 이름 붙일 후보다.

정리

값에 이름을 붙여 코드가 읽히고, 변경이 안전해졌다. 다음은 화면에 보이는 글자 차례다 — '금액' 같은 텍스트를 코드에 박으면 다국어가 안 된다 → CH02-L06(Text Symbol).
