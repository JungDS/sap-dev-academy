Report Event·Selection Screen 심화

실습 — 예매현황 리포트 (이벤트·검증)

이벤트·검증 종합 — 기본값·화면제어·입력/존재/권한 검증·조회·표시를 이벤트마다 제자리에.

Track 1Chapter 15Lesson 12중급

🎫 종합 — 이벤트로 엮은 예매 조회 리포트

이 챕터의 마지막은 를 따로 외우는 게 아니라 한 리포트 흐름으로 묶는 실습이다. 사용자는 공연 코드를 보고(기본값) → 잘못된 입력을 고치고(검증) → 권한을 통과한 뒤 → 예매 목록을 조회한다. 이걸 한 번에 만들면 INITIALIZATION·AT SELECTION-SCREEN OUTPUT·AT SELECTION-SCREEN·START-OF-SELECTION이 각각 왜 필요한지 연결된다.

요구사항

  • 최초 실행 시 기본 공연 코드 C001을 제안한다.
  • 고급 조건 필드(pa_note)는 모드가 A일 때만 활성화한다.
  • 공연 코드가 비어 있거나 존재하지 않으면 실행을 막는다.
  • 현재 사용자에게 조회 권한이 없으면 실행을 막는다.
  • 검증을 통과하면 zbooking을 조회한다. 결과가 없으면 상태 메시지를, 있으면 SALV로 표시한다.

REPORT zch15_concert_booking.

PARAMETERS: pa_conc TYPE zconcert-concert_id,
            pa_mode TYPE c LENGTH 1,
            pa_note TYPE c LENGTH 20.

DATA: gv_chk     TYPE zconcert-concert_id,
      gv_status  TYPE zbooking-status,
      gt_booking TYPE STANDARD TABLE OF zbooking,
      lo_alv     TYPE REF TO cl_salv_table,
      gs_screen  TYPE screen.

SELECT-OPTIONS so_stat FOR gv_status.

INITIALIZATION.                              " ① 첫 기본값
  pa_conc = 'C001'.
  so_stat-sign = 'I'. so_stat-option = 'EQ'. so_stat-low = 'R'.
  APPEND so_stat.

AT SELECTION-SCREEN OUTPUT.                   " ② 화면 직전 제어
  LOOP AT SCREEN INTO gs_screen.
    IF gs_screen-name = 'PA_NOTE' AND pa_mode <> 'A'.
      gs_screen-active = '0'.                 " 고급 모드 아닐 때 숨김
      MODIFY SCREEN FROM gs_screen.
    ENDIF.
  ENDLOOP.

AT SELECTION-SCREEN ON pa_conc.                " ③ 입력·존재·권한 검증
  IF pa_conc IS INITIAL.
    MESSAGE '공연 코드를 입력하세요' TYPE 'E'.
  ENDIF.

  SELECT SINGLE concert_id FROM zconcert INTO gv_chk
    WHERE concert_id = pa_conc.
  IF sy-subrc <> 0.
    MESSAGE '없는 공연 코드입니다' TYPE 'E'.
  ENDIF.

  AUTHORITY-CHECK OBJECT 'Z_CONCERT'
    ID 'CONCERT' FIELD pa_conc
    ID 'ACTVT'   FIELD '03'.
  IF sy-subrc <> 0.
    MESSAGE '이 공연을 볼 권한이 없습니다' TYPE 'E'.
  ENDIF.

START-OF-SELECTION.                           " ④ 검증 후 본 처리(조회+표시)
  SELECT * FROM zbooking INTO TABLE gt_booking
    WHERE concert_id = pa_conc
      AND status IN so_stat.

  IF gt_booking IS INITIAL.
    MESSAGE '조회된 예매가 없습니다' TYPE 'S'.
    RETURN.
  ENDIF.

  TRY.
      cl_salv_table=>factory(
        IMPORTING r_salv_table = lo_alv
        CHANGING  t_table      = gt_booking ).
      lo_alv->display( ).
    CATCH cx_salv_msg.
      MESSAGE 'ALV 표시 중 오류가 발생했습니다' TYPE 'E'.
  ENDTRY.

💡 책임이 별로 나뉜다. 기본값=INITIALIZATION, 화면 제어=OUTPUT, 입력/존재/권한 검증=AT SELECTION-SCREEN ON pa_conc, 조회·표시=START-OF-SELECTION. SALV 표시는 END-OF-SELECTION의 위치와 경계에서 본 대로 legacy/LDB용 이벤트가 아니라 START-OF-SELECTION 안에 둔다(신규 단순 리포트엔 END-OF-SELECTION 불필요).

직접 실행해 보기 — 예매 리포트 실행 시뮬레이터

시나리오를 골라 ▶ 실행하면, 가 차례로 흐르며 어디서 통과하고 어디서 막히는지, 어떤 메시지가 나가고 결과 테이블이 어떻게 채워지는지 보인다. 상태 필터(so_stat)를 R↔C로 바꿔 결과가 달라지는 것도 확인하자.

검증 시나리오

시나리오 입력 기대 결과
기본 실행 pa_conc=C001, so_stat=R 예매 목록 조회
코드 누락 pa_conc 공백 E 메시지 · 화면 복귀
없는 공연 pa_conc=C999 E 메시지 · 화면 복귀
권한 없음 pa_conc=C002(권한 없음) E 메시지 · 화면 복귀
결과 없음 존재·권한 OK, 0건 S 메시지 · 종료
고급 모드 pa_mode=A pa_note 입력 가능

흔한 실수와 주의

  • 검증을 START-OF-SELECTION으로 미룸 → 입력·존재·권한 검증은 조회 전 AT SELECTION-SCREEN에서 한다. 권한 검증을 조회 뒤로 미루지 않는다.
  • 필드를 숨기면 값도 사라진다고 착각 → pa_note를 숨겨도 내부 변수 값은 그대로다. 숨김과 값 삭제는 다른 문제다.
  • END-OF-SELECTION을 통합 실습에 넣기 → 신규 단순 리포트는 START-OF-SELECTION 하나로 충분하다.
  • 메시지 문구만 같게 → "없는 코드"와 "권한 없음"은 원인이 다른 문제다. 다른 문구로 막는다.

정리

이 실습의 완성 기준은 "코드가 돈다"가 아니라 책임이 별로 분리되어 있느냐다. 사용자가 값을 보기 전·입력한 뒤·실행하기 전·조회를 시작한 뒤의 일이 제자리에 있으면, 리포트는 읽기 쉬워지고 오류는 빨리 잡히며 피드백은 정확해진다.
