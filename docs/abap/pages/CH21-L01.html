Global Class 생성과 객체

SE24로 전역 클래스를 만들고 객체를 생성한다 — 흩어진 로직을 한 덩어리로.

Track 1Chapter 21Lesson 1고급

왜 — 흩어진 로직을 한 개념으로 🧩

지금까지 콘서트 예매 로직은 FORM get_remaining, FORM check_capacity, FORM book_seats처럼 따로따로 있었다. 처음엔 단순해 보인다. 그런데 화면 프로그램·ALV 리포트·배치가 같은 예약 판단을 재사용하기 시작하면 문제가 드러난다.

  • 어느 FORM이 최신 규칙인지 헷갈린다.
  • 공연 ID·회차 같은 상태를 매번 로 넘겨야 한다.
  • 정원 초과 처리가 프로그램마다 조금씩 달라진다.

"예약 관리자"라는 업무 개념은 분명히 있는데, 코드에는 그 개념이 흩어져 있다. 데이터(잔여석)와 행동(예약·취소)을 한 덩어리로 묶고 싶다. 그게 이고, 그 틀이 다.

🎫 콘서트 예매 앱의 핵심 로직을 ZCL_BOOKING_MANAGER 클래스로 묶어 간다.

전역 클래스 — SE24 🏛️

SE24(또는 ADT)에서 ZCL_BOOKING_MANAGER를 만든다. Local Class로 모듈화에서 봤듯, 는 DEFINITION(밖에서 무엇을 볼 수 있는지·어떤 속성/가 있는지)과 IMPLEMENTATION(메서드가 실제로 하는 일)으로 나뉜다.

CLASS zcl_booking_manager DEFINITION
  PUBLIC
  CREATE PUBLIC.
  PUBLIC SECTION.
    METHODS book IMPORTING iv_seats TYPE i.
ENDCLASS.

CLASS zcl_booking_manager IMPLEMENTATION.
  METHOD book.
    " 예약 처리는 뒤 레슨에서 채운다.
  ENDMETHOD.
ENDCLASS.

객체 만들기 — NEW 🏗️

만 선언했다고 예약 관리자가 생긴 건 아니다. 실제로 일을 시키려면 **()**를 만들어야 한다. 객체를 만드는 연산자가 NEW다.

ABAP
1
2
DATA(lo_mgr) = NEW zcl_booking_manager( ).   " 객체 생성
lo_mgr->book( iv_seats = 2 ).                " -> 로 인스턴스 메서드 호출

lo_mgr는 객체 자체가 아니라 객체를 가리키는 참조 변수다. classic 방식으로 쓰면 이 구조가 더 또렷하게 보인다.

ABAP
1
2
3
DATA lo_mgr TYPE REF TO zcl_booking_manager.   " 객체를 담을 변수
CREATE OBJECT lo_mgr.                          " 객체 생성(classic)
lo_mgr->book( iv_seats = 2 ).

NEW zcl_booking_manager( )와 CREATE OBJECT lo_mgr는 둘 다 객체를 만들고 instance constructor를 호출한다. 앞으로는 modern NEW를 쓰되, 기존 코드·SAP GUI 예제에서 CREATE OBJECT를 만나도 같은 개념으로 읽으면 된다.

흔한 실수와 점검 🚧

  • 객체 없이 호출 → 참조 변수 선언은 빈 명함을 만든 것일 뿐. NEW(또는 CREATE OBJECT)로 객체부터 만든다. lo_mgr가 initial이면 lo_mgr->book( )은 실패한다.
  • = 객체 라고 착각 → zcl_booking_manager는 설계도, lo_mgr는 그 설계도로 만든 특정 객체를 가리키는 변수. 같은 클래스로 객체를 여러 개 만들면 각자 다른 상태를 가진다.
  • 무조건 Global Class → 한 프로그램 안에서만 짧게 쓰는 보조 구조면 Local Class도 된다. 여러 프로그램·후속 챕터가 공유할 업무 로직이라야 Global Class가 제값을 한다.

클래스와 객체를 만들었다. 이제 그 안에 담을 속성·메서드와 공개 범위를 보자 → Attribute / Method / Visibility.
