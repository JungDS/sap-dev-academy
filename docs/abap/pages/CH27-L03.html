IF 사다리로 분기가 길어질 때

"일반/VIP/조기예매" 요금 계산을 IF/CASE로 길게 나누면 추가할 때마다 그 코드를 건드려야 한다. Strategy는 각 방식을 구현으로 분리해 갈아끼운다.

INTERFACE zif_price_strategy.
  METHODS calc IMPORTING iv_seats TYPE i RETURNING VALUE(rv) TYPE p.
ENDINTERFACE.

" 사용: 전략을 주입받아 그대로 호출
METHOD checkout.
  DATA(lv_price) = mo_strategy->calc( iv_seats = 2 ).
ENDMETHOD.
  • 새 요금 정책 = 새 추가(기존 코드 불변 — OCP 원칙).
  • 어떤 전략을 쓸지는 Factory Pattern·설정으로 주입(선택과 실행을 나눈다).

전략을 갈아끼워 보기

전략을 바꿔도 호출부(checkout)는 한 줄 그대로인데, 가격 계산이 통째로 바뀌는 걸 확인해 보라. 좌석 수도 바꿔 가며.

흔한 실수와 점검

  • 전략 안에 전략 분기 → 전략 내부에서 또 IF로 나누면 의미 없다. 한 전략 = 한 방식.
  • 과설계 → 방식이 1~2개로 고정이면 Strategy가 과할 수 있다(IF 하나로 충분할 때도). 패턴은 미래를 예언하는 도구가 아니라 실제 변화 압력을 관리하는 도구다.
  • 선택 책임을 호출부에 → checkout이 "VIP면 이 class, 조기면 저 class"까지 판단하면 다시 정책 목록을 알게 된다. 선택은 Factory/설정, 실행은 Strategy로 나눈다.
  • 전략에 DB조회·메시지·화면을 → 요금 strategy는 계산만 맡는다. 화면 메시지는 controller/view, DB 조회는 model/repository로 분리해야 테스트가 쉽다(다음 레슨 MVC).

정리

  • Strategy = 갈아끼울 알고리즘을 구현으로 분리, 호출부는 그대로.
  • 새 정책 = 새 (OCP). 전략 선택은 팩토리·설정으로 주입.

동작을 유연하게 했다. 이제 화면·로직·데이터를 분리하는 구조 → MVC(다음 레슨).
