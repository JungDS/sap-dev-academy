INSERT / UPDATE / MODIFY / DELETE 실무 기준

직접 DB를 바꾼다 — 네 가지 DML과 감사필드.

CH08은 읽기였다 — 이제 쓰기

CH08에서 배운 SELECT는 읽기 전용이었다. 데이터를 화면에 보여 줄 수는 있었지만, "예매가 들어왔다", "취소됐다" 같은 변화를 DB에 남기지는 못했다. 실무 프로그램의 절반은 결국 데이터를 바꾸는 일이다.

DB의 행을 직접 바꾸는 네 개의 문장을 묶어 (Data Manipulation Language)이라고 부른다. 읽기(SELECT)와 짝이 되는 쓰기 4형제다.

문 한 줄 동작 키가 이미 있으면 키가 없으면
INSERT 새 행 추가 오류(덤프) 추가
UPDATE 기존 행 수정 수정 아무 일 없음(sy-subrc=4)
MODIFY 있으면 수정·없으면 추가(upsert) 수정 추가
DELETE 행 삭제 삭제 아무 일 없음(sy-subrc=4)

INSERT zbooking FROM @ls_booking.                          " 새 예매 1건
UPDATE zbooking SET status = 'X' WHERE booking_id = @lv_id. " 상태만 수정
MODIFY zbooking FROM @ls_booking.                          " 있으면 수정·없으면 추가
DELETE FROM zbooking WHERE booking_id = @lv_id.            " 예매 취소

네 문장의 차이가 헷갈리는 게 정상이다. 아래에서 직접 실행해 보면 한 번에 잡힌다.

직접 바꿔 보기

아래 미니 ZBOOKING에 네 연산을 실행해 보자. 같은 키로 INSERT를 두 번 하면 무슨 일이 일어나는지, UPDATE/DELETE에서 WHERE를 빼면 몇 행이 바뀌는지 눈으로 확인해 보라.

한 건이 아니라 통째로 — FROM TABLE

한 행씩 넣는 대신, 의 내부 테이블을 한 번에 DB로 보낼 수 있다. CH07에서 손으로 한 줄씩 입력하던 번거로움, CH08에서 "조회 결과 itab을 통째로 저장하고 싶다"던 아쉬움이 여기서 풀린다. 행마다 왕복하지 않아 훨씬 빠르다.

ABAP
1
2
INSERT zgugudan FROM TABLE @lt_gugu.        " 81행을 한 번에
MODIFY zbooking FROM TABLE @lt_booking.     " 여러 건 upsert

누가·언제 바꿨나 — 감사필드 자동 stamp

업무 테이블에는 보통 "이 행을 누가·언제 만들었나"를 남기는 컬럼(created_by· created_on)을 둔다. 나중에 문제를 추적하거나 책임을 가릴 때 필수다. 값은 CH04에서 만난 로 채운다 — 시스템이 로그온 사용자·날짜·시각을 알아서 들고 있다.

ls_booking-created_by = sy-uname.   " 로그온 사용자 ID
ls_booking-created_on = sy-datum.   " 오늘 날짜
ls_booking-created_at = sy-uzeit.   " 현재 시각
INSERT zbooking FROM @ls_booking.

위 플레이그라운드에서 INSERT/MODIFY로 새 행을 만들면 created_by·created_on이 자동으로 찍히는 걸 볼 수 있다.

⚠️ 표준 테이블은 직접 쓰지 않는다

SCARR·SFLIGHT 같은 SAP 표준 테이블은 INSERT/UPDATE로 직접 바꾸면 안 된다. 표준 데이터는 보이지 않는 검증·후속 처리(다른 테이블 동기화, 번호 채번 등)가 딸려 있는데, 직접 은 그걸 모두 건너뛰어 데이터를 조용히 망가뜨린다. 표준 데이터는 반드시 BAPI/표준 API를 경유한다(자세히는 CH30). 직접 DML은 내가 만든 테이블(ZBOOKING 등)에만 쓴다.

흔한 실수와 점검

  • INSERT 중복키 → 이미 있는 키면 런타임 덤프. "있으면 갱신"이 목적이면 MODIFY.
  • WHERE 없는 UPDATE/DELETE → 전체 행이 바뀌거나 사라진다. 사고의 단골 — 조건을 반드시.
  • 표준 테이블 직접 → 금지. BAPI 경유(CH30).
  • sy-subrc 미확인 → 변경이 정말 됐는지 항상 확인(다음 레슨의 COMMIT 판단과 직결).

정리

  • 쓰기 4형제 = INSERT(추가)·UPDATE(수정)·MODIFY(upsert)·DELETE(삭제).
  • 여러 건은 FROM TABLE로 한 번에 — 빠르다.
  • 업무 테이블엔 감사필드(created_by·created_on)를 SY 변수로 stamp.
  • 표준 테이블은 직접 금지 → BAPI.

데이터를 바꾸는 법은 익혔다. 그런데 이 변경은 언제 진짜 확정되고, 중간에 실패하면 어떻게 되돌릴까? COMMIT/ROLLBACK → 다음 레슨.
