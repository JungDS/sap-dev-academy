읽기를 넘어 — 쓰기·검증·서비스까지

CDS View Entity 기본 구조에서 만든 CDS는 데이터를 읽기/모델링하는 데 강하다. ZI_Concert·ZC_Concert를 만들면 프로그램이 테이블을 직접 읽지 않고 재사용 가능한 모델을 소비할 수 있다. 하지만 예매 앱의 핵심은 읽기에서 끝나지 않는다.

사용자는 예매를 생성하고, 좌석 수를 수정하고, 취소 버튼을 누르고(삭제), 저장 전에 정원 초과를 검증해야 한다. 그리고 그 기능은 Fiori 화면이나 OData API로 노출되어야 한다. 이걸 프로그램마다 직접 짜면 구조가 금방 무너진다 — 화면·리포트·서비스마다 저장 로직·권한·메시지가 제각각이 된다.

이 전부를 하나의 비즈니스 로 묶는 현대 SAP 표준이 (RESTful Application Programming Model)다.

RAP는 "파일 하나"가 아니다 — 계층의 연결

계층 역할
Root Interface View RAP BO의 중심 entity(뿌리). 보통 define root view entity로 만든다.
Projection View 외부에 보여 줄 소비 모델(ZC_*).
BDEF (Behavior Definition) 어떤 operation을 허용할지 선언(create·update·delete·validation·determination·action).
Behavior Pool BDEF에 선언한 custom 동작의 실제 ABAP 코드를 담는다.
Service Definition 어떤 CDS entity를 서비스로 노출할지 선언.
Service Binding OData 같은 protocol과 연결해 실제 호출 가능한 서비스로 만든다.

managed vs unmanaged

시나리오 누가 저장하나
managed 런타임/provider가 표준 저장 처리의 상당 부분을 자동으로 맡는다.
unmanaged 기존 로직·특수 저장 절차를 개발자가 더 직접 구현한다(레거시 통합).

신규 학습·신규 개발은 보통 managed에서 시작한다(이해하기 좋고 코드가 적다). 단, managed라고 "코드 0"은 아니다 — validation·action·메시지·권한 같은 custom 로직은 behavior pool에 들어간다.

직접 조립해 보기

아래 보드에서 6계층을 하나씩 켜 보자. 빠진 계층마다 어떤 사용자 행동이 불가능해지는지가 핵심이다. "이 파일이 무엇을 하나"보다 "이 파일이 없으면 무엇을 못 하나"로 보면 구조가 또렷해진다.

어떻게 확인하는가

파일 이름을 외우기보다 **"계층이 빠짐없이 이어지는가"**를 본다.

확인 질문 정상 신호
root entity가 있는가? define root view entity ZI_Booking ...가 있다.
소비 projection이 있는가? ZC_Booking이 ZI_Booking을 projection한다.
behavior가 선언되어 있는가? BDEF에 define behavior for ZI_Booking이 있다.
동작 코드 위치가 있는가? behavior pool의 local handler class가 있다.
서비스로 노출되는가? service definition이 expose ZC_Booking as Booking;을 포함한다.
protocol binding이 있는가? Service Binding이 활성화되어 URL/preview를 제공한다.

흔한 실수와 점검 🚧

  • 를 CDS의 다른 이름으로 생각 → CDS는 RAP의 data model을 구성하지만, RAP는 behavior와 service까지 포함한다.
  • create/update/delete를 직접 DB DML과 혼동 → 이건 BDEF에서 BO operation을 허용하는 선언이다. 테이블을 직접 바꾸는 SQL과 LUW는 INSERT / UPDATE / MODIFY / DELETE 실무 기준에서 다룬다.
  • managed = 코드 0으로 오해 → 표준 저장은 줄지만 custom 로직은 behavior pool에 들어간다.
  • 서비스부터 만들기 → 아래 계층(root·BDEF)이 부실하면 서비스가 있어도 트랜잭션 앱으로 동작하지 않는다.

큰 그림을 봤다. 이제 RAP의 토대인 root Interface View부터 아래에서 위로 하나씩 쌓자 → Interface View ZI_* 설계 (Root).
