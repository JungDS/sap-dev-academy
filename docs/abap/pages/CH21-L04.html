OO ABAP 기본 설계

Static Method와 Instance Method

클래스로 직접(=>) vs 객체로(->) — 객체 상태가 필요한가로 가른다.

Track 1Chapter 21Lesson 4고급

객체가 필요한가, 아닌가 ⚖️

모든 에 가 필요한 건 아니다. 공연 ID를 대문자로 정리하는 일은 특정 예약 객체의 상태 없이도 가능하다. 반대로 잔여석 계산은 그 공연·회차를 알아야 하니 객체 상태가 필요하다.

  • 멤버(METHODS·DATA) — 객체마다 따로. 객체로 -> 호출.
  • 정적 멤버(CLASS-METHODS·CLASS-DATA) — 공용. 객체 없이 => 호출.

" 인스턴스 — 객체가 필요
DATA(lo_mgr) = NEW zcl_booking_manager( iv_concert = 'C001' iv_perf = '01' ).
DATA(lv_remaining) = lo_mgr->remaining( ).

" 정적 — 클래스로 직접(객체 없이)
DATA(lv_concert) = zcl_booking_manager=>normalize_concert_id( 'c001' ).

me-> — 메서드 안에서 자기 자신 🪞

안에서 자기 의 다른 멤버를 가리킬 때 me->를 쓴다. 그 객체의 상태(mv_concert 등)로 잔여석을 계산하는 메서드는 이렇게 생겼다.

METHOD remaining.
  SELECT COALESCE( SUM( seats ), 0 )
    FROM zbooking
    WHERE concert_id = @mv_concert
      AND perf_no    = @mv_perf
      AND status    <> 'C'
    INTO @DATA(lv_booked).

  rv_remaining = mv_cap - lv_booked.
ENDMETHOD.

me->는 대부분 생략할 수 있지만, 지역 변수·와 이름이 비슷할 때 me->mv_cap처럼 쓰면 의도가 분명해진다.

언제 정적, 언제 인스턴스 🧭

  • 상태가 필요 없는 유틸리티(포맷·계산) → 정적.
  • 객체별 상태(잔여석 등)를 다루면 → .

🧷 =>와 ->를 기계적으로 외우지 말고 왼쪽에 무엇이 오는지 보자. 명이 왼쪽이면 =>, 참조 변수가 왼쪽이면 ->다.

흔한 실수와 점검 🚧

  • 상태를 읽는 를 정적으로 → 잔여석은 공연·회차마다 다르니 여야 한다.
  • 단순 helper를 전부 인스턴스로 → 객체 상태가 필요 없는 포맷 변환은 정적이 더 자연스럽다(괜히 객체를 만들지 않는다).
  • 정적 메서드에서 인스턴스 멤버 → 정적 메서드 안엔 자기 객체(me)가 없다. 인스턴스 멤버를 못 쓴다.
  • CLASS-DATA를 전역 변수처럼 → 여러 객체가 공유하므로, 한 객체의 변경이 다른 객체에도 영향을 줄 수 있다.

호출 방식을 정리했다. 이제 종류가 달라도 같은 규약을 따르게 하는 Interface → Interface 기본 설계.
