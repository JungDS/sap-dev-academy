🎫 조각을 하나의 예매 흐름으로

이제 이 챕터의 조각을 하나의 콘서트 예매 흐름으로 묶는다. 목표는 "RAP 앱 전체를 완벽히 운영 배포한다"가 아니라, RAP 구조를 실제 예매 시나리오에 연결해 보는 것이다. 실습의 핵심은 둘이다 — 정원 초과 예매는 validation으로 막고, 취소는 action으로 명시 실행한다. 여기에 생성 시 기본 상태를 determination으로 채우면 세 로직의 차이를 한 번에 확인할 수 있다.

저장은 언제 일어나나 — EML 개념 지도

입문자가 RAP에서 가장 크게 오해하는 지점은 "예매를 만들면 곧바로 DB에 저장된다"는 것이다. 실제로는 두 단계다. 예매자 정훈영이 새 예매를 넣으면, 그 변경은 먼저 (RAP가 변경을 잠시 모아 두는 )에만 반영된다. DB에 실제로 쓰이는 것은 그다음이다.

프로그램에서 RAP BO를 직접 다루는 언어를 (Entity Manipulation Language)이라 한다. 흐름은 이렇다.

문장 무엇을 하나
MODIFY ENTITIES 생성·수정·삭제 요청을 buffer에 넣는다(아직 DB 아님).
COMMIT ENTITIES 를 트리거해 buffer를 검사·조정한 뒤 DB에 all-or-nothing으로 저장한다.
ROLLBACK ENTITIES 현재 트랜잭션의 buffer·lock을 정리(취소)한다.

핵심 두 가지만 기억하면 된다.

  • 저장 트리거는 COMMIT ENTITIES다. MODIFY ENTITIES 직후 테이블을 SELECT하면 아직 안 보일 수 있다 — 버퍼에만 있으니까.
  • behavior pool(provider)은 스스로 COMMIT하지 않는다. 위 validate_capacity·cancel 안에서 우리가 본 IN LOCAL MODE는 같은 BO를 내부에서 다루는 특수 형태이고, 실제 저장 확정은 Fiori/OData 런타임이나 외부 consumer가 맡는다.

🔎 여기서는 "언제 저장되나"라는 개념 지도까지만. 외부 프로그램이 MODIFY → COMMIT/ROLLBACK ENTITIES로 RAP BO를 직접 호출하고 failed/reported를 처리하는 실전 코드, 그리고 Draft·Lock·ETag 같은 동시성 심화는 RAP 심화 Capstone에서 다룬다.

흔한 실수와 점검 🚧

  • validation 실패를 화면 메시지로만 → 실패 instance를 failed에, 사용자 메시지를 reported에 기록해야 runtime·UI가 제대로 반응한다.
  • 정원 계산을 현재 입력 좌석만 보기 → 이미 예약된 좌석·취소 제외·같은 회차 합계·현재 transaction 변경분을 함께 봐야 한다(실습에선 이 기준을 설계 질문으로 적어 둔다).
  • action과 determination 섞기 → 취소는 사용자가 명시 실행하므로 action, 생성 시 기본 상태는 자동 결정이므로 determination이 자연스럽다(Validation / Determination / Action 개요).
  • IN LOCAL MODE 남용 → behavior implementation에서 해당 RAP BO 자체를 읽는 특수 형태다(Behavior Implementation 기초). 일반 소비 프로그램에 습관적으로 붙이는 문법이 아니다.
  • 직접 DML로 우회 → behavior pool 안에서 테이블을 직접 바꾸면 RAP buffer·를 우회할 수 있다. 직접 DB 변경은 INSERT / UPDATE / MODIFY / DELETE 실무 기준에서 따로 다룬다.

이 과정에서 키운 콘서트 예매 앱이 RAP 트랜잭션 앱으로 완성됐다. Track-1은 "읽기 프로그램"에서 "현대 SAP 트랜잭션 앱의 기본 구조"까지 한 바퀴를 돌았다. 이제 Track-2 실무 디테일 — 직접 DB 변경(DML) → INSERT / UPDATE / MODIFY / DELETE 실무 기준.
