오류 로그와 재처리 구조

변경이 실패했을 때 — 기록하고 다시 처리한다.

실패는 반드시 남긴다

대량 변경에서 일부가 실패하면, 무엇이·왜 실패했는지 남겨야 나중에 재처리할 수 있다. 매 변경 뒤 sy-subrc를 확인하고, 실패 건을 로그/오류 테이블에 모은다.

DATA lt_error TYPE TABLE OF zbooking.

LOOP AT lt_booking INTO DATA(ls).
  INSERT zbooking FROM @ls.
  IF sy-subrc <> 0.
    APPEND ls TO lt_error.        " 실패 건 모으기
  ENDIF.
ENDLOOP.
  • 표준 Application Log(BAL, SLG1)에 메시지를 남기는 방식도 흔하다(정식 학습은 CH35-L05).
  • 실패 건을 별도 테이블/스풀에 남겨 재처리 잡으로 다시 돌린다.

재처리 패턴

  • 실패 건만 모아 두었다가 원인 해결 후 다시 실행.
  • 멱등성(같은 걸 두 번 처리해도 안전)을 고려 — MODIFY(upsert)·중복 체크.

흔한 실수와 점검

  • 실패 무시 → 일부 실패를 모른 채 COMMIT. 항상 sy-subrc 점검.
  • 로그 없음 → 무엇이 실패했는지 못 찾는다. 최소한 키·사유 기록.

실패를 다뤘다. 대량 처리는 ·성능도 챙겨야 한다 → Package 처리(다음 레슨).
