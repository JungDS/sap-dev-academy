잠그기 — ENQUEUE

앞 레슨에서 만든 Lock Object를 활성화하면 ENQUEUE_EZ_BOOKING·DEQUEUE_EZ_BOOKING 함수가 자동으로 생겼다. 우리는 를 호출(CH10의 CALL FUNCTION)만 하면 된다. 키를 넘기면 그 행만 잠긴다.

CALL FUNCTION 'ENQUEUE_EZ_BOOKING'
  EXPORTING  booking_id     = lv_id
  EXCEPTIONS foreign_lock   = 1     " 이미 남이 잠갔음
             system_failure = 2
             OTHERS         = 3.

IF sy-subrc <> 0.
  MESSAGE '다른 사용자가 편집 중입니다' TYPE 'E'.   " 정중히 거절
ENDIF.

핵심은 foreign_lock 다. 이미 다른 사용자가 잠근 행을 또 잠그려 하면 여기로 떨어진다(sy-subrc=1). 이걸 무시하고 변경하면 동시 충돌이 나므로, 반드시 분기해서 "편집 중"이라고 안내하고 멈춰야 한다.

풀기 — DEQUEUE

ABAP
1
2
CALL FUNCTION 'DEQUEUE_EZ_BOOKING'
  EXPORTING booking_id = lv_id.

두 사용자로 직접 — 누가 이기나?

아래에서 User A와 B가 같은 예매(1001)를 두고 잠금을 다툰다. A가 먼저 ENQUEUE한 뒤 B가 ENQUEUE하면 어떻게 되는지, A가 DEQUEUE/COMMIT하면 B가 언제 가능해지는지 직접 눌러 보라. 가운데 SM12(잠금 목록)에 누가 잠갔는지 뜬다.

표준 패턴 — 잠금부터 해제까지

  1. ENQUEUE(잠금) → 2. 읽기 → 3. 변경(DML) → 4. COMMIT WORK(확정) → 5. DEQUEUE(해제).

COMMIT WORK/ROLLBACK WORK는 보통 잠금을 자동 해제한다(다음 레슨). 그래도 흐름을 명확히 하려고 DEQUEUE를 명시하기도 한다.

흔한 실수와 점검

  • foreign_lock 미처리 → 이미 잠긴 걸 무시하면 충돌. sy-subrc 분기로 거절·안내.
  • 잠금만 하고 안 풂 → 다른 사용자가 영영 못 쓴다. 반드시 DEQUEUE(또는 COMMIT 자동 해제).
  • 잠금을 너무 오래 쥠 → 편집 화면을 켜 둔 채 자리를 비우면 남이 막힌다(다음 레슨).

정리

  • 활성화로 생긴 ENQUEUE_*/DEQUEUE_*를 호출해 잠그고 푼다.
  • foreign_lock(이미 남이 잠금)은 무시 말고 정중히 거절한다.
  • 패턴 = ENQUEUE → 읽기 → 변경 → COMMIT → (DEQUEUE).

잠그고 푸는 법을 익혔다. 그런데 언제 자동으로 풀리고, 는 어떻게 다룰까 → 다음 레슨.
