실무 데이터 변경과 트랜잭션 제어

COMMIT WORK / ROLLBACK WORK

변경을 확정하거나 되돌린다 — 트랜잭션 제어.

바꿨다고 끝이 아니다

앞 레슨에서 INSERT/UPDATE로 데이터를 바꿨다. 그런데 사실 DML 문은 변경을 예약할 뿐이다. 그 예약을 진짜로 확정해 디스크에 못 박는 것이 이고, 중간에 문제가 생기면 예약을 통째로 취소해 직전 상태로 되돌리는 것이 다.

DATA lv_failed TYPE abap_bool.

INSERT zbooking FROM @ls_fri.                " 금요일 회차 예매
IF sy-subrc <> 0. lv_failed = abap_true. ENDIF.

INSERT zbooking FROM @ls_sat.                " 토요일 회차 예매
IF sy-subrc <> 0. lv_failed = abap_true. ENDIF.

IF lv_failed = abap_false.
  COMMIT WORK.        " 둘 다 성공했을 때만 확정
ELSE.
  ROLLBACK WORK.      " 하나라도 실패하면 둘 다 취소 — 반쪽 저장 방지
ENDIF.

마지막 한 줄만 보면 안 된다. 두 INSERT 뒤에 IF sy-subrc = 0을 한 번만 쓰면 두 번째 문장만 보게 되어 첫 INSERT 실패를 놓친다. COMMIT WORK는 "마지막 문장이 성공"이 아니라 **"업무 단위 전체가 성공"**일 때만 호출한다 — 그래서 문장마다 결과를 누적한다.

왜 중요한가 — 전부 아니면 전무 (원자성)

정훈영이 금·토 두 회차를 한 묶음으로 예매한다고 하자. 두 예매는 업무상 함께 성립해야 한다 — "금요일은 잡혔는데 토요일은 실패"한다면? 고객은 한 자리만 얻고 다른 자리는 놓친, 반쪽짜리 묶음 예매가 남는다. 이런 반쪽 저장을 막으려고, 묶여야 할 변경들을 하나의 단위로 다룬다 — 전부 확정 (COMMIT)되거나 전부 취소(ROLLBACK)되거나. 이 "전부 아니면 전무" 성질이 원자성이다.

아래에서 둘째 INSERT를 일부러 실패시키고 COMMIT을 눌러, 반쪽 저장이 실제로 어떻게 나는지 직접 만들어 보라. 그리고 올바른 선택(ROLLBACK)과 비교해 보라.

COMMIT 시점은 신중하게

  • 너무 늦으면(누락) → COMMIT을 안 하면 변경이 확정되지 않아 사라질 수 있다.
  • 너무 자주(남발) → 묶여야 할 작업을 중간중간 쪼개 COMMIT하면, 또다시 반쪽 저장 위험. 하나의 논리 단위(=업무상 함께 성공/실패해야 하는 묶음) 단위로만 COMMIT한다.
  • COMMIT WORK AND WAIT → 저장이 완료된 뒤 다음 코드를 진행해야 할 때 쓴다 (기본 COMMIT WORK는 저장을 비동기로 맡기고 바로 진행 — 자세한 동작은 다음 레슨의 LUW).

흔한 실수와 점검

  • COMMIT 누락 → 변경이 확정 안 돼 사라진다. 작업 끝에 확정 책임을 분명히.
  • 실패 무시 후 COMMIT → 반쪽 저장. sy-subrc=4인 UPDATE는 dump가 아니라 조용히 지나가니, 반드시 결과를 보고 실패면 ROLLBACK. "조용히 지나감"은 성공이 아니다.
  • 남발하는 COMMIT → 논리 단위를 쪼개지 말 것.
  • DB cursor 중간에 COMMIT/ROLLBACK → ROLLBACK WORK는 열린 database cursor를 닫을 수 있다. SELECT … ENDSELECT 같은 긴 cursor 흐름 중간에 제어를 넣는 습관은 피한다.

정리

  • DML은 예약 → COMMIT WORK로 확정, ROLLBACK WORK로 전부 취소.
  • 묶음은 원자성(전부 아니면 전무)으로 — 반쪽 저장을 막는다.
  • 실패하면 COMMIT이 아니라 ROLLBACK. 완료를 기다려야 하면 AND WAIT.

확정·취소를 배웠다. 그런데 ""이라는 말이 DB와 SAP에서 의미가 다르다. 그리고 COMMIT WORK 한 줄이 실제로 어떻게 저장으로 이어지는지 → LUW(다음 레슨).
