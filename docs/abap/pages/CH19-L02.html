New Open SQL / Modern ABAP SQL

@ Host Variable과 Host Expression

ABAP 값을 SQL에 안전하게 넣는다 — @변수와 @( 식 ).

Track 1Chapter 19Lesson 2중급

왜 — SQL은 DB에서, 변수는 ABAP에서 🌉

SELECT 문장은 데이터베이스에서 실행된다. 그런데 ABAP 변수 값은 애플리케이션 서버 에 있다. 이 둘을 이으려면 "이 이름은 DB 컬럼이 아니라 ABAP 값이다"라는 표시가 필요하다. 그 표시가 바로 @다.

classic에서는 이게 이름 규칙(lv_)에만 기댔다 — 문법으로는 드러나지 않았다.

@ Host Variable — "이건 ABAP 값이다" 🏷️

ABAP 프로그램에서 선언한 데이터 객체(변수·상수·필드)를 SQL 문장의 자리(operand)에 넣은 것이 host variable이다. @는 그것을 SQL 안에서 host variable로 알아보게 하는 escape 문자다.

DATA(lv_carr) = 'LH'.
DATA(lv_date) = sy-datum.

SELECT *
  FROM sflight
  WHERE carrid = @lv_carr
    AND fldate >= @lv_date
  INTO TABLE @DATA(lt_flight).

여기서 carrid·fldate는 DB 컬럼, @lv_carr·@lv_date는 ABAP 값이다.

@( ) Host Expression — 식도 통째로 넘긴다 🧮

변수 하나가 아니라 계산식의 결과도 SQL에 넘길 수 있다. 이때는 @( ... )로 감싼다 — 이것이 host expression이다.

ABAP
1
2
3
4
5
6
DATA(lv_base) = 100.

SELECT carrid, connid, seatsmax
  FROM sflight
  WHERE seatsmax > @( lv_base + 100 )
  INTO TABLE @DATA(lt_large).

@( lv_base + 100 )은 ABAP에서 계산한 결과를 SQL operand로 넘긴다. 단, 아무 식이나 밀어 넣는 통로가 아니다 — 그 결과는 SQL 자리가 요구하는 타입으로 lossless(값 손실 없이) 변환될 수 있어야 한다.

🧩 "식"이 둘이라 헷갈린다 — 실행되는 곳이 다르다. @( lv_base + 100 )(host expression)은 ABAP에서 계산해 SQL에 넘긴다. 반면 CASE·COALESCE·SUBSTRING 같은 SQL expression(다음 레슨들)은 DB에서 계산된다.

세 가지로 확인하기 ✅

일부러 다음 세 경우를 실행해 본다.

  1. 정상 변수 lv_carr = 'LH' → 결과 조회, sy-subrc = 0.
  2. 결과 없음 lv_carr = 'ZZ' → 빈 테이블, sy-subrc = 4, sy-dbcnt = 0.
  3. 타입/값 부적합 → lv_carr의 길이·값이 컬럼 타입과 안 맞으면 문법 경고·오류·런타임 데이터 오류.

3번을 "DB가 이상하다"로 보지 말자. **"ABAP 값이 SQL operand 자리의 타입 규칙을 못 맞췄다"**가 정확한 해석이다. 디버거에선 SELECT 직전에 lv_carr·lv_date·lv_base 값을 먼저 확인하고, 실행 뒤엔 어떤 host 값이 들어갔는지 함께 기록한다.

흔한 실수와 점검 🚧

  • 컬럼에 @ → DB 컬럼(carrid)엔 붙이지 않는다. @는 ABAP 값에만.
  • @( ) 괄호 누락 → 식은 반드시 @( ... )로 감싼다.
  • 에 @ → 'LH' 같은 SQL 리터럴은 문장 안의 값이라 escape하지 않는다.
  • 너무 복잡한 host 식 → 입문 단계에선 산술·단순 내장값까지만. 메서드 호출·객체 생성식은 객체지향을 배운 뒤에.

ABAP 값을 안전하게 넣었다. 이제 결과 받을 테이블도 SQL 문장에서 바로 선언하자 → INTO TABLE @DATA(...) Inline Target.
