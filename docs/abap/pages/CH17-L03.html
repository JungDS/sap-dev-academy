출력용 Internal Table 준비

그리드에 보여줄 데이터를 내부 테이블에 담는다 — SELECT와 결과 확인.

무엇을 보여줄까

ALV는 DB 테이블을 화면에 마법처럼 붙이는 도구가 아니다. 화면에 보여 줄 데이터는 먼저 프로그램 안의 (Internal Table 기초)에 담겨야 한다. Grid ALV는 그 내부 테이블을 읽어 행과 열로 그린다.

ABAP
1
2
3
4
DATA lt_booking TYPE STANDARD TABLE OF zbooking.

SELECT * FROM zbooking INTO TABLE lt_booking
  WHERE concert_id = p_concert.       " 이 공연의 예매만
  • 행 타입(ZBOOKING)이 로 타입팅돼 있으면 다음 레슨의 Field Catalog가 컬럼 제목·타입을 자동으로 끌어 쓴다.
  • 여기는 classic 구간이라 SELECT ... INTO TABLE ... WHERE ... 형태를 쓴다(@ host marker·콤마 필드 목록은 New Open SQL / Modern ABAP SQL에서).

조회 직후 세 가지 확인

빈 그리드가 기술 오류인지 정상적인 "결과 없음"인지 구분하려면, 조회 직후 셋을 본다.

확인 의미
sy-subrc 결과가 있었나 — 0=있음, 4=보통 결과 없음
sy-dbcnt DB에서 몇 건 넘어왔나
lt_booking 행 수 ALV가 그릴 실제 데이터 건수
ABAP
1
2
3
IF sy-subrc <> 0.
  MESSAGE '해당 공연의 예매 데이터가 없습니다.' TYPE 'S'.
ENDIF.

💡 이 메시지는 프로그램을 멈추려는 게 아니라, 빈 표가 정상(없는 공연)임을 알리는 안내다(조회 실패와 MESSAGE (기초)).

직접 조회해 보기 — 공연별 데이터 준비

공연 번호를 바꿔 조회하고, sy-subrc·sy-dbcnt·lt_booking 행 수가 어떻게 달라지는지 보자. C001은 결과가 있고, C999는 0건(sy-subrc=4), 조건 없이 조회하면 데이터가 한꺼번에 들어와 느려질 수 있다.

흔한 실수와 점검

  • 빈 테이블을 오류로만 보기 → 존재하지 않는 공연을 조회하면 빈 표가 정상 결과다. sy-subrc·sy-dbcnt·메시지를 함께 봐야 한다.
  • SELECT * 남용 → 입문 예제엔 단순해서 쓰지만, 실무는 필요한 필드만(필드 최적화는 New Open SQL / Modern ABAP SQL 이후 SQL 심화에서).
  • 행 타입 불명 → 구조 타입이 분명한 테이블이라야 그리드가 컬럼을 잡는다.

정리

  • 이 레슨은 "ALV에 무엇을 보여 줄 것인가" 를 준비하는 단계다 — 데이터를 lt_booking에 담는다.
  • 조회 결과는 sy-subrc·sy-dbcnt·행 수로 확인하고, 빈 결과도 정상일 수 있음을 기억한다.
