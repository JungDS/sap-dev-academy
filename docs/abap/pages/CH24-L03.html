DB LUW와 SAP LUW 차이

트랜잭션의 두 단위 — DB LUW와 SAP LUW.

Track 2Chapter 24Lesson 3중급

"트랜잭션"이라는 말이 두 가지다

앞 레슨에서 COMMIT WORK로 변경을 확정했다. 그런데 SAP에서 (함께 성공/실패해야 하는 작업 묶음)은 두 층으로 나뉜다.

  • DB LUW(Logical Unit of Work) — 데이터베이스가 보장하는 한 묶음. 한 번의 실제 쓰기~`COMMIT` 사이. 짧고 기술적이다.
  • SAP LUW — 사용자가 여러 화면·단계를 거치는 업무 단위. 길고 업무적이다.

SAP은 사용자가 여러 화면을 거치는 동안 변경을 곧바로 쓰지 않고 모아 두었다가, 마지막 COMMIT WORK 한 번에 한꺼번에 DB에 반영한다. 이 실제 쓰기를 맡는 일꾼이 Update Module(업데이트 함수)이다.

함수를 불렀는데 DB에 없다? — 타이밍이 핵심

여기서 입문자가 가장 많이 헷갈린다. CALL FUNCTION ... IN UPDATE TASK는 함수를 지금 실행하는 게 아니라, "COMMIT WORK 할 때 실행해 줘"라고 **예약(등록)**만 한다. 그래서 호출 직후 DB를 조회하면 아직 아무것도 없다. 아래에서 단계를 진행하며 대기 큐와 DB가 언제 채워지는지 직접 확인해 보라.

  • IN UPDATE TASK — 지금이 아니라 COMMIT 시점에 실행하도록 등록(지연 실행).
  • PERFORM <form> ON COMMIT — 비슷하게 " 때 이 루틴을 실행"하도록 등록한다. 단, subroutine은 parameter interface가 없는 유산(legacy) 패턴이라 신규 설계엔 권장하지 않는다 — 보이면 읽을 수 있는 수준으로만 알아 두고, 새 코드는 method wrapper로 감싼다.
  • 기본 COMMIT WORK는 update를 비동기로 맡기고 진행한다. 저장 완료를 기다려야 하면 앞 레슨의 COMMIT WORK AND WAIT.

IN UPDATE TASK 직후 sy-subrc는 정의되지 않는다. 이 문장은 실행이 아니라 등록이라, 성공 여부를 sy-subrc로 판정하지 않는다. 저장 결과는 COMMIT WORK 결과와 update log(SM13)로 확인한다.

흔한 실수와 점검

  • 즉시 실행 기대 → IN UPDATE TASK는 COMMIT 때 실행. 그 전엔 DB에 없다(그리고 호출 직후 sy-subrc는 무의미).
  • COMMIT 없이 끝 → 등록만 하고 COMMIT을 안 하면 update task는 영영 실행되지 않는다. 프로그램·이 끝나면 등록이 사라질 수 있으니 저장 책임을 분명히 한다.
  • update 함수 안에서 COMMIT/ROLLBACK → 금지. update 처리 중에는 경계를 다시 흔들지 않는다.
  • 표준은 BAPI → 표준 데이터 변경은 BAPI가 이 LUW 처리를 알아서 관리(직접 update task 짤 일 적음). ABAP Cloud에서는 임의 update task보다 released API·RAP behavior를 먼저 검토한다.

정리

  • 엔 DB LUW(DB 보장, 짧음)와 SAP LUW(업무 단위, 여러 화면)가 있다.
  • IN UPDATE TASK/ON COMMIT은 등록일 뿐 — 실제 실행·DB 기록은 COMMIT WORK 시점.
  • 저장 완료를 기다려야 하면 AND WAIT.

트랜잭션 단위를 이해했다. 그런데 대량 변경 중 일부가 실패하면 어떻게 기록하고 다시 처리할까 → 다음 레슨.
