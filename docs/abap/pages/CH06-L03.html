Internal Table

단일 행 제어

한 행을 콕 집어 넣고, 찾고, 고치고, 지운다.

Track 1Chapter 6Lesson 3초급

행을 콕 집어 다루기

전체를 훑지 않고 특정 한 행을 넣고, 찾고, 고치고, 지우는 일이 잦다. 결과는 그때그때 로 확인한다(0=성공).

넣기 — APPEND vs INSERT

  • APPEND gs_... TO gt_.... — 맨 끝에 추가(STANDARD 테이블의 기본).
  • INSERT gs_... INTO gt_... INDEX n. — n번째 위치에 끼워 넣기(Index Table 전용).
  • INSERT gs_... INTO TABLE gt_.... — 키에 맞는 자리에 넣기.
ABAP
1
2
3
APPEND gs_person TO gt_person.            " 끝에
INSERT gs_person INTO gt_person INDEX 1.  " 맨 앞에
INSERT gs_person INTO TABLE gt_person.    " 키 기준 자리

INTO(위치 기반)와 INTO TABLE(키 기반)은 다른 문장이다. HASHED는 번호가 없어 INTO TABLE만 가능하다. SORTED는 INDEX 문법 자체는 되지만 삽입이 정렬 순서를 깨면 런타임 오류가 난다. 그래서 SORTED/HASHED엔 INTO TABLE을 쓴다(자리는 시스템이 잡는다).

찾기 — READ TABLE

ABAP
1
2
3
4
READ TABLE gt_person INTO gs_person WITH KEY name = '정훈영'.
IF sy-subrc = 0.
  WRITE: / '찾음:', gs_person-age.
ENDIF.
  • INDEX n — n번째 행을 번호로 직접 읽기(예: READ TABLE gt_person INTO gs_person INDEX 1.).
  • WITH KEY field = ... — 아무 필드 조합(free key)으로 찾기(STANDARD는 순차 탐색).
  • WITH TABLE KEY ... — 선언에 정의된 키로 찾기(SORTED/HASHED에 최적).
  • 못 찾으면 sy-subrc = 4. 반드시 확인한다.

찾기에 성공하면 시스템 필드 에 그 행의 번호가 들어온다. "몇 번 행에서 찾았나"다. 잠시 뒤 고치기에서 바로 써먹는다.

값은 필요 없고 있는지만 볼 땐 TRANSPORTING NO FIELDS로 복사를 생략한다(빠름).

ABAP
1
2
3
READ TABLE gt_person TRANSPORTING NO FIELDS WITH KEY name = '정훈영'.
IF sy-subrc = 0.  " 존재만 확인(Work Area 복사 없음)
ENDIF.

정렬 — SORT

키로 자주 찾을 테이블은 먼저 정렬해 두는 게 유리하다. 문법은 간단하다.

ABAP
1
2
SORT gt_person BY name.              " name 기준 오름차순(기본)
SORT gt_person BY age DESCENDING.    " 내림차순

참고로 SORTED TABLE은 키 순서가 선언으로 고정돼 있어 SORT로 다시 정렬할 수 없다 (SORT의 대상은 STANDARD·HASHED).

빠른 탐색 — BINARY SEARCH

STANDARD 테이블에서 단건을 자주 찾으면 순차 탐색이 느리다. 키로 정렬해 두고 BINARY SEARCH를 붙이면 이진 탐색으로 빨라진다.

ABAP
1
2
3
SORT gt_person BY name.                  " 반드시 같은 키로 먼저 정렬
READ TABLE gt_person INTO gs_person
     WITH KEY name = '정훈영' BINARY SEARCH.

⚠️ BINARY SEARCH는 그 키로(오름차순) 정렬돼 있어야 정확하다. 정렬 안 하고 쓰면 있는 행도 "없음"(sy-subrc ≠ 0)으로 나온다. 아래에서 직접 실패를 목격해 보자.

고치기 · 지우기 — MODIFY · DELETE

찾은 행을 고치는 최빈 패턴은 이것이다. READ가 채워 준 sy-tabix로 바로 MODIFY.

ABAP
1
2
3
4
5
6
READ TABLE gt_person INTO gs_person WITH KEY name = '정훈영'.
IF sy-subrc = 0.
  gs_person-age = 31.
  MODIFY gt_person FROM gs_person INDEX sy-tabix
         TRANSPORTING age.               " age 필드만 갱신(나머지 보존)
ENDIF.
  • MODIFY gt_... FROM gs_... INDEX n. — n번 행을 통째로 교체.
  • MODIFY TABLE gt_... FROM gs_.... — 키로 행을 찾아 교체.
  • TRANSPORTING f1 f2 — 그 필드만 갱신하고 나머지는 그대로.
  • DELETE gt_person WHERE age < 20. — 조건에 맞는 행 모두 삭제. 하나라도 지웠으면 sy-subrc = 0, 지운 행이 없으면 4.

흔한 실수와 점검

  • 미확인 → READ 실패 시 Work Area엔 이전 값이 그대로 남는다. 그걸 새로 읽은 값인 줄 알고 쓰는 버그가 잦다.
  • BINARY SEARCH 전에 SORT 안 함 → 잘못된 "없음" 결과.
  • INTO vs INTO TABLE 혼동 → 위치 기반인지 키 기반인지 구분.
  • 를 오래 보관 → 그 사이 정렬·삭제가 일어나면 번호의 의미가 바뀐다. READ 직후 바로 쓴다.

한 행은 자유자재로 다룬다. 이제 여러 행을 한꺼번에 훑고 집계하는 법이다 → 다중 행 제어.
