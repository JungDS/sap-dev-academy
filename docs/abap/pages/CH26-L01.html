Factory Pattern

객체 생성을 한 곳에 모은다 — Factory.

NEW를 여기저기 흩뿌리면

코드 곳곳에서 NEW zcl_...( )로 객체를 만들면, 생성 방식이 바뀔 때(예: 종류에 따라 다른 클래스) 전부 고쳐야 한다. 생성을 한 곳(팩토리) 에 모은다.

CLASS zcl_booking_factory DEFINITION PUBLIC.
  PUBLIC SECTION.
    CLASS-METHODS create
      IMPORTING iv_type       TYPE c
      RETURNING VALUE(ro_obj) TYPE REF TO zif_booking.
ENDCLASS.

CLASS zcl_booking_factory IMPLEMENTATION.
  METHOD create.
    CASE iv_type.
      WHEN 'V'.    ro_obj = NEW zcl_vip_booking( ).      " VIP 예매
      WHEN OTHERS. ro_obj = NEW zcl_booking_manager( ).  " 일반 예매
    ENDCASE.
  ENDMETHOD.
ENDCLASS.
  • 호출부는 zcl_booking_factory=>create( 'V' )만 — 어느 클래스인지 몰라도 된다.
  • 인터페이스(zif_booking) 반환으로 다형성과 결합(CH20).

흔한 실수와 점검

  • 팩토리 우회 → 어떤 곳은 직접 NEW, 어떤 곳은 팩토리 → 일관성 깨짐. 생성은 팩토리로.
  • 거대 팩토리 → 종류가 너무 많으면 분리·구성을 고려.

생성을 모았다. 그런데 딱 하나만 있어야 하는 객체는? → Singleton(다음 레슨).
