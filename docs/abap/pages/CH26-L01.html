Lock Object와 동시성 제어

Lock Object 설계 기준

동시 변경을 막는 장치 — Lock Object와 잠금 모드.

Track 2Chapter 26Lesson 1중급

동시에 같은 예매를 고치면

INSERT / UPDATE / MODIFY / DELETE 실무 기준에서 데이터를 안전하게 바꾸는 법을 배웠다. 그런데 실무 시스템은 여러 사람이 동시에 쓴다. A·B 두 직원이 같은 예매 1001(2석)을 동시에 열고, A가 3석으로 고쳐 저장한 뒤 B가 오래된 화면에서 1석으로 저장하면 — 결과는 1석이다. A의 3석이 오류 없이 사라진다. 이 현상을 Lost Update(잃어버린 갱신: 먼저 저장한 변경이 나중 저장에 덮여 사라짐)라고 한다.

COMMIT/ROLLBACK은 내 안만 챙길 뿐, 두 사람이 같은 예매를 동시에 편집했다는 사실은 못 막는다. 그래서 "이 행은 지금 내가 작업 중"이라고 알리는 잠금 장치가 필요하다 — 다.

Lock Object — SE11에서 만든다

SE11 → → EZ_BOOKING(관례: E + 테이블명)을 만든다. 구성은 단 두 가지다.

  • Primary Table — 잠글 대상 테이블(ZBOOKING).
  • Lock Argument — 무엇을 기준으로 잠글지. 보통 키 필드(booking_id) — "1001번 예매만 잠금".
  • 활성화하면 잠그고/푸는 ENQUEUE/DEQUEUE 함수가 자동 생성된다(ENQUEUE_EZ_BOOKING· DEQUEUE_EZ_BOOKING). 우리가 함수 본문을 짤 필요 없이 호출만 하면 된다(다음 레슨).

잠금 모드 — E / S / X

잠금에는 "어떤 동시 접근을 허용할지" 정하는 모드가 있다.

모드 뜻 다른 사용자에게 허용
E Exclusive(쓰기) 아무도 못 — 가장 흔함
S Shared(읽기) 다른 S(읽기) 는 OK, E(쓰기) 는 차단
X eXclusive 비누적 같은 사용자도 중첩 불가(가장 강함)

말로는 헷갈리니, A가 어떤 모드로 잡고 있을 때 B의 요청이 허용되는지/거절되는지 직접 바꿔 보자.

대부분의 변경 잠금은 E(Exclusive)다 — "내가 고치는 동안 아무도 손대지 마".

흔한 실수와 점검

  • Lock Argument 과다 → key를 안 넘기면 한 건이 아니라 더 넓은 범위가 잠긴다. 한 사람이 한 예매를 고치는데 다른 모든 예매 변경이 막히면 시스템은 안전해지는 게 아니라 멈춘다. 보통 키 단위로 좁게.
  • SAP Lock이 SQL을 자동으로 막는다고 착각 → SAP Lock은 중앙 lock table의 **논리적 약속(자발적 규약)**이다. 잠긴 행이라도 프로그램이 lock check를 안 하면 ABAP SQL로 그냥 접근·변경된다. 즉 "DB가 모든 UPDATE를 거절하는 마법"이 아니다 — 변경 프로그램이 먼저 ENQUEUE_*를 호출하고 실패하면 변경을 멈춰야 의미가 있다.
  • 잠금 없이 변경 → 동시 충돌 위험. 변경 전에 잠근다(ENQUEUE / DEQUEUE Function Module의 표준 패턴).
  • 모드 오해 → 단순 조회 보호엔 S, 변경엔 E. X(강한 배타)·O(optimistic)는 특수 상황.

정리

  • 동시 변경 충돌은 COMMIT으로 못 막는다 → (SE11)로 잠근다.
  • 구성 = Primary Table + Lock Argument(보통 키). 활성화 시 ENQUEUE/DEQUEUE 자동 생성.
  • 모드 = E(쓰기·기본)·S(읽기 공유)·X(강한 배타). S끼리만 동시 허용.

잠금 장치(Lock Object)를 설계했다. 이제 실제로 잠그고 푸는 함수를 호출해 보자 → 다음 레슨.
