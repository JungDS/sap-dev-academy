대량 변경 시 Package 처리

수십만 건을 나눠서 — 패키지 단위 COMMIT.

Track 2Chapter 25Lesson 5중급

한 번에 다 하면 터진다

수십만 건을 하나의 으로 바꾸면 어떻게 될까? COMMIT 전까지의 모든 변경이 ( 세그먼트)에 쌓이고, 건드린 행마다 잠금이 걸린 채 유지된다. 결국 메모리·DB 잠금·로그가 폭발하고, 중간에 한 건만 실패해도 전부 롤백된다.

해법은 단순하다 — 일정 건수()마다 처리하고 COMMIT해서, 쌓인 부담을 주기적으로 비운다.

DATA lv_cnt TYPE i.

LOOP AT lt_big INTO DATA(ls).
  MODIFY ztarget FROM @ls.
  lv_cnt += 1.
  IF lv_cnt >= 5000.          " 5천 건마다
    COMMIT WORK.              " 미커밋 버퍼를 비운다
    CLEAR lv_cnt.
  ENDIF.
ENDLOOP.
COMMIT WORK.                  " 루프 후 남은 것 확정(빠뜨리기 쉬움)

패키지 크기, 직접 바꿔 보기

를 작게/크게 잡거나 "단일(전체)"로 두면 COMMIT 횟수와 최대 미 버퍼가 어떻게 달라질까? 아래에서 크기를 바꿔, 버퍼가 톱니처럼 차고 비우는 패턴과 단일 COMMIT의 버퍼 폭발을 비교해 보라.

  • 패키지 크기는 ·성능을 보며 조정한다(보통 1천~1만). 작을수록 자원은 안전하나 COMMIT 횟수가 늘어 오버헤드가 커진다 — 절충점을 찾는다.
  • 패키지마다 COMMIT해 잠금·로그· 세그먼트를 비운다.
  • 중간 실패 시 어디까지 처리됐는지(재시작점)를 관리한다 — 오류 로그와 재처리 구조의 오류 테이블/로깅과 함께.

원자성 ↔ 재시작은 트레이드오프다. 패키지 COMMIT은 부분 성공과 재처리를 허용하는 업무에 어울린다. 반대로 **"전체가 모두 성공해야만 저장"**인 업무에는 패키지 COMMIT을 쓰면 안 된다 — 앞 패키지가 이미 확정되어 뒤에서 실패해도 전체를 한 번에 ROLLBACK할 수 없기 때문이다. 그런 경우는 전체 한 LUW로 묶거나 사전 검증을 강화한다.

흔한 실수와 점검

  • 전체 단일 COMMIT → 대량에서 자원 폭발. 반드시 로 쪼갠다.
  • 루프 뒤 마지막 COMMIT 누락 → 패키지 배수가 아닌 나머지 건이 확정 안 됨. 루프 후 한 번 더 — 단, 실제 변경이 있었을 때만 commit하도록 lv_pending 플래그를 두면 더 명확하다.
  • 재시작점 없음 → 중간 실패 후 처음부터 다시. 진행 위치를 기록(오류 로그와 재처리 구조).
  • 전체 성공 필요 업무에 패키지 COMMIT → 앞 패키지가 이미 확정돼 전체 ROLLBACK이 불가. 부분 성공을 허용하는 업무에만 쓴다.

정리

  • 대량 변경은 단위 COMMIT으로 — 미 버퍼를 주기적으로 비운다.
  • 크기는 1천~1만에서 절충(작을수록 안전, COMMIT 오버헤드↑).
  • 루프 뒤 남은 건 마지막 COMMIT + 재시작점 관리를 잊지 않는다.

🧩 도전 과제 (서브퀘스트)

선택 과제 — 이 과정에서 키운 콘서트 예매 앱에 이 챕터를 적용한다.

  1. 실습 — 예매 입력 화면 (Dynpro)에서 만든 예매 입력 화면의 SAVE에 실제 INSERT zbooking을 붙이고 COMMIT WORK로 확정한다.
  2. 저장 시 감사필드(created_by=sy-uname·created_on=sy-datum)를 자동 stamp한다.
  3. (심화) 엑셀에서 받은 예매 100건을 단위 COMMIT(이 레슨)으로 일괄 등록한다.

힌트: INSERT / UPDATE / MODIFY / DELETE 실무 기준(INSERT·감사필드)·COMMIT WORK / ROLLBACK WORK(COMMIT)·이 레슨(패키지). 표준 테이블은 직접 DML 금지(BAPI).
