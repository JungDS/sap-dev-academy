DDIC 관계와 입력도움말(F4)

Value Table과 Foreign Key의 차이

Domain의 Value Table은 '제안', Foreign Key는 '실제 관계'.

Track 1Chapter 9Lesson 2초급

비슷해 보이는 둘

을 만들 때 을 지정할 수 있고, 테이블 필드엔 를 건다. 둘 다 "어떤 테이블의 값으로 제한"하는 것 같은데 뭐가 다른가? 헷갈리기 쉬운 지점이라 분명히 짚는다.

Value Table = Domain의 '제안'

  • 은 수준에서 "이 도메인 값은 보통 이 테이블에서 온다"고 제안하는 것이다.
  • 그 자체로는 검증을 강제하지 않는다. 어디까지나 기본 제안.

Foreign Key = 테이블 필드의 '실제 관계'

  • 는 테이블 필드 수준에서 거는 실제 관계다.
  • 이게 있어야 입력검증·F4가 실제로 동작한다.
  • Foreign Key를 만들 때, 에 이 있으면 시스템이 그걸 Check Table로 자동 제안한다 → 둘은 협력 관계.

보너스 — Domain의 변환 루틴(ALPHA)

에는 변환 루틴(Conversion Routine) 도 달 수 있다. 대표가 ALPHA다. 공연번호·사번처럼 숫자로 된 ID를, 내부 저장은 앞자리 0을 채우고(0000000042) 화면 표시는 0을 떼서(42) 보여 주는 식으로 형식을 자동 변환한다.

  • 같은 ID라도 "내부 형식"과 "표시 형식"이 다를 수 있다는 점을 기억해 둔다.
  • 코드에서 조회·비교 시 형식이 안 맞아 "없음"이 나오면 변환 루틴(ALPHA) 때문일 수 있다. 표준 변환 기능으로 내부/표시 형식을 맞춘다.

CH03에서 "변환 루틴은 개념만"이라 했던 그 ALPHA가 여기서 실제로 쓰인다 — 나선형 재방문.

직접 해 보기

두 스위치(·)를 조합해 C999가 통과처럼 보이는지/거부되는지 직접 확인해 보자. ALPHA 변환도 함께 만져 볼 수 있다.

흔한 실수와 점검

  • 만 믿기 → 그것만으론 검증 안 된다. 필드에 를 걸어야.
  • Check Table ≠ Value Table → 보통 같게 쓰지만, Foreign Key에서 다른 Check Table을 지정할 수도 있다.

값의 존재는 관계로 제한했다. 그런데 코드만 든 마스터에 사람이 읽을 이름을 붙이려면? → 다음 레슨(Text Table).
