OO ABAP 기본 설계

Inheritance / Redefinition

공통은 부모에, 차이만 자식에 — 물려받아 확장·재정의한다.

Track 1Chapter 20Lesson 7고급

왜 — 공통은 한 번만 🧬

일반 예매와 VIP 예매가 대부분 같고 일부 규칙만 다르다고 하자. 두 를 처음부터 따로 만들면 잔여석 계산·정원 초과 ·공연/회차 상태 보관 같은 공통 로직이 중복된다.

상속은 공통을 부모에 두고, 차이만 자식에서 확장하게 해 준다. ABAP 클래스는 단일 상속(부모는 하나)이다.

CLASS zcl_vip_booking DEFINITION PUBLIC
  INHERITING FROM zcl_booking_manager.
  PUBLIC SECTION.
    METHODS book REDEFINITION.        " 부모 메서드를 다시 정의
ENDCLASS.

CLASS zcl_vip_booking IMPLEMENTATION.
  METHOD book.
    IF iv_seats > 4.                  " VIP 전용 추가 규칙
      RAISE EXCEPTION TYPE zcx_fully_booked.
    ENDIF.
    super->book( iv_seats = iv_seats ).   " 부모의 원래 동작 재사용
  ENDMETHOD.
ENDCLASS.
  • INHERITING FROM — 부모를 한다.
  • REDEFINITION — 부모의 를 새로 구현(오버라이드).
  • super-> — 부모의 원래 구현을 부른다(자식 검사 후 부모 동작 재사용).

정적 타입 vs 동적 타입 🎭

부모 타입 참조 변수에 자식 를 담을 수 있다. 그래도 재정의된 는 실제 객체(자식) 구현으로 실행된다.

ABAP
1
2
3
4
DATA lo_mgr TYPE REF TO zcl_booking_manager.   " 정적 타입 = 부모
lo_mgr = NEW zcl_vip_booking( iv_concert = 'C001' iv_perf = '01' ).
"                              ↑ 동적 타입 = 자식
lo_mgr->book( iv_seats = 2 ).   " 자식의 redefined book 이 실행된다

변수의 정적 타입은 zcl_booking_manager지만, 담긴 객체의 동적 타입은 zcl_vip_booking이다. book이 재정의돼 있으면 호출은 자식 구현으로 들어간다 — 이것이 다음 레슨에서 다룰 다형성의 핵심이다.

ABSTRACT · FINAL 🚧

  • ABSTRACT — 직접 를 못 만들고 상속 전용 틀로 둔다(공통 정책의 추상 부모).
  • FINAL — 더 이상 상속·재정의 금지(완성된 ).

흔한 실수와 점검 🚧

  • REDEFINITION 없이 오버라이드 → 부모 를 바꾸려면 definition에 METHODS book REDEFINITION.이 있어야 한다.
  • 자식에서 부모 private 접근 → private은 자식에게도 안 보인다. 자식에게 열 의도면 부모에서 protected로 설계한다.
  • 상속으로 모든 변형 해결 → "VIP는 예약 관리자의 특수한 종류"면 상속이 맞지만, "출력 가능"·"검증 가능" 같은 횡단 규약은 Interface가 더 적합하다.
  • super->를 무심코 켜고 끔 → 부모 동작을 유지하며 앞뒤로 규칙을 더할지, 완전히 대체할지 설계 의도를 먼저 정한다.

상속으로 확장했다. 그런데 부모 타입 변수에 담긴 자식을 원래 자식 타입으로 다시 다루려면? → 다형성 — CAST와 CASE TYPE OF.
