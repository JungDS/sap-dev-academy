Editable Grid ALV와 입력 검증

DATA_CHANGED_FINISHED Event

변경이 모두 반영된 뒤 — 합계·연동 갱신.

Track 2Chapter 28Lesson 3고급

변경이 끝난 뒤에 할 일

DATA_CHANGED Event는 값이 바뀐 직후(아직 내부 테이블 반영 전) 그 셀을 검증했다. 그런데 모든 후처리를 그 순간에 하면 안 된다. 좌석 수를 여러 번 고치는 중에 매 입력마다 총합·잔여석·상태를 다 다시 계산하면 화면이 흔들리고 가 꼬인다.

data_changed_finished는 변경이 내부 테이블에 반영된 뒤 발생한다. 합계·잔여석 같은 연동(파생) 값 재계산에 적합하다.

METHODS on_changed_finished
  FOR EVENT data_changed_finished OF cl_gui_alv_grid
  IMPORTING e_modified.

METHOD on_changed_finished.
  IF e_modified <> abap_true.
    RETURN.                          " 바뀐 게 없으면 후처리 생략
  ENDIF.

  CLEAR gv_total_seats.
  LOOP AT lt_booking INTO DATA(ls_booking).
    gv_total_seats = gv_total_seats + ls_booking-seats.
  ENDLOOP.

  DATA(ls_stbl) = VALUE lvc_s_stbl(
    row = abap_true col = abap_true ).
  go_grid->refresh_table_display( is_stable = ls_stbl ).
ENDMETHOD.
  • e_modified — 실제로 바뀐 게 있을 때만(abap_true) 재계산한다.
  • 끝에 stable refresh로 편집하던 행·컬럼 위치를 유지하며 다시 그린다.

합계 재계산 체험

SEATS를 고치고 확정(Enter나 다른 칸으로 이동)하면 data_changed_finished가 총 좌석 합계를 다시 계산하는 걸 확인해 보라.

흔한 실수와 점검

  • 두 를 같은 용도로 → 검증은 data_changed(바뀐 셀), 후처리는 data_changed_finished(변경 완료 후). 검증을 finished에서 늦게 하면 값이 이미 들어간 뒤 오류를 보게 된다.
  • 후처리가 또 변경을 유발 → 재계산이 사용자 입력값을 덮어쓰거나 또 다른 변경 이벤트를 부르면 무한 갱신이 된다. 파생값 갱신은 최소한으로.
  • e_modified 미확인 → 안 바뀌었는데 매번 재계산·refresh하면 성능·UX가 나빠진다(실무 ALV는 행이 많고 후처리가 여러 서비스를 부를 수 있다).

정리

  • data_changed_finished = 변경이 내부 테이블에 반영된 뒤 후처리.
  • e_modified로 실제 변경을 확인하고, 합계·잔여석 같은 파생값을 계산한 뒤 stable refresh.
  • 셀별 즉시 검증(data_changed)과 라운드 후 후처리(data_changed_finished)는 역할이 다르다.

후처리까지 했다. 이제 같은 컬럼이라도 행마다 입력 가능 여부를 다르게 하는 Cell Style → Cell Style 기반 입력 제어(다음 레슨).
