AMDP / ADBC / Pushdown

운영 리스크와 DB 종속성

강력함의 대가 — 종속성·디버깅·이식성을 관리한다.

Track 2Chapter 33Lesson 5고급

강력한 만큼 위험도 크다

AMDP·Native SQL은 빠르지만 대가가 있다.

  • DB 종속 — HANA SQLScript는 다른 DB로 못 옮긴다(이식성↓).
  • 디버깅 어려움 — ABAP 디버거(CH04)로 SQLScript 내부를 못 본다.
  • 검증 약함 — Open SQL의 강한 구문 검사가 없다.
  • 클라우드 제약 — ABAP Cloud(CH23)에서 Native SQL은 제한적.

운영 원칙

  • 최소화 — 정말 필요할 때만. 대부분은 SQL/CDS.
  • 격리·문서화 — AMDP/Native는 별도 로 모으고 이유를 남긴다.
  • 테스트 — 단위 테스트(CH26)·대표 데이터로 검증.

흔한 실수와 점검

  • 편하다고 남발 → 종속성·유지보수 부채가 쌓인다.
  • 이식 고려 안 함 → DB 전환·클라우드 전환에서 큰 비용.

🧩 도전 과제 (서브퀘스트)

선택 과제 — DB 가까이에서 한다.

  1. 공연별 예매 집계를 CDS/Open SQL로 먼저 구현한다(1순위).
  2. 절차적 로직이 꼭 필요하면 같은 집계를 AMDP(SQLScript)로 만들어 비교한다.
  3. (심화) AMDP/ADBC의 DB 종속성·이식성·클라우드 제약을 표로 정리해 선택 기준을 세운다.

힌트: L01(Pushdown 판단)·L02(AMDP)·L03(ADBC)·L04(비교)·L05(리스크).
