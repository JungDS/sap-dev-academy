CDS View Entity 기초

CDS View Entity 기본 구조

DB 계층의 현대적 읽기 모델 — define view entity.

Track 1Chapter 23Lesson 1고급

왜 또 다른 "뷰"가 필요할까

Database View와 Open SQL JOIN 비교에서 본 classic Database View는 SE11이라는 화면(GUI)에서 클릭으로 정의하는 방식이었다. 학습 초반에는 그걸로 충분하다. 그런데 시스템이 커지면 세 가지 불편이 쌓인다.

  • 같은 조회가 프로그램마다 반복된다. 콘서트 목록을 보여주는 리포트·ALV·배치 점검이 전부 zconcert를 직접 읽는다면, 프로그램마다 필드 선택·별칭·권한·라벨을 다시 맞춰야 한다.
  • 데이터의 "의도"가 코드 밖에서 안 보인다. 테이블에는 값만 있다. "이 금액은 어떤 통화와 짝인가", "이 공연은 어떤 회차와 연결되나" 같은 의미는 여기저기 흩어진다.
  • 요즘 개발 흐름은 텍스트 소스를 좋아한다. ADT(Eclipse)·Git 기반에서는 화면으로 그린 뷰보다 텍스트로 관리되는 모델이 훨씬 추적하기 좋다.

이 불편의 해결책이 바로 다. 핵심 한마디는 "프로그램을 하나 더 만든다"가 아니라 **"읽기 모델을 재사용 가능한 이름으로 만든다"**이다.

무엇인가 — 모델을 선언하는 파일

에서 CDS는 Core Data Services(핵심 데이터 서비스)의 약자다. ABAP의 REPORT·FORM·CLASS와 달리, CDS는 DDL(Data Definition Language, 데이터를 정의하는 언어)로 작성하고 ADT(Eclipse)에서 만든다. 문법의 심장은 다음 한 줄이다.

@AccessControl.authorizationCheck: #NOT_REQUIRED
define view entity ZI_Concert
  as select from zconcert
{
  key concert_id,
      artist,
      venue,
      capacity
}

읽는 순서는 이렇다.

코드 뜻
@AccessControl...: #NOT_REQUIRED 이 첫 예제에선 권한 검사를 요구하지 않는다(권한은 뒤 레슨에서).
define view entity ZI_Concert ZI_Concert라는 CDS View Entity를 만든다.
as select from zconcert 기반 데이터 소스는 zconcert 테이블이다.
key concert_id concert_id를 키 요소로 노출한다.
artist, venue, capacity 소비처가 읽을 일반 필드다.

{ } 안의 select list는 INTO TABLE @DATA(...) Inline Target에서 익힌 SELECT 감각과 닮았지만, 지금 데이터를 가져오는 문장이 아니다. SELECT ... INTO TABLE ...처럼 당장 읽어 오는 게 아니라, 모델을 선언하는 자리다. 활성화하면 ABAP Dictionary에 CDS entity가 생기고, 그 다음부터 ABAP SQL이나 다른 CDS가 ZI_Concert라는 이름을 데이터 소스처럼 쓴다.

계산 요소도 간단히 만든다. 회차 테이블 zperf의 날짜를 또렷한 DATS 타입으로 노출하고 싶다면:

@AccessControl.authorizationCheck: #NOT_REQUIRED
define view entity ZI_Perf
  as select from zperf
{
  key concert_id,
  key perf_no,
      perf_date,
      cast( perf_date as abap.dats ) as event_date
}

여기서 event_date는 테이블의 물리 컬럼이 아니라 CDS가 조회될 때 보여 주는 계산 요소다. 입문자가 가장 오해하는 지점이 여기다 — "테이블에 컬럼이 새로 생겼다"가 아니다. DB 구조는 그대로고, CDS entity를 읽을 때 보이는 요소를 정의했을 뿐이다.

직접 관찰해 보기

아래 위젯에서 원본 테이블 → DDL 작성 → 활성화 → Data Preview → ABAP SQL 소비의 다섯 단계를 따라가 보자. 필드 오타 주입을 켜면 활성화가 어떻게 실패하는지(CDS는 오류가 있으면 활성화되지 않는다), 끄면 SELECT FROM ZI_Concert로 프로그램이 테이블 대신 모델을 소비하는 모습까지 이어진다.

어떻게 확인하는가

ADT에서 DDL Source를 만들고 활성화한 뒤 세 가지를 본다.

  1. 활성화 오류 — 필드명 오타나 key 위치 오류가 있으면 활성화 단계에서 바로 드러난다. CDS는 오류가 있으면 활성화 자체가 안 된다.
  2. Data Preview — ZI_Concert를 열어 미리 보면 concert_id·artist·venue·capacity가 보인다. L01은 단순 select라 행 수가 원본 zconcert와 같아야 정상이다. 크게 다르면 where·join을 의심한다.
  3. ABAP SQL로 읽기 — 모델이 잘 만들어졌으면 프로그램에서 이름으로 읽는다.
ABAP
1
2
3
SELECT *
  FROM ZI_Concert
  INTO TABLE @DATA(lt_concerts).

이 코드는 INTO TABLE @DATA(...) Inline Target에서 배운 모던 ABAP SQL(@DATA·host 변수 @)을 그대로 쓴다. 중요한 확인 포인트는 FROM zconcert가 아니라 **FROM ZI_Concert**라는 점 — 프로그램이 테이블에 직접 붙지 않고 CDS 모델을 소비하기 시작했다.

흔한 실수와 점검 🚧

  • CDS를 ABAP 프로그램처럼 생각 → DDL Source에는 WRITE·LOOP·IF·sy-subrc가 없다. 실행 흐름이 아니라 결과 구조를 읽는 파일이다.
  • key를 DB 기본키와 100% 같은 것으로 착각 → CDS의 key는 이 뷰가 노출하는 키 의미다. 보통 테이블 키를 따라가지만, projection· 모델에선 설계에 따라 달라질 수 있다(이 레슨에선 테이블 키를 그대로 올리는 수준만).
  • #NOT_REQUIRED를 실무 보안 설정으로 외움 → 아직 권한을 안 배웠으니 단순 조회용 최소 설정으로 쓴 것뿐이다. 진짜 권한이 필요한 CDS는 뒤에서 DCL과 함께 다룬다.

뷰 하나는 만들었다. 그런데 실무는 같은 데이터라도 소비처마다 다른 모양이 필요해서, 뷰를 계층으로 나눈다 — 재사용 기반(Interface)과 소비 출구(Projection) → Interface View와 Projection View 구분.
