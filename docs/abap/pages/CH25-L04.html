오류 로그와 재처리 구조

변경이 실패했을 때 — 기록하고 다시 처리한다.

실패는 반드시 남긴다

수백 건을 한 번에 처리하다 보면 일부만 실패하는 일이 흔하다(중복키, 참조 오류 등). 이때 실패를 그냥 흘려보내면 나중에 "무엇이·왜 빠졌는지" 알 길이 없다. 그래서 매 변경 뒤 sy-subrc를 확인하고, 실패한 건을 오류 테이블/로그에 모아 둔다.

DATA lt_error TYPE TABLE OF zbooking.

LOOP AT lt_booking INTO DATA(ls_booking).
  INSERT zbooking FROM @ls_booking.
  IF sy-subrc <> 0.                    " 단건 중복키면 sy-subrc=4
    APPEND ls_booking TO lt_error.     " 실패 건만 따로 모은다
  ENDIF.
ENDLOOP.

TYPES: BEGIN OF ts_error,
         booking_id TYPE zbooking-booking_id,
         reason     TYPE string,
       END OF ts_error.
DATA lt_error TYPE TABLE OF ts_error.

LOOP AT lt_booking INTO DATA(ls_booking).
  INSERT zbooking FROM @ls_booking.
  IF sy-subrc <> 0.
    APPEND VALUE #( booking_id = ls_booking-booking_id
                    reason     = |INSERT 실패: sy-subrc={ sy-subrc }| ) TO lt_error.
  ENDIF.
ENDLOOP.

처리 → 실패 수집 → 재처리, 직접 해 보기

아래에서 예매 5건을 일괄 처리하면 2건(중복키)이 실패한다. 실패가 오류 테이블에 어떻게 모이고, 원인을 고친 뒤 실패분만 다시 처리하는 흐름을 따라가 보라.

재처리 패턴

  • 실패 건만 오류 테이블/스풀에 남겨 두었다가, 원인 해결 후 그것만 다시 실행한다 (처음부터 전부 다시 돌리지 않는다 — 오류 테이블이 곧 재시작점).
  • 멱등성(같은 걸 두 번 처리해도 결과가 안전)을 고려한다 — INSERT 대신 MODIFY(upsert)나 사전 중복 체크. 재처리가 또 중복키로 깨지면 안 되니까.
  • 표준 Application Log(BAL, T-code SLG1)에 성공·경고·오류 메시지를 남기는 방식이 현업에서 흔하다(정식 학습은 Application Log와 오류 추적).

흔한 실수와 점검

  • 실패 무시 → 일부 실패를 모른 채 COMMIT. 항상 sy-subrc 점검(COMMIT WORK / ROLLBACK WORK의 COMMIT 판단과 직결).
  • 로그 없음 → 무엇이 실패했는지 못 찾는다. 최소한 키 + 사유는 기록.
  • 재처리가 또 깨짐 → 멱등성 미고려. MODIFY/중복 체크로 두 번 처리해도 안전하게.

정리

  • 실패는 흘려보내지 말고 오류 테이블/로그에 남긴다(sy-subrc 점검).
  • 재처리는 실패분만 — 오류 테이블이 재시작점.
  • 멱등성(MODIFY·중복 체크)으로 재처리를 안전하게.

실패 처리를 다뤘다. 그런데 애초에 수십만 건을 한 번에 바꾸면 ·잠금이 터진다. 나눠서 처리하는 법 → 다음 레슨(Package 처리).
