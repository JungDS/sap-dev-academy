CDS View Entity 기초

DCL / Authorization 개요

누가 어떤 행을 볼 수 있나 — CDS 접근 제어(DCL).

Track 1Chapter 22Lesson 6고급

왜 모델에 권한을 붙일까

데이터 모델을 만들고 UI 힌트까지 붙였다고 끝이 아니다. 실제 시스템에서는 **"누가 어떤 데이터를 볼 수 있는가"**가 중요하다. 같은 ZI_Concert라도 서울 공연 담당자는 서울 venue만, 부산 담당자는 부산 venue만 봐야 할 수 있다.

이 권한 필터를 프로그램마다 AUTHORITY-CHECK와 WHERE 조건으로 반복하면 위험하다. 어떤 프로그램은 권한을 적용하고, 어떤 프로그램은 빠뜨릴 수 있다. CDS의 장점은 읽기 모델 가까이에 접근 제어를 선언해 여러 소비처가 같은 보안 규칙을 따르게 하는 것이다.

무엇인가 — DCL로 행 단위 권한

DCL(Data Control Language)은 CDS entity에 대한 접근 규칙을 별도 DCL source로 정의한다.

@MappingRole: true
define role ZI_Concert_Role {
  grant select on ZI_Concert
    where ( venue ) = aspect pfcg_auth( Z_VENUE_AUTH, VENUE, ACTVT = '03' );
}
코드 뜻
@MappingRole: true 이 DCL role이 CDS access control role임을 표시.
define role ZI_Concert_Role 접근 규칙 이름을 정의.
grant select on ZI_Concert ZI_Concert를 SELECT할 때의 허용 조건.
where ( venue ) = aspect pfcg_auth(...) 사용자의 PFCG 권한을 기준으로 venue 값을 필터링.

대상 CDS 뷰는 권한 검사를 의도에 맞게 선언해야 한다.

@AccessControl.authorizationCheck: #CHECK
define view entity ZI_Concert
  as select from zconcert
{
  key concert_id,
      artist,
      venue,
      capacity
}

#CHECK는 access control이 있어야 한다는 기대를 분명히 드러낸다. #NOT_REQUIRED는 access control이 없어도 된다는 뜻으로 초기 단순 조회 예제에 적합하다. 이 챕터에서는 이 둘의 차이만 확실히 잡는다.

중요한 구분이 하나 있다 — DCL role은 사용자에게 PFCG role을 배정하는 행위 그 자체가 아니다. DCL은 CDS access rule이고, aspect pfcg_auth는 classic PFCG 권한 의 값을 조건으로 참조할 뿐이다. 권한 객체 설계와 사용자 권한 부여(SU01/PFCG)는 별도 보안 설정이다.

직접 비교해 보기

아래에서 사용자를 바꾸고 @AccessControl을 #CHECK ↔ #NOT_REQUIRED로 토글해 보자. 같은 SELECT인데 사용자 권한에 따라 보이는 행이 달라지고, #NOT_REQUIRED면 권한 밖 venue까지 노출되는 것을 직접 확인할 수 있다.

어떻게 확인하는가

  1. DCL source 활성화 — @MappingRole: true 누락, role 이름 오류, 대상 entity 오타, 권한 ·필드명 오류가 있으면 활성화 단계에서 드러난다.
  2. 대상 뷰의 @AccessControl — 권한을 보여 주는 예제에서는 #CHECK로 의도를 명확히 한다.
  3. 사용자별 결과 비교 — 같은 SELECT FROM ZI_Concert를 실행했을 때 사용자 A(SEOUL)와 B(BUSAN)가 보는 행이 달라져야 한다.
  4. 권한 없는 행이 안 보이는가 — 단순히 오류가 없다고 통과가 아니다. 행 단위 권한은 결과집합의 내용이 핵심이다.

흔한 실수와 점검 🚧

  • DCL을 만들면 자동으로 무조건 적용된다고 생각 → entity의 access control 설정·privileged access·소비 방식에 따라 적용 여부를 확인해야 한다.
  • #NOT_REQUIRED를 "권한이 필요하다"로 이해 → 이름과 반대다. 권한 예제에서는 #CHECK를 써서 의도를 분명히 한다.
  • DCL role과 사용자 role assignment 혼동 → define role은 CDS access rule이지, SU01/PFCG에서 사용자에게 role을 부여하는 작업이 아니다.
  • 권한을 UI에서만 막기 → 버튼·컬럼을 숨겨도 모델을 직접 읽는 다른 소비처가 있으면 우회된다. 행 단위 권한은 데이터 모델 가까이에서 일관되게 적용해야 한다.

CDS로 데이터를 현대적으로 모델링·노출·보호했다. 이제 배운 CDS를 콘서트 앱에 모아 적용해 ZI_Concert·ZI_Perf·ZC_Concert를 만들자 → 실습 — 콘서트 CDS 뷰 ZI_/ZC_.
