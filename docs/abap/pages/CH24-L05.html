대량 변경 시 Package 처리

수십만 건을 나눠서 — 패키지 단위 COMMIT.

Track 2Chapter 24Lesson 5중급

한 번에 다 하면 터진다

수십만 건을 한 트랜잭션으로 바꾸면 메모리·로그·잠금이 폭발한다. 일정 건수(패키지) 마다 처리하고 COMMIT한다.

DATA lv_cnt TYPE i.

LOOP AT lt_big INTO DATA(ls).
  MODIFY ztarget FROM @ls.
  lv_cnt += 1.
  IF lv_cnt >= 5000.          " 5천 건마다
    COMMIT WORK.
    CLEAR lv_cnt.
  ENDIF.
ENDLOOP.
COMMIT WORK.                  " 남은 것 확정
  • 패키지 크기는 메모리·성능을 보며 조정(예: 1천~1만).
  • 패키지마다 COMMIT해 잠금·로그·롤백 세그먼트를 비운다.
  • 중간 실패 시 어디까지 처리됐는지(재시작점) 관리가 중요(L04의 로깅과 함께).

흔한 실수와 점검

  • 전체 단일 COMMIT → 대량에서 자원 폭발. 패키지로 쪼갠다.
  • 재시작점 없음 → 중간 실패 후 처음부터 다시. 진행 위치를 기록.

🧩 도전 과제 (서브퀘스트)

선택 과제 — 콘서트 예매 앱(CH09~)에 이 챕터를 적용한다.

  1. CH16-L08에서 만든 예매 입력 화면의 SAVE에 실제 INSERT zbooking을 붙이고 COMMIT WORK로 확정한다.
  2. 저장 시 감사필드(created_by=sy-uname·created_on=sy-datum)를 자동 stamp한다.
  3. (심화) 엑셀에서 받은 예매 100건을 패키지 단위 COMMIT(L05)으로 일괄 등록한다.

힌트: L01(INSERT·감사필드)·L02(COMMIT)·L05(패키지). 표준 테이블은 직접 DML 금지(BAPI).

직접 데이터 변경·트랜잭션·대량처리를 다뤘다. 그런데 여러 사용자가 동시에 같은 데이터를 바꾸면? 충돌을 막는 잠금 → CH25(Lock Object).
