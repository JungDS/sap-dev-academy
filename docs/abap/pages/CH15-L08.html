Selection Screen 고급 — 블록·그룹 검증과 커스텀 F1·F4

필드 하나를 넘어 — 블록/라디오그룹 검증과, 코드로 직접 만드는 F1 도움말·F4 입력 도움.

필드 하나를 넘어선 검증·도움말

Chapter 15 · AT SELECTION-SCREEN 입력 검증에서 AT SELECTION-SCREEN ON <필드>로 한 필드를 검증했다. 그런데 실무 화면은 더 넓은 단위로 반응해야 할 때가 많다 — 블록 안 여러 필드가 함께 맞아야 하고, 라디오 선택에 따라 조건이 달라지고, F1 도움말이나 F4 입력 도움을 직접 제공해야 한다. 선택화면은 단순 입력칸 묶음이 아니라, 작은 dialog 화면처럼 반응할 수 있다.

블록 · 라디오그룹 단위 검증

필드 하나가 아니라 블록 전체의 조합, 또는 라디오 그룹 선택을 한 번에 검증한다. (블록을 화면에 그리는 SELECTION-SCREEN BEGIN OF BLOCK … 문법은 다음 레슨 Selection Screen UI 구성에서 자세히 본다. 여기서는 그 블록을 한 단위로 검증하는 에 집중한다.)

" 블록 b_date 전체의 조합 규칙을 한 번에 검증
AT SELECTION-SCREEN ON BLOCK b_date.
  IF p_from IS INITIAL OR p_to IS INITIAL.
    MESSAGE '기간을 모두 입력하세요' TYPE 'E'.
  ENDIF.
  IF p_from > p_to.
    MESSAGE '시작일이 종료일보다 늦습니다' TYPE 'E'.
  ENDIF.

" 라디오 그룹 g1 선택을 검증
AT SELECTION-SCREEN ON RADIOBUTTON GROUP g1.
  IF p_sum IS INITIAL AND p_det IS INITIAL.
    MESSAGE '요약/상세 중 하나를 선택하세요' TYPE 'E'.
  ENDIF.
  • ON BLOCK <블록> — 한 블록의 여러 필드를 묶어 조합을 검증.
  • ON RADIOBUTTON GROUP <그룹> — 라디오 선택과 다른 입력의 정합성을 검증.

코드로 만드는 F1 도움말 — ON HELP-REQUEST

Data Element의 기본 문서로 부족할 때, 특정 필드의 F1(도움말) 을 코드로 만든다.

ABAP
1
2
AT SELECTION-SCREEN ON HELP-REQUEST FOR p_conc.
  MESSAGE '공연 코드는 ZCONCERT에 등록된 CONCERT_ID입니다' TYPE 'I'.

ON HELP-REQUEST는 필드 의미를 보충하는 짧은 도움말에 적합하다. 긴 교육 문서를 팝업으로 띄우는 용도로 남용하지 않는다.

코드로 만드는 F4 입력 도움 — ON VALUE-REQUEST

선언적 Search Help(Chapter 9 · PARAMETERS와 DDIC F4 Help 연결)로 부족할 때 — 목록을 그때그때 DB에서 만들어야 할 때 — F4 입력 도움을 코드로 채운다.

PARAMETERS p_carr TYPE scarr-carrid.

DATA: BEGIN OF gs_carr,
        carrid   TYPE scarr-carrid,
        carrname TYPE scarr-carrname,
      END OF gs_carr.
DATA gt_carr LIKE STANDARD TABLE OF gs_carr.

AT SELECTION-SCREEN ON VALUE-REQUEST FOR p_carr.
  " 1) 목록 데이터를 조회 (classic)
  SELECT carrid carrname
    FROM scarr
    INTO TABLE gt_carr.

  " 2) F4 팝업으로 보여 주고, 고른 값을 화면 필드에 반영
  CALL FUNCTION 'F4IF_INT_TABLE_VALUE_REQUEST'
    EXPORTING retfield    = 'CARRID'     " 돌려줄 컬럼
              dynpprog    = sy-repid
              dynpnr      = sy-dynnr
              dynprofield = 'P_CARR'     " 값을 채울 화면 필드
    TABLES    value_tab   = gt_carr.     " 보여 줄 목록
  • value_tab — 팝업에 띄울 목록(코드에서 만든다).
  • retfield/dynprofield — 고른 값을 어느 화면 필드에 돌려줄지.
  • ⚠️ 이 함수는 고른 값을 P_CARR에 운반해 주지만, 선택화면의 다른 필드 값까지 자동으로 읽어 오지는 않는다. 다른 필드의 현재 화면 값이 필요하면 별도 API (DYNP_VALUES_READ 등)가 필요하다.

직접 반응시켜 보기 — Selection Screen 반응 실험실

네 가지 고급 를 직접 일으켜 보자. 블록 날짜를 비우거나 뒤집어 블록 검증을, 라디오를 바꿔 그룹 이벤트를, p_conc에서 F1을, p_carr에서 F4를 호출해 어떤 ABAP 이벤트가 발생하고 선택값이 화면 필드로 어떻게 들어오는지 확인한다.

실수와 주의

  • F4가 필요하면 무조건 function module부터 → 먼저 DDIC Search Help·check table로 자동 제공되는 도움말을 고려한다. 코드 F4는 목록이 동적이거나 특수할 때만.
  • 블록 이름 불일치 → ON BLOCK의 이름은 정의한 BLOCK 이름과 똑같아야 한다.
  • 검증 중복 → 블록 검증과 전체 검증이 같은 오류를 두 번 띄우지 않게 한다.
  • 검증과 도움말 혼동 → ON·ON BLOCK은 막는 것, F1·F4는 돕는 것으로 목적이 다르다.

정리

  • 선택화면은 정적 입력칸이 아니라, 블록·라디오그룹·F1·F4 로 사용자 행동에 반응하는 화면이다.
  • 검증(ON BLOCK·ON RADIOBUTTON GROUP)과 도움말(ON HELP/VALUE-REQUEST)을 나눠서 생각한다.
  • 자동 DDIC 도움말 vs 직접 구현의 경계를 먼저 판단한다 — 되도록 선언적으로.

검증·도움말 이벤트까지 다뤘다. 이제 선택화면을 시각적으로 구성하는 법 (블록·프레임·줄 배치)을 보자 → Selection Screen UI 구성.
