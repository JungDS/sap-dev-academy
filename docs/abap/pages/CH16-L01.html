정형 화면을 넘어서

Selection Screen UI 구성까지의 선택화면(PARAMETERS·SELECT-OPTIONS)은 SAP이 정해 준 틀이다. 조회 조건을 빠르게 받기엔 좋지만, "공연을 고르고, 좌석 수를 넣고, 예매 버튼을 누르면 검증이 돌고, 뒤로 를 누르면 닫히는" 내가 직접 설계한 업무 화면을 만들기엔 한계가 있다.

그 화면이 (Dynamic Program — 화면 하나의 레이아웃 + 흐름 로직)이고, 화면 중심 프로그램이 이다. 사실 우리가 써 온 선택화면도 특수한 형태의 Dynpro였다 — 이제 Screen Painter로 화면을 직접 그리는 일반 Dynpro로 넘어간다.

ℹ️ Dynpro는 classic SAP GUI 화면 기술이다. 기존 시스템 유지보수와 classic 화면 이해를 위해 배운다(최신 UI는 Fiori/UI5). 숨기지 않고 경계를 분명히 해 둔다.

화면의 두 박자 — PBO / PAI

화면 하나가 뜨고 처리되는 흐름은 두 박자다.

  • PBO(Process Before Output) — 화면을 그리기 직전(초기값·상태 준비).
  • PAI(Process After Input) — 사용자가 버튼·Enter를 누른 뒤(입력 처리).

이 두 박자는 화면이 닫힐 때까지 반복된다. "위에서 아래로 한 번 실행"이 아니라, 화면이 보이고 → 사용자가 반응하고 → 다시 화면이 준비되는 순환으로 생각해야 한다.

흔한 실수와 점검

  • T-code 없이 실행 → 은 리포트처럼 F8로 못 연다. T-code로만 실행된다(SE93로 생성).
  • 리포트 기대 → START-OF-SELECTION이 아니라 PBO/PAI 흐름이다.
  • PBO/PAI를 AT SELECTION-SCREEN OUTPUT/AT SELECTION-SCREEN과 같다고 치환 → "전/후" 감각은 비슷해도, flow logic + dialog module은 다른 구조다.
  • 화면 번호 0100을 규칙으로 오해 → 0100은 관례적 예시일 뿐, 시스템이 꼭 0100만 쓰는 건 아니다.

화면 프로그램의 뼈대를 잡았다. 이제 실제 화면 번호와 Screen Painter로 화면을 그려 보자 → Screen Number와 Screen Painter.
