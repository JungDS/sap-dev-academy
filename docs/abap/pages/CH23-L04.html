CDS View Entity 기초

Annotation과 의미 부여

뷰·필드에 업무 의미를 단다 — @Annotation.

Track 1Chapter 23Lesson 4고급

왜 값만으로는 부족할까

테이블과 뷰는 값을 담거나 계산할 수 있다. 그러나 값만으로는 업무 의미가 충분히 전달되지 않는다.

  • artist라는 필드는 화면에 "아티스트"라고 보여야 할 수 있다.
  • 항공편 price는 단순 숫자가 아니라 특정 통화(currency)와 짝을 이루는 금액이다.
  • capacity는 숫자지만 좌석 수라는 수량 의미를 가진다.

프로그램마다 라벨을 직접 쓰고, 화면마다 목록 순서를 직접 정하고, 금액과 통화의 관계를 따로 처리하면 같은 데이터가 소비처마다 다르게 보인다. CDS (@로 시작하는 metadata 선언)은 이 의미를 모델 가까이에 붙인다. 그래서 UI·분석·검색·권한 같은 도구가 CDS의 metadata를 읽어 일관되게 행동한다.

무엇인가 — @로 시작하는 metadata

은 CDS entity 위에 붙으면 뷰 전체에, 필드 위에 붙으면 그 요소에 적용된다.

@AccessControl.authorizationCheck: #NOT_REQUIRED
@EndUserText.label: '공연'
define view entity ZC_Concert
  as select from ZI_Concert
{
  @EndUserText.label: '공연 ID'
  key concert_id,

  @UI.lineItem: [{ position: 10 }]
  @EndUserText.label: '아티스트'
      artist,

  @UI.lineItem: [{ position: 20 }]
  @EndUserText.label: '공연장'
      venue,

  @EndUserText.label: '정원'
      capacity
}
Annotation 역할
@EndUserText.label 사용자에게 보일 설명 라벨.
@UI.lineItem 목록형 UI에서 어떤 필드를 어떤 순서로 보여줄지 힌트(position).
@Semantics.* 필드의 업무 의미(이름·주소·금액·통화·수량·단위 등).
@AccessControl.authorizationCheck CDS 접근 제어를 어떻게 다룰지(권한은 뒤 레슨).

@Semantics는 특히 짝을 조심해야 한다. 금액 필드는 통화 필드를, 수량 필드는 단위 필드를 가리켜야 한다. L01의 항공편 모델 ZI_Flight에는 price와 currency가 실제로 있으니, 이 둘로 금액-통화 짝을 선언한다.

@Semantics.amount.currencyCode: 'currency'

@Semantics.currencyCode: true

수량도 같은 원리다(@Semantics.quantity.unitOfMeasure: 'unit' ↔ @Semantics.unitOfMeasure: true). 가장 흔한 잘못은 금액 필드가 자기 자신을 통화로 가리키는 것(price가 currency가 아니라 price를 가리킴)이다.

입문자는 "annotation은 주석이니까 없어도 실행만 되면 된다"고 생각하기 쉽다. 하지만 CDS annotation은 일반 주석이 아니라 활성화된 metadata다. 도구가 읽으므로 틀린 annotation은 화면·분석·서비스에서 실제 문제가 된다.

직접 확인해 보기

아래에서 라벨을 켜고/끄고(헤더가 기술 필드명 ↔ 업무 라벨로 바뀜), 금액의 통화 짝을 맞춰 보고(자기참조면 오류), 목록 순서를 바꿔(position 재할당) annotation의 효과를 직접 보자.

어떻게 확인하는가

  • 위치 — 뷰 전체 라벨은 define view entity 위에, 필드 라벨·UI annotation은 그 필드 바로 위에 있어야 한다.
  • 반영 — Data Preview나 metadata outline에서 라벨이 반영되는지 본다(적어도 활성화 오류가 없어야 한다).
  • 짝 존재 — @Semantics.amount.currencyCode: 'currency_code'라고 썼다면 같은 select list에 currency_code 요소가 실제로 있어야 한다.
  • UI는 힌트 수준 — 이 챕터는 Fiori Elements 화면을 만드는 곳이 아니다. @UI.lineItem은 position 중심으로 "이렇게 보이도록 힌트를 준다"까지만 확인한다.

흔한 실수와 점검 🚧

  • 주석 위치 틀림 → 필드 annotation은 바로 아래 필드에 붙는다. @UI.lineItem을 artist 위에 썼다고 생각했는데 사이에 빈 줄·다른 필드가 끼면 의도와 다른 필드에 붙을 수 있다.
  • UI 주석을 기반(ZI_)에 과다 → ZI_는 재사용 기반이다. UI 표현은 소비 계층 ZC_나 Metadata Extension이 더 자연스럽다(역할을 섞지 않기).
  • @Semantics 자기참조 → 금액 필드는 다른 통화 필드를 가리켜야 한다. 자기 자신을 가리키면 의미가 무너진다.
  • 임의 annotation → annotation은 SAP가 정의한 것만 도구가 읽는다. 이름이 그럴듯하다고 읽어 주는 게 아니다.

의미를 달았다. 그런데 UI 주석이 늘어나면 본문이 복잡해진다 — 주석을 별도 파일로 빼는 Metadata Extension → Metadata Extension 기초.
