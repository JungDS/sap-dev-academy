Enhancement / BAdI / User Exit

Clean Core 관점의 확장 기준

업그레이드에 강한 확장 — Clean Core.

Track 2Chapter 29Lesson 5고급

표준은 깨끗하게, 확장은 분리

CH23-L08의 Clean Core 원칙이 확장에도 그대로 적용된다. 표준을 직접 고치지 않고, 공개 확장점(BAdI)·Released API로만 확장해야 업그레이드에 강하다.

기준

  • 확장점(BAdI) 우선 — SAP이 보장한 자리만.
  • Released API만 — 비공개 ·테이블 직접 접근 금지(클라우드에서 깨짐).
  • Key User / Developer Extensibility — S/4HANA Cloud의 공식 확장 계층을 쓴다.
  • 확장 코드는 별도 패키지로 분리해 추적·이송을 명확히.

온프레미스의 유연함(Implicit·Modification)과 클라우드의 규율(BAdI·Released) 사이에서, 가능한 한 Clean Core 쪽을 택하는 게 미래지향적이다.

친화 여부 가려내기

각 확장 방식이 Clean Core에 맞는지 퀴즈로 가려 보라.

흔한 실수와 점검

  • 비공개 API 확장 → 업그레이드·클라우드 전환에서 깨진다.
  • 확장 추적 불가 → 별도 패키지·문서로 무엇을 왜 확장했는지 남긴다.

🧩 도전 과제 (서브퀘스트)

선택 과제 — 표준을 건드리지 않고 확장한다.

  1. 적절한 BAdI를 찾아(SE18) 예매 관련 검증 로직을 구현(SE19)하는 시나리오를 설계한다.
  2. BAdI가 없는 지점은 Enhancement Point(Implicit)로 최소 개입하는 방안을 제시한다.
  3. (심화) 같은 확장을 Clean Core(Released API·별도 패키지) 관점에서 재검토한다.

힌트: L02(Enhancement Point)·L03(BAdI)·L04(Implicit/Explicit)·L05(Clean Core).
