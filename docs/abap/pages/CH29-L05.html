Clean Core 관점의 확장 기준

업그레이드에 강한 확장 — Clean Core.

Track 2Chapter 29Lesson 5고급

표준은 깨끗하게, 확장은 분리

확장은 요구를 빨리 푸는 도구지만, 잘못 쌓이면 표준을 점점 무겁게 만든다. 처음엔 작은 exit 하나였는데, 몇 년 뒤엔 업그레이드마다 깨지는 표준 변경·어디서 실행되는지 모르는 implicit· 내부 테이블 직접 접근 Z 로직이 쌓인다. 기능은 많아도 바꾸기 어려운 시스템이 된다.

Clean Core(ABAP Cloud와 Released API 원칙에서 본 그 원칙)는 표준 핵심을 깨끗이 유지하고, 확장은 표준이 허용한 안정적 지점·공개 계약으로 분리하자는 사고방식이다. 유행어 암기가 아니라, 앞에서 배운 확장 수단을 운영에 넣을 때 "미래 업그레이드·Cloud-ready를 해치지 않는가" 를 판단하는 기준이다.

💡 핵심 질문은 "기술적으로 가능한가"가 아니라 "공개된 계약인가." 오늘 on-premise라 내부 테이블을 직접 읽을 수 있어도, 그게 미래에도 안정적인 계약이란 뜻은 아니다.

좋은 확장의 4가지 특성

  1. 표준 소스를 직접 수정하지 않는다.
  2. SAP가 제공한 명시적 확장 계약(BAdI·Enhancement Point)을 우선한다.
  3. Released API·안정적 interface로만 표준과 통신한다(비공개 ·테이블 직접 접근 X).
  4. 고객 코드는 별도 package·명확한 이름·transport·테스트 근거를 가진다.

ABAP Cloud 관점에선 세 단어를 구분한다.

용어 뜻
Released API ABAP Cloud에서 접근이 허용(release)된 객체
Developer Extensibility ADT에서 released API로 upgrade-stable 커스텀 코드
Key User Extensibility 지정된 확장점에서 key user 도구로 하는 제한적 확장

친화 여부 가려내기

각 확장 방식이 Clean Core에 맞는지 퀴즈로 가려 보라.

흔한 실수와 점검

  • "Clean Core는 Cloud 얘기"로 치부 → on-premise도 업그레이드·패치·운영 분석·개발자 교체가 있다. 깨끗한 코어는 classic에서도 비용을 줄인다.
  • Released API = SAP 표준 로 착각 → SAP가 만든 객체라고 다 release된 API는 아니다. release contract·제한 언어 버전 visibility 기준이 따로 있다.
  • implicit을 Clean Core 구멍으로 → modification이 아니어도 내부 변수·순서에 깊게 붙으면 업그레이드에 약하다. "표준 안 고쳤다"만으로 부족 — "안정적 계약을 썼다"까지.
  • 확장을 문서화 안 함 → 어느 BAdI에 어떤 filter로, 어떤 업무 규칙을, 비활성화하면 뭐가 사라지는지 없으면 장애 분석이 지옥. 문서가 곧 운영 안전장치.

정리

  • 이 챕터의 결론 = "확장을 할 수 있다"가 아니라 "표준을 깨끗하게 유지하며 확장한다."
  • 우선순위 = BAdI·Released API 같은 공개 계약 > Customer Exit·implicit(레거시·제한적) > Modification(최후).
  • 확장 요구를 받으면 코드부터 쓰지 말고 확장 지점과 책임 범위를 먼저 판단한다.

🧩 도전 과제 (서브퀘스트)

선택 과제 — 표준을 건드리지 않고 확장한다.

  1. 적절한 BAdI를 찾아(SE18) 예매 검증 로직을 구현(SE19)하는 시나리오를 설계한다.
  2. BAdI가 없는 지점은 Enhancement Point로 최소 개입하는 방안을 제시한다(어느 수단을 쓸지 판단).
  3. (심화) 같은 확장을 Clean Core(Released API·별도 패키지) 관점에서 재검토한다.
