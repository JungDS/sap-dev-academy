내부 테이블을 정하는 세 가지

TYPE TABLE OF ts_person 한 줄에는 사실 세 가지 속성이 들어 있다. 이 셋을 알면 테이블을 의도대로 설계할 수 있다.

  1. Line Type(행 모양) — 한 행의 타입.
  2. Primary Key(기본 키) — 행을 찾고 구분하는 기준.
  3. Table Kind(종류) — 내부적으로 어떻게 저장·탐색하나.

키까지 명시하면 선언 한 줄에 셋이 다 보인다. 테이블 타입 이름은 tt_(table type) 접두어가 관례다.

ABAP
1
2
3
4
TYPES tt_person TYPE STANDARD TABLE OF ts_person
                WITH NON-UNIQUE KEY name.

DATA gt_person TYPE tt_person.

① Line Type — 행은 꼭 Structure가 아니어도 된다

행 모양은 보통 Structure지만, 숫자 하나·문자 하나 같은 단일(elementary) 타입일 수도 있다.

ABAP
1
2
3
DATA gt_num    TYPE TABLE OF i.           " 행이 정수 하나
DATA gt_person TYPE TABLE OF ts_person.   " 행이 Structure
DATA gt_amt    TYPE TABLE OF zde_amount.  " 행이 DDIC Data Element

Chapter 5 · DDIC Structure에서 본 "DDIC 타입으로 쓸 수 있는 것"과 같은 원리다. 나중에 배울 DB 테이블(Transparent Table)의 행 모양도 그대로 Line Type으로 줄 수 있다(다음 챕터에서 직접 해 본다).

② Primary Key — 행을 무엇으로 찾나

키는 "이 행을 무엇으로 찾고 구분할 것인가"이다. 사번이 유일하다면 사번이 키 후보고, 구구단이라면 dan+mul 조합이 한 줄을 식별한다.

키를 명시하지 않고 TYPE TABLE OF만 쓰면(지금까지 우리가 한 방식) standard key가 잡힌다. 행 타입의 문자 계열(문자·) 필드 전부를 묶은 느슨한 키다. 위 예처럼 WITH NON-UNIQUE KEY name으로 직접 정할 수도 있다(UNIQUE면 중복 금지).

Internal Table의 키는 안에서 행을 찾는 기준일 뿐이다. DB의 Primary Key(중복 저장을 막는 제약 장치)와는 다르다(그쪽은 다음 챕터에서 만난다).

③ Table Kind — 세 종류

종류 저장 / 탐색 번호() 접근 언제
STANDARD 입력 순서 유지, 키 탐색은 순차 O 기본. 대부분의 처리
SORTED 키 순서 자동 유지, 이진 탐색 O 정렬 유지·빠른 키/범위 탐색
HASHED 해시로 단건 탐색(행 수와 무관) X 유일 키 대량 데이터 단건 조회
  • STANDARD·SORTED는 Index Table이다. 행마다 번호가 붙어 "n번 행"으로 접근할 수 있다. HASHED는 행을 관리하는 인덱스가 없어 번호 접근이 안 되고 키로만 찾는다.
  • HASHED는 키가 UNIQUE(중복 금지)여야 한다.
  • 처음엔 STANDARD로 충분하다. SORTED/HASHED는 접근 패턴이 분명하고 성능이 절실할 때 다시 본다.

로컬 vs 전역(DDIC) Table Type

Structure처럼 테이블 타입도 로컬(TYPES)·전역(SE11 Table Type)으로 둘 수 있다.

SE11 → Data Type → Table Type → ZTT_PERSON → 행 타입에 Structure ZST_PERSON → 활성화. 그러면 여러 프로그램이 같은 테이블 타입을 공유한다.

ABAP
1
DATA gt_person TYPE ztt_person.   " 전역 Table Type

흔한 실수와 점검

  • 행 타입과 테이블 타입 혼동 → ts_person은 한 행의 모양, TABLE OF ts_person은 여러 행의 모양. 행 모양을 먼저 정한다.
  • 처음부터 SORTED/HASHED 남발 → 기본은 STANDARD. 측정 후 바꾼다.
  • HASHED에 번호 접근 → HASHED는 가 없다. 키로만 찾는다.
  • 키를 DB 제약으로 오해 → Internal Table의 키는 안 탐색 기준이다.

테이블을 어떻게 정의하는지 알았다. 이제 한 행을 콕 집어 넣고, 찾고, 고치는 법이다 → 단일 행 제어.
