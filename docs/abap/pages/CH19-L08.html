Track 1Chapter 19Lesson 8중급

🎫 SELECT를 모던으로 — 단, 결과는 검증

공연·상태로 예매 필터·공연별 예매현황 리포트에서 만든 콘서트 조회를, 이 챕터의 모던 SQL(@·콤마·@DATA)로 다듬는다. 목표는 앱을 새로 만드는 게 아니라 이미 동작하는 조회를 모던 SQL 경계 안에서 안전하게 바꾸고, 업무 결과가 그대로인지 검증하는 것이다.

🚧 이 실습이 하지 않는 것: 예매 저장(DML)·lock·OO manager 클래스·CDS/RAP·예외 클래스. 모두 다음 챕터들의 주제다.

" classic
DATA lt_book TYPE TABLE OF zbooking.
SELECT * FROM zbooking
  INTO TABLE lt_book
  WHERE concert_id IN so_conc.

" modern
SELECT * FROM zbooking
  WHERE concert_id IN @so_conc
  INTO TABLE @DATA(lt_book).

so_conc는 selection screen에서 만들어진 range table이다. SQL 안에선 ABAP 값이므로 **@so_conc**로 표시한다.

② 공연별 집계 — LEFT OUTER JOIN · COALESCE · CASE

SELECT c~concert_id, c~artist, c~capacity,
       COALESCE( SUM( b~seats ), 0 ) AS booked,
       CASE WHEN COALESCE( SUM( b~seats ), 0 ) >= c~capacity THEN 'FULL'
            ELSE 'OPEN' END AS seat_status
  FROM zconcert AS c
  LEFT OUTER JOIN zbooking AS b
    ON  b~concert_id = c~concert_id
    AND b~status <> 'C'
  WHERE c~concert_id IN @so_conc
  GROUP BY c~concert_id, c~artist, c~capacity
  INTO TABLE @DATA(lt_stat).

여기 모던 SQL이 여럿 모였다: 콤마 목록, @so_conc, @DATA, COALESCE(예매 없는 공연의 합계 null → 0), CASE(매진 상태 컬럼).

🧩 별칭을 같은 SELECT 안에서 다시 못 쓴다. SUM( b~seats ) AS booked로 만든 booked를 같은 SELECT list의 CASE에서 바로 쓸 수 없다(SELECT list alias는 같은 list의 SQL expression operand로 불가). 그래서 CASE 안에서 COALESCE( SUM( b~seats ), 0 )를 반복한다. 길어 보여도 규칙을 지킨 코드다.

③ 가장 조심할 곳 — 취소 조건은 ON에 ⚠️

취소 예매(status = 'C') 제외 조건을 어디에 두느냐가 결과를 바꾼다.

  • ON에 두면(… ON b~concert_id = c~concert_id AND b~status <> 'C'): 예매가 없는 공연(C003)도 목록에 남는다(booked = 0).
  • WHERE에 두면(… WHERE b~status <> 'C'): outer join 뒤 WHERE가 오른쪽 null 행을 지워, 예매 없는 공연 C003가 사라진다.

예매 없는 공연도 보여 주는 게 목적이므로, 오른쪽 테이블 조건은 ON에 둔다. 위 체험에서 직접 바꿔 보자.

결과 검증 ✅

샘플 데이터(so_conc = C001·C002·C003):

concert_id artist cap
C001 정훈영 Band 100
C002 ABAP Trio 50
C003 UI5 Night 80

zbooking: 손흥민 C001 2(N), 아이유 C001 3(N), 차은우 C001 1(C=취소), 정훈영 C002 50(N).

concert_id booked seat_status 이유
C001 5 OPEN 취소 1건 제외, 2 + 3 만 합산
C002 50 FULL 예약 좌석이 capacity와 같음
C003 0 OPEN 예매 없음 → SUM null → COALESCE로 0

실행 후 확인할 것: ① lt_book 행 수 = sy-dbcnt ② booked 값 ③ seat_status가 capacity 기준과 맞는지 ④ C003가 사라지지 않는지 ⑤ 취소 C가 합계에서 빠졌는지.

결과 메시지 — string template

SQL이 아니라 ABAP 출력이다. 지난 챕터의 string template로 확인한다.

ABAP
1
2
3
LOOP AT lt_stat INTO DATA(ls_stat).
  WRITE / |{ ls_stat-concert_id } { ls_stat-artist } 예약 { ls_stat-booked }/{ ls_stat-capacity } - { ls_stat-seat_status }|.
ENDLOOP.

직접 해 볼 과제

  1. 공연·상태로 예매 필터 조회를 @·@DATA로 바꾼다.
  2. 공연별 예매현황 리포트 를 콤마·@DATA로 바꾼다(필드 사이 콤마 주의).
  3. (도전) SQL Expression의 CASE로 매진 상태 컬럼을, COALESCE로 null을 0으로 처리한다.

모던 SQL 자기점검 (펼치기)
  • WHERE concert_id IN @so_conc에서 @를 왜 붙였나? 안 붙이면 어떻게 되나?
  • SELECT list와 GROUP BY를 모두 콤마로 맞췄나? 공백과 섞으면?
  • @DATA( ) target에 무엇이 만들어졌는지 디버거로 봤나?
  • 취소 조건을 ON에 뒀나 WHERE에 뒀나? 예매 없는 공연이 남으려면 어디에?
  • 예매 없는 공연의 booked가 null이 아니라 0으로 나오나? 무엇이 그렇게 만드나?
  • booked alias를 같은 SELECT의 다른 식에서 바로 쓰지 않았나?
  • 실행 뒤 sy-subrc·sy-dbcnt를 확인했나?

흔한 실수와 점검 🚧

  • @ 누락 → 모던 SELECT의 ABAP 변수·so_conc엔 @. 컬럼엔 안 붙인다.
  • 콤마/공백 혼용 → 모던은 필드 콤마. classic 공백과 섞지 않는다.
  • 취소 조건을 WHERE에 → 예매 없는 공연이 사라진다. 보존하려면 ON에.
  • COALESCE 오해 → DB null 대체다. 이미 0인 값을 바꾸는 게 아니다.

modern SQL로 조회를 다뤘다. 그런데 CTE·window 같은 더 복잡한 조회가 남았다. 고급 SQL로 이어 가자 → 고급 SQL은 언제 필요한가.
