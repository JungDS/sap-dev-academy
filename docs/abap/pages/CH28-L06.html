화면 검증과 저장 검증은 다르다

셀 단위 검증(L05)을 통과해도, 저장 직전 전체 규칙(중복·정원 합계·권한)을 다시 본다. 화면 검증은 입력 편의, 저장 검증은 데이터 무결성의 마지막 방어선이다.

COMMIT WORK.
  • check_changed_data( ) — 편집 중인 값을 내부 테이블로 강제 반영.
  • 최종 검증 통과 후에만 DML.

저장 흐름 체험

값을 고치고 저장을 눌러 보라. 잘못된 값이 있으면 저장이 거부되고, 모두 통과하면 MODIFY + COMMIT이 실행되는 흐름을 확인할 수 있다.

흔한 실수와 점검

  • check_changed_data 누락 → 마지막 셀 편집이 반영 안 된 채 저장될 수 있다.
  • 화면 검증만 믿기 → 저장 전 전체 규칙을 다시 확인(무결성).

🧩 도전 과제 (서브퀘스트)

선택 과제 — 예매 좌석수를 표에서 직접 편집한다.

  1. 예매 목록 Grid의 SEATS 컬럼을 편집 가능(edit)으로 만들고 입력 를 등록한다.
  2. DATA_CHANGED에서 좌석수가 잔여석을 넘으면 그 셀에 오류를 등록(add_protocol_entry)한다.
  3. (심화) 저장 직전 check_changed_data 후 정원 규칙을 최종 검증하고 MODIFY+COMMIT(CH24).

힌트: L01(edit)·L02(DATA_CHANGED)·L05(검증/오류표시)·L06(저장 전 검증).

입력·검증·저장까지 완성했다. 이제 SAP 표준을 건드리지 않고 확장하는 법 → CH29(Enhancement·BAdI).
