Inline Declaration

위에서 선언, 아래서 사용 — 멀어지면 잡음

classic ABAP은 변수를 먼저 DATA로 선언하고 한참 아래에서 사용한다. 학습 초기엔 좋다 — 어떤 변수가 있는지 위에서 한눈에 보인다. 그런데 코드가 길어지면, 맨 위에 선언한 ls_book을 80줄 아래서 쓰면 읽는 사람은 "이 변수 값이 어디서 들어오지?"를 계속 추적해야 한다. 선언이 업무 의도와 떨어지면 친절함이 아니라 잡음이 된다.

Inline declaration은 값이 필요한 바로 그 자리에서 변수를 선언한다.

📌 이 챕터부터 모던 문법을 쓴다. Grid ALV 기초까지는 일부러 classic만 썼다 — 이제 경계를 넘었다.

DATA() — 값이 만들어지는 자리에서 선언

DATA(변수명)은 아무 데서나 쓰는 축약이 아니다. ABAP이 "여기엔 어떤 타입이 와야 하는지" 를 아는 위치(declaration position)에서만 쓴다. 즉 "타입을 안 써도 된다"가 아니라 "문맥이 타입을 제공한다" 가 핵심이다.

위치 classic modern
READ TABLE … INTO INTO ls_book (선언 별도) INTO DATA(ls_book)
LOOP AT … INTO INTO ls_row (선언 별도) INTO DATA(ls_row)
계산 결과 대입 DATA lv_total TYPE i. 후 대입 DATA(lv_total) = …

" modern — READ TABLE의 행 타입을 ABAP이 알아 ls_book 타입 자동 결정
READ TABLE lt_booking INTO DATA(ls_book) WITH KEY booking_id = '0001'.

LOOP AT lt_booking INTO DATA(ls_row).
  WRITE: / ls_row-customer.
ENDLOOP.

DATA(lv_total) = ls_book-seats * 10000.   " 오른쪽 결과 타입으로 결정

💡 inline 변수도 그냥 변수다 — 마법처럼 사라지는 임시값이 아니라, 선언 이후 같은 처리 블록에서 쓸 수 있다. 다만 너무 많은 문장을 한 줄에 몰면 다시 읽기 어렵다.

FINAL() — 다시 안 바뀌는 값

값을 한 번만 정하고 이후 변경을 막고 싶으면 FINAL()로 선언한다(읽기 전용 의도).

ABAP
1
FINAL(lv_max) = 100.    " 이후 lv_max 재대입 시 문법 오류

⚠️ FINAL()은 누적 변수에는 맞지 않는다. 합계를 더해 가는 lv_total을 FINAL로 선언하면 다음 누적 줄에서 막힌다. 한 번 정하고 바꾸지 않을 값에만 쓴다.

직접 판정해 보기 — Inline 선언 위치 판정기

문장 카드를 골라 DATA() 인라인이 허용되는지/보류인지, FINAL()로 바꾸면 재대입이 왜 막히는지 확인하자. SELECT 안의 인라인은 다음 챕터에서 다룬다는 것도 본다.

흔한 실수와 점검

  • 이미 선언한 이름을 또 DATA() → 같은 처리 범위에서 중복 선언이다. 값을 다시 넣으려면 기존 변수에 대입한다.
  • 타입을 알 수 없는 위치에서 DATA() → 문맥이 타입을 제공해야 쓸 수 있다.
  • SELECT에 DATA() → SQL 안의 인라인은 별도 escape가 필요한 New Open SQL이라 Classic Open SQL과 Modern ABAP SQL 비교에서 — 이 챕터의 인라인은 LOOP·READ·대입 등 SQL 밖에서.

정리

  • Inline declaration의 핵심은 짧게 쓰기가 아니라 선언 위치를 의도 가까이로 옮기는 것이다.
  • DATA()는 값이 만들어지는 자리에서 변수를 만들고(문맥이 타입 제공), FINAL()은 이후 바꾸면 안 되는 값을 표시한다.

선언을 의도 가까이로 옮겼다. 이제 구조체·테이블을 한 식으로 채우는 VALUE를 보자 → VALUE Constructor Expression.
