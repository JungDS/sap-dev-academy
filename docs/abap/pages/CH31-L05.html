잘못된 입력을 그 자리에서

DATA_CHANGED Event에서 검증하는 법을 배웠다. 여기선 거기에 UX를 더한다 — 사용자는 "어느 행·어느 셀·왜" 틀렸는지 알아야 고친다. 여러 셀이 동시에 틀릴 수 있으니, 셀은 빨갛게 + 오류는 목록으로 모아 보여 준다.

METHOD on_data_changed.
  LOOP AT er_data_changed->mt_good_cells INTO DATA(ls_cell).
    IF ls_cell-fieldname <> 'SEATS'.
      CONTINUE.
    ENDIF.
    TRY.
        DATA(lv_seats) = CONV i( ls_cell-value ).
      CATCH cx_sy_conversion_no_number.
        er_data_changed->add_protocol_entry(
          i_msgid = 'ZMSG' i_msgty = 'E' i_msgno = '003'
          i_msgv1 = '숫자를 입력하세요'
          i_fieldname = ls_cell-fieldname
          i_row_id    = ls_cell-row_id ).
        CONTINUE.
    ENDTRY.
    IF lv_seats < 1 OR lv_seats > 10.
      er_data_changed->add_protocol_entry(
        i_msgid = 'ZMSG' i_msgty = 'E' i_msgno = '004'
        i_msgv1 = '좌석 수는 1부터 10까지'
        i_fieldname = ls_cell-fieldname
        i_row_id    = ls_cell-row_id ).
    ENDIF.
  ENDLOOP.
  er_data_changed->display_protocol( ).      " 누적 오류 목록 표시
ENDMETHOD.
  • add_protocol_entry — 오류 메시지 + 셀 위치(i_fieldname·i_row_id)를 함께 등록.
  • display_protocol( ) — 누적된 오류 목록을 사용자에게 보여 준다.
  • i_msgid는 메시지 클래스(SE91)다 — 메시지 ID·번호로 "어디가 왜 틀렸는지"를 엮는다.

오류를 만들어 보기

SEATS에 잘못된 값(0·11·문자·빈칸)을 넣어, 셀이 빨개지고 오류 목록에 모이는 걸 확인해 보라. 정상값으로 고치면 오류가 사라진다.

흔한 실수와 점검

  • 추상적 오류 메시지 → "입력 오류"보다 "좌석 수는 1부터 10까지" 처럼 다음에 할 행동을 담는다.
  • 오류 등록하고 저장 통과 → 이 단계의 프로토콜은 화면 단계 피드백이다. 저장 전 전체 검증은 다음 레슨에서 다시 — 입력 편의(여기)와 무결성 방어선(저장)은 역할이 다르다.
  • 무조건 팝업 → 많은 셀을 연속 편집하는 화면에선 하단 오류 목록이 팝업보다 편하다. 화면 성격에 맞게 패널·상태바·팝업을 고른다.

정리

  • Editable ALV 검증은 사용자가 어느 셀을 어떻게 고칠지 알려 줘야 한다.
  • add_protocol_entry(메시지+위치) → display_protocol(목록)으로 오류를 모아 보여 준다.
  • 이 단계는 입력을 친절하게 고치게 하는 자리 — 최종 저장 가능 여부는 저장 직전에 다시 판단한다.

입력 단계 검증을 했다. 마지막으로 DB에 쓰기 직전 최종 검증 → 변경 데이터 DB 반영 전 검증(다음 레슨).
