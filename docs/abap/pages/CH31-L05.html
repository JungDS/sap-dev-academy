목록 조회 — GET_ENTITYSET

DPC_EXT 에서 EntitySet의 ..._GET_ENTITYSET 를 재정의해 데이터를 돌려준다. CH08의 SELECT가 여기 쓰인다.

METHOD concertset_get_entityset.
  " URL 필터를 ABAP 조건으로
  DATA(lt_filter) = io_tech_request_context->get_filter( )->get_filter_select_options( ).

  SELECT concert_id, artist, venue, capacity
    FROM zconcert
    ORDER BY concert_id                       " OFFSET엔 정렬이 필요
    INTO CORRESPONDING FIELDS OF TABLE @et_entityset
    UP TO @iv_top ROWS
    OFFSET @iv_skip.         " $top(개수) + $skip(건너뛰기) 페이징
ENDMETHOD.
  • et_entityset — 돌려줄 목록(모델 구조).
  • $filter·$top·$skip·$orderby 같은 OData 쿼리를 ABAP 조건으로 매핑.
  • 단건은 GET_ENTITY.

흔한 실수와 점검

  • 필터 무시 → $filter를 안 받으면 전체 반환(성능·정합 문제). 매핑한다.
  • 페이징 무시 → 대량은 $top/$skip을 SELECT에 반영.

신규 개발은 RAP(CH23) 이 OData를 더 적은 코드로 노출한다 — SEGW는 기존 자산 이해용.

🧩 도전 과제 (서브퀘스트)

선택 과제 — 예매 데이터를 표준 메시지·서비스로 연계한다.

  1. 예매 확정 시 IDoc(Outbound)으로 외부에 통지하는 흐름을 설계한다(Basic Type·Partner Profile).
  2. 수신 실패 IDoc(상태 51)을 WE02로 찾아 BD87로 재처리한다.
  3. (심화) 예매 목록을 SEGW OData(GET_ENTITYSET·$filter)로 노출한다 — 신규라면 RAP(CH23)이 더 적은 코드.

힌트: L01L03(IDoc/ALE)·L04L05(Gateway/OData).

시스템 연계(IDoc·OData)를 다뤘다. 이제 느린 프로그램을 빠르게 — 성능 분석·튜닝 → CH32.
