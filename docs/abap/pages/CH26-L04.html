화면·로직·데이터를 분리한다 — MVC.

Track 2Chapter 26Lesson 4고급

한 프로그램에 다 섞으면

조회·계산·화면·가 한 리포트에 뒤섞이면 유지보수가 지옥이다. MVC로 역할을 나눈다.

계층 역할 예
Model 데이터·업무 로직 ZCL_BOOKING_MODEL(조회·계산)
View 표시 ALV/Dynpro 출력
Controller 흐름 조율 이벤트 받아 Model 호출 → View 갱신

DATA(lo_model) = NEW zcl_booking_model( iv_concert = p_concert ).

  • 화면을 바꿔도 Model은 그대로, 로직을 바꿔도 View는 그대로.
  • 테스트도 Model만 떼어 검증 가능(다음 레슨).

한 계층을 바꾸면 어디까지 영향이 갈까

시나리오를 골라, 한 계층을 바꿀 때 나머지가 그대로인지 직접 확인해 보라 — 이게 역할을 나누는 진짜 이유다.

흔한 실수와 점검

  • Model에 화면 코드 → 표시 관심사는 View로. Model은 순수 로직(화면을 몰라야).
  • 과분리 → 작은 리포트까지 3계층은 과할 수 있다. 규모에 맞게.

정리

  • Model(로직)·View(표시)·Controller(흐름)로 관심사를 나눈다.
  • 한 계층 변경이 나머지에 안 번진다 → 유지보수·테스트가 쉬워진다.

구조를 나눴다. 그 덕에 테스트 가능한 설계가 가능해진다 → 다음 레슨.
