Draft — 미저장 변경 보관

L04의 draft table로, 사용자가 입력하다 만 변경이 초안으로 보관된다. 다른 기기에서 이어 하거나, 실수로 닫아도 복구된다. managed RAP이 초안 처리를 대부분 자동으로 해 준다.

권한 — DCL과 인스턴스 권한

  • DCL(CH23-L06)로 "누가 어떤 공연의 예매를 볼지" 행 단위 제어.
  • BDEF의 authorization(global/instance)으로 동작별 권한(예매 생성·취소 권한).

운영 마무리

  • ATC·ABAP Unit 통과(CH38) → 이송(DEV→QAS→PRD).
  • Application Log로 추적, Clean Core(Released API만, CH24-L08·CH32).

🎓 여정의 끝 — 그리고 시작

정훈영의 첫 WRITE 'Hello'(CH01)에서 출발해, 변수·타입·구조·테이블·SQL을 익히고, 화면과 ALV를 만들고, 모던 문법·OO·CDS·RAP으로 콘서트 예매 풀스택 앱을 완성했다. Track-2에서 ·동시성··성능·운영까지 — 현업 개발자의 한 바퀴를 돌았다.

여기서 끝이 아니다. 실제 프로젝트, UI5/Fiori 심화, 새 릴리스의 기능 — 배운 토대 위에 계속 쌓아 가면 된다. 0부터 시작한 당신은, 이제 개발자다. 🚀

흔한 실수와 점검

  • 권한 나중에 → 권한·Draft는 설계 초기에 함께 고려.
  • 여기서 멈춤 → 작은 기능이라도 직접 만들어 보며 실력을 굳힌다.

🧩 도전 과제 (서브퀘스트)

선택 과제 — Capstone을 끝까지 완성한다.

  1. CH09부터 쌓은 콘서트 예매 앱을 RAP+Fiori로 통합해 List/Object Page를 띄운다.
  2. 정원 초과 validation과 취소 action을 직접 구현해 동작을 확인한다.
  3. (심화) Draft·Authorization(DCL) 을 더하고, ATC·이송(CH38)까지 운영 관점으로 마무리한다.

힌트: L02~L05(ZI_/ZC_·BDEF/BIMP·Action/Validation)·L06(Service/Fiori)·L07(Auth/Draft/운영).
