대량 데이터 처리와 Package 설계

수백만 건을 안정적으로 — 패키지·병렬·푸시다운.

Track 2Chapter 32Lesson 5고급

규모가 커지면 설계가 다르다

수백만 건은 "잘 짠 SELECT" 이상의 설계가 필요하다.

  • 패키지 처리(CH24-L05) — 일정 건수마다 처리·COMMIT로 메모리·잠금 관리.
  • 병렬 처리 — 작업을 나눠 동시에(aRFC·병렬 프레임워크). 단 복잡·자원 고려.
  • Code Pushdown — 가공을 ABAP이 아니라 DB(HANA)에서 하도록 — CDS·AMDP(CH33)· modern SQL 식(CH19). "데이터를 ABAP으로 끌어와 LOOP" 대신 DB에서 집계해 결과만.

흔한 실수와 점검

  • 전부 메모리로 → 수백만 건을 한 번에 itab에 담으면 메모리 폭발. 패키지·스트리밍.
  • ABAP에서 집계 → DB가 잘하는 일(집계·정렬)은 DB로 내린다.

🧩 도전 과제 (서브퀘스트)

선택 과제 — 예매현황 리포트를 빠르게 만든다.

  1. 예매현황 리포트를 ST05로 트레이스해 가장 느린 SELECT를 찾는다.
  2. LOOP 안 SELECT가 있으면 FOR ALL ENTRIES/JOIN(CH13)으로 제거한다.
  3. (심화) 대량 집계를 Code Pushdown(GROUP BY를 DB에서)으로 바꿔 ABAP LOOP 합산을 없앤다.

힌트: L01(ST05)·L02(SAT)·L04(SELECT-in-LOOP)·L05(Pushdown).

성능 분석·튜닝을 다뤘다. DB에서 직접 도는 로직 — AMDP·ADBC → CH33.
