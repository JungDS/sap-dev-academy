Selection Screen 권한/존재 여부 검증 기초

입력 단계에서 "있는 값인가"와 "볼 수 있는 사용자인가"를 분리해 막는다.

Track 1Chapter 15Lesson 7중급

없는 코드와 못 보는 코드는 다르다

선택화면에 잘못된 공연 코드가 들어왔다고 하자. 그런데 "잘못됨"에는 두 가지 전혀 다른 경우가 섞여 있다.

  • 없는 코드 — DB에 그런 공연 자체가 없다. → 존재(data) 문제
  • 못 보는 코드 — 공연은 있지만, 이 사용자에게 볼 권한이 없다. → 보안 문제

이 둘을 "그냥 데이터가 없네"로 뭉뚱그리면, 사용자는 엉뚱한 안내를 받고, 운영 시스템에서는 권한 없는 사람이 데이터에 접근하는 사고로 이어질 수 있다. 그래서 검증을 존재 검증과 권한 검증으로 나눠서 한다.

이 레슨은 Chapter 15 · AT SELECTION-SCREEN 입력 검증에서 배운 입력 검증을, "존재"와 "권한" 두 관문으로 한 단계 더 끌어올린 것이다.

존재 검증 — 진짜 있는 키인가

먼저 입력한 코드가 DB에 실제로 있는지 확인한다. 로 한 건만 찾아보고, 못 찾으면(sy-subrc <> 0) 막는다. 필드 하나(p_conc)만 검사하므로 에서 한다.

PARAMETERS p_conc TYPE zconcert-concert_id.

DATA gv_concert_id TYPE zconcert-concert_id.

AT SELECTION-SCREEN ON p_conc.
  " ① 존재 검증 — 이 공연 코드가 진짜 있나?
  SELECT SINGLE concert_id
    FROM zconcert
    INTO gv_concert_id
    WHERE concert_id = p_conc.
  IF sy-subrc <> 0.
    MESSAGE e003(zmsg) WITH p_conc.   " "없는 공연 코드입니다: &1"
  ENDIF.

💡 gv_concert_id는 프로그램 맨 위에서 선언한 전역 변수라 gv_로 시작한다. e003(zmsg)는 Chapter 15 · AT SELECTION-SCREEN 입력 검증에서 본 메시지 클래스 호출(번호·&1 자리표시자)을 그대로 이어 쓴 것이다.

권한 검증 — 볼 수 있는 사용자인가

존재하는 코드라도, 이 사용자가 그 공연을 볼 권한이 있는지는 또 다른 문제다. ABAP은 명시적 권한 확인에 를 쓴다. (권한 객체)에 정의된 필드 값을 기준으로 접근 가능 여부를 검사하고, 결과를 sy-subrc로 돌려준다.

  " ② 권한 검증 — 이 사용자가 볼 수 있나? (존재 검증을 통과한 뒤)
  AUTHORITY-CHECK OBJECT 'Z_CONCERT'
    ID 'CONCERT' FIELD p_conc
    ID 'ACTVT'   FIELD '03'.          " 03 = 조회(display) 활동
  IF sy-subrc <> 0.
    MESSAGE e004(zmsg) WITH p_conc.   " "이 공연을 볼 권한이 없습니다: &1"
  ENDIF.

sy-subrc의 의미(공식문서 기준):

sy-subrc 뜻
0 통과 — 권한 있음(또는 체크가 꺼져 있어 통과 처리)
4 권한은 있으나 그 값에 대한 권한이 부족
12 그 권한 객체에 대한 권한이 아예 없음

핵심은 단순하다. sy-subrc = 0이면 통과, 그 외에는 막는다.

⚠️ 'Z_CONCERT'·'CONCERT' 같은 권한 객체·필드 이름과 활동 코드는 교육용 예시다. 실제 시스템에서는 SU21 등에서 정의된 권한 객체·필드를 확인해야 한다.

직접 통과시켜 보기 — 존재·권한 이중문

공연 코드 3개를 차례로 두 관문에 통과시켜 보자. C001(있음·권한 있음), C002(있음·권한 없음), C999(없는 코드)가 각각 어느 관문에서 막히는지, 그리고 막힐 때 어떤 메시지가 나가는지 본다. "없는 코드"와 "권한 없음"은 색도 문구도 달라야 한다.

실수와 주의

  • "데이터가 안 보이면 권한 없음"으로 추정 → 존재 검증과 권한 검증은 분리한다. 안 보이는 이유가 없어서인지 권한이 없어서인지는 전혀 다른 문제다.
  • AUTHORITY-CHECK 뒤 sy-subrc 미확인 → 명령만 쓰고 결과를 안 보면 검사한 의미가 없다. 반드시 IF sy-subrc <> 0으로 분기한다.
  • 교육용 권한 객체 이름을 실무에 그대로 사용 → 실제 객체·필드는 시스템 설계마다 다르다.
  • modern 혼입 → 여기는 classic 구간이라 @DATA/@ 같은 New Open SQL은 쓰지 않는다.

정리

  • 선택화면 검증은 문법 검사가 아니라 업무 접근의 첫 관문이다.
  • 로 존재를, 로 권한을 확인한다 — 둘은 별개의 관문이다.
  • 두 검증을 분리하면 오류 메시지가 정확해지고, 위험한 입력이 조회 로직에 닿기 전에 막힌다.

기본 검증까지 익혔다. 다음은 선택화면을 블록·그룹으로 묶어 검증하고, 입력칸에 커스텀 F1(도움말)·F4(입력 도움) 까지 붙이는 고급 를 보자 → Selection Screen 고급 — 블록·그룹 검증과 커스텀 F1·F4.
