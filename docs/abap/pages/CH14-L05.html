Classic DDIC View·유지보수 객체

Table Maintenance Generator / SM30

테이블/뷰에 표준 유지보수 화면을 생성 — SM30로 운영.

Track 1Chapter 14Lesson 5중급

입력 화면을 자동 생성

테이블이나 에 표준 입력 화면을 자동으로 만들어 주는 게 다. 만든 뒤엔 SM30으로 데이터를 입력·수정한다.

생성 (SE11 → Utilities)

SE11 → 테이블/뷰 → Utilities → .

  • Authorization Group(권한 그룹), 화면 번호, Maintenance Type(1단계/2단계) 지정.
  • 생성하면 표준 유지보수 화면(추가·수정·삭제)이 만들어진다.

운영 (SM30)

TEXT
1
SM30 → 테이블/뷰 입력 → Maintain → 행 추가·수정·삭제 → 저장
  • SM30 — 단일 뷰/테이블 유지보수.
  • 저장 시 (이송 대상이면) 에 기록된다(Chapter 1 · 로컬 오브젝트와 이송의 그 흐름).

SM30 문이 열리기까지

SM30은 "마법 주소"가 아니다. 테이블 활성화 → 유지보수 허용(Display/Maintenance) → TMG 생성 → 권한까지 모두 맞아야 열린다. 어느 하나라도 빠지면 화면이 안 열리거나 저장이 막힌다. 직접 게이트를 켜며 무엇이 막는지 확인해 보자.

One Step vs Two Step — 화면 단계 수

Generator를 만들 때 Maintenance Type(유지보수 유형)을 고른다.

  • One Step(1단계) — 목록 화면 하나에서 바로 입력·수정한다. 필드가 적은 단순 테이블에 알맞다.
  • Two Step(2단계) — 목록(개요) 화면 + 상세 화면 둘로 나뉜다. 한 행에 필드가 많아 한 화면에 다 안 들어갈 때, 목록에서 행을 고르고 상세 화면에서 편집한다.

💡 필드가 적으면 One Step, 많으면 Two Step. 나중에 바꿔 재생성할 수도 있다.

Maintenance Dialog — 장단점

Generator가 만들어 주는 표준 유지보수 화면(Maintenance Dialog)은 공짜로 얻는 게 많다.

장점 단점
코딩 없이 입력·수정·삭제 화면 자동 생성 레이아웃 커스터마이즈가 제한적
입력검증·F4·이송 연동이 그대로(Chapter 9) 복잡한 업무 로직엔 부족
권한 그룹으로 접근 통제 (일부는 이벤트로 보완)

단순 마스터데이터 유지보수엔 이걸로 충분하다. 화면을 마음대로 꾸미거나 복잡한 로직이 필요하면 별도 화면(Chapter 16 · Dynpro 기초)을 고려한다.

흔한 실수와 점검

  • Generator 없이 SM30 → "유지보수 불가" 오류. 먼저 Generator로 화면 생성.
  • 권한 그룹 미지정 → 운영에서 권한 통제가 안 된다.
  • Delivery Class·Recording Routine 무시 → 시스템 간 이동이 필요한 customizing 데이터와 이동 안 하는 local 데이터는 관리 정책이 다르다(이송 잡힘 여부에 영향).
  • 생성 성공 ≠ 유지보수 가능 → 반드시 SM30에서 열어·저장·재조회까지 확인한다.

이제 한 테이블에 표준 화면을 붙였다. 그런데 연관된 여러 테이블을 함께 유지보수하려면? → 다음 레슨(View Cluster).
