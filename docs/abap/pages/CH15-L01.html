ABAP Report Event 전체 흐름

실행형 프로그램의 이벤트 순서 — 언제 무엇이 실행되나.

Track 1Chapter 15Lesson 1중급

🖥️이번 Lesson에서 다루는 트랜잭션 코드

위에서 아래로? 아니다

지금까지 START-OF-SELECTION. 아래에 코드를 몰아 뒀다. 그런데 "화면 뜨기 전 기본값", "입력 검증", "조회 후 마무리" 는 서로 다른 시점에 실행돼야 한다. 실행형 프로그램은 한 줄씩 위에서 아래로 도는 게 아니라, 시스템이 정해진 순서대로 각 구획을 호출한다.

🍳 비유 — 식당의 하루: 오픈 준비(재료 세팅) → 주문 접수(검증) → 조리(본 처리) → 마감. 손님이 와도 "조리"부터 하지 않는다. ABAP도 똑같이 시점이 정해져 있다.

한눈에 보는 이벤트 흐름

블록을 꼭 다 써야 하는 건 아니다. 아래에서 탭을 1단계 → 5단계로 누르면, 이벤트 키워드가 하나도 없는 코드부터 전체 생명주기까지 코드와 흐름이 함께 점점 확장된다. 왼쪽 흐름도와 오른쪽 코드를 좌우로 대조하며 "코드를 더하면 실행 흐름이 어떻게 늘어나는가"를 보자.

아래 표는 5단계(최종 구조) 의 전체 이벤트를 한 번에 정리한 것이다.

시점(이벤트) 언제 도나 주로 하는 일
프로그램 적재 직후 1회 거의 안 씀(초기 상수 등)
선택화면 뜨기 전 1회 기본값 세팅
화면 그리기 직전(PBO) 화면 동적 제어(숨김/회색)
사용자가 실행/Enter(PAI) 입력 검증
검증 통과 후 메인 로직 실행(DB 조회·가공)
본 처리 끝 결과 출력·마무리(ALV 등)
  • 각 이벤트 키워드 아래에 그 시점 코드를 둔다.
  • 키워드를 안 쓰면 코드는 암묵적으로 START-OF-SELECTION에 묶인다.
각 이벤트, 한 줄 더 자세히
  • INITIALIZATION — 화면이 보이기 전이라, 사용자가 바꾸기 전의 기본값을 넣기 좋다.
  • AT SELECTION-SCREEN OUTPUT — PBO(Process Before Output). 화면을 그리기 직전이라 필드를 숨기거나 비활성화한다.
  • AT SELECTION-SCREEN — PAI(Process After Input). 사용자가 값을 넣고 실행을 누른 직후라 검증의 자리.
  • START-OF-SELECTION / END-OF-SELECTION — 본 처리와 마무리. 보통 조회는 START, 출력은 END.

왜 자리가 중요한가

기본값은 , 검증은 , 조회는 — 자리를 맞춰야 의도대로 동작한다. 자리를 틀리면 "검증했는데 이미 조회가 끝나 있다" 같은 일이 벌어진다.

흔한 실수와 점검

⚠️ 검증을 에 두면? 이미 메인 로직이 시작된 뒤다. 검증은 반드시 .

⚠️ 키워드 누락? 코드가 전부 본처리로 묶여 순서가 꼬인다.

정리

  • 실행형 프로그램은 정해진 순서로 각 구획을 호출한다(위→아래 X).
  • 기본값·검증·메인 로직·출력은 각자의 에 둔다.
  • 그림의 점선(검증 실패 복귀) 이 핵심 — 검증을 통과해야 본처리로 간다.
