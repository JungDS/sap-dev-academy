Structure (Local · DDIC)

DDIC Structure

SE11에서 DDIC Structure를 만들어 Structure Type을 전역으로 공유한다.

Track 1Chapter 5Lesson 2초급

Structure도 전역으로

Chapter 3에서 타입을 전역(·)으로 올렸듯, Type도 전역으로 올릴 수 있다. 여러 프로그램·화면·테이블이 같은 Structure를 공유하게 하는 게 Structure다.

SE11에서 Structure 만들기

SE11 → Data Type → → ZST_PERSON → Create.

  • (필드)를 추가하고, 각 필드 타입은 보통 로 준다 (라벨·도움말까지 상속).
Component Typing 예 데이터 타입 길이 설명
NAME Data Element ZDE_NAME CHAR 20 이름
AGE Data Element ZDE_AGE NUMC 3 나이
AMOUNT Data Element ZDE_AMOUNT DEC 8 (소수 2) 금액

데이터 타입·길이는 Data Element가 참조하는 에서, 설명(라벨)은 Data Element에서 온다. Chapter 3의 계층이 여기서 그대로 재사용된다.

활성화하면 전역 Structure Type이 된다.

프로그램에서 쓰기

ABAP
1
2
3
4
DATA gs_person TYPE zst_person.

gs_person-name = '정훈영'.
WRITE gs_person-name.

Local (BEGIN OF)와 쓰는 법은 똑같다. 차이는 어디에 생성했는가뿐이다. 프로그램 내부에 생성(선언)하면 Local, ABAP Dictionary(SE11)에서 생성(정의)하면 Global이다. 전역으로 정의해 두면 여러 프로그램이 같은 모양을 공유하고, 한 곳만 고치면 전부 반영된다. 아래에서 그 차이를 직접 비교해 보자.

를 로 준 덕에 라벨까지 얹혀 있는 모습을 트리로 확인해 보자. gs_person-name에 값을 넣는 대입도 눌러 볼 수 있다.

DDIC 타입으로 쓸 수 있는 것 — Domain만 빼고

변수나 의 타입으로 줄 수 있는 객체는 여럿이다. 단 은 직접 못 준다(Chapter 3에서 봤듯 Domain은 를 통해서만).

DDIC 객체 변수/Component 타입으로? 비고
Data Element ✅ 가장 흔함(라벨·도움말 상속)
✅ Structure 안에 Structure(중첩)
Transparent Table ✅ 그 테이블의 행 한 줄 모양
View ✅ 뷰의 행 모양
Table Type ✅ 내부 테이블 타입(→ 다음 챕터)
Domain ❌ 변수 타입 직접 불가(DE 경유)
  • Component에 Data Element 대신 다른 Structure를 주면 중첩 Structure가 된다(다음 레슨에서 INCLUDE·APPEND와 함께).

💡 심화(지금은 가볍게) — Component가 전부 문자(C·N 등)인 Structure는, 통째로 한 문자 변수처럼 다룰 수도 있다(예: year·month·day 8자리 Structure에 sy-datum을 통째로 대입). "그런 트릭이 있다" 정도만 알아 두자.

언제 로컬, 언제 DDIC?

  • 그 프로그램에서만 잠깐 쓰는 임시 묶음 → 로컬(BEGIN OF).
  • 그 외에는 전부 를 권장. 사용처가 워낙 다양해서 일일이 나눌 필요가 없다. "여럿이 공유할 때만"으로 좁혀 생각하지 말자. 한 프로그램에서 한 번만 쓰더라도 DDIC로 만드는 경우가 흔하다.

흔한 실수와 점검

  • 에 Built-in Type 직접 지정 → SE11에서 Component 타입은 대신 Built-in Type(CHAR·NUMC 같은 기본 타입을 직접 지정)으로도 줄 수 있다. 동작은 하지만 라벨·도움말을 잃는다. 가능하면 Data Element를 참조한다.
  • 이름 규칙/활성화 → Z/Y 시작, 활성화(Ctrl+F3) 안 하면 사용 불가.
  • 밑줄(_) 위치 제한 → DDIC ·테이블 이름은 2·3번째 자리에 _가 올 수 없다(이름 끝 _도 불가). ZS_PERSON(3번째가 _)은 거부되고 ZST_PERSON(4번째부터)은 되는 이유다. 실무에서는 ZSMM·ZSSD·ZSPP나 ZMMS·ZSDS·ZPPS처럼 앞자리를 밑줄 없이 채운 이름을 자주 본다.
