로컬 타입의 벽

CH02의 는 편했지만 그 프로그램 안에서만 산다. 회사 전체가 쓰는 "금액", "통화코드" 같은 타입을 매 프로그램이 또 정의하는 건 낭비고, 정의가 갈리면 사고다.

😵 A 프로그램은 금액을 DEC 8,2, B 프로그램은 DEC 11,2로 정의했다면? 같은 "금액"인데 자리수가 달라 데이터가 어긋난다. 한 곳에서 한 번만 정의하고 싶다.

이제 타입을 전역으로 올린다 — 그 시작이 의 이다.

DDIC와 Domain

(SAP Dictionary)는 시스템 전체의 데이터 정의를 한 곳에서 관리하는 중앙 사전이다. 그 안에서 은 순수한 기술 속성만 담는다.

Domain이 아는 것 Domain이 모르는 것
데이터 타입·길이 (예: DEC 8자리, 소수 2) 이 값이 "금액"인지 "단가"인지 (의미)
(선택) 허용 값 목록 = 화면에 뜰 라벨

Domain은 "어떻게 생긴 값인가"만 안다. "이 값이 무슨 의미인가"는 다음 레슨의 가 맡는다. 역할이 갈린다.

SE11에서 Domain 만들기 — 직접

SE11(Dictionary) → Domain → 이름 ZDO_AMOUNT → Create. 말로만 보면 추상적이니, 아래 시뮬레이터에서 직접 만들어 보자. 값을 채우고 💾 저장 → ✓ 검사 → 🔥 활성화 순서로 눌러, DDIC 객체가 살아나는 과정을 눈으로 따라가 보라.

🖱️ 예제 칩 "상태 도메인" 을 누르면 CHAR 1 + O/C가 채워진다. 값 범위 탭에서 고정값을 추가/삭제하며 검사 결과가 어떻게 바뀌는지도 시험해 보자.

활성화 — DDIC의 철칙

방금 시뮬레이터에서 봤듯, 저장만 하면 객체는 "비활성(inactive)" 상태다. 이 상태로는 런타임에서 쓸 수 없다. (Ctrl+F3)를 해야 런타임 객체가 생성되고, 비로소 다른 객체가 이 Domain을 참조할 수 있다.

상태 의미
비활성 (저장만 함) 작업 중인 초안 — 런타임에서 못 씀
활성 (활성화 완료) 런타임 객체 생성됨 — 참조·실행 가능

⚠️ 활성화한 뒤 정의를 바꾸면 다시 비활성이 된다. 바뀐 내용을 반영하려면 재활성화해야 한다. (시뮬레이터에서 활성화 후 길이를 고쳐 보면 배지가 바로 "비활성"으로 돌아간다.)

고정값(Fixed Values)으로 입력 제한

상태값처럼 정해진 값만 받아야 한다면 Value Range(값 범위) 탭에 을 넣는다.

O = Open   (열림)
C = Closed (닫힘)

이렇게 두면 그 Domain을 쓰는 화면 필드는 O/C만 허용하게 된다.

이 고정값, 나중에 어디서 다시 만나나? (더 보기)

여기 적은 은 화면의 입력 도움말(값 선택 목록)과 입력 검증으로 그대로 되돌아온다(CH08). 즉 "한 번 정의 → 여러 곳에서 공짜로 재사용"이 의 힘이다.

흔한 실수와 점검

⚠️ 이름 규칙 — 전역 객체도 Z/Y로 시작. (시뮬레이터에서 AMOUNT로 검사하면 바로 막힌다.)

⚠️ Domain에 라벨을 기대 — 라벨·의미는 Domain이 아니라 소관(다음 레슨).

⚠️ 저장만 하고 끝 — 비활성이라 아무도 못 쓴다. 반드시 까지.

  • 저장 시 패키지·이송요청을 묻는 건 CH01-L05에서 다뤘다($TMP면 로컬).

정리

  • 같은 타입을 프로그램마다 다시 정의하는 낭비를 막으려 타입을 전역으로 올린다 → 의 .
  • 은 기술 속성(타입·길이·소수·)만 담는다. 의미·라벨은 몫.
  • DDIC 객체는 해야 런타임에 산다 — 저장만 하면 비활성.

기술 속성은 정의했다. 그런데 이 값이 화면에 어떤 이름표로 뜨고 무슨 의미인지는 누가 정하나? → (다음 레슨).
