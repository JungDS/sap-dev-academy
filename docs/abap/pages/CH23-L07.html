CDS View Entity 기초

실습 — 콘서트 CDS 뷰 (ZI_/ZC_)

콘서트앱 — 데이터 모델을 재사용 가능한 CDS 계층으로.

Track 1Chapter 23Lesson 7고급

🎫 흩어진 문법을 하나로

지금까지 배운 문법을 따로 보면 각각은 작아 보인다. 그러나 실무 가치는 조합에서 나온다. 예매자 정훈영이 보는 공연 목록 하나에도 이 계층이 전부 깔린다. 콘서트 앱의 테이블 위에 ZI_Concert·ZI_Perf·ZC_Concert를 만들면 이런 변화가 생긴다.

  • 프로그램은 테이블 대신 CDS entity를 읽는다.
  • 공연과 회차의 관계는 association으로 모델에 남는다.
  • 화면 목록에 필요한 필드는 **소비 뷰(ZC_)**에서 고른다.
  • UI 주석은 Metadata Extension으로 분리한다.
  • 권한이 필요하면 DCL로 row-level filter를 붙인다(뷰마다 각각).

이 실습은 RAP 아키텍처 개요로 가기 전의 마지막 기반 작업이다. RAP를 만들지는 않지만, RAP가 올라갈 수 있는 읽기 모델의 뼈대를 만든다.

순서대로 만들기

활성화에는 순서가 있다. ZI_Concert가 ZI_Perf를 association target으로 참조하므로, target이 먼저 있어야 한다.

먼저 회차 기반 뷰를 만든다.

@AccessControl.authorizationCheck: #NOT_REQUIRED
define view entity ZI_Concert
  as select from zconcert
  association [0..*] to ZI_Perf as _Perf
    on $projection.concert_id = _Perf.concert_id
{
  key concert_id,
      artist,
      venue,
      capacity,
      _Perf
}

그 위에 소비용 뷰를 기반 위에 얹어(nesting) 만든다. extension을 받을 수 있게 @Metadata.allowExtensions: true도 켠다.

마지막으로 UI 목록 주석을 Metadata Extension(DDLX)으로 분리한다(Metadata Extension 기초).

annotate entity ZC_Concert with

이 실습에서는 잔여석 나 예매 수 계산을 구현하지 않는다. "공연과 회차가 관계로 연결되었다"는 기반까지만 완성한다.

직접 빌드해 보기

아래 빌더에서 순서대로 만들어 보자. ZI_Perf를 건너뛰고 ZI_Concert를 먼저 만들면 어떤 오류가 나는지도 꼭 확인하자(활성화 순서 무결성).

확인 체크리스트

점검 항목 기대 결과
ZI_Perf 활성화 회차 키 concert_id·perf_no가 보인다.
ZI_Concert 활성화 _Perf association이 있고, source concert_id가 select list에 있다.
ZC_Concert 활성화 소비 뷰가 필요한 필드와 _Perf를 노출한다.
Metadata Extension 활성화 @UI.lineItem이 DDLX에 있고 본문은 UI 주석으로 오염되지 않는다.
ABAP SQL 조회 프로그램이 ZC_Concert를 데이터 소스로 읽는다.

흔한 실수와 점검 🚧

  • 활성화 순서 무시 → ZI_Concert가 ZI_Perf를 참조하므로 target이 먼저 존재해야 한다.
  • 소비 뷰에서 association 빠뜨림 → 소비자가 회차 경로를 따라가야 한다면 ZC_Concert에서도 _Perf를 노출해야 한다(빼는 것도 설계일 수 있으니 실수인지 의도인지 구분).
  • @Metadata.allowExtensions: true 빠뜨림 → DDLX만 만들어서는 붙지 않는다.
  • 기반에만 권한 걸고 안심 → DCL은 위 계층으로 되지 않는다(DCL 개요). ZC_Concert를 보호하려면 그 뷰에도 access control을 따로 건다.
  • RAP 내용 섞기 → 이 챕터의 최종 산출물은 CDS 뷰와 metadata extension이다. RAP의 동작 정의(behavior)·서비스 바인딩·액션은 작성하지 않는다.
  • 잔여석 계산을 성급히 → 단순해 보여도 예매 상태·취소··권한·성능을 함께 봐야 한다. 여기서는 설계만 한다.

콘서트 앱의 읽기 모델을 CDS로 끌어올렸다. 이제 이 기반 위에 **트랜잭션 서비스(RAP)**를 얹어 만들고·바꾸고· 저장하는 단계로 → RAP 아키텍처 개요.
