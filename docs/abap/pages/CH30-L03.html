BAPI가 없을 때 — 화면을 자동으로

표준 BAPI가 없는 트랜잭션은, 사용자가 화면에 입력하는 과정을 흉내 내 자동 등록한다. 그게 (Batch Data Communication)다. 화면·필드·값을 BDCDATA에 채워 트랜잭션을 돌린다.

DATA lt_bdc TYPE TABLE OF bdcdata.

" 화면·필드·값을 차곡차곡(헬퍼 FORM으로)
PERFORM fill USING 'X' 'SAPMXXXX' '0100'.   " 화면
PERFORM fill USING ' ' 'FIELD-X'  '값'.      " 필드

CALL TRANSACTION 'XX01' USING lt_bdc
  MODE 'N' UPDATE 'S'                         " N=무화면, S=동기
  MESSAGES INTO lt_msg.
  • MODE: A(전체화면)·E(오류만)·N(무화면, 배치).
  • 결과 메시지를 받아 성공/실패를 판정(L05의 재처리와 결합).
  • (실무) MODE/UPDATE를 일일이 적는 대신 OPTIONS FROM <옵션구조체>(ctu_params 타입) 하나로 화면모드·갱신 등을 한 번에 지정하기도 한다.
  • BAPI가 있으면 BAPI 우선 — BDC는 화면 변경에 취약하다.

두 가지 방식 — Call Transaction vs Session

방식 특징
Call Transaction 즉시 실행·결과 바로 확인(위 예시). 단순·동기.
Session(배치 입력) 큐에 쌓아 두고 SM35에서 실행·재처리. 대량·야간에 안정적.

BDCDATA는 SHDB로 녹화

화면·필드 이름을 손으로 채우긴 번거롭다. SHDB(Transaction Recorder)로 트랜잭션을 한 번 녹화하면 BDCDATA를 채우는 뼈대 코드를 자동 생성해 준다.

흔한 실수와 점검

  • 화면 흐름 변경 → 표준 화면이 바뀌면 가 깨진다. BAPI 가능하면 그쪽.
  • MODE 'A'로 대량 → 배치는 N. A는 디버깅용.

화면 자동 입력을 봤다. 이제 외부 파일(엑셀)을 읽어 들이자 → 다음 레슨.
