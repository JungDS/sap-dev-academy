예매 RAP 런타임 시뮬레이터 (determination · validation · action)

CH24의 조각을 한 흐름으로 모았습니다. 예매를 생성하면 determination이 상태를 채우고 validation이 정원을 검사합니다. 정원을 넘으면 failed/reported로 거부되고, 취소는 action으로 명시 실행돼요.

Behavior Pool 안에서 테이블을 직접 바꾸지 마세요. 직접 DML은 RAP transactional buffer와 save sequence를 우회할 수 있어요 — RAP operation과 EML 흐름으로 설계합니다. 직접 DB 변경·LUW는 다음 트랙의 주제예요. validation은 막는 검사, determination은 값 자동 결정, action은 사용자 명시 실행 — 같은 로직이 아닙니다.
